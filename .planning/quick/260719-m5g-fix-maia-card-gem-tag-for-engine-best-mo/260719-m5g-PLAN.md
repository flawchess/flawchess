# Quick Task 260719-m5g: Maia card gem tag for engine best move before it is played

## Problem

In the Maia "Human Move Probability" card, the engine's best move is NOT marked as a
gem when the user played a NON-best move at that ply — even though the on-board gem
badge DOES show once the move is actually played. Reported on prod game 1247936, move
46: user played Rc7??, engine best was Kd3 (a gem), but the card never marked Kd3.

## Root cause

`qualityBySanWithGem` (`frontend/src/pages/Analysis.tsx` ~1313) tags `reconciledBestSan`
as gem/great only via stored data. The backend (`classify_best_move` /
`_build_best_move_candidates`) writes a `game_best_moves` row ONLY for plies where the
user PLAYED the engine best move. So when the user played a non-best move, the stored
lookup is empty, the guard `nextNode?.san === reconciledBestSan` fails, and the line
~1330 short-circuit `if (gameHasStoredBestMoveData) return qualityBySan;` returns BEFORE
the live `classifyGem` fallback runs. The card therefore never live-classifies the
engine best move. The separate `gemByNode` arrival path (~1906) grades the parent
position on demand and correctly shows the gem once the move is played — hence the
inconsistency.

## Fix (two edits, both inside `qualityBySanWithGem`)

1. Gate the ~1330 short-circuit on `playedBestHere = nextNode?.san === reconciledBestSan`.
   The stored short-circuit is authoritative ONLY when the user played the best move
   (stored data then definitively says whether THAT move was gem/great). When the played
   move was non-best, fall through to the live `classifyGem` fallback for
   `reconciledBestSan`.

2. Change the live fallback's Maia rung from the reactive slider (`selectedElo`) to the
   pinned mover rating `pinnedEloForMover(sideToMoveFromFen(position))`. Aligns the card's
   pre-play gem with the on-board `gemByNode` badge (also pinned) and with the Phase 172 /
   SEED-106 D-01 decision that the gem rung is rating-at-game-time, never the slider. Also
   fixes a pre-existing sideline inconsistency (off-mainline fallback used the slider while
   the arrival badge used pinned). Update deps: add `pinnedEloForMover`, drop `selectedElo`.

## Constraints / non-goals

- No backend / stored-data / gemByNode changes.
- Keep the classifyGem C2 gate (`playedIsBest: bestSan === reconciledBestSan`, 163-REVIEW WR-01).
- Live exploration overlays (Maia chart, WDL bar, FlawChess Engine) keep using `selectedElo`;
  only gem CLASSIFICATION switches to pinned.
- Live fallback stays gem-only (matches gemByNode); `great` remains owned by the stored route.
