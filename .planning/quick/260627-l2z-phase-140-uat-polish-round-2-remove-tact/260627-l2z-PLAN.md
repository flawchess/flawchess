# Quick Task 260627-l2z — Plan

Five UAT items on the `/analysis` full-game board (game mode = `?game_id=X&ply=Y`).
Built on Quick 260627-jqk (`useGameOverlay`, precomputed blue arrow + eval bar).

## Context (from research)

- **Tactic mode entry (`?game_id=X&flaw_ply=Y`) is reachable only by hand-typed URL / tests.**
  No production link generates a `flaw_ply` URL (all "Analyze" buttons use `?ply=`).
  Removal = deleting the `isTacticMode` code paths in `Analysis.tsx`, the
  `TacticModeOverlay.tsx` component, and `Analysis.tactic.test.tsx`.
- `TacticModeOverlay.tsx` exports `buildPvArrow` / `buildRootArrows` / `isBlackToMove`,
  imported by `Analysis.tsx`. `buildPvArrow` (depth-countdown arrow) is reused by item 3,
  so it must be **relocated** to a lib helper before the component is deleted.
  `buildRootArrows` and `isBlackToMove` become dead and are removed.
- Stale arrows: `useStockfishEngine` does not clear `pvLines`/`evalCp`/`evalMate` when the
  analyzed FEN changes — the prior position's PV lingers until the new search reports,
  leaving orphaned grey/blue arrows. Precomputed blue + eval already render immediately
  on the main line via `useGameOverlay`.
- Game-mode "tactic overlay" (`useGameOverlay`) only computes arrows on the **main line**
  (`hasPly = onMain && k>=0`). On a contextual PV sideline (chip-click → `insertPvLine`)
  it has no precomputed data, so the depth-labelled overlay does not render (item 3).
- `tacticNodeColors` (depth-0 punchline coloring) is `isTacticMode`-gated; it must be
  re-derived for the game-mode PV sideline nodes (item 4).
- `boardFlipped` is never auto-set in game mode. Player color = `gameData.user_color`
  (`'white'`/`'black'`); convention elsewhere is `flipped = user_color === 'black'`.
