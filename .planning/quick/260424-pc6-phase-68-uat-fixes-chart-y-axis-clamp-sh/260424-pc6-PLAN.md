<issue_2_diagnosis_hypotheses>
Before coding the fix for Issue 2, the executor must diagnose. Likely causes, ranked:

1. **Render-order layering**: Recharts paints SVG elements in the order of JSX children. The current code renders `<g>` wrappers around `<Area>` AFTER `<ChartTooltip>` but BEFORE the `<Line>`s. If Recharts hoists the `<Line>` paths above the `<g>` in the final SVG (it sometimes reorders based on internal z-index), the bands render but the lines sit on top — which is correct — BUT if ComposedChart's SVG output places the bands AFTER the lines, fine; if before with opacity 0.18, the bands would be visible. Verify by opening devtools.

2. **Ranged Area with null gaps**: Recharts 2.x `<Area dataKey>` with tuple values `[low, high]` and null-valued points mixed in can fail to render any area at all in some configurations — the renderer may short-circuit when it encounters nulls early. Workaround: split into continuous "runs" where the band is active, or set `connectNulls={true}` if the visual gap is acceptable.

3. **`allowDataOverflow={false}` + clamped domain**: with the new 20-80 y-axis, raw values (like 12 or 88) will be clipped. If the band extends outside the clipped range, Recharts may refuse to render any part of the band (not just the clipped portion). Fix: set `allowDataOverflow={true}` OR clamp the band tuples to [20, 80] before passing to the Area.

4. **`<g>` wrapper breaks Recharts' internal ref wiring**: Recharts walks its children to compute axis extents and layout. Wrapping `<Area>` in a plain `<g>` may confuse Recharts' child-scanning logic in some versions, causing the area to not register with the axis and thus never paint. Workaround: drop the `<g>` wrapper, attach `data-testid` to the Area's root SVG group via its `id` or `className`, and let the test assert via `container.querySelector`. (Tests were passing due to jsdom's lenient rendering of the `<g>` tag regardless of whether the nested `<Area>` produced paths.)

Instruct the executor to start with hypothesis 1 (open the page in the dev environment, inspect DOM), then 4, then 3, then 2. Do not guess — pick a hypothesis, confirm with browser devtools or a minimal repro, then apply the targeted fix.
</issue_2_diagnosis_hypotheses>
