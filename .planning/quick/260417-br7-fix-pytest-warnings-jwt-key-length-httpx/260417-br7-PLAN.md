# Quick Task 260417-br7: Fix pytest warnings

## Problem

`uv run pytest` produces two flavors of warning noise across ~75+ tests:

1. **`InsecureKeyLengthWarning`** (PyJWT) — `SECRET_KEY` defaults to `"change-me-in-production"` (23 bytes), below the 32-byte SHA256 recommendation. Fires on every JWT encode/decode in auth/guest/oauth tests.
2. **`DeprecationWarning`** (httpx) — 7 tests pass `cookies={...}` per-request to `client.get()`. httpx deprecates per-request cookies; set on the client instance instead.

Both are fixable at low effort. No suppression needed.

## Tasks

### Task 1: Fix JWT SECRET_KEY length in tests

**File:** `tests/conftest.py`

**Change:** Set `SECRET_KEY` env var to a 64-hex-char (32-byte) value before `app.core.config` is imported. Place the assignment next to the existing `SENTRY_DSN` override (same pattern: must precede `from app.core.config import settings`).

**Verify:** `uv run pytest 2>&1 | grep -c InsecureKeyLengthWarning` returns 0.

### Task 2: Fix httpx per-request cookies deprecation

**Files:**
- `tests/test_oauth_csrf.py` (lines 133, 153, 177)
- `tests/test_guest_google_promotion.py` (lines 365, 424, 454, 490)

**Change:** Move `cookies={...}` from the `client.get(...)` / `client.request(...)` call into the `httpx.AsyncClient(...)` constructor. Each AsyncClient in these tests is created fresh per-test, so cookies-on-client is equivalent to cookies-on-request.

**Verify:** `uv run pytest 2>&1 | grep -c DeprecationWarning` returns 0.

## must_haves

- `uv run pytest` passes with zero `InsecureKeyLengthWarning` and zero httpx `DeprecationWarning`
- All 775 tests still pass
- No code behavior changes (cookies still sent, JWT still signs/decodes)
