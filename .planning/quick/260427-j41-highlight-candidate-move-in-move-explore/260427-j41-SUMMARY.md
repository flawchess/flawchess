# Quick task 260427-j41: Highlight candidate move in Move Explorer Summary

When the user clicks the "Moves" link on an OpeningFindingCard (Opening → Insights), the candidate row in MoveExplorer now renders a 4px severity-colored left border, the matching board arrow pulses for ~1.8 s, and the row auto-scrolls into view — closing the deep-link UX gap left after quick-task 260427-h3u.

## What Changed

### 1. `frontend/src/components/board/ChessBoard.tsx`
- `BoardArrow` gains optional `isHighlightPulse?: boolean`.
- `ArrowOverlay` applies `className="animate-arrow-pulse"` to the matching `<path>` and inlines `animationDuration` + `animationIterationCount` from `ARROW_PULSE_DURATION_MS` (600 ms) × `ARROW_PULSE_ITERATIONS` (3) = ~1.8 s pulse window. The static `opacity` prop is left untouched; `animation-fill-mode: forwards` plus a 100 % keyframe stop at 0.75 hand the path back to the static `ARROW_OPACITY` after the pulse.
- No new color constants — pulse only modulates opacity.

### 2. `frontend/src/index.css`
- Added `@keyframes arrow-pulse` (0 % opacity 0.45 → 50 % 1 → 100 % 0.75) and a `.animate-arrow-pulse` helper that sets only `animation-name`, `animation-timing-function: ease-in-out`, and `animation-fill-mode: forwards`. Duration/iterations come from the inline JS-provided style so the constants live in exactly one place.

### 3. `frontend/src/components/move-explorer/MoveExplorer.tsx`
- New props: `highlightedMove?: { san: string; color: string } | null` and `onHighlightConsumed?: () => void`.
- The matching row renders inline `borderLeft: ${HIGHLIGHT_BORDER_WIDTH_PX}px solid ${color}`. Merges with the existing `opacity` style for low-game rows so neither value clobbers the other.
- The matching row also receives a ref so the parent can `scrollIntoView({ block: 'center', behavior: 'smooth' })` exactly once per highlighted SAN (guarded by `lastScrolledSanRef`).
- Highlight-consumption signal lives in a `useEffect` keyed on `position`/`moves`/`highlightedMove` so the React 19 dev double-render does not double-fire `onHighlightConsumed`. Row clicks signal inline in `handleRowClick` after the existing branches.
- Existing data-testid `move-explorer-row-${san}` is reused — the highlighted row is the same `<tr>`, no NEW interactive element added.

### 4. `frontend/src/pages/Openings.tsx`
- New state: `const [highlightedMove, setHighlightedMove] = useState<{ san: string; color: string } | null>(null)`.
- `handleOpenFinding` now sets the highlight to `{ san: finding.candidate_move_san, color: getSeverityBorderColor(finding.classification, finding.severity) }` BEFORE `navigate('/openings/explorer')`, so MoveExplorer paints the highlight on its first frame after the route change.
- `boardArrows` useMemo decorates the matching arrow with `isHighlightPulse: true` (color unchanged — see decision above) and adds `highlightedMove` to its dependency list.
- New `<MoveExplorer>` props: `highlightedMove={highlightedMove}` and `onHighlightConsumed={() => setHighlightedMove(null)}`.
- New derived-state block `prevTabForHighlight` clears the highlight when `activeTab !== 'explorer'`, mirroring the existing `prevTab` pattern.
- Imports `getSeverityBorderColor` directly from `@/lib/openingInsights`.

## Desktop + Mobile Coverage

`MoveExplorer` is mounted exactly once via the shared `moveExplorerContent` variable — both the desktop sidebar layout and the mobile drawer layout reuse the same instance. No duplication needed. The `highlightedMove` prop and the row's inline border-left work identically across breakpoints.

## Clear-Trigger Decision Matrix

| Trigger | Source | Mechanism |
|---|---|---|
| Position changes (board move played, navigation, bookmark load) | `position` prop | `useEffect` in MoveExplorer fires `onHighlightConsumed` |
| Moves array reference changes (filter-driven query refetch) | `moves` prop reference | Same `useEffect`, comparing refs in refs |
| Any row clicked in MoveExplorer | `handleRowClick` | Inline `onHighlightConsumed?.()` after existing branches |
| Tab change away from `/openings/explorer` | `activeTab` derived from `location.pathname` | `prevTabForHighlight` block in Openings.tsx |

All four converge on the same parent setter `setHighlightedMove(null)`. Idempotent: calling it twice is harmless.
