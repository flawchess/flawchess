<objective>
Refactor `query_opening_transitions` from a three-layer SQL construction (standard-start subquery JOIN + window-function CTE + outer GROUP-BY) to a single flat aggregation. Move `entry_san_sequence` resolution and `resulting_full_hash` derivation into the Python pipeline.

Purpose: Eliminate the structural complexity that caused the planner to misestimate join cardinality (debug session `.planning/debug/opening-insights-timeout-u84.md`). The Alembic migration `e925558020b9` (extended statistics) is the safety net; this refactor removes the *cause* — a query shape simple enough that the planner doesn't need the hint.

Output: A flat aggregation that returns one row per `(entry_hash, move_san)` with sufficient information for the service layer to reconstruct everything the previous shape returned, plus a small repository helper to batch-resolve SAN prefixes. No behavior change at the HTTP boundary.

Note: This is a quick task (single feature, single branch `simplify-opening-insights`, single commit per atomic task). No HTTP contract changes — frontend untouched.
</objective>
