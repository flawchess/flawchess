# Quick Task 260616-rm6: Fix the Quick Scan and platform-count counter lag

## Problem

During a large active import, three indicators on the Import page move at three
different cadences from three different sources, so the slower two visibly lag
the fast one and read as "stuck":

| Indicator | Source | Cadence |
|---|---|---|
| "X fetched, Y saved" | `useImportPolling` → import-job counter | 2s |
| "N games" header (per platform) | `useUserProfile` → live `COUNT(*)` on `games` | 5s |
| "Quick Scan: X / Y" banner | `useEvalCoverage` → `/imports/eval-coverage` | 3s self-poll + 5s invalidation |

Observed on prod during the 2026-06-16 dual 20k import: the header showed
"9,030 games" while "saved" was 14,400, and Quick Scan appeared frozen until a
reload. Root cause is NOT missing invalidation (the page already invalidates
`userProfile` / `gameCount` / `eval-coverage` every 5s) — it's the cadence and
source mismatch. The header count is a backend `COUNT(*)` that legitimately
trails the job's optimistic `games_imported`, and the eval-coverage self-poll can
briefly stop (refetchInterval returns `false`) when `pct_complete` momentarily
hits 100 with the helper workers keeping evals in lockstep, leaving the 5s
invalidation as the only rescue.

## Decision (user, 2026-06-16)

**Drive the header from the live job** — frontend-only, lowest risk. During an
active import, source the per-platform "N games" header from the live import-job
`games_imported` (already polled every 2s) so it climbs in lockstep with "saved".
Also tighten the Quick Scan rescue cadence so the banner doesn't visibly stall.

Out of scope: backend `COUNT(*)` visibility/latency investigation (deferred).

## Tasks

### Task 1 — Drive the per-platform header count from the live import job

**File:** `frontend/src/pages/Import.tsx`

- Add an optional `onProgress?: (platform: string, gamesImported: number) => void`
  prop to `ImportProgressBar`. In a `useEffect` keyed on `data?.games_imported`
  and `isActive`, report `data.platform` + `data.games_imported` up while the job
  is active. (The hook runs before the `platformFilter` early-return, so the
  duplicate per-platform-block instances all report the same value — deduped by
  the parent's set-state guard. Same `['import', jobId]` query → one request.)
- In `ImportPage`, add `liveImported` state (`Record<string, number>`) and a
  memoized `handleProgress` callback (`useCallback`) that stores the max-seen
  value per platform. Pass it to every `ImportProgressBar`.
- Compute displayed counts:
  `displayChessComCount = Math.max(profile.chess_com_game_count, liveImported['chess.com'] ?? 0)`
  (and lichess equivalent). `Math.max` keeps incremental syncs correct (library
  count dominates the small new-games delta) while first imports show the
  live-climbing job counter. Never displays a number below the true committed
  count.
- Use the computed counts in the two header `<span>`s (replace the bare
  `profile.chess_com_game_count` / `profile.lichess_game_count`).
- Respect `noUncheckedIndexedAccess`: narrow the `Record` access with `?? 0`.

**verify:** With no active job, header shows `profile.*_game_count` (unchanged —
existing `Import.stateMachine` test renders with `activeJobIds={[]}` and expects
profile counts). With an active job whose `games_imported` exceeds the profile
count, header shows `games_imported`.

**done:** Header count climbs at the 2s import-poll cadence during a first import.

### Task 2 — Tighten the Quick Scan rescue cadence

**File:** `frontend/src/pages/Import.tsx`

- Lower `GAME_COUNT_REFRESH_INTERVAL_MS` from `5000` to `3000` so the
  `['imports', 'eval-coverage']` invalidation (the rescue for the self-poll's
  pct=100 stop) fires at the same 3s cadence as the eval-coverage self-poll,
  halving the worst-case Quick Scan stall window (5s → 3s).
- Added load is modest (one indexed COUNT + one eval-coverage fetch per 3s during
  import); the 2026-06-16 stress test showed the DB comfortably handling these at
  5s with the box at ~50% memory. The job-driven header (Task 1) is the primary
  smoothness mechanism, so this is a robustness tweak, not the load-bearing fix.

**verify:** `GAME_COUNT_REFRESH_INTERVAL_MS === 3000`; the readiness-invalidation
contract test (`Import.stateMachine`) still passes (the `['imports','readiness']`
call is untouched).

**done:** Quick Scan banner refresh interval ≤ 3s during active import.
