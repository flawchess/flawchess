# Quick Task 260701-93s: Remove tier-4 blob fallthrough from `claim_eval_job(scope="idle")`

Implements the **corrected** decided fix in
`.planning/seeds/SEED-072-tier3-ttl-claim-double-claim-waste.md`.

## Problem (verified against deployed code)

Phase 145 wired tier-4 (flaw-blob gating backfill) into `claim_eval_job(scope="idle")`
as a fallthrough after tier-3. Phase 146 removed the inline server walk that used to
fill blobs on the `/submit` path (`_apply_submit` now forces `blob_map={}`) but left the
tier-4 routing in the idle `/lease` path. Result on prod: `/lease?scope=idle` never
returns 204 (there is always a ~3.3M NULL-blob tier-4 backlog), so remote workers full-ply
**re-evaluate already-complete games** (~5:1 submit:completion), never fall through to their
dedicated rung-4 `/flaw-blob-lease`, and the gating backfill is starved.

## Fix

In `app/services/eval_queue_service.py::claim_eval_job`, delete the "Tier-3 empty → try
tier-4 blob-backfill" block inside the `if scope == "idle":` branch so that when
`_claim_tier3_derived` returns `None`, the idle scope returns `None` → HTTP 204. Remote
workers then fall to `/flaw-blob-lease` (which calls `_claim_tier4_blob` directly), the
only post-146 path that actually writes blobs.

**Bundled `scope=None` path is intentionally retained** with a clarifying comment: its sole
consumer is the in-process server-pool drain (`eval_drain.py::run_one_full_eval_tick`),
which DOES write blobs via the MultiPV-2 pass (`_build_flaw_multipv2_blobs` →
`_run_multipv2_pass`), so tier-4 drains correctly there (not the broken `/submit` path).
