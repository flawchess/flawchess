# Quick Task 260318-w15: Fix Google SSO Requiring Double-Click to Sign In

**One-liner:** 401 interceptor guarded against /login page and /auth/ routes; Google sign-in gets one silent retry with 500ms delay before surfacing error toast.

## What Was Done

Two files patched to fix Google SSO requiring a double-click:

**Root cause:** The axios 401 interceptor was unconditionally clearing localStorage, wiping the query cache, and redirecting to `/login` on any 401 response — including the `/auth/google/authorize` request during the OAuth flow. This cancelled the in-flight auth request and reset the UI, requiring a second click.

### Task 1: Guard 401 interceptor (client.ts)

Added two guards before the redirect/cache-clear block:
- `onLoginPage`: `window.location.pathname === '/login'` — avoids page reload that cancels in-flight requests when already on login page
- `isAuthRoute`: request URL starts with `/auth/` — auth endpoints handle their own states; global interceptor must not interfere

Both guards still call `Promise.reject(error)` so calling code can handle errors normally.

### Task 2: Retry + error logging in Google sign-in (LoginForm.tsx)

- Added `console.error('Google sign-in failed:', error)` for debugging
- Converted `handleGoogleSignIn()` to accept `attempt = 0` parameter
- On first failure: waits 500ms, retries with `attempt + 1`
- On second failure: shows existing error toast, resets `googleLoading`
- Fixed `onClick` to use `() => handleGoogleSignIn()` wrapper (prevents MouseEvent being passed as `attempt` — TypeScript caught this during build)
