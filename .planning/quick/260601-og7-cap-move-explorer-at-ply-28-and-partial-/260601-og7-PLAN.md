<objective>
Cap the move explorer at ply 28 and rebuild the three Zobrist-hash indexes on `game_positions`
as PARTIAL (`WHERE ply <= 28`), reclaiming ~3 GB of index footprint with zero data loss and no
table rewrite. Implements SEED-033 (all decisions settled in the seed — read it first).

Purpose: The three hash indexes (`ix_gp_user_full_hash_move_san`, `ix_gp_user_white_hash`,
`ix_gp_user_black_hash`) total ~4.8 GB and only matter for opening-depth position matching.
A single shared cap value (`MAX_EXPLORER_PLY = 28`) ties the explorer's reachable depth to the
index boundary so the win is unconditional: if the explorer can never reach past the cap, no
query ever looks up a hash past it.

Output: shared backend constant, non-transactional CONCURRENTLY rebuild migration, `AND ply <= MAX_EXPLORER_PLY`
predicates on the capped hash queries, and a hard frontend explorer cap mirroring the constant.
</objective>

upgrade(): for each of the three hash indexes, CREATE the new partial index CONCURRENTLY, then DROP

temp name then it cannot be renamed concurrently — instead use this safe ordering per index:

  2. `op.drop_index(<original_name>, table_name="game_positions", postgresql_concurrently=True)`
  3. `op.execute('ALTER INDEX "<original_name>_partial" RENAME TO "<original_name>"')`  (rename is a fast catalog-only metadata op, safe outside CONCURRENTLY; the model in Task 1 keeps the original names so the final names must match)

restores the pre-migration state exactly (full indexes, original names).
