**1b. Create `frontend/src/prerender.tsx`**

This is the SSR entry point that `vite-prerender-plugin` calls at build time. It uses `renderToString` from `react-dom/server` and `StaticRouter` from `react-router-dom` for SSR-compatible routing.

```tsx
import { renderToString } from 'react-dom/server';
import { StaticRouter } from 'react-router-dom';
import { Routes, Route } from 'react-router-dom';
import { HomePageContent } from './pages/Home';
import { PrivacyPage } from './pages/Privacy';

export async function prerender(data: { url: string }) {
  const html = renderToString(
    <StaticRouter location={data.url}>
      <Routes>
        <Route path="/" element={<HomePageContent />} />
        <Route path="/privacy" element={<PrivacyPage />} />
      </Routes>
    </StaticRouter>
  );
  return { html };
}
```

Key points:
- Uses `StaticRouter` (not `BrowserRouter`) for SSR — `StaticRouter` is exported from `react-router-dom` in v7 (previously from `react-router-dom/server` in v6, verify the correct import)
- Renders only the public page components, NOT the full App (avoids auth, query client, Sentry)
- `PrivacyPage` uses `useEffect` for title — this is fine, `useEffect` is a no-op during SSR
- The `prerender` function is async as required by the plugin API

**Important SSR compatibility check for StaticRouter:**
- In react-router-dom v7, `StaticRouter` may need to be imported from `react-router-dom/server` or `react-router` — check the actual export. If not available, install `react-router` as a dependency and import from `react-router/server`.
- If `StaticRouter` is not available, an alternative is to render the components without a router, using a simple conditional on `data.url` to pick the right component. The Link components in PublicHeader and footers would need to render as plain `<a>` tags in that case — but try StaticRouter first.

**1c. Configure plugin in `frontend/vite.config.ts`**

Add the plugin to the Vite config:

```ts
import { vitePrerenderPlugin } from 'vite-prerender-plugin'
```

Add to plugins array (AFTER react() and tailwindcss(), BEFORE VitePWA()):

```ts
vitePrerenderPlugin({
  renderTarget: '#root',
  prerenderScript: path.resolve(__dirname, 'src/prerender.tsx'),
  additionalPrerenderRoutes: ['/privacy'],
}),
```

Options explained:
- `renderTarget: '#root'` — inject prerendered HTML into the `<div id="root">` element
- `prerenderScript` — absolute path to the prerender entry point
- `additionalPrerenderRoutes: ['/privacy']` — `/` is discovered automatically from index.html, but `/privacy` needs to be explicitly listed since it's not linked from the entry HTML (it IS linked from the homepage footer, but the plugin may not discover it automatically — safer to list explicitly)

**1d. No changes to main.tsx**

Keep `createRoot` (not `hydrateRoot`). When React loads client-side, it replaces the prerendered content. This is simpler than hydration and avoids hydration mismatches for authenticated users who get redirected away from the homepage.

The brief moment where prerendered HTML shows before React takes over is a feature, not a bug — users see content instantly instead of a blank page.
