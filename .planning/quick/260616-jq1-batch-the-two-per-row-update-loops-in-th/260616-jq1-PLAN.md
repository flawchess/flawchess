- **Loop A — main eval write** (`_apply_full_eval_results`, the `for target in targets:` write loop, ~lines 445-498): one `UPDATE game_positions SET ...` per ply, ~40-80 rows/game. This is the larger N+1.

Output: a batched eval-drain write path; existing eval-drain tests still pass; commit references `Fixes FLAWCHESS-6B`.

<key_facts>
GamePosition columns touched: `eval_cp` / `eval_mate` are `SmallInteger` (nullable), `best_move` is `String(5)` (nullable), `pv` is `Text` (nullable, unbounded — this is the oversized-string risk in Loop B). Table is `game_positions`. The write key per row is `(game_id, ply)`.

`bindparam`, `select`, `update`, and `import sqlalchemy as sa` are already imported at the top of eval_drain.py.

Loop A column-set logic that MUST be reproduced exactly (per row, when `not is_terminal`):
- `is_lichess_eval_game=True`: write `best_move` ONLY, and ONLY when `best_move is not None`. Never touch eval_cp/eval_mate (lichess %evals are authoritative). If best_move is None → no UPDATE for that row.
- `is_lichess_eval_game=False` (engine game):
  - `eval_cp, eval_mate = _post_move_eval(pos_eval, ply)` (the SEED-044 +1 shift — already computed; do not touch it).
  - `best_move` written whenever `best_move is not None`, INDEPENDENT of eval.
  - If `eval_cp is None and eval_mate is None`: leave eval NULL; increment `failed_ply_count` UNLESS `target.ends_game` (SEED-049 terminal exemption — no increment).
  - Else: write both `eval_cp` and `eval_mate`.
  - A row with neither best_move nor an eval gets no UPDATE (the current `if values:` guard).
- Terminal targets (`target.is_terminal`) are skipped entirely (eval-only donor, no row).

`failed_ply_count` accounting is pure Python over `pos_eval` / `best_move_by_ply` / `target.ends_game` — it does NOT depend on how the UPDATE is issued. Keep computing it in the same per-target pass; only the DB write is batched. This is the safest seam: compute the full list of `(ply, values_dict)` write-rows AND `failed_ply_count` in one pass (no DB I/O), then emit the batched UPDATE(s) at the end.

Loop B fault tolerance that MUST be preserved: a single bad/oversized `pv` row must not abort the flaw rows + oracle counts already written, nor the other PV rows. The current code catches per-row exceptions and Sentry-captures. The real failure mode is an oversized pv string. Pre-filter is the right move: validate/collect surviving `(pv_ply, pv_string)` pairs up front, then ONE batched UPDATE. Keep a Sentry capture path for any skipped row (`set_context` with game_id/pv_ply, `set_tag("source", ...)`, `capture_exception` — NEVER embed variables in the message string, per CLAUDE.md Sentry rules).
