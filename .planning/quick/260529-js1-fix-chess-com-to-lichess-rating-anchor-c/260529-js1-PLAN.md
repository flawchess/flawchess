<objective>
Fix the chess.comâ†’Lichess rating-anchor conversion bug in the `user_rating_anchors`
SQL pipeline. The blended-anchor SQL (`_per_user_cte_median_anchor_blended`)
converts every chess.com game's native rating by matching it against a lookup
table of chess.com **Blitz** anchors, regardless of the game's actual time
control. For rapid/bullet buckets the native rating is a chess.com rapid/bullet
rating, so the SQL matches the wrong scale and skips the intra-TC inversion that
the canonical converter `convert_chesscom_to_lichess` applies. Measured impact:
rapid/classical anchors inflated (~85 pts for dev user 28 rapid), bullet
deflated, blitz correct.

Fix approach (locked, per task constraints): make `_chesscom_conversion_values_sql`
emit a per-bucket lookup table keyed on **native chess.com rating** for that
bucket's source TC, where each row's lichess-equivalent is produced by the
EXISTING `convert_chesscom_to_lichess` (single source of truth â€” no SQL-side
inversion port, no duplicated interpolation math). The existing nearest-anchor
LATERAL join then matches native ratings against native anchors, which is
correct. This also makes `convert_chesscom_to_lichess` / `_invert_intra_tc` /
`CHESSCOM_INTRA_TC` LIVE code (currently dead outside their own tests).

Bucket â†’ chess.com source-TC mapping (locked): bulletâ†’bullet, blitzâ†’blitz,
rapidâ†’rapid, classicalâ†’rapid (chess.com has no classical category; chess.com
games bucketed `classical` carry rapid ratings â€” chess.com Daily games are
already dropped upstream by the `time_control_str LIKE '1/%'` filter, so the
`classical` bucket only ever sees rapid-rated chess.com games here).

Purpose: correct peer-relative percentile anchors for chess.com users in
non-blitz buckets.
Output: corrected SQL conversion + a pure-Python equivalence test + repaired
existing snapshot-row test.
</objective>
