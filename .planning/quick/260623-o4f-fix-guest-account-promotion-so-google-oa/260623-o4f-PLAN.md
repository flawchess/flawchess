# Quick Task 260623-o4f — Guest promotion safety net for Google OAuth

## Problem

Prod users 198 (guest) + 199 (registered Google) are the same person: the guest
imported 5965 games, then signed up via Google, which minted a NEW account (199)
and re-imported, orphaning the guest. Root cause: guest→registered linkage lives
only in the frontend's choice of OAuth endpoint. `LoginForm.tsx` has no promote
branch (its Google button always hits plain `/auth/google/authorize`), and the
backend `UserManager` does not override `oauth_callback`, so the stock flow
creates a fresh user whenever the Google email matches no existing account (a
guest's `guest_*@guest.local` email never matches).

## Approach

Two layers:

1. **Frontend (close the common path + kill the divergence):** Extract the
   guest-aware Google sign-in into one shared helper used by BOTH `LoginForm` and
   `RegisterForm`, so the two buttons can never drift again. The helper routes to
   `/auth/google/authorize-promote` (with the saved `guest_token` as a Bearer
   header) when a guest session exists, falling back to plain
   `/auth/google/authorize` otherwise.

2. **Backend (server-side safety net):** Make `/auth/google/authorize` itself
   guest-aware via an OPTIONAL authenticated-user dependency. When the caller is
   an active guest, the endpoint builds the promote-audience state JWT (embedding
   `guest_user_id`) and points the Google `redirect_uri` at the existing,
   already-registered `callback-promote` route — so the in-place promotion runs
   no matter which frontend entry point started the flow (Login tab, a
   logged-in guest, or any future button).

**Deviation from the literal ask:** the task text said "have the plain
`/auth/google/callback` promote the guest." Instead we route guests through the
existing tested `callback-promote` handler (reused, not duplicated). Same end
result — a guest is promoted in place — with no new Google redirect URI to
register and no second promotion branch to maintain inside the plain callback.

## Tasks

### Task 1 — Backend: guest-aware `/auth/google/authorize`
- **files:** `app/users.py`, `app/routers/auth.py`
- **action:**
  - `app/users.py`: add `current_active_user_optional = fastapi_users.current_user(active=True, optional=True)`.
  - `app/routers/auth.py`: import `current_active_user_optional`; add it as an
    optional dependency to `google_authorize`. When the resolved user is a guest,
    embed `guest_user_id` + `_OAUTH_PROMOTE_STATE_AUDIENCE` in the state JWT and
    set the redirect to `.../api/auth/google/callback-promote`; otherwise keep the
    existing plain `_OAUTH_STATE_AUDIENCE` + `.../callback` behavior. Add a
    bug-fix comment explaining the orphan it prevents.
- **verify:** `uv run pytest tests/test_guest_google_promotion.py tests/test_oauth_csrf.py -q`
- **done:** A guest's authorize call yields a `callback-promote` redirect_uri and
  a promote-audience state with `guest_user_id`; anonymous/non-guest callers are
  unchanged.

### Task 2 — Frontend: shared guest-aware Google sign-in helper

  `guest_token` branch (authorize-promote with Bearer header, fall back to plain

- **verify:** `cd frontend && npm run lint && npx tsc -b && npm test -- --run`
- **done:** Both Google buttons go through the helper; no behavioral regression
  for registered users; knip clean.
