Two pieces of Phase 140 UAT round-6 feedback:

**Item 1 — Free-move classification icon in the move list.**
When playing a free move on the analysis board and the live engine evaluates it as a
blunder/mistake, the board already paints the severity glyph (item 4, commit e116912c).
The MOVE LIST (VariationTree) does not — it only renders glyphs from the precomputed
`flawMarkerByNodeId` (game-mode markers), which excludes free-play nodes. Surface the

**Item 2 — Revert the missed-tactic "ply-1" attribution; show should-have-played as a
violet arrow at the flaw ply.**
Commits `cfaa7856` (board/minimap/eval-tooltip) and `e116912c` (Analysis.tsx
`flawMarkerByNodeId` split + activePvNodeId) shifted the missed-tactic one ply earlier
(onto the decision node). The user calls this conceptually wrong. Root cause: an earlier
change (UAT "thl" item 2, `useGameOverlay.ts:177`) switched the blue best-move arrow from
`best_move[k]` (the move that should have been played INSTEAD of the played move) to
`best_move[k+1]` (the FOLLOWING best move from the displayed position). That switch made
the missed-tactic look misplaced, prompting the ply-1 shift. Revert the ply-1 shift and,
at the flaw ply, draw the should-have-played move as a NEW violet arrow.

## Implementation Decisions (LOCKED)

### Reference design: the flaw cards (FlawCard.tsx) are correct and UNCHANGED
The small flaw cards in the results list show the DECISION position (one ply before the
flaw): blue arrow = the move that should have been played (`FlawCardData.best_move` =
"Engine best move FROM the pre-flaw position"); red/orange arrow = the played blunder/
mistake. This is the ground-truth semantic for "the move that should have been played."
Do NOT change FlawCard.

### Game card (LibraryGameCard) + analysis board (useGameOverlay): post-flaw position
These surfaces are navigable and land you on the position AFTER the flawed move. There:
- **Missed tactic** → draw the should-have-played move (the same move FlawCard's blue
  arrow shows, the missed move) as a NEW **violet** arrow, carrying the **missed-tactic
  depth label**, attributed to the **flaw ply** (reverting the ply-1 shift). The played
  flaw is already shown via the severity glyph + colored last-move square.
- **Allowed tactic** → NO violet arrow. The existing blue following-best arrow already
  points at the allowed-tactic line (the opponent's refutation continuation). Unchanged.
- **Clean ply** → blue following-best arrow, as today. Unchanged.

### DECISION 1 — Missed-flaw arrows: VIOLET + BLUE (both show)
On a missed-tactic flaw position, show BOTH the new violet should-have-played arrow AND
the existing blue following-best arrow (engine's best continuation from the resulting
position). The missed-depth label moves OFF the blue arrow (revert cfaa7856's labeling of
blue with `missedAhead`) and ONTO the violet arrow at the flaw ply.

### DECISION 2 — Allowed "+1" depth offset: GAME CARD + ANALYSIS BOARD ONLY
`ALLOWED_DECISION_DEPTH_OFFSET` (+1 for allowed orientation) lives in the SHARED
`toDisplayDepthForOrientation` / `tacticDepthBadge` / `resolveVisibleTactic`. Drop the +1
ONLY on the navigable surfaces (LibraryGameCard, useGameOverlay, VariationTree, EvalChart,
Analysis.tsx) where missed/allowed are no longer co-anchored on one board. KEEP the +1 on
the static FlawCard, where both depths still share the decision board and the anchoring
adjustment remains correct. Parameterize the badge/display function by an `anchored`
flag. Do NOT touch the backend filter mirror in `library_repository.py` (depth filter
stays as-is).

### Violet arrow color + DECISION 3: missed = violet everywhere
"Violet" does not exist yet (best-move arrow is blue-500; missed label is light-blue hue
258). Add a NEW violet/purple arrow color constant (+ matching light-violet label color)
in `frontend/src/lib/theme.ts`. It must read as visually distinct from the blue best-move
and grey 2nd-best arrows — it signals a counterfactual "the move you should have played."

**DECISION 3 (user follow-up):** Unify the MISSED-tactic color to violet. Recolor the
`TAC_MISSED` family in theme.ts — `TAC_MISSED`, `TAC_MISSED_BG`, `TAC_MISSED_BORDER`,
`TAC_MISSED_LABEL` — from light-blue (hue 258) to violet/purple (hue ~295-305, keep the
existing lightness/chroma relationships so it stays readable on the dark UI). This makes
the **missed-tactic tags/chips** (e.g. the "missed: hanging…" chip) AND the **missed-tactic
entries in the EvalChart tooltip** render violet, matching the new should-have-played
arrow (which should use this same violet family — no separate hue). The allowed family
(`TAC_ALLOWED*`, light red) is unchanged. Scope note: this recolors EVERY missed-tactic
surface (flaw cards' missed chips included) since the color family is shared — that is
intended (missed = violet is now the unified semantic). The flaw card's should-have-played
ARROW itself stays `BEST_MOVE_ARROW` (blue); only its depth label follows the violet
`TAC_MISSED_LABEL`. Do not introduce a second, separate violet constant — reuse the
recolored `TAC_MISSED` family for the new arrow so there is one violet.

### DECISION 4 (user follow-up): allowed → distinct magenta-purple
Recolor the `TAC_ALLOWED` family — `TAC_ALLOWED`, `TAC_ALLOWED_BG`, `TAC_ALLOWED_BORDER`,
`TAC_ALLOWED_LABEL` — from light-red (hue 25) to a red-leaning **magenta-purple (hue ~330)**,
preserving each constant's lightness/chroma. This recolors the allowed tactic tags/chips,
the allowed arrows (flaw-card red flaw arrow, PV lead-in arrow), the allowed depth numbers,
and the allowed eval-chart tooltip entries (all driven by the shared `TAC_ALLOWED*` family).
To keep missed vs allowed clearly distinguishable (they are semantic opposites), set the
MISSED family hue to the blue-violet side (~285-290) rather than ~300, so missed (~287) and
allowed (~330) are ~40° apart. This OVERRIDES the "~295-305" figure in DECISION 3 for the
missed family. **Apply DECISION 4 as a follow-up AFTER the main executor completes** (it is
mid-flight editing theme.ts for the missed recolor; do not edit theme.ts concurrently). Then
re-run the full frontend gate (`npm run lint && npm test -- --run && npx tsc -b`); update any
color snapshot expectations to the new hues (allowed is no longer red).

### DECISION 5 (user follow-up): more distinct colors — magenta (missed) + teal (allowed)
DECISION 4's blue-violet/magenta-purple pairing read as "two purples." Pushed the two
orientations into the only two FREE zones of the board palette (everything else is taken:
red 25 blunder, orange 55 mistake, yellow inaccuracy, green 145 clean-move, blue 260
best-move arrow): **missed = magenta (hue 330)**, **allowed = teal (hue 200)** — ~130°
apart. Critically, magenta sits ~70° off the blue best-move arrow that renders BESIDE the
missed should-have-played arrow (DECISION 1), so they no longer read as two blues (violet
287 was only ~27° from that blue). Recolor `TAC_MISSED*` → hue 330 and `TAC_ALLOWED*` →
hue 200 (lightness/chroma unchanged, labels too). This SUPERSEDES DECISION 3's violet and
DECISION 4's hues. Applied + gate re-run (1214 tests green, tsc/lint/knip clean).

## Specific Ideas / Risk

### CRITICAL: ply-indexing off-by-one is the recurring failure mode
The `best_move` ply indexing has caused repeated off-by-one bugs across UAT rounds (thl
item 2, w8k item 5). `EvalPoint.best_move` is documented "best move FROM this position",
while `FlawCardData.best_move` is documented "best move FROM the pre-flaw position" — and
`useGameOverlay` currently sources the blue arrow from `bestMoveByPly.get(k+1)`. Do NOT
assume the index. The should-have-played (violet) move on the game card / analysis board
must render the SAME move the FlawCard blue arrow shows for that flaw. VERIFY parity
empirically against a known missed-tactic flaw (e.g. compare the violet arrow's from/to
on the analysis board against the FlawCard for the same game+ply) before declaring done.
