<objective>
Promote the lichess-eval residual lane in `_claim_tier3_derived` into the tier-3
recency-weighted Efraimidis–Spirakis (ES) user lottery, so lichess-eval games
(`full_pv_completed_at IS NULL AND lichess_evals_at IS NOT NULL`) compete for draws
in the same lottery as needs-engine games instead of running ONLY when the
needs-engine backlog is globally empty.

Motivation (verified against prod 2026-07-23): user 235's 63.5k needs-engine
import (~5 days to drain at ~545 games/h) currently starves user 28's 3 returning
lichess-eval games — the residual lane only fires when NO needs-engine candidate
exists anywhere. The τ½=1d recency-weighted user lottery was designed precisely so
a returning user wins ~30–40% of draws against a mass importer, but the lichess-eval
population never entered it. This is a DELIBERATE precedence change relative to
174-07/SEED-109 (which broadened the residual predicate while explicitly preserving
"same precedence, no new starvation dynamic").

Purpose: fix lichess-eval starvation by unifying the two populations into one
lottery; remove the now-redundant residual fallback lane.
Output: a rewritten `_claim_tier3_derived` (unified Step-1/Step-2, no residual
lane) plus updated/added tests. No migration (existing partial indexes cover both
predicate branches), no new deps, no schema change, no frontend change.
</objective>

<design_notes>
INDEX DECISION (constraint #4, resolved — NO migration): both branches of the
unified predicate are already backed by partial indexes on `games(user_id)`:
  - `ix_games_needs_engine_full_evals` — WHERE full_evals_completed_at IS NULL AND lichess_evals_at IS NULL (Phase 119)
  - `ix_games_lichess_pv_backfill_pending` — WHERE lichess_evals_at IS NOT NULL AND full_pv_completed_at IS NULL (174-07)
Expressing Step-1's candidate as an OR of the two branch predicates lets PostgreSQL
BitmapOr the two partial indexes. Do NOT add a new index; do NOT combine into a
single broader index (the two partials already exist and stay small via
self-termination). If EXPLAIN during manual verification shows a seq scan, note it
in the SUMMARY — but the residual lane already ran the lichess branch globally today
with these indexes, so the union is not a regression.

GUEST ASYMMETRY (constraint #2, quick 260719-fsz): guests eligible ONLY via
lichess-eval games, never needs-engine games. Express this INSIDE the SQL
predicates (guard the needs-engine branch with the user's is_guest = false),
and pass `include_guests=True` to `_es_weighted_user_pick` so the outer
`u.is_guest = false AND` filter is dropped and the per-branch guard governs.

RESIDUAL REMOVAL: the old residual lane's population (lichess-eval games for all
users incl. guests) is fully subsumed by the unified Step-1/Step-2 union, so the
residual `_es_weighted_game_pick` call and its trailing user-id lookup are removed.
If Step-2 returns None (a race: the picked user's games drained between Step-1 and
Step-2), return None — the next tick re-draws. Do NOT reintroduce a fallback lane.

SECURITY (constraint #3): all new SQL fragments are trusted hardcoded literals
composed into `_es_weighted_user_pick` / `_es_weighted_game_pick`; only :picked_user,
τ, and floor VALUES are bound as params. No f-string-interpolated values.
</design_notes>
