## Approach & residual risk (Endgames now in scope)

The user accepted the larger scope. Investigation result: **no eval data backfill
is required.** The endgame span-entry rows already carry `eval_cp` / `eval_mate`
(`query_endgame_entry_rows` projection, lines ~251-252), and the aggregator
already iterates them. The Score/Eval values are produced by the SAME shared
helpers the openings path uses — `_build_wdl_stats` (Wilson score CI +
confidence + p_value + last_played_at) and `compute_eval_confidence_bucket`
(eval mean / CI / confidence / p_value). No new statistical code is written.

Residual nuances (must be honored, not silently resolved):
1. **Endgame eval baseline is color-agnostic.** Endgame stats apply NO color
   filter (D-02, both colors mixed per category). Openings picks the per-color
   baseline by `request.color`. For endgames there is no single color, so use
   `EVAL_BASELINE_PAWNS_WHITE` (0.25) — this matches the existing
   "color is None → white baseline" convention already in
   `openings_service.analyze` line ~228. The Eval bullet baseline tick will
   therefore always sit at +0.25 on the Endgames Games subtab. This is a
   deliberate, documented choice; note it in the SUMMARY.
2. **`score_p_value` backward compat.** `EndgameCategoryStats.score_p_value`
   already exists and is consumed by `EndgameTypeCard` (Stats subtab). Do NOT
   change its gating semantics: keep `score_p_value = p_value if total >=
   PVALUE_RELIABILITY_MIN_N else None`. The new `p_value` field (ungated, from
   `_build_wdl_stats`) is additive and separate; the panel reads `p_value` +
   `confidence` + `score`, the existing card keeps reading `score_p_value`.
3. **Eval is per-span, score is per-game-per-class.** The WDL/score counts
   aggregate one row per (game, endgame_class) span (a game spanning two classes
   counts in both — existing D-02 behavior). The eval mean uses the same span
   rows' entry eval (user-perspective cp, mate/NULL excluded, `|cp| >= 2000`
   trimmed). This matches how `conversion`/`type_achievable_score_gap` already
   aggregate, so it is consistent with the rest of the card — but it is NOT
   identical to openings' "one MG-entry eval per matched position." Acceptable
   and documented; the bullet semantics ("eval at endgame entry for this type")
   are correct for the endgame context.

No item is dropped. If, during execution, the eval data turns out to be absent
for a meaningful share of endgame rows, do NOT silently omit the Eval row —
keep it (the panel already em-dashes when `eval_n == 0`) and note the coverage
gap in the SUMMARY.
</risk>
