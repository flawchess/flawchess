## Design Resolution: who computes PV-continuation-node second-best (Option B)

RESEARCH Open Question #1 flagged a real tension. The Phase-141 gate
(`apply_forcing_line_filter`, already shipped) walks a multi-node `Sequence[PvNode]` and
requires **>=2 solver nodes** — so each blob needs best-vs-second data at every node *along
the refutation / best-move PV continuation*, not just at the flaw ply. Those continuation
positions are hypothetical (reached by playing the PV moves) and are NOT in
`engine_result_map`. D-01's literal framing ("assembly from engine_result_map", "not new
positions") therefore cannot, by itself, produce gate-usable blobs.

**Resolution: Option B — the server walks each flaw's PV line locally.**

Rationale:
1. **Correctness is non-negotiable.** A node-0-only blob always fails the gate's one-mover
   discard (`len(solver_nodes) < 2`). The blob MUST be multi-node, so the continuation
   positions MUST be evaluated somewhere.
2. **Option A is infeasible without a contract round-trip.** Workers lease a whole game and
   evaluate per-ply BEFORE classification; they do not know which plies are flaws, so they
   would have to walk every ply's PV (huge wasted compute) or wait for a server round-trip.
   Confining the walk to the few flaw plies, server-side, is far cheaper.
3. **It mirrors an accepted pattern.** `_fill_engine_game_flaw_pvs` (SEED-056, eval_drain.py:991)
   already does targeted local re-evaluation of flaw-adjacent plies, session-free, on the
   module pool. D-05 already extends that exact pattern to multipv=2 for the eval-gap subset.
   Option B is the natural generalization — one mechanism, not two.
4. **D-01 / D-03 are still honored.** The whole-game per-ply pass still goes to multipv=2
   (no separate worker pass), and the worker-supplied per-ply second-best (D-03 SubmitEval
   fields, Plan 03) is reused for the **root node** of each line — the load-bearing SC3 use
   that keeps the schema non-vestigial. Only the deeper PV-continuation nodes are local.
5. **Cost is bounded and in budget.** Flaw plies are few per game (~2-5); the PV cap is
   `PV_CAP_PLIES` (12); two lines per flaw → at most ~120 local multipv2 evals/game at the
   existing 1M budget (D-06), on the existing module pool (no new pool, RSS unchanged).

Tradeoff acknowledged vs D-01's "reuses the remote fleet / marginal cost" wording: the
deep-node walk runs on the server in both the local-drain and remote-submit paths, so the
fleet only offloads the whole-game per-ply pass, not the continuation-node compute. This is
unavoidable given the locked gate shape; D-06 keeps the budget at 1M and Plan 04's histogram
validates it (raise to 1.5-2M only if the SC4 test fails).

Per-node second-best source:
- **Node 0** of each line (the flaw ply itself / flaw_ply+1 — both real game plies): use
  `second_best_map[ply]` when present (worker- or local-Step-3-supplied), else recompute
  locally (D-05).
- **Nodes 1..N** (continuation positions): always local `evaluate_nodes_multipv2`.
