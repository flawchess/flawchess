<objective>
Upgrade the long-running fleet worker (`scripts/remote_eval_worker.py`) to continuously drain the tier-4 flaw-blob queue — the functional core of this phase. Without it, every game the live `/submit` now defers (Plan 01) stays NULL-blob forever and tactic tags never get gated.

Implements:
- **D-04** — after exhausting tier-1/2/3, the worker polls `/eval/remote/flaw-blob-lease`, evaluates the leased continuation FENs at MultiPV-2, and echoes token + result to `/eval/remote/flaw-blob-submit`. Worker stays token-opaque (D-04a). Server endpoints already exist (Phase 145, reused unchanged).
- **D-03 consequence / Claude's Discretion (RESEARCH-confirmed)** — reduce the full-ply pass from MultiPV-2 to MultiPV-1 (`evaluate_nodes_with_pv`) now that per-ply second-best is dropped from the wire (single-consumer grep in RESEARCH §1). The tier-4 blob rung keeps MultiPV-2.
- **Claude's Discretion (RESEARCH §5)** — lower `HTTP_TIMEOUT_S` from the 120s SEED-071 stopgap back to 30.0 (the live `/submit` now runs no engine; 10x margin over expected p99).

Purpose: Close the offload loop — deferred live games and the corpus backlog drain through the existing Phase-145 tier-4 machinery + D-07 gated retag.

Output: A fleet worker that drains tier-4 continuously, evaluates the full-ply pass ~20% cheaper, and uses the restored 30s timeout.

## Artifacts this phase produces (Plan 02)
- `_eval_flaw_blob_positions(pool, positions) -> list[dict]` — new helper; MultiPV-2 fan-out keyed by token; returns `{token, best_cp, best_mate, second_cp, second_mate, second_uci}`.
- `_handle_flaw_blob_response(client, pool, sf_version, dry_run, loop, blob_lease_resp) -> bool` — new handler mirroring `_handle_entry_ply_response`.
- Modified `_run_cycle` — rung-4 inserted between the tier-3 204 and the sleep; docstring updated.
- Modified `_eval_positions` — `evaluate_nodes_with_pv` (4-tuple), second-best dropped from the full-ply output dict; docstring updated.
- `HTTP_TIMEOUT_S: float = 30.0` (stopgap comment removed).
- No new files, no schema, no new EnginePool, no `STOCKFISH_POOL_SIZE` change.
</objective>
