# Phase 114.1: Code Review Report

**Reviewed:** 2026-06-10
**Depth:** standard
**Files Reviewed:** 23
**Status:** issues_found

## Summary

Phase 114.1 replaces `games.move_count` (full-move count) with `games.ply_count`
(exact half-move count) across the backend write path, all readers/schemas, the
benchmark §5 chapter, and three frontend game cards. I traced every correctness
concern raised in the phase context and found the core migration math, the
backfill identity, the display-equivalence helper, and the §5 FLOOR/CEIL
arithmetic to all be **correct**. No BLOCKER-class defects were found.

Verified correct:

- **Backfill identity.** `process_game_pgn` writes `ply_count = len(nodes)` and the
  final position row is stored at `ply = len(nodes)`. game_positions plies are
  `0..len(nodes)` contiguous, so `max(ply) GROUP BY game_id` exactly reproduces the
  import-time value. Migration and import write path agree.
- **Display equivalence (the off-by-one concern).** The OLD column was
  `move_count = (len(nodes) + 1) // 2` (confirmed via git history of zobrist.py).
  The NEW label is `Math.ceil(ply_count / 2)` with `ply_count = len(nodes)`. For
  all non-negative integers `(n + 1) // 2 == ceil(n / 2)`, so the rendered
  "N Moves" label is byte-identical to the prior behavior (ply=1→1, ply=2→1,
  ply=3→2, ply=4→2). No off-by-one.
- **§5 FLOOR/CEIL arithmetic.** The old game_positions scan counted white-mover
  even plies and black-mover odd plies in `[1, N]`, which equal `floor(N/2)` and
  `ceil(N/2)` respectively. The new `FLOOR(ply_count/2.0)` / `CEIL(ply_count/2.0)`
  reproduce these exactly. The `/ 2.0` (not `/ 2`) divisor correctly forces float
  division, and `NULLIF(user_moves, 0)` plus `WHERE user_moves > 0` preserve the
  drop-on-zero behavior. NULL ply_count routes to NULL → dropped, matching the
  old position-less drop. Arithmetically equivalent.
- **Nullable safety.** Every reader treats `games.ply_count` as `int | None`:
  backend schemas declare `int | None`, services pass it through verbatim, and the
  three frontend cards guard with `game.ply_count !== null` before calling
  `plysToFullMoves`. No None/undefined crash or NaN render path exists.
- **Downgrade round-trip.** `move_count = (ply_count + 1) / 2` (Postgres integer
  division) reproduces the original full-move count and leaves NULL rows NULL.

The findings below are quality/robustness issues, not correctness failures.

## Warnings

### WR-01: `plysToFullMoves` is unguarded against negative / NaN / fractional input

**File:** `frontend/src/lib/chess.ts:16-18`
**Issue:** The helper does `Math.ceil(plyCount / 2)` with no validation. Every
current caller guards with `ply_count !== null` first, so this is safe *today*,
but the helper's contract is "takes a number" and a `NaN` or negative input would
silently produce `NaN` / a negative move count in the UI rather than failing
loudly. Because `ply_count` is the only nullable field in this family and the
guard lives at the call site (not in the helper), a future caller that forgets the
`!== null` check would render `Math.ceil(null / 2) = 0` (JS coerces `null`→0) — a
plausibly-wrong "0 Moves" rather than an omitted line. This is a latent
foot-gun introduced by centralizing the math without centralizing the guard.
**Fix:** Make the helper total over its declared domain, e.g.:
```ts
export function plysToFullMoves(plyCount: number): number {
  if (!Number.isFinite(plyCount) || plyCount <= 0) return 0;
  return Math.ceil(plyCount / 2);
}
```
Or tighten the signature to `plyCount: number` and keep the null-guard at call
sites as the single documented contract (current approach) — but then add a code
comment on the helper stating "caller must pre-filter null/non-positive."

### WR-02: §5 user_moves silently coerces a NULL `ply_count` analyzed game out of the cohort with no diagnostic

**File:** `scripts/benchmarks/chapter5.py:243-251`
**Issue:** `user_moves_per_game` computes `FLOOR(g.ply_count / 2.0)` /
`CEIL(g.ply_count / 2.0)`. If any game in `base_games` (which requires
`evals_completed_at IS NOT NULL`, i.e. analyzed) has `ply_count IS NULL`, the
CASE yields NULL → `per_game_delta` NULLIF/`WHERE user_moves > 0` drops it
**silently**. The phase intends this (matches the old position-less drop), and in
practice an analyzed game always has positions so ply_count is non-NULL. But the
old query derived user_moves from a `COUNT(*)` over game_positions, which can
**never** be NULL — it was structurally impossible to hit this path. The new
query introduces a NULL-propagation route that depends on an unstated invariant
("every analyzed game has a non-NULL ply_count"). If a backfill gap or a future
import bug ever leaves an analyzed game with NULL ply_count, §5 will under-count
that cohort cell with zero signal. Since chapter5 is benchmark-computation only
(no endpoint, no Sentry), the drift would be invisible.
**Fix:** Either assert the invariant in the §5 cohort query so a violation is
loud, or document it. A cheap guard is to surface NULL-ply_count analyzed games in
the viability/cohort diagnostic, e.g. add a count of
`base_games WHERE ply_count IS NULL` to `CohortSummary` so a non-zero value flags
the gap in the generated report rather than silently shrinking denominators. At
minimum, add a SQL comment at line 243 stating the relied-upon invariant
("base_games is analyzed ⇒ has positions ⇒ ply_count non-NULL; a NULL here means
a backfill gap and the game is dropped").

## Info

### IN-01: `games.ply_count` uses `Integer` (4 bytes) where `SmallInteger` (2 bytes) suffices

**File:** `alembic/versions/20260610_200128_07994baf3b15_replace_move_count_with_ply_count.py:38`, `app/models/game.py:120`
**Issue:** Plies in a real chess game never approach the `int32` ceiling — the
longest possible games stay well under the `SmallInteger` ceiling of 32,767
(prod max full-move counts are in the low hundreds). CLAUDE.md's database design
rule states "don't use BIGINT where SmallInteger suffices"; the same logic applies
to Integer-vs-SmallInteger. Note this is **not a regression**: the prior
`move_count` column was already `Integer` (downsized from BIGINT in the
20260324 migration), so the phase is a like-for-like type swap. Flagging for
awareness, not as a defect introduced by this phase.
**Fix:** If a follow-up touches this column, use
`sa.SmallInteger()` in the migration and
`ply_count: Mapped[int | None] = mapped_column(SmallInteger, nullable=True)` in the
model. Low priority — the 2-byte saving across ~590k rows is marginal and not worth
a standalone migration.

### IN-02: `formatTimeControl` / `formatDate` triplicated verbatim across three game cards

**File:** `frontend/src/components/results/GameCard.tsx:32-64`, `frontend/src/components/results/LibraryGameCard.tsx:106-139`, `frontend/src/components/library/FlawCard.tsx:57-88`
**Issue:** Three identical copies of `formatDate`, `formatTimeControl`, and the
`SECONDS_PER_DAY` constant, each annotated "copied verbatim … D-05 forbids shared
import." This phase did not introduce the duplication (it predates 114.1), but the
phase touched all three files to add the `plysToFullMoves` import, making it the
natural moment to observe that the `plysToFullMoves` helper was correctly
extracted to `lib/chess.ts` while the equally-shared `formatTimeControl`/`formatDate`
were not. The "D-05 forbids shared import" rationale applies to *components* (don't
make LibraryGameCard import GameCard), not to pure display utilities that belong in
`lib/`. Out of scope for this phase; noting the inconsistency.
**Fix:** A future cleanup could move `formatTimeControl` + `formatDate` to
`lib/chess.ts` alongside `plysToFullMoves` (pure functions, no component coupling,
satisfies D-05). Do not do this inside 114.1 without a scope sign-off.

### IN-03: Migration comment cites prod measurements that cannot be verified at review time

**File:** `alembic/versions/20260610_200128_07994baf3b15_replace_move_count_with_ply_count.py:14-17`
**Issue:** The docstring states the backfill "runs in 3.2s at 33MB peak over 43.9M
game_positions rows" and "the write touches ~590k games rows." These are useful for
the no-HUMAN-gate justification, but they are point-in-time prod measurements baked
into a permanent artifact; if the backfill is ever replayed on a much larger prod
dataset (or on the benchmark DB at 43.9M+ rows), the "sub-10s total / no NULL
window" claim may no longer hold and the migration could hold a longer lock than
the comment implies. This is informational — the measurement is plausible and the
single-transaction design is sound for the stated scale.
**Fix:** None required. If desired, soften the language to "measured ~3.2s on prod
as of 2026-06-10 at ~44M rows; re-verify timing before replaying on a materially
larger dataset."
