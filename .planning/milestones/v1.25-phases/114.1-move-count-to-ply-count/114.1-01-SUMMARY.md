## What Was Built

Single-wave cut-over from `games.move_count` (full-move count, was `(len(nodes)+1)//2`) to `games.ply_count` (exact half-move count, `len(nodes)`):

- **Alembic migration `07994baf3b15`** (down_revision `d9e3f1a4b5c6`): adds nullable `ply_count`, backfills via `UPDATE games g SET ply_count = sub.mp FROM (SELECT game_id, max(ply) AS mp FROM game_positions GROUP BY game_id) sub WHERE g.id = sub.game_id`, then drops `move_count`. `downgrade()` reverses it. Round-trips cleanly.
- **`app/models/game.py`**: `move_count` replaced by `ply_count: Mapped[int | None] = mapped_column(nullable=True)`.
- **`app/services/zobrist.py`**: `GameProcessingResult` TypedDict key renamed `move_count` → `ply_count`; computation changed from `(len(nodes)+1)//2` to `len(nodes)`.
- **`app/services/import_service.py`**: `_PositionRowsResult.move_counts` → `ply_counts`; Stage 5 bulk UPDATE writes `.values(ply_count=bindparam("b_pc"))`.
- **API schemas**: `GameRecord.ply_count`, `LibraryGameRecord.ply_count`, `FlawListItem.ply_count` (all `int | None`).
- **Backend readers**: all 4 sites pass `ply_count=...` directly, no COALESCE fallback (D-03).
- **Tests**: fixtures renamed to assert `ply_count` with half-move expected values.
