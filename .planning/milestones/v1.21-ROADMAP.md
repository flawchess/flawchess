# v1.21 Time-Control-Aware Endgame Metrics — Roadmap (ARCHIVED)

**Shipped:** 2026-05-31
**Phases:** 97, 98, 99, 99.1
**Milestone goal:** Make the entire Endgame Metrics and Endgame Type Breakdown reporting time-control-aware, so a player is judged against the norms of the speed they actually play rather than a blended average. Replace the aggregated Conversion/Parity/Recovery cards with per-time-control cards, restructure the type breakdown into collapsible per-TC accordion cards, and bring peer-relative percentile chips to the per-TC rates — then demote the large generated cohort-percentile lookup out of Python source into a DB table.

## Summary

Four phases (one an internal refactor) building on the v1.17 Endgame Metrics section and the v1.19 per-TC percentile/anchor pipeline:

- **Phase 97** replaced the single aggregated Conversion/Parity/Recovery cards with one card per time control (bullet/blitz/rapid/classical), each carrying its own gauge trifecta, WDL bar, and Score Gap chart. Conversion/Recovery gauges use TC-specific neutral bands (benchmark TC d≈0.9); Parity and Score Gap keep the shared global band (both collapse on TC). Cards self-suppress below a per-TC games floor. New backend per-TC rate aggregation (`_compute_per_tc_metric_cards`) + a TC-keyed band registry (`TC_METRIC_BANDS`) codegen'd into `endgameZones.ts`.
- **Phase 98** restructured the Endgame Type Breakdown from a 3-col grid of five per-type cards into full-width vertically-stacked collapsible cards, one per TC, with the user's primary TC (time-weighted) expanded by default. Each card holds a 2×2 grid of four type tiles (rook/minor_piece/pawn/queen — Mixed dropped); Conversion/Recovery gauges return, banded against each card's own per-(class × TC) IQR, and Score Gap is banded per-TC for one consistent card grammar.
- **Phase 99** added peer-relative percentile chips (the v1.19 `PercentileChip` primitive) to the per-TC Conversion, Parity, and Recovery rate cards. 12 new per-(metric, TC) ENUM values computed via the shared pooled-per-user `canonical_slice_sql.py` builder parameterised by TC, cohort-matched on the per-(user, TC) rating anchor, 4-bullet disclosure tooltip.
- **Phase 99.1** moved the generated `COHORT_PERCENTILE_CDF` registry (3.1 MB / ~130k lines) out of `app/services/global_percentile_cdf.py` into a `benchmark_cohort_cdf` DB table seeded from a compact `app/data/` artifact via `scripts/seed_cohort_cdf.py`; the module shrank to ~250 lines of logic. Internal refactor, byte-for-byte chip parity, no behaviour change.

PRs: #160 (Phase 97; rating-anchor follow-up #158/#159), #163 (Phase 98; deployed via release #164), #167 (Phase 99), #168 (Phase 99.1). Benchmark-generator side work landed in #166. Boundary commit on `main`: `dcd22fef` (pre-Phase-97).

## Phase Details

### Phase 97: Endgame Metrics by Time Control

**Goal**: Restructure the Endgame Metrics section of the Endgames page into per-time-control cards (bullet/blitz/rapid/classical), mirroring the existing `EndgameTimePressureSection` per-TC card pattern. Each TC card renders the Conversion/Parity/Recovery gauge trifecta plus WDL and Score Gap charts scoped to that time control. The single aggregated Conversion/Parity/Recovery cards are removed entirely (no aggregated cards on top). Conversion and Recovery gauges use **TC-specific neutral bands** (benchmark Cohen's d ≈ 0.9 on the TC axis — they genuinely differ per TC; bands sourced from `reports/benchmark/benchmarks-latest.md` §3.2.1 per-TC p25/p75); Parity keeps the shared global band and Score Gap keeps the shared global band (both collapse on the TC axis, d < 0.15, so a single band is correct for every TC). Cards self-suppress below a minimum-games floor because Conversion/Recovery are conditional rates with thin per-TC denominators. Requires a new backend aggregation path that computes per-TC conversion/recovery/parity rate values and a new TC-keyed band structure in `app/services/endgame_zones.py` regenerated into `frontend/src/generated/endgameZones.ts`. Per-TC conversion/recovery **percentile badges deferred** to Phase 99.
**Depends on**: none (builds on the v1.19 per-TC percentile/anchor pipeline + v1.17 Endgame Metrics section)
**Requirements**: standalone — no requirement IDs (endgame stats UX refinement)
**Success Criteria** (what must be TRUE):

  1. The Endgame Metrics section renders one card per time control the user has sufficient games in (fixed order bullet/blitz/rapid/classical), with no aggregated Conversion/Parity/Recovery cards remaining.
  2. Each TC card shows the Conversion/Parity/Recovery gauge trifecta plus WDL and a Score Gap chart, all scoped to that time control.
  3. Conversion and Recovery gauge neutral bands are TC-specific; Parity and Score Gap bands are the shared global bands on every card.
  4. Cards self-suppress below a minimum-games floor; the floor is validated against dev-DB distributions during planning.
  5. Per-TC conversion/parity/recovery rate values are computed by the backend and exposed via the endgame overview response; the TC-keyed bands are threaded to the frontend via the regenerated `endgameZones.ts` (CI drift gate green).
  6. Desktop and mobile layouts both render the per-TC cards responsively.
  7. Backend (`pytest`, `ty`, `ruff`) and frontend (`lint`, `test`, `knip`) gates all pass.

### Phase 98: Per-TC Collapsible Endgame Type Cards

**Goal**: Restructure the Endgame Type Breakdown so time control is a per-card *view dimension* instead of a pooled band or a TC-mix blend. Replace the 3-col grid of five per-type cards with **one full-width collapsible card per time control** (bullet/blitz/rapid/classical), the **user's primary TC expanded by default** (primary = argmax of coarse time spent `games_in_bucket × NOMINAL_DURATION[bucket]`, no recency weighting) and the rest collapsed. Inside each TC card, a **2×2 grid of four type tiles** — rook, minor_piece, pawn, queen — with **Mixed dropped** (least-actionable catch-all; pawnless stays hidden). Because each card is a single TC, the Conversion and Recovery gauges removed on 2026-05-29 return showing **that TC's own per-(class × TC) benchmark band** (the d≈1.2–1.7 metrics judged against the right reference) with no TC-mix weighting. **Score Gap (ΔES) is banded per-TC as well, for visual consistency** (statistically TC-flat at d≈0.13, so the four bands are near-identical — the redundancy is chosen, not a bug). TC cards/tiles self-suppress below the games floor (à la Lichess Tutor's 30-games-per-speed model). Applies the mode-3 "collapsible TC cards" pattern; full-width-stacked layout is a hard constraint. Endgame ELO Timeline and Endgame Metrics by TC (Phase 97) are out of scope.
**Depends on**: Phase 97 (reuses `EndgameGauge`, the `endgameZones.ts` codegen, and the per-(class × TC) rate aggregation path)
**Requirements**: standalone — no requirement IDs (endgame stats UX refinement)
**Success Criteria** (what must be TRUE):

  1. The Endgame Type Breakdown renders as full-width, vertically stacked collapsible cards, one per time control (only TCs with sufficient games appear).
  2. The user's primary TC card is expanded by default (primary = argmax of coarse time spent, over TCs passing the games floor); others collapse behind a keyboard-accessible chevron with `data-testid`s.
  3. Each expanded TC card shows a 2×2 grid of four type tiles (rook, minor_piece, pawn, queen). Mixed is no longer shown; pawnless stays hidden.
  4. Each tile's Conversion and Recovery gauges are back, banded against that card's own per-(class × TC) benchmark IQR — no TC-mix-weighted blend.
  5. Score Gap is banded per-(class × TC) like the other tile metrics (forced per-TC for visual consistency despite being TC-flat).
  6. Per-(class × TC) bands are generated into `endgameZones.ts` via `endgame_zones.py` + `gen_endgame_zones_ts.py`, CI drift gate green (no eligible-count weighting payload).
  7. A TC card and/or its tiles self-suppress below the games floor; a user concentrated in one TC sees only that TC's card.
  8. The backend `/stats` endgame breakdown exposes per-(class × TC) rates and counts grouped for per-TC rendering; the LLM insights path is unaffected (response shape preserved or additively extended).
  9. Desktop and mobile both render the collapsible per-TC cards and 2×2 tile grid responsively.
  10. Backend + frontend gates pass; layout tests updated; a `CHANGELOG.md` entry records the restructure and the Conv/Recov re-introduction.

- [x] 98-02-PLAN.md — Frontend: primaryTc util + restored 5-element tile + collapsible per-TC accordion section + test updates + CHANGELOG

### Phase 99: Percentile Badges for Conversion, Parity, and Recovery

**Goal**: Add peer-relative percentile chips (the v1.19 `PercentileChip` primitive) to the per-TC Conversion, Parity, and Recovery rate cards delivered in Phase 97 — the deferred badge work called out in Phase 97. Requires new per-(metric, TC) cohort CDF materialization in `user_benchmark_percentiles` (today only the ΔES score-gap family has stored percentiles). Mirrors the Phase 94.3 Time Pressure per-TC chip pattern: the 3 rate families × 4 TCs are computed via the pooled-per-user methodology in `canonical_slice_sql.py` parameterised by TC, cohort-matched on the per-(user, TC) rating anchor, and surfaced with the 4-bullet disclosure tooltip per `feedback_percentile_chip_tooltip_disclosure`. Conversion and Recovery are conditional rates with thin per-TC denominators, so the inclusion floor and sparse-cell suppression carry over from Phase 97.
**Depends on**: Phase 97 (per-TC rate aggregation + TC card layout); Phase 94.3 (per-TC pooled CDF + chip pattern)
**Requirements**: standalone — no requirement IDs (endgame stats UX refinement)
**Success Criteria** (what must be TRUE):

  1. Each per-TC card renders a percentile chip on its Conversion, Parity, and Recovery rates, gated on the per-(metric, TC) inclusion floor — below floor, no chip renders.
  2. 12 new metrics (`{conversion_rate, parity_rate, recovery_rate}_{bullet, blitz, rapid, classical}`) are added to the `user_benchmark_percentiles` ENUM and computed via the shared pooled-per-user builder parameterised by TC — drift between CDF construction and per-user lookup remains structurally impossible.
  3. The cohort CDFs are generated under the existing per-(metric, ELO anchor, TC) sliding-window protocol, with the regen report archived.
  4. Chips are cohort-matched on the per-(user, TC) rating anchor and carry the 4-bullet disclosure tooltip, first two bullets TC-scoped.
  5. The backfill script populates the 12 new metrics on dev (and prod via tunnel after sign-off); desktop + mobile parity; backend + frontend gates pass.

### Phase 99.1: Move Cohort CDF Out of Source into a DB Table

**Goal**: Eliminate the source bloat in `app/services/global_percentile_cdf.py` (3.1 MB / 130,369 lines, of which ~130,119 are float literals for the generated `COHORT_PERCENTILE_CDF` dict and only ~250 are logic) by relocating the CDF breakpoint data into a `benchmark_cohort_cdf` DB table, unlocking SQL analysis of the breakpoints as a side benefit. The lookup is not on a request hot path (`interpolate_cohort_percentile` is called only by `compute_stage_a` / `compute_stage_b` background tasks, ~32 lookups per import). The generator emits a compact seed file to `app/data/` instead of rewriting the `.py`; a manual idempotent seed script (`scripts/seed_cohort_cdf.py`, `ON CONFLICT DO UPDATE`) loads it, wired into `bin/run_local.sh`. The module shrinks to ~250 lines, retaining `CdfMetricId`, `CdfTable`, breakpoint constants, and the interpolation math.
**Depends on**: Phase 99 (consumes the regenerated cohort CDF + the `interpolate_cohort_percentile` lookup and its two background callers)
**Requirements**: standalone — no requirement IDs (internal refactor / tech-debt)
**Success Criteria** (what must be TRUE):

  1. `global_percentile_cdf.py` no longer contains the generated registry; ~250 lines, BEGIN/END sentinels gone. `CdfMetricId`, `CdfTable`, breakpoint constants, and `_interpolate_with_table` remain.
  2. A `benchmark_cohort_cdf` table (new model + Alembic migration, schema only) holds all breakpoint cells, with `snapshot_month` as an audit column.
  3. The interpolation lookup reads from the table (async DB access in the new `benchmark_cohort_cdf_repository.load_cohort_cells` batched prefetch per CONTEXT D-04; the module stays pure/sync taking a prefetched `CdfTable`); both background callers updated and pass.
  4. `gen_global_percentile_cdf.py` emits a compact seed file to `app/data/`; the regen drift test compares regenerated seed-file content rather than source bytes.
  5. `scripts/seed_cohort_cdf.py` performs an idempotent `ON CONFLICT DO UPDATE` load; `bin/run_local.sh` seeds on first-time local setup; chips render unchanged end-to-end.
  6. Backend + frontend gates pass; chip output is byte-for-byte equivalent to pre-refactor for a representative user (pure relocation, no behavior change).

## Milestone Summary

**Decimal Phases:**

- Phase 99.1: Move Cohort CDF Out of Source into a DB Table (inserted after Phase 99 as a structural-debt follow-up; addresses SEED-030 Track B).

**Key Decisions:**

- TC-specific neutral bands for Conversion/Recovery (benchmark TC d≈0.9), shared global band for Parity and Score Gap (both collapse on TC, d<0.15). Source of truth: `reports/benchmark/benchmarks-latest.md`.
- Per-(class × TC) bands with no TC-mix blending (supersedes the earlier blended-band approach in `notes/endgame-typecard-tcmix-gauges.md`); each single-TC card shows that TC's own reference.
- Score Gap banded per-TC for one consistent card grammar despite being statistically TC-flat — known, chosen redundancy.
- Mixed dropped from the type tiles as the least-actionable catch-all; pawnless stays hidden.
- Primary-TC default-expand = argmax of coarse time spent (`games_in_bucket × NOMINAL_DURATION[bucket]`), no recency weighting; accordion resets to primary TC on filter change.
- Cohort CDF lives in `benchmark_cohort_cdf` (DB), seeded from a compact `app/data/` artifact; interpolation stays pure/sync over a batched-prefetched table (CONTEXT D-04).

**Issues Resolved:**

- Endgame Type Breakdown "no data" despite a header game count (per-class spans now aggregated like the pooled stats; Mixed/pawnless excluded so header count == sum of tiles).
- Endgames page locked forever for sparse-endgame users (readiness gate recognises a committed rating anchor with no percentile rows as "analysis ran, user too sparse").

**Technical Debt Incurred / Carried:**

- Prod backfill of the 12 new rate-percentile metrics deferred to deploy (todo `2026-05-31-phase-99-prod-backfill-rate-percentiles`).
- SEED-030 Track A (split oversized multi-concern modules) remains open; Track B (CDF demotion) closed by Phase 99.1.
