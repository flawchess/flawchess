<objective>
Bring the 11 majors-behind frontend dependencies up to their latest major, one coupled cluster at a time, ordered low-risk → high-risk so the gate signal sharpens as work proceeds. This is tooling/dependency maintenance — no new application features.

The phase is structured as **6 strictly-sequential waves, one cluster per wave**. Clusters MUST NOT run in parallel: bisectability (criterion 2) requires exactly one cluster in flight at a time. Each cluster is its own branch → full local gate → squash-merge to `main`, then the next cluster branches off the updated `main` (D-06). Each cluster is one atomic commit, so a gate failure bisects to exactly one cluster.

1. Wave 1 — lucide-react 1.x  (+ shadcn 4.9.0 straggler, D-04)

Purpose: keep the frontend toolchain current and supported; the high-risk recharts bump is last so the gate is already proven on simpler clusters first.
Output: bumped version constraints in `frontend/package.json` + `package-lock.json`, plus a small set of config/code edits required by the bumps (recharts CartesianGrid axis binding, chart.tsx type updates, TS6 baseUrl handling, possibly eslint config). There are NO new symbols or files.

## Execution rules specific to this plan (READ FIRST)

- **One cluster = one wave = one branch = one squash-merge to `main` (D-06).** Do NOT batch multiple clusters into one branch or one commit. Branch off the current `main`, do the cluster, run the FULL local gate, squash-merge to `main` per CLAUDE.md § "Version Control", then branch the next cluster off updated `main`.
- **Sequential only.** Never start a later cluster before the previous cluster has merged green. GSD's parallel-within-wave default does not apply here.
- **Re-snapshot exact versions first (Claude's Discretion).** The research version table is from 2026-05-31 and will drift. The first task of EACH cluster runs `npm outdated <pkgs>` / `npm view <pkg> version` from `frontend/` to confirm the live latest target before editing `package.json`. Use the live latest major as the target; the research numbers are the floor.
- **Full local gate definition** (CLAUDE.md § "Pre-PR checklist") run before EVERY cluster merge:
  - Backend (unchanged code, still gated): `uv run ruff format app/ tests/` , `uv run ruff check app/ tests/ --fix` , `uv run ty check app/ tests/` , `uv run pytest -n auto -x`
  - Frontend: `( cd frontend && npm run lint && npm test -- --run && npm run build && npm run knip )`
- **Escape hatches (represent, do NOT pre-trigger):** research cleared all clusters, so no deferral is baked in. But if a single package cannot be made green, pin it back with a documented reason (precedent: quick task 260531-jga) — criterion 1. If a peer-compat blocker appears for cluster 4 or 5 (research says none), spend a BOUNDED effort on npm `overrides` / a known-good intermediate combo, then defer the cluster with the blocker + unblocking version recorded — D-02. Do not ship a fragile over-pinned lockfile to force a bump.
- **Supply-chain check per cluster** (see `<threat_model>`): confirm each bumped package is the legitimate registry package at the intended version, review the `package-lock.json` diff for unexpected transitive additions, and run `npm audit --audit-level=high` after install. Block on high-severity.
- **CHANGELOG:** append a terse user-facing bullet under `## [Unreleased]` → `### Changed` when the phase's clusters land (per CLAUDE.md changelog rules). One bullet for the dependency-upgrade phase is fine.
