# Roadmap: FlawChess v1.14 Score-Based Opening Insights

**Status:** 🚧 Planning (opened 2026-04-28)
**Source:** SEED-007 (Option A only — Wilson on score, 0.50 pivot, no user-baseline) + SEED-008 (label reframe). Both seeds folded into this milestone and closed.
**Design note:** [.planning/notes/opening-insights-v1.14-design.md](../notes/opening-insights-v1.14-design.md)
**Requirements:** [.planning/REQUIREMENTS.md](../REQUIREMENTS.md) — 14 active requirements, INSIGHT-SCORE-01..07 + INSIGHT-UI-01..07

## Milestone Goal

Migrate Opening Insights and Move Explorer color coding from loss-rate to chess score `(W + 0.5·D)/N`, gate findings on effect size against a 0.50 pivot, and annotate them with low/medium/high confidence badges. Replaces today's "Major/Minor Weakness" framing with a calibrated discovery surface that holds up under future LLM narration.

The conceptual pivot: **effect size decides what shows up, confidence annotates how sure we are**. Standard stats says "pre-register α, threshold on p, interpret effect size." For a discovery UI it's the right inversion — the user's question is "is this worth my attention" not "is this statistically significant." A 0.51-score line at n=1000 isn't interesting even with high confidence; a 0.55-score line at n=20 is interesting even with low confidence.

## Phase Details

### Phase 75: Backend — score metric and confidence annotation

**Goal:** `opening_insights_service` and `openings_repository` classify findings on chess score `(W + 0.5·D)/N` against a 0.50 pivot with effect-size thresholds, computes Wilson 95% confidence intervals per finding, and exposes a `confidence: "low" | "medium" | "high"` field on the `OpeningInsightFinding` API contract. The CI consistency test mirrors the score-based thresholds against `arrowColor.ts`.

**Depends on:** v1.13 shipped (this is a refinement of the same `opening_insights_service` from Phase 70). No schema changes — score is computed from existing `w / d / l / n_games` aggregates.

**Requirements:** INSIGHT-SCORE-01, INSIGHT-SCORE-02, INSIGHT-SCORE-03, INSIGHT-SCORE-04, INSIGHT-SCORE-05, INSIGHT-SCORE-06, INSIGHT-SCORE-07

**Plans:** 4 plans

Plans:
- [ ] 75-01-PLAN.md — Constants module rewrite + frontend pure-additive exports + CI consistency test rewrite
- [ ] 75-02-PLAN.md — OpeningInsightFinding API contract: remove loss_rate/win_rate, add confidence + p_value
- [ ] 75-03-PLAN.md — Service _classify_row + _compute_confidence helper + repository HAVING clause + service/repository test updates
- [ ] 75-04-PLAN.md — REQUIREMENTS.md amendments (D-05 → INSIGHT-SCORE-04 trinomial Wald; D-09 → INSIGHT-SCORE-06 confidence + p_value)

**Success Criteria** (what must be TRUE):

1. `POST /api/insights/openings` returns `OpeningInsightFinding[]` with `score: float`, `severity: "major" | "minor"`, and `confidence: "low" | "medium" | "high"` fields. The legacy `loss_rate` / `win_rate` fields are removed (or kept solely for backward-compat — discuss-phase decision); score is the canonical metric.
2. Findings classify on score against the 0.50 pivot with effect-size thresholds: minor at `|score − 0.5| ≥ 0.05`, major at `≥ 0.10`. Strict boundaries; no asymmetry between weakness and strength sides.
3. Each surviving finding carries a Wilson 95% confidence bucket — `high` / `medium` / `low` — derived from the interval half-width or equivalent p-value bucket. Exact bucket boundaries are resolved during Phase 75 `/gsd-discuss-phase`.
4. `MIN_GAMES_PER_CANDIDATE` reduced from 20 to 10. Findings with `n_games ∈ [10, 19]` surface with confidence `(low)` rather than being filtered out — discovery framing.
5. CI consistency test asserts score-based threshold lock-step between backend classification and `frontend/src/lib/arrowColor.ts`. Test catches any divergence (regression of the v1.13 INSIGHT-CORE-05 invariant under the new metric).

**Open questions for `/gsd-discuss-phase`:**
- Wilson half-width buckets: e.g. `high` at half-width ≤ 0.10, `medium` at ≤ 0.20, `low` otherwise — confirm against real data before locking.
- Whether to keep `loss_rate` / `win_rate` in the API for one milestone (deprecation window) or replace cleanly. Cleanest is replace; only payload consumer is the frontend in this same milestone.
- Where confidence is computed — pure SQL via Wilson formula, or post-aggregation in Python. Push to SQL if straightforward; otherwise Python-side over the existing aggregates.

### Phase 76: Frontend — score-based coloring, confidence badges, label reframe

**Goal:** Move Explorer arrows and moves-list rows are colored by score (effect-size only). The existing `(low)` indicator on moves-list rows extends to `(low) / (medium) / (high)` and consumes the new backend confidence field. Opening Insights cards show a confidence badge next to a softened severity label, with section titles reframed per SEED-008 ("Worth a closer look" / "Played confidently"). A `?` explainer popover on the section title walks users through the framing.

**Depends on:** Phase 75 (consumes the new `confidence` field and score-based thresholds).

**Requirements:** INSIGHT-UI-01, INSIGHT-UI-02, INSIGHT-UI-03, INSIGHT-UI-04, INSIGHT-UI-05, INSIGHT-UI-06, INSIGHT-UI-07

**Success Criteria** (what must be TRUE):

1. `arrowColor.ts` colors arrows by score (effect-size only) using the same thresholds as backend Phase 75. No confidence cue (no opacity/dashing/dotting) — board reads at a glance via color intensity. Arrows stay user-agnostic-at-a-position. The CI consistency test still passes.
2. Move Explorer moves-list row tint uses score (same encoding as arrows). Existing WDL bars on each row are unchanged — they remain the literal-data display. The existing `(low)` indicator at games < 10 extends to a three-level system: `(low)` / `(medium)` / `(high)`, with `(high)` rows unmarked.
3. `OpeningInsightsBlock` section titles soften per SEED-008 — "Worth a closer look (White/Black)" and "Played confidently (White/Black)" replace "White/Black Opening Weaknesses/Strengths". Severity language inside cards softens correspondingly. Final copy resolved during Phase 76 `/gsd-discuss-phase`.
4. Each `OpeningFindingCard` shows a confidence badge `(low) / (medium) / (high)` adjacent to the severity word. Badge consumes the new `confidence` field. Underlying numbers (`L/N`, `n_games`) stay visible.
5. A `?` icon on the section title opens an explainer popover covering the score / sample-size / confidence framing in user-friendly language.
6. All changes apply equally to mobile layouts (Move Explorer mobile, Insights block mobile drawer/single-column rendering). Verified at 375px width — no horizontal scroll, ≥ 44px touch targets, all interactive elements have `data-testid` per CLAUDE.md frontend rules.

**Open questions for `/gsd-discuss-phase`:**
- Exact section-title and severity copy (tone calibration follows `feedback_llm_prompt_design.md`).
- Whether the explainer popover content lives inline in `OpeningInsightsBlock.tsx` or in a shared `<HelpPopover>` component reused elsewhere.
- Whether the `(low)/(medium)/(high)` indicator on the moves-list row should match the visual style of the existing `(low)` cue (small gray text in parens) or take a more prominent badge form. Default: match existing.

## Out of Scope (already decided in `/gsd-explore`)

- User-baseline / Bayesian shrinkage thresholds — explicitly rejected. See REQUIREMENTS.md § Out of Scope and the design note for full rationale.
- Showing raw p-values on cards.
- Confidence cue on board arrows (opacity/dashing).
- Dropping the WDL bars from the move explorer.
- LLM narration of opening findings — future seed; v1.14 lands the calibrated data plumbing that LLM would consume.
