## Milestone Goal

Migrate Opening Insights and Move Explorer color coding from loss-rate to chess score `(W + 0.5·D)/N`, gate findings on effect size against a 0.50 pivot, and annotate them with low/medium/high confidence badges. Replaces the prior "Major/Minor Weakness" framing with a calibrated discovery surface that holds up under future LLM narration.

The conceptual pivot: **effect size decides what shows up, confidence annotates how sure we are**. Standard stats says "pre-register α, threshold on p, interpret effect size." For a discovery UI it's the right inversion — the user's question is "is this worth my attention" not "is this statistically significant." A 0.51-score line at n=1000 isn't interesting even with high confidence; a 0.55-score line at n=20 is interesting even with low confidence.

## Phase Details

### Phase 75: Backend — score metric and confidence annotation

**Shipped behavior:** `opening_insights_service` and `openings_repository` classify findings on chess score `(W + 0.5·D)/N` against a 0.50 pivot with effect-size thresholds (`±0.05` minor, `±0.10` major), compute trinomial Wald 95% half-width per finding (variance `(W + 0.25·D)/N − score²` — actual variance of the chess result distribution, not binomial Wilson), and expose `confidence: "low" | "medium" | "high"` plus `p_value: float` on `OpeningInsightFinding`. Strict `≤`/`≥` boundaries, half-width buckets `≤ 0.10 → high`, `≤ 0.20 → medium`, else `low`. `loss_rate` / `win_rate` removed cleanly. `MIN_GAMES_PER_CANDIDATE` dropped 20 → 10. CI consistency test enforces threshold lock-step with `arrowColor.ts`.

**Requirements:** INSIGHT-SCORE-01..07 (all met)

Plans:
- [x] 75-01-PLAN.md — Constants module rewrite + frontend pure-additive exports + CI consistency test rewrite
- [x] 75-02-PLAN.md — OpeningInsightFinding API contract: remove loss_rate/win_rate, add confidence + p_value
- [x] 75-03-PLAN.md — Service `_classify_row` + `_compute_confidence` helper + repository HAVING clause + service/repository test updates
- [x] 75-04-PLAN.md — REQUIREMENTS.md amendments (D-05 → INSIGHT-SCORE-04 trinomial Wald; D-09 → INSIGHT-SCORE-06 confidence + p_value)

**Key decisions:**
- Trinomial Wald CI over binomial Wilson (D-05) — chess result distribution `X ∈ {0, 0.5, 1}` is trinomial; binomial Wilson over-states uncertainty when draws are common. Standard formula in chess match statistics (BayesElo, Ordo).
- API contract extends with both `confidence` (the half-width bucket, user-facing) and `p_value` (two-sided test of observed score vs 0.50, tooltip-grade) — D-09.
- Strict `≤`/`≥` boundaries throughout — eliminates the prior `loss_rate > 0.55` asymmetry.

### Phase 76: Frontend — score-based coloring, confidence badges, label reframe

**Shipped behavior:** `arrowColor.ts` colors arrows by score (effect-size only, no confidence cue on arrows). Move Explorer moves-list row tint uses score, with extended mute rule (`game_count < 10 OR confidence === 'low'`). Conf column added to moves-list with sort key `(confidence DESC, |score - 0.50| DESC)`. `OpeningFindingCard` renders score-based prose ("You score X% as <Color> after <san>") with a level-specific confidence indicator and tooltip carrying the directional p-value. `UNRELIABLE_OPACITY` mute applied when `n_games < 10` OR `confidence === 'low'`. Four `InfoPopover` triggers added to `OpeningInsightsBlock` section headers covering the score / sample-size / confidence framing. Mobile parity verified at 375px.

**Requirements:** INSIGHT-UI-01..07 — INSIGHT-UI-04 descoped 2026-04-28 per Phase 76 D-04 (severity word never appears as user-facing text; confidence badge + sort calibration carry the SEED-008 calibration intent without title rewrites).

**Key decisions:**
- Sort key `(confidence DESC, |score - 0.50| DESC)` (D-03) — high-confidence findings rise to the top within each severity bucket; effect-size is the within-confidence tiebreak.
- Single `compute_confidence_bucket` module shared across `opening_insights_service` and the move-explorer payload — CI structural test asserts there's only one implementation.
- INSIGHT-UI-04 descoped — confidence badge + sort calibration deliver SEED-008's intent without rewriting "Weakness/Strength" copy that's already serving its purpose.

**Inline follow-up (PR #71, between Phase 76 and Phase 77):** Force grey arrow + skip row tint when `confidence === 'low'`. The board now reads cleanly at-a-glance: low-confidence findings still surface in the table (with the badge) but don't visually claim authority on the board.

### Phase 77: Troll-opening watermark on Insights findings

**Shipped behavior:** `troll-face.svg` renders as a 30%-opacity bottom-right watermark on `OpeningFindingCard` (mobile + desktop) AND a small inline icon next to qualifying SAN rows in `MoveExplorer` (desktop only via `hidden sm:inline-block`) when the user-side position matches a curated troll-opening set. Matching is **frontend-only** via a side-only FEN piece-placement key — no backend schema change, no Zobrist hash, no API contract change. Curation is offline via a Node/TS script that prints candidates to stdout for human pruning per CONTEXT.md D-01.

**Requirements:** TBD at discuss-phase, captured as D-01..D-11 in `77-CONTEXT.md`. All decisions met in shipped code.

Plans (4 plans across 2 waves):
- [x] 77-01-PLAN.md — Wave 1 — Asset move + theme constant (`TROLL_WATERMARK_OPACITY = 0.30`) + matcher utility (`deriveUserSideKey`, `isTrollPosition`) + unit tests
- [x] 77-02-PLAN.md — Wave 1 — Curation script (`frontend/scripts/curate-troll-openings.ts`) + hand-pruned static data module (autonomous: false — D-01 human checkpoint)
- [x] 77-03-PLAN.md — Wave 2 — `OpeningFindingCard` watermark (single absolute-positioned sibling covering both mobile and desktop branches, `pointer-events-none`, decorative `<img>` with `alt=""` + `aria-hidden="true"`) + tests
- [x] 77-04-PLAN.md — Wave 2 — Move Explorer inline icon (desktop-only via `hidden sm:inline-block`, `sideJustMoved` derivation contract, defensive throw on input contract) + tests

**Key decisions (from 77-CONTEXT.md):**
- Frontend-only matching via side-only FEN piece-placement key (D-08) — no Zobrist hash, no backend schema, no API contract change. The curation set is small and read-only; lookup happens once per finding render.
- Per-ply emission (both colors) in the curation script (Pitfall 2) — pushes ambiguity into the human review step rather than guessing at "the defining position" of an opening.
- Decorative `<img>` idiom over CSS background — keeps the asset cacheable and lets browser image decoding pipeline handle scaling; `alt=""` + `aria-hidden="true"` keep it out of the accessibility tree.

## Out of Scope (decided in `/gsd-explore`)

- User-baseline / Bayesian shrinkage thresholds — explicitly rejected. See REQUIREMENTS.md § Out of Scope and the design note for full rationale.
- Showing raw p-values on cards (only the `(low)/(medium)/(high)` badge surfaces; `p_value` lives in tooltip).
- Confidence cue on board arrows (opacity/dashing) — board reads at-a-glance via effect-size color only; PR #71 added a stronger rule (force grey + skip row tint when low confidence) that achieves the same outcome via the existing color system.
- Dropping the WDL bars from the move explorer.
- Engine-eval-based weakness detection — same rationale as v1.13.
- Troll-opening watermark on Bookmarks / Endgames surfaces — Phase 77 covers Insights finding cards + Move Explorer rows only.
