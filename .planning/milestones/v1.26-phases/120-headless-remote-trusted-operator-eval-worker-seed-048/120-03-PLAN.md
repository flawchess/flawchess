<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker machine → prod HTTPS API | The worker is the trusted-operator client; it holds the operator token (from `--token`/env) and presents it over TLS. |
| operator token at rest in the worker process | Token passed via CLI arg or `EVAL_OPERATOR_TOKEN` env; lives in the operator-controlled process only. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-120-01 | Information Disclosure | operator token in worker | mitigate | Token read from `--token` or `EVAL_OPERATOR_TOKEN` env; sent only as the `X-Operator-Token` request header over HTTPS; NEVER printed/logged (startup log prints base_url + workers only). Verified by grep in acceptance criteria. |
| T-120-02 | Tampering (engine-generation drift) | worker SF version reporting (D-5) | mitigate | Worker reads its real version via `get_stockfish_version()` at startup and reports it on every submit; the server (120-02) rejects a mismatch. The worker does not fabricate the version. |
| T-120-03 | Tampering (ply misalignment / store poisoning) | worker eval mapping (D-2) | mitigate | Worker returns `evaluate_nodes_with_pv` results position-keyed and unmodified — no post-move shift, no row construction. It cannot mis-align storage; the server owns the convention. Verified by grep (no `+ 1` / `post_move` in `_eval_positions`). |
| T-120-SC | Tampering | npm/pip installs | accept | No new packages this phase (120-RESEARCH.md Package Legitimacy Audit: none). Worker uses only existing deps (`httpx`, `chess`, `EnginePool`). No install task; no legitimacy checkpoint required. |
</threat_model>
