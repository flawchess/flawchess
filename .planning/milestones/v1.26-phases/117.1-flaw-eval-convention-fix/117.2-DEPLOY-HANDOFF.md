## What's shipping (and why)

A single **data-only Alembic migration**: `alembic/versions/20260614_130000_wipe_eval_only_residue.py` (revision `20260614130000`, down_revision `20260614120000`).

**Background:** SEED-044 fixed an off-by-one where the full-game analyzer stored engine evals pre-move instead of post-move (shipped in release #190 / Phase 117.1, already in prod). While verifying, we found a second, older issue: **3,497 engine games (3 users)** carry dense `eval_cp` from a *pre-Phase-117 eval-only pass* — every ply has an eval but there's **no `best_move`, no flaw classification, and no `full_evals_completed_at`**. The Library marks a game "analyzed" purely on **≥90% eval coverage**, so these show as *analyzed-with-no-flaws* (the symptom the user reported).

**The migration** NULLs `eval_cp/eval_mate/best_move/pv` for engine games (`lichess_evals_at IS NULL`) that have **no full marker** and **≥90% coverage**, so they drop out of the coverage-based "analyzed" set until the background drain re-materializes them properly (post-move eval + best_move + flaws + marker). They're already queued for the drain (`full_evals_completed_at IS NULL`, tier-3 backlog).

**Why not gate the Library on `full_evals_completed_at` instead?** Because **9,655 lichess games have flaws but no marker** (their flaws came from import/backfill/reclassify classification, which never sets the drain-only marker). Gating on the marker would hide them. The wipe is the lichess-safe choice — it's gated on `lichess_evals_at IS NULL`, so lichess is never touched.

### 2. Migration had the intended effect + lichess untouched (prod DB — needs the tunnel: `bin/prod_db_tunnel.sh`, MCP `mcp__flawchess-prod-db__query`)
```sql
WITH eval_only AS (
  SELECT gp.game_id
  FROM game_positions gp JOIN games g ON g.id=gp.game_id
  WHERE g.lichess_evals_at IS NULL AND g.full_evals_completed_at IS NULL
  GROUP BY gp.game_id
  HAVING (SUM(CASE WHEN gp.eval_cp IS NOT NULL OR gp.eval_mate IS NOT NULL THEN 1 ELSE 0 END)::float/COUNT(*)) >= 0.90
)
SELECT
  (SELECT count(*) FROM eval_only) AS eval_only_residue_games,                         -- expect ~0  (was 3497)
  (SELECT count(*) FROM games WHERE lichess_evals_at IS NOT NULL) AS lichess_total,     -- expect 39876 UNCHANGED
  (SELECT count(*) FROM games WHERE lichess_evals_at IS NOT NULL AND full_evals_completed_at IS NULL
     AND EXISTS (SELECT 1 FROM game_flaws f WHERE f.game_id=games.id)) AS lichess_flawed_no_marker;  -- expect 9655 UNCHANGED
```

## If something's wrong

- **`gh pr checks` says "no checks reported":** the PR is conflicted (can't happen here per the ancestor check, but if it does) — `git merge -s ours origin/production -m "chore(release): forward-port"` then `git push origin main`.
- **`lichess_total` or `lichess_flawed_no_marker` dropped after deploy:** the migration wrongly touched lichess data. This should be impossible (every statement is gated `lichess_evals_at IS NULL`), but if it happens, **stop** and report — the down migration is a no-op (can't restore), so escalate rather than improvise.
- **CI red on the release PR:** it's a pure data-migration + test; failures are almost certainly unrelated flakes or drift — see `deploy` skill `references/ci-fixes.md`. Fix on `main`, push, re-watch.
- **Server SHA mismatch at the end of `bin/deploy.sh`:** deploy didn't converge — do not claim success; check `ssh flawchess "cd /opt/flawchess && docker compose ps"` and the deploy logs.
