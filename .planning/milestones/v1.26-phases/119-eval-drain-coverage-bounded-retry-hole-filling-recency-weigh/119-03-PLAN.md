<objective>
Two locked, non-seed cleanups (ROADMAP §E/F/G):

E. Pulse the EvalCoverageBadge CPU icon (EvalCoverageBadge.tsx:81) while analysis is incomplete, matching the sibling surfaces that already pulse. Decision locked 2026-06-14: gate on `analyzedN < totalN`, NOT on inFlightCount.

F. Remove the `in_flight_count` plumbing end-to-end. Per RESEARCH-NOTES: tier-2 auto-enqueue was removed in Phase 118 and tier-3 is a derived pick with no eval_jobs row, so `count_in_flight_evals` (= eval_jobs pending|leased) can only ever count tier-1 explicit jobs — already signalled by each card's pulsating "Analyze" button — and is structurally blind to the dominant tier-3 backlog drain. It was never an honest progress indicator for the common case.

G (reader half): the dead `ix_eval_jobs_user_active` index DROP is in the 119-01 migration. This plan removes its sole reader, `count_in_flight_evals`.

CRITICAL nuance the ROADMAP under-specifies: GamesTab and FlawsTab currently use the inFlightCount `>0 → 0` transition to fire a one-shot query invalidation that flips a card from "Analyzing…" to the analyzed view (and refreshes flaw views) after a tier-1 analysis completes. Removing inFlightCount naively would break that refresh. Repoint those effects to an `analyzedCount` transition (analyzedCount is still polled via trackFullAnalysis and rising is the observable "a game finished" signal). The per-card NoAnalysisState keeps its own localized onInFlightChange (D-118-11 — already independent of aggregate inFlightCount).

Purpose: an honest progress signal (the pulsing badge tracks real coverage), less dead plumbing, a dropped index.
Output: pulsing badge, in_flight_count fully removed, refresh effects repointed, knip-clean.
</objective>
