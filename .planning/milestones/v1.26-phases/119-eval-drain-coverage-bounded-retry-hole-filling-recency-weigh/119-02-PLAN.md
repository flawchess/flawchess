<objective>
Replace `_claim_tier3_derived`'s winner-take-all `last_activity DESC` top key (eval_queue_service.py:204-241, D-118-04) with an Efraimidis–Spirakis recency-weighted user lottery (SEED-046), and in the SAME rewrite fix the live tier-3 lichess-%eval leak bug (RESEARCH-NOTES "Live prod bug" section).

The leak: the current WHERE clause is `full_evals_completed_at IS NULL` ONLY — it does NOT exclude lichess %eval games. D-118-04's intended deprioritization via `Game.lichess_evals_at.isnot(None).asc()` (line 230) is the LAST order key and `played_at DESC` (line 226) breaks the tie first, so the lichess tiebreaker almost never fires. Net effect: the drain runs a full 1M-node engine pass on lichess games that `needs_engine_full_evals` says to skip (~70% wasted throughput measured on prod user 28), and the coverage badge appears frozen.

The SEED-046 candidate predicate (`full_evals_completed_at IS NULL AND lichess_evals_at IS NULL` = exactly `needs_engine_full_evals`) IS the fix. The new selection must REPLACE the WHERE clause and DROP the dead line-230 tiebreaker — NOT layer the lottery on top of the old predicate.

Purpose: fast catch-up on return (a returning user leaps to the front), no wasted lichess re-eval, self-unfreezing badge.
Output: rewritten _claim_tier3_derived with weighted lottery + residual PV-backfill tier + τ/floor constants + EXPLAIN evidence; renamed is_analyzed→is_lichess_eval_game; updated tests.
</objective>
