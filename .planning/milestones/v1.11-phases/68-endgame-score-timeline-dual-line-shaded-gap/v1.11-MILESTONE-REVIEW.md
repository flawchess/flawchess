### DEAD-01 (Warning) — `frontend/src/generated/endgameZones.ts` is never imported

**File:** `frontend/src/generated/endgameZones.ts` (entire file, 40 LOC)
**Finding:** The file is code-generated from the Python zone registry by `scripts/gen_endgame_zones_ts.py`, enforced against drift by a CI step ("Zone drift check" in `.github/workflows/ci.yml:43-46`), and explicitly ignored by knip (`frontend/knip.json:18`) — but `rg "generated/endgameZones"` across `frontend/src` returns zero consumer imports. The frontend still reads the duplicated inline constants in `EndgameScoreGapSection.tsx:79-105` (`FIXED_GAUGE_ZONES`, `ENDGAME_SKILL_ZONES`) and `EndgameClockPressureSection.tsx:18,23` (`NEUTRAL_PCT_THRESHOLD`, `NEUTRAL_TIMEOUT_THRESHOLD`), plus `EndgamePerformanceSection.tsx:44-45` (`SCORE_GAP_NEUTRAL_MIN/MAX`).

**Evidence:** Comment in `tests/services/test_endgame_zones_consistency.py:8-11`: "_This test is throwaway — it gets deleted when Phase 66 switches FE consumers to import from frontend/src/generated/endgameZones.ts._" Phase 66 shipped but the switchover never happened. The generated file plus its codegen script plus the regex-based consistency test form a three-way drift-guard for constants that nobody consumes from the codegen'd source.

**Suggested fix:** Either (a) complete the Phase 66 switchover: swap the three inline constant blocks for imports from `@/generated/endgameZones`, delete the throwaway consistency test, remove the `frontend/knip.json:18` exclusion; or (b) if the direction changed and FE is meant to keep its own constants, delete the generated file, the `scripts/gen_endgame_zones_ts.py` script, the CI drift check, and the knip exclusion. The current state has all three sources present and only one of them (the regex consistency test) actually wired.

### DEAD-03 (Info) — `_finding_time_pressure_vs_performance` computes a finding that is filtered out downstream

**File:** `app/services/insights_service.py:384, 866-914`; consumer gate at `app/services/insights_llm.py:78`
**Finding:** `_finding_time_pressure_vs_performance` emits a `SubsectionFinding` that `_assemble_user_prompt` then filters out via `_SKIPPED_SUBSECTIONS = frozenset({"time_pressure_vs_performance"})`. The finding affects `findings_hash` (cache key), which is the only reason it still exists, but the docstring on the emitter admits the value "is NOT a clock-diff percentage in the registry sense" and sets `is_headline_eligible=False` permanently. The 10-bucket chart is rendered separately by `_format_time_pressure_chart_block`.

**Evidence:** The scalar's zone assignment reuses `avg_clock_diff_pct` as a metric id (line 906) but the value is a mean score in [0, 1], not a clock-diff percent. The registry entry for `avg_clock_diff_pct` is `(-10, +10)` so a score of 0.5 always resolves to "weak" — bogus, but it never reaches the prompt so nobody notices.

**Suggested fix:** Either (a) drop the finding entirely and have the chart payload (`time_pressure_chart` on `EndgameTabFindings`) own the cache-invalidation for this signal, or (b) pick a genuine metric id that matches the value semantics. Current state is a cache-side-effect disguised as a finding.

## 3. Bugs & Correctness Issues

### BUG-01 (Info) — `response_json` serialisation of `SubsectionFinding.value=NaN` depends on Pydantic behaviour, not explicit test

**File:** `app/services/insights_service.py:1314-1333` (`_compute_hash`)
**Finding:** The hash recipe relies on `model_dump_json` emitting NaN as `null` (then parsed back via `json.loads`). This is correct Pydantic v2 behaviour, but there is no explicit regression test. If a future dependency bump changes NaN handling (e.g. to `"NaN"` string), the hash silently changes and invalidates every cached entry without a user-visible symptom beyond cache-miss storms.

**Evidence:** Implicit reliance on `model_dump_json` → NaN → null conversion.

**Suggested fix:** Add a `test_compute_hash_handles_nan_values` unit test that constructs an `EndgameTabFindings` with at least one `value=float("nan")` finding and asserts the hash is stable across two calls. This already exists indirectly via `_empty_finding` integration tests, but not as a direct guard.

### BUG-02 (Info) — `compute_player_profile` uses `datetime.date.today()` without timezone context

**File:** `app/services/insights_service.py:267` (and companion at line 569 in `insights_llm.py`)
**Finding:** `today = datetime.date.today()` uses local process timezone. In production (UTC container) this is UTC. In tests running elsewhere it may drift. The `last_3mo` cutoff is `today - timedelta(days=90)` and the stale threshold uses the same anchor.

**Evidence:** Every other time-related call in the service uses `datetime.datetime.now(datetime.UTC)` (e.g. `compute_findings` at line 178: `as_of=datetime.datetime.now(datetime.UTC)`). The `today()` call is the one inconsistent spot.

**Suggested fix:** Replace `datetime.date.today()` with `datetime.datetime.now(datetime.UTC).date()`. Near-zero practical risk (prod is UTC), but aligns with the rest of the service.
