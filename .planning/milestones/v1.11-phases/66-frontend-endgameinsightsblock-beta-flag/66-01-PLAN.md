## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→/users/me/profile (authenticated) | Authenticated user reads own profile row including beta_enabled |
| client→/users/me/profile (PUT — authenticated) | Authenticated user submits arbitrary JSON body; pydantic filters to UserProfileUpdate fields |
| migration apply | Schema change executed by deploy user during CI/deploy or manual `alembic upgrade head` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-66-01 | Information Disclosure | GET /users/me/profile | accept | Endpoint already behind `current_active_user` dep; beta_enabled is not a secret — it is a UX-visible feature gate. No PII in the column. |
| T-66-02 | Tampering (mass-assignment) | PUT /users/me/profile with `{beta_enabled: true}` in body | mitigate | `UserProfileUpdate` Pydantic model (unchanged in this plan) only declares `chess_com_username` and `lichess_username`. Pydantic v2 `extra="ignore"` default silently drops unknown fields. Task 3 Test 3 asserts this guard holds. |
| T-66-03 | Denial of Service (migration lock) | Alembic upgrade on users table in prod | accept | `ALTER TABLE users ADD COLUMN … NOT NULL DEFAULT false` is fast in PostgreSQL 11+ (metadata-only default backfill). Users table is not high-write; brief ACCESS EXCLUSIVE lock is acceptable. |
| T-66-04 | Elevation of Privilege (direct DB flip abuse) | Flipping beta_enabled via psql | accept | Intentional by BETA-01 design — direct DB op is the ONLY way to flip the flag. Cohort control is a human process, not a code boundary. |
</threat_model>
