## Critical Validation Cases (Nyquist minimum)

The plan-checker should confirm at least these test cases exist in PLAN tasks:

1. **Convergence case:** Seed two games for the same user where game A plays `1.e4 e5 2.Nf3` and game B plays `1.Nf3 e5 2.e4` (or analogous pair). Both reach the same `result_hash`. Assert: row for `2.Nf3` candidate has `game_count == 1` (move-played) but `wins+draws+losses == 2` (resulting-position).
2. **Filter parity:** Resulting-position WDL must apply identical filters (time control, platform, color, recency) as move-played count. Assert via a fixture where one of the two converging games is filtered out — both counts must drop in lockstep.
3. **Single-order case (no transposition):** When only one move order reaches the position, `game_count` and `wins+draws+losses` must agree exactly.
4. **OpeningInsightFinding `n_position_reached`:** New schema field is plumbed end-to-end (repo → service → schema → response payload) and is `>= n_games` (move-played count) by construction.
5. **Wilson reuse:** Confirm `compute_confidence_bucket` is called with resulting-position W/D/L without any signature change; identical helper, different inputs.
6. **Gate stability:** `OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE` (n>=20) gate continues to operate on **move-played** count, not resulting-position count.
