## Summary

Phase 80.1 swaps WDL aggregation from move-played to resulting-position on two surfaces (Move Explorer + Opening Insights). The repository plumbing (`query_transposition_wdl`, `query_resulting_position_wdl`) is well-structured: filter parity is explicit, empty-list short-circuits avoid wasted round trips, missing-hash semantics (omit, not zero-tuple) are consistent across the two siblings, and SQL shapes mirror the established `query_next_moves` / `query_transposition_counts` patterns.

Service-layer rewires are minimal and respect the locked no-touch list (D-07): no schema field additions, no constant changes, no FE changes. Sequential awaits honor the CLAUDE.md AsyncSession constraint. The `SimpleNamespace` adapter pattern in `compute_insights` keeps `_classify_row` / `_compute_score` / `compute_confidence_bucket` / `wilson_bounds` byte-identical, which is the cleanest way to satisfy D-07.

No blockers. Two warnings: a real semantic concern about cross-color WDL blending in `compute_insights` (the new pos_wdl is called with `color=None` while the parent query is color-specific, so white_strengths/weaknesses cards show denominators that mix in black-perspective games and vice-versa), and a misleading comment that claims the two queries have parity when they do not. The test churn is well-absorbed via the `pos_wdl=None → mirror move-played` default in `_run_compute`, but one restructured assertion in `test_get_next_moves_populates_score_confidence_p_value` lost teeth and is now tautological. Minor: the prodcheck script does not enforce a read-only role at the DB level and uses an unparametrized type hint.

## Warnings

### WR-01: `compute_insights` blends cross-color games into the resulting-position WDL

**File:** `app/services/opening_insights_service.py:509-521`
**Issue:** `query_resulting_position_wdl` is invoked with `color=None`, while `query_opening_transitions` (which produced the rows being enriched) explicitly filters by `Game.user_color == color` at `app/repositories/openings_repository.py:836`. The pos WDL therefore aggregates BOTH white-perspective and black-perspective games for any `resulting_full_hash` that is reachable from both colors. The resulting `OpeningInsightFinding` in `white_strengths` / `white_weaknesses` (and the black equivalents) carries a denominator and W/D/L that include off-color games — a semantic that contradicts the phase's stated goal that "the row's headline number matches what the user sees after clicking through" (CONTEXT.md line 9; click-through position summaries in `analyze()` at `openings_service.py:162` apply `color=request.color`). The current behavior is documented and tested in `tests/repositories/test_opening_insights_repository.py::test_query_resulting_position_wdl_color_none_passes_through`, but the test only confirms the SQL contract — it does not justify the UX choice. For users who play both colors, common transposition-rich positions (e.g. Sicilian / English transpositions) will show differently-numbered cards versus their click-through summaries.

The locked decisions (D-03, D-05) intentionally kept a single `n_games` field, but neither D-03 nor D-05 explicitly addresses cross-color blending; this was not raised in CONTEXT.md or RESEARCH.md as an explicit consideration. If the planner intended `color=None` here, the comment at line 518 (see WR-02) is inconsistent with that intent; if the planner did not consider it, this is a real regression versus the click-through alignment goal.

**Fix:** Pass the row's color into `query_resulting_position_wdl` so the per-color finding enriches against per-color resulting-position WDL. Two viable shapes:

```python
# Option A: per-color batch call (mirrors color filter from query_opening_transitions)
pos_wdl_by_color: dict[str, dict[int, tuple[int, int, int]]] = {}
for color in colors_to_query:
    color_hashes = list({int(r.resulting_full_hash) for r in rows_by_color[color]})
    if color_hashes:
        pos_wdl_by_color[color] = await query_resulting_position_wdl(
            session=session,
            user_id=user_id,
            hash_list=color_hashes,
            ...,
            color=color,  # match the per-color filter from query_opening_transitions
            ...,
        )
# Then in the inner loop:
pos = pos_wdl_by_color[color].get(int(row.resulting_full_hash))
```

```python
# Option B: keep color=None but document the divergence loudly and align the
# click-through (analyze()) call sites to also use color=None when the user
# arrives via an Insights card. More invasive; spans frontend.
```

Option A is the simpler fix and preserves the click-through semantics. Either way, the call must align with whatever the position summary endpoint uses on click-through; today they diverge on the color axis.

### WR-02: Misleading filter-parity comment in `compute_insights`

**File:** `app/services/opening_insights_service.py:518`
**Issue:** The inline comment reads `color=None,  # mirror query_opening_transitions filter behavior`. This is misleading: `query_opening_transitions` does color-filter, just via an explicit predicate on the outer SELECT (`Game.user_color == color` at `app/repositories/openings_repository.py:836`) instead of via `apply_game_filters(color=...)`. The comment as written suggests the two queries are filter-equivalent and that `color=None` is the intentional way to "mirror" the parent — a future maintainer reading only this line would conclude the queries match when they do not. This is the same code path called out in WR-01.

**Fix:** Replace the comment with one that names the intentional divergence (or, if WR-01 is taken, make this site per-color and remove the misdirection):

```python
# Before:
color=None,  # mirror query_opening_transitions filter behavior

# After (if keeping color=None — documents the divergence honestly):
color=None,  # NOTE: query_opening_transitions filters by Game.user_color == color
            # via an explicit WHERE predicate, not via apply_game_filters. Passing
            # color=None here intentionally aggregates across both perspectives so
            # transposition matching ignores which side the user was on. See
            # WR-01 in 80.1-REVIEW.md for the alignment trade-off versus
            # the click-through position summary (which IS color-filtered).

# After (preferred, paired with WR-01 fix):
color=color,  # match query_opening_transitions' explicit per-color filter
```

## Info

### IN-01: Restructured `test_get_next_moves_populates_score_confidence_p_value` assertion is now tautological

**File:** `tests/test_openings_service.py:668-679`
**Issue:** The post-Phase 80.1 restructuring derives the helper input from the response itself:

```python
pos_n = entry.wins + entry.draws + entry.losses
expected_confidence, expected_p, _expected_se = compute_confidence_bucket(
    entry.wins, entry.draws, entry.losses, pos_n
)
expected_score = (entry.wins + 0.5 * entry.draws) / pos_n
assert entry.score == pytest.approx(expected_score, abs=1e-9)
```

`expected_score` is mechanically `(entry.wins + 0.5 * entry.draws) / (entry.wins + entry.draws + entry.losses)`. This assertion only fails if the service computes `score` from a formula different from `(W + D/2) / (W + D + L)`. If the service had a bug where it populated `entry.wins` from the wrong query (e.g. used `gc` denominator with move-played W/D/L), this assertion would still pass — `entry.wins`, `entry.draws`, `entry.losses` would all be the buggy values, and `expected_score` would match the buggy `entry.score`. The denominator-semantics coverage now lives entirely in `TestNextMovesTranspositionWdl`, which is fine, but this test no longer adds independent signal beyond "score formula is internally consistent."

**Fix:** Either delete this redundant test (the three transposition tests cover the meaningful invariants) or strengthen it by asserting against an independently-computed expected value. Since the fixture has no transposition (single-order parity), the test could assert against the move-played W/D/L counts seeded directly:

```python
# Seeded: 2 wins + 1 loss (no draws). Move-played n == pos n == 3.
assert entry.wins == 2
assert entry.draws == 0
assert entry.losses == 1
assert entry.game_count == 3
expected_confidence, expected_p, _ = compute_confidence_bucket(2, 0, 1, 3)
assert entry.score == pytest.approx((2 + 0) / 3, abs=1e-9)
assert entry.confidence == expected_confidence
```

Marked Info rather than Warning because the new `TestNextMovesTranspositionWdl` class adequately covers the WDL-source contract.

### IN-03: `prodcheck_80_1.py` does not enforce read-only at the DB session level

**File:** `scripts/prodcheck_80_1.py:90-91`
**Issue:** The script docstring claims read-only "by construction" because `compute_insights` does no writes. This is true today, but if a future change introduced a write inside `compute_insights` or one of its transitive calls (e.g. a cache-warm side-effect), the script would silently issue writes against prod. The comment acknowledges DSN-level enforcement is the user's responsibility ("DSN should use a read-only DB role"), but no defense-in-depth is in place at the session level.

**Fix:** Open the session in read-only mode so any accidental future write fails loudly:

```python
async with session_factory() as session:
    await session.execute(text("SET TRANSACTION READ ONLY"))
    request = OpeningInsightsRequest()
    result = await compute_insights(session=session, user_id=user_id, request=request)
```

Alternatively, assert the connected role is read-only at startup:

```python
async with engine.connect() as conn:
    role = (await conn.execute(text("SELECT current_user"))).scalar_one()
    if "ro" not in role.lower() and "readonly" not in role.lower():
        print(f"ERROR: DSN role '{role}' does not appear read-only.", file=sys.stderr)
        return 2
```

Marked Info because the script is a one-off D-09 sanity tool with explicit usage instructions, not a long-lived production code path.
