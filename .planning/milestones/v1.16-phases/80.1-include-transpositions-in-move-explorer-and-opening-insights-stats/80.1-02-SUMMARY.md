# Phase 80.1 Plan 02: Move Explorer Service Field Swap — Summary

`get_next_moves` now consumes the new `query_transposition_wdl` (Plan 01) and swaps the WDL-derived fields on Move Explorer rows from move-played to resulting-position. `game_count` stays move-played (D-01); `wins / draws / losses / win_pct / draw_pct / loss_pct / score / confidence / p_value` reflect resulting-position aggregation (D-02). The 57%→61% click-through mismatch on the Openings → Moves tab is closed by construction.

## What Changed

### `app/services/openings_service.py`

**Imports:**
- Added `import logging` and `query_transposition_wdl` to the existing `app.repositories.openings_repository` import group.
- Module-level `logger = logging.getLogger(__name__)` declared after all imports (E402 compliance).

**`get_next_moves` (function modified, lines 432–488 of new file):**
1. Added a single `await query_transposition_wdl(...)` call after the existing `query_transposition_counts` call. Filter parameters are byte-for-byte identical to the parent `query_next_moves` call (Pitfall 1 mitigation; threat T-80.1-05).
2. Replaced the assembly-loop body (was lines 451–480, now 467–498):
   - `gc = row.game_count` retained for `game_count` and `transposition_count` arguments.
   - New `pos = pos_wdl.get(row.result_hash)` lookup.
   - Defensive fallback branch (`if pos is None`) logs via `logger.warning` and reuses `row.wins / row.draws / row.losses` with `pos_total = gc`. Should not fire under correct filter parity but surfaces in dev/prod if it does (Pitfall 4 mitigation).
   - When `pos` is present: `w, d, lo = pos; pos_total = w + d + lo`.
   - All percentage / score / confidence / p_value derivations use `pos_total` as the denominator instead of `gc`.
   - The `compute_confidence_bucket(w, d, lo, pos_total)` call is the only invocation in the production code path; the fallback branch's `pos_total` happens to equal `gc` but is named consistently for clarity.

**No changes** to: imports for the schema, the `_fetch_result_fens` helper, the cutoff/result_hashes computation, the sort logic, or the response wrapping. Only the field-population loop and the new query call changed.

**No frontend changes.** Per D-07, schema field names (`wins / draws / losses / score / etc.`) stay identical; only their semantics shift. The Move Explorer Score-column `InfoPopover` copy is unchanged.

### `tests/test_openings_service.py`

**Restructured `TestNextMovesScoreConfidence::test_get_next_moves_populates_score_confidence_p_value`:**
- The expected-value cross-check now derives `pos_n = entry.wins + entry.draws + entry.losses` and feeds `pos_n` into both `compute_confidence_bucket(...)` and the `expected_score` formula. When the test fixture has no transposition (as is the case here — 2 wins + 1 loss all from a single SOURCE), `pos_n == entry.game_count` so the assertion still holds, but the documentation is now correct under the new dual-N contract.

**Added `TestNextMovesTranspositionWdl` class (3 tests):**

1. **`test_wdl_includes_transposition_games`** (Validation case #1, canonical convergence):
   - Game A: e4 from `0xCC01` → `0xCC03` (win). Game B: d4 from `0xCC02` → `0xCC03` (loss, transposition).
   - Asserts `entry.move_san == "e4"`, `entry.game_count == 1`, `entry.wins == 1`, `entry.losses == 1`, `entry.wins + entry.losses == 2`, `entry.score == 0.5` (1 win + 1 loss out of 2 pos visits), not `1.0` (the move-played win rate).

2. **`test_single_order_game_count_equals_wdl_total`** (Validation case #3, single-order parity):
   - Single Game A from `0xCC04` → `0xCC05`.
   - Asserts `entry.game_count == 1` AND `entry.wins + entry.draws + entry.losses == 1`.

3. **`test_transposition_wdl_filter_parity`** (Validation case #2, T-80.1-05 mitigation):
   - Game A (rated=True, e4) + Game B (rated=False, transposition d4), both reaching `0xCC08`.
   - With `rated=True`: drops Game B from BOTH `game_count` AND pos WDL (`game_count == 1` AND `wins+draws+losses == 1`).
   - With `rated=None`: Game B in pos WDL only (`game_count == 1` AND `wins+draws+losses == 2`).

## Deviations from Plan

### Auto-fixed Issues

None — plan executed as written.

### Implementation choices

**1. [Implementation choice — module logger]** The plan said "Log via existing logging if a fallback occurs." `app/services/openings_service.py` had no prior logger. I added `import logging` and a module-level `logger = logging.getLogger(__name__)` — consistent with stdlib logging pattern used elsewhere in `app/services/`. This is the standard pattern for Python service modules and required no new dependencies.

**2. [Cosmetic — E402]** Initial draft placed `logger = logging.getLogger(__name__)` between the repository imports and the schema imports, triggering ruff E402 ("module level import not at top of file"). Fixed by moving the logger declaration after all imports. No functional impact.

## Threat Surface Scan

No new attack surface. Both function calls (`query_transposition_counts` and `query_transposition_wdl`) operate on the same user-scoped `game_positions` filter pattern, route through the same `apply_game_filters` 8-parameter set, and expose no new fields in `NextMoveEntry`. T-80.1-04 (information disclosure), T-80.1-05 (filter-parity tampering), T-80.1-06 (extra DB round-trip DoS) are all mitigated as documented in the plan's threat model — and the canonical filter-parity test (`test_transposition_wdl_filter_parity`) provides ongoing regression coverage for T-80.1-05.

No threat flags.
