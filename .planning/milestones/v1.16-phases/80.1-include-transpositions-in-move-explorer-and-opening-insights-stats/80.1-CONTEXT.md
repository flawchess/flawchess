# Phase 80.1: Include transpositions in Move Explorer and Opening Insights stats - Context

**Gathered:** 2026-05-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Switch the WDL/score aggregation source on two surfaces from *games-that-played-the-exact-candidate-move-from-this-exact-entry* (move-played) to *all-games-visiting-the-resulting-position* (transposition-inclusive), so the row's headline number matches what the user sees after clicking through.

**Two surfaces in scope:**

1. **Move Explorer Moves rows** (`Openings → Moves` tab, `MoveExplorer.tsx`)
   - `game_count` column **stays** as games-that-actually-played-this-move (move-played count). The existing `TranspositionInfo` popover at `MoveExplorer.tsx:359-365` ("Position reached in N total games (M via other move orders)") already explains the gap and stays as-is.
   - `wins / draws / losses / win_pct / draw_pct / loss_pct / score / confidence / p_value` are recomputed over all games visiting `result_hash` (the candidate's resulting position full_hash).
   - The 57%→61% click-through mismatch the roadmap calls out goes away by construction.

2. **Opening Insights findings** (`OpeningInsightFinding`, `OpeningFindingCard`)
   - Classification (weakness vs. strength), Wilson score-test confidence, ranking key, and the displayed score bar all switch to resulting-position WDL.
   - The `OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE` (n>=10) **surfacing gate stays on move-played count** — only flag positions the user reached via the called-out move meaningfully often. (Locked by ROADMAP.)
   - MG-entry eval / CI / confidence on the cards (added in quick task 260506-u2b) are already transposition-inclusive — `query_opening_phase_entry_metrics_batch` keys off `resulting_full_hash`. **No work needed on the eval pillar.**

**Out of scope (locked "keep as-is" 2026-05-07):**

- No `n_position_reached` / `move_played_count` field added to `OpeningInsightFinding`.
- No tooltip / info-popover copy changes (Move Explorer Score column InfoPopover and `OpeningFindingCard` score-badge tooltip stay as-is).
- No score-pivot / effect-size threshold recalibration in this phase. Sanity-check during execute is fine; recalibration is its own follow-up if numbers actually shift.
- No change to Move Explorer Moves row sort order (stays by `game_count` desc).
- No change to Move Explorer score muting threshold semantics (mute trigger keeps reading the row's WDL denominator — which is the field we're switching, so the mute now triggers off resulting-position N by default; this is the intended consequence of the switch and matches the resulting-position semantics).

</domain>

<decisions>
## Implementation Decisions

### Move Explorer Moves rows (D-01)
- **D-01:** In `get_next_moves` (or wherever the Moves rows are assembled), keep `game_count` from `query_next_moves` (move-played count), but recompute every WDL-derived field (`wins / draws / losses / win_pct / draw_pct / loss_pct / score / confidence / p_value`) from a single batch query over `result_hash` that returns W/D/L per resulting position. Denominator for percentages and for Wilson confidence/p_value = resulting-position W+D+L (same as the position summary the user lands on after clicking).
- **D-02:** Implementation choice (extend `query_transposition_counts` to also return W/D/L per `result_hash`, or add a sibling `query_transposition_wdl`) is left to the planner — both shapes are acceptable. Single batch query keyed on the list of `result_hash` values returned by `query_next_moves`. Filters must mirror `query_next_moves` exactly (apply the same `apply_game_filters` invocation) so the row's WDL is consistent with the rest of the page.

### Opening Insights findings (D-03)
- **D-03:** Switch classification, Wilson confidence, ranking, and the displayed bar from move-played WDL to resulting-position WDL. Concretely: `query_opening_transitions` returns one row per `(entry_hash, candidate_san)`; for the surviving rows, recompute W/D/L over games visiting `resulting_full_hash` (independent of which move order brought them there) and replace `_classify_row`, `_wilson_bounds`, ranking key, and `OpeningInsightFinding.{score, wins, draws, losses, n_games, ci_low, ci_high, p_value, confidence}` with the resulting-position values.
- **D-04:** **Surfacing gate stays move-played.** The `n_games >= OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE` HAVING clause in `query_opening_transitions` continues to gate on the move-played count. Don't touch that filter — only the post-filter classification/score numbers flip to resulting-position. (ROADMAP-locked.)
- **D-05:** **`n_games` field on `OpeningInsightFinding` becomes resulting-position N** (since the field drives the displayed denominator on the bar / Wilson CI). Keep it as a single `n_games` field; do **NOT** add a parallel `n_position_reached` / `n_move_played` field. ("Keep as-is" — user 2026-05-07.) The honesty trade-off (card displays resulting-position N while the surfacing gate uses move-played N) is accepted; the existing transposition framing across the app is consistent enough.
- **D-06:** Implementation hint: the resulting-position WDL aggregation already exists conceptually in the code that powers the position summary the user lands on. The simplest addition is a second pass / sub-CTE that joins `game_positions` on `resulting_full_hash` to compute resulting-position W/D/L per candidate row, in the same single round trip as `query_opening_transitions`. Planner picks the cheaper SQL shape.

### No-touch list (D-07)
- **D-07:** **Locked "keep as-is" 2026-05-07** — do not modify any of the following in this phase:
  - `OpeningInsightFinding` schema (no new fields).
  - Move Explorer Score column `InfoPopover` copy ("Score = (W + 0.5·D)/N" stays).
  - `OpeningFindingCard` score-badge tooltip copy.
  - Move Explorer Moves row sort order (stays `game_count` desc).
  - Move Explorer score muting threshold value or constant. The mute trigger continues to read the row's WDL denominator; that denominator switches from move-played N to resulting-position N as a natural consequence of D-01, and that's the intended behavior.
  - `OPENING_INSIGHTS_SCORE_PIVOT`, `OPENING_INSIGHTS_MAJOR_EFFECT`, `OPENING_INSIGHTS_MINOR_EFFECT` constants. No recalibration in this phase. If finding volumes shift meaningfully under the new aggregation, that's a follow-up phase, not this one.

### Test churn (D-08)
- **D-08:** Every test asserting WDL/score numbers in `tests/services/test_opening_insights_*` and `tests/services/test_openings_service_*` (and any move-explorer / stats fixture asserting per-row score / win_pct) needs updating. Expected — call it out in PLAN.md so executor doesn't treat the cascade as a regression. Executor regenerates expected values from fixture data using the new aggregation rule, not from prior assertions.

### Sanity check during execute (D-09)
- **D-09:** During execute-phase, run prod-DB queries (read-only, via the existing `flawchess-prod-db` MCP) on a couple of representative users to confirm the post-switch finding volumes haven't collapsed (e.g. zero strengths) or exploded. If volumes shift dramatically, **flag and stop** — don't auto-recalibrate. The user wants to see the numbers before committing to threshold changes. Threshold recalibration is explicitly out of scope (D-07).

### Claude's Discretion
- **Schema field semantics.** `OpeningInsightFinding.{score, wins, draws, losses, n_games, ci_low, ci_high, p_value, confidence}` all flip to resulting-position. The `Field()` validator on `n_games` is `ge=OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE` today — check whether resulting-position N is always ≥ move-played N (it is, by definition: every move-played game also visits the resulting position), so the validator stays satisfied. Confirm during planning.
- **Ranking key.** The Phase 75 sort uses Wilson lower-bound for weaknesses and upper-bound for strengths. After D-03 those bounds are computed on resulting-position WDL — same formula, different inputs. No code change to the ranking helper itself; just feed it the new W/D/L.
- **Backend payload shape.** Single round trip; no new top-level response field needed for either surface.
- **`MoveExplorer.tsx` Moves row test.** Update fixtures so the row's `game_count` and `wins+draws+losses` can legitimately differ — today many fixtures have them equal because no transpositions were modeled.

</decisions>

<deferred>
## Deferred Ideas

- **`n_position_reached` field on `OpeningInsightFinding`.** Honesty disclosure on the card so users can see "Score across X games via any move order; you played [move] in Y of them." Considered and rejected for this phase ("keep as-is" 2026-05-07). Could be revisited in a UX-polish follow-up if the resulting-position framing turns out to confuse users.
- **Tooltip / info-popover copy refresh.** Move Explorer Score column InfoPopover and `OpeningFindingCard` score-badge tooltip both currently say "Score = (W+0.5·D)/N" without mentioning transposition aggregation. A future copy pass could clarify "across all games reaching this position, via any move order." Not done in this phase.
- **Score-pivot / effect-size threshold recalibration.** If the post-switch sanity check (D-09) shows finding volumes shifting materially (e.g. resulting-position scores cluster tighter and fewer findings cross the ±0.05 / ±0.10 effect bands), recalibration is its own follow-up phase. Use the benchmark DB and per-user prod queries to derive new pivots; don't change them under cover of this phase.
- **Move Explorer Moves row sort by resulting-position score.** Today sorts by `game_count` (move-played) desc. After this phase, sorting by `score` could surface "best resulting positions you've reached via this move" but that's a UX shift, not a bug fix; keep deferred.
- **Move Explorer muting threshold tied to move-played count.** With this phase's switch, the row's WDL denominator becomes resulting-position N, and the n<20 mute trigger consequently keys off that. If users feel rows get unmuted "too easily" when transpositions inflate the denominator, anchor the mute on move-played count instead — as a follow-up tweak.
