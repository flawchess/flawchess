# Milestone v1.28: Tactic Tagging

**Status:** In progress
**Phases:** 124, 125, 126 — 3 phases
**Source:** SEED-039 (tactic family / cause-of-error flaw tags). Decision record: `.planning/notes/tactic-tagging-architecture.md`.

## Overview

v1.28 adds the "cause of error" axis to the existing flaw taxonomy — naming *why* a flawed move was bad in tactical terms by reading the already-stored refutation `pv` at `flaw_ply + 1`. No new engine pass: v1.27's remote eval pipeline already stores the refutation line for both colors, and `game_flaws` already materializes both sides' mistakes/blunders, so the detector is pure CPU.

The milestone delivers in three phases:

- **Phase 124 — Schema + Tactic Detector.** A single Alembic migration adds `tactic_motif` (nullable SmallInteger enum, one per flaw) and `tactic_piece` (nullable SmallInteger, python-chess PieceType) to `game_flaws`. The cook-heuristic detector is reimplemented in original code (no AGPL source copied — heuristics are not copyrightable) and wired into both the `classify_game_flaws` eval-drain flow-through and `backfill_flaws.py`. A hand-labeled per-motif fixture set validates precision-first accuracy before any backfill runs.
- **Phase 125 — Backfill Tactic Motifs.** Idempotent `backfill_flaws.py` pass over ~131k self-eval'd games (`full_evals_completed_at IS NOT NULL`). Lichess-eval-only flaws (~13.6k games) keep `tactic_motif = NULL` — they fill in as a side effect of the existing tier-3 idle-fleet full-eval pipeline with no bespoke tooling.
- **Phase 126 — Comparison Stats + Frontend.** `GET /api/library/tactic-comparison` returns per-motif player vs opponent rates (normalized per game or per 100 blunders) with Wilson-based significance verdict, honoring all game filters. The Library surface gains motif chips on flaw cards and a MiniBulletChart you-vs-opponent motif comparison grid (reusing the v1.25 pattern), with full mobile parity at 375px.

Key dependencies already satisfied by v1.27: `game_positions.pv` is stored at `flaw_ply + 1` for both colors; `game_flaws` materializes both players' flaws; `is_opponent_expr(ply, games.user_color)` is a tested query-time helper. The genuine technical risk is the detector itself (reimplementing + validating the cook heuristics — Q-011 in `.planning/research/questions.md`).

## Phases

### Phase 124: Schema + Tactic Detector

**Goal**: The system can detect and store a tactic motif for any flawed move that has a stored refutation PV
**Depends on**: Nothing (first phase of this milestone; builds on v1.26/v1.27 infrastructure)
**Requirements**: TACSCH-01, TACSCH-02, TACDET-01, TACDET-02, TACDET-03, TACDET-04
**Success Criteria** (what must be TRUE):
  1. A new Alembic migration adds nullable `tactic_motif` (SmallInteger enum) and `tactic_piece` (SmallInteger) columns to `game_flaws`; the dev DB migrates cleanly and existing rows carry NULL in both columns
  2. Given a stored `game_positions.pv` at `flaw_ply + 1`, the detector returns at most one motif name from the implemented MVP motif set (finalized at phase-discussion time — provisional starting point: fork, pin, skewer, hanging-piece, back-rank/mate, double-check) with a fixed priority order when multiple motifs fire on the same refutation line
  3. The detector leaves `tactic_motif = NULL` when confidence is low rather than guessing; a hand-labeled per-motif fixture set (drawn from own prod flaws, not from cook.py output) passes with precision-first accuracy
  4. Motif detection runs inside `classify_game_flaws` (eval-drain flow-through) and `backfill_flaws.py` (recompute path) for both the player's and the opponent's flaws, with no new Stockfish invocation

### Phase 125: Backfill Tactic Motifs

**Goal**: All existing self-eval'd game flaws carry their tactic motif and piece tags; coverage is honest and verifiable
**Depends on**: Phase 124 (validated detector + migrated schema)
**Requirements**: TACSCH-03
**Success Criteria** (what must be TRUE):
  1. Running `backfill_flaws.py` over all games with `full_evals_completed_at IS NOT NULL` (~131k) populates `tactic_motif` and `tactic_piece` for every flaw row where the detector fires with high confidence (NULL rows reflect genuine low-confidence positions, not skipped runs)
  2. Lichess-eval-only flaws (games with `lichess_evals_at IS NOT NULL` and no `full_evals_completed_at`) keep `tactic_motif = NULL` with no bespoke backfill job; their PVs fill in via the existing tier-3 idle-fleet full-eval pipeline
  3. Backfill is idempotent: re-running it over the same games produces the same result without duplicating or corrupting rows
**Plans**: TBD

### Phase 126: Comparison Stats + Frontend

**Goal**: Players can see which tactic motifs they allow more or less than their opponents, with significance gating and full mobile parity
**Depends on**: Phase 125 (populated `tactic_motif` rows)
**Requirements**: TACCMP-01, TACCMP-02, TACCMP-03, TACUI-01, TACUI-02, TACUI-03
**Success Criteria** (what must be TRUE):
  1. `GET /api/library/tactic-comparison` returns per-motif rates normalized per game (or per 100 blunders) for player vs opponents, with the player/opponent split derived at query time via the existing `is_opponent_expr` helper — no new column
  2. Each motif comparison entry carries a significance verdict computed by the project's existing Wilson-based chess-score significance utility; the endpoint honors all game filters (time control, platform, rated, opponent type, recency, color) and severity filter, consistent with other Library flaw surfaces
  3. Each flaw card in the Library displays its `allowed` motif as a family-colored chip with a definition popover, using the same chip pattern as the existing flaw-tag taxonomy
  4. The you-vs-opponent motif comparison surface (MiniBulletChart grid: measure + CI + benchmark zone where available, per-motif tooltips disclosing definition, sign convention, and filter interaction) renders on the Library page with a section-level sample gate below which the comparison is withheld
  5. All chips, comparison bullets, and interactive elements render correctly at 375px mobile width with `data-testid` + ARIA labels matching the project's browser-automation rules
