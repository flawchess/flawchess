# Phase 185 — UI Design Contract

> Visual and interaction contract for the Bots page roster transpose (rung-rows x style-columns) and the persona win-stars row. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Star icon gaps (`gap-0.5`≈2px rounds to the existing Tailwind scale, see Exceptions), avatar-to-name gap |
| sm | 8px | Grid cell gap (`gap-2`, unchanged from the current `PersonaCard` grid), header-row-to-grid gap |
| md | 16px | Card internal padding (`p-2`→ kept at 8px per existing `PersonaCard`, see Exceptions), page horizontal padding (`p-4`) |
| lg | 24px | Gap between the player-rating line and the grid (`gap-6` on the root flex column, unchanged) |
| xl | 32px | Not used this phase |
| 2xl | 48px | Not used this phase |
| 3xl | 64px | Not used this phase |

Exceptions:
- `PersonaCard`'s existing `p-2` (8px) internal padding and `gap-1` (4px) internal stack gap are UNCHANGED — this phase only appends a stars row inside the existing card, it does not restyle the card shell.
- The stars row uses Tailwind's `gap-0.5` (2px) between the 3 star glyphs — an intentional exception to the 4px-multiple rule for icon-internal spacing, consistent with `frontend/src/lib/theme.ts`'s existing icon-toggle patterns (small glyphs read cramped at 4px+ gaps). Icon size itself (14px) and outer row spacing all stay on the 4px grid.
- **Executor note:** the non-4-multiple values in this contract (`gap-0.5` 2px star gap; pre-existing `p-2`/`gap-1` card shell) are intentional, documented exceptions carried in this spec — do NOT "fix" them to 4px-grid values during implementation.

## Typography

Unchanged from the existing `PersonaCard`/`PersonaGrid` — this phase adds no new text sizes or weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / player-rating caption | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label / persona name, ELO label | 14px (`text-sm`) | 500 (medium) for name, 400 (regular) for ELO label | 1.5 |
| Heading / style column header | 14px (`text-sm`) | 600 (semibold) | 1.2 |
| Display | not used | — | — |

**This phase introduces zero new font weights.** The rendered surface shows three weights — 400, 500, 600 — but only 400 (regular, body/ELO) and 600 (semibold, style headers) are new-to-this-phase elements; 500 (medium) is not part of this phase's declared palette.

**Locked pre-existing exception (carried over from Phase 183/184):** `font-medium` (500) on the persona name label (`PersonaCard.tsx` line 58) is a pre-existing legacy weight this phase does not touch, add, or remove. It is documented here as an exception, not counted toward this phase's weight budget. Do not "fix" it to 400 or 600 to force a clean 2-weight surface — that would be an unplanned scope change to a component this phase deliberately leaves untouched.

**New elements introduced by this phase** (style header row, stars row) use only 400/600 — no third weight is added anywhere new. No new `text-xs` or smaller is introduced anywhere (CLAUDE.md floor).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Unchanged — persona cards remain a one-tap `onSelect` action (no visible button label; `aria-label="{persona.name}, {persona.calibratedLabel} ELO"`, extend to include win count per Interaction Notes below). "Custom" button at the grid's bottom is unchanged. |
| Empty state heading | Not applicable — the grid always renders all 24 personas (`PERSONA_REGISTRY` is a static compile-time constant); there is no "no personas" empty state. The win-stars row itself has its own zero-state (see below), not a page-level empty state. |
| Empty state body (win-stars, wins = 0) | No copy — render 3 grey-outline stars silently. Do not show a "0 wins" or "no wins yet" text label; the outline stars ARE the zero-state, consistent with a rating/review-star convention. `aria-label="0 wins"` on the row for screen readers only (visually silent). |
| Error state | Not applicable this phase for the grid itself (no network fetch on `PersonaGrid`/`PersonaCard` — win counts are fetched once at `Bots.tsx` page level per RESEARCH.md Pattern 3). If `useBotPersonaWins()` errors or is still loading, `PersonaCard` receives `winsForPersona: undefined` and renders the same 3-grey-outline-star zero-state as an honest "no data yet" rather than fabricating a 0 that could misrepresent an actual win history once the query resolves. This is a deliberate loading/error-state merge: both look identical (all-outline stars), because a wrong `0` read as "you have never won" would be actively misleading if the fetch simply failed — silence is safer than a false negative here. |
| Destructive confirmation | Not applicable — no destructive action introduced by this phase. |

| Loading / not-yet-fetched | Win-stars row before `useBotPersonaWins()` resolves | ✅ covered | Renders identically to the zero-win state (3 grey-outline stars) — `winsForPersona` prop defaults to `undefined`, treated as 0 for display purposes only (never blocks card interactivity; the card remains clickable during the win-count fetch) |

## Layout & Interaction Notes (non-template, phase-specific detail)

These supplement the template sections above with the concrete transposed-grid contract, since the standard template fields (spacing/typography/color) don't fully capture a grid restructuring.

### Grid structure

- **Header row:** one row, 4 cells, one per style in `STYLE_SECTION_ORDER` (`Attacker`, `Trickster`, `Grinder`, `Wall`), each cell's text colored via the existing `STYLE_ACCENT` map (verbatim, unchanged hex/oklch values). `text-sm font-semibold`, centered over its column. Rendered as a CSS grid row (not a `<table>` — keep the existing div/grid approach), matching the persona-cell grid's own column template so header and body columns align exactly.
- **Body rows:** 6 rows, one per `Rung` ascending 800 (top) → 1800 (bottom). Each row is `RUNGS.flatMap`'s outer loop; within a row, 4 `PersonaCard`s in `STYLE_SECTION_ORDER` order (inner loop) — DOM order becomes rung-major, replacing the current style-major order (Pitfall 2: the existing DOM-order test must be REWRITTEN, not patched).
- **No row labels:** per the locked decision, no rung number/label renders in a left-hand gutter or row header. `PersonaCard`'s own `calibratedLabel` text is the only strength indicator per row.
- **Responsive columns:** the grid stays 4 columns at all breakpoints (never collapses to fewer columns on mobile, since the whole point of the transpose is 4 style columns) — this differs from the CURRENT `grid-cols-3 sm:grid-cols-6` pattern (which reflowed by breakpoint because it iterated 6-per-style-section). Recommend `grid-cols-4` unconditionally; verify at plan time that 4 columns of `PersonaCard` (48px avatar + text) remain legible at a 320px viewport (4 columns × ~72-80px each) — if cramped, reduce `AVATAR_SIZE_PX` is explicitly OUT of scope (card internals unchanged), so the fallback is horizontal scroll or a slightly reduced grid gap, not a card redesign.
- **`data-testid` migration:** the current `data-testid="bots-persona-section-${style}"` wraps a per-style `<section>` (6 cards). After the transpose there is no natural per-style DOM container of that shape — recommend moving this test-id to the header-row cell (`data-testid="bots-persona-header-${style.toLowerCase()}"`) rather than dropping it, so tests can still assert per-style presence/accent color at the header level (Pitfall 2 follow-through). Every existing `data-testid="bots-persona-card-${persona.id}"` on individual cards is UNCHANGED.

### Stars row

- **Position:** appended at the bottom of `PersonaCard`'s existing vertical stack (avatar → name → ELO label → **stars row**, new). Card shell padding/gap otherwise unchanged.
- **Rendering:** exactly 3 `lucide-react` `Star` icons at `size={14}`, `gap-0.5` between them, horizontally centered under the ELO label. `Math.min(wins, MAX_DISPLAY_STARS)` filled (gold, `fill="currentColor"`, `color: STAR_FILLED`) from left to right, remainder outline (`fill="none"`, `color: STAR_EMPTY`). `MAX_DISPLAY_STARS = 3` as a named constant (no magic number).
- **Accessibility:** the 3 `Star` glyphs are `aria-hidden="true"` individually; the wrapping `<span>` carries a single `aria-label={`${wins} win${wins === 1 ? '' : 's'}`}` (or `"0 wins"` for the loading/error/zero-merged state) so screen readers get one clean announcement instead of 3 redundant icon labels. This aria-label is a SEPARATE announcement from the card's own `aria-label` (persona name + ELO) — do not merge them into one long string; keep the stars row's `aria-label` on its own `<span>` wrapper so assistive tech reads persona identity first, then win count, as two distinct pieces of info within the same button.
- **Data source:** `winsForPersona: number | undefined` passed down through `PersonaGrid` → `PersonaCard` as a plain prop (Pattern 3 — no `useQuery` inside either component). `undefined` renders identically to `0` (all-outline).
- **Cap ownership:** backend `GET /bots/persona-wins` returns the RAW win count (uncapped) per RESEARCH.md's Assumption A2 recommendation; the frontend applies `Math.min(wins, 3)` at render time. This keeps a future "hover for exact count" or detail view a one-line addition later without a backend contract change.

### Focal point

- The header row's accent-colored style names are the primary visual anchor; the eye reads them first, then descends the rung rows.

### What does NOT change

- `PersonaCard`'s avatar circle (48px), name (`text-sm font-medium`), and ELO label (`text-sm text-muted-foreground`) — all untouched.
- The "Custom" button at the grid's bottom, the player-rating caption line above the grid, and the `InfoPopover` — all untouched, still rendered above/below the (now-transposed) grid in the same relative position.
- Card tap target, hover state (`pointer-fine:hover:bg-inactive-bg-hover`), and border/background (`border-border bg-card`) — unchanged.
