## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added extensionless relative TypeScript import resolution to frontend-alias-hook.mjs**
- **Found during:** Task 3 (calibration-parity.check.mjs)
- **Issue:** `frontend-alias-hook.mjs` only rewrote `@/`-prefixed specifiers. `selectBotMove.ts`/`mctsSearch.ts` (and their many siblings under `frontend/src/lib/engine/`) import each other via extensionless relative specifiers (`./mctsSearch`, `./types`, `./select`, ...) — this resolves fine under TypeScript's `bundler` moduleResolution (Vite) but Node's native ESM loader requires an explicit extension. Phase 165's gem-elo harness never hit this because every module it imports via `@/` (`maiaEncoding.ts`, `gemMove.ts`, `liveFlaw.ts`, `uciParser.ts`) only imports other modules via `@/` or bare package specifiers — no relative imports. Without a fix, `import { selectBotMove } from '@/lib/engine/selectBotMove'` failed with `ERR_MODULE_NOT_FOUND` on the very first transitive relative import, blocking Task 3 entirely (CAL-02's whole point).
- **Fix:** Added a narrow resolve-hook branch: for a relative specifier (`./`/`../`) whose importer lives under `frontend/src`, if the extensionless path does not exist on disk but the `.ts` sibling does, resolve to that file. Falls through to `nextResolve` for every other case (bare packages, scripts' own relative imports, genuine extensioned imports) — unchanged behavior for every existing call site.
- **Files modified:** `scripts/lib/frontend-alias-hook.mjs`
- **Verification:** Re-ran both `gem-parity.check.mjs` (unaffected, still PASS) and a tiny `gem-elo-calibration.mjs` smoke run (still produces its usual TSV) after the change, plus the new `calibration-parity.check.mjs` (PASS) and a negative-control mutation of that same check (fails with a nonzero exit, confirming the assertions are load-bearing).
- **Committed in:** `505947a9` (part of the Task 3 commit)
