<objective>
Turn the already-shipped ECO database into a queryable **prefix index**, and commit the corpus-parity property test that keeps it honest.

`frontend/src/lib/openings.ts` today fetches `/openings.tsv` once, strips move numbers, and builds a `Map<sanSequence, Opening>` keyed by the FULL line — used only to *name* an opening. The book needs the inverse query: given the SAN history so far, is `history + candidateMove` still a prefix of some ECO line? That is a `Set<string>` of every prefix of every line, derived from the SAME parse.

**Why the corpus-parity test is the load-bearing artifact here:** the book's failure mode is invisible. A prefix set that matches nothing still lets the bot produce perfectly legal moves — it just quietly searches every ply and nobody notices. RESEARCH.md resolved SAN parity empirically with a throwaway script (0 illegal / 0 mismatches over all 3,641 lines), but a throwaway script guards nothing. A future `openings.tsv` refresh or a `chess.js` major bump could silently reintroduce a mismatch. This plan commits that check as a permanent property test over the REAL shipped asset.

Purpose: the candidate-generation half of SC2.
Output: `loadOpeningPrefixSet()` + `frontend/src/lib/__tests__/openings.test.ts`.
</objective>

Change `buildLookup()`'s cached promise from `Promise<Map<string, Opening>>` to `Promise<OpeningLookup>`, where `OpeningLookup` is a new exported interface with two fields: `fullLineMap: Map<string, Opening>` (exactly today's map, unchanged semantics) and `prefixSet: Set<string>`.

Inside the existing row loop, after computing the line's SAN token array (split `pgnToSanSequence(pgn)` on spaces and drop empties), do BOTH:
- `fullLineMap.set(tokens.join(' '), { eco, name })` — today's behavior, preserved.
- for `len` from 1 through `tokens.length`, `prefixSet.add(tokens.slice(0, len).join(' '))` — every non-empty prefix of the line, not just the full line. This is the entire delta. Corpus is 3,641 rows, longest line 36 SAN tokens, so the total insert count is a few tens of thousands of short strings, built once and cached at module scope.

Keep the existing `.catch()` that resets `lookupPromise = null` and rethrows — the retry-on-transient-failure behavior must survive.

Update `findOpening()` to read `(await buildLookup()).fullLineMap`; its exported signature and longest-match-walk behavior are unchanged. `preloadOpenings()` is unchanged.

Add a new export `loadOpeningPrefixSet(): Promise<ReadonlySet<string>>` returning `(await buildLookup()).prefixSet`. This is the single seam the bot's book (plan 04) consumes. It must NOT swallow errors — it returns the underlying rejected promise so its caller can decide (plan 04 latches the bot permanently out of book on a rejection rather than silently degrading).

Add a module doc comment on `openings.ts` naming both consumers and their two DIFFERENT query shapes, so nobody later conflates them: `findOpening()` walks `len` DOWN from the history length against `fullLineMap` (naming an already-played sequence, longest match wins); the book walks each LEGAL MOVE against `prefixSet` with key `[...history, candidateSan].join(' ')` (a fixed-length forward membership test). Same parsed data, opposite direction.

Do not add a second `fetch('/openings.tsv')` anywhere.

Write these tests:

**`describe('SAN parity (whole corpus)')` — the standing guard.** Parse the real TSV directly in the test (do not go through `openings.ts` — this test validates the ASSET against `chess.js`, not the module). For every data row: strip move numbers the same way `pgnToSanSequence` does, replay each SAN token through a fresh `chess.js` `Chess()` inside a try/catch, and compare the returned `move.san` against the TSV's own token. Accumulate every failure into an array of `{ line, eco, ply, tsvSan, chessJsSan }` records and assert `expect(failures).toEqual([])` — an array assertion, so a regression NAMES the offending ECO line instead of just printing `false`. Also assert `expect(rowCount).toBeGreaterThan(3000)` so a truncated/empty asset (fetch stub broken, file moved) fails loudly instead of passing vacuously on zero rows.

**`describe('prefix set')`.**
- Against a SYNTHETIC two-row TSV (stub `fetch` with a hand-written header + two rows, e.g. an `1. e4 c6 2. d4 d5` line and an `1. d4` line): assert `loadOpeningPrefixSet()` returns exactly the expected prefix strings (`'e4'`, `'e4 c6'`, `'e4 c6 d4'`, `'e4 c6 d4 d5'`, `'d4'`) and nothing else. Deterministic, corpus-independent — this is what pins the prefix-expansion logic itself.
- Against the REAL TSV: assert `prefixSet.has('e4')`, `prefixSet.has('e4 e5')`, `prefixSet.has('d4 d5')` are all true; and assert the set does NOT contain `'Nf3 Nf6 Ng1 Ng8'` (a knight shuffle — verified absent from the corpus: `grep -c "Nf3 Nf6 Ng1" frontend/public/openings.tsv` returns 0). Do NOT use a Bongcloud-style sequence as the negative case — `grep -ci bongcloud` returns 1, the corpus DOES name it. If you pick a different negative case, grep-verify its absence first.
- Assert `findOpening(['e4', 'c6'])` still resolves a non-null `{eco, name}` against the real TSV — proof Task 1's refactor did not break the pre-existing name lookup.
