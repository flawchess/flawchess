# Phase 169.5 Plan 04: Wire the Book into the Game Loop Summary

**`useBotGame` now answers the early plies from the ECO book (one Maia eval, zero searches, zero Stockfish grades), latches out of book one-way, warms both engines on mount, and refuses any draw it has no evaluation for.**

## Performance

- **Duration:** ~55 min
- **Completed:** 2026-07-13
- **Tasks:** 3 (committed together — tasks 2 and 3 both extend the same `describe('book')` block, and task 1's hook changes cannot pass their own gate without the mocks task 2 adds)
- **Files modified:** 4

## Accomplishments
- `runBotTurn` resolves the bot's move through a local `resolveMove()` returning `{ uci, fromBook }`: while in book it calls the module-scope `resolveBookMove` helper (prefix set → live SAN history → legal moves → one Maia policy → `selectBookMove`); on `null` the one-way `hasLeftBookRef` latch fires at that single decline point and the turn falls through to `selectBotMove`
- The `Promise.all([resolveMove(), delay(...)])` shape is preserved, so a near-instant book move still lands inside the reveal band instead of snapping back at zero latency
- Book and searched moves share ONE commit path: the same `chargeableElapsedMs()` → `flagIfOutOfTime()` → `commitMove()` pipeline. A book move is debited its real elapsed time and gets the Fischer increment; there is no untimed bypass and no second commit path
- Both engines are warmed on mount (`pool.warm()`, `queue.warm()`), and `loadOpeningPrefixSet()` is kicked off fire-and-forget so the ECO fetch is in flight before the bot's first turn
- **Latent 169 D-01 defect fixed:** `lastRootPracticalScoreRef` is now `number | null`, initialized and reset to `null` (= no eval computed yet this game), and `wouldBotAcceptDraw` refuses on that sentinel before it looks at the board. Suppressing the post-commit grade on book plies made this reachable for real, not hypothetically: the draw gate's endgame condition opens on **queens-off alone**, and the shipped corpus contains queens-off lines inside the ply cap (`openings.tsv:1065` trades queens by ply 9)
- 7 new hook tests + 1 new draw-gate unit test; the full frontend suite is green at **151 files / 1929 tests**

## Decisions Made
- **`mockPolicy` defaults to `{}` in `beforeEach`.** An empty policy clears no candidate above `BOOK_POLICY_FLOOR`, so the book declines on the bot's first turn and every pre-existing (pre-169.5) test keeps exercising the searched-move path exactly as before. The book tests opt IN by resolving a real policy. Without this, the book would silently engage in ~30 unrelated tests
- **The line-724 draw test was made honest, not weakened.** It passed *before* this plan only because `mockGrade` resolved an empty `Map`, leaving the ref at its untouched `0.5` — i.e. it asserted the very bug this plan fixes. It now seeds a real near-equal `MoveGrade`, so the bot accepts because it genuinely evaluated the position. Strictly stronger than what it replaced
- **The clock test asserts `whiteClockMs > baseMs`.** With a debit of at most ~1.6s and a 2s increment, the bot's clock can only end up *above* where it started if `commitMove` applied the increment — a bypass path fails it. (An exact `toBe(base - debit + inc)` was tried first and is wrong: under fake timers the async continuation resolves at the end of the advance window, so the debit tracks the advance, not the reveal delay.)
- **Ply-cap ownership is explicit.** `BOOK_PLY_CAP` itself is tested at the pure layer (`openingBook.test.ts`); the hook's obligation is to hand the book the REAL move history, which is pinned by its own test rather than by scripting 16 legal plies
