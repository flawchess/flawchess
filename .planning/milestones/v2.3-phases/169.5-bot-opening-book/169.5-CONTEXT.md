# Phase 169.5: Bot Opening Book - Context

**Gathered:** 2026-07-13
**Status:** Ready for planning
**Source:** `/gsd-explore` session 2026-07-13 (the exploration served as the discuss round — every decision below was made and confirmed by the user in that session)

<domain>
## Phase Boundary

**The problem.** Today the bot searches every ply from the starting position. `useBotGame` starts from `new Chess()` and every bot turn runs the full Phase 168.5 MCTS budget (`FLAWCHESS_BOT_MAX_NODES = 50`, `FLAWCHESS_BOT_CONCURRENCY = 4`, Stockfish leaf evals) — measured at ~5.4s median, ~12.7s worst case. The opening is where that search is *least* useful and *most* expensive: the tree is widest and the eval is flattest, so the two-sided early-stop rule (clear-winner OR near-tie-flatness) is least likely to fire and the search usually runs to the full node budget. On top of that, move 1 additionally pays a cold start — the `WorkerPool` and `MaiaQueue` are created lazily, Stockfish WASM workers spawn on first use, and the Maia ONNX session loads its weights — so the first move is the worst move of the game for both latency and the bot's clock.

**This phase.** While in the opening, the bot answers from a book instead of searching: candidates are the legal moves that keep the game a prefix of some line in the already-shipped ECO database, weighted by Maia's policy at the bot's ELO. The result is near-instant, spends no search budget, costs the bot almost none of its clock, and varies across games. The book window is also used to warm the engine so the first *searched* move is not cold.

**In scope**: the book module (ECO index + candidate filter + policy-weighted selection + exit rule), its wiring into the bot's move path, engine prewarm during the book window, tests.

**Out of scope**: the troll/persona book (SEED-098's Trickster) — but the design must leave the hook. Any UI. Any change to `mctsSearch`, `botBudget.ts` constants, the clock, or the ELO calibration. Any backend endpoint.
</domain>

<decisions>
## Implementation Decisions

### Book source

- **D-01**: The book is the already-shipped `frontend/public/openings.tsv` — the 3,641-line ECO database (`eco` / `name` / `pgn` columns, PGN as a SAN line like `1. e4 c6 2. d4 d5`), today read only by `frontend/src/lib/openings.ts` to *name* openings. No new asset, no backend endpoint, no network dependency beyond the same-origin static file the app already fetches. Bot play stays fully client-side.
  - Rejected: a `game_positions`-backed DB book (real popularity + WDL, but adds a server round-trip and a new aggregate to an all-client bot).
  - Rejected: a hand-curated static book, incl. reusing the harness's 16-opening `OPENING_BOOK` (narrow, hand-maintained, the human deviates out of it immediately).

### Move weighting

- **D-02**: Among in-book candidate moves, the bot samples weighted by **Maia's policy at the bot's configured ELO**. `openings.tsv` carries no frequencies and no WDL, and the overwhelming majority of its 3,641 lines are obscure (Amar Opening, Sodium Attack, Creepy Crawly, Drunken Knight). Uniform sampling over book continuations would make the bot play garbage openings the large majority of the time. Maia supplies the human-frequency prior the file lacks; the ECO file supplies the theory guardrail and the opening's *name*. Cost: one Maia NN eval (~100ms), zero Stockfish.
- **D-05**: Reuse `botSampling.ts`'s existing raw-policy sampling (the machinery behind `selectBotMove`'s `blend <= 0` regime — `samplePolicy`), renormalized over the in-book candidate subset. **Do NOT use `applyPolicyTemperature` from `policyTemperature.ts`** — Phase 166 **D-02 (locked)** keeps that transform out of the bot path entirely: its "flatten = more engine-like" polarity is an artifact of the analysis *ranking* pipeline, where Stockfish grading rescues the surfaced move. In a raw-sampling path there is no rescue, so it would make the bot noisier and weaker and would break the Phase 168 calibration curve. Diversity comes from *sampling* the policy, not from re-tempering it.

### Exit rule

- **D-03**: The bot leaves book when **no in-book continuation clears a Maia policy floor**, OR **a ply cap is reached** — whichever comes first. Both thresholds are named constants in the book module (no magic numbers), tuned by feel.
  - The policy floor is self-tuning: it fires exactly when no theory move looks human-plausible at that ELO, which is the honest definition of "out of book".
  - The ply cap exists because ECO's deep mainlines (e.g. a 10-ply Najdorf) would otherwise let the bot replay theory for 20 plies without ever engaging the engine.
  - Leaving book is a **one-way transition within a game** — the bot does not re-enter book after searching once.
  - Note for the planner: ECO's 3,641 lines cover nearly every sane early position (even 1.h4 is a named opening), so the membership test filters very little in the first few plies. The book's real jobs are (a) a principled stop signal, (b) making the opening *nameable* so a persona can bias it later, and (c) keeping the bot inside theory. Maia at the bot's ELO already *is* a human opening book; ECO is the guardrail and the vocabulary, not the source of move quality.

### Calibration

- **D-04**: **This phase does NOT perturb the ELO calibration, and is not a strength change requiring re-calibration.** Read alongside the D-19 block in `frontend/src/lib/engine/botBudget.ts` before worrying about this. `scripts/calibration-harness.mjs` already starts *every* anchor game from a mid-opening FEN drawn from `scripts/lib/calibration-openings.mjs`'s `OPENING_BOOK` (harness line ~1105: `const opening = OPENING_BOOK[idx % OPENING_BOOK.length]` → `startFen: opening.fen`), so the calibrated ELO was **never measured on plies 1–8 in the first place**. A play-time book moves the app *toward* the harness's regime, not away from it. Do not add a re-calibration task to this phase.

### Persona hook

- **D-06**: The troll/persona book is **out of scope** for this phase, but the design must leave the hook open. `frontend/src/data/trollOpenings.ts` (curated Bongcloud-tier positions) and `frontend/src/lib/trollOpenings.ts` (`deriveUserSideKey`) already exist, and SEED-098's Trickster persona plans to bias the book toward them. Structure book selection so a future persona can **re-weight the candidate set** without a rewrite — do not hard-code the Maia-policy weighting as the only possible weighting function. Do not build the persona layer here.

### Claude's Discretion

- The exact policy-floor value and ply-cap value (named constants, tuned by feel).
- How the ECO TSV is indexed for prefix lookup (trie, prefix Map, keyed by SAN sequence — `openings.ts` already builds a `Map<sanSequence, Opening>`; reuse or extend rather than re-parsing the file a second time).
- Where the book decision is wired: inside `selectBotMove` vs. ahead of it in `useBotGame`'s bot turn. **Constraint**: `selectBotMove` is imported *unchanged* by both the app and `scripts/calibration-harness.mjs` (Phase 166 contract). The harness must NOT gain a book (it already starts from book FENs; giving it a second book would corrupt the anchor games). Whichever wiring is chosen must preserve that.
- Whether prewarm is a new exported helper or an effect in the existing hook.
- Test strategy for "openings vary across games" (a seeded-PRNG distribution assertion is preferable to a flaky randomness test).
</decisions>

<specifics>
## Specific Ideas

- Measured today (Phase 168.5-04): median ~5.4s per move, worst case ~12.7s, at the locked 50-node budget. Target for a book move: within the existing `REVEAL_DELAY_MIN_MS` (500ms) / `REVEAL_DELAY_MAX_MS` (1500ms) reveal band — i.e. the synthetic human-pacing delay becomes the binding constraint, not the engine.
- The `openings.tsv` SAN-sequence Map that `openings.ts` already builds is keyed by the *full* line ("e4 c6 d4 d5"). A prefix lookup needs the set of lines *extending* the current sequence — deriving that from the same parsed data (a prefix index built once, lazily, alongside the existing name Map) avoids a second fetch and a second parse.
- Diversity check worth writing: from the standard start position, repeated seeded runs should produce a spread of first moves roughly tracking Maia's policy at that ELO (mostly 1.e4/1.d4, some 1.c4/1.Nf3, rarely junk) — not a single deterministic move, and not a uniform spray over all 20 legal moves.
</specifics>

<deferred>
## Deferred Ideas

- **Troll/persona book weighting** (SEED-098 Trickster) — hook only, per D-06.
- **A FlawChess DB book** built from `game_positions` (real popularity + WDL per rating band, from our own imported games). Genuinely on-brand and strictly better weights, but it makes bot play depend on the server. Revisit only if the Maia-policy weighting proves inadequate in play.
- **Book for the human side / opening trainer** — out of scope; this phase is about what the *bot* plays.
