<objective>
Build the store path: a PGN-only `normalize_flawchess_game`, the
`store_bot_game` service (rating derivation + `_flush_batch` reuse + side-table
insert + idempotency + single-transaction commit), and the thin `bots` router.

Purpose: This is the phase's payload — POST /bots/games persists a finished bot
game as a first-class `platform='flawchess'` Library game, analyzable exactly like
an imported game, idempotent on the client UUID, with a server-computed rating.
Delivers STORE-01/02/03/05/06.

Output: `normalize_flawchess_game`, a `game_repository` id-lookup helper,
`store_bot_game_service.store_bot_game`, `app/routers/bots.py`, and its registration
in `app/main.py`.

Assumption recorded (RESEARCH Open Question 1 / A1): D-14's request schema has no
termination-reason field, and resignation/timeout are not derivable from board state.
This plan derives checkmate/stalemate/draw from the final board via python-chess,
reads a PGN `[Termination "..."]` header when present (closed vocab mapped 1:1 to the
Termination Literal), and defaults everything else to "unknown". This is a known gap
flagged to Phase 169 (the PGN emitter), NOT a blocker, and does NOT add a request field.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (POST /bots/games) | Untrusted client-supplied PGN + settings persisted for the authenticated user. |
| JWT → user_id | user_id is derived from the token, never the request body. |
| guest ↔ registered | Guests use the same authed path; must not reach another user's data. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-167-07 | Denial of Service | PGN parse in normalize_flawchess_game | high | mitigate | chess.pgn.read_game wrapped to return None (never raise past the service); router maps None → 422, never 500; pgn max_length bound applied at the schema (Plan 01). |
| T-167-08 | Tampering | player rating / platform / is_computer_game | high | mitigate | Server derives rating from user_rating_anchors and sets platform/is_computer_game/username in the normalizer; the request schema has no such field (D-05/D-14). |
| T-167-09 | Tampering / Info Disclosure | game_uuid idempotency scope | medium | mitigate | Idempotency keyed on (user_id, platform, platform_game_id) via uq_games_user_platform_game_id — two users may reuse the same UUID with no collision; a re-submit never overwrites another user's game (D-11). |
| T-167-10 | Elevation of Privilege | guest auth path | high | mitigate | user_id = current_active_user.id from the JWT, never the body; guests are ordinary User rows on the same dependency — a guest cannot store into another account (D-13/ASVS V4). |
| T-167-11 | Repudiation / data loss | partial write across game + settings | medium | mitigate | Single transaction: _flush_batch + settings insert + commit are one unit (D-10); a failure rolls back both. |
</threat_model>
