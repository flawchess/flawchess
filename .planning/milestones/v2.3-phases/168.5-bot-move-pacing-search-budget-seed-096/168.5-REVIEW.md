## Warnings

### WR-01: `--resume` with a non-`.tsv` path silently overwrites the main results TSV with the summary (data loss)

**File:** `scripts/calibration-harness.mjs:1257`
**Issue:** `const summaryPath = mainTsvPath.replace(/\.tsv$/, '-summary.tsv')`. On `--resume`, `mainTsvPath` is the user-supplied path verbatim. If it does not end in `.tsv` (e.g. the user resumed from `cp run.tsv run.bak`), `replace` is a no-op, `summaryPath === mainTsvPath`, and `emitEloSummary`'s `fs.writeFileSync` **truncates the primary results matrix** — destroying the entire (potentially multi-hour, just-appended) sweep artifact. `loadPriorSweep` validates content, not the filename, so nothing upstream blocks this.
**Fix:**
```js
// in parseArgs, case 'resume':
const resumePath = path.resolve(requireFlagValue(value, key));
if (!resumePath.endsWith('.tsv')) {
  throw new Error(`--resume: expected a .tsv file, got ${resumePath} — the summary sibling path is derived by extension`);
}
args.resume = resumePath;
```
(or assert `summaryPath !== mainTsvPath` before `emitEloSummary`).

### WR-02: `setupHarnessEngines` leaks the spawned Stockfish pool (and hangs the harness) when a later bring-up step throws

**File:** `scripts/calibration-harness.mjs:314-320`
**Issue:** Inside `setupHarnessEngines`, if `resolveFrontendModule('chess.js')` or `makeNodeProviders` throws AFTER `createStockfishPool` succeeded, the N live Stockfish child processes are never terminated — the function throws before returning, so `main()`'s CR-01 `pool?.quitAll()` guard sees `pool === undefined`. Child stdio handles keep the Node event loop alive, so the harness (and the determinism/pruning checks, which call `setupHarnessEngines` first) **hangs with orphaned engine processes** instead of exiting. This is exactly the failure class the file's own CR-01 (`main()`) and CR-02 (`createStockfishPool`) comments fixed at the other two bring-up seams — this middle seam was missed.
**Fix:**
```js
export async function setupHarnessEngines({ stockfishProcs = STOCKFISH_POOL_DEFAULT_SIZE } = {}) {
  const maiaCtx = await createMaiaSession();
  const pool = await createStockfishPool({ size: stockfishProcs });
  try {
    const { Chess } = await resolveFrontendModule('chess.js');
    const providers = makeNodeProviders(maiaCtx.session, maiaCtx.ort, pool.grade);
    return { providers, pool, Chess, maiaCtx };
  } catch (err) {
    pool.quitAll();
    throw err;
  }
}
```

### WR-03: stop-rule block nests 6 levels deep — breaches the project hard limit of 4

**File:** `frontend/src/lib/engine/mctsSearch.ts:489-504`
**Issue:** Inside `mctsSearch`'s function body: `while` (1) → `for (const result of results)` (2) → `if (budget.stopRule)` (3) → `if (extremes)` (4) → `if (argmax === stable) / if (minNodes && window)` (5) → `if (clearWinner || nearTieFlatness)` (6). CLAUDE.md sets nesting depth "soft 3, hard 4 inside any function body" as the firm rule. The stop-rule check is also a self-contained concern (rolling stability state + two-sided threshold test) with a natural seam.
**Fix:** Extract a helper holding the mutable stability state, e.g.:
```ts
interface StopRuleState { stableArgmaxUci: string | null; stableCheckCount: number; }
function stopRuleSatisfied(root: EngineNode, rule: BotStopRule, nodesEvaluated: number, state: StopRuleState): boolean {
  const extremes = rootChildValueExtremes(root);
  if (!extremes) return false;
  state.stableCheckCount = extremes.argmaxUci === state.stableArgmaxUci ? state.stableCheckCount + 1 : 1;
  state.stableArgmaxUci = extremes.argmaxUci;
  if (nodesEvaluated < rule.minNodes || state.stableCheckCount < rule.stabilityWindow) return false;
  return (
    extremes.topValue - extremes.runnerUpValue >= rule.marginThreshold ||
    extremes.maxValue - extremes.minValue <= rule.epsilonThreshold
  );
}
// call site: if (budget.stopRule && stopRuleSatisfied(root, budget.stopRule, nodesEvaluated, stopState)) earlyStop = true;
```
Behavior-identical (pure over the same values, no wall-clock), and the loop body drops back within limits.

### WR-04: no automated drift guard between the hook's `FLAWCHESS_BOT_*` constants and the harness's D-11 mirror block

**Files:** `frontend/src/hooks/useFlawChessEngine.ts:71-85`, `scripts/calibration-harness.mjs:113-128`
**Issue:** The bot budget (50/8/4) and the four stop-rule fields are duplicated by hand in two files. The phase's own trace (T-168.5-04-01, quoted in the harness comment) names divergence as the failure mode: "a divergence from the hook's own values calibrates a bot that does not ship." Yet nothing enforces equality: `useFlawChessEngine.test.tsx` asserts only structure (`< 400`, `=== 8`, integer, well-formed), never the exact values the harness mirrors, and `loadPriorSweep`'s budget guard compares the harness against *itself*. A one-sided retune of either file passes every gate and silently mis-calibrates.
**Fix:** The harness already imports frontend modules via the `@/` alias hook (`selectBotMove`, `botSampling`, `maiaEncoding`). Move the four `FLAWCHESS_BOT_*` constants into a dependency-light module (e.g. `frontend/src/lib/engine/botBudget.ts` — no React/Sentry/Worker imports), re-export from the hook for app callers, and import it directly in the harness, deleting the mirror block. Failing that, minimally pin exact values in `useFlawChessEngine.test.tsx` (`expect(FLAWCHESS_BOT_MAX_NODES).toBe(50)` etc.) with a comment binding them to the harness mirror.

### WR-05: pruning check's coverage assertion (part a) is tautological

**File:** `scripts/lib/calibration-pruning.check.mjs:98-116`
**Issue:** `gridKeys` is constructed FROM `fixtureRows` (line 98), then each row is asserted to be a member of `gridKeys` (line 103) — this can never fail regardless of any code change, and it never exercises the real sweep loop's "every anchor gets a row" invariant in `main()`/`processCellAnchor`. The PASS line ("skip_reason coverage — 4/4 gridKeys present") reports a guarantee the check does not actually establish, giving false confidence in D-15's "no silent coverage gaps" requirement. (Part (b), the `loadPriorSweep` round-trip, is a genuine check and is fine.)
**Fix:** Build `gridKeys` independently from axis constants (`for (const anchor of FIXTURE_ANCHORS) gridKeys.add(cellKey(FIXTURE_BOT_ELO, FIXTURE_BOT_BLEND, anchor))`) and assert `fixtureRows` covers it exactly (both directions: every key has a row, every row's key is in the grid), or drop the coverage claim from the PASS message and keep only the games=0↔skip_reason consistency + round-trip assertions.
