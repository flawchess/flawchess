## Accomplishments

- Implemented both D-15 pruning mechanisms in `calibration-harness.mjs`'s grid loop: static bracketing skips any anchor rated more than `ANCHOR_ELO_WINDOW` (400 Elo) from a bot-cell's nominal rating; dynamic cutoff walks the in-window anchors outward from the bot-cell's own rating (split into a weaker side and a stronger-or-equal side) and prunes further-out anchors on whichever side hits a near-0 or near-100% result (`DYNAMIC_CUTOFF_SCORE_EPS=0.05`).
- Every pruned anchor is a REAL `games=0` TSV row carrying a populated `skip_reason` (`out_of_window` / `lost_cutoff` / `won_cutoff`) — `gridKeys` is never shrunk at runtime, preserving `--resume`'s grid-membership invariant (Pitfall 4).
- Fixed a real bug this restructuring would otherwise have introduced: `loadPriorSweep`'s games-count guard previously required every prior row's `games` to equal `--games-per-cell` exactly, which would throw on ANY `--resume` of a pruned sweep (a skip row is `games=0` by design). Split the check into a played-row branch (must equal `--games-per-cell`) and a skip-row branch (must equal exactly `0`).
- Fixed the `gameIndex` fast-forward on a resumed cell to read that cell's own recorded `games` count (0 for a pruned row, `--games-per-cell` for a played one) instead of blindly assuming `--games-per-cell` — a pruned resume cell now correctly consumes zero gameIndex slices on both fresh and resumed runs (D-09 byte-identity).
- Extracted `anchorRatingFor` from `calibration-harness.mjs` into `calibration-anchors.mjs` (additive, no behavior change) so both the pre-existing D-05 advisory Elo summary and the new D-15 ordering helper share one rating lookup built on `SF_SKILL_ELO`/`MAIA_ELO_LADDER`.
- Added `scripts/lib/calibration-pruning.check.mjs`: a standalone, engine-free structural check (synthesized TSV fixture covering all three `skip_reason` markers) proving (a) every `gridKey` has a row with pruned rows carrying a non-empty `skip_reason`, and (b) the fixture round-trips through the exported `loadPriorSweep` without throwing.
- SC5: rebudgeted `calibration-determinism.check.mjs` to construct `playGame`'s budget at the SHIPPED bot budget — removed the old `DETERMINISM_MAX_NODES=20`/`DETERMINISM_MAX_PLIES=8` override (a Plan 02 artifact from the 400-node era); `playGame` now defaults to `FLAWCHESS_BOT_MAX_NODES`/`_MAX_PLIES` and has always unconditionally carried `stopRule: FLAWCHESS_BOT_STOP_RULE` + the pinned `FLAWCHESS_BOT_CONCURRENCY` since 168.5-04. Ran the check against real engines: **PASS — same seed reproduces a byte-identical 29-ply blend=1 game** (result=win, reason=checkmate), ~3 minutes wall-clock for both runs combined.
- Ran a bounded D-17 pilot through the REAL harness pipeline (not a hand-built probe): two short cells (`--elo 1500 --blends 1 --anchors maia1500 --games-per-cell 1`, n=15 bot moves; `--elo 1500 --blends 0.5 --anchors maia1500 --games-per-cell 1`, n=28 bot moves), combined n=43 blend>0 bot moves — **median 3.32s (inside the 3-8s target band), p95 16.72s (under the 20s ceiling)**. Confirms the shipped budget's pacing before handing the full sweep to the operator.
- Documented the operator runbook for the full D-16 ~24h re-calibration sweep below — NOT launched by this plan.

## Decisions Made

- **Dynamic-cutoff traversal ambiguity, resolved as bracket-and-expand.** D-15's stated rule ("iterate anchors in ascending strength order; after a near-0 result skip stronger remaining anchors; after a near-100% result skip weaker anchors") is under-specified for a single whole-window ascending pass: in strict ascending (weakest→strongest) order, "remaining weaker anchors" to skip after a win-sweep is a vacuous set (weaker anchors, by construction, are always already played by the time a stronger one triggers a sweep). The only traversal where BOTH halves of the stated rule have genuine, non-vacuous effect is one that splits the in-window anchors at the bot-cell's own nominal rating and walks each side OUTWARD (closest-to-bot anchor first): the weaker side descending, the stronger side ascending. This is what got implemented (`orderAnchorsForDynamicCutoff`). It matches the D-14/D-15 rationale ("Maybe we can stop testing a FlawChess bot vs even stronger anchors when it lost all games") and additionally makes the symmetric "stop testing weaker anchors once thoroughly dominant" case meaningful. Flagging this explicitly since it's a genuine interpretation call within Claude's Discretion (CONTEXT.md), not a literal transcription of the plan's prose — worth a second look if the eventual operator sweep's skip pattern looks surprising.
- **`ANCHOR_ELO_WINDOW=400`.** Chosen so a bot-cell's window covers roughly ±4 Maia rungs (100-Elo steps) either side, wide enough to bracket at least one anchor above and below the bot's own rating for every D-14 default bot ELO (1100/1500/1900) while still excluding the most extreme cross-family anchors (e.g. a 1100-rated bot never plays the sf5≈2200 anchor).
- **`DYNAMIC_CUTOFF_SCORE_EPS=0.05`.** At the D-14 default `~10 games/cell`, a score of exactly `0` (all losses) or `1` (all wins) triggers the cutoff; a single stray draw (`0.5/10 = 0.05`) sits exactly at the boundary and does NOT prevent a cutoff from firing on an otherwise-total sweep, which is the intended "already-decided" signal — a genuinely mixed result (even one win or loss among ten) will not trigger it.
- **`loadPriorSweep`'s games-count guard split by row kind, not a value substitution.** Rejected the simpler "treat any `games=0` as automatically valid" rule in favor of an explicit `skip_reason !== ''` check plus a NEW assertion (`games` must be exactly `0` for a skip row) — this also catches a corrupted TSV where a skip row's games count was tampered with or a real bug wrote `games=0` without a reason.
- **`calibration-pruning.check.mjs` is fully synthesized (no real engines), per the plan's own instruction** — the round-trip property under test (`loadPriorSweep`'s guards) is pure data-shape logic, independent of what actually produced the TSV rows; using fabricated tallies keeps the check fast (milliseconds) and deterministic, appropriate for a structural regression check rather than an integration proof.

## Operator Runbook: Full D-16 Re-Calibration Sweep

**This is an OPERATOR-launched, out-of-band step (D-16). No executor task blocks on it — do not run this from within a GSD plan/phase execution.**

### Command

```bash
node --import ./scripts/lib/frontend-alias-hook.mjs scripts/calibration-harness.mjs \
  --elo 1100,1500,1900 \
  --blends 0,0.5,1 \
  --anchors maia700,maia900,maia1100,maia1300,maia1500,maia1700,maia1900,maia2100,maia2300 \
  --games-per-cell 10 \
  --seed 1 \
  --stockfish-procs 4 \
  --out-dir reports/data
```

- **Grid (D-14):** ELO `{1100, 1500, 1900}` × blend `{0, 0.5, 1}` × the trimmed Maia-rung anchor set above (bracketing every cell within the shipped `ANCHOR_ELO_WINDOW=400`; Stockfish-skill anchors dropped per D-14 — the raw-Maia rungs alone bracket all three bot ELOs). `blend=0` cells are cheap (no search, Maia-argmax-only) and anchor the human-end scale.
- **`--games-per-cell 10`** per D-14's "~10 games/cell" target.
- D-15 pruning is automatic — no flag needed. Every anchor outside a bot-cell's `±400` window is skipped instantly (static bracketing); within-window anchors are cut off dynamically once a cell's outcome trends decisively one way.
- **`--stockfish-procs 4`** matches this plan's measurement environment; raise it if the reference box has more idle cores (bounded by `nproc`).

### Resumability (D-16)

If the sweep is killed (machine restart, manual interrupt, crash), **re-run the EXACT SAME command** with `--resume <the-emitted-tsv-path>` appended:

```bash
node --import ./scripts/lib/frontend-alias-hook.mjs scripts/calibration-harness.mjs \
  --elo 1100,1500,1900 --blends 0,0.5,1 \
  --anchors maia700,maia900,maia1100,maia1300,maia1500,maia1700,maia1900,maia2100,maia2300 \
  --games-per-cell 10 --seed 1 --stockfish-procs 4 --out-dir reports/data \
  --resume reports/data/calibration-harness-<TIMESTAMP>.tsv
```

`--resume` skips every already-completed `(bot_elo, bot_blend, anchor)` cell (played OR D-15-pruned — both are "complete" rows) and appends the remainder to the SAME file, so the finished map is byte-identical to an uninterrupted run. It refuses (throws) on any mismatch of `--games-per-cell`, `--seed`, the bot budget, or the grid axes — if the command above is copied verbatim, none of those can drift.

### Retry loop

For a genuinely unattended overnight/weekend run, wrap the command in a simple restart-on-failure loop (the harness's own `ENGINE_RETRY_ATTEMPTS` already retries individual timed-out engine calls in-place — this outer loop only covers a harder crash, e.g. an OOM-killed Stockfish process the in-process retry couldn't recover):

```bash
TSV=""
until node --import ./scripts/lib/frontend-alias-hook.mjs scripts/calibration-harness.mjs \
  --elo 1100,1500,1900 --blends 0,0.5,1 \
  --anchors maia700,maia900,maia1100,maia1300,maia1500,maia1700,maia1900,maia2100,maia2300 \
  --games-per-cell 10 --seed 1 --stockfish-procs 4 --out-dir reports/data \
  ${TSV:+--resume "$TSV"}; do
  TSV=$(ls -t reports/data/calibration-harness-*.tsv | head -1)
  echo "sweep exited non-zero, resuming from $TSV in 30s..."
  sleep 30
done
```

### Wall-clock envelope (D-16, ~24h)

3 ELOs × 3 blends × up to 9 anchors × 10 games/cell = up to 810 games BEFORE D-15 pruning; D-15's static window alone drops roughly half of the raw cross-product (each bot-cell only plays anchors within ±400 Elo — for the 1100/1500/1900 grid against the 700-2300 anchor ladder, each cell brackets 4-5 of the 9 anchors), and the dynamic cutoff further truncates most cells (per this plan's pilot, and Plan 04/02's measurements, blend=1 moves cost ~3-9s median at the shipped budget; blend=0 cells are near-free). This is expected to comfortably fit the ~24h envelope on the 16-core reference box at `--stockfish-procs 4`; if a live run's own throughput report (printed at the end) projects over-budget, either raise `--stockfish-procs` (bounded by available cores) or trim `--games-per-cell`.
