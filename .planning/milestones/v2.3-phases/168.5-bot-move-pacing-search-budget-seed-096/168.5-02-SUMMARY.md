# Phase 168.5 Plan 02: Deterministic Stockfish Grading + Hardened Watchdog Summary

**Depth-only `go` + `Clear Hash` per call in nodeGrade/evalPositionCp (D-10), decoupled watchdog constants + retry-in-place in the Stockfish pool (D-11), and a measured Clear-Hash throughput cost (~0.33s/expansion, no meaningful slowdown vs the pre-fix baseline).**

## Accomplishments

- `nodeGrade` and `evalPositionCp` now send `setoption name Clear Hash` in their per-call reset block and issue depth-only `go` commands (no `movetime` token), so a grade/adjudication is a pure function of `(position, depth, clean hash)` — load-independent (D-10).
- Watchdog timeouts (`GRADING_WATCHDOG_TIMEOUT_MS=60000`, `ADJUDICATION_WATCHDOG_TIMEOUT_MS=20000`) are new, independently-sized constants, replacing the old movetime-derived `X + SLACK_MS` formula that no longer made sense once `movetime` was removed (D-11).
- Added `ADJUDICATION_TARGET_DEPTH=10`, shallower than grading's depth-14, since adjudication runs after every ply of every game and its Clear-Hash cost compounds fastest.
- `stockfish-pool.mjs`'s `withEngine` now retries a timed-out operation in place via `stopAndSync()` up to `ENGINE_RETRY_ATTEMPTS=2` times (3 total attempts) before propagating failure — gated strictly to the `waitFor` timeout error class so a genuinely deterministic error (illegal move, parse failure) is never retried.
- Measured the Clear-Hash throughput cost on bounded, real-engine runs (see below) and confirmed both watchdog constants sit comfortably above the worst observed latency.

## Measurement (Task 3)

**Method:** with Task 1+2 landed, ran three bounded, real-engine (Maia ONNX + Stockfish WASM-over-UCI) instrumented probes — no full grid sweep (that is Plan 05's operator sweep, D-16). All scripts were ad-hoc, scratchpad-only (not committed):

1. **Harness cell run** (`node scripts/calibration-harness.mjs --elo 1100 --blends 1 --anchors maia1100 --games-per-cell 1 --stockfish-procs 4`, killed after 4 bot moves logged): bot-move times 119.27s, 108.97s, 98.19s, 85.04s (mean ≈ 102.9s) at the current 400-node/8-ply budget.
2. **Single instrumented bot move** (`selectBotMove` called directly, blend=1, 400-node budget, concurrency=4, wrapping `pool.grade` to record per-call latency): total move time 131.1s; **400 `grade()` calls: mean=514ms, p95=1228ms, max=1834ms**; **per-expansion cost (total/n) = 0.328s**.
3. **Adjudication sweep** (`evalPositionCp` called once per FEN across all 33 `OPENING_BOOK` entries, single Stockfish process, depth-10 + Clear Hash): **n=33, mean=18ms, max=57ms**.

**Comparison to SEED-096's pre-fix baseline** (~0.28s/expansion, min 96s/median 109s/mean 111s/max 146s over 13 blend=1 bot moves at 400 nodes, unfixed movetime-capped grading):

- Per-expansion cost with the D-10 fix: **0.328s vs 0.28s baseline (~1.17x)** — within noise/contention variance, not a meaningful regression. All measurements ran while a separate, pre-existing full-grid production sweep (an unrelated user-started background process, `reports/data/calibration-harness-2026-07-12T10-24-36-664Z.tsv`) was also actively consuming 4 Stockfish procs on the same 16-core box, so this number likely reflects MORE CPU contention than the original baseline measurement, not less.
- Full bot-move times (85.04s–131.1s across 5 samples) fall within or near the pre-fix baseline's own observed range (96s–146s, n=13) — **no order-of-magnitude slowdown from adding `Clear Hash` to every grading call.**

**`ADJUDICATION_TARGET_DEPTH=10` justified:** mean 18ms / max 57ms per adjudication call is negligible relative to the ~100s+ bot-move grading cost, even though adjudication runs after every ply of every game (far more often than bot-move grading). No need to lower to 8 — 10 is comfortably cheap. Kept as-set from Task 1's starting value; no further code change required.

**Watchdog headroom confirmed:**
- `GRADING_WATCHDOG_TIMEOUT_MS = 60,000ms` vs worst observed single `grade()` call latency **1,834ms** → **~33x margin**.
- `ADJUDICATION_WATCHDOG_TIMEOUT_MS = 20,000ms` vs worst observed single `evalPositionCp()` call latency **57ms** → **~350x margin**.

Both constants sit well above the worst observed real-engine latency, satisfying D-11's "generous, independently-sized" requirement; no adjustment needed.

## Decisions Made

- `ADJUDICATION_TARGET_DEPTH` finalized at 10 (Task 1's starting value), confirmed cheap by Task 3's measurement — no lowering to 8 needed.
- `GRADING_WATCHDOG_TIMEOUT_MS=60000` / `ADJUDICATION_WATCHDOG_TIMEOUT_MS=20000` confirmed generous (33x/350x margin over worst observed latency) — no adjustment needed.
- `ENGINE_RETRY_ATTEMPTS=2` (3 total attempts) chosen per the plan's suggested starting value; retry gated strictly to the `waitFor` timeout error message pattern, never a bare catch-all, so illegal-move/parse errors propagate immediately.
- `GRADING_MOVETIME_SAFETY_CAP_MS`/`SLACK_MS`/`ADJUDICATION_MOVETIME_MS`/`ADJUDICATION_SLACK_MS` fully deleted (not just removed from the `go`/watchdog path) after confirming via repo-wide grep that no other file imports or references them.

## Issues Encountered

- The plan's Task 3 action suggested running one full `--games-per-cell 1` game via the CLI harness. A full blend=1 game (~40+ bot moves at ~100-130s each) would have taken well over an hour to complete, and a separate real production sweep was already running concurrently on the same machine, competing for CPU. Instead of running (and waiting on) a full game, this plan ran a partial harness cell (killed after 4 bot moves, enough for a representative sample) plus two small, targeted, instrumented microbenchmarks (a single `selectBotMove` call wrapping `pool.grade` for per-call latency stats, and a `evalPositionCp` sweep over all 33 opening-book positions for adjudication latency) — both exercise the identical fixed production code paths (`nodeGrade`/`evalPositionCp` via the real pool), just with richer per-call instrumentation and a bounded time cost. This satisfies the plan's intent (a bounded, non-full-grid, real-engine measurement) while keeping the measurement itself tractable.
- A pre-existing, unrelated full-grid production sweep (`reports/data/calibration-harness-2026-07-12T10-24-36-664Z.tsv`, started independently before this plan's execution) was running throughout — left untouched (not started or stopped by this plan); its presence is noted as a contention caveat on the measured numbers above, not treated as an error.
