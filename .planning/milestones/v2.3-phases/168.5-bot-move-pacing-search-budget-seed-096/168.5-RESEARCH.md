## User Constraints (from CONTEXT.md)

### Locked Decisions

**Area A — Clock/fixed-strength fork (SC1)**
- **D-01: Fixed strength + synthetic bot clock.** One global search budget → one calibrated ELO per config (preserves everything SEED-096 locked). The bot's search NEVER reads or bends to the clock. The bot's displayed clock is debited a scripted amount per move, not real compute time.
- **D-02: Synthetic debit = fraction-of-remaining schedule** (e.g. `remaining/N + share-of-increment`; exact form is Phase 169's to implement, the model is locked here). By construction the bot cannot lose on time — flag-on-time is a user-side-only end condition.
- **D-03: Small TC-independent randomized minimum reveal delay** (~0.5–1.5s ballpark) floors near-instant moves (blend=0 computes in ~0.09s and feels robotic). Display-only, implemented in Phase 169's game loop; strength untouched. SEED-096's TC-scaled pacing envelope stays deferred.
- **D-04: Phase 169 SC1 is amended during this phase's execution** (part of 168.5 SC1) to: "the bot paces its replies (not instant): search runs at the fixed shipped budget regardless of clock, a small randomized reveal delay floors fast moves, and the bot's clock is debited a fraction-of-remaining synthetic think time (the bot never flags)."
- **D-04b: Pacing theater is UX-only — the harness has NONE of it.** The synthetic clock debit (D-02) and reveal delay (D-03) live exclusively in Phase 169's game loop. The harness plays moves back-to-back at raw compute speed: no artificial waits, no delay floors, no clock simulation (consistent with SEED-096 "no per-game time pressure in calibration"). Sweep wall-clock and the D-17 moveMs stats therefore measure pure search time.

**Area B — Early-stop rule + budget**
- **D-05: Two-sided stop rule — stop when the decision is SETTLED, either way:**
  - *Clear winner*: top candidate's practicalScore leads the runner-up by ≥ margin, stable for K applied expansions (obvious recaptures, only-moves).
  - *Near-tie flatness*: top candidates all within a small ε of each other and the ranking has stopped churning (opening routine, dead-equal positions) — more search won't change anything that matters; for `blend < 1` the softmax sampling picks among them anyway.
  Both sides share a min-nodes floor. Rationale: a margin-only rule gets openings exactly backwards (near-equal moves → no clear winner → full-budget thinks on routine moves). Only genuinely contested positions (moderate gaps, best move still flipping) run toward the full budget — user's explicit intent: "spend more time in complicated positions, less in opening routine."
- **D-06: The rule is grade-based and deterministic** — evaluated in canonical apply order (per SEED-096), no wall-clock input, no explicit Maia-entropy signal in v1 (grade-margin dynamics capture complexity implicitly). All thresholds (margin, ε, K, min-nodes floor) are named constants tuned via the harness.
- **D-07: Pacing target: ~5s median, ~15s worst-case** for `blend > 0` on the reference box. `MAX_NODES` sized so a full-budget contested move stays ~15s (~50 nodes at the measured ~0.28s/expansion — validate, don't trust).
- **D-08: Same stop rule at ALL blends.** The mid-blend softmax-sampling shift (early stop changes the score vector feeding the softmax) is accepted and re-measured by the SC4 sweep — no special-casing, no blend-scaled thresholds.
- **D-09: Pin bot-play search concurrency to one fixed constant** (bot games only; the analysis board keeps device-adaptive `computePoolSize`). The harness uses the same pinned value — app == harness holds exactly, effective strength is device-independent, and mctsSearch's per-concurrency-level determinism becomes full determinism for bot play.

**Area C — SEED-095 deterministic grades + watchdog**
- **D-10: Depth-only `go` + `Clear Hash` per `go` in the harness.** ALL harness searches (bot grading AND adjudication) become depth-bounded with NO movetime cutoff, plus `setoption name Clear Hash` before every `go`. Result depends only on (position, depth, clean hash) — load-independent; D-09 determinism becomes a hard guarantee instead of probabilistic. Measure the throughput cost (grading is the sweep bottleneck), don't assume it.
- **D-11: Watchdog = generous timeout + retry-in-place.** Raise the timeout well above worst observed; on breach: `stopAndSync` recovery, retry the same `go` N times, engine restart as last resort. The in-flight game survives; only after N failures does the cell abort. Plus an operational `--resume` retry loop for sweeps.
- **D-12: The app keeps its movetime safety cap** (workerPool grading UX guard on slow devices). The harness measures the canonical depth-14 uncapped strength; rare device-side grading-depth wobble is accepted (status quo Phase 168 already shipped).
- **D-13: The D-09 determinism check stays an on-demand script**, run at the new budget + stop rule as this phase's SC5 and documented as the pre-sweep sanity step. NOT added to CI (engine spawning too heavy per push).

**Area D — Re-calibration sweep (SC4)**
- **D-14: Full coarse grid with trimmed anchors:** ELO {1100, 1500, 1900} × blend {0, 0.5, 1} × trimmed anchor set (Maia rungs bracketing each cell; Stockfish-skill anchors droppable for this run) × ~10 games/cell. blend=0 cells are cheap (no search) and anchor the human-end scale. This is the FIRST real strength map — 168-03 never launched the grid run.
- **D-15: Anchor pruning, both mechanisms (user-requested pragmatism):**
  - *Static bracketing*: per bot cell, only schedule anchors within an ELO window of the cell's nominal rating.
  - *Dynamic cutoff*: run a cell's anchors in ascending strength order; after the bot scores ~0 vs an anchor, skip all stronger anchors for that cell; symmetrically, after a 100% sweep, skip weaker anchors.
  Every skipped matchup is logged in run metadata — no silent coverage gaps. Window size / cutoff ε are named constants for the planner.
- **D-16: Wall-clock envelope ~24h** (overnight/weekend) on the reference box, operator-launched with `--resume` + the D-11 retry loop — plans must NOT block an executor agent on the sweep. Size games-per-cell/anchors to fit the envelope at measured per-move cost.
- **D-17: SC4 concrete pass criterion:** `blend > 0` per-move median in **3–8s** AND p95 ≤ **20s** across all sweep moves on the reference box (Adrian's 16-core dev box). The band catches both "still too slow" and a degenerate over-eager stop rule.
- **D-18: Accept & record if top-end strength drops.** The map documents real strength at the shipped budget — that IS the deliverable. No in-phase retune loop; relabeling bots / strength recovery is future work (SEED-091 #3 deferral). Pacing wins over nominal top-end strength.

## Summary

This phase is a **retune + hardening** phase over already-shipped, already-read code — there is no new library, no new package, and no new architectural layer. Three existing subsystems change: (1) `frontend/src/lib/engine/mctsSearch.ts` gains an additive, opt-in early-stop check evaluated at the exact point `onSnapshot` already fires (after every applied backup, in canonical dispatch order — this determinism property is already structurally guaranteed by the existing code, the stop rule just needs to read it); (2) `frontend/src/hooks/useFlawChessEngine.ts` gains a **second, bot-specific** budget profile (retuned `MAX_NODES`, pinned concurrency, stop-rule knobs) that must NOT silently overwrite the existing analysis-board profile (400 nodes, adaptive concurrency, no time pressure) — this is the single biggest architectural risk in the phase and is flagged prominently below; (3) `scripts/lib/calibration-providers.mjs` / `scripts/lib/stockfish-pool.mjs` / `scripts/calibration-harness.mjs` get a determinism fix (depth-only `go` + `Clear Hash`) whose throughput cost is explicitly unmeasured — the harness currently loses its transposition table on every single call, which could be a large, not-yet-quantified slowdown that eats into the D-16 24h sweep envelope.

The measured baseline (SEED-096 "Measured 2026-07-12") is unambiguous: every `blend > 0` move pays the full 400-node budget today (~96–146s, median 109s) — there is no fast tail, "openings settle fast" is a property early-stopping must CREATE, not one the current engine already has. This means the two-sided stop rule (D-05) is not optional polish; without it a smaller `MAX_NODES` alone (D-07's ~50-node estimate) is the ONLY lever that currently helps ALL positions uniformly, while early stopping is what makes routine positions feel instant. Both levers ship together per D-05/D-07/D-08.

**Primary recommendation:** Introduce a dedicated, additive `SearchBudget.stopRule` (optional field, `undefined` = today's unchanged full-budget behavior) plus a new bot-play-only constant block in `useFlawChessEngine.ts` (e.g. `FLAWCHESS_BOT_MAX_NODES`, `FLAWCHESS_BOT_CONCURRENCY`, `FLAWCHESS_BOT_STOP_RULE`) sitting alongside — never replacing — the existing `FLAWCHESS_ENGINE_MAX_NODES`/`_MAX_PLIES` analysis-board defaults. Mirror only the bot constants into the harness's D-11 block. Fix SEED-095 (depth-only + Clear Hash) FIRST and measure its throughput cost BEFORE sizing the D-16 sweep, per the seed's own explicit sequencing.

### Pattern 1: Bot-play budget as an ADDITIVE profile, never an overwrite

**What:** `useFlawChessEngine.ts` currently has exactly ONE constant set (`FLAWCHESS_ENGINE_MAX_NODES = 400`, `FLAWCHESS_ENGINE_MAX_PLIES = 8`) and ONE concurrency call site (`concurrency: computePoolSize()`), used by the `/analysis` page's live engine card. This phase's mandate — per SEED-096 D-07 ("Rough sizing for a ~15s move target: ~50 nodes") — is scoped explicitly to **bot play**, not analysis. Retuning the SHARED constant in place would silently cut the analysis board's search quality from 400 nodes to ~50 nodes, an unrelated regression with no UX justification (analysis has no pacing pressure — it already anytime-emits and the user is not waiting on a clock).

D-09 already establishes this split for concurrency explicitly: *"Pin bot-play search concurrency to one fixed constant (bot games only; the analysis board keeps device-adaptive `computePoolSize`)."* The same split logic must extend to `MAX_NODES` and the new stop-rule knobs, even though CONTEXT.md's canonical_refs describe the retuning as happening "in `useFlawChessEngine.ts`" — that file is simply where the constant LIVES, not evidence that the SAME constant must serve both callers.

**When to use:** Any time this phase touches `useFlawChessEngine.ts`'s constants.

**Recommended shape** (planner should size the exact constant names/location, this is illustrative):
```typescript
// frontend/src/hooks/useFlawChessEngine.ts — EXISTING, unchanged (analysis board):
const FLAWCHESS_ENGINE_MAX_NODES = 400;
const FLAWCHESS_ENGINE_MAX_PLIES = 8;
// budget.concurrency = computePoolSize()  (unchanged call site)

// NEW — bot-play-only profile, exported so Phase 169 + the harness can read it
// without invoking the analysis-board's React hook:
export const FLAWCHESS_BOT_MAX_NODES = 50; // [ASSUMED — validate via measurement, D-07]
export const FLAWCHESS_BOT_CONCURRENCY = 4; // [ASSUMED — pin from measured device profile, D-09]
export const FLAWCHESS_BOT_STOP_RULE = {
  marginThreshold: 0.05,   // [ASSUMED — tune via harness, D-06]
  epsilonThreshold: 0.02,  // [ASSUMED — tune via harness, D-06]
  stabilityWindow: 3,      // [ASSUMED — "K applied expansions", D-05]
  minNodes: 8,             // [ASSUMED — shared floor, D-05]
};
```
The harness's D-11 mirror block (`scripts/calibration-harness.mjs` lines ~87–104) then mirrors `FLAWCHESS_BOT_*`, not `FLAWCHESS_ENGINE_MAX_NODES` — continuing the existing comment-mirror convention (Claude's Discretion notes this is an open choice; the comment-mirror pattern is lower-risk than adding a new `@/` import surface purely for constants, and matches Phase 168's own precedent).

**Where Phase 169 fits:** Phase 169 (`useBotGame`, not yet built) most plausibly calls `selectBotMove` DIRECTLY (mirroring exactly how the harness's `playGame()` already does it — see `scripts/calibration-harness.mjs` lines 443–459), NOT via the `useFlawChessEngine` React hook (that hook's anytime-emit/throttle machinery is built for the analysis board's incremental line display, not for "compute one move and return a UCI string"). This phase's job is to land the bot-specific constants + stop-rule support so Phase 169 has something concrete to import — it does NOT need to wire a live bot-play call site of the hook itself.

### Pattern 2: Stop-rule evaluation point — reuse the existing canonical-apply-order loop

**What:** `mctsSearch.ts`'s main loop already has exactly the right hook point: the `for (const result of results)` loop (lines 431–438) applies each dispatched expansion **in canonical order** (via `Promise.all`'s order-preserving resolution, never raw arrival order) and calls `onSnapshot(...)` after every one. SEED-096's evaluation notes explicitly call for "evaluate the stop condition in canonical apply order" — this is exactly that loop, no new mechanism needed.

**When to use:** Implementing D-05/D-06.

**Mechanics:**
1. After `applyExpansion(result, rootMover)` and the `nodesEvaluated += 1` increment (i.e., right where `onSnapshot` already fires), read `root.children` directly for their `.value` (this IS `RankedLine.practicalScore` — see `treeCommon.ts`'s `buildRankedLines`, `practicalScore: child.value`). Do NOT call `buildSnapshot`'s sorted `buildRankedLines` for this check — that sorts by findability-weighted `rankScore`, not `practicalScore`, and is needless allocation on every expansion. Scan `root.children.values()` directly for the top-2 `.value`s and the current argmax `.uci`.
2. Track a small rolling window (size `stopRule.stabilityWindow`) of "which uci was the argmax at each check" — both sides of the two-sided rule need "stability"/"stopped churning", which is a multi-check property, not a single-snapshot one.
3. **Clear-winner check:** `top.value - runnerUp.value >= stopRule.marginThreshold`, AND this has held for `stopRule.stabilityWindow` consecutive checks (the argmax UCI hasn't changed across the window).
4. **Near-tie-flatness check:** `max(childValues) - min(childValues) <= stopRule.epsilonThreshold` across ALL current root children (not just top-2), AND the argmax hasn't changed for `stopRule.stabilityWindow` checks.
5. Both checks gate on `nodesEvaluated >= stopRule.minNodes` first (shared floor — prevents stopping on a single expansion's noise).
6. On satisfying either check, set a local `earlyStop = true` and let it terminate BOTH the inner dispatch-buffering loop and the outer `while` — a labeled break or an added `&& !earlyStop` term in the outer `while` condition, mirroring the existing `!signal.aborted` pattern.

**Pitfall — `EngineSnapshot.budgetExhausted` is a false dichotomy once early-stop exists.** Its doc comment reads *"True once `maxNodes`/`maxPlies` stopped the search (not an abort)"* — a binary. Early-stop is neither budget exhaustion nor abort; it is a third, deliberate stop reason. Grep confirms no UI currently *reads* `budgetExhausted` for display (it flows through `useFlawChessEngine.ts`'s returned state but nothing renders it yet), so there is no live regression risk, but silently mapping "stopped early" to `budgetExhausted: true` would be misleading to future consumers (and to the harness's own `budgetExhausted` reporting, if it ever surfaces per-move stop reason in TSV output). Recommend a new field (e.g. `stopReason: 'budget' | 'early-stop' | null`, or simply an added `stoppedEarly: boolean` alongside `budgetExhausted`) rather than overloading the existing boolean — additive to `EngineSnapshot`/`treeCommon.ts`'s `buildSnapshot`.

**Pitfall — the stop rule must NOT engage for the analysis board's calls.** Since `SearchBudget.stopRule` is optional/`undefined` for existing callers (analysis board, every current test in `mctsSearch.test.ts`), and the search loop should simply skip the stop-rule check entirely when `budget.stopRule` is undefined, this is naturally backward-compatible — but it is exactly the kind of contract every existing `mctsSearch.test.ts` budget object (`{ maxNodes, elo, maxPlies, concurrency }`, no `stopRule` key) implicitly exercises. Confirm the omitted-field path is covered by at least one new explicit test (`stopRule: undefined` behaves byte-identical to today) alongside the new stop-rule-enabled tests.

### Pattern 3: SEED-095 fix mechanics — depth-only `go` + Clear Hash, decoupled watchdog

**What:** Today `nodeGrade`/`evalPositionCp` (`scripts/lib/calibration-providers.mjs`) send a `go` with BOTH a `depth` clause AND a `movetime` clause — Stockfish stops at whichever is hit first, which under CPU load means the ACTUAL reached depth (and therefore the eval) is load-dependent. D-10 removes the `movetime` clause entirely (pure `go depth N`, no wall-clock self-limit) and adds `setoption name Clear Hash` immediately before every `go` (bot grading, adjudication — anchor Stockfish-skill moves are explicitly OUT of scope for this determinism guarantee, since `Skill Level` weakening is itself a source of intentional engine-internal randomness the D-09 determinism check already sidesteps by using a Maia anchor, not an SF-skill anchor, for its own proof).

**Two real consequences the planner must size, not assume:**

1. **The harness-side "watchdog" (`StockfishUciEngine.waitFor`'s `timeoutMs`) is CURRENTLY DERIVED from the movetime cap** — e.g. `nodeGrade` calls `waitFor(predicate, GRADING_MOVETIME_SAFETY_CAP_MS + SLACK_MS)` (2500+2500=5000ms). Once `movetime` is removed from the `go` command, there is no longer a natural upper bound the engine self-enforces — the watchdog constant MUST become an independently-sized, generous constant (D-11: "raise the timeout well above worst observed"), not a formula derived from a movetime value that no longer exists in the command. This is a real code-shape change to every `waitFor` call site in `calibration-providers.mjs` and `calibration-anchors.mjs`, not just a constant-value bump.
2. **`Clear Hash` before every single `go` forfeits the transposition table entirely, every call** — Stockfish's depth-14 search normally benefits from TT hits across the many `go`s issued within one game (same engine instance reused for the whole game per `pool.newGameAll()`'s existing per-GAME reset). Resetting the hash per-CALL (not per-game) means every grading/adjudication search starts cold. SEED-096 explicitly flags "measure the throughput cost... don't assume it" — this is the single largest unquantified risk to the D-16 24h sweep envelope. **Recommend measuring this on a small bounded run (e.g. 1-2 games at blend=1) BEFORE committing to the D-14 grid size**, mirroring exactly how the 2026-07-12 "Measured" bounded run already validated the MAX_NODES sizing question.

**Open question left to the planner (not resolved in CONTEXT.md):** adjudication's search depth. `ADJUDICATION_MOVETIME_MS = 500` (a cheap quick eval) is one of "the constants D-10/D-11 rework" per canonical_refs, but no `ADJUDICATION_TARGET_DEPTH` value is decided. Reusing `GRADING_TARGET_DEPTH = 14` for adjudication (which runs after EVERY ply of EVERY game, far more often than bot-move grading) would multiply the Clear-Hash throughput cost dramatically. SEED-095's original proposal offered "reset hash before every go OR depth-cap adjudication" as alternative fixes; D-10 combines both, but a SHALLOWER adjudication-specific depth constant (e.g. 8–10) is very likely the right choice to keep adjudication cheap while still deterministic (depth-capped + Clear Hash). Flag this as a **Claude's Discretion gap**, not a locked decision — size it from measurement.

### Anti-Patterns to Avoid
- **Retuning `FLAWCHESS_ENGINE_MAX_NODES` in place.** Silently degrades `/analysis`'s engine card — see Pattern 1.
- **Deriving the new harness watchdog timeout as a formula off the (now-removed) movetime constant.** The watchdog must be sized independently once `movetime` leaves the `go` command — see Pattern 3.
- **A margin-only stop rule.** SEED-096/D-05 is explicit that this "gets openings exactly backwards" — near-equal moves in routine positions never trigger a margin-only rule, so it would run EVERY opening move to full budget. The near-tie-flatness side of D-05 exists specifically to catch this case.
- **Wall-clock-based stop conditions of any kind** (a `performance.now()` check inside the stop rule, or a setTimeout-based cap). SEED-096 point 3 is explicit: a wall-clock cap is machine-dependent, breaks D-09 determinism, and makes measured ELO depend on which box ran the sweep. Every stop signal must be a pure function of grade/tree state.
- **Assuming early-stop is "strength-neutral by construction."** SEED-96's evaluation notes are explicit that the visit-futility trick (which WOULD be strength-neutral) doesn't apply here — FlawChess selects by `practicalScore` (value), not visit count, so any margin-based stop is genuinely lossy and MUST be re-measured by the D-14/SC4 sweep, not assumed safe.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Watchdog timeout / retry-on-timeout | A new promise-race/timeout wrapper | Extend the EXISTING `StockfishUciEngine.waitFor(predicate, timeoutMs)` + `stopAndSync()` (`scripts/lib/node-engine-providers.mjs`) | This machinery already exists, already handles process-death/stdin-error edge cases (`#failPendingWaiters`) — the phase only needs to (a) decouple the timeout value from movetime and (b) add a retry-N-times loop AROUND calls to `nodeGrade`/`evalPositionCp`, not reimplement the underlying wait |

## Common Pitfalls

### Pitfall 1: Overwriting the analysis board's search quality
**What goes wrong:** Retuning `FLAWCHESS_ENGINE_MAX_NODES` from 400 to ~50 in place silently cripples the `/analysis` page's engine card (a shipped, in-production feature with no pacing pressure).
**Why it happens:** The single constant is named generically (`FLAWCHESS_ENGINE_MAX_NODES`) and CONTEXT.md's canonical_refs describe the retune target AS this file/line, inviting an in-place edit.
**How to avoid:** Introduce a parallel, additively-named bot-play constant block (Pattern 1); leave the existing analysis constants untouched.
**Warning signs:** Any diff that changes the VALUE of `FLAWCHESS_ENGINE_MAX_NODES` (not just adds new constants nearby) should be treated as a red flag requiring explicit justification.

### Pitfall 2: Watchdog timeout silently orphaned from the removed movetime cap
**What goes wrong:** After dropping `movetime` from the `go` command (D-10), every `waitFor(predicate, GRADING_MOVETIME_SAFETY_CAP_MS + SLACK_MS)` call site still references the old (now-irrelevant) movetime-derived constant, so the harness starts timing out on legitimately-long depth-14 searches that used to be capped by movetime but no longer are.
**Why it happens:** `GRADING_MOVETIME_SAFETY_CAP_MS`/`SLACK_MS`/`ADJUDICATION_MOVETIME_MS`/`ADJUDICATION_SLACK_MS` are currently BOTH the UCI `go` parameter AND the watchdog sizing input — removing the UCI usage without also resizing the watchdog input leaves a stale, too-small timeout.
**How to avoid:** Introduce an independent watchdog constant (D-11) sized from real measurement of depth-14-with-Clear-Hash worst-case latency, not derived from the (now unused) movetime value.
**Warning signs:** Any `waitFor` call whose timeout expression still contains `GRADING_MOVETIME_SAFETY_CAP_MS`/`ADJUDICATION_MOVETIME_MS` after the `movetime` clause is removed from the corresponding `go` command.

### Pitfall 3: Clear-Hash throughput cost blows the D-16 24h envelope
**What goes wrong:** Resetting the hash table before every single `go` (rather than once per game) could substantially slow every grading/adjudication call, shrinking the number of games/anchors that actually fit in the ~24h sweep envelope — discovered only AFTER the full sweep is launched.
**Why it happens:** The determinism fix and the throughput/sizing decision are coupled but were decided somewhat independently in CONTEXT.md ("measure, don't assume").
**How to avoid:** Run a small bounded measurement (mirroring the 2026-07-12 "Measured" run's own methodology) with the D-10 fix applied, BEFORE finalizing D-14's grid size / D-16's games-per-cell.
**Warning signs:** A bounded pilot run's moves/sec is markedly worse than the pre-fix baseline documented in SEED-096 (blend=1: ~0.28s/expansion, ~110s/move at 400 nodes) after accounting for the new smaller MAX_NODES.

### Pitfall 4: Anchor-pruning loop restructuring breaks TSV durability invariants
**What goes wrong:** D-15's dynamic cutoff (skip stronger/weaker anchors once a cell's outcome is decided) requires restructuring `playCell()`/`main()`'s current nested-loop shape (fixed `--games-per-cell` per `(botElo, botBlend, anchor)` triple, one durable TSV row written per completed cell) into something that can decide, PARTWAY through a cell's anchor list, to skip the remainder. This risks breaking the existing WR-01 durability guarantee (a crash mid-sweep keeps every completed cell's row) and the `--resume` (SEED-097) skip-logic, both of which assume a stable, exhaustive `(botElo, botBlend, anchor)` grid enumerated up front.
**Why it happens:** The grid is currently generated exhaustively (`gridKeys` built from the full cross-product) before any game is played; D-15 needs per-cell, mid-sweep decisions that shrink the effective grid at RUN TIME, which `--resume`'s "prior cell not in current grid → refuse" guard was explicitly NOT designed to tolerate.
**How to avoid:** Treat a skipped anchor as a REAL row in the main TSV (games=0, with a `skip_reason` column or similar — CONTEXT.md's Claude's Discretion explicitly calls out "Sweep TSV schema extensions (early-stop stats, skipped-matchup log)" as open) rather than an absent row, so `--resume`'s grid-membership check and the durability model both stay intact.
**Warning signs:** A `--resume` run that throws "prior cell ... is not in the current grid" after a D-15-pruned run, or a summary TSV silently missing rows for skipped anchors with no logged reason (violating D-15's own "no silent coverage gaps" requirement).

### Pitfall 5: knip flagging new exported constants
**What goes wrong:** If bot-play constants are exported from `useFlawChessEngine.ts` or a new module but Phase 169 (which would consume them) isn't built yet, `npm run knip` (CI-gating per CLAUDE.md) may flag them as unused exports.
**Why it happens:** knip detects dead exports; this phase intentionally ships constants ahead of their Phase 169 consumer.
**How to avoid:** Either (a) have the harness `.mjs` files import them via the `@/` alias (satisfying knip's usage detection, and formalizing the "import" option from CONTEXT.md's Claude's Discretion note), or (b) if keeping the comment-mirror convention (no cross-import), add a `knip.json` `ignoreExports` entry with a comment explaining the Phase-169-not-yet-built reason, mirroring the project's existing precedent (`onnxruntime-web` in `ignoreDependencies` for an analogous "invisible to knip's glob" reason).
**Warning signs:** `npm run knip` failing in CI after this phase's frontend changes land.
