# Phase 168.5 Plan 03: Bot move pacing/search budget — mctsSearch stop-rule + bot budget profile Summary

**Two-sided grade-margin early-stop wired into mctsSearch's canonical apply loop, plus a distinct, additive FLAWCHESS_BOT_* budget profile in useFlawChessEngine.ts — both provisional pending Plan 04's harness measurement.**

## Performance

- **Duration:** ~20 min
- **Completed:** 2026-07-12

## Accomplishments

- `SearchBudget.stopRule?: BotStopRule` (marginThreshold/epsilonThreshold/stabilityWindow/minNodes) and `EngineSnapshot.stopReason: 'budget' | 'early-stop' | null` added as optional/sibling fields — `stopRule` omitted is byte-identical to today's full-budget search (proven by a dedicated omitted-vs-explicitly-undefined test)
- `mctsSearch`'s existing `for (const result of results)` apply loop now evaluates a two-sided stop rule after every applied expansion: clear-winner (top child leads the runner-up by `marginThreshold`, argmax stable for `stabilityWindow` checks) OR near-tie-flatness (all root children within `epsilonThreshold`, argmax stable), both gated by a shared `minNodes` floor — reads `root.children`'s `.value` directly, no wall-clock signal anywhere
- New `FLAWCHESS_BOT_MAX_NODES` (50, provisional), `FLAWCHESS_BOT_MAX_PLIES` (8), `FLAWCHESS_BOT_CONCURRENCY` (4, provisional), `FLAWCHESS_BOT_STOP_RULE` exported from `useFlawChessEngine.ts` in a visually separated block, sitting alongside — never replacing — the untouched `FLAWCHESS_ENGINE_MAX_NODES = 400` / `FLAWCHESS_ENGINE_MAX_PLIES = 8` / `computePoolSize()` call site
- 6 new unit tests across `mctsSearch.test.ts` (clear-winner, near-tie-flatness, min-nodes floor, undefined no-op) and the new `useFlawChessEngine.test.tsx` (bot-profile distinctness + well-formedness) — full frontend suite (142 files, 1812 tests) green, `tsc -b --noEmit` clean, `knip` clean with no config changes needed

## Decisions Made

- Stop-rule evaluation reads `root.children`'s own `.value` fields directly (== `RankedLine.practicalScore`), never the findability-sorted `buildRankedLines` — matches Pattern 2's explicit guidance and avoids needless per-expansion allocation.
- A single shared argmax-stability counter (reset whenever the argmax UCI changes, incremented otherwise) serves both sides of the two-sided rule, since "the argmax has settled for K checks" is one concept both conditions gate on.
- `buildSnapshot`'s new `stopReason` parameter defaults to `null` rather than becoming a required argument — keeps `fallbackExpectimax.ts` (untouched, out of this plan's declared file scope) compiling without modification, at the cost of that runner never reporting `'budget'` explicitly (it stays `null` even when its own `budgetExhausted` is true). Acceptable: this plan's mandate is `mctsSearch`'s stop rule, not `fallbackExpectimax`'s parity.
- Task 4's test asserts on all four `FLAWCHESS_BOT_*` exports (not just the two named in the plan's acceptance criteria) — this made every export knip-visible without an `ignoreExportsUsedInFile`/`ignore` config change, avoiding an unplanned `knip.json` edit while still satisfying the plan's literal acceptance bullets.
- All `FLAWCHESS_BOT_*` numeric values (50/8/4/margin 0.05/epsilon 0.02/window 3/minNodes 8) are PROVISIONAL per RESEARCH.md Pattern 1 — every constant carries a `PROVISIONAL` comment; Plan 04 measures against the harness and locks final values here and in the harness's D-11 mirror block.
