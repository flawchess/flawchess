<objective>
Wire the persistence writes into `useBotGame`: snapshot on every move, snapshot with the
clock fold on tab-hide/pagehide, enqueue-and-clear on game end, clear on new game.

Purpose: this is the plan that makes RESUME-01's "persists on every move" and RESUME-02's
"an abandoned game leaves no server trace" true. SC2 is made STRUCTURAL here: the POST can
only ever be fed from a queue that only `finalizeGame` writes to.

Output: `useBotGame.ts` writes/clears/enqueues at exactly four call sites, plus the tests
that go red when any of them is removed.

## The two traps in this plan (both from RESEARCH.md)

1. **Stale closure.** The existing `visibilitychange` effect (useBotGame.ts:741-757) has `[]`
   deps ON PURPOSE — it only touches refs. The moment a hide handler also needs to READ
   `activeColor` / `clockBaseRef` / the latches to build a snapshot, a `[]`-deps closure
   silently reads values frozen at first mount. That is exactly the Phase 169 half-invariant
   shape: invisible to tsc, eslint, knip, and a green suite. Give the hide-time write its
   OWN effect with a real dependency array.
2. **Double-subtracting the fold.** `foldClockBasesForSnapshot` must produce a value written
   into the snapshot payload only. If a write mutates `clockBaseRef.current`, the next real
   `commitMove()` subtracts the same elapsed time a second time (once via the fold, once via
   its own `chargeableElapsedMs()` read) and the user's clock silently double-drains.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| game loop → localStorage (write) | Writes fire on every move and on every tab-hide. A throwing write (QuotaExceeded, Safari private mode) must not break the game in progress. |
| game loop → pending-store queue | The queue is the ONLY input to the server POST. Anything that can write to it can put a game in the user's Library. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-170-09 | Tampering (integrity of the store-once guarantee) | `enqueuePendingStore` call sites | high | mitigate | SC2 depends on `finalizeGame` being the SOLE enqueue site. A second enqueue site (e.g. in `commitMove`, or in a future "autosave to server" feature) would silently put UNFINISHED games on the server, violating RESUME-02 outright. Mitigation: exactly one call site, placed behind the WR-03 first-outcome-wins latch, with a source assertion in the acceptance criteria and a `store-once` test that goes red if the enqueue is moved. |
| T-170-02 | Denial of Service | `writeSnapshot` firing on every move | medium | mitigate (Plan 01) | Every write is already inside `botGameSnapshot`'s try/catch and degrades to a silent no-op on a storage throw — so a full/blocked localStorage disables resume but never breaks the game in progress. This plan adds no new unguarded storage access. Manual verification (Safari private browsing) is tracked in `170-VALIDATION.md`'s Manual-Only table. |
| T-170-10 | Tampering (clock integrity) | the D-01 fold double-subtracting | medium | mitigate | If a snapshot write mutated `clockBaseRef.current`, the user's clock would silently double-drain on the next move — an integrity bug in the one thing the phase's goal calls "fairly paused". Mitigation: `foldClockBasesForSnapshot` returns a copy; the live ref is never written by a snapshot path; revert-proof #2 pins it. |
| T-170-01 | Tampering | user hand-edits the snapshot mid-game to gain clock | low | accept | Same disposition as Plan 01's T-170-01: single-player, unrated, self-affecting only; the server derives every scored field from the PGN and the JWT. Client-side clock fairness here is a UX property, not a security control. |
</threat_model>
