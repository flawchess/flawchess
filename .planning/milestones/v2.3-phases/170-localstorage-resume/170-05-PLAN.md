<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| shared browser → `/bots` | Two different accounts may use the same browser profile. A snapshot or a queued finished game written by user A must never be resumed, or stored, under user B's account. |
| localStorage → gate render | A tampered/corrupted snapshot reaching the gate must not white-screen the page. |
| client → `POST /bots/games` | The drain fires automatically on page load, without user action. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-170-04 | Information Disclosure | cross-user snapshot / queue on a shared browser | medium | mitigate | `Bots.tsx` resolves `ownerKey` from `useUserProfile().data?.email` and passes it to EVERY snapshot/queue call, and it waits for the profile to settle before the first read. Combined with Plan 01's per-owner key scoping, user B loading `/bots` cannot see, resume, or auto-store user A's game — the keys are physically different. This mirrors `useUserFlag.ts`'s existing per-email scoping and its stated rationale. Residual (accepted, documented): a guest's `anon`-scoped snapshot is not carried across a later registration; Phase 171 owns the guest flow and can migrate the key at promotion time. |
| T-170-11 | Elevation of Privilege / Tampering | the drain firing automatically on page load | low | accept (mitigated upstream) | The drain POSTs with the current session's JWT. A game can therefore only ever land in the Library of whoever is currently authenticated — the server derives `user_id` from the token and ignores any client claim (Phase 167). Combined with T-170-04's key scoping, an automatic drain cannot attribute A's game to B. No consent prompt is needed for the user's OWN finished game; storing it IS the product. |
| T-170-02 | Denial of Service | a corrupted snapshot crashing the `/bots` render | medium | mitigate | The gate is only rendered from a snapshot that already passed `readSnapshot`'s parse + version + shape validation and, on corruption, was cleared and captured once. `Bots.tsx` never reads localStorage directly. A failed read degrades to the no-snapshot branch: `/bots` auto-starts a fresh game exactly as today. |
| T-170-12 | Repudiation / data loss (user-facing integrity) | Discard destroying a queued FINISHED game | medium | mitigate | The discard path calls `clearSnapshot` ONLY. It never touches `flawchess_bot_pending_store:*`. If the two shared a key (or discard cleared both), a user whose last game failed to store and who then discarded a separate in-progress snapshot would lose that finished game permanently, with no trace and no error. Pinned by a source assertion (`Bots.tsx` contains no `removePendingStore`) and by Plan 04's `newGame`-leaves-the-queue test. |
</threat_model>
