<objective>
Write the frontend's first call site for the Phase 167 `POST /bots/games` endpoint, and the
drain loop that decides — per HTTP outcome — whether a queued finished game survives to the
next page visit.

Purpose: RESUME-02's "each finished game is stored exactly once". The server already makes a
double-store structurally impossible (`uq_games_user_platform_game_id` keyed on the
client-minted `game_uuid`); this plan's job is the mirror invariant on the client — never
lose a finished game before a confirmed store, and never pin a permanently-unstorable one.

Output: `frontend/src/types/bots.ts`, a `botsApi` block in `frontend/src/api/client.ts`,
`frontend/src/hooks/useStoreBotGame.ts` (mutation + drain), and its test file.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (`POST /bots/games`) | The client supplies `game_uuid`, `pgn`, and the settings that produced the game. Everything else the game record contains (result, termination, per-move clocks, TC bucket, rating anchor, `user_id`) is DERIVED SERVER-SIDE from the PGN and the JWT — Phase 167 D-15. The client is not trusted to assert any of them. |
| localStorage queue → client (read) | A tampered/corrupted queue entry can only produce a request the server then validates and bounds. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-170-05 | Denial of Service | `POST /bots/games` request body | medium | transfer | An oversized/forged PGN is already rejected server-side by `StoreBotGameRequest`'s `Field(max_length=MAX_BOT_PGN_LENGTH)` plus the ELO/blend/tc_preset bounds (Phase 167, shipped and unchanged this phase). A normal snapshot payload is ~2 KB, orders of magnitude under the bound. No new client-side length cap is added — duplicating the server's bound on the client would be defence theatre with a second value to drift. |
| T-170-06 | Spoofing / Elevation of Privilege | game attribution (`user_id`) | low | accept (already mitigated upstream) | The client CANNOT specify whose Library a game lands in: the endpoint depends on `current_active_user` and derives `user_id` from the JWT (`app/routers/bots.py`). A tampered queue entry stores to the tamperer's own account or gets a 401. No new control is needed here. |
| T-170-07 | Tampering (replay / double-store) | queue drain re-POSTing an already-stored game | low | mitigate | The drain treats `created: false` as SUCCESS and removes the entry — combined with the stable `gameUuid` (Plan 03, D-11) and the server's `uq_games_user_platform_game_id` idempotency constraint keyed on `platform_game_id = game_uuid`, a re-POST after a lost response is a structural no-op rather than a duplicate game. Removing the entry on `created:false` is exactly what prevents a permanently-retried "already stored" game. |
| T-170-03 | Denial of Service (self) | never-draining queue | medium | mitigate | 422 entries are dropped rather than retried forever (they can never succeed). 401/5xx entries are kept but the queue is capped at `MAX_PENDING_STORE_ENTRIES` (Plan 01), so an extended outage bounds rather than unbounds localStorage growth. |
</threat_model>
