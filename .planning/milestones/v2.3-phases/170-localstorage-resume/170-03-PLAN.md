<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| snapshot (localStorage) → game loop | A restored `BotGameSnapshot` seeds the live game's refs. A malformed PGN reaching `restoreChess` would throw inside a lazy `useState`/`useRef` initializer, i.e. during render — a white-screen crash rather than a graceful degradation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-170-02 | Denial of Service | `useBotGame` lazy initializers seeded from `resume.pgn` | medium | mitigate | The hook is only ever handed a snapshot that already passed `readSnapshot`'s parse + version + shape validation (Plan 01) — an unparseable value never becomes a `BotGameSnapshot`. `readSnapshot` is the single entry point; the hook does not read localStorage itself. Residual risk: a well-formed JSON payload carrying a syntactically-invalid `pgn` string (a hand-tampered value) would still throw in `restoreChess`. Accepted as low: it is self-inflicted, single-user, and recoverable by clearing site data — matching T-170-01's disposition. Do NOT add a try/catch inside a render-phase initializer to "fix" this; that hides real bugs. |
| T-170-08 | Denial of Service (resource) | bot search started before the user confirms Resume | medium | mitigate | The `live` gate. Without it, merely LOADING `/bots` with a stale snapshot on the bot's turn spawns Stockfish workers AND runs a full search — burning battery/CPU on a game the user may be about to discard, and (if the bot's clock was low) potentially flagging it mid-boot for no reason. Gating the three clock/search effects while leaving the bring-up effect unconditional gives warm engines with zero work done. |
| T-170-07 | Tampering (double-store) | `gameUuid` re-minting on resume | high | mitigate | If `gameUuid` were re-minted on resume, a resumed-then-finished game would be stored under a NEW `platform_game_id`, defeating the server's `uq_games_user_platform_game_id` constraint and producing a genuine duplicate game in the user's Library — a direct violation of SC3. Mitigation: `gameUuidRef` is seeded from `resume.gameUuid` and re-minted ONLY in `newGame()`, proven by the `stable-uuid` test. This is the one high-severity threat in the phase and it is fully mitigated. |
</threat_model>
