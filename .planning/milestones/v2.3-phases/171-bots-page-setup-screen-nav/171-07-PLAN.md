<objective>
Close D-21 (the CONTEXT amendment): store the finished game ON FINISH, surface the confirmed save, and
prevent the double-POST that this creates.

RESEARCH.md Pitfall 3: D-20's "once the store CONFIRMS" has no signal to observe today. Phase 170 wired
`finalizeGame()` ŌåÆ `enqueuePendingStore()` (localStorage only); the actual POST fires only on the NEXT `/bots`
mount, via `useDrainPendingStore`. As shipped, a "Saved to your Library" row gated on mutation status would
never appear on the result screen the user is looking at ŌĆö it would activate on a SECOND visit, by which point
the dialog is gone.

The fix adds a finish-time `useStoreBotGame().mutate()`. That creates exactly one new risk, and closing it is
half this plan's job: the next mount's drain must not re-POST a game that already stored. The queue entry is
removed on store success, and a test pins "finish ŌåÆ store ŌåÆ remount ŌåÆ no second POST".

Purpose: PLAY-10 / SC3 / SC4 ŌĆö the finished game demonstrably lands in the Library, and the guest is told what
that does and does not include.
Output: a finish-time store trigger, a deduped queue, and the confirmed-save affordance on both result surfaces.
</objective>
