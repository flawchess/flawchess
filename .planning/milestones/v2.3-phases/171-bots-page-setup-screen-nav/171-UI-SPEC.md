# Phase 171 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope:** the setup screen (`BotsPage`'s no-snapshot branch), the nav entries for `/bots`
(desktop `NavHeader`, mobile `MobileBottomBar` + `MobileMoreDrawer`), and the post-game
"Saved to your Library" + guest-caveat affordance on `GameResultDialog`/`GameResultStrip`.
The clocked board, clocks, move list, resign/draw controls, sounds, and result copy itself
were already specified and shipped in Phase 169/170 — **not re-specified here**.

## Spacing Scale

Declared values (must be multiples of 4), matching the app-wide convention already used by
`Bots.tsx` / `PresetRangeFilter.tsx`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (`gap-1`), preset-chip internal padding |
| sm | 8px | Compact stacked control spacing (`gap-2`), chip grid gaps (`gap-1`→ rounds to this bucket) |
| md | 16px | Setup-screen card padding (`p-4`), gap between setup controls (`gap-4`) |
| lg | 24px | Section separation between control groups on the setup screen |
| xl | 32px | — (not used at this phase's scale) |
| 2xl | 48px | — (not used at this phase's scale) |
| 3xl | 64px | — (not used at this phase's scale) |

Exceptions:
- Preset-button min tap height is **44px** (`h-11` mobile / `h-7` desktop, exactly
  `PresetRangeFilter`'s existing responsive pattern) — a deliberate touch-target
  exception, not a spacing-scale violation.
- Slider thumb hit target is **44px** minimum per `components/ui/slider.tsx`'s existing
  comment (iOS HIG / Material) — inherited, not introduced by this phase.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / control label | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label — active/emphasized (preset active, summary value, ELO number) | 14px (`text-sm`) | 600 (semibold, `font-medium`/`font-semibold` — Tailwind's `font-medium` maps to 500 in this app's existing usage, treat as the "emphasized" weight) | 1.5 |
| Heading — setup screen section title / dialog title | 20px (`text-xl` per `DialogTitle` primitive) | 600 (semibold) | 1.2 |
| Display — none | n/a | n/a | n/a |

Only 2 weights in play: **400 regular** (body/labels/copy) and **600 semibold**
(section headings, active-state emphasis, the Start button label). Do not introduce a
third weight (no `font-bold`/`font-black` anywhere in this phase's new UI).

**Hard floor:** no `text-xs` anywhere in the setup screen, nav items, or post-game
affordance — CLAUDE.md's floor applies. The one permitted exception is the ELO info
popover's `InfoPopover` body, which is an existing hover/tap-triggered transient surface
(`MetricStatPopover`-family pattern) and ships at `text-xs` by established convention.

| Role | Value | Usage |

| Destructive | `--destructive` `oklch(0.577 0.245 27.325)` | Reserved for: none new in this phase — no destructive action is introduced (Start/New-game/color/TC pickers are all non-destructive). Resign's existing destructive-red confirm button (Phase 169) is untouched. |

Accent is NOT used for: setup-screen body copy, the ELO number display, unselected
preset chips, or the color/TC selector rows — those stay in `text-foreground` /
`text-muted-foreground` per the existing `PresetRangeFilter` convention (only the
active/selected element gets the accent treatment).

Nav dot/lock treatment (unchanged mechanism, extended to `/bots`): a locked nav item is
`opacity-40` with no accent tint; the existing red notification dot convention
(`bg-red-500`) is NOT applicable to `/bots` (Bots carries no "new content" badge in this
phase).

## Component Contracts (phase-specific, beyond reusable primitives)

### Play-style control (D-01)

**New component**, NOT a literal `PresetRangeFilter` reuse — that component is hard-wired
to a 2-value range (`[lo, hi]`, `minStepsBetweenThumbs`). Build a sibling
single-thumb variant (e.g. `PlayStyleControl.tsx`) that visually matches
`PresetRangeFilter`'s shell (label + `InfoPopover` + summary row, 2-button preset grid,
`Slider`) but wires a single `value: [blend]` to `components/ui/slider.tsx` (already
supports single-thumb via `value={[v]}`, per its own doc comment).

- Preset grid: `grid-cols-2` (2 buttons: Human, Engine), same `h-11 sm:h-7` chip sizing,
  same active/inactive classes as `PresetRangeFilter`'s preset buttons.
- Slider: `min=0.05 max=1.00 step=0.05`, single thumb, `thumbLabels={['Play style']}`.
- **Human-preset-active visual convention** (Claude's Discretion, resolved here): when
  `blend === 0`, the slider's thumb sits parked at its own minimum position (0.05) since
  0 is not a reachable slider value — dim the slider (the whole slider block, track +
  thumb, at `opacity-50`, matching the existing `data-[disabled]:opacity-50` slider
  convention) and show the Human summary line instead of a numeric blend value. Moving
  the slider directly (dragging while dimmed) exits Human mode immediately — the drag
  gesture itself sets `blend` to whatever position the drag lands on (no separate
  "unlock" step; this matches how `sliderToRange`-style controls behave elsewhere: the
  slider is always the source of truth for its own domain, and Human is simply a value
  outside that domain, expressed only via the preset button).
- Active-preset detection: `blend === 0` → Human active; `blend === 1` → Engine active;
  anything else → no preset highlighted (both buttons in their inactive style), same
  `activePreset: string | null` contract as `PresetRangeFilter`.

### ELO picker (D-04/D-05/D-07)

Claude's Discretion, resolved: **reuse `components/analysis/EloSelector.tsx` as-is**,
unmodified. It already: derives min/max/step from `MAIA_ELO_LADDER` (locked, full
600–2600 ladder per D-04), renders a single-thumb `Slider` + numeric readout + optional
reset-to-default control, and its own `InfoPopover` — but its bundled `EloInfoTooltip`
copy is analysis-board-specific ("The engines play at this rating... defaults to the
mover's rating") and does NOT carry D-05's bot-specific honesty caveat. **Do not
edit `EloSelector.tsx`'s copy** (it's shared with `/analysis`); instead render the setup
screen's own label + `InfoPopover` (the ELO info popover copy specified above) ABOVE or
BESIDE the reused `EloSelector`, OR pass a wrapping label row and suppress
`EloSelector`'s own inline "ELO" + tooltip pair if its props allow (check for a
label-suppression escape hatch at plan time — if none exists, accept the minor
duplication of an "ELO" caption rather than forking the component). The `defaultElo`/
`onReset` props wire directly to D-07's lichess-blitz-equivalent default (or 1500
fallback) exactly as the analysis board's free-play path will after D-08.

### Color picker (D-12)

3-button chip row (`grid-cols-3`), same visual language as the play-style presets
(inactive `bg-inactive-bg` / active `bg-toggle-active`): **White / Black / Random**.
No icons needed — text labels are sufficient at this control's scale. `data-testid`
`setup-color-white` / `setup-color-black` / `setup-color-random`.

### Time control picker (D-14)

A 3-group chip grid (Blitz / Rapid / Classical sub-headers, each a `grid-cols-3` or
`grid-cols-2` row of TC chips — 4/3/2 items respectively), same chip visual language.
Each chip label is the raw preset (e.g. "3+0", "10+5", "30+20"); the sub-header text
("Blitz"/"Rapid"/"Classical") is a plain `text-sm text-muted-foreground` label above
each group, not a button. Active chip = `bg-toggle-active` per the shared convention.
`data-testid` pattern: `setup-tc-{bucket}-{preset}` e.g. `setup-tc-rapid-10-0`.

### Setup screen layout

Single-column stacked card on both mobile and desktop (no side-by-side layout needed —
this is a small control set, not a dashboard): ELO picker → Play style control → Color
picker → Time control picker → Start button, in that order top-to-bottom, `gap-4` between
groups, `p-4` container padding, `max-w-md` (narrower than the in-game board's `max-w-5xl`
— this is a form, not a board page). Reuses `data-testid="bots-page"` container semantics
(same root div, different content) per `Bots.tsx`'s existing `BotsPage`/`BotsGame` split.

### Post-game "Saved to Library" + guest caveat (D-20)

Renders as an additional row BELOW the existing Analyze/New-game action row in both
`GameResultDialog` (dialog footer area, as a `DialogDescription`-style line under the
title) and `GameResultStrip` (an additional line below the button row, matching the
strip's `text-sm` body copy). Guards:
- Only renders once `useStoreBotGame`'s mutation status is `success` (never renders in
  `pending`/`idle`/`error` — no partial-store hedge copy needed per D-20's explicit
  no-dependency-on-POST framing for the Analyze button).
- Guest caveat renders only when `profile.is_guest === true`, appended as a second line
  directly below "Saved to your Library" (not merged into one paragraph — two short
  lines read faster than one long one).

## Nav Contract (D-16/D-17/D-18)

- **Desktop `NAV_ITEMS`** (App.tsx): insert `{ to: '/bots', label: 'Bots', Icon: Bot }`
  (lucide `Bot` icon — matches the "bot" concept; confirm it doesn't collide visually
  with any existing icon at a glance) as the **2nd** entry: `Library, Bots, Openings,
  Endgames`.
- **`NavHeader`'s lock rule**: extend the exemption from `to !== '/library' && to !==
  '/admin'` to also exclude `/bots` — `locked = to !== '/library' && to !== '/admin' &&
  to !== '/bots' && !navUnlocked`. `/bots` is never dimmed, never `aria-disabled`,
  regardless of `navUnlocked`.
- **Mobile `BOTTOM_NAV_ITEMS`**: same insertion, same order, same lock exemption in
  `MobileBottomBar` (`locked = to !== '/library' && to !== '/bots' && !navUnlocked`).
  This grows the bar from 3 to 4 items + the existing More button = 5 total, the
  established mobile ceiling (D-18) — no layout change needed beyond the array growing
  (the bar already uses `flex-1` per-item sizing that accommodates N items).
- **`MobileMoreDrawer`**: renders `NAV_ITEMS` directly, so the Bots entry appears there
  automatically once added to `NAV_ITEMS` — same lock-exemption edit as `NavHeader`.
- **`ROUTE_TITLES`**: add `'/bots': 'Bots'` so the mobile header page-title shows "Bots".
- **`isActive()`**: add a `/bots` branch — `if (to === '/bots') return
  pathname.startsWith('/bots');` (matches the existing prefix-match convention for
  Library/Openings/Endgames, future-proofing if `/bots` ever grows sub-routes).
- Every nav change above touches BOTH desktop (`NavHeader`) and mobile
  (`MobileBottomBar` + `MobileMoreDrawer`) per CLAUDE.md's "apply to mobile too" rule —
  this is explicit in D-18 already, restated here as a hard visual-contract line item.
