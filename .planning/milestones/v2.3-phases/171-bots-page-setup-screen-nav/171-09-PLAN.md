<objective>
Close UAT gap 2 (minor, test 3): the bot board does not highlight the last played move.

Root cause (already diagnosed â€” do NOT re-investigate): a **pure wiring gap, not a bug**. The
shared `ChessBoard` component already implements last-move highlighting via an optional
`lastMove={{from,to}}` prop (`ChessBoard.tsx:66-73` prop, `:336-353` memoized `squareStyles`), and
`Bots.tsx:335` is the **only** `ChessBoard` caller in the app that never passes it â€” because
`useBotGame` never exposes a `lastMove` at all. Its `fenAtPly` helper replays SAN and throws away
the `chess.move()` return values that already carry `.from` / `.to`.

Fix: capture what is already being computed, expose it, pass it.

Purpose: the single most basic board affordance on every chess site. Its absence reads as broken.
Output: `useBotGame` exposes `lastMove`; the bot board renders the standard yellow highlight.
</objective>

<reuse_mandate priority="hard">
## REUSE the existing pattern â€” do not invent one

- **Rendering already exists.** `ChessBoard.tsx:66-73` (the `lastMove` prop) and `:336-353` (the
  memoized `squareStyles` that paint `MOVE_HIGHLIGHT_SQUARE`). Do not touch `ChessBoard.tsx`.
- **The derivation reference is `useChessGame.ts:95-118`** (`computeInitialChessState`'s replay
  loop: it captures `move.from` / `move.to` when `i === ply - 1`). `useBotGame.fenAtPly` is the
  same loop **minus that capture**. Copy the shape.
- **Pass `lastMove` ONLY.** Inherit `ChessBoard`'s default `MOVE_HIGHLIGHT_SQUARE` (`theme.ts:16`).
  Do **NOT** add a new theme constant, do **NOT** pass a `lastMoveColor` override, do **NOT**
  hard-code a colour (`lastMoveColor` is reserved for the analysis board's severity coding).
</reuse_mandate>

<decision priority="hard">
## Derive lastMove from `viewedPly`, NOT from the live tail

The bots board has move-list navigation (`viewPly` / `returnToLive`), so the highlight **must
follow the move list** â€” lichess/chess.com behaviour, and consistent with `useChessGame`, which
derives its own `lastMove` from `currentPly`.

Deriving from the live game (`moveHistory[moveHistory.length - 1]`) would leave a **stale
highlight** on the live tail move while the user scrubs back through history. That is the bug this
decision pre-empts; a test pins it.
</decision>
