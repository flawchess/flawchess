# Phase 57.1: Code Review Report

**Reviewed:** 2026-04-18
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

The Phase 57.1 change set switches the Endgame ELO timeline from a rolling-mean opponent-rating anchor to a per-combo asof-joined actual-rating anchor via `bisect.bisect_right`, adds `per_week_endgame_games` to the schema + frontend types, and swaps `<LineChart>` to `<ComposedChart>` so muted per-week volume bars can sit under the Elo lines on a hidden right Y-axis.

Overall the work is cleanly structured: the orchestrator pre-computes per-combo `(dates, ratings)` arrays and hands them to a pure helper, the helper's per-week counter is correctly bound to endgame events (not emission events), test coverage for the new asof semantics is solid (forward-fill, same-day last-game wins, per-combo isolation, per-week-vs-trailing-window), and the frontend keeps hooks above the early-return branches so Rules-of-Hooks are respected.

One correctness bug is worth addressing before the feature is considered locked: the asof cutoff `next_monday_dt` inherits the time component of the triggering `played_at`, so the cutoff drifts into the next ISO week by up to 24 hours. In practice this can let a Monday-morning rating from week N+1 leak into week N's `actual_elo`, which subtly contradicts "asof end of this ISO week" semantics documented in the phase plan. Second warning is a minor shadowing risk in the tests' helper vs production-path invariant.

No security issues, no hardcoded secrets, no debug artifacts, no dangerous functions detected. Type safety is strong throughout (Literal unions, explicit return types, no `any` on the TS side except in documented narrow-cast sites).

## Warnings

### WR-01: Asof cutoff bleeds into next ISO week by up to 24 hours

**File:** `app/services/endgame_service.py:1023-1030`
**Issue:** The asof cutoff `next_monday_dt` is computed as `played_at - timedelta(days=iso_weekday - 1) + timedelta(days=7)`, which preserves the time-of-day of the triggering `played_at`. The docstring (line 1024) says the cutoff is "start of NEXT Monday (Pitfall 2): inclusive of any Sunday game" — i.e. midnight at the ISO week boundary — but the actual cutoff is `<played_at_time> on next Monday`.

Concrete failure mode: suppose the user plays a Sunday endgame game at 14:00 (week N) and a non-endgame game Monday at 08:00 with a fresh rating (week N+1). Window already has ≥ 10 endgame games. When the Sunday event triggers emission for week N:
  - `played_at = 2026-01-11 14:00` (iso_weekday=7)
  - `monday_dt = 2026-01-05 14:00`
  - `next_monday_dt = 2026-01-12 14:00`
  - `bisect_right(actual_elo_dates, 2026-01-12 14:00)` includes the Monday 08:00 game.
  - Week N's emitted `actual_elo` uses the Monday 08:00 rating — a value from week N+1.

This means the paired-line "gap" plotted for week N can be off by whatever Elo change happened in the first hours of Monday. Not a crash, but a silent correctness error that violates the asof-at-ISO-week-end contract documented in 57.1-CONTEXT.md and the schema docstring.

The existing test suite passes incidentally because `test_asof_same_day_uses_last_game` uses a 12:00 Sunday trigger with all three competing same-day games before 16:00 (well before the 24-hour drift would matter), and the other tests either use a single time-of-day for the whole fixture or don't place games in the first hours of the following Monday.

**Fix:** Pin `monday_dt` and `next_monday_dt` to midnight at the ISO boundary instead of carrying the triggering `played_at`'s time:
```python
# Preserve tzinfo (played_at is tz-aware in production — DateTime(timezone=True))
# so the bisect comparison stays on the same tz axis.
monday_dt = datetime.combine(
    (played_at - timedelta(days=iso_weekday - 1)).date(),
    time.min,
    tzinfo=played_at.tzinfo,
)
monday = monday_dt.date()
next_monday_dt = monday_dt + timedelta(days=7)
```
(Needs `from datetime import time` or equivalent.) Add a regression test seeding a Sunday endgame game and a Monday-morning all-game with a different rating — the asserted `actual_elo` should be the Sunday rating, not the Monday one.

### WR-02: Per-week count can be reported as 0 when an "all" event triggers emission for a future ISO week

**File:** `app/services/endgame_service.py:1004-1046`
**Issue:** The per-week counter is incremented only inside `if side == "endgame"`, but the emission block (lines 1016-1046) runs for every event in the merged stream, including `"all"` events. If the window is already saturated (≥ 10 endgame games from earlier weeks) and the NEXT event is an `"all"` event in a new ISO week with no endgame games yet, an emission is still written for that new week's key — with `per_week_endgame_games = 0`, because no endgame event has fired for that week.

Effect: the frontend volume bar shows 0 for that week (visually invisible, OK), the paired lines still carry values (using the stale trailing window's skill and the asof rating at the all-event), and the tooltip shows "Games this week: 0 (visible combos)". That last piece is misleading — a user reading the tooltip would conclude that week had no endgame games, but the presence of the point implies otherwise. It also contradicts the "only weeks with endgame activity produce points" reading of the docstring (line 962: "only emits on endgame events" — this is stated explicitly in the test `test_actual_elo_asof_forward_fill` docstring, line 3335).

Why it slipped through: the unit tests interleave endgame and all-game events at matching timestamps, so every emission-eligible event happens to be an endgame event. Real data routinely has all-game events on days with no endgame games.

**Fix:** Gate the emission on `side == "endgame"` too — only emit when the current event is itself an endgame event. That also matches the documented intent ("the helper only emits on endgame events"). Move the emission block inside the `if side == "endgame":` branch:
```python
for played_at, side, row in events:
    if side == "endgame":
        endgame_window.append(row)
        endgame_window = endgame_window[-window:]
        iso_year_evt, iso_week_evt, _ = played_at.isocalendar()
        per_week_count[(iso_year_evt, iso_week_evt)] = (
            per_week_count.get((iso_year_evt, iso_week_evt), 0) + 1
        )

        if len(endgame_window) < MIN_GAMES_FOR_TIMELINE:
            continue
        skill = _endgame_skill_from_bucket_rows(endgame_window)
        if skill is None:
            continue

        iso_year, iso_week, iso_weekday = played_at.isocalendar()
        # ... (asof + emission body unchanged)
```
Add a regression test: 10 endgame games in week 1 plus 3 all-games in week 2 with no endgame games → result must not contain a week-2 point.

## Info

### IN-01: `endgame_window[-window:]` slice on every endgame event is O(n) per event

**File:** `app/services/endgame_service.py:1007`
**Issue:** `endgame_window = endgame_window[-window:]` rebuilds a new list of up to `window` elements every time an endgame event is processed. For a user with many endgame games this is O(N * W) where W = 100 (the timeline window). Not a correctness issue; flagged as style because `collections.deque(maxlen=window)` expresses the intent directly and trims in O(1).

**Fix (optional):**
```python
from collections import deque
endgame_window: deque[tuple[Any, ...]] = deque(maxlen=window)
# ...
endgame_window.append(row)  # auto-trims at maxlen
```
Consumers that pass the window to `_endgame_skill_from_bucket_rows` already accept `Sequence`, so no API change needed.

### IN-02: `renderLegend()` is called on every render instead of being passed as a component

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:392`
**Issue:** `<ChartLegend content={renderLegend()} />` invokes the function at render time and passes the returned React node. Recharts' `content` prop also accepts a component/function reference, which lets it control when to re-render the legend. Minor optimization, not a bug. The existing pattern works because React reconciles the identical tree cheaply.

**Fix (optional):** `<ChartLegend content={renderLegend} />` (pass the function itself) — matches Recharts' recommended usage and avoids creating a new React element on every parent render.

### IN-03: `data-testid` on `<Bar>` probably does not reach the DOM

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:402`
**Issue:** `<Bar ... data-testid="endgame-elo-volume-bars" />` — Recharts does not generally forward unknown HTML attributes from series components (`<Line>`, `<Bar>`) to the underlying SVG. The file itself acknowledges this for `<Line>` in the block comment at lines 282-284 (testid moved to the legend button). If no test currently relies on this selector, it is dead identification. If a test DOES rely on it, the test is likely matching nothing.

**Fix:** Either wrap the `<ComposedChart>` in a container div with this testid (the `ChartContainer` already carries `endgame-elo-timeline-chart`), remove this attribute, or verify in a rendered snapshot that Recharts does forward it on `<Bar>` in this version.

### IN-04: Long template-literal keys in the per-row record are magic strings

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:116-119, 348-351`
**Issue:** The suffixes `_endgame_elo`, `_actual_elo`, `_games_in_window`, `_per_week_games` are repeated across the chartData builder, the barChartData builder, and the tooltip renderer. A typo in any one would silently produce `undefined` values at runtime with no type-checker help (the record index signature is `string | number | undefined`). Minor maintainability concern.

**Fix (optional):** Extract a small builder:
```ts
const rowKey = (key: string, suffix: 'endgame_elo' | 'actual_elo' | 'games_in_window' | 'per_week_games') =>
  `${key}_${suffix}` as const;
```
Union-typed suffix keeps typos out, no runtime cost.
