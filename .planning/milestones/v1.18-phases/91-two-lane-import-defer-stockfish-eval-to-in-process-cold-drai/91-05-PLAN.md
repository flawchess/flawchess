<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Lifespan ↔ background tasks | Tasks are trusted in-process coroutines; cancellation contract enforced |
| Lifespan → engine pool | `stop_engine()` is the FINAL teardown step — must run after drain task ends |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-19 | Denial of Service | Drain task hangs on shutdown, blocks Uvicorn from terminating | mitigate | `task.cancel()` raises `CancelledError` inside the coroutine; the loop in `run_eval_drain` does NOT catch CancelledError (Plan 91-02 invariant), so the task terminates within one await. If a `_TIMEOUT_S = 2.0` engine call is in-flight, cancellation propagates through `engine.evaluate` (the EnginePool respects asyncio cancellation). |
| T-91-20 | Tampering | Future edit moves `await stop_engine()` BEFORE drain task await | mitigate | If `stop_engine` runs first, an in-flight drain evaluation gets a "engine shut down" error, captured by drain's general `except Exception:` clause — logged + Sentry, no crash. Mitigation by ordering convention (acceptance_criteria gate). |
| T-91-21 | Repudiation | Drain crashes during normal runtime, silently restarts | accept | The drain's own internal exception handlers (Plan 91-02) log + capture + sleep + continue. Loop never exits except on cancellation. This is the intended behavior — operationally, log scraping for `source=eval_drain` Sentry events is the audit trail. |
</threat_model>
