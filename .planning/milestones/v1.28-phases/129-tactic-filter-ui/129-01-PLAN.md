<objective>
Extend the backend tactic-filter layer (TACUI-04) and tactic-comparison endpoint (TACUI-05) so the
frontend in plans 02/03 can filter flaws by depth + a 3-value orientation and render two bullets per
family. This is the shared-type-producing wave: it locks `TacticBullet.orientation` (schema option A,
per RESEARCH A2) which `types/library.ts` mirrors in wave 2.

Two genuinely new capabilities fall out of the locked decisions:
1. A `max_tactic_depth` filter param bounding the active orientation's depth column, with forced
   mates exempt (D-04/D-05).
2. An `"either"` orientation state = OR across both `missed_*` and `allowed_*` column sets, widening
   the 128 binary `Literal` to `Literal["either","missed","allowed"]` (D-07/D-08).
Plus the comparison endpoint returns BOTH orientation rates per family, ranked top-6 by Missed
(D-13/D-14).

Purpose: every frontend filter axis and the two-bullet grid need backend support that the 128
contract does not provide. Landing it first (single wave) lets the shared schema be locked before
wave 2 touches `types/library.ts`.
Output: extended filter clauses at both SQL sites, a dual-orientation comparison response, and the
backend tests that prove each behavior.
</objective>
