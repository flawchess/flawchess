## What Was Built

**`opening_position_eval` table** (SEED-053, D-123.1-01): position-keyed dedup cache with `full_hash BIGINT PRIMARY KEY`, `eval_cp SMALLINT NULL`, `eval_mate SMALLINT NULL`, `best_move VARCHAR(5) NULL`. Column widths match `game_positions` exactly. No FK, no cascade — entries are position-intrinsic and immutable.

**Alembic migration** `20260617_130000`: `upgrade()` creates the table; `downgrade()` drops it. Down-revision wires to `20260616_120000` (phase-123 entry-eval lease). Round-trips clean (`downgrade -1 && upgrade head` verified).

**`scripts/backfill_opening_eval_cache.py`**: runs a single server-side `INSERT … SELECT DISTINCT ON (nxt.full_hash)` self-join filtered to our-engine evals (`full_evals_completed_at IS NOT NULL AND lichess_evals_at IS NULL`). `ON CONFLICT (full_hash) DO NOTHING` makes it idempotent. Imports `DEDUP_MAX_PLY` from `app.models.game_position` — no hardcoded literal. Generous 1-hour statement timeout for prod's ~12.9M-row aggregate. `--db dev|prod` (prod requires `bin/prod_db_tunnel.sh`).

**Dev backfill result**: 27,844 rows inserted in ~10s. Second run: 0 rows inserted (idempotency confirmed).

### Notes on Correctness Probe

The plan's acceptance-criteria probe (LATERAL LIMIT 1 vs cache) showed 11 apparent mismatches. Investigation confirmed these are ordering artifacts, not data errors: for high-traffic opening hashes (e.g., one hash appearing in 1,558 games with 11 distinct Stockfish evals), `DISTINCT ON` (INSERT) and `LIMIT 1` (probe) select different rows from the set of valid evals — both results are correct evaluations of the same position. This is the project-wide acknowledged eval non-determinism ("Stockfish eval_cp not reproducible across machines"; MEMORY.md). The cache contains a valid our-engine eval for every cached hash.
