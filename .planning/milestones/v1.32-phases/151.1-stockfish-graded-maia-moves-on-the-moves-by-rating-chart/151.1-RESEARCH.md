## Summary

Phase 151.1 is a pure frontend extension of code shipped in Phase 151 ‚Äî no new package, no new
backend, no new route. The entire feasibility question reduces to: **can a second, independently
configured Stockfish WASM worker (searchmoves + MultiPV = up to ~6) run alongside the existing
MultiPV=2 eval-bar worker and the Maia onnxruntime-web worker, on the SAME vendored
`stockfish-18-lite-single.js` binary, without touching the first worker's state machine or
consumers?** Every piece of this app already has a near-identical precedent to mirror:
`useStockfishEngine.ts` is the template for the grading worker's lifecycle (Worker creation,
UCI init, streaming `info depth` commit, stale-guard, tab-hide pause), `useMaiaEngine.ts` is the
template for its ephemeral per-position FIFO cache, and `liveFlaw.ts`/`flawThresholds.ts` already
implement the ENTIRE grading pipeline (`evalToExpectedScore` ‚Üí `classifyLiveSeverity`) that this
phase needs ‚Äî it just needs two new buckets ("best", "good") layered on top of the existing three.

**Primary recommendation:** Build a **second, fully independent Web Worker instance** (same
`ENGINE_PATH`, new hook e.g. `useStockfishGradingEngine`) rather than time-sharing or mode-switching
the existing `useStockfishEngine` instance. This is the only approach that structurally guarantees
SC3's "without disturbing the eval bar / engine card" ‚Äî the primary hook's `pvMapRef`, state
machine, and every downstream consumer (`EngineLines`, `VariationTree`, `bestSanFromPv`) stay
byte-for-byte untouched. Grade the position with `go depth <D> searchmoves <ucis...> movetime
<safety-cap>` after `setoption name MultiPV value <candidateCount>`, parse each `info` line's
`pv[0]` (NOT its `multipv` index ‚Äî index is rank, not move identity, and can reorder as depth
increases) to key a `Map<SAN, {evalCp, evalMate, depth}>`, and derive the 5 quality buckets by
running the existing `evalToExpectedScore`/`classifyLiveSeverity` pipeline against the max eval
found among the candidates (see Code Examples). Cache grades per-FEN so ELO-slider drags ‚Äî
which only change the *displayed subset*, never the eval ‚Äî never re-trigger a search unless the
newly-revealed move hasn't been graded yet.

### Pattern 1: Second independent Worker instance, not a mode-switch

**What:** `useStockfishGradingEngine` creates its OWN `new Worker(ENGINE_PATH)` (identical vendored
path), with its own `workerRef`, `stateRef`, `pvMapRef`-equivalent, and lifecycle effect ‚Äî a
structural sibling to `useStockfishEngine`, not a shared instance.

**When to use:** Whenever a UI surface needs a materially different search configuration
(MultiPV count, searchmoves restriction, stop condition) than an already-running engine instance
that other consumers depend on staying stable.

**Example (adapted from the real `useStockfishEngine.ts` pattern, verified in this repo):**
```typescript
// Source: frontend/src/hooks/useStockfishEngine.ts (mirrored, not copied verbatim)
const worker = new Worker(ENGINE_PATH); // '/engine/stockfish-18-lite-single.js'
worker.postMessage('uci');
// ... on 'uciok':
worker.postMessage(`setoption name MultiPV value ${candidateSans.length}`);
worker.postMessage('isready');
// ... on 'readyok', when a (fen, candidateUcis) pair is ready to grade:
worker.postMessage(`position fen ${fen}`);
worker.postMessage(
  `go depth ${GRADING_TARGET_DEPTH} searchmoves ${candidateUcis.join(' ')} movetime ${GRADING_MOVETIME_SAFETY_CAP_MS}`,
);
```

### Pattern 2: Identify a graded move by `pv[0]`, never by `multipv` index

**What:** In a restricted-searchmoves + MultiPV=N search, `multipv` is a RANK (1 = currently best
among the candidates), not a stable move identity. As search depth increases, two candidates can
swap rank (the whole point of D-05's progressive refinement ‚Äî a move that looks "best" at depth 4
can be correctly demoted once a refutation is found at depth 12). The existing
`useStockfishEngine.commitPvSnapshot` pattern keys its map by `multipv` because with an
UNRESTRICTED search + MultiPV=2, that's an acceptable simplification for "top 2 lines" display.
For grading, key the map by the UCI move at `parsed.pv[0]`, converted to SAN.

**When to use:** Any time `searchmoves` restricts the move set and every restricted move must keep
its own identity across depth-iterations.

**Example:**
```typescript
// NEW pattern ‚Äî contrast with useStockfishEngine's pvMapRef.set(parsed.multipv, ...)
const parsed = parseInfoLine(line); // reuse uciParser.ts verbatim ‚Äî no changes needed there
if (parsed !== null && parsed.pv[0] !== undefined) {
  const san = bestSanFromPv(gradingFen, parsed.pv[0]); // reuse Analysis.tsx's bestSanFromPv logic
  if (san !== null) {
    gradeMapRef.current.set(san, {
      evalCp: toWhitePov(parsed.scoreCp),
      evalMate: toWhitePov(parsed.scoreMate),
      depth: parsed.depth,
    });
  }
}
```

### Pattern 3: Quality-bucket derivation reusing the existing grading pipeline verbatim

**What:** `evalToExpectedScore` + `classifyLiveSeverity` (from `liveFlaw.ts`) already implement
exactly the drop-based 3-band classification (inaccuracy/mistake/blunder) this phase's yellow/
orange/red map onto. "best" and "good" are the only new concepts, both derived from a simple
comparison against the maximum expected score found among the graded candidates.

**Example:**
```typescript
// NEW: frontend/src/lib/moveQuality.ts
import { evalToExpectedScore, classifyLiveSeverity, type MoverColor } from '@/lib/liveFlaw';

export type MoveQuality = 'best' | 'good' | 'inaccuracy' | 'mistake' | 'blunder';

export interface MoveGrade {
  evalCp: number | null;
  evalMate: number | null;
  depth: number;
}

export function classifyMoveQuality(
  gradeMap: Map<string, MoveGrade>,
  mover: MoverColor,
): Map<string, { quality: MoveQuality; expectedScore: number }> {
  const scores = new Map<string, number>();
  let bestSan: string | null = null;
  let bestEs = -Infinity;
  for (const [san, grade] of gradeMap) {
    const es = evalToExpectedScore(grade.evalCp, grade.evalMate, mover);
    scores.set(san, es);
    if (es > bestEs) {
      bestEs = es;
      bestSan = san;
    }
  }
  const result = new Map<string, { quality: MoveQuality; expectedScore: number }>();
  for (const [san, es] of scores) {
    if (san === bestSan) {
      result.set(san, { quality: 'best', expectedScore: es });
      continue;
    }
    const severity = classifyLiveSeverity(bestEs, es); // null | inaccuracy | mistake | blunder
    result.set(san, { quality: severity ?? 'good', expectedScore: es });
  }
  return result;
}
```

Note: `classifyLiveSeverity`'s existing signature (`esBefore, esAfter`) exactly fits ‚Äî `esBefore` is
the position's best-achievable expected score (found via the grading search itself, since D-02
guarantees SF-best is always in the candidate set), `esAfter` is a given candidate's expected
score. No new sigmoid or threshold logic is needed anywhere in this pipeline.

### Anti-Patterns to Avoid

- **Reusing `FlawSeverity` for the new 5-bucket type:** `FlawSeverity` (`types/library.ts`) is a
  cross-stack contract mirroring the backend's literal severity enum (`inaccuracy | mistake |
  blunder`) ‚Äî it is consumed by badges, backend-shaped API responses, and glyph maps elsewhere.
  Adding `'best' | 'good'` to it would corrupt that contract. Define a NEW, frontend-only
  `MoveQuality` type (see Pattern 3) instead.
- **Keying grading info-lines by `multipv` index:** see Pattern 2 ‚Äî this silently produces wrong
  colors as the search deepens and ranks reorder.
- **Sharing one Worker/hook for both eval-bar and grading:** breaks SC3's isolation requirement and
  risks the FLAWCHESS-7V "stop/go race ‚Üí RuntimeError: unreachable" pitfall already fixed once in
  `useStockfishEngine` ‚Äî don't reintroduce that class of bug by cramming two search modes into one
  state machine.
- **Computing the candidate set from `perElo`'s peak-across-all-ELO** (what `capMovesByPeak` does
  today): D-02 requires cumulative mass **at the selected ELO**, not peak probability across the
  whole ladder ‚Äî these are different rankings and produce different line sets. `capMovesByPeak`
  must be REPLACED (or heavily rewritten), not reused as-is, for the candidate-selection role
  (though the underlying `perElo` curve is still what feeds each line's shape across the x-axis).

## Common Pitfalls

### Pitfall 1: Treating `multipv` index as move identity in a searchmoves-restricted search
**What goes wrong:** Colors flicker or attach to the wrong SAN as the search deepens.
**Why it happens:** `useStockfishEngine`'s existing `pvMapRef` is keyed by `multipv` (1..N), which
is a stable simplification ONLY because with an unrestricted search + MultiPV=2, "the current best
line" and "the current second-best line" are the relevant concepts ‚Äî not "what did move X score".
Grading needs the latter.
**How to avoid:** Key the grading map by `pv[0]` (see Pattern 2).
**Warning signs:** A move's color visibly jumps to a DIFFERENT move's color between two depth
updates without the move itself changing rank in a sensible way ‚Äî a sign the map key is wrong.

### Pitfall 2: Re-running the full grading search on every ELO-slider tick
**What goes wrong:** Dragging the ELO selector feels janky/laggy, or worse, spams `stop`+`go` at
the grading worker, hitting the FLAWCHESS-7V race class of bug.
**Why it happens:** The candidate SET (which SANs are displayed) is ELO-dependent (D-02), but the
GRADE (eval per SAN) is position-only, per the seed's explicit note ("Stockfish grades are
position-only (ELO-independent) and cacheable per move; only the displayed subset changes with the
ELO slider"). A naive implementation re-derives `shownSans` on every ELO change and re-sends a
fresh `go` for it.
**How to avoid:** Cache grades per FEN (`Map<san, MoveGrade>`, board-session-scoped FIFO, mirrors
`useMaiaEngine`'s `MAIA_CACHE_MAX`/`cacheResult` pattern). On an ELO-drag-triggered candidate-set
recompute, diff the new `shownSans` against the cache's already-graded keys for this FEN; only
issue a new `go` (with `searchmoves` = the UNION of previously-graded + newly-needed SANs) if the
diff is non-empty. Debounce the ELO-drag-driven diff check the same way `RAPID_STEP_DEBOUNCE_MS`
debounces FEN changes, so a held-drag gesture coalesces into one search, not one per tick.
**Warning signs:** Grading-worker `postMessage` call count scales with ELO-slider drag events
rather than with distinct board positions visited.

### Pitfall 3: Assuming the restricted search's "best" line matches the primary engine's `bestSan`
**What goes wrong:** The primary `useStockfishEngine` (MultiPV=2, unrestricted, movetime 1500ms)
and the grading worker (MultiPV=N, searchmoves-restricted, different depth budget) can, at shallow
depths or in razor-sharp positions, report *slightly* different top moves or evals for the "same"
position ‚Äî because search budgets, node counts, and even move-ordering heuristics differ between
the two independent searches.
**Why it happens:** They are two separate engine processes with separate `go` parameters; nothing
guarantees byte-identical convergence, especially before both reach a stable/settled depth.
**How to avoid:** Per D-02, the candidate set already forces `bestSan` (from the PRIMARY engine)
into `searchmoves`, so the grading search evaluates it directly ‚Äî use the grading search's OWN
best-scoring candidate (Pattern 3's `bestEs`/`bestSan`) as the "best" bucket and esBefore baseline,
not the primary engine's `bestSan` string reused as a lookup key into the grading map. They will
usually agree, but treating the grading search as self-consistent (its own internal ranking is
the source of truth for classifying ITS OWN evals) avoids any cross-search disagreement leaking
into a wrong "best" bucket assignment.
**Warning signs:** The move visually marked "best" (thick stroke, from the primary engine's
`bestSan` prop) doesn't get the dark-green "best" color ‚Äî it's graded as "good" instead. This is
NOT necessarily a bug (the two searches can genuinely disagree at different depths/budgets) but
should be called out during UAT so it isn't mistaken for a wiring defect.

### Pitfall 4: Triple-worker resource pressure on mobile Safari
**What goes wrong:** Slow first paint, dropped frames, or Safari's WKWebView Worker/WASM memory
budget getting tight when the primary Stockfish worker, the NEW grading Stockfish worker, and the
Maia onnxruntime-web worker are all alive simultaneously on every analysis-board visit.
**Why it happens:** All three are separate OS threads each hosting a nontrivial WASM/ONNX runtime;
mobile Safari has historically been the tightest memory/CPU environment among the app's supported
browsers (per the existing D-3 "single-thread WASM only, no COOP/COEP" constraint from Phase 136,
already chosen specifically because mobile Safari doesn't support the SharedArrayBuffer-based
multi-thread build).
**How to avoid:** SC4 explicitly calls for a measured latency check + graceful degradation path
(cap candidate count, lower grading depth, or debounce grading behind the primary eval) ‚Äî build the
grading pass with those three knobs as named constants from day one (not as an afterthought), and
make the hands-on mobile Safari measurement a required plan-time task, not a "trust it'll be fine"
assumption.
**Warning signs:** Real-device (not desktop-Chrome-devtools-throttled) testing shows first-shallow-
grade latency exceeding roughly 1-2 seconds after Maia is ready, or dropped/stalled `info` streaming.

### Pitfall 5: Colliding grading-worker commands with the FLAWCHESS-7V stop/go race
**What goes wrong:** Rapid-fire board navigation (arrow-key auto-repeat) sends `stop` then
immediately `position`+`go` to the grading worker before the stale `bestmove` response for the
previous `go` arrives, trapping the single-thread WASM engine in `RuntimeError: unreachable`
(exactly the bug already fixed once in `useStockfishEngine`, see its `stateRef.current ===
'stopping'` early-return branch and its code comment referencing FLAWCHESS-7V).
**Why it happens:** Any second worker built without carrying over that exact serialization
discipline reintroduces the class of bug the primary engine already had to fix.
**How to avoid:** Copy `useStockfishEngine`'s `analyze()`/`handleLine()` state machine shape
(`idle`/`thinking`/`stopping`, `stopPendingRef`, deferred re-`go` on the stale `bestmove`) into the
grading hook verbatim ‚Äî don't re-derive it from scratch.
**Warning signs:** Console errors reading `RuntimeError: unreachable`, or the grading worker
appearing to silently stop responding after a burst of rapid navigation.

import { Chess } from 'chess.js';

/** Convert a SAN move (relative to `fen`) into its UCI string (e2e4, e7e8q, ...), or null if illegal. */
export function sanToUci(fen: string, san: string): string | null {
  try {
    const chess = new Chess(fen);
    const move = chess.move(san);
    return `${move.from}${move.to}${move.promotion ?? ''}`;
  } catch {
    return null;
  }
}

### Candidate-set selection (D-02/D-06/D-07) ‚Äî replaces `capMovesByPeak` for this surface
```typescript
// NEW ‚Äî frontend/src/lib/moveQuality.ts (or a sibling file); unit-testable pure function.
// Source: SEED-083 D-2 (maiachess's 0.95 cumulative-mass rule) + 151.1-CONTEXT.md D-06/D-07.
const CUMULATIVE_MASS_THRESHOLD = 0.95;
const CANDIDATE_HARD_CAP = 5;

function nearestByElo(perElo: MoveCurvePoint[], target: number): MoveCurvePoint | undefined {
  // mirror useMaiaEngine's nearestByElo ‚Äî could be extracted/shared instead of duplicated
  return perElo.reduce<MoveCurvePoint | undefined>((closest, p) => {
    if (closest === undefined) return p;
    return Math.abs(p.elo - target) < Math.abs(closest.elo - target) ? p : closest;
  }, undefined);
}

export function selectCandidatesByMass(
  perElo: MoveCurvePoint[],
  selectedElo: number,
  playedSan: string | null,
  bestSan: string | null,
): string[] {
  const rung = nearestByElo(perElo, selectedElo);
  if (rung === undefined) return [];
  const sorted = Object.entries(rung.moveProbabilities).sort((a, b) => b[1] - a[1]);
  const massSet: string[] = [];
  let cumulative = 0;
  for (const [san, prob] of sorted) {
    if (cumulative >= CUMULATIVE_MASS_THRESHOLD) break;
    massSet.push(san);
    cumulative += prob;
  }
  const top5 = massSet.slice(0, CANDIDATE_HARD_CAP);
  const keep = new Set(top5);
  if (bestSan != null) keep.add(bestSan);
  if (playedSan != null) keep.add(playedSan); // D-07: always show played, even below the cut
  // Preserve probability-descending order for stable stacking/legend order.
  return sorted.map(([san]) => san).filter((san) => keep.has(san));
}
```

## Open Questions

1. **What exact depth/movetime budget keeps grading "streaming-live" on mobile Safari without
   starving the primary MultiPV=2 eval or the Maia inference?**
   - What we know: the primary engine uses `movetime 1500 nodes 2000000`; CSSLab's own reference
     targets depth 18 (min depth 12), depth-bounded with no movetime/nodes cap, for likely a
     SINGLE search config across FEWER lines than FlawChess's ~5-6 hard cap.
   - What's unclear: whether depth 18 is realistic on mobile Safari with MultiPV up to ~6 running
     as a SECOND concurrent worker (vs. desktop, where CSSLab's target audience skews toward).
   - Recommendation: start with a conservative target depth (e.g. 14) + a wall-clock safety
     `movetime` cap (e.g. 2500-3000ms) as named constants, and treat both as an SC4 tuning knob to
     adjust after the mandatory hands-on latency measurement (mirrors the "measure and judge" VALID-
     01 posture already established for Maia in Phase 151).

2. **Should the grading worker be created eagerly (mounted alongside the primary engine and Maia
   on every analysis-board visit) or lazily (only spun up once Maia's `perElo` for the current
   position is ready, since grading needs the candidate SAN list first anyway)?**
   - What we know: Maia's candidate probabilities must exist before `searchmoves` can be built, so
     grading is inherently sequenced AFTER (or in parallel awaiting) the Maia result for a given
     FEN, unlike the primary engine which starts immediately on FEN change.
   - What's unclear: whether creating the Worker itself (WASM instantiation, ~10s of ms to ~100+ms)
     ahead of having anything to grade wastes resources vs. paying that startup cost lazily on
     first need.
   - Recommendation: create the Worker instance once per component mount (cheap, avoids re-paying
     WASM instantiation on every position), but only send `position`/`go` once a candidate SAN list
     is available ‚Äî i.e. mirror `useStockfishEngine`'s existing `enabled` gate pattern, just gate
     the `go` on "Maia ready for this FEN" rather than gating Worker creation itself.
