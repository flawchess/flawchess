# Phase 151.1: Stockfish-graded Maia moves on the Moves-by-Rating chart - Context

**Gathered:** 2026-07-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Grade each shown Maia candidate move with the **existing client-side Stockfish** and recolor the
**"Moves by Rating" chart** lines (+ their end-of-line SAN labels) by **move quality** — so the
chart surfaces the *human trap* (a popular move Stockfish grades as a mistake), not just which
moves humans play. Replace Phase 151's fixed top-6-by-peak cap with a "moves people actually play"
candidate set. Browser-only, ephemeral, **zero DB writes / zero backend / no schema change** —
same posture as Phase 151.

**In scope:** quality-coloring of the chart lines/labels; the new candidate-selection rule; the
Stockfish grading pass (acquisition + progressive streaming); tooltip quality detail.

**Explicitly OUT of scope:** any move-list table (maiachess-style) — chart-only (D-1 below); any
DB write / persistence of grades; the Phase 152 flaw-overlay verdict banner + Maia-WDL
practical-severity reframe; changing FlawChess's severity thresholds; touching the engine-card
variation tree behavior.

</domain>

<decisions>
## Implementation Decisions

### Pre-locked by SEED-083 (carried forward — the /gsd-explore session that promoted this phase)
- **D-01 — Surface: chart-only.** No new move-list table. Color encodes **quality**; the
  end-of-line SAN labels carry **identity**; played/best emphasis is **stroke width** (independent
  of color).
- **D-02 — Candidate set: Maia cumulative-probability ≥ 0.95 (at the selected ELO) ∪ {SF-best}**,
  replacing the top-6-by-peak cap (this supersedes SURF-03's default N≈6 rule for this surface).
- **D-03 — Colors: 5 buckets on FlawChess's OWN existing thresholds** (`liveFlaw.ts` /
  `flawThresholds.ts`), do NOT copy maiachess's coarser 3-band scheme:
  dark-green = SF-best · light-green = good (drop < 0.05) · yellow = inaccuracy (0.05–0.10) ·
  orange = mistake (0.10–0.15) · red = blunder (≥ 0.15). Only "best" + "good" are new on top of
  the existing 3 severity bands. Theme colors live in `theme.ts`.
- **D-04 — Eval acquisition: ONE Stockfish root search** with UCI `searchmoves = candidates` +
  `MultiPV = |candidates|` — not N child-position searches, no coverage gap, no per-move floor.

### Grading UX (discussed this session)
- **D-05 — Progressive / streaming grading.** Lines appear immediately when Maia is ready (neutral
  color); the moment the first **shallow** Stockfish eval arrives, each move is colored by its
  current best-available grade; colors then **keep refining as the search deepens** — exactly like
  `useStockfishEngine` already streams and updates the 2 best lines on each `info depth …` line. No
  wait-for-final, no blank hold. A new board navigation **cancels** the in-flight grade (existing
  stale-guard pattern). This is also how maiachess's `streamEvaluations` staged-deepening behaves.

### Line-count cap (discussed this session)
- **D-06 — Hard cap at ~5.** Take the 0.95-mass set, keep the **top-5 by probability**, always
  ∪ {SF-best} → ~5–6 lines max. Bounds legibility at the ~360px left-column width **and** bounds
  grading cost to ≤ ~6 moves per navigation. Sharp positions naturally show fewer.

### Played-move handling (discussed this session)
- **D-07 — Always show + grade the played move** (game mode) even when it falls below the
  top-5/0.95 cut — the "you played a red move here" case is the highest-value coaching signal.
  Played **and** SF-best keep **thicker stroke width** so they stay findable regardless of quality
  color. Honors SURF-02 (emphasize played + engine-best). Free play has no played move — fine.

### Tooltip / label detail (discussed this session)
- **D-08 — Quality word + eval in the tooltip; chart face unchanged.** Color stays the
  at-a-glance signal. The hover tooltip adds, per move, its **classification word** (best / good /
  inaccuracy / mistake / blunder) and its **eval**. End-of-line SAN labels stay colored-SAN only
  (no extra text). The per-move eval shown is the **white-POV eval Stockfish assigns to that move**
  (from the `searchmoves`/MultiPV info lines); the quality word comes from `classifyLiveSeverity`
  on the **mover's expected-score drop vs the best available move** (the same
  `evalToExpectedScore` → drop → band pipeline the app already uses). Prob% stays in the tooltip.

### Claude's Discretion (planner / researcher to resolve)
- **Engine contention / second analysis mode** — the existing `useStockfishEngine` drives the eval
  bar + engine card at `MultiPV=2`. The grading pass needs a *different* config (`searchmoves` +
  `MultiPV = |candidates|`) and would contend with the single-thread WASM worker if run
  simultaneously. Decide: a **second dedicated Stockfish worker** for grading vs. time-sharing /
  a mode-switch on the existing engine. Must NOT disturb the eval-bar / engine-card variation tree.
- **Grade cache scope** — Stockfish grades are **position-only** (ELO-independent) and keyable per
  (position, move); only the *displayed subset* changes when the ELO slider drags (D-02 candidate
  set is ELO-dependent). An ephemeral board-session cache (mirror `LIVE_EVAL_CACHE_MAX` /
  `useMaiaEngine`'s 256-entry FIFO) avoids re-grading on ELO drags and back-nav. No persistence.
- **Grading search depth / stop condition** — target depth vs. movetime vs. node budget for the
  grading search, and whether it reuses `useStockfishEngine`'s existing budget; tune for the ≤~6
  candidate case so streaming feels live without starving the primary eval.
- **Exact 5-color theme tokens** in `theme.ts` (map to the existing WDL / severity palette;
  reuse `severityGlyph.ts` / flaw-color conventions where they exist for consistency).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `liveFlaw.ts` (`classifyLiveSeverity` + `evalToExpectedScore`) — the entire grading pipeline
  already exists client-side; 151.1 adds only "best" (SF #1) and "good" (clean, non-best) buckets.
- `useStockfishEngine.ts` — already streams MultiPV `info depth` updates with a stale-guard and
  tab-hide pause; the grading pass is a variant (`searchmoves` + `MultiPV = |candidates|`).
- `MovesByRatingChart.tsx` — `capMovesByPeak` / `colorForSan` / `pivotRows` / tooltip are the exact
  seams to change (new cap rule in D-06; quality coloring replacing identity in `colorForSan`;
  tooltip detail in D-08). Stroke-width emphasis (`EMPHASIZED_STROKE_WIDTH`) already exists → reuse
  for D-07.

### Established Patterns
- Client WASM engine in a Web Worker, lazy-loaded, tab-hide paused, ephemeral session cache
  (Stockfish + Maia both do this). Grading rides the same pattern; nothing persists.
- Colors with semantic meaning live in `theme.ts` (never hard-coded in components).
- `data-testid` on interactive/chart containers; `text-sm` floor (tooltip may use `text-xs` per the
  hover-tooltip exception); apply changes to desktop + mobile Human tab surfaces alike.

### Integration Points
- The grading engine plugs into `Analysis.tsx` alongside the existing `useStockfishEngine`
  instance (eval bar + engine card) — resolve the single-thread-worker contention (Claude's
  Discretion above) without disturbing those.
- `MovesByRatingChart` receives `perElo` (candidate probabilities) + needs a new per-move
  grade map (SAN → {evalCp, severity}) streamed from the grading engine.

</code_context>
