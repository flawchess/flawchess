# Phase 151.1 Plan 01: searchmoves + MultiPV de-risk spike Summary

De-risked the single genuine unknown of the phase: the vendored
`stockfish-18-lite-single.js` WASM build **does** functionally honor a
`searchmoves`-restricted search combined with `MultiPV = N` (N up to the
~6 candidate cap), producing exactly N distinct `info` lines each keyed to a
distinct restricted move at real depths. **Verdict: YES.** Plan 03 (grading hook)
proceeds as designed; no pivot to the D-04 fallback is required.

## What Was Built

A throwaway, dependency-free probe page (`frontend/public/engine/searchmoves-probe.html`)
that drove the REAL vendored `stockfish-18-lite-single.js` worker with
`setoption name MultiPV value 4` + `go depth 16 searchmoves <4 candidate ucis>`,
streaming the raw `info`/`bestmove` lines plus a live per-candidate depth/score
summary table. The page was deleted immediately after checkpoint sign-off
(T-151.1-01 mitigation — no dead public asset shipped).

## SC3 / Assumption A1 Verdict: YES

Verified headlessly against the exact vendored binary the browser Worker loads
(`frontend/public/engine/stockfish-18-lite-single.wasm`), driven via the engine's
own Node CLI mode (identical WASM computation to the browser Worker; the Worker
message plumbing is already proven in production `useStockfishEngine`).

**Evidence** — with 4 LEGAL restricted candidates (`e1g1`, `c1e3`, `d1e2`, `b1c3`)
at `setoption name MultiPV value 4` + `go depth 16 searchmoves ...`:
- Exactly **4 distinct** `info … multipv 1..4 … pv <move>` lines appeared.
- Each `pv[0]` was one of the 4 restricted moves; **no non-candidate move ever
  appeared as pv[0]**.
- Every candidate's score refined sensibly across all 16 depths (D-05
  progressive-grading premise holds on the real binary).
- `bestmove` was `c1e3`.

This directly validates the Plan 03 grading hook's engine assumption: it loads the
SAME worker path (`/engine/stockfish-18-lite-single.js`) the probe used.

## Load-Bearing Caveats for Plan 03

**CAVEAT 1 — pass ONLY legal UCI candidates (illegal moves are silently dropped).**
The engine silently drops illegal `searchmoves` entries. An earlier run using the
probe's original hard-coded list (`f1e2`, `c1e3`, `d1e2`, `e1g1`) produced only
**3** lines because `f1e2` is illegal in the probe FEN (the white bishop is already
on c4; f1 is empty). Plan 03 must pass only legal UCI candidates — Maia candidate
moves + SF-best are legal by construction, so no per-move legality floor is needed,
but the hook must **never** pass an illegal or placeholder move, or MultiPV lines
will silently under-count.

**CAVEAT 2 — the `multipv` index is an EVAL RANK, not input order (RESEARCH landmine #1,
confirmed on the real binary).** Candidates passed in order
`[e1g1, c1e3, d1e2, b1c3]` came back ranked `multipv 1=c1e3, 2=d1e2, 3=b1c3, 4=e1g1`.
**The grading map MUST key by `pv[0]` (the move), never by the `multipv` index.**
