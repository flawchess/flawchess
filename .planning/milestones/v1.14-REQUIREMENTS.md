# Requirements: FlawChess v1.14 — Score-Based Opening Insights

**Milestone goal:** Migrate Opening Insights and Move Explorer color coding from loss-rate to chess score `(W + 0.5·D)/N`, gate findings on effect size against a 0.50 pivot, and annotate them with low/medium/high confidence badges. Replaces today's "Major/Minor Weakness" framing with a calibrated discovery surface that holds up under future LLM narration.

**Source:** SEED-007 (Option A only — Wilson on score) + SEED-008 (label reframe), folded together so the math, the labels, and the calibration cue ship under one consistent framing. Design decisions captured in `.planning/notes/opening-insights-v1.14-design.md`.

## v1.14 Requirements

### Backend — score metric and confidence annotation (INSIGHT-SCORE)

- [x] **INSIGHT-SCORE-01** — Score `(W + 0.5·D)/N` is now the canonical metric in `opening_insights_service.py` and `openings_repository.py`. Raw `w/d/l/n_games` retained as the literal-data display.
- [x] **INSIGHT-SCORE-02** — Pivot held at 0.50 (no user-baseline shrinkage). Rationale captured in the v1.14 design note.
- [x] **INSIGHT-SCORE-03** — Effect-size gate shipped with strict `≤`/`≥` boundaries: minor at 0.45/0.55, major at 0.40/0.60. Constants in `opening_insights_constants.py`.
- [x] **INSIGHT-SCORE-04** — Trinomial Wald CI shipped (variance `(W + 0.25·D)/N − score²`); pure-Python, no scipy. Half-width buckets `≤ 0.10 → high`, `≤ 0.20 → medium`, else `low` — strict `≤` boundaries. Pivoted from Wilson per Phase 75 D-05.
- [x] **INSIGHT-SCORE-05** — `MIN_GAMES_PER_CANDIDATE` dropped 20 → 10. Low-confidence candidates surface with the `(low)` badge as discovery signal.
- [x] **INSIGHT-SCORE-06** — `OpeningInsightFinding` payload now exposes `confidence: "low" | "medium" | "high"` + `p_value: float` (two-sided Z-test vs 0.50). `loss_rate` / `win_rate` removed cleanly. `severity` retained.
- [x] **INSIGHT-SCORE-07** — `tests/services/test_opening_insights_arrow_consistency.py` updated; threshold lock-step between backend classification and `frontend/src/lib/arrowColor.ts` enforced in CI.

### Frontend — score-based coloring, confidence badges, label reframe (INSIGHT-UI)

- [x] **INSIGHT-UI-01** — `arrowColor.ts` migrated to score (effect-size only). Same thresholds as backend.
- [x] **INSIGHT-UI-02** — Move Explorer moves-list row tint by score; WDL bars unchanged.
- [x] **INSIGHT-UI-03** — Three-level `(low) / (medium) / (high)` indicator shipped on moves-list rows. High-confidence unmarked (default).
- [x] **INSIGHT-UI-04** — **DESCOPED 2026-04-28** per Phase 76 D-04. Severity word never appears as user-facing text (only drives border color). Confidence badge + `(confidence DESC, |score - 0.50| DESC)` sort carry the SEED-008 calibration intent without title rewrites.
- [x] **INSIGHT-UI-05** — Confidence badge shipped on `OpeningFindingCard.tsx` (`OpeningFindingCard` shipped score-based prose with confidence indicator + tooltip-grade p-value, plus `UNRELIABLE_OPACITY` mute when `n_games < 10` OR `confidence === 'low'`).
- [x] **INSIGHT-UI-06** — Four `InfoPopover` triggers on `OpeningInsightsBlock` section headers cover the score / sample-size / confidence framing. Reuses existing `InfoPopover` component.
- [x] **INSIGHT-UI-07** — Mobile parity verified at 375px (no horizontal scroll, ≥ 44px touch targets, all interactive elements have `data-testid`).

**Post-Phase-76 inline hotfix (PR #71, 2026-04-28):** Force grey arrow + skip row tint when `confidence === 'low'`. Strengthens the at-a-glance read: low-confidence findings still surface in the table (with the badge) but don't visually claim authority on the board. Beyond the original requirements; bookkept as a v1.14 decision.

## Future Requirements (deferred)

- **LLM narration of opening insights** — was previously deferred behind v1.13. v1.14 lands the calibrated data plumbing (effect size + confidence) that future LLM narration would consume; the LLM layer itself remains a separate future seed.
- **Population-relative weakness signals** — gated on full benchmark ingest (SEED-006). Not part of v1.14 because the design explicitly rejects user/population baselines (see Out of Scope below).

## Out of Scope

- **User-baseline / Bayesian shrinkage thresholds** — explicitly rejected. Lichess and chess.com matchmaking already center users near 50% score; the opponent-strength filter handles users whose baseline drifts. User-baseline would re-do work the platform and the filter UI already do, *and* would break user-agnostic arrow color reads on shared positions. See `.planning/notes/opening-insights-v1.14-design.md` § "What we rejected, and why".
- **Showing raw p-values on cards** — the low/medium/high badge is the user-facing surface; the Wilson math underneath derives the badge but is never exposed.
- **Confidence cue on board arrows (opacity/dashing)** — the board reads at-a-glance via effect-size color only; confidence lives on cards and moves-list rows where text can carry the badge.
- **Dropping the WDL bar from the move explorer** — the bar remains the literal-data display; the score-based color is the *threshold/classification* layer on top of it.
- **Changing the dedupe rules** in `_dedupe_continuations` and `_dedupe_within_section` — working as intended, out of scope.
- **Engine-eval-based weakness detection** — same rationale as v1.13.
