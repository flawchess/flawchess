## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Untrusted user input (filter state) crosses to POST /api/insights/openings |
| client→React Router | Deep-link click triggers client-side navigation; no untrusted data crosses a security boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-71-01 | Tampering | POST /api/insights/openings body | mitigate | Pydantic schema with `extra="forbid"` (already enforced in Phase 70 — see app/schemas/opening_insights.py:22). The hook only sends fields whitelisted in the schema. |
| T-71-02 | Information disclosure | OpeningInsightFinding payload | accept | All findings are derived from the authenticated user's own games. No cross-user leakage possible (router uses `current_active_user`). |
| T-71-03 | Denial of service | Heavy filter combinations / repeated refetches | mitigate | TanStack Query staleTime (30s) prevents thrashing. The hook query key normalizes `recency='all' → null`, preventing duplicate distinct keys. Backend already index-scannable per Phase 70. |
| T-71-04 | Repudiation | Deep-link navigation | accept | Pure client-side state mutation; no audit log requirement. |
| T-71-05 | Spoofing / XSS | Finding fields rendered as text in cards | mitigate | All finding fields come from a Pydantic-validated server response; React's default JSX text rendering escapes HTML. No `dangerouslySetInnerHTML` is used anywhere in the new components. |
| T-71-06 | Elevation of privilege | Filter forwarding | accept | Filters are user's own; no role/scope escalation possible. |
</threat_model>
