## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01:** Replace the triple PGN parse with a single unified function that walks the mainline once, yielding (hashes, classification, eval, clock, move_san) per ply. `hashes_for_game` and the separate classification/eval loops in `_flush_batch` are replaced by this new function.

**D-02:** `move_count` is derived from the unified function output (`(len(hash_tuples) - 1 + 1) // 2`), and `result_fen` is already returned. The third PGN parse for these is eliminated entirely.

**D-03:** Eliminate the SELECT for `Game.pgn` after `bulk_insert_games`. Instead, build a `platform_game_id -> PGN` lookup dict from the batch, then do a lightweight SELECT for `(id, platform_game_id)` to map new game IDs back to their PGNs.

**D-04:** Replace per-game UPDATE for `move_count`/`result_fen` with a bulk UPDATE using CASE expressions — one query per batch instead of 10-30 individual UPDATEs.

**D-05:** Increase `_BATCH_SIZE` from 10 to 25-30. The position INSERT chunking (1,700 rows/chunk) already prevents oversized queries. Larger batches mean fewer DB commits and round-trips.

**D-06:** Focus on processing speed only — no concurrent archive fetching.

**D-07:** No multiprocessing or ProcessPoolExecutor for CPU-bound work.

**D-08:** Re-sync speed is not a concern. Optimization targets initial imports of large game libraries (1,000+ games).

## Summary

The import pipeline currently processes games at ~20 games/sec (chess.com) and ~16-24 games/sec (lichess) for initial imports. Network I/O accounts for only ~30s of a 432s import run — the bottleneck is CPU-bound processing in `_flush_batch`. Three distinct bottlenecks were identified and confirmed by code inspection:

**Bottleneck 1: Triple PGN parse.** Each game is parsed three times by `chess.pgn.read_game()`. Benchmarking confirms triple parsing takes 3x longer than single parsing (0.453ms vs 0.149ms per game at 10,000 game scale). `hashes_for_game` already walks the mainline; `classify_position` and eval extraction run in a second pass; `move_count`/`result_fen` trigger a third pass. All four data needs can be served in one mainline walk.

**Bottleneck 2: Unnecessary DB round-trip.** After `bulk_insert_games` returns new IDs, the code executes `SELECT Game.id, Game.pgn WHERE id IN (new_ids)` — redundantly fetching PGN text that was already available in the in-memory batch. This is both a latency cost (one extra round-trip per batch) and a data-transfer cost (PGN text can be 1-10KB per game).

**Bottleneck 3: Per-game UPDATE statements.** Each game gets an individual `UPDATE games SET move_count=X, result_fen=Y WHERE id=Z` — 10 round-trips per batch instead of 1. SQLAlchemy 2.x `case()` with the simple CASE form (`case(dict, value=Game.id)`) supports this as a single query.

**Batch size increase from 10 to 25-30** reduces the number of DB commits proportionally (2.5-3x fewer). Memory analysis confirms batch_size=30 produces ~2MB of in-memory position rows — safe for the 7.6GB+2GB-swap production server.

**Primary recommendation:** Implement the unified `process_game_pgn()` function, eliminate the redundant SELECT, replace per-game UPDATEs with one CASE-expression UPDATE per batch, and set `_BATCH_SIZE = 28`. These changes together target ~2x throughput improvement with no architectural changes.

### SQLAlchemy 2.x CASE Expression (verified)

The `case()` function in SQLAlchemy 2.x supports the simple CASE form when `value=` is provided:

```python
from sqlalchemy import case, update
from app.models.game import Game

case(
    {1: 45, 2: 32},          # id -> move_count mapping
    value=Game.id,            # CASE game.id WHEN 1 THEN 45 WHEN 2 THEN 32 END
)
```

Produces: `CASE games.id WHEN %(param_1)s THEN %(param_2)s WHEN %(param_3)s THEN %(param_4)s END`

This is valid SQLAlchemy 2.x syntax (verified by running against the installed version). The 1.x dict-as-first-arg form still works in 2.x for the simple CASE pattern.

## Architecture Patterns

### Pattern 1: Unified PGN Processing Function

**What:** Replace `hashes_for_game` with a new function (e.g., `process_game_pgn`) that walks the mainline exactly once and returns all per-ply data plus game-level data.

**When to use:** Called once per game in `_flush_batch`, replacing three separate calls.

**Ply loop order (critical — all pre-push except eval):**

```python
# Source: verified from current import_service.py + zobrist.py logic
for ply, node in enumerate(nodes):
    # 1. Pre-push: hashes from current board state
    white_hash, black_hash, full_hash = compute_hashes(board)
    # 2. Pre-push: classification from current board state
    classification = classify_position(board)
    # 3. Pre-push: move SAN (board must be in pre-move state)
    move_san = board.san(node.move)
    # 4. On node: clock annotation
    clock_seconds = node.clock()
    # 5. On node: eval annotation (eval of position AFTER this move)
    pov = node.eval()
    eval_cp, eval_mate = _extract_eval(pov)
    # 6. Advance board
    board.push(node.move)
    # Yield or accumulate ply data here
```

After loop: compute final position hashes + result_fen; derive `move_count = (len(nodes) + 1) // 2`.

**Return type:** A TypedDict or dataclass (consistent with CLAUDE.md: use TypedDicts for internal structured data).

```python
# Proposed return from process_game_pgn():
class PlyData(TypedDict):
    ply: int
    white_hash: int
    black_hash: int
    full_hash: int
    move_san: str | None
    clock_seconds: float | None
    eval_cp: int | None
    eval_mate: int | None
    material_count: int
    material_signature: str
    material_imbalance: int
    has_opposite_color_bishops: bool
    piece_count: int
    backrank_sparse: bool
    mixedness: int

class GameProcessingResult(TypedDict):
    plies: list[PlyData]
    result_fen: str | None
    move_count: int
```

**Backward compatibility for `hashes_for_game`:** Tests call `hashes_for_game` directly (17+ test cases in `test_zobrist.py`). Options:
- Keep `hashes_for_game` as a thin wrapper that calls `process_game_pgn` and extracts the hash/move_san/clock fields — zero test changes required.
- Or update all tests to call the new function — more work, no functional advantage.

The thin wrapper approach is recommended (zero test churn, clean public API).

**`compute_hashes` stays unchanged** — it is called independently by analysis endpoints and must remain a standalone function.

### Pattern 2: platform_game_id Lookup for D-03

**What:** Build a lookup dict from the batch before inserting, then after `bulk_insert_games`, do a lightweight SELECT for `(id, platform_game_id)` only. Look up PGN via the in-memory dict.

```python
# Build before bulk_insert:
pgn_by_platform_id: dict[str, str] = {
    (g.platform_game_id if isinstance(g, NormalizedGame) else g["platform_game_id"]): 
    (g.pgn if isinstance(g, NormalizedGame) else g.get("pgn", ""))
    for g in batch
}

# After bulk_insert_games returns new_ids:
result = await session.execute(
    select(Game.id, Game.platform_game_id).where(Game.id.in_(new_game_ids))
)
for game_id, platform_game_id in result.fetchall():
    pgn = pgn_by_platform_id.get(platform_game_id, "")
    if pgn:
        # process pgn...
```

**Why this is better than current code:** Current code does `SELECT Game.id, Game.pgn` which transfers large PGN text back from the DB. New code selects only `platform_game_id` (small string). The PGN was already in Python memory — no need to round-trip it.

### Pattern 3: Bulk UPDATE with CASE Expressions for D-04

**What:** Accumulate `{game_id: move_count}` and `{game_id: result_fen}` during the ply loop, then issue a single UPDATE per batch.

```python
# Accumulate during processing:
move_counts: dict[int, int] = {}
result_fens: dict[int, str | None] = {}

# After processing all games in batch:
if move_counts:
    await session.execute(
        update(Game)
        .where(Game.id.in_(list(move_counts.keys())))
        .values(
            move_count=case(move_counts, value=Game.id),
            result_fen=case(
                {k: v for k, v in result_fens.items() if v is not None},
                value=Game.id,
                else_=None,
            ),
        )
    )
```

**Edge case:** `case()` with an empty dict will raise. Guard with `if move_counts:` before executing. Also, `result_fen` can be `None` — filter None values from the CASE dict and use `else_=None` for the fallback.

**asyncpg parameter limit check:** For batch_size=28, a bulk CASE UPDATE generates 4 params per game (WHEN key, THEN value for each of two columns) × 28 games = 112 params, well under the 32,767 limit. Plus the IN clause adds 28 params = 140 total. Safe.

### Pattern 4: Batch Size Selection

**Selected value: `_BATCH_SIZE = 28`**

Rationale:
- The 25-30 range is locked in D-05
- 28 is a nice round number that stays comfortably within the range
- Memory: 28 games × 80 rows = 2,240 position rows × ~800 bytes (Python dict) ≈ 1.8MB — safe for production
- chunking: 2,240 rows splits to one 1,700-row chunk + one 540-row chunk (2 DB roundtrips for positions, same as before)
- Previous OOM was at batch_size=50 **before** chunking; with chunking the constraint is memory, not query size

### Recommended `_flush_batch` Structure After Refactor

```
_flush_batch(session, batch, user_id):
  1. Build pgn_by_platform_id dict from batch             [local, O(N)]
  2. bulk_insert_games(session, game_dicts) -> new_ids    [1 DB round-trip]
  3. if not new_ids: return 0
  4. SELECT (id, platform_game_id) WHERE id IN (new_ids)  [1 DB round-trip]  
  5. For each (game_id, platform_game_id):
       pgn = pgn_by_platform_id[platform_game_id]
       result = process_game_pgn(pgn)                     [1 PGN parse/game]
       Accumulate position_rows, move_counts, result_fens
  6. bulk_insert_positions(session, position_rows)        [N/1700 chunks]
  7. Bulk UPDATE move_count/result_fen via CASE           [1 DB round-trip]
  8. session.commit()                                     [1 round-trip]
  9. return len(new_ids)
```

**Round-trips per batch (new vs old):**
- Old: 1 (bulk_insert) + 1 (SELECT pgn) + N (per-game UPDATE) + 1 (commit) = 3+N
- New: 1 (bulk_insert) + 1 (SELECT id+platform_game_id) + ceil(rows/1700) (positions) + 1 (CASE UPDATE) + 1 (commit) = 4+ceil(rows/1700)
- For batch_size=28: old = 3+28=31, new = 4+2=6 (5.2x fewer round-trips)

## Common Pitfalls

### Pitfall 1: Board State Mismatch Between Hashes and Classification

**What goes wrong:** If `classify_position(board)` is called AFTER `board.push()` instead of before, classification data is for the NEXT position, not the current one. This would corrupt all position classifications silently.

**Why it happens:** The loop has both pre-push and post-push operations. Rearranging order breaks the invariant.

**How to avoid:** Enforce a strict per-ply order in `process_game_pgn`: always compute all pre-push values (hashes, classification, move_san) BEFORE calling `board.push(node.move)`.

**Warning signs:** Unit tests for `classify_position` on known positions would catch hash/classification ply mismatches.

### Pitfall 2: Empty CASE Dict Raises

**What goes wrong:** `case({}, value=Game.id)` raises a `CompileError` or similar because SQLAlchemy requires at least one WHEN clause.

**Why it happens:** If all games in a batch are duplicates, `new_ids` is empty and the loop never populates `move_counts`. A subsequent `update().values(move_count=case({}, value=Game.id))` would fail.

**How to avoid:** The existing early return `if not new_game_ids: return 0` already handles this. Additionally, guard the bulk UPDATE call: `if move_counts: await session.execute(...)`.

### Pitfall 3: result_fen = None in CASE Dict

**What goes wrong:** `case({1: None, 2: 'fen'}, value=Game.id)` may produce unexpected SQL or type errors because `None` is a Python null, not a SQL `NULL` literal.

**Why it happens:** `process_game_pgn` returns `result_fen=None` for empty/invalid PGNs. Including None values in the CASE dict can cause issues depending on SQLAlchemy version.

**How to avoid:** Filter the CASE dict to only include non-None result_fens. For games where result_fen is None, the column remains NULL (its default), which is correct behavior.

```python
fen_case_map = {gid: fen for gid, fen in result_fens.items() if fen is not None}
if fen_case_map:
    result_fen_case = case(fen_case_map, value=Game.id, else_=None)
```

### Pitfall 4: Test Mocks That Bypass process_game_pgn

**What goes wrong:** Tests in `test_import_service.py` mock `hashes_for_game` at the module level. After refactoring, if the new function name changes, those mocks stop working and the tests silently pass without testing the right code path.

**Why it happens:** Tests that mock `app.services.import_service.hashes_for_game` use string references. If `import_service.py` no longer imports `hashes_for_game` but instead uses `process_game_pgn`, the mock targets become stale.

**How to avoid:**
- If keeping `hashes_for_game` as a thin wrapper imported by `import_service.py`, existing mocks remain valid.
- If replacing the import with `process_game_pgn`, update all mock targets in `test_import_service.py`.
- Add a test that validates `process_game_pgn` output shape against known PGN input (verifies no regressions in data extraction).

### Pitfall 5: platform_game_id Dict Miss

**What goes wrong:** If a batch contains a duplicate `platform_game_id` (two NormalizedGame objects with the same ID), the dict comprehension silently drops one. The INSERT uses ON CONFLICT DO NOTHING so this is already expected — but the PGN lookup must use the same key.

**Why it happens:** The `pgn_by_platform_id` dict is built from the batch; the DB uniqueness constraint means only one of the duplicates is inserted. The returned `platform_game_id` from the SELECT will always be in the dict (since the inserted game came from the batch).

**How to avoid:** No special handling needed — the SELECT only returns IDs of newly inserted games, which all exist in the batch. The last-write-wins dict comprehension is safe because both duplicates would have the same PGN anyway (same game).

### Pitfall 6: ty Type Checking with TypedDict

**What goes wrong:** `ty` (the Astral type checker) runs in CI and blocks the build on type errors. TypedDicts with optional keys or heterogeneous value types can trigger errors.

**Why it happens:** `PlyData` TypedDict mixes `int`, `str | None`, `float | None`, `bool` values. Function signatures returning these must declare explicit types.

**How to avoid:**
- Use `TypedDict` from `typing` with explicit field types
- Annotate `process_game_pgn` return type explicitly: `-> GameProcessingResult | None`
- For None returns (empty/invalid PGN), return `None` rather than a partial structure
- Run `uv run ty check app/ tests/` before committing

## Code Examples

### Unified PGN Processing Loop (verified pattern)

```python
# Source: derived from zobrist.py:hashes_for_game + import_service.py:_flush_batch
# All pre-push operations before board.push(). Eval is on the node (post-move semantics).

def process_game_pgn(pgn_text: str) -> GameProcessingResult | None:
    try:
        game = chess.pgn.read_game(io.StringIO(pgn_text))
    except Exception:
        return None
    if game is None:
        return None

    nodes = list(game.mainline())
    if not nodes:
        return None

    board = game.board()
    plies: list[PlyData] = []

    for ply, node in enumerate(nodes):
        wh, bh, fh = compute_hashes(board)
        classification = classify_position(board)
        move_san: str = board.san(node.move)
        clock_seconds: float | None = node.clock()
        pov = node.eval()
        eval_cp: int | None = None
        eval_mate: int | None = None
        if pov is not None:
            w = pov.white()
            eval_cp = w.score(mate_score=None)
            eval_mate = w.mate()
        
        # Compute endgame_class here or leave to _flush_batch — either works
        board.push(node.move)
        plies.append(PlyData(ply=ply, white_hash=wh, ...))

    # Final position
    wh, bh, fh = compute_hashes(board)
    plies.append(PlyData(ply=len(nodes), white_hash=wh, ..., move_san=None, clock_seconds=None))

    result_fen = board.board_fen()
    move_count = (len(nodes) + 1) // 2
    return GameProcessingResult(plies=plies, result_fen=result_fen, move_count=move_count)
```

### hashes_for_game as Thin Wrapper (backward compat)

```python
# Source: existing zobrist.py API — wrapper preserves test compatibility
def hashes_for_game(
    pgn_text: str,
) -> tuple[list[tuple[int, int, int, int, str | None, float | None]], str | None]:
    """Thin wrapper around process_game_pgn for backward compatibility."""
    result = process_game_pgn(pgn_text)
    if result is None:
        return [], None
    hash_tuples = [
        (p["ply"], p["white_hash"], p["black_hash"], p["full_hash"], p["move_san"], p["clock_seconds"])
        for p in result["plies"]
    ]
    return hash_tuples, result["result_fen"]
```
