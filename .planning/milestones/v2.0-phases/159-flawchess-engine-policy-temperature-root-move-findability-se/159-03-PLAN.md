<objective>
Thread A of SEED-085: a Maia policy-temperature knob that flattens (T>1, more human fallibility) or sharpens (T<1, toward Stockfish) the user's-side move distribution, feeding both the search and the Thread B findability `P_you` so the two compose. This plan delivers the pure transform and the search-core wiring only; the UI slider is plan 159-04.

Per the research's central finding, the two threads compose with zero extra glue: temperature is applied before `truncateAndRenormalize` (D-07), whose output becomes `child.prior`, which is exactly what plan 159-01's `rankScore` reads. There is no combiner to write. The load-bearing constraints are D-05 (user's side ONLY — flattening the opponent has the opposite effect and would cancel the dial) and Pitfall 1 (T=1 must be a bit-identical no-op via short-circuit, or every existing user's default search silently drifts).

Purpose: implement temperature at the orchestrator layer (not maiaQueue), so the Maia (fen, elo) cache stays a pure raw oracle and the "is this the user's side" decision lives where rootMover is already known.
Output: `policyTemperature.ts`, a `sideMatchesMover` helper, a `SearchBudget.policyTemperature` field, and mirrored edits + tests in both runners.
</objective>
