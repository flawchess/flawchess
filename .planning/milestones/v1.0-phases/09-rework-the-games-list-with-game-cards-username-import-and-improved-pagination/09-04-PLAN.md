2. **app/users.py** -- Add a `current_user_or_dev_bypass` dependency that checks `settings.ENVIRONMENT`. When `ENVIRONMENT == "development"`, return a mock/first user from DB instead of requiring JWT. When `ENVIRONMENT == "production"`, delegate to `current_active_user`. Export this as `current_active_user` (rename the original to `_jwt_current_active_user`). Implementation:

```python
from app.core.config import settings
from app.core.database import get_async_session

# Keep original JWT-based dependency
_jwt_current_active_user = fastapi_users.current_user(active=True)

async def _dev_bypass_user(
    session: AsyncSession = Depends(get_async_session),
) -> User:
    """In development mode, return the first active user without auth."""
    from sqlalchemy import select as sa_select
    result = await session.execute(
        sa_select(User).where(User.is_active == True).limit(1)
    )
    user = result.unique().scalar_one_or_none()
    if user is None:
        from fastapi import HTTPException
        raise HTTPException(status_code=401, detail="No active user found (dev mode)")
    return user

if settings.ENVIRONMENT == "development":
    current_active_user = _dev_bypass_user
else:
    current_active_user = _jwt_current_active_user
```

Note: All routers already import `current_active_user` from `app.users`, so this change propagates automatically.

5. **Alembic migration** -- Run `uv run alembic revision --autogenerate -m "add white black username rating columns"`. Then edit the generated migration to add a backfill step after the ADD COLUMN operations:

```python
# Backfill from existing user_color + opponent_username + user_rating + opponent_rating
op.execute("""
    UPDATE games SET
        white_username = CASE WHEN user_color = 'white' THEN opponent_username ELSE opponent_username END,
        black_username = CASE WHEN user_color = 'black' THEN opponent_username ELSE opponent_username END,
```

Wait -- the backfill logic is: if user_color='white', then white_username is UNKNOWN (we don't have the per-game username stored), and black_username=opponent_username. If user_color='black', white_username=opponent_username and black_username is UNKNOWN. For ratings: if user_color='white', white_rating=user_rating and black_rating=opponent_rating. Vice versa.

Correct backfill SQL:
```python
op.execute("""
    UPDATE games SET
        white_username = CASE WHEN user_color = 'white' THEN NULL ELSE opponent_username END,
        black_username = CASE WHEN user_color = 'black' THEN NULL ELSE opponent_username END,
        white_rating = CASE WHEN user_color = 'white' THEN user_rating ELSE opponent_rating END,
        black_rating = CASE WHEN user_color = 'black' THEN user_rating ELSE opponent_rating END
""")
```

Note: white_username is NULL when user played as white because we don't know the per-game username (it may differ from current username on chess.com). The user's own username for existing games will remain NULL until they re-import. New imports will populate it correctly.
