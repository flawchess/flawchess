   - `PositionSuggestion(BaseModel)` with fields: `white_hash: str`, `black_hash: str`, `full_hash: str`, `fen: str`, `moves: list[str]`, `color: str` (white|black), `suggested_match_side: str` (mine|both), `game_count: int`, `opening_name: str | None`, `opening_eco: str | None`.    - `SuggestionsResponse(BaseModel)` with field `suggestions: list[PositionSuggestion]`.
   - `MatchSideUpdateRequest(BaseModel)` with field `match_side: str` (mine|opponent|both). No `color` field needed since the bookmark already stores `color`.

2. In `app/repositories/position_bookmark_repository.py`, add four async functions:

   a. `get_existing_full_hashes(session, user_id) -> set[int]`: SELECT DISTINCT full_hash from position_bookmarks WHERE user_id. Used for deduplication. NOTE: position_bookmarks does NOT have a full_hash column — the bookmark stores `target_hash` which could be white_hash, black_hash, or full_hash depending on match_side. To get the full_hash for dedup, we need to recompute it from the stored FEN using `compute_hashes`. Alternative simpler approach: fetch all bookmarks for the user, replay each FEN with `chess.Board(fen)` + `compute_hashes(board)`, collect the full_hash values. Since a user has at most ~50 bookmarks, this is fast.

   b. `get_top_positions_for_color(session, user_id, color, ply_min, ply_max, limit, exclude_full_hashes) -> list[tuple]`: GROUP BY (white_hash, black_hash, full_hash) on game_positions JOIN games WHERE user_id AND ply BETWEEN ply_min AND ply_max AND Game.user_color == color. ORDER BY COUNT(DISTINCT game_id) DESC. Over-fetch (limit + len(exclude) + 10), post-filter excluding full_hashes in exclude set, return first `limit` results. Each row is (white_hash, black_hash, full_hash, game_count).

   c. `suggest_match_side(session, user_id, color, white_hash, black_hash) -> str`: Compute ratio of COUNT(DISTINCT full_hash) / COUNT(DISTINCT game_id) for the user's color hash. If ratio <= 1.5 return "mine", else "both". Use `hash_col = GamePosition.white_hash if color == "white" else GamePosition.black_hash` and filter by the corresponding hash value.

   d. `update_match_side(session, bookmark_id, user_id, new_match_side) -> PositionBookmark | None`: Fetch the bookmark by id+user_id. Recompute target_hash: parse bookmark.fen with `chess.Board(fen)`, call `compute_hashes(board)` to get (white_hash, black_hash, full_hash), then resolve target_hash using the same logic as frontend's `resolveMatchSide`: if match_side=="both" -> full_hash, if "mine" -> white_hash when color=white / black_hash when color=black, if "opponent" -> the opposite. Update bookmark.match_side and bookmark.target_hash. Flush and return.

3. In `tests/test_bookmark_repository.py`, add tests:

   - `test_get_top_positions_excludes_hashes`: Same setup but pass exclude_full_hashes containing one of the result hashes, verify it's excluded.
   - `test_suggest_match_side_mine`: Create positions where the same white_hash maps to few distinct full_hashes, assert returns "mine".
   - `test_suggest_match_side_both`: Create positions where the same white_hash maps to many distinct full_hashes, assert returns "both".
   - `test_update_match_side`: Create a bookmark with match_side="both", call update_match_side with "mine", verify target_hash changed to the white_hash or black_hash as appropriate.

   Use the existing `db_session` fixture from conftest.py. Create test data by inserting Game and GamePosition rows directly via session.

  <action>
1. Add `GET /position-bookmarks/suggestions` endpoint. IMPORTANT: Define this route BEFORE the `PUT /position-bookmarks/reorder` and `/{bookmark_id}` routes to avoid path conflicts.

   The endpoint:
   - Requires `current_active_user` dependency (authenticated).
   - Calls `get_existing_full_hashes(session, user_id)` to get dedup set.
   - Calls `get_top_positions_for_color` twice (once for "white", once for "black") with ply_min=6, ply_max=14, limit=5, passing the exclude set.
   - For each result row (white_hash, black_hash, full_hash, game_count):
     - Fetch one representative game's PGN: SELECT game_id, ply FROM game_positions WHERE full_hash = X AND user_id = Y LIMIT 1, then SELECT pgn FROM games WHERE id = game_id. (Or join in one query.)
     - Reconstruct FEN + SAN moves at that ply using python-chess: `chess.pgn.read_game(io.StringIO(pgn))`, replay moves[:ply], collect `board.san(move)` for each, get `board.fen()`.
     - Compute opening name: call `find_opening(" ".join(f"{i//2+1}." + san if i%2==0 else san for i, san in enumerate(moves)))` — or simpler: construct a minimal PGN string from the SAN moves and pass to `find_opening`. The `find_opening` function accepts a PGN string and normalizes it internally.
     - Call `suggest_match_side(session, user_id, color, white_hash, black_hash)` for the heuristic.
   - Combine white + black results into `SuggestionsResponse`.
   - Return `response_model=SuggestionsResponse`.

   Performance note: For a typical user with < 50k games, the GROUP BY query runs in < 1s. The per-suggestion PGN replay (max 10 suggestions) adds < 100ms total.

2. Add `PATCH /position-bookmarks/{bookmark_id}/match-side` endpoint. This route is safe after `/{bookmark_id}` routes because the path has an additional `/match-side` literal segment.

   The endpoint:
   - Requires `current_active_user` dependency.
   - Accepts `MatchSideUpdateRequest` body.
   - Calls `update_match_side(session, bookmark_id, user_id, request.match_side)`.
   - Returns 404 if bookmark not found.
   - Returns `response_model=PositionBookmarkResponse`.
  </action>
