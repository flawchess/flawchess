## Phase Details

### Phase 84: Data plumbing — per-type cohort p50 + mirror-rate audit

**Goal:** Confirm per-type cohort `p50` is reachable from the frontend for Section 3 bullets, and audit that mirror-bucket peer rates (`opponent_score` / `opponent_games`) are already exposed on `/api/endgames/overview` for Section 2. Prerequisite for the bullet centre ticks in Section 3 and the peer bullets in Section 2 + 3.
**Depends on:** Nothing (first v1.17 phase)
**Requirements:** DATA-01, DATA-02
**Success Criteria** (what must be TRUE):
  1. `frontend/src/generated/endgameZones.ts` exposes an explicit per-type `p50` for each class × bucket (rook/minor_piece/pawn/queen/mixed × conversion/recovery), regenerated from `app/services/endgame_zones.py` via `scripts/gen_endgame_zones_ts.py`.
  2. CI drift guard on the codegen continues to pass after the schema extension.
  3. A written audit confirms `opponent_score` + `opponent_games` are already populated on every `MaterialRow` (Section 2) and on each per-type breakdown (Section 3) of `/api/endgames/overview` — or, if any are missing, a minimal backend extension is shipped to expose them.
**Plans:** 3 plans
- [ ] 84-01-PLAN.md — Extend PerClassBands with p50, regenerate endgameZones.ts, add TestPerClassP50 (DATA-01)
- [ ] 84-02-PLAN.md — Extend ConversionRecoveryStats with 4 opponent fields, wire mirror identity in _aggregate_endgame_stats (DATA-02)
- [ ] 84-03-PLAN.md — Append inline audit section to 84-01-SUMMARY.md per D-10 (DATA-01 + DATA-02)
**Context:** Confirmed during roadmap drafting: `PER_CLASS_GAUGE_ZONES` carries only `[p25, p75]` per class × bucket, not the explicit `p50` tick. The Section 2 `MaterialRow` schema already carries `opponent_score` + `opponent_games`, so DATA-02 is almost certainly an audit-only confirmation; per-type mirror data needs verification too. Phase may collapse into Phase 85/86 plan-phase if DATA-02 is confirmed already-present and DATA-01 is a one-plan codegen extension. Flag this assumption explicitly in the plan-phase.

### Phase 85: Section 1 — Games with vs without Endgame cards

**Goal:** Replace the `EndgamePerformanceSection` table with two side-by-side cards (No / Yes) on lg+, stacked on mobile. Each card carries a WDL bar + Score row + cohort score bullet vs 0.50. A full-width Score Gap footer bullet (Yes − No vs 0) spans both cards. Significance gating follows `EndgameStartVsEndSection` convention (`n ≥ 10` ∧ `p < 0.05` ∧ value outside neutral band).
**Depends on:** Phase 84 (mirror-rate audit complete — not strictly blocking since Section 1 uses no peer bullet)
**Requirements:** SEC1-01, SEC1-02, SEC1-03, SEC1-04, SEC1-05, SEC1-06, SEC1-07
**Success Criteria** (what must be TRUE):
  1. On lg+ the Endgames page shows two side-by-side cards labelled "Games without Endgame" / "Games with Endgame", stacked on mobile (375px parity).
  2. Each card shows its WDL bar and a Score row with chess score `(W + 0.5·D)/n` + Wilson CI whiskers + cohort score bullet centred at 0.50 with the standard neutral band.
  3. A full-width footer Score Gap bullet (Yes − No, centre 0, neutral band `[SCORE_GAP_NEUTRAL_MIN, SCORE_GAP_NEUTRAL_MAX]`) spans both cards.
  4. Card score-row font color paints only when `n ≥ MIN_GAMES_FOR_RELIABLE_STATS` AND `p < 0.05` AND value lands outside the neutral band; an `InfoPopover` on each Score row explains the cohort frame and the rating-tier confound.
  5. The legacy `EndgamePerformanceSection` table component is removed from the codebase (knip clean).

**UI hint:** yes

### Phase 86: Section 2 — Endgame Metrics 4-card layout (Conv / Parity / Recov / Skill)

**Goal:** Replace `EndgameScoreGapSection` table + 4-gauge strip with 4 side-by-side cards on lg+, stacked on mobile, in order Conversion → Parity → Recovery → Endgame Skill. Conv/Parity/Recov share an identical layout (gauge → percent + games → WDL bar → cohort bullet vs per-bucket p50 → peer bullet `You − Opp` vs 0). Skill card has no WDL bar and no peer bullet. Mirror-bucket opponent baseline logic (`opponentRate` / `MIRROR_BUCKET` / `MIN_OPPONENT_BASELINE_GAMES`) is preserved and feeds the peer bullets. Open question on Skill-bullet sig-test handling resolved during plan-phase per SEC2-08.
**Depends on:** Phase 84 (DATA-02 mirror-rate audit) and Phase 85 (Section 1 score-bullet pattern locked in for reuse).
**Requirements:** SEC2-01, SEC2-02, SEC2-03, SEC2-04, SEC2-05, SEC2-06, SEC2-07, SEC2-08, SEC2-09, SEC2-10
**Success Criteria** (what must be TRUE):
  1. User sees 4 side-by-side cards (Conv / Parity / Recov / Skill) on lg+, stacked on mobile; Conv/Parity/Recov each show gauge + percent + WDL bar + cohort bullet vs per-bucket p50 + peer bullet `You − Opp` vs 0.
  2. Skill card shows gauge + percent + cohort bullet vs p50 only (no WDL, no peer bullet); its sig-test handling is documented in the plan-phase decision log.
  3. Cohort-bullet neutral bands match the gauge bands exactly (reuse `FIXED_GAUGE_ZONES.{conversion,parity,recovery}`); peer bullets use the mirror-bucket opponent rate per-card with Wald-z sig test vs 0, gated on `MIN_OPPONENT_BASELINE_GAMES`.
  4. An `InfoPopover` on each cohort bullet explains the global p50 frame and the rating-tier confound; an `InfoPopover` on each peer bullet explains the filter-responsive mirror-bucket interpretation.
  5. The legacy `EndgameScoreGapSection` table and its 4-gauge strip are removed (knip clean).
**Plans:** TBD
**UI hint:** yes
**Context:** This is the largest phase by logic LOC. If plan-phase estimates exceed the project's plan-size heuristic, consider splitting into Phase 86a (cards + cohort bullets + Skill sig-test decision) and Phase 86b (mirror-bucket peer-bullet plumbing). Skill bullet sig-test resolution (SEC2-08) is the main scope risk — options are skip / approximate via component CIs / compute on raw outcomes.

### Phase 87: Section 3 — Per-type Endgame Type Breakdown cards

**Goal:** Replace the grouped `EndgameWDLChart` and extend `EndgameConvRecovChart` into 5 full per-type cards (rook / minor_piece / pawn / queen / mixed; pawnless hidden per `HIDDEN_ENDGAME_CLASSES`). Each card carries side-by-side Conv + Recov gauges, a WDL bar, Conv cohort bullet, Recov cohort bullet, Conv peer bullet (`You − Opp` vs 0), Recov peer bullet (`You − Opp` vs 0), and a Games deep-link to `/endgames/games?type=…`. Mobile density check on real device before locking; fallback path is to drop the per-type peer bullets and keep only cohort bullets if scroll bloat is unacceptable.
**Depends on:** Phase 84 (DATA-01 per-type p50 codegen) and Phase 86 (two-bullet doctrine and peer-bullet plumbing established).
**Requirements:** SEC3-01, SEC3-02, SEC3-03, SEC3-04, SEC3-05, SEC3-06, SEC3-07
**Success Criteria** (what must be TRUE):
  1. User sees 5 per-type cards in a 3-col grid on lg+, 2-col on sm, 1-col on mobile, each labelled by class with side-by-side Conv + Recov gauges, a WDL bar, two cohort bullets, two peer bullets (or fallback: cohort-only after real-device check), and a Games deep-link `/endgames/games?type=…`.
  2. Per-type cohort bullets use the per-class explicit `p50` from `PER_CLASS_GAUGE_ZONES[<class>].{conversion,recovery}` (Phase 84 output); per-type peer bullets use the per-type mirror-class rate, gated on `MIN_OPPONENT_BASELINE_GAMES` with a sparse-n indicator.
  3. Real-device mobile density check is performed and documented; if peer bullets are dropped, the decision log records the loss and updates the `InfoPopover` copy accordingly.
  4. The legacy `EndgameWDLChart` component is removed; `EndgameConvRecovChart` no longer exists as a standalone gauge-only component (extended into the per-type card shell). Knip clean.

**UI hint:** yes

### Phase 88: Polish — popovers, gating decisions, automation rules, 375px parity

**Goal:** Resolve the two polish decisions left for plan-phase (peer-bullet neutral band; gauge significance gating), apply CLAUDE.md browser-automation rules (`data-testid`, ARIA labels, semantic HTML) consistently across all new card surfaces, and verify mobile parity at 375px across Sections 1 / 2 / 3.
**Depends on:** Phases 85, 86, 87 (all section card layouts shipped)
**Requirements:** POLISH-01, POLISH-02, POLISH-03, POLISH-04
**Success Criteria** (what must be TRUE):
  1. Peer-bullet neutral band decision is documented and applied uniformly (either reuse `SCORE_GAP_NEUTRAL_MIN/MAX` across Conv/Parity/Recov, or introduce dedicated `CONV_DIFF_NEUTRAL_*` / `RECOV_DIFF_NEUTRAL_*` from the benchmarks distribution).
  2. Gauge significance gating decision is documented and applied (keep gauge always-colored with bullet font-gated, or extend gating to the gauge for visual consistency).

  4. All three sections render correctly at 375px (no horizontal scroll, 44px tappable targets, popovers reachable, peer-bullet fallback documented if triggered in Phase 87).
