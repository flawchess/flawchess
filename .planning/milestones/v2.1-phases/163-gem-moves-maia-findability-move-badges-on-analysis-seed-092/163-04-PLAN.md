<objective>
Wire gem detection into the analysis page. The hard part (RESEARCH Pitfall 1) is that both gem conditions must be evaluated at the PARENT position — the position before the move that reached the current node — but `useMaiaEngine`/`useStockfishGradingEngine` only expose current-position data. Solve it with two page-level per-FEN retention caches (mirroring the proven `engineEvalByFen`), a parent-keyed `gemCandidate` memo, and a `gemByNode` sticky cache (mirroring `liveFlawByNode`). Then surface it: a violet Gem board marker, a `GemIcon` in the move list, and a current-position `qualityBySanWithGem` override that recolors the chart curve + names the gem in the popover.

Purpose: Lazy, per-visited-ply, sticky gem badges on all visited nodes (mainline + free variations, both colors), re-deriving C1 when the ELO slider moves.
Output: two per-FEN caches, gemCandidate memo, gemByNode sticky cache, board + move-list + chart surfacing, and integration tests.
</objective>
