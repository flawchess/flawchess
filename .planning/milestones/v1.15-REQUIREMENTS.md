# Requirements: FlawChess v1.15 — Eval-Based Endgame Classification

**Milestone goal:** Replace the material-imbalance + 4-ply persistence proxy for endgame conversion/recovery classification with Stockfish eval (depth 15) populated into the existing `eval_cp` / `eval_mate` columns on `game_positions`. Backfill historical span-entry positions across benchmark + prod, eval new span-entry positions during import going forward, refactor endgame queries to threshold on eval, and remove the proxy entirely (hard cutover).

**Source:** Validation report at `reports/conv-recov-validation-2026-05-02.md` flagged that the proxy holds at ~81.5% agreement vs Stockfish on the populated subset (22% lichess-only eval coverage) but misses ~24% of substantive material-edge sequences. Queen and pawnless classes underperform structurally.

## v1.15 Requirements (Phase 78)

### Engine Integration (ENG)

- [x] **ENG-01**: Stockfish (recent stable version, pinned) is available inside the backend Docker image and runs as a long-lived UCI process — not a per-call subprocess fork. — *Shipped: pinned sf_17 AVX2 with SHA-256 supply-chain verification (later bumped to sf_18); CI runner installs `stockfish` via apt; `STOCKFISH_PATH` env var threaded end-to-end.*
- [x] **ENG-02**: A thin engine wrapper module exposes a single async-friendly API for evaluating a board position at depth 15, returning `(eval_cp, eval_mate)` with white-perspective sign convention matching the existing column semantics in `app/services/zobrist.py`. — *Shipped: `app/services/engine.py` with `evaluate(board)` returning `(eval_cp, eval_mate)`.*
- [x] **ENG-03**: The wrapper is shared by the backfill script and the import path so engine lifecycle (startup, hash size, depth) is configured in exactly one place. — *Shipped: same module consumed by `scripts/backfill_eval.py` and `app/services/import_service.py`.*

### Backfill (FILL)

- [x] **FILL-01**: A backfill script identifies endgame span-entry rows where both `eval_cp` and `eval_mate` are NULL, replays SAN from the game's `pgn` column to that ply, evaluates the position, and writes back. A span entry is `MIN(ply)` of a `(game_id, endgame_class)` group with `count(ply) ≥ ENDGAME_PLY_THRESHOLD`. — *Shipped: `scripts/backfill_eval.py`.*
- [x] **FILL-02**: The script is idempotent (safe to re-run; skips rows already populated), resumable (interruption mid-run does not require restart from scratch), and dedupes evaluations by `full_hash` so identical positions are not re-evaluated. — *Shipped (relaxed): row-level idempotency only via `WHERE eval_cp IS NULL AND eval_mate IS NULL`. `full_hash` dedup dropped per Phase 78 amendment 2026-04-30 — added complexity for marginal CPU savings on a one-shot backfill.*
- [x] **FILL-03**: The script runs to completion against the benchmark database first; the operator validates results before running it against prod. — *Shipped: combined Phase 78+79 backfill executed benchmark → prod in plan 79-04.*
- [x] **FILL-04**: After the prod backfill completes, every endgame span-entry row in prod has either `eval_cp` or `eval_mate` populated (existing lichess `%eval` annotations are trusted and not overwritten). — *Shipped: prod backfill verified complete in 79-04 cutover.*

### Import Pipeline (IMP)

- [x] **IMP-01**: When a game is imported, after endgame classification has marked positions with `endgame_class`, the import worker evaluates each per-class span-entry position and writes `eval_cp` / `eval_mate` to those rows where the lichess `%eval` annotation did not already populate them. — *Shipped: `_flush_batch` eval pass between `bulk_insert_positions` and the `move_count` UPDATE, same transaction.*
- [x] **IMP-02**: Import-time evaluation does not block other imports for an unbounded duration — typical games (1-3 span entries × ~70 ms at depth 15) add well under 1 second per game to the import path. — *Shipped: per-batch parallel `EnginePool` (PR #79) keeps the eval pass within budget; `STOCKFISH_POOL_SIZE` defaults to 1 outside prod, ships 2 in prod.*

### Endgame Service Refactor (REFAC)

- [x] **REFAC-01**: `query_endgame_entry_rows`, `query_endgame_bucket_rows`, and `query_endgame_elo_timeline_rows` in `app/repositories/endgame_repository.py` are rewritten to threshold on `eval_cp` and `eval_mate` at the span-entry row instead of `material_imbalance` at entry plus a contiguity-checked persistence lookup at entry + 4 plies. — *Shipped.*
- [x] **REFAC-02**: Conversion / parity / recovery classification follows the rule: apply user-color sign flip first, then `(eval_mate > 0) OR (eval_cp ≥ 100) → conversion`; mirror for recovery; else parity. Mate scores at any value count as max conversion / recovery. — *Shipped: `_classify_endgame_bucket(eval_cp, eval_mate, user_color)` is the single helper; `eval_mate` short-circuits to ±1,000,000 cp.*
- [x] **REFAC-03**: `_MATERIAL_ADVANTAGE_THRESHOLD`, `PERSISTENCE_PLIES`, the `array_agg(... ORDER BY ply)[PERSISTENCE_PLIES + 1]` patterns, and the contiguity case-expression are deleted from the codebase. The proxy is gone — hard cutover, no fallback. — *Shipped.*
- [x] **REFAC-04**: Index `ix_gp_user_endgame_game` is updated via Alembic migration so the new eval-based query stays index-only (the `INCLUDE` columns reflect what the rewritten queries actually read). — *Shipped: Alembic `c92af8282d1a` reshapes INCLUDE from `material_imbalance` to `eval_cp` / `eval_mate`.*
- [x] **REFAC-05**: The `material_imbalance` column on `game_positions` is preserved — it remains useful for other features and is not coupled to the conv/recov classification anymore. — *Shipped: column retained, `tests/test_aggregation_sanity.py` still consumes it.*

## Out of Scope (this milestone)

- **Re-evaluating positions that already have a lichess `%eval` annotation** — the validation report demonstrated lichess evals are accurate enough for classification; re-evaluating burns CPU for no agreement gain. Trust them.
- **Adding new columns** — `eval_cp` / `eval_mate` already exist; no schema growth was needed for the eval pass. Phase 79 added the `phase` SmallInteger column (one column).
- **Tuning the ±100 cp threshold or experimenting with per-class thresholds** — uniform ±100 cp boundary used. Per-class threshold tuning explicitly out of scope.
- **Removing or deprecating the `material_imbalance` column** — has other consumers; decoupled but kept (REFAC-05).
- **Eval coverage outside endgame span entries and middlegame entry** — opening positions are not part of this milestone; tactical filters and per-ply eval timeline data belong to SEED-010 Library.
