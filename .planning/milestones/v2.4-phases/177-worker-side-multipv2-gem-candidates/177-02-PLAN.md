<objective>
Add a dedicated, fully isolated tier-4b lease/submit endpoint pair so the ~416k already-analyzed
games can be gem-candidate-backfilled by the remote fleet, with the worker doing exactly the N
runner-up MultiPV-2 searches and the server writing a minimal row set with no reclassification.

Purpose: The SEED-076 incremental filter strips already-eval'd plies from an engine-game lease, so
a fully-analyzed tier-4b game leases with nothing for the worker to run a full pass on. The server
must therefore compute the candidate plies (played == stored best_move, out-of-book) from its own
stored best_move/pv columns and hand the worker just those FENs. The submit path must NOT re-run the
delete-then-insert reclassify (FLAWCHESS-8D churn) — it writes game_best_moves + stamps completion,
touching nothing else (S-06). This is the whole payoff of the phase (near-linear backfill scaling).

Output: the BestMove* schema pair, the two endpoints, the inverse-shift candidate reconstruction,
the minimal writer, and the config comment fix.
</objective>

<design_divergence_flag>
⚠ DELIBERATE DIVERGENCE FROM CONTEXT.md D-01's LITERAL WORDING — READ BEFORE EXECUTING.

CONTEXT.md D-01 says workers should reach tier-4b by "extending the idle fall-through: scope=idle
falls through to tier-4b after tier-3 empties." Verified against current code, that LITERAL design
is infeasible and would REGRESS blob backfill:

- `claim_eval_job(scope="idle")` deliberately returns None (→ 204) once tier-3 empties (SEED-072,
  eval_queue_service.py:771-778) SPECIFICALLY so the worker falls through to its dedicated
  `/flaw-blob-lease` rung — the only post-146 path that can write blobs.
- If scope=idle instead returned tier-4b work, the worker's idle rung would never 204 (tier-4b has
  ~416k games ≈ effectively unbounded), so the flaw-blob rung would never be reached → the worker
  fleet (~85% of tier-4 blob throughput) STOPS draining blob backfill. That is a direct SEED-072
  regression.

This plan instead honors CONTEXT.md **D-02** literally — "Mirrors the flaw-blob-lease/submit PAIR
pattern" — with a dedicated `/bestmove-lease` + `/bestmove-submit` pair consumed as a NEW worker
rung-5 (Plan 04), placed AFTER the flaw-blob rung-4 (preserving TIER_BLOB_BACKFILL(4) <
TIER_BESTMOVE_BACKFILL(5), Pitfall 6). This preserves D-01's INTENT (tier-4b is lowest priority,
fresh work first, single BEST_MOVE_BACKFILL_ENABLED gate) via the only mechanism that does not
starve blob backfill. RESEARCH.md and PATTERNS.md both independently recommend this dedicated pair.

If the developer prefers the literal idle-fall-through despite the blob-starvation cost, STOP and
re-plan — this changes Plan 02 + Plan 04 structurally. Default: proceed with the dedicated pair.
</design_divergence_flag>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker → server (/bestmove-lease, /bestmove-submit) | Semi-trusted, operator-token authenticated. Server computes candidate plies itself and recomputes them at submit; it trusts only the worker's runner-up numbers, never which plies it chose (S-02/D-03). |
| /bestmove-submit → DB (game_best_moves) | Minimal write path, structurally isolated from the reclassify write path (game_flaws) — no shared write-session code (D-02). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-177-05 | Tampering | submitted ply outside game range on /bestmove-submit | high | mitigate | In-range check `0 <= ply < game_length`, 422 before write (Task 3). |
| T-177-06 | Tampering | submitted ply in-range but not a real candidate | medium | mitigate | Server recomputes the candidate set (D-03) and drops non-members; never trusts worker ply choice (Task 3). |
| T-177-07 | Denial of Service (data integrity) | tier-4b submit accidentally sharing the reclassify write path | high | mitigate | Structural isolation — dedicated schema + handler, no apply_full_eval / _classify_and_fill_oracle, mirroring the proven flaw-blob isolation (Task 3). |
