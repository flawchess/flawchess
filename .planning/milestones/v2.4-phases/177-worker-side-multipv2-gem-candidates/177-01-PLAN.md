<objective>
Shift the gem-candidate MultiPV-2 runner-up search off the server's hot path for the fresh
analysis lanes (tier-1/2/3): gate the atomic lane to v2 workers, thread worker-computed
second-best evals through the already-dormant `second_best_map` parameter of
`_build_best_move_candidates`, and make the residual server-side fallback observable.

Purpose: Today `_apply_atomic_submit` passes `second_best_map=None` (eval_remote.py ~line 1170),
so EVERY played==best candidate takes a synchronous backend `evaluate_nodes_multipv2` call
inside the submit request — the exact server-pool load this phase removes. The mechanism to
remove it already exists (`fallback_targets = [t for t in candidate_targets if t.ply not in
second_best_map]`); this plan makes `second_best_map` non-empty for v2 submissions and forces
a clean fleet upgrade via lease-level version gating (S-03/Pitfall 4).

Output: schema fields for the new wire data, version-gated `/atomic-lease`, tamper-guarded
`second_best` consumption, and a source-tagged fallback branch.
</objective>

<assumption_delta_decision>
Noun: worker protocol version (was singular v1; becomes v1 + v2).
Decision: add-alongside — `WORKER_SCHEMA_VERSION` bumps to 2 and is gated at the LEASE for the
atomic lane; v1 workers idle harmlessly (204) rather than being hard-rejected. The version-gating
identity lives at the lease request (per-request query param), not per-worker registration, which
correctly names the throughput-relevant boundary (a v1 binary must produce no atomic-lane work,
regardless of how it authenticated).
Rationale: S-03 locks lease-level gating; a per-request param is the smallest surface (mirrors the
existing `scope` Query param on the same endpoint) and needs no new persisted state.
</assumption_delta_decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| worker → server (/atomic-lease, /atomic-submit) | Semi-trusted, operator-token authenticated. The server trusts submitted NUMBERS but never worker JUDGEMENT — it re-applies played==best, out-of-book, and inaccuracy gates itself (S-02). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-177-01 | Tampering | second_best ply in /atomic-submit | high | mitigate | Structural in-range check `0 <= ply < game_length`, 422 before any write (Task 3), mirroring the existing blob-node guard. |
| T-177-02 | Tampering | second_best eval for a non-candidate ply (in-range but not played==best) | medium | mitigate | Server recomputes candidates itself in _build_best_move_candidates; a submitted ply that is not a real candidate is simply never read (map lookup miss), never trusted (S-02). |
| T-177-03 | Denial of Service | stale v1 worker polling /atomic-lease post-deploy | low | accept | 204 is cheap (no DB write beyond the existing sweep); worker idle_sleep backoff bounds request rate. Accepted by design (S-03 rollout). |
| T-177-04 | Spoofing | unauthenticated access to atomic endpoints | high | mitigate | Unchanged `Depends(require_operator_token)` on both endpoints (X-Operator-Token, hmac.compare_digest, fail-closed). |
| T-177-SC | Tampering | npm/pip/cargo installs | n/a | accept | No new packages introduced this plan (RESEARCH: zero new deps) — audit table N/A. |
</threat_model>
