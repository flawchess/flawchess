## Decisions Made
- **No `user_id` in the key** — best-move candidacy is a property of the position, not the user (unlike `game_flaws`, which is inherently a user's mistake). `(game_id, ply)` composite PK is sufficient and provably unique.
- **Composite PK is the sole uniqueness mechanism** — no redundant `UniqueConstraint` on `(game_id, ply)`.
- **Raw cp storage (D-05)** — `best_cp`/`second_cp` are `SmallInteger` centipawns, mirroring `game_positions.eval_cp`; the cp→expected-score conversion stays query-time, keeping Gem/Great margins retunable with zero re-analysis.
- **No stored tier flag** — Gem vs Great is decided at query time from the stored floats (GEMS-01/07, D-4).

## Notes on Prohibition Verification (naive-grep caveat)
- The plan's Task 1 automated check `test "$(grep -icE 'is_gem|is_great|Boolean|Float' ...)" = "0"` reports a single match: the line `maia_prob: Mapped[float] = mapped_column(REAL, ...)`. That match is the type annotation on the one legitimately-required REAL column (identical to the established `bot_game_settings.py` `play_style_blend: Mapped[float]` pattern), not a forbidden column. The substantive prohibitions hold: no Boolean columns, no stored gem/great flag, and `maia_prob` is the ONLY REAL/float column (cp/mate are all `SmallInteger`). Docstring/comment prose was reworded to remove the words "boolean"/"float" so the check only trips on the unavoidable `Mapped[float]` annotation. Keeping the correct annotation over gaming the string check is the right call.
