<objective>
Wire the Plan-04 machinery into the eval-apply pipeline: a candidate-row builder, called from the single
shared `apply_full_eval` write-session body, that for each out-of-book played==best ply passing the inaccuracy
gate derives the mover's pinned ELO, scores the played move with backend Maia-3, and INSERTs a `game_best_moves`
row in the same commit as the rest of the eval — so EVERY newly analyzed game gets candidate rows regardless of
whether the local drain or a remote worker produced the Stockfish evals (GEMS-03), with the worker protocol
untouched. Explicitly handle Pitfall 1 (missing second-best on the remote-worker lane) with a targeted
backend-owned MultiPV-2 fallback, and measure steady-state RSS against the OOM budget (D-03b).

Purpose: This is the load-bearing integration. `apply_full_eval` is the ONE function both lanes call in-process
on the backend, so inserting here is what makes "every newly analyzed game" true without any worker change.
Pitfall 1 is a real, previously-undocumented gap: SEED-108's "zero extra Stockfish cost" premise only holds for
the local lane, so the builder must be prepared to spend bounded extra Stockfish on played==best plies that lack
runner-up data.

Output: the candidate-row builder in `app/services/eval_apply.py` (+ any helper in `eval_drain.py`),
`scripts/measure_maia_rss.py`, and eval-apply tests for the gate, the Pitfall-1 fallback, and row insertion.
</objective>
