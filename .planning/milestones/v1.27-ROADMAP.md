# Milestone v1.27: Remote Eval Worker Fan-Out & In-App Feedback

**Status:** ✅ SHIPPED 2026-06-16
**Phases:** 121, 122, 123 — 3 phases
**Total Plans:** 6 (121: 1 · 122: 2 · 123: 3)

## Overview

v1.27 is a post-v1.26 cleanup-and-scale milestone built from three independent seeds, grouped at close per the standalone-then-regroup pattern (cf. v1.20). Two of the three phases extend the v1.26 remote-eval-worker contract (SEED-048 / Phase 120) so off-box compute helps with more than just the idle tier-3 backlog; the third adds an in-app feedback channel.

Each phase shipped to production standalone as it completed (releases #199, #202, #203) and soaked in prod before this grouping:

- **Phase 121 — Remote-worker tier-1 claiming (SEED-048).** A remote eval worker can now claim tier-1 single-game "analyze" requests, not just the tier-3 idle backlog, so when the server pool is mid-game on another job a second idle machine can pick up a freshly-enqueued single-game analysis and shorten click-to-pickup latency. FCFS: the server's in-process drain still usually wins tier-1 when idle (no network hop, no poll interval), so this deliberately targets the server-busy overflow case.
- **Phase 122 — In-app feedback button (SEED-049).** A low-friction in-app feedback channel (guests included) so users can submit likes / dislikes / suggestions tied to the exact page they were on, persisted to a new `feedback` table and fired as a tagged Sentry signal so feedback pings the team instead of rotting in a table nobody reads.
- **Phase 123 — Remote-worker entry-ply fresh-import drain (SEED-051).** Extends the headless remote eval worker pool to also drain entry-ply (import-time, depth-15) eval in parallel on big first imports, cutting first-import latency by roughly the worker fan-out factor via a three-rung priority ladder: tier-1 single-game > entry-ply fresh-import drain > tier-3 idle backlog.

### Phase 123: Remote-worker fan-out for entry-ply (import-time) eval on big first imports (SEED-051)

**Status**: ✅ Implemented, tested, and live in prod — shipped 2026-06-16 (release #203). UAT done, confirmed on a real 5,132-game first import (user 28): entry-ply drain split server-pool (2,573) / worker `ws80` (1,800) with identical 3.7 evals/game density, 100% `evals_completed_at` with zero stuck or expired-unstamped leases, and code-review BLOCKER CR-01 (zero-target lease livelock) verified fixed live (9 zero-target leased games stamped, not re-leased). Mixed-fleet backward compat holds: un-upgraded workers still drain full-ply under the `remote-worker` fallback and never touch entry-ply.
**Goal**: Extend the headless remote eval worker pool (SEED-048 / Phase 120) — which today only drains full-ply tier-1/3 — to also drain **entry-ply** (import-time, depth-15) eval in parallel on **big first imports**, cutting first-import latency (time until a brand-new user sees flaws / phase-transition evals populate) by roughly the worker fan-out factor. The worker gains a second, higher-priority work type via a three-rung priority ladder: tier-1 single-game (top) > entry-ply fresh-import drain (new, batched depth-15) > tier-3 idle backlog (bottom), checked between full-ply games (no preemption, no reserved capacity). Incremental syncs stay server-pool-only via a runtime backlog-depth gate, so the lease/round-trip tax is only paid when it pays off.
**Depends on**: Phase 120 (remote worker lease/submit protocol + headless CLI); Phase 121 tier-1 claiming (#199)
**Source**: SEED-051 (decisions D-1…D-5 locked 2026-06-16) · **Plans**: 3 · completed 2026-06-16 (release #203); UAT 2/2 pass

- [x] 123-01 — Foundation: lease columns migration + shared SKIP-LOCKED LIFO claim helper + tuning constants + D-01 server-side lease
- [x] 123-02 — Batched /entry-lease + /entry-submit endpoints, scope param, D-5 backlog gate, X-Worker-Id (operator-token auth)
- [x] 123-03 — Worker CLI: D-06 ladder + depth-15 entry-ply path + distinctive --worker-id

Scope (the delta is small — reuses the SEED-048 worker + SEED-044 storage convention):

1. **One nullable lease column on `games`** + migration (D-3) — `entry_eval_lease_expiry` (optionally `entry_eval_leased_by`); the queue stays the existing predicate `games.evals_completed_at IS NULL`, claimed via a `SKIP LOCKED` LIFO (id DESC) lease. No new table, no per-position storage.
2. **Batched entry-ply lease endpoint** — claim N fresh games, server derives target plies (reuse `_collect_eval_targets` / phase-transition selection), return a flat `{game_id, ply, fen}[]`. Worker stays a dumb Stockfish-over-HTTP node (D-2).
3. **Batched entry-ply submit endpoint** — accept `{game_id, ply, eval_cp, eval_mate}[]`, apply SEED-044 convention, classify flaws, stamp `evals_completed_at` per fully-covered game, clear the lease.
4. **Worker CLI: depth-15 mode** + the between-full-ply-games priority check (D-1).
5. **D-5 backlog-depth gate** — bounded existence probe (`… LIMIT 1 OFFSET 299`) at lease time; invite workers only when backlog ≥ threshold (starting knob: 300 games / 50-game batches); tail (≲300 games) falls back to the server pool for free.

Open/deferred (not v1): entry-ply lease TTL sizing; routing `run_eval_drain` through the same lease so the server pool can't double-evaluate leased games (v1 vs fast-follow TBD); backlog-gate threshold tuning against real server-pool throughput once live; macOS background-scheduling caveat (SEED-048) unchanged.

### Key Decisions

- **Tier-1 remote claim is FCFS overflow, not biased routing** (SEED-048, Phase 121) — the server in-process drain still wins tier-1 when idle; remote claim only helps the server-busy case. Biasing tier-1 to the faster box and interruptible tier-3 are explicit deferred follow-ons.
- **Opaque job-token threaded lease→submit** (Phase 121) — the lease carries `eval_jobs.id` (`None` for tier-3), submit echoes it, the handler stamps `status='completed'` only when present, with a `status='leased'` stamp guard. Tier-3 behavior is unchanged.
- **Feedback fires a Sentry signal, not just a DB row** (SEED-049, Phase 122) — username / ELO-bucket / platform derived from the user record (not denormalized), tagged `source="feedback"`, so the team is pinged instead of polling a table.
- **Entry-ply fan-out via a three-rung worker ladder, no preemption** (SEED-051, Phase 123) — tier-1 > entry-ply fresh-import drain > tier-3, checked between full-ply games; no reserved capacity. The worker stays a dumb Stockfish-over-HTTP node; the server derives eval targets so server-pool and worker produce identical density.
- **One nullable lease column, queue predicate unchanged** (D-3, Phase 123) — `games.evals_completed_at IS NULL` claimed via `SKIP LOCKED` LIFO (id DESC); no new table, no per-position storage.
- **Backlog-depth gate keeps the lease tax pay-for-itself** (D-5, Phase 123) — a bounded existence probe (`LIMIT 1 OFFSET 299`) invites workers only when backlog ≥ ~300 games; the tail falls back to the server pool for free, so incremental syncs stay server-pool-only.
- **CR-01 zero-target lease livelock fixed** (Phase 123 REVIEW-FIX) — games whose derived target set is empty are stamped complete rather than re-leased forever; verified live (9 zero-target games stamped, not re-leased).

### Deferred / Out of Scope

- Biasing tier-1 to the faster box; interruptible tier-3 (SEED-048 follow-ons).
- Entry-ply lease TTL sizing; routing `run_eval_drain` through the same lease so the server pool can't double-evaluate leased games; backlog-gate threshold tuning against real server-pool throughput (SEED-051 open items).
- macOS background-scheduling caveat for off-box workers (SEED-048) unchanged.
