# Phase 94.1 Plan 12: WR-01 N+1 collapse on eval-drain Stage B fan-out — Summary

Collapsed the eval-drain post-commit Stage B gate from a per-user `count_pending_evals` loop (up to 10 sequential DB round-trips per drain batch) to a single aggregated `sa.values()` LEFT JOIN query, and locked the single-query contract with an event-listener regression test. Closes REVIEW.md WR-01.

## What Was Built

- **New repository helper `users_with_zero_pending(session, user_ids: Sequence[int]) -> list[int]`** in `app/repositories/game_repository.py`. Construction: inline `sa.values()` VALUES-table over the input user_ids LEFT OUTER JOIN'd to `games` on `(Game.user_id == uid_col) AND Game.evals_completed_at.is_(None)`, `GROUP BY uid_col`, `HAVING func.count(Game.id) == 0`. The LEFT JOIN over the inline VALUES table is what makes users-with-no-games return correctly (test 4 locks this). Empty input short-circuits to `[]` without issuing SQL.
- **Eval-drain rewrite** at `app/services/eval_drain.py:556-564`. The per-user `count_pending_evals` loop inside the post-commit read session is replaced with ONE call to `users_with_zero_pending(read_session, affected_user_ids)`, followed by the existing `asyncio.create_task(compute_stage_b(uid))` fan-out. Stage B behavioural semantics are unchanged: only the gate query is collapsed. Sentry capture pattern preserved.
- **6 regression tests** in `tests/repositories/test_game_repository_zero_pending.py`:
  - `test_empty_input_returns_empty_list` — empty input short-circuits
  - `test_all_users_have_zero_pending` — all 3 users with completed evals are returned
  - `test_mixed_pending_counts` — only the zero-pending user is returned
  - `test_user_with_zero_games_returns_user` — LEFT JOIN preserves users with no games
  - `test_aggregate_matches_per_user_loop` — aggregated result set matches the per-user `count_pending_evals` loop on a deterministic 5-user mixed cohort (this is the WR-01 equivalence contract)
  - `test_users_with_zero_pending_issues_single_query` — event-listener-based query-count test asserts exactly ONE `FROM`-bearing statement is issued (this is the WR-01 single-query contract)

## How It Works

The Stage B fan-out path was the eval-drain hot path's biggest N+1 risk: with `_DRAIN_BATCH_SIZE=10`, the worst case was 10 sequential `count_pending_evals` round-trips inside one read session per drain tick before the next batch could begin. The aggregated query issues one round-trip regardless of batch size.

The SQL emitted (Postgres dialect, sketch):

```sql
WITH input_uids(uid) AS (VALUES (1), (2), (3))
SELECT input_uids.uid
FROM input_uids
LEFT JOIN games
  ON games.user_id = input_uids.uid AND games.evals_completed_at IS NULL
GROUP BY input_uids.uid
HAVING count(games.id) = 0
```

For a user with no games at all, the LEFT JOIN emits one row with `games.id IS NULL`, and `count(games.id)` collapses to 0 — so they're correctly classified as zero-pending. For a user with all evals completed, every joined `games.evals_completed_at IS NULL` row is filtered by the ON predicate (not the WHERE clause), again yielding a 0 count. For a user with at least one pending eval, the count is ≥ 1 and the HAVING filters them out.
