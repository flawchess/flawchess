# Phase 94.1: Canonical-Slice User Percentile Materialisation - Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the apples-to-oranges comparison shipped in Phase 94 by materialising each user's canonical-slice metric value + percentile in a new `user_benchmark_percentiles` table, decoupling the chip from UI filter state. The chip becomes a **trait** of the user (like Elo), not a **view** of their data — computed from a canonical slice that mirrors the benchmark CTE (`status='completed'` + ±100 ELO opponent at game time + sparse-cell `(2400, classical)` exclusion + 36-month recency + standard variant, pooled across TCs with no per-TC cap), stored with the lookup percentile and a `cdf_snapshot`, and refreshed via two stages hooked into the existing two-lane import pipeline:

- **Stage A** — eval-independent. Fires at import completion. Computes `score_gap`.
- **Stage B** — eval-dependent. Fires when the cold drain has emptied the user's pending-eval rows. Computes `achievable_score_gap`, `section2_score_gap_conv`, `section2_score_gap_parity`.

A one-shot backfill script (`scripts/backfill_user_percentiles.py --target dev|prod`) populates existing users on rollout so the chip lights up for the entire user base, not just users who import after the deploy. Phase 94's per-request `interpolate_percentile` call site is rewired to read from the table; tooltip copy is updated to make the canonical-slice framing explicit.

**Out of scope:** Phase 95 LLM payload + prompt rework (consumes the materialised percentile but lives in its own phase). Re-lookup-after-CDF-regeneration as a scheduled job (manual-run script + `cdf_snapshot` column are sufficient for v1.19). Per-type / Time Pressure / Opening Insights percentile chips.

</domain>

<decisions>
## Implementation Decisions

### Stage B trigger semantics (discussed)

- **D-01: Per-batch group-by-user check in `eval_drain.py`.** After each drain batch commits (`_mark_evals_completed` + `session.commit()` at `app/services/eval_drain.py:547-548`), group the just-drained `game_ids` by `user_id`. For each affected user, query `COUNT(*) WHERE user_id = X AND evals_completed_at IS NULL`. For users whose count is now zero, fire Stage B as an `asyncio.create_task` background job. Low latency (~drain tick interval, seconds). One extra count query per affected user per drain batch — cheap on the existing `ix_games_evals_pending` partial index.
- **D-02: No periodic sweeper coroutine.** The failure mode (Stage B raises, process crashes mid-batch, transient DB outage between mark-completed and Stage B fire) is recovered by either (a) the user's next import (Stage A first, then their next drained batch re-triggers Stage B), or (b) a manual `scripts/backfill_user_percentiles.py --target prod` run. The backfill script IS the safety net for this phase — explicit, on-demand, no third coroutine to maintain.
- **D-03: Stage A hook lives in `_complete_import_job` (`app/services/import_service.py:480`).** Fires after `session.commit()` at the end of the function, NOT inside the import transaction (per ROADMAP success criterion 3: "does not run inside the import transaction"). Background task via `asyncio.create_task`. Computes `score_gap` only.
- **D-04: Stage A and Stage B are non-blocking — neither extends import latency.** Both are fired-and-forgotten as background tasks. Stage A is enqueued by the import worker; Stage B is enqueued by the drain coroutine. Errors inside Stage A / Stage B are captured to Sentry (per CLAUDE.md Backend Rules — `sentry_sdk.capture_exception` with `set_context` for `user_id` / `metric`) but do not propagate to the trigger site.

### Schema type details (discussed)

- **D-05: `metric` column = Postgres ENUM type.** `CREATE TYPE benchmark_metric AS ENUM ('score_gap', 'achievable_score_gap', 'section2_score_gap_conv', 'section2_score_gap_parity')`. Mirrors the `CdfMetricId` Literal in `scripts/gen_global_percentile_cdf.py`. Stricter than VARCHAR + CHECK at the cost of an `ALTER TYPE ... ADD VALUE` migration when future metrics get badged — acceptable for the v1.19 set and well-trodden in Alembic. Note for planner: the codebase doesn't use Postgres ENUMs elsewhere, so this is a deliberate first-use; document the pattern at the migration.
- **D-06: `percentile` column = `DOUBLE PRECISION NULL`.** Matches Phase 94's API return type (`float | None`) exactly — no precision mismatch at the chip-render seam, no rounding inside the DB layer. NULL semantics match the "below inclusion floor → no chip" path (mirrors how `_p_value` / `_ci_*` emit `None` today). Yes, the chip rounds to integer for display, but storing the float preserves whatever sub-integer precision `interpolate_percentile()` returns (useful for the Phase 95 LLM payload and any future viz).
- **D-07: `cdf_snapshot` = `DATE`, no secondary index.** Snapshot date e.g. `2026-05-23`. Primary key `(user_id, metric)` is the only index. The manual re-lookup-after-CDF-regeneration script does a full table scan filtering `WHERE cdf_snapshot < new_date` — acceptable at the current user-count scale for a one-shot manual run. Revisit if the re-lookup becomes scheduled.
- **D-08: Full schema (locked).**
  ```
  user_benchmark_percentiles
    user_id        INTEGER       NOT NULL  FK -> users.id  ON DELETE CASCADE
    metric         benchmark_metric ENUM NOT NULL
    value          DOUBLE PRECISION NOT NULL  -- user's canonical-slice metric value
    percentile     DOUBLE PRECISION NULL      -- NULL when below inclusion floor
    n_games        INTEGER NOT NULL           -- sample size at compute time
    cdf_snapshot   DATE NOT NULL              -- which CDF this percentile maps to
    computed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
    PRIMARY KEY (user_id, metric)
  ```
  Recompute is UPSERT (`INSERT ... ON CONFLICT (user_id, metric) DO UPDATE`). Exactly one row per `(user_id, metric)` at any time.

### Locked by ROADMAP / design note (not re-discussed)

- **D-09: Canonical slice = benchmark cohort definition.** `status='completed'` + ±100 ELO opponent (game-time) + sparse-cell `(2400, classical)` exclusion + 36-month recency + standard variant + per-metric inclusion floor. Pooled across TCs (no per-TC cap). Source of truth: `.claude/skills/benchmarks/SKILL.md` §1 + the CTE machinery in `scripts/gen_global_percentile_cdf.py`. Locked by ROADMAP success criterion 2.
- **D-10: Per-metric inclusion floors (from design note, mirror Phase 93/94 floors).** `score_gap` ≥30 endgame AND ≥30 non-endgame games; `achievable_score_gap` ≥20 endgame-entry games; `section2_score_gap_conv` ≥20 spans per entry-eval bucket; `section2_score_gap_parity` ≥20 spans per entry-eval bucket. A user failing the floor for a metric gets either no row OR `percentile=NULL` (planner picks — recommend `percentile=NULL` + the `value` still stored if computable, so future floor changes don't require recompute; if `value` itself isn't defined below the floor, no row).
- **D-11: CTE-sharing mechanism = Option 1 (extract shared module).** The note's preferred choice; the user did not pick this for further discussion, so the default Option 1 path holds. Extract `_canonical_selected_users_cte`, `_per_user_cte_score_gap`, `_per_user_cte_achievable`, `_per_user_cte_section2`, `_equal_footing_filter_sql`, `_sparse_exclusion_sql`, `_elo_bucket_expr` from `scripts/gen_global_percentile_cdf.py` into a shared module (recommended path: `app/services/canonical_slice_sql.py` — under `app/` so both the script and the per-user compute service can import without a script→app import). Refactor `gen_global_percentile_cdf.py` to consume the shared module. The two structural differences live in the consumer layer: benchmark CDF iterates over `selected_users` and aggregates per-(user, cell); per-user path filters by single `user_id` from `users` and collapses to one pooled value per user per metric. Locked by ROADMAP success criterion 7.
- **D-12: Chip read path rewire.** Phase 94's per-request `interpolate_percentile(metric_id, value)` call inside `app/services/endgame_service.py` is removed; the 4 `{metric}_percentile` fields read `(value, percentile)` from `user_benchmark_percentiles` for the current `user_id`, ignoring filter state. The row's filter-applied metric value continues to render from the existing per-request compute. Locked by ROADMAP success criterion 4.
- **D-13: Tooltip copy update.** The existing chip popover copy (Phase 94 D-08: skill-isolating vs improvement-focus flavors) gets a canonical-slice framing addition — make explicit that the chip reflects the user's career under canonical conditions (status='completed', ±100 ELO opponents, last 36 months, pooled across TCs) while the row's value reflects active UI filters. Within the discipline of `feedback_popover_copy_minimalism.md` (WHAT + sign convention, no jargon, no caveats); planner picks the exact strings. Locked by ROADMAP success criterion 4.
- **D-14: Backfill script (`scripts/backfill_user_percentiles.py`).** CLI: `--target dev|prod` (mirrors `scripts/import_stress_monitor.py` lines 13-19), optional `--user-id <id>`, optional `--metric <id>`. Idempotent (UPSERT). Eval-dependent metrics computed only for users whose cold drain has completed; pre-cold-drain users get Stage-A-only backfill rows. Emits per-metric per-inclusion-floor-reason summary table. Operational nuances (user ordering, batched commits, n_games=0 skip strategy) are planner's call — not deferred for further discussion. Locked by ROADMAP success criterion 6.

### Claude's Discretion

The user picked Stage B trigger + Schema type details as the only two areas worth a full discussion. The remaining decisions default to the ROADMAP / design note path. Planner has flexibility on:

- **Shared-module file path and naming.** Recommend `app/services/canonical_slice_sql.py` — under `app/` so both the script and the per-user compute service can import; "canonical_slice_sql" names the canonical-slice-game-set role explicitly. Open to other names.
- **Per-user compute service location.** Recommend a new `app/services/user_benchmark_percentiles_service.py` (or similar) that exports `compute_stage_a(user_id) -> None` and `compute_stage_b(user_id) -> None`, each calling into the shared canonical-slice SQL + `interpolate_percentile()` from `app/services/global_percentile_cdf.py`, then UPSERTing into the table.
- **Backfill operational details.** Order of users (recommend creation-asc to surface oldest accounts first), batched commits (recommend per-user commit — simpler error recovery than batched), pre-skip users with zero games via a single LEFT JOIN at the top (avoids per-user query overhead), progress logging (per-user line at INFO is fine at v1.19 scale; per-100 users summary line is also OK).
- **Inclusion-floor failure representation.** Recommend: if `value` is computable but `n_games < floor`, store the row with `percentile = NULL`; if `value` itself isn't computable (zero games in slice), no row. Document in PLAN.md.
- **Whether to add an Alembic test asserting the ENUM contains exactly the 4 expected values** — recommend yes, a single pytest assertion against `pg_enum` to catch accidental drift.
- **Stage A / Stage B background-task error handling.** Recommend: wrap each in a `try/except Exception` that logs + `sentry_sdk.capture_exception()` with `set_context("percentile_compute", {"user_id": …, "stage": "A"|"B"})`. Never propagate to the trigger site (drain coroutine / import worker must not crash).

These are HOW decisions inside the locked WHAT. Planner can resolve them in PLAN.md without coming back to the user, but should document the chosen shape briefly.

</decisions>

## Existing Code Insights

### Reusable Assets
- **Canonical CTE machinery in `scripts/gen_global_percentile_cdf.py`** — the 7 `_*` builders are extracted verbatim into the new shared module (D-11). No re-derivation of the canonical slice.
- **`interpolate_percentile(metric_id, value)` in `app/services/global_percentile_cdf.py`** — the per-user compute service calls this after deriving `value` from the canonical slice. Same helper Phase 94 used per-request; now called once per metric per user at Stage A / Stage B firing time.
- **`async_session_maker()` + `_BATCH_SIZE` patterns in `import_service.py` / `eval_drain.py`** — established session-scoping pattern for short write windows. Stage A / Stage B background tasks use the same pattern (open a short session, UPSERT, commit, close).
- **`scripts/import_stress_monitor.py` `--target dev|prod` argparse** — copy-paste the connect-resolution helper (dev → local Docker, prod → tunnel + refuse-if-down).
- **`ix_games_evals_pending` partial index** — already exists on `WHERE evals_completed_at IS NULL`. The Stage B per-user pending-count query (D-01) hits this index efficiently.

### Established Patterns
- **Background-task fan-out via `asyncio.create_task`** — Stage A and Stage B follow the same pattern the import orchestrator already uses for fire-and-forget jobs.
- **UPSERT via `INSERT ... ON CONFLICT ... DO UPDATE`** — Phase 91 / Phase 92 introduced this idiom on `import_jobs` and `game_positions`. The new `user_benchmark_percentiles` recompute path uses the same SQLAlchemy `insert().on_conflict_do_update()` call shape.
- **Per-metric reliability-floor gate emitting `NULL` to the wire** — backend already does this for `_p_value` / `_ci_*` when `n < PVALUE_RELIABILITY_MIN_N`. Phase 94.1 mirrors that semantic at the DB layer (`percentile NULL` when below floor; chip absent on the FE).
- **Sentry error capture pattern** — `sentry_sdk.set_context("import", {"job_id": …})` + `sentry_sdk.capture_exception(exc)` per CLAUDE.md Backend Rules. Stage A / Stage B background-task errors follow the same pattern with `set_context("percentile_compute", {...})`.
- **Alembic migration + seed-on-startup discipline** — migrations run automatically on backend container startup via `deploy/entrypoint.sh`. The new ENUM type + table land in one migration.

### Integration Points
- **Backend service → schema → frontend type** (Phase 94 pattern, unchanged): `endgame_service.py` reads `(value, percentile)` from `user_benchmark_percentiles` for the current user, writes the 4 `{metric}_percentile` fields onto the response model. TypeScript types regenerate automatically from the FastAPI OpenAPI schema.
- **Import worker → Stage A**: `_complete_import_job` fires `asyncio.create_task(compute_stage_a(user_id))` after the commit. Drain coroutine → Stage B: after each `_mark_evals_completed` commit, group game_ids by user, count pending per user, fire `asyncio.create_task(compute_stage_b(user_id))` for users at zero.
- **Backfill script → Stage A + Stage B**: `scripts/backfill_user_percentiles.py --target prod` runs both stages in-process for every user (filtering Stage B to users whose pending-eval count is already zero — matches steady-state hook behavior). Idempotent UPSERT.
- **No router changes**: Phase 94's endpoint shape is unchanged. Only the data source for the 4 `{metric}_percentile` fields moves from per-request `interpolate_percentile()` to the new table.
- **Migration: 1 new ENUM type + 1 new table.** No existing-column changes. The chip read-path rewire in `endgame_service.py` is a pure code change (no schema impact).

## Specific Ideas

- **Shared module name = `app/services/canonical_slice_sql.py`** (recommendation, planner can override). Under `app/` so both the per-user compute service and `scripts/gen_global_percentile_cdf.py` import from the same place.

  ```
   achievable_score_gap           upserted=634, skipped=282 (below_floor=190, no_eval=92)

   section2_score_gap_parity      upserted=438, skipped=478 (below_floor=386, no_eval=92)
  ```
- **Postgres ENUM first-use** — document in the migration's comment / `op.execute(...)` block and in `app/models/user_benchmark_percentiles.py` why ENUM was chosen over VARCHAR+CHECK (4-value tight set mirroring `CdfMetricId`; project's first ENUM use is intentional).

<deferred>
## Deferred Ideas

- **Phase 95 LLM payload + prompt rework consuming the percentiles from `user_benchmark_percentiles`** — locked to Phase 95 per ROADMAP. Phase 94.1 only persists the data.
- **Periodic sweeper coroutine for Stage B safety net** — explicitly rejected for v1.19 (D-02). The backfill script is the safety net. Revisit if Stage B failures become a recurring operational issue.
- **Scheduled CDF-snapshot re-lookup job** — ROADMAP success criterion 5 says the manual-run script + `cdf_snapshot` column are sufficient for v1.19. The script itself is out of scope for Phase 94.1; it gets defined when the first benchmark CDF regeneration after v1.19 ships triggers it.
- **Per-type / Time Pressure / Opening Insights percentile chips** — out of scope (no canonical-slice methodology defined yet for those metric families).
- **Postgres ENUM secondary index on `metric`** — not needed at v1.19's user-count scale; PK on `(user_id, metric)` is the only access pattern. Revisit if the table grows millions of rows.
- **Storing a history of percentile computations (audit / timeline of "my chip over time")** — explicitly rejected by the design note's UPSERT-in-place schema. Future timeline-of-chip viz would justify a separate `user_benchmark_percentiles_history` table; not v1.19 scope.

</deferred>
