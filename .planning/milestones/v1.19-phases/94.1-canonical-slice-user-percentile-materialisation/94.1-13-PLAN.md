<objective>
Close two correctness gaps surfaced during Plan 94.1-11 HUMAN-UAT:

1. The chip shows Top 6% for user 28's achievable_score_gap when the floor-respecting value indicates Top 71%. Root cause: `_compute_metric_for_user` reports `value_raw` computed with `apply_floor=False`, averaging tiny below-floor cells (n=1, n=3) into the user's reported value alongside the legitimate n=362 cell. The benchmark CDF was built with `apply_floor=True`, so the single-user reporting must match.

2. Stage B re-fires multiple times during an active import as eval batches drain and per-user pending-eval count flickers between zero and non-zero. Each re-fire produces a different value because the canonical-slice CTE input set has grown. Users briefly see partial intermediate percentiles.

The minimum-viable correct fix is to:

- **Drop the `n_cells_floor` column entirely.** Nothing reads its numeric value (only `== 0` as a binary gate). The Plan 11 rename is reverted by this drop migration.
- **Single-query value computation with `apply_floor=True`.** No row is written for below-floor users (the chip suppresses naturally).
- **Active-import gate on Stage B fan-out.** Stage B fires once, after eval-drain settles AND no in_progress/pending import_jobs exist for the user.
- **Re-backfill dev DB** so existing rows reflect the corrected formula.

This plan does NOT attempt to resolve the deeper semantic question of whether the chip should display a per-user-averaged-across-cells value or per-cell values. That's a SPEC discussion (option β vs α vs γ — see Plan 11 SUMMARY supersession note) and a candidate for a separate phase.
</objective>
