## Decisions Made

- **Delete the 4 heavy-seed scaffolds outright.** Tombstone skip markers point at another file but still register as test code; deletion makes the new real-data module the unambiguous canonical home and matches the plan's `<acceptance_criteria>` (no `Realized by tests/...` pointer comments).
- **SC-7 parity test narrowed to score_gap.** Per Plan 09 Task 2 action step 2's explicit escape clause: eval-bearing metrics need realistic per-ply `eval_cp` values to exercise the Lichess sigmoid, which is a heavier seed surface; that surface is covered by the new real-data module's Stage B test instead.
- **Stage B test strong-asserts only `achievable_score_gap`.** Per the plan: with 35 eval-bearing games rotated across 3 eval buckets, the `achievable_score_gap` floor (≥20 entry-eval games) is met, but the `section2_*` per-bucket span floor (≥20 spans per entry-eval bucket) requires far more games. Soft-asserting section2_* is the right calibration.
- **Below-floor test is tolerant of either no-row OR row-with-NULL-percentile.** Per CONTEXT D-10, both outcomes satisfy the contract; the precise behavior depends on which cells survive the elo-bucket NULL gate against the small seed.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Downstream `su.tc_bucket` referenced a column that did not exist on the single_user CTE**

- **Found during:** Task 2 (running the SC-7 parity test against a real DB)
- **Issue:** REVIEW.md WR-03 had flagged this as a possible runtime blocker. The single_user `selected_users` CTE projected only `user_id`, but the per-metric CTEs all reference `su.tc_bucket AS tc_bucket` in their `rows` / `gap_rows` / `spans` projections. Running the SC-7 parity query produced `UndefinedColumnError: column su.tc_bucket does not exist`. The bug was latent before Plan 09 because the same SQL never executed (the bindparam tokeniser failed earlier on `:user_id::int`).
- **Fix:** Add `NULL::text AS tc_bucket` to the single_user CTE so the downstream projection resolves. The per-user path pools across all TCs anyway (D-09: no per-TC cap), so a NULL placeholder is semantically correct — the sparse-cell exclusion `(2400, 'classical')` is trivially satisfied against NULL.
- **Files modified:** `app/services/canonical_slice_sql.py`, `tests/services/test_canonical_slice_sql.py` (added `test_single_user_cte_emits_tc_bucket_column` regression guard).
- **Verification:** `uv run pytest tests/services/test_canonical_slice_per_user_vs_benchmark_parity.py tests/services/test_canonical_slice_sql.py -x` passes.
- **Committed in:** `997feaad` (part of Task 2 commit).

**2. [Rule 3 - Blocking] Sentry-mock tests previously relied on the now-fixed bindparam ArgumentError to trip**

- **Found during:** Task 2 (running `test_user_benchmark_percentiles_service.py` after lifting skip markers)
- **Issue:** The Sentry-mock tests patched only `interpolate_percentile` to raise, but the new (fixed) code path now reaches `_compute_metric_for_user` first, which silently returns `(None, 0)` for a user with no games. The early-exit path then short-circuits before `interpolate_percentile` is called, so the mocked side_effect never fires and no Sentry context is captured.
- **Fix:** Also patch `_compute_metric_for_user` to return `(0.05, 5)` so we deterministically reach `interpolate_percentile`. Now the mocked raise fires every run.
- **Files modified:** `tests/services/test_user_benchmark_percentiles_service.py`.
- **Verification:** Both Sentry-mock tests now pass deterministically.
- **Committed in:** `997feaad` (part of Task 2 commit).
