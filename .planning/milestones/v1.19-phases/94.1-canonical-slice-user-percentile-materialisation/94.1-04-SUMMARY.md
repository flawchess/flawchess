## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] PercentileRow as dataclass instead of TypedDict**

- **Found during:** Task 2 verification (test_upsert_inserts_when_no_row_exists)
- **Issue:** Plan 01 tests use attribute access (`row.value`, `row.percentile`, `row.n_games`, `row.cdf_snapshot`). TypedDict is a dict subclass and doesn't support attribute access.
- **Fix:** Changed `PercentileRow` from `TypedDict` to a frozen dataclass. CLAUDE.md allows TypedDict for internal data but frozen dataclass is equally valid for this use case and matches the test contract.
- **Files modified:** `app/repositories/user_benchmark_percentiles_repository.py`
- **Commit:** 3bbc81c1

**2. [Rule 1 - Bug] Implicit varchar→benchmark_metric cast for asyncpg compatibility**

- **Found during:** Task 1 verification (test_pk_rejects_duplicate_metric_for_same_user_via_plain_insert)
- **Issue:** asyncpg sends Python `str` bind parameters as `$N::VARCHAR` in prepared statements. Postgres rejects this for ENUM columns without an explicit `::benchmark_metric` cast. Plan 01's test 5 uses raw `text().bindparams()` with a Python `str`, which fails with `DatatypeMismatchError` before even reaching the PK constraint check.
- **Fix:** Added `CREATE CAST (varchar AS benchmark_metric) WITH INOUT AS IMPLICIT` in the migration's `upgrade()` step (after ENUM creation, before table creation). The cast is safe: invalid strings still fail at runtime; it only removes the implicit type rejection for valid ENUM labels sent as varchar.
- **Downgrade cleanup:** `DROP CAST IF EXISTS (varchar AS benchmark_metric)` added in `downgrade()` before `benchmark_metric_enum.drop()`.
- **Files modified:** `alembic/versions/20260524_000000_add_user_benchmark_percentiles.py`
- **Commit:** e14f2d3c
