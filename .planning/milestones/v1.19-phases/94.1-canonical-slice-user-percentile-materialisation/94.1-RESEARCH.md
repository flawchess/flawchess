# Phase 94.1: Canonical-Slice User Percentile Materialisation - Research

**Researched:** 2026-05-23
**Domain:** SQLAlchemy 2.x async migrations + Postgres ENUM + canonical-slice SQL extraction + two-stage background compute pipeline
**Confidence:** HIGH (most decisions locked in CONTEXT.md; code seams concretely identified in the running codebase)

## Summary

Phase 94.1 is a structural rewire: nothing changes about the user-facing chip's appearance or the published CDF, but the per-request `interpolate_percentile(filter_applied_value)` call in `endgame_service.py` (4 call sites at lines 1369-1394 and 2344-2348) is replaced by a per-user materialised value/percentile pair stored in a new `user_benchmark_percentiles` table. The replacement value is computed against the user's *canonical slice* (mirror of the benchmark cohort CTE) so the chip is filter-independent. Compute happens via two background tasks: Stage A fires at `_complete_import_job` (`app/services/import_service.py:480`) after commit; Stage B fires at the end of each `eval_drain.py` tick (line 547-548) once `_mark_evals_completed` commits, for each user whose `evals_completed_at IS NULL` count just dropped to zero. A one-shot backfill (`scripts/backfill_user_percentiles.py --target dev|prod`) populates existing users on rollout.

Every locked decision in CONTEXT.md (D-01 through D-14) and the seven ROADMAP success criteria are internally consistent and grounded in patterns the codebase already runs in production: the `pg_insert().on_conflict_do_nothing()` pattern (game_repository.py:30) maps cleanly to `on_conflict_do_update()`; the `postgresql.ENUM(... create_type=False)` migration shape (alembic/versions/20260408_155149_3e4018d62102_*.py) is the precedent for D-05's `benchmark_metric` ENUM; the `--db dev|benchmark|prod` argparse pattern in `scripts/backfill_eval.py` plus the `--target dev|prod` flavour in `scripts/import_stress_monitor.py` together specify the `--target dev|prod` CLI exactly. The shared canonical-slice SQL module (D-11) is a mechanical extraction of 7 helpers from `scripts/gen_global_percentile_cdf.py` lines 214-509 into `app/services/canonical_slice_sql.py`, with `selected_users` becoming a parameter — benchmark consumer keeps the existing CTE; per-user consumer substitutes `SELECT u.id AS user_id FROM users u WHERE u.id = :user_id`.

**Primary recommendation:** Plan as 6 tasks across 4 waves — (Wave 0) shared SQL module extraction + Alembic migration + model; (Wave 1) per-user compute service + UPSERT path; (Wave 2) Stage A + Stage B hook wiring; (Wave 3) endgame_service.py rewire + tooltip copy + backfill script. Wave 0 is the gate — refactoring `gen_global_percentile_cdf.py` to consume the shared module must land green before Stage A/B hooks can be wired, because the per-user compute service depends on the same shared module.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Stage B trigger semantics:**
- **D-01: Per-batch group-by-user check in `eval_drain.py`.** After each drain batch commits (`_mark_evals_completed` + `session.commit()` at `app/services/eval_drain.py:547-548`), group the just-drained `game_ids` by `user_id`. For each affected user, query `COUNT(*) WHERE user_id = X AND evals_completed_at IS NULL`. For users whose count is now zero, fire Stage B as an `asyncio.create_task` background job. Low latency (~drain tick interval, seconds). One extra count query per affected user per drain batch — cheap on the existing `ix_games_evals_pending` partial index.
- **D-02: No periodic sweeper coroutine.** The failure mode is recovered by (a) the user's next import (Stage A first, then their next drained batch re-triggers Stage B), or (b) a manual `scripts/backfill_user_percentiles.py --target prod` run. The backfill script IS the safety net for this phase.
- **D-03: Stage A hook lives in `_complete_import_job` (`app/services/import_service.py:480`).** Fires after `session.commit()` at the end of the function, NOT inside the import transaction. Background task via `asyncio.create_task`. Computes `score_gap` only.
- **D-04: Stage A and Stage B are non-blocking.** Errors inside Stage A / Stage B are captured to Sentry (per CLAUDE.md Backend Rules) but do not propagate to the trigger site.

**Schema type details:**
- **D-05: `metric` column = Postgres ENUM type.** `CREATE TYPE benchmark_metric AS ENUM ('score_gap', 'achievable_score_gap', 'section2_score_gap_conv', 'section2_score_gap_parity')`. Mirrors the `CdfMetricId` Literal. Project's first ENUM with this name — document the pattern at the migration.
- **D-06: `percentile` column = `DOUBLE PRECISION NULL`.** Matches Phase 94's API return type (`float | None`). NULL semantics match the "below inclusion floor → no chip" path.
- **D-07: `cdf_snapshot` = `DATE`, no secondary index.** Primary key `(user_id, metric)` is the only index.
- **D-08: Full schema (locked).**
  ```
  user_benchmark_percentiles
    user_id        INTEGER       NOT NULL  FK -> users.id  ON DELETE CASCADE
    metric         benchmark_metric ENUM NOT NULL
    value          DOUBLE PRECISION NOT NULL
    percentile     DOUBLE PRECISION NULL
    n_games        INTEGER NOT NULL
    cdf_snapshot   DATE NOT NULL
    computed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
    PRIMARY KEY (user_id, metric)
  ```

**Locked by ROADMAP / design note:**
- **D-09: Canonical slice = benchmark cohort definition.** `status='completed'` + ±100 ELO opponent (game-time) + sparse-cell `(2400, classical)` exclusion + 36-month recency + standard variant + per-metric inclusion floor. Pooled across TCs.
- **D-10: Per-metric inclusion floors** (verbatim from `scripts/gen_global_percentile_cdf.py:58-61`):
  - `score_gap` — ≥30 endgame AND ≥30 non-endgame games
  - `achievable_score_gap` — ≥20 endgame-entry games
  - `section2_score_gap_conv` — ≥20 spans per entry-eval bucket
  - `section2_score_gap_parity` — ≥20 spans per entry-eval bucket
- **D-11: CTE-sharing mechanism = Option 1 (extract shared module).** Extract `_canonical_selected_users_cte`, `_per_user_cte_score_gap`, `_per_user_cte_achievable`, `_per_user_cte_section2`, `_equal_footing_filter_sql`, `_sparse_exclusion_sql`, `_elo_bucket_expr` from `scripts/gen_global_percentile_cdf.py` into `app/services/canonical_slice_sql.py`. Refactor `gen_global_percentile_cdf.py` to consume.
- **D-12: Chip read path rewire.** Phase 94's per-request `interpolate_percentile(metric_id, value)` calls inside `app/services/endgame_service.py` are removed; the 4 `{metric}_percentile` fields read `(value, percentile)` from `user_benchmark_percentiles` for the current `user_id`, ignoring filter state.
- **D-13: Tooltip copy update.** Add canonical-slice framing to existing chip popover copy. Within the discipline of `feedback_popover_copy_minimalism.md`.
- **D-14: Backfill script.** `scripts/backfill_user_percentiles.py --target dev|prod` (+ optional `--user-id`, `--metric`). Idempotent UPSERT.

### Pattern 1: SQLAlchemy Async UPSERT with ON CONFLICT DO UPDATE

**What:** The idiom for "upsert exactly one row per natural key" using PostgreSQL's `INSERT ... ON CONFLICT (...) DO UPDATE`. Already in production at `app/repositories/game_repository.py:30` (DO NOTHING variant).

**When to use:** Stage A / Stage B / backfill — each recompute overwrites the prior `(user_id, metric)` row in place.

**Example (extending `app/repositories/game_repository.py:30` pattern):**
```python
# Source: project precedent — game_repository.py:11-35 (do_nothing)
#                            extends to do_update per SQLAlchemy 2.x docs
from sqlalchemy.dialects.postgresql import insert as pg_insert
from sqlalchemy.ext.asyncio import AsyncSession

from app.models.user_benchmark_percentile import UserBenchmarkPercentile

async def upsert_percentile(
    session: AsyncSession,
    *,
    user_id: int,
    metric: str,           # one of the 4 ENUM values
    value: float,
    percentile: float | None,
    n_games: int,
    cdf_snapshot: date,
) -> None:
    stmt = pg_insert(UserBenchmarkPercentile).values(
        user_id=user_id,
        metric=metric,
        value=value,
        percentile=percentile,
        n_games=n_games,
        cdf_snapshot=cdf_snapshot,
    )
    stmt = stmt.on_conflict_do_update(
        index_elements=["user_id", "metric"],
        set_={
            "value": stmt.excluded.value,
            "percentile": stmt.excluded.percentile,
            "n_games": stmt.excluded.n_games,
            "cdf_snapshot": stmt.excluded.cdf_snapshot,
            "computed_at": func.now(),   # server-side NOW() refresh
        },
    )
    await session.execute(stmt)
```

### Pattern 2: Postgres ENUM via Alembic with `create_type=False`

**What:** Project precedent for creating a Postgres ENUM type alongside a column. Lifecycle is controlled by the migration (`enum.create(op.get_bind(), checkfirst=True)` on upgrade; `enum.drop(...)` on downgrade); the model uses `SAEnum(..., create_type=False)` so SQLAlchemy never tries to create or drop the type itself.

**Source:** `alembic/versions/20260408_155149_3e4018d62102_convert_result_user_color_termination_.py` (the same migration that introduced `gameresult`, `color`, `termination`, `timecontrolbucket` enums to the `games` table) — and `app/models/game.py` lines 21-40 for the model side.

**When to use:** Phase 94.1's `benchmark_metric` ENUM (D-05).

**Example (composed from the two precedents):**
```python
# Source: alembic/versions/20260408_155149_3e4018d62102_*.py + app/models/game.py
# In the new migration (e.g. 20260524_*_add_user_benchmark_percentiles.py):

from sqlalchemy.dialects import postgresql

benchmark_metric_enum = postgresql.ENUM(
    "score_gap",
    "achievable_score_gap",
    "section2_score_gap_conv",
    "section2_score_gap_parity",
    name="benchmark_metric",
    create_type=False,
)

def upgrade() -> None:
    # Step 1: create the type *before* the table references it
    benchmark_metric_enum.create(op.get_bind(), checkfirst=True)

    # Step 2: create the table
    op.create_table(
        "user_benchmark_percentiles",
        sa.Column("user_id", sa.Integer(),
                  sa.ForeignKey("users.id", ondelete="CASCADE"), nullable=False),
        sa.Column("metric", benchmark_metric_enum, nullable=False),
        sa.Column("value", sa.Float(), nullable=False),       # DOUBLE PRECISION
        sa.Column("percentile", sa.Float(), nullable=True),   # DOUBLE PRECISION NULL
        sa.Column("n_games", sa.Integer(), nullable=False),
        sa.Column("cdf_snapshot", sa.Date(), nullable=False),
        sa.Column("computed_at", sa.DateTime(timezone=True),
                  nullable=False, server_default=sa.func.now()),
        sa.PrimaryKeyConstraint("user_id", "metric"),
    )

def downgrade() -> None:
    # Order matters: drop table THEN drop the type (the type cannot be
    # dropped while a column still references it)
    op.drop_table("user_benchmark_percentiles")
    benchmark_metric_enum.drop(op.get_bind(), checkfirst=True)
```

### Pattern 3: Background-Task Fan-Out via `asyncio.create_task` with Own Session

**What:** Fire-and-forget a background coroutine that opens its own short-lived session. CLAUDE.md's hard rule: never share `AsyncSession` across coroutines. The codebase already does this in three places: `app/routers/imports.py:69` (`asyncio.create_task(import_service.run_import(job_id))`), and each helper in `app/services/import_service.py` opens `async with async_session_maker()` per scope (bootstrap, per-batch, completion).

**When to use:** Stage A hook in `_complete_import_job`; Stage B hook after `_mark_evals_completed` commit; backfill script per-user iteration.

**Example:**
```python
# Source: app/routers/imports.py:69 + app/services/import_service.py:489-499
import asyncio
import sentry_sdk

from app.core.database import async_session_maker

async def compute_stage_a(user_id: int) -> None:
    """Background task — never propagates exceptions back to trigger site."""
    try:
        async with async_session_maker() as session:
            # 1) run the score_gap canonical-slice CTE for this user
            # 2) interpolate_percentile("score_gap", value)
            # 3) UPSERT user_benchmark_percentiles
            ...
            await session.commit()
    except asyncio.CancelledError:
        raise  # lifespan shutdown contract (WR-07 / eval_drain.py:550-554)
    except Exception as exc:
        # CLAUDE.md Backend Rules: capture with set_context, never variable in message
        sentry_sdk.set_context("percentile_compute",
                               {"user_id": user_id, "stage": "A"})
        sentry_sdk.capture_exception(exc)
```

```python
  avg(metric_value) AS pooled_value,        # collapses per-(user, cell) to one
```

### Anti-Patterns to Avoid

- **Sharing one `AsyncSession` between Stage A trigger and Stage A compute** — explicit CLAUDE.md hard rule. Open a fresh session inside `compute_stage_a()`.
- **Letting Stage A / Stage B exceptions propagate** — would crash the import worker or eval-drain coroutine. Wrap each in `try/except Exception` + Sentry capture (D-04).
- **Embedding `user_id` in error messages** — fragments Sentry grouping. Use `sentry_sdk.set_context("percentile_compute", {...})` per CLAUDE.md.
- **Running Stage A inside the import transaction** — explicit ROADMAP success criterion 3. Hook MUST be after `session.commit()` at line 499.
- **Computing percentile inside the request handler** — D-12 removes the 4 per-request `interpolate_percentile` calls from `endgame_service.py`. The Phase 94 helper itself stays (still used by the new compute service), only the call sites move.
- **Variable-name shadowing of the existing `score_gap_percentile`-style field names** — D-12 keeps the same `{metric}_percentile` field names on the response model; only the data source changes (from helper call to table read). FE contract is unchanged.
- **Per-batched commit in the backfill script** — recommend per-user commit (CONTEXT.md "Claude's Discretion" recommendation). Simpler error recovery, idempotent, no risk of a half-applied batch on Ctrl-C.

## Common Pitfalls

### Pitfall 1: Stage B fires too often (or too rarely)

**What goes wrong:** Either Stage B fires after every drain batch for every active user (wasted work; same percentile recomputed dozens of times during a long import) OR it never fires (because the user's pending-eval count never reaches zero in a single batch).

**Why it happens:** The `count(*) WHERE evals_completed_at IS NULL` check must be done *per affected user* (not globally) and the trigger condition is "this user's count just transitioned to zero" — not "this user has any zero-pending state". A user with 500 games in the drain queue will have their count read as `> 0` for many drain ticks before reaching zero exactly once.

**How to avoid:** Run the count query *after* `_mark_evals_completed` commits, group `game_ids` by `user_id` first, query count for each affected user, fire `create_task(compute_stage_b)` only for users whose count is now exactly zero. The transition-detection is automatic because the count drops to zero exactly once per import-completion-from-the-eval-side.

**Warning signs:** `compute_stage_b` invoked multiple times for the same user in a short window; Sentry duplicate-context events.

### Pitfall 2: Stage B fires before Stage A has run for a brand-new user

**What goes wrong:** A user's first import completes, Stage A fires for `score_gap`; before Stage A's UPSERT commits, the eval-drain batch processes their endgame-entry plies and Stage B fires for the 3 eval-dependent metrics. The two writes race.

**Why it happens:** Both stages are `asyncio.create_task` background jobs. They operate on different `(user_id, metric)` PK rows so there's no conflict at the table level, but ordering is not guaranteed.

**How to avoid:** No prevention needed at the data layer — the PK is `(user_id, metric)`, the 4 metrics are disjoint, and each stage writes exactly the metrics it owns. The races are benign.

**Warning signs:** None — this is actually the correct behaviour. Document it in `compute_stage_a` / `compute_stage_b` docstrings.

### Pitfall 3: ENUM column rejects values that should pass

**What goes wrong:** Inserting `metric='score_gap'` fails with `invalid input value for enum benchmark_metric`.

**Why it happens:** The Alembic migration uses `create_type=False` on the model side but the type wasn't actually created during `upgrade()` (forgot `enum.create(op.get_bind(), checkfirst=True)`). OR the migration ran on dev but not on prod (CI/CD didn't run alembic upgrade).

**How to avoid:** (a) In `upgrade()`, call `benchmark_metric_enum.create(op.get_bind(), checkfirst=True)` BEFORE `op.create_table(...)`. (b) Verify with a single pytest assertion against `pg_enum` after migration runs (CONTEXT.md discretion item — recommended).

**Warning signs:** `psycopg2.errors.InvalidTextRepresentation` or `asyncpg.exceptions.InvalidTextRepresentationError` on first UPSERT.

### Pitfall 4: Downgrade fails because table still references type

**What goes wrong:** `op.execute("DROP TYPE benchmark_metric")` fails with `cannot drop type benchmark_metric because column ... depends on it`.

**Why it happens:** The downgrade order is wrong — must drop the table FIRST, then drop the type.

**How to avoid:** Order in `downgrade()`:
```python
op.drop_table("user_benchmark_percentiles")
benchmark_metric_enum.drop(op.get_bind(), checkfirst=True)
```

**Warning signs:** Migration test `pytest tests/integration/test_alembic_migrations.py` fails on downgrade.

### Pitfall 5: `compute_stage_b` runs before the user has any endgame-entry evals

**What goes wrong:** A user with import-completed but no Stockfish evaluation triggers Stage B; the canonical-slice query returns zero rows for the 3 eval-dependent metrics; `compute_stage_b` writes nothing OR writes `n_games=0` rows.

**Why it happens:** Stage B trigger condition is "count of `evals_completed_at IS NULL` is zero". For a user with no games requiring eval (e.g., chess.com import where lichess `%eval` headers don't apply), the count is `0` from the start.

**How to avoid:** Stage B compute service handles "zero canonical-slice games" cleanly — emit no row for that metric (per CONTEXT.md Claude's Discretion: "if `value` itself isn't computable (zero games in slice), no row"). The trigger semantic is still correct.

**Warning signs:** Sentry events with `n_games=0` and "zero canonical-slice games" log line.

### Pitfall 6: Backfill script connects to wrong DB

**What goes wrong:** `--target prod` connects to dev (or vice versa) because the env-var override `BACKFILL_PROD_DB_URL` is misconfigured.

**Why it happens:** The script reuses `_db_url` from `scripts/backfill_eval.py` shape. That script has a safety guard refusing to run if the resolved URL doesn't match expected tokens.

**How to avoid:** Copy `scripts/backfill_eval.py`'s `_assert_*_db` safety guard verbatim:
- `--target dev`: refuse unless URL contains `localhost:5432`
- `--target prod`: refuse unless URL contains `localhost:15432` AND `bin/prod_db_tunnel.sh` is up (use `curl` or `nc` to verify the tunnel)

**Warning signs:** `Refusing to run: resolved DB URL does not contain...` startup error.

### Pitfall 7: Tooltip copy gets verbose

**What goes wrong:** The new canonical-slice framing turns the popover into a paragraph of methodology; users skim past it.

**Why it happens:** Tempting to explain "canonical slice = status='completed' + ±100 ELO + 36-month + sparse-cell + pooled-TC".

**How to avoid:** Per `feedback_popover_copy_minimalism.md`: WHAT + sign convention only, no jargon, no caveats. The current Phase 94 copy is 1-2 sentences (see PercentileChip.tsx:71-86). The D-13 update must stay at that length — add ONE phrase clarifying "your career under canonical conditions (not the current filtered view)".

**Warning signs:** Popover overflows the Radix `Content` max-width; user complaints in HUMAN-UAT.

### Pitfall 8: Per-user CTE drift from benchmark CTE after extraction

**What goes wrong:** Six months from now someone edits `_per_user_cte_score_gap` for the benchmark path and forgets the per-user consumer; the chip silently starts reading a different slice from the published CDF.

**Why it happens:** D-11 was chosen precisely to prevent this, but the human protection is shaky — both consumers should be exercised by a test on every CI run.

**How to avoid:** Add a unit test `tests/services/test_canonical_slice_sql.py` that:
1. Builds both consumer queries for each metric and asserts the SQL string of the *shared fragments* (`_equal_footing_filter_sql`, `_sparse_exclusion_sql`, `_elo_bucket_expr`) is byte-identical between consumers.
2. Builds the full benchmark-source query and full single-user-source query and asserts the only differences are the documented two (selected_users join + per-TC equality).

**Warning signs:** A future PR diff touches `canonical_slice_sql.py` but the test still passes — verify the test actually exercises every fragment.

## Open Questions (RESOLVED)

1. **`cdf_snapshot` semantic**
   - What we know: column is `DATE NOT NULL`, "records which `GLOBAL_PERCENTILE_CDF` revision the percentile was looked up against" (PCTL-08).
   - What's unclear: is the value the benchmark DB snapshot month (`BENCHMARK_DB_SNAPSHOT_MONTH = "2026-03"` → `date(2026, 3, 1)` or `date(2026, 3, 31)`), or the compute-time date (`date.today()`)? CONTEXT.md D-07 example value is `2026-05-23` which is a compute date.

   - Implemented by Plan 05 (94.1-05-PLAN.md — `compute_stage_a` / `compute_stage_b` pass `cdf_snapshot=date.today()` to `upsert_percentile`).

2. **Per-user pooled value formula confirmation**
   - What we know: benchmark CDF calls `percentile_cont(...) WITHIN GROUP (ORDER BY metric_value) FROM per_user_values` where `per_user_values` has one row per `(user, elo_bucket, tc_bucket)` cell that cleared the floor.
   - What's unclear: when a single user spans multiple `(elo_bucket, tc_bucket)` cells (e.g., 1600-blitz + 2000-rapid because their rating climbed), should the per-user pooled value be:
     - (a) `avg(metric_value)` across cells (each cell weighted equally), OR
     - (b) weighted-`avg(metric_value)` by `n_games_in_cell`?
   - **RESOLVED:** (a) unweighted, matching the benchmark's `percentile_cont` which treats each `per_user_values` row equally regardless of underlying `n_games`. Verify in PLAN.md by inspecting the exact `per_user_values` shape.
   - Implemented by Plan 05 (94.1-05-PLAN.md — `_compute_metric_for_user` does `SELECT avg(metric_value) FROM per_user_values`, unweighted across cells).

3. **Stage B trigger when only some metrics' inclusion floors pass**

   - What's unclear: does Stage B write 2 rows (achievable + conv with percentile non-null, parity with `percentile=NULL`) or 3 rows or 1 row?

### Known Threat Patterns for {Python / FastAPI / SQLAlchemy async / Postgres} stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SQL injection via composed CTE string | Tampering | Use parameter binding (`text("... WHERE u.id = :user_id").bindparams(user_id=user_id)`); never f-string user-controlled values into the SQL fragments in `canonical_slice_sql.py`. The shared module builds *static* SQL — only the `user_id` is parameterised. |
| Information disclosure via cross-user percentile read | Information Disclosure | Service-layer SELECT MUST filter by the authenticated user's `user_id`, never accept it as a query param. |
| Race-condition double-UPSERT (Stage A + Stage B for the same `(user_id, metric)`) | Tampering | Not actually a risk — the 4 metrics are partitioned between stages, so no two stages write the same PK. Document in compute service docstring. |
| Backfill script connects to wrong DB | Tampering | `_assert_target_safe` guard refuses on URL mismatch (Pattern 4). |
| Sentry message contains user PII | Information Disclosure | Use `set_context("percentile_compute", {"user_id": user_id, "stage": "A"})` per CLAUDE.md, never embed `user_id` in the exception message string (fragments grouping AND leaks). |
