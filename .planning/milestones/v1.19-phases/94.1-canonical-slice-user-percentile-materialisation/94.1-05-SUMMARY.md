# Phase 94.1 Plan 05: Compute Orchestrator (Stage A + Stage B) Summary

**One-liner:** Per-user canonical-slice compute service with dual-query floor detection, Sentry-isolated error handling, and injectable session factory for Plan 08 backfill.

## What Was Built

`app/services/user_benchmark_percentiles_service.py` — the single non-trivial behavioral file in Phase 94.1. Exports:

- `STAGE_A_METRIC: CdfMetricId = "score_gap"` — constant identifying Stage A's metric
- `STAGE_B_METRICS: tuple[CdfMetricId, ...] = ("achievable_score_gap", "section2_score_gap_conv", "section2_score_gap_parity")` — Stage B's three eval-dependent metrics
- `compute_stage_a(user_id, *, session_maker=None)` — hooked by Plan 06 into `_complete_import_job`
- `compute_stage_b(user_id, *, session_maker=None)` — hooked by Plan 06 into `eval_drain.py`

Internal helper (not exported): `_compute_metric_for_user(session, user_id, metric_id) -> tuple[float | None, int]`

## Pooled-Value Formula

**Query 1 (raw value):** `WITH selected_users AS (SELECT :user_id::int AS user_id), <per_user_cte_for(metric_id, source="single_user", apply_floor=False)> SELECT avg(metric_value)::float AS value_raw FROM per_user_values`

Produces `avg(metric_value)` across ALL `(elo_bucket, tc_bucket)` cells regardless of floor. Returns None when the user has zero canonical-slice games.

**Query 2 (floor check):** Same CTE chain with `apply_floor=True`, outer `SELECT count(*)::int AS n_cells_floor FROM per_user_values`

Returns count of cells that passed the per-metric HAVING gate (0 = below floor, > 0 = at/above floor).

This two-query approach avoids modifying `canonical_slice_sql.py` (which would break the byte-identical regression gate in `tests/scripts/test_gen_global_percentile_cdf_unchanged.py`) and avoids CTE name conflicts from a dual-CTE single-query design.

## Below-Floor Representation

Chosen convention (CONTEXT.md Claude's Discretion):

- `value_raw IS NOT None AND n_cells_floor == 0` → write row with `percentile=NULL`, `value=value_raw`, `n_games=0`
- `value_raw IS None` (zero canonical-slice games) → write no row

This allows future floor-change without recompute (the value is already stored). The `n_games` column stores the floor-passing cell count, not the raw game count (see Decisions).

## Sentry Capture Pattern

Three capture sites per CLAUDE.md Backend Rules:

- Stage A outer `except Exception`: `set_context("percentile_compute", {"user_id": user_id, "stage": "A"})` + `capture_exception(exc)`
- Stage B per-metric inner `except Exception`: `set_context(..., {"user_id": ..., "stage": "B", "metric": metric_id})` + `capture_exception(exc)`
- Stage B outer `except Exception`: `set_context(..., {"user_id": ..., "stage": "B"})` + `capture_exception(exc)`

No `user_id` embedded in exception message strings (V4 — T-94.1-11). `asyncio.CancelledError` re-raised in all 3 sites (lifespan shutdown contract, eval_drain.py:550-554).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Design Gap] Dual-CTE single-query approach replaced with two separate queries**

- **Found during:** Task 1 implementation
- **Issue:** The plan described a single-query dual-CTE approach (`WITH selected_users AS (...), <cte_raw> AS per_user_values_raw, <cte_floor> AS per_user_values_floor ...`). This is not possible: `per_user_cte_for` returns a multi-CTE string with shared internal names (`endgame_game_ids`, `rows`, `per_user`, `per_user_values`). Including both raw and floor variants in one `WITH` clause causes CTE name conflicts.
- **Fix:** `_compute_metric_for_user` runs two separate SQL executions — one for `value_raw` (`apply_floor=False`) and one for `n_cells_floor` (`apply_floor=True`). Both use `text(sql).bindparams(user_id=user_id)` (V5 security preserved).
- **Files modified:** `app/services/user_benchmark_percentiles_service.py` (new file, design adjustment)
- **Commit:** 3020dc35

**2. [Rule 1 - Design Gap] n_games stores floor-passing cell count instead of raw game count**

- **Found during:** Task 1 implementation
- **Issue:** The plan implied `n_games_floor` as an actual game count. `canonical_slice_sql.py`'s `per_user_values` CTE only exposes `(metric_value, elo_bucket, tc_bucket)` — no game count column. Adding one would modify the byte-identical CTE output, breaking `test_gen_global_percentile_cdf_unchanged.py`'s regression gate (ROADMAP SC-7 invariant).
- **Fix:** `n_games` stores `n_cells_floor` (count of floor-passing `(elo, tc)` cells). Semantics: 0 = below floor, > 0 = at/above floor. This is correct for the `percentile=NULL` decision and stored as a "sample size proxy at compute time". Documented in module docstring and here.
- **Impact:** The `n_games` column in `user_benchmark_percentiles` reads as cell-count not game-count. Future plans that need actual game count can add a `per_user_values` column to `canonical_slice_sql.py` separately (as a non-breaking change if they don't touch the benchmark CTE output).
- **Commit:** 3020dc35

**3. [Rule 1 - Design Gap] Floor check uses n_cells_floor > 0 instead of >= SCORE_GAP_MIN_ENDGAME_N**

- **Found during:** Task 1 implementation
- **Issue:** Plan's Stage A floor check `if n_cells_floor >= SCORE_GAP_MIN_ENDGAME_N` assumes `n_cells_floor` is a game count. It's actually a cell count (max a few cells per user), so `>= 30` would always be False.
- **Fix:** Floor check is `if n_cells_floor == 0: percentile = None`. If any cell passes the HAVING floor, the user has passed the inclusion floor (the HAVING clause enforces ≥ 30 endgame + ≥ 30 non-endgame). The Python check is a confirmation, not a recount.
- **Commit:** 3020dc35
