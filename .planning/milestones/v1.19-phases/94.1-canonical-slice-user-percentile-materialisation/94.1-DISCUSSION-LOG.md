## Stage B trigger semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Per-batch group-by-user check | After each drain batch commit, group just-drained game_ids by user_id, query pending count per affected user, fire Stage B for users at zero. Low latency (~drain tick). 1 count query per affected user per batch. | ✓ |
| Periodic sweeper coroutine | Separate coroutine ticks every N min, scans for users whose pending eval count = 0 AND whose user_benchmark_percentiles.computed_at is stale. Self-healing. Higher latency. | |
| Hybrid: per-batch primary + sweeper safety net | Per-batch as primary, sweeper as safety net. Belt-and-braces but two trigger paths. | |
| users.stage_b_pending flag column | Boolean on users, set TRUE at import, cleared by drain when pending hits zero, watcher coroutine reacts to transition. Cleanest event semantics but adds schema migration + third coroutine. | |

**User's choice:** "I'll let you pick. To me, Per-batch group-by-user check sounds good."
**Notes:** Claude committed to per-batch alone (not the hybrid) — the backfill script (`scripts/backfill_user_percentiles.py --target prod`) is the recovery path for Stage B failures, eliminating the need for a sweeper coroutine. Captured as D-01 + D-02. The hook site (`app/services/eval_drain.py:547-548`, after `_mark_evals_completed` + commit) and Stage A's parallel hook site (`app/services/import_service.py:480`, after `_complete_import_job`'s commit) are documented in D-01 + D-03.

## Schema type details

### metric column type

| Option | Description | Selected |
|--------|-------------|----------|
| VARCHAR + CHECK constraint | VARCHAR(64) with CHECK IN (...). Easy to add new metric IDs via Alembic. | |
| Postgres ENUM type | CREATE TYPE benchmark_metric AS ENUM (...). Stricter; ALTER TYPE ADD VALUE for new metrics. Project's first ENUM use. | ✓ |
| Plain VARCHAR, no DB constraint | Trust Python-side Literal for validation; no CHECK. | |

**User's choice:** Postgres ENUM type
**Notes:** Captured as D-05. Note for planner: codebase doesn't use Postgres ENUMs elsewhere — deliberate first-use; document the pattern at the migration. Adding future metrics requires a two-step (Python Literal update + ALTER TYPE migration), well-trodden in Alembic.

### percentile column type

| Option | Description | Selected |
|--------|-------------|----------|
| SMALLINT NULL (1..99) | Matches design note + chip "top X%" rounding. 2 bytes. | |
| DOUBLE PRECISION NULL | Matches Phase 94's API return type float \| None. Preserves sub-integer precision from interpolate_percentile(). 8 bytes. | ✓ |
| SMALLINT with no range constraint | Cheaper migration; allows out-of-band writes. | |

**User's choice:** DOUBLE PRECISION NULL
**Notes:** Captured as D-06. Resolves a potential precision-mismatch between Phase 94's API contract (`float | None`) and the materialised storage. The design note's `SMALLINT` recommendation is superseded — the chip still rounds for display, but the float is preserved for the Phase 95 LLM payload and any future viz.

### cdf_snapshot column type + index

| Option | Description | Selected |
|--------|-------------|----------|
| DATE + no extra index | DATE matches design note. Re-lookup script does full table scan — fine at v1.19 scale. | ✓ |
| DATE + secondary index on cdf_snapshot | Cheaper re-lookup filter. Small storage cost. | |
| TEXT (version string / git SHA) | Survives same-day re-runs of gen_global_percentile_cdf.py. Slightly more complex. | |

**User's choice:** DATE + no extra index
**Notes:** Captured as D-07. Same-day CDF re-runs are unlikely in v1.19; the manual `gen_global_percentile_cdf.py` invocation cadence is "occasional, ops-driven". Revisit if the re-lookup-after-CDF-regeneration becomes scheduled.

## Claude's Discretion

Areas where the user explicitly delegated to Claude, or that defaulted to the design note's preferred path:

- **CTE-sharing mechanism** — defaulted to Option 1 (shared module path `app/services/canonical_slice_sql.py`). The design note explicitly preferred Option 1 ("the more durable choice"); the user did not pick this for re-discussion, so Option 1 stands. Captured as D-11.
- **Backfill operational details** — user ordering, batched commits, n_games=0 skip strategy, progress reporting. Recommendations captured in CONTEXT.md "Claude's Discretion" section (creation-asc ordering, per-user commit, LEFT JOIN pre-skip, INFO-level per-user logging).
- **Shared-module file path + per-user compute service location** — recommendations only; planner can override.
- **Inclusion-floor failure representation** — recommend `percentile=NULL` when below floor but `value` computable; no row when `value` itself isn't computable.
- **Stage A / Stage B background-task error handling** — recommend `try/except Exception` + Sentry capture, never propagate to trigger site.
