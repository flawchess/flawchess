# Phase 94.4 Plan 05b: Percentile Service Cohort-CDF Cutover Summary

## One-liner

Rewrites `user_benchmark_percentiles_service.py` to compute per-(user, TC) anchors first (Lichess-precedence + chesscom_raw_rating capture) and then per-(metric, TC) percentiles via `interpolate_cohort_percentile`; retires the post-94.3 12-tuple STAGE_B_METRICS and the legacy `interpolate_percentile` stub.

## What was built

This is the middle plan of the Wave 2 atomic-cutover cluster (05a → 05b → 05c, D-10a). 05a shipped the schema reshape; this plan rewrites the service that writes against the new schema. 05c follows with the API shaper + frontend types; the three plans land together in one squash-merged PR with no intermediate deploy.

### 1. Service rewrite (`app/services/user_benchmark_percentiles_service.py`)

**`compute_anchors_for_user(session, user_id) -> dict[TimeControlBucket, RatingAnchorRow]`** — new helper implementing D-12 Lichess-precedence + D-07 bullet 4 raw-rating capture:

1. For each TC in `(bullet, blitz, rapid, classical)`:
   - Probe `_user_has_lichess_games_in_tc` (single existence query against `games`).
   - If yes → `platform='lichess'`; call `per_user_cte_median_anchor(tc, source='single_user', platform='lichess')`. `chesscom_raw_rating=None`.
   - If no → `platform='chesscom'`; call the same builder with `platform='chesscom'` to read the raw chess.com median. Capture as `chesscom_raw_rating` BEFORE conversion (this is the value the Plan 07 tooltip discloses). Then route through `convert_chesscom_to_lichess(raw, source_tc=mapped_tc, target_tc=tc)` to produce the Lichess-equivalent that becomes the persisted `anchor_rating`. If conversion returns None (Daily, or out-of-range), skip the TC — chip suppresses naturally.
2. `upsert_anchor` writes the row; the helper returns a `dict[TC, RatingAnchorRow]` for the percentile loop.
3. Per-TC exceptions trip an inner `try/except` that captures to Sentry with `set_context("percentile_compute", {"user_id", "stage": "anchor", "tc"})` and continues with the next TC.

**`compute_stage_a`** — rewritten per RESEARCH Pitfall 9 ordering: anchors FIRST, then per-TC `score_gap` percentiles. Each TC's percentile is interpolated against the cohort CDF at `(metric='score_gap', anchor=<Lichess-equivalent>, tc)` and upserted with the widened 3-column PK including `time_control_bucket`. Sentry context carries `{user_id, stage='A', metric, tc}` on per-TC errors.

**`compute_stage_b`** — rewritten to iterate `STAGE_B_METRIC_FAMILIES` (7-tuple) × the user's above-floor anchored TCs. Anchors are fetched once at the top via `fetch_anchors_for_user`; if no anchors exist Stage B is a no-op. Each (family, tc) cell dispatches through `_compute_metric_for_user_per_tc` which builds the per-TC SQL via `_per_user_cte_for_family_and_tc` (mirrors `scripts/gen_global_percentile_cdf.py::_per_user_cte_for_metric_and_tc` for D-10 source-mode parity). Per-cell errors are isolated; the Sentry context carries `{user_id, stage='B', metric, tc}`.

**Retirements**:

- `STAGE_B_METRICS` 12-tuple of TC-suffixed names → gone.
- `_compute_metric_for_user` (pooled-across-TCs single-row helper) → replaced by `_compute_metric_for_user_per_tc` (per-TC family-dispatched).
- Import of `interpolate_percentile` → gone; the import + every call site now uses `interpolate_cohort_percentile`.

**TC → ChessComSourceTC mapping** — added at module level. `'classical'` maps to `'daily'` so chess.com classical inputs route through `convert_chesscom_to_lichess(..., source_tc='daily')` which returns None (Daily has no ChessGoals mapping, RESEARCH Pitfall 11). The mapping is a `dict[TimeControlBucket, ChessComSourceTC]` rather than an inline branch to keep the suppression policy at one place.

### 3. Bug fix (`app/services/canonical_slice_sql.py`) — Rule 1 auto-fix

Plan 02's `per_user_cte_median_anchor` accepted `platform: Literal['lichess', 'chesscom']` and emitted `AND g.platform = '{platform}'` directly. But the `games` table stores chess.com as `'chess.com'` (with the dot, per `app/services/normalization.py:261`). The bug was latent — the benchmark DB (Plan 04's only consumer) is Lichess-only, so the `'chesscom'` branch was never exercised. Fix: translate the literal at SQL-emit time (`'chesscom'` → `'chess.com'`; `'lichess'` unchanged). Inline comment documents the discrepancy and where storage normalisation happens.
