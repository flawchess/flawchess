# Phase 94.2 Plan 03: Pooled per-user compute service rewrite — Summary

Close the cross-plan red from 94.2 wave 1: rewrite the per-user compute service against the pooled SQL surface introduced in Plan 01, the regenerated CDF from Plan 02, and re-add the `n_games` column (D-9-amend correction to D-9) so the binding inclusion-floor count threads from the pooled CTE all the way to the storage row.

## What changed

### Schema (new Alembic migration `b8f4d92c1e25`)

- `op.add_column("user_benchmark_percentiles", sa.Column("n_games", sa.Integer(), nullable=False, server_default="0"))`.
- Downgrade drops the column (data-lossy, acceptable — Plan 06's backfill repopulates).
- Round-trip verified locally: `upgrade head` → `downgrade -1` → `upgrade head`, all clean against dev DB.

### ORM (`app/models/user_benchmark_percentile.py`)

- New `n_games: Mapped[int] = mapped_column(Integer, nullable=False)`.
- Module docstring updated: replaces the stale "n_cells_floor dropped" line with the n_games re-add history (94.1-04 add → 94.1-11 rename → 94.1-13 drop → 94.2-03 re-add under pooled semantics).
- Class docstring documents per-metric n_games meaning under the pooled model: `score_gap` → endgame games count; `achievable_score_gap` → endgame-entry-with-d_i count; `section2_score_gap_conv` / `section2_score_gap_parity` → bucket span counts.

### Compute service (`app/services/user_benchmark_percentiles_service.py`)

- `_compute_metric_for_user` rewritten to consume the Plan 01 pooled SQL surface:
  - Drops the `apply_floor=True` kwarg from `per_user_cte_for` (the parameter is gone post-Plan-01).
  - SQL becomes `WITH selected_users + per_user_values → SELECT metric_value::float, n_games::int FROM per_user_values LIMIT 1` — no `avg(metric_value)` wrapper because `per_user_values` already emits one row per user under the pooled model.
  - Return type widened from `float | None` to `tuple[float, int] | None` per D-9-amend and RESEARCH Pitfall 4.
  - V5 bindparam preserved: `text(sql).bindparams(user_id=user_id)` (load-bearing per 94.1 Plan 09 fix).
- `compute_stage_a` and `compute_stage_b` destructure the `(value, n_games)` tuple and pass both to `upsert_percentile`. Public signatures `(user_id, *, session_maker=None) -> None` are byte-identical (PCTL-09).
- Module docstring rewritten to describe the Phase 94.2 pooled flow; floor convention paragraphs replaced.

### Repository (`app/repositories/user_benchmark_percentiles_repository.py`)

- `upsert_percentile` gains a required `n_games: int` keyword parameter.
- The INSERT `.values(...)` block includes `n_games=n_games`; `on_conflict_do_update(set_=...)` includes `"n_games": stmt.excluded.n_games` so refresh UPSERTs replace the prior count.
- Docstring updated to document the new parameter under Phase 94.2 (D-9-amend).

### Tests

- `tests/services/test_user_benchmark_percentiles_service.py` rewritten: 16 tests covering the new pooled contract (single-query, no apply_floor/elo_bucket/avg(metric_value), tuple return shape, None on empty/null n_games, signature preservation pinned via `inspect.signature`, Stage A/B fan-out with n_games threading, per-metric isolation, real-DB zero-canonical happy path, Sentry non-propagation for both stages). All 16 pass.
- `tests/repositories/test_user_benchmark_percentiles_repository.py` updated: all five `upsert_percentile` call sites now pass `n_games=`. 6 tests, all pass.
- `tests/services/test_endgame_service_chip_decoupling.py` updated: two `upsert_percentile` call sites now pass `n_games=`. 4 tests, all pass.
- `tests/integration/test_alembic_migrations.py` updated: `_EXPECTED_COLUMNS` frozenset gains `n_games` so the post-upgrade schema inventory matches the new shape. 2 tests, all pass.
- `tests/services/test_user_benchmark_percentiles_service_real_data.py`: removed `test_user_28_shape_floor_respecting_value_excludes_below_floor_cells` and its per-cell helper/constants. The 4 remaining tests (qualifying-user happy path, Stage B happy path, idempotent UPSERT, below-floor no-row) continue to exercise the pooled compute end to end and all pass.
