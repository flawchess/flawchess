## How It Works

The Phase 94.2 pooled methodology computes one metric value per user from
their recent-1000-per-TC × 36-month pool of games, and emits no storage
row at all if the user is below the ≥30 inclusion floor for that metric
(the pooled CTE's HAVING clause filters them out). Under 94.1's per-cell
methodology the same situation was modelled as "no canonical games" — a
phrase that becomes semantically incorrect under pooling (it's not "no
games", it's "pooled count below the floor"). Renaming the summary
counter to `skipped_below_pooled_floor` aligns the operator-facing log
output with the methodology actually executing under the hood.

The backfill script's compute path is unchanged structurally — it still
calls `compute_stage_a` / `compute_stage_b`, which now internally
delegate to the pooled `_compute_metric_for_user` from Plan 03 — so the
script gains the pooled methodology automatically. Only the per-metric
counter and the summary log emission required updates.
