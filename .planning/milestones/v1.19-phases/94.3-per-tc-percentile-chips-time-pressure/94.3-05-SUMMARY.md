## Task 1: Argparse widening

**Change**: `scripts/backfill_user_percentiles.py:512-525`. Replaced the inline 4-tuple with `list((STAGE_A_METRIC, *STAGE_B_METRICS))` so the choice list tracks the service-layer source-of-truth tuple dynamically. The script already imports both names at module load (lines 92-97) for the per-metric summary counters, so the dynamic form is safe with no import cycle risk.

**Verification**:
- `uv run python scripts/backfill_user_percentiles.py --help` lists all 16 metric IDs under `--metric` choices.
- `grep -oE '(time_pressure_score_gap|clock_gap|net_flag_rate)_(bullet|blitz|rapid|classical)' <help output> | sort -u | wc -l` → 12 (all per-TC IDs).
- `uv run ruff check scripts/ + uv run ty check scripts/` → clean.
- `uv run ruff format --check app/ tests/ scripts/` → 191 files already formatted.
- `uv run pytest tests/scripts/test_backfill_user_percentiles.py -x` → 7 passed (existing tests unchanged — the original 4 labels remain a subset of the 16, so `for metric_label in _EXPECTED_METRIC_LABELS` still passes).

**Commit**: `8f27e4e9`.

**Why no drift-guard test**: The dynamic form makes drift impossible (the choices list IS the tuple by construction). A separate test would be redundant — there's no value being asserted that could become stale.

### Cells flagged for Plan F awareness (low / zero dev coverage)

All 3 **classical-TC** cells (`time_pressure_score_gap_classical`, `clock_gap_classical`, `net_flag_rate_classical`) show `upserted=0` for user 2 and have zero rows across the 3 users processed so far. This matches Plan A's candidacy report flagging classical as the thinnest cohort across the user base. Two interpretations:

1. **User 2 specifically does not play enough classical to clear the ≥30 pooled floor.** This is genuine below-floor behavior (the `_compute_metric_for_user` CTE's HAVING clause emits no row); the chip will suppress naturally on Plan F's render.
2. **The pattern likely holds for many users in prod**, since classical is the rarest time control across the FlawChess user base. Plan F's chip-suppression escape hatch (per `94.3-01-SUMMARY.md` "Cells flagged for suppression") may need to apply preemptively to classical cells, or the chip-render path needs to handle the empty-percentile-row case correctly (it already does — `_compute_time_pressure_cards` defaults to `None`, mapping to no chip).

**Recommendation for Plan F / Plan G**: cross-reference the prod backfill summary (Task 3 deferred) against this dev result. If prod also shows `upserted=0` on classical-TC cells for most users, validate that the Plan F frontend suppresses the chip cleanly (no broken UI). No code change required from this plan.

### Why the multi-user run was terminated

19 users × ~6 min/user = ~2 hours of compute, exceeding the autonomous-wave budget. The script is **idempotent** (per its module docstring: "re-running produces no state drift — the UPSERT updates `computed_at` but leaves `value` / `percentile` unchanged when inputs are the same"). The remaining 16 dev users will pick up the 12 new metrics on their next steady-state Stage B drain (eval_drain.py hook). If Plan F's dev smoke-test ever needs full dev coverage, re-running `uv run python scripts/backfill_user_percentiles.py --target dev` against the now-widened argparse is a one-command operation that completes in the background.
