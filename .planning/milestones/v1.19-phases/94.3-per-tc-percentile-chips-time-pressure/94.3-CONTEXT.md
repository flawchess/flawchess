# Phase 94.3: Per-TC Percentile Chips on Time Pressure Cards — Context

**Gathered:** 2026-05-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the Phase 94.2 pooled-per-user percentile contract to the Time Pressure section. Where Phase 94's 4 ΔES chips are global-pooled (one CDF per metric across all TCs), the Time Pressure surface is **per-TC by construction** (cards are bullet / blitz / rapid / classical) and the user-meaningful comparison is "top X% of *bullet* players", not "top X% of all players". Add 12 new metrics — `{time_pressure_score_gap, clock_gap, net_flag_rate} × {bullet, blitz, rapid, classical}` — each computed via Phase 94.2's pooled-per-user methodology parameterised by TC.

Three new per-TC pooled-aggregate SQL builder families added to `app/services/canonical_slice_sql.py`, consumed by both `scripts/gen_global_percentile_cdf.py` (12 new CDF keys) and `app/services/user_benchmark_percentiles_service.py` (12 new compute calls). Storage table shape (`user_id, metric, value, percentile, n_games, cdf_snapshot, computed_at`) is unchanged — the `benchmark_metric` ENUM gains 12 new values via a reversible Alembic migration (mirrors the Phase 94.2 D-9-amend pattern). All 12 metrics run in Stage B (post-cold-drain). Backfill extended for one-shot population.

Frontend reuses `PercentileChip` verbatim (existing component, banded color + flame stack + Radix popover shell) with one new prop to handle the Net Flag Rate direction inversion. Three chip slots on each of the 4 `TimePressureTcCard` instances; the per-quintile bullet chart beneath each chip is unchanged.

Time Pressure Score Gap is a **binary collapse** of the existing per-quintile bullet chart — `score(user-pressured games) − score(opp-pressured games)` where pressured = `clock_pct < 40%` at endgame entry — chosen over a per-(TC, quintile) decomposition to avoid mixed-construct CDFs from inclusion-floor variance.

**Locked carryover from Phase 94.2 (no schema redesign):** `user_benchmark_percentiles` table shape + UPSERT contract, Stage A/B trigger pattern, backfill script harness (`scripts/backfill_user_percentiles.py --target dev|prod`), shared `app/services/canonical_slice_sql.py` module, `interpolate_percentile(metric_id, value)` helper, single ≥30 inclusion floor on the pooled set, suppression-below-floor (no row stored), `PercentileChip` component + 4-bullet tooltip disclosure mandate per `feedback_percentile_chip_tooltip_disclosure`.

**Out of scope:** Phase 95 LLM payload + prompt rework. Per-type / Opening Insights percentile chips. Per-(TC, ELO) percentile cells (94.2 D-7 stance — rating accepted as honest, ELO pooled within TC). Per-(TC, quintile) decomposition (binary collapse explicitly chosen).

## Implementation Decisions

### Chip placement on TimePressureTcCard (discussed)

- **D-1: Per-row chip placement matches Phase 94's pattern.** Three chip slots per card:
  - **Clock Gap chip** — on the existing `ClockGapHeaderRow` (3-column "You / Gap / Opp" row), right-aligned next to the centre Gap value. The chip annotates the same metric (`mean_diff_pct`) that the row already displays.
  - **Net Flag Rate chip** — on the existing `NetFlagRateRow`, right-aligned next to the value + info popover trigger.
  - **Time Pressure Score Gap chip** — on the Quintiles subtitle line ("Score by time-pressure"), right-aligned. The chip summarises the chart beneath it; no new metric row introduced. The chart itself stays unannotated per the seed's "chip = headline trait, chart = breakdown" framing.

  Mobile parity (375px): chips right-align on the same row as their metric row when there's space; planner decides wrap behaviour at narrow widths (acceptable to wrap to a second line on small screens as long as the chip stays visually associated with its metric label).

### Net Flag Rate chip direction inversion (discussed)

- **D-2: Direction prop on `PercentileChip`. Storage stays uniform.** `user_benchmark_percentiles.percentile` always stores the raw cohort percentile of the user's value (no metric-dependent storage semantics). The chip component grows a `direction: 'higher_is_better' | 'lower_is_better'` prop (or equivalent — planner's call on prop shape vs flavor-bound default). For `lower_is_better`, the chip flips:
  - Text formatter: `top_x = round(percentile)` instead of `round(100 - percentile)` — so a user at raw percentile 5 (better than 95% of cohort on net_flag_rate) renders "Top 5%".
  - Color bands: green for low raw percentile (good), red for high raw percentile (bad). Same `ZONE_SUCCESS` / `GAUGE_NEUTRAL` / `ZONE_DANGER` tokens; the threshold dispatch flips.
  - Flame tiers: trigger on low raw percentile (p1/p5/p10) instead of p99/p95/p90.

  Score Gap chips (Time Pressure Score Gap, Clock Gap) remain `higher_is_better` — default direction unchanged. Only the four `net_flag_rate_{tc}` flavors flip.

- **D-3: Net Flag Rate popover gets an explicit "Lower is better" line.** First content line of the 4-bullet body (before benchmark composition): `"Lower is better — you have fewer net timeouts than 95% of bullet players."` (or per-TC equivalent). Belt-and-suspenders alongside the direction prop's visual flip — disambiguates inversion for users who notice that one chip on the card reads inverted from its siblings. **This is a deliberate addition to the 4-bullet contract for the four `net_flag_rate_{tc}` flavors only.** All other chips (Time Pressure Score Gap, Clock Gap, the 4 Phase 94 chips) keep the 4-bullet structure verbatim.

### Clock Gap chip semantics (discussed)

- **D-4: "Top X%" on Clock Gap = furthest positive gap.** Mean `(user_clock - opp_clock) / base_time` at endgame entry, positive = user advantage. Skill-aligned framing — better clock managers consistently enter endgames with time to spare. Raw cohort percentile maps directly to the chip: a user whose mean gap is at the 95th percentile of the per-TC cohort renders "Top 5%" with green band + flames. Confirmed against the seed §Open Question 2 default; alternative interpretations (closest-to-zero, smallest-deficit) explicitly rejected.

  Implication: a user with a strongly negative mean gap (chronic clock deficit) lands at a low percentile and reads "Top 80%+" / red band. The chip is honest about clock-management skill without folding the distribution or asymmetrically clamping.

### Cohen's d candidacy extension (discussed)

- **D-5: Per-(metric × TC) Cohen's d candidacy table IS a Phase 94.3 deliverable.** The researcher extends `reports/benchmarks-gap-metrics-percentile-candidacy.md` (or writes a sibling at `reports/benchmarks-time-pressure-percentile-candidacy.md` — planner's call) with 12 new cells:
  - Per `(metric_family, tc)` pair: Cohen's d of the metric distribution against rating (game-time ELO bucket) within the per-TC cohort.
  - Per-cell tier classification using the same ladder as the Phase 94.2 candidacy report: **rating-invariant** (d ≤ 0.2), **mild coupling** (d ≤ 0.35), **moderate coupling** (d ≤ 0.6), **heavy rating-proxy** (d > 0.6) — exact bin labels planner refines from the existing report's wording.
  - Methodology section explicitly mirrors the Phase 94.2 report's canonical-slice CTE construction, restricted to one TC.

  This calibrates the 4th tooltip bullet (rating-correlation framing) per chip — there are 12 separate framings, not 3 metric-family or 1 generic. Net Flag Rate at bullet likely tracks rating much harder than at classical; the per-cell calibration is non-trivial and required for honest disclosure.

  The candidacy extension is the **only** pre-flight benchmark-DB artifact required for Phase 94.3 — see D-6 for what was deliberately *not* commissioned.

- **D-6: No other pre-flight probes against the benchmark DB.** The seed's `<Open Design Questions>` 3 (inclusion floor calibration) and 4 (cutpoint inflection) are resolved by **locking the seed defaults**:
  - **Inclusion floor: ≥30 games in each of the two pressured cells (user-pressured AND opp-pressured)** for `time_pressure_score_gap_{tc}`. ≥30 games on the pooled set per TC for `clock_gap_{tc}` and `net_flag_rate_{tc}` (mirrors Phase 94.2 D-6). Below floor → no row stored (chip suppressed; same contract as Phase 94.1/94.2).
  - **Cutpoint locked at <40%** for the pressured/un-pressured boundary, matching the per-quintile bullet chart's left half (Q0+Q1). No empirical inflection check.
  - **p1..p99 resolution** for all 12 CDFs. Per-TC cohort sizes (~300–500 estimate) are sufficient; tail-SE widening at extremes accepted as the deliberate trade-off for methodological parity with the Phase 94 chip surface.

  Rationale: the user judgment is that benchmark cohort sizes are large enough to support p1..p99 granularity without per-cell stability probes, and that pressured-cell coverage variance across TCs is not large enough to warrant per-TC floor tuning. If the backfill summary surfaces unexpectedly low coverage at classical, that's a runtime signal to revisit — not a blocker for the design.

### Claude's Discretion

The user picked the 4 areas above for discussion. Remaining HOW choices the planner can resolve in PLAN.md without coming back to the user:

- **Exact `direction` prop shape** — bool vs string union vs flavor-bound default. Recommended: a discriminated union on `flavor` where `net_flag_rate_{tc}` flavors auto-set `lower_is_better` and the prop isn't exposed at the call site. Keeps `EndgameTimePressureCard` call sites symmetric (planner specifies flavor; chip handles direction).
- **Net Flag Rate "Lower is better" line placement in the popover body** — recommend prepending it as a standalone line before the 4-bullet structure (visually distinct from the bullets), but in-bullet placement (e.g. as a leading sub-line on bullet 1) is acceptable if the planner finds it reads more naturally.
- **Time Pressure Score Gap chip on the Quintiles subtitle line — exact alignment** — recommend matching the existing right-aligned info-popover trigger position on subtitle lines (`time-pressure-card-${card.tc}-quintiles-subtitle` pattern). Planner verifies via screenshot on desktop + 375px.
- **Candidacy report file location** — either extend `reports/benchmarks-gap-metrics-percentile-candidacy.md` with a new "Time Pressure metric family" section (single-file, sibling tables) or write a sibling `reports/benchmarks-time-pressure-percentile-candidacy.md` (separate-file, links from the original). Recommend extend for grep-ability of "all candidacy tables in one file"; sibling is fine if the original gets unwieldy.
- **Cohen's d cell labels in the report** — reuse the Phase 94.2 candidacy report's existing tier ladder verbatim (rating-invariant / mild / moderate / heavy). Exact thresholds (d ≤ 0.2 / 0.35 / 0.6) refined from the existing report's published thresholds, not invented.
- **Tooltip 4th-bullet copy strings per (metric × TC) tier** — exact wording is planner's call within `feedback_percentile_chip_tooltip_disclosure`'s 4-bullet contract and `feedback_popover_copy_minimalism`'s discipline (WHAT + sign convention only, no jargon).
- **Plan boundaries inside the phase** — recommend a similar wave structure to Phase 94.2 (canonical-slice builder refactor → CDF regen → per-user service compute → backfill → frontend) with the candidacy report as a Plan-01 deliverable feeding the tooltip copy in the frontend plan. Planner finalises.
- **Migration shape** — `ALTER TYPE benchmark_metric ADD VALUE` 12 times, batched in one migration; reversible via DROP-and-recreate-ENUM (postgres ENUM doesn't support remove value). Recommend documenting the "12 new values, irreversible without table rewrite" cost in the migration docstring.
- **Backfill operational details** — same defaults as Phase 94.2 (per-user commits, creation-asc ordering, per-100-users summary log line, dev-then-prod-after-sign-off).
- **Test scaffolding** — Vitest assertions for the direction prop (lower-is-better Net Flag chip color band + flame trigger at low percentile), TS test for `time_pressure_score_gap_bullet` flavor popover including the "Lower is better" line, Pytest goldens for the 3 new SQL builders.
