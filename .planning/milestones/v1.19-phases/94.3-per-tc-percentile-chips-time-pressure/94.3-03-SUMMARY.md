# Phase 94.3 Plan 03: Widen CDF registry to 16 entries — Summary

`CdfMetricId` Literal widened from 4 to 16 entries; `GLOBAL_PERCENTILE_CDF` regenerated against the benchmark DB with the 12 new per-(metric × TC) time-management entries; byte-identical SQL goldens restructured into per-metric `.sql` fixtures and parametrised over all 16 metrics; prior 94.2 report archived. Plan D can now reference any of the 12 new metric IDs at runtime without `KeyError`.

- `_metric_display_name` widened with the per-family map producing `Time Pressure Score Gap (<tc>)` / `Clock Gap (<tc>)` / `Net Flag Rate (<tc>)`. Unknown IDs raise `KeyError`.
- `interpolate_percentile` signature widened from `MetricId` to `MetricId | CdfMetricId` (Rule 3 — see Deviations below).

## Deviations from Plan

### [Rule 3 — Auto-fix blocking issue] Widen `interpolate_percentile` signature to `MetricId | CdfMetricId`

- **Found during:** Task 1 GREEN — ty check.
- **Issue:** Plan claims `interpolate_percentile(metric_id, value)` signature stays unchanged because "the 12 new metric IDs flow through the existing dispatch automatically". This is true at runtime but breaks type-checking: `interpolate_percentile` was annotated `metric_id: MetricId` where `MetricId` is the 19-entry Literal from `app/services/endgame_zones.py`. The 12 new per-TC IDs are NOT members of `MetricId` (the per-TC family is new in Phase 94.3 and lives in the CDF registry only), so two existing call sites in `app/services/user_benchmark_percentiles_service.py:155` and `:211` failed ty after Task 1 widened `CdfMetricId`.
- **Fix:** Widened the parameter type to `MetricId | CdfMetricId`. Updated the docstring to explain the union and the rationale (caller convenience for the broad MetricId iteration path + the new CdfMetricId-only IDs). Kept the existing `# ty: ignore[invalid-argument-type]` on the `GLOBAL_PERCENTILE_CDF.get(metric_id)` call (the union is wider than the registry key type by design — `None` is returned for IDs not in the registry).
- **Why not widen `endgame_zones.MetricId`:** The endgame_zones Literal is consumed by `assign_zone`, `ZONE_REGISTRY` lookups, and dozens of frontend-facing surfaces. Widening it would force every other ZONE_REGISTRY consumer to handle the 12 new IDs even though they have no zone band. Union at the single call site is the minimum-surface fix.
- **Files modified:** `app/services/global_percentile_cdf.py` (one signature line + docstring update).
- **Commit:** `90c1b08a`.

### [Refactor / scope-adjacent] `_archive_prior_report` phase-tag detection

- **Found during:** Task 2 — running the regen script.
- **Issue:** The script hard-coded `94.1` in the archive filename (`global-percentile-cdf-94.1-{today}.md`) — a leftover from when Phase 94.2 archived Phase 94.1 reports. Running unchanged under Phase 94.3 would have archived the Phase 94.2 report as `global-percentile-cdf-94.1-2026-05-24.md`, which is misleading.
- **Fix:** Added `_detect_prior_phase_tag(prior_path)` which parses the prior report's top heading (`# Global Percentile CDF — ... (Phase XX.Y)`) to extract the phase tag. New archive name: `global-percentile-cdf-<date>-phase-<tag>.md`. Falls back to `unknown` if the heading doesn't match.
- **Files modified:** `scripts/gen_global_percentile_cdf.py` (one new helper + one-line filename change in `_archive_prior_report`).
- **Commit:** `d7d02a81`.
