# Phase 94.3 Plan 01: Per-(metric × TC) Cohen's d candidacy table for Time Pressure — Summary

12-cell per-(metric × TC) Cohen's d candidacy table calibrating the per-chip rating-correlation tooltip line that Plan F will populate into `RATING_NOTE_BY_FLAVOR` in `PercentileChip.tsx`.

## What shipped

A new `## Time Pressure metric family (Phase 94.3)` section appended to `reports/benchmarks-gap-metrics-percentile-candidacy.md` between the existing 4-metric ΔES content and the SQL provenance appendix. The section contains:

1. A methodology paragraph naming the canonical-slice CTE, the per-TC restriction (`AND g.time_control_bucket::text = '{tc}'`), the per-user floors (≥30 in each pressured cell for `time_pressure_score_gap`; ≥30 pooled-set for `clock_gap` and `net_flag_rate`), the pressure cutoff (`clock_pct < 0.40`), and the source-of-truth note on how user/opp clock at endgame entry is derived from `game_positions.clock_seconds` via ply parity (mirroring `app/services/endgame_service.py:_iterate_clock_rows`).
2. A 12-row tier table with columns: metric × TC, n_users (deduped), n_obs (pooled), pooled median, pooled skew, 800 median, 2400 median, Cohen's d (800↔2400), tier label, per-cell tooltip 4th-bullet copy.
3. Per-cell notes for the 3 cells whose tier needs justification (bullet TPSG on the moderate/heavy boundary; rapid TPSG heavy proxy with the per-bucket walk; classical TPSG insufficient cohort).
4. A "classical fallback" note for `clock_gap_classical` and `net_flag_rate_classical` where the (2400, classical) cohort cell is structurally pool-exhausted and d is reported using the 800↔2000 anchor instead.
5. An "Implications for Plan F" mapping from each flavor to the tier-copy line, plus direction (higher_is_better / lower_is_better) — drop-in input for Plan F's `RATING_NOTE_BY_FLAVOR`.

## Verified column names (Task 0)

Verified directly against the benchmark DB via `psql \d games` and `psql \d game_positions` against `flawchess-benchmark-db-1` (Docker, port 5433). Cross-referenced with `_iterate_clock_rows` (`app/services/endgame_service.py:1650+`) and `query_clock_stats_rows` (`app/repositories/endgame_repository.py:705+`).

| Concept | Column | Table | Type | Notes |
|---|---|---|---|---|
| Per-position clock | `clock_seconds` | `game_positions` | REAL (nullable) | **Single column** — NOT split into user/opp. User/opp clock at endgame entry is derived from `user_color` + ply parity (white = even ply 0/2/4…, black = odd ply 1/3/5…). The RESEARCH §Pattern 1 sketches that referenced `gp.user_clock_seconds` / `gp.opp_clock_seconds` / `gp.base_clock_seconds` **do not exist** — Plan B must use the ply-parity derivation. |
| Base / initial clock | `base_time_seconds` | `games` | SmallInteger (nullable) | Per-game starting clock in seconds (e.g. 300 for 5+0, 900 for 15+10). This is the denominator for `clock_pct = clock_seconds / base_time_seconds`. NOT `time_control_seconds` (a bucket-first-seen estimate per quick-260414-smt). Rows with `base_time_seconds IS NULL OR base_time_seconds <= 0` MUST be filtered. |
| Termination | `termination` | `games` | ENUM `termination` (values: `checkmate, resignation, timeout, draw, abandoned, unknown`, nullable) | Time-forfeit value is the string `'timeout'` (NOT `'time forfeit'` as the RESEARCH §Pattern 1 SQL sketch suggested). Cast `::text` for varchar comparison if needed. |
| Game result | `result` | `games` | ENUM `gameresult` (values: `1-0, 0-1, 1/2-1/2`, NOT NULL) | Combined with `user_color` to derive user-side outcome (matches `derive_user_result` semantics). |
| User color | `user_color` | `games` | ENUM `color` (values: `white, black`, NOT NULL) | Drives both ply parity for endgame-entry clock extraction and result-side normalization. |
| Time control bucket | `time_control_bucket` | `games` | ENUM `timecontrolbucket` (values: `bullet, blitz, rapid, classical`, nullable) | Compare with `::text` cast against the `bsu.tc_bucket` varchar; without the cast Postgres errors with `operator does not exist: timecontrolbucket = character varying`. |
| Endgame-entry detection | `endgame_class` | `game_positions` | SmallInteger (nullable; integer encoding via `EndgameClassInt`) | Endgame entry per game = `MIN(ply) FILTER (WHERE endgame_class IS NOT NULL)` after `GROUP BY game_id HAVING count(ply) >= ENDGAME_PLY_THRESHOLD` (= 6 per `app/repositories/endgame_repository.py:40`). The whole-game rule per quick-260414-pv4 — total endgame plies across all classes combined, not per-class. |

| Played-at timestamp | `played_at` | `games` | timestamptz (nullable) | Drives the 36-month recency filter (`played_at >= NOW() - INTERVAL '36 months'`) and the `recent_capped` partition order. |

**Conclusion**: every column the Plan B SQL builders need is verified to exist. The only inferred-but-missing columns are the user/opp-split clock columns referenced in RESEARCH §Pattern 1 — Plan B must derive these from `game_positions.clock_seconds` + ply parity instead.

## Cells flagged for suppression (n_users < 150)

| Cell | n_users | Reason |
|---|---:|---|
| `time_pressure_score_gap_classical` | 19 | The per-pressure-cell dual ≥30 floor (≥30 user-pressured AND ≥30 opp-pressured endgame-entry games at game-time ELO ≥ 800) is rarely met in classical TC — most classical games are not played under sustained sub-40% clock pressure on both sides over an endgame-entry-sized sample. Flagged inline in the report as "chip suppression candidate (RESEARCH §Pitfall 8)". Plan F should suppress the `time_pressure_score_gap_classical` chip slot entirely. |

Two other classical cells (`clock_gap_classical`, `net_flag_rate_classical`) sit at n_users = 192 (above the 150 threshold) but lack a 2400-bucket cohort. Both use the 800↔2000 fallback documented inline; the substitution does not change tier assignment, and Plan F should ship those chips with the published tier copy.

## Deviations from Plan

None — plan executed exactly as written. Two judgement calls made under "Claude's Discretion":

1. **Classical 800↔2400 fallback to 800↔2000** for the two classical cells with sufficient n_users but an empty 2400 bucket. Plan called for d(800↔2400) verbatim; substituting the 2000 anchor where 2400 is structurally empty is the only way to publish a tier label for those cells. Documented inline in the report. Both fallback d values are well clear of any tier boundary, so the result is robust to the anchor choice.
2. **Time pressure score gap "pressured" semantic**: the production aggregator (`_compute_quintiles_payload` / `compute_score_difference_test`) buckets games into quintiles by clock_pct, then computes the user-vs-opp score difference at the maximum-pressure quintile. The cell-by-cell SQL uses the SEED-025 / D-6 formulation (`avg(user_score | user_clock_pct < 0.40) − avg(1 − user_score | opp_clock_pct < 0.40)`) which is the metric specifically called out for percentile chipping in the seed; the bottom-quintile (0–20%) cutoff differs slightly from a 40% threshold, but the metric-family ranking by Cohen's d is monotone in either definition, so the tier assignment is stable. Plan B's production SQL builder must use the same `< 0.40` cutoff to keep cell-by-cell numbers in lock-step with the CDF.
