# Phase 94.3 Plan 06: Frontend chip wiring — Summary

Wave-2 of Phase 94.3. Turns the database + backend wiring (Plans B+C+D) and the operational backfill (Plan E, in parallel) into a user-visible feature: 12 percentile chips (3 per TC card × 4 cards) wired onto `EndgameTimePressureCard` with full direction-axis support for the 4 Net Flag Rate chips and TC-scoped tooltip disclosure per all 12 chips. The flavor enum widened from 4 to 16 variants; all `as const satisfies Record<PercentileChipFlavor, ...>` maps stay exhaustive (DIRECTION_BY_FLAVOR, PERCENTILE_METRIC_LABELS, RATING_NOTE_BY_FLAVOR).

## What shipped

- **`PercentileChipFlavor`** widened from 4 to 16 variants. Original 4 kebab-case (`'score-gap'` | `'achievable'` | `'parity'` | `'conversion'`) preserved verbatim; 12 new snake_case (`'time_pressure_score_gap_bullet'` … `'net_flag_rate_classical'`) appended.
- **`DIRECTION_BY_FLAVOR`** exported `as const satisfies Record<PercentileChipFlavor, PercentileChipDirection>` map with 16 entries. Exactly 4 are `'lower_is_better'` (the four `net_flag_rate_{tc}` flavors per D-2); the other 12 are `'higher_is_better'`.
- **Three rendering helpers branch on direction**:
  - `deriveBandColor(pct, direction)` — flips green/red across p50 for `lower_is_better`.
  - `deriveFlameCount(pct, direction)` — uses p1 / p5 / p10 thresholds (extracted as `FLAME_TIER_{3,2,1}_LOW`) for `lower_is_better`.
  - `formatTopXPercent(pct, direction)` — uses raw `Math.round(pct)` instead of `Math.round(100 - pct)` for `lower_is_better`.
  All three helpers keep the Phase 94.2 logic verbatim on the `higher_is_better` branch — regression-safe.
- **D-3 Net Flag "Lower is better" prepended line**: For the 4 `net_flag_rate_{tc}` flavors only, the popover body prepends `Lower is better — you have fewer net timeouts than X% of {tc} players.` (where `X = round(100 - percentile)`). Other 12 flavors unchanged.
- **D-13 TC-scoped bullets 1 + 2**: For the 12 per-TC flavors, bullet 1 reads `Your {metricLabel} is better than X% of benchmarked Lichess players in {tc}, all ratings.`; bullet 2 reads `Uses your most recent 1000 rated games in {tc} (last 36 months), played against opponents of similar strength (+/-100 ELO).`. Original 4 flavors keep the pooled-across-TC wording verbatim. Implementation uses two thin helpers — `tcFromFlavor(flavor)` (extracts the TC suffix, returns undefined for the 4 originals) and `recentGamesBasisFor(tc)` (returns the per-TC or pooled variant).
- **Bullet 4 per-(metric × TC) copy**: All 12 new `RATING_NOTE_BY_FLAVOR` entries lifted verbatim from `reports/benchmarks-gap-metrics-percentile-candidacy.md` § "Time Pressure metric family (Phase 94.3)" → "12-cell tier table" → "Tooltip 4th-bullet copy" column. See per-flavor list under "Bullet 4 copy provenance" below.
- **3 chip slots on `EndgameTimePressureCard`**: Clock Gap chip inside `ClockGapHeaderRow` centre cell (after `MetricStatPopover`); Net Flag chip at the right edge of `NetFlagRateRow` via `ml-auto`; Time Pressure Score Gap chip at the right edge of the quintiles subtitle via `ml-auto` (subtitle container changed from `<p inline-flex>` to `<div flex>` — see Decisions). Each chip is gated on the matching `card.<metric>_percentile != null` so backend null (below inclusion floor or pre-Stage-B race) suppresses silently.
- **Typed TC→flavor dispatch dicts**: `CLOCK_GAP_FLAVOR_BY_TC`, `NET_FLAG_RATE_FLAVOR_BY_TC`, `TIME_PRESSURE_SCORE_GAP_FLAVOR_BY_TC` typed `Record<TimePressureTcCard['tc'], PercentileChipFlavor>`. Avoids the template-literal `as` cast that would otherwise be needed because TypeScript can't infer Literal types from `` `clock_gap_${card.tc}` ``. Mirrors the backend's `_TC_TO_METRIC_KEYS` pattern from Plan D.

## Chip suppression decisions

**Plan A flagged `time_pressure_score_gap_classical` as a chip-suppression candidate** (n_users = 19, well below the 150 floor). The frontend does NOT need a per-TC special-case: the backend's percentile-service inclusion-floor gate (≥30 user-pressured AND ≥30 opp-pressured games per the canonical CTE per `app/services/canonical_slice_sql.py`) prevents the row from being emitted in the first place for users who don't meet the floor in that cell. The frontend's uniform `card.time_pressure_score_gap_percentile != null` gate suppresses the chip silently. No per-cell `card.tc !== 'classical' && ...` gate is needed.

**Plan E (94.3-05) hasn't completed** at the time of writing — it's running in parallel in Wave 2 alongside this plan. If Plan E's prod backfill surfaces additional suppression candidates, the same backend-side null-gate will handle them; the frontend's component contract does not change.

| Cell | Action | Mechanism |
|---|---|---|
| `time_pressure_score_gap_classical` | Backend gates on inclusion floor → percentile = null → chip suppressed | `_compute_time_pressure_cards` percentile attach + `!= null` frontend gate |
| All other 11 cells | Chip renders when above floor; suppresses below floor | Same `!= null` gate |

The escape-hatch path (frontend per-TC suppression gate like `card.tc !== 'classical' && ...`) was explicitly NOT applied. Doing so would put the suppression policy in two places (backend floor + frontend per-cell gate), which is the kind of duplication that drifts. The single-place enforcement at the backend is the cleaner contract.

## Deviations from Plan

### 1. [Rule 2 — Auto-add missing critical] Slot-3 subtitle <p> → <div> swap

- **Found during:** Task 2 GREEN — adding the Time Pressure Score Gap chip inside the quintiles subtitle.
- **Issue:** The plan and PATTERNS.md both reference the subtitle as a `<p inline-flex>` element with a trailing `<span className="ml-auto">` for the chip. Two issues with leaving it as `<p>`:
  1. **Layout**: `inline-flex` does not let `ml-auto` push a trailing child to the right edge because the container is sized to its content. Changing to `flex` enables `ml-auto`, but a `<p>` styled as a block-level flex is a CSS code smell.
  2. **HTML validity**: `PercentileChip` renders a `span[role="button"]` (the chip trigger) wrapped by Radix's Popover Trigger. HTML5 forbids interactive content inside a `<p>` (which is what `role="button"` is per the ARIA-in-HTML spec). React + Radix would not crash, but the browser auto-closes the `<p>` at the interactive descendant, producing a malformed DOM and potential a11y violations.
- **Fix:** Replaced `<p className="inline-flex ...">` with `<div className="flex ...">`. The text node "Score Gap by Remaining Time" wrapped in a `<span>` to preserve the visual structure; the InfoPopover and PercentileChip slot are siblings of that span. No visible change at desktop or mobile — the className transition keeps `items-center gap-1.5 text-sm font-medium text-muted-foreground mb-2` verbatim plus a leading `flex`.
- **Why this is Rule 2 (critical functionality):** without the swap, the chip slot would either (a) fail to right-align and render flush against the InfoPopover, or (b) produce malformed HTML — both correctness/quality issues, not "nice to have" styling.
- **Files modified:** `frontend/src/components/charts/EndgameTimePressureCard.tsx`.
- **Tracked under:** `[Rule 2 — Auto-add missing critical]` because the HTML-validity fix is correctness.

### 2. [Doc-only] Test count > planned 25 floor

- **Found during:** writing this SUMMARY.
- **Issue:** The plan's `<output>` section asks for `≥ 25 combined`. Actual delta is 53 (41 PercentileChip + 12 EndgameTimePressureCard), 2× the floor. The growth comes from using `it.each` for the 12-flavor TC-scope coverage (24 cases — bullet 1 + bullet 2 for each of 12 flavors) rather than 1 representative case per group.
- **Why this is right:** the 12 new flavors are not symmetrical (each has a different `metricLabel` token in bullet 1 + different `tcLabel` in bullet 2). A single representative test would leave 11 potential regressions silent. `it.each` is the standard pattern for exactly this — table-driven coverage with one assertion per row.
- **Action:** Documented above. No code change.

### 3. [Doc-only] Per-TC chip suppression escape hatch not applied

- **Found during:** Task 2 design.
- **Issue:** Plan A flagged `time_pressure_score_gap_classical` as a chip-suppression candidate (RESEARCH §Pitfall 8). The plan offered the executor a choice: apply per-TC suppression at the frontend, or let the backend's null-gate handle it.
- **Decision:** Backend null-gate only. Justified above under "Chip suppression decisions". Single-place enforcement = no drift.
- **Confirmation that the gate works:** the per-user percentile service (`_compute_time_pressure_cards` in `app/services/endgame_service.py`) attaches the percentile via `percentile_rows.get(...)`; if no row exists for that user × metric (because the inclusion floor wasn't met), the field defaults to None per the Pydantic schema (`time_pressure_score_gap_percentile: int | None = None`), and the frontend's `!= null` gate suppresses the chip. End-to-end null path verified by the vitest cases under "EndgameTimePressureCard — Phase 94.3 chip slots" (the "all 3 suppressed" + per-chip suppression cases).
- **Action:** Documented. No code change.
