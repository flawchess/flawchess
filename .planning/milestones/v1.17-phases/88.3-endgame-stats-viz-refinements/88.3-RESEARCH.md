## Summary

Phase 88.3 makes three independent visual-representation fixes to the `/endgames/stats` page. No backend changes, no new API fields, no benchmark refresh. All work is frontend-only and the three changes can be planned as parallel waves.

**Change 1 (Inactivity-gap annotations):** The two ordinal-axis timeline charts ("Endgame Score Gap over Time" and "Endgame ELO Timeline") must render a visible axis-break glyph plus compact label at each gap between consecutive data points that exceeds an inactivity threshold. Both charts use ISO-Monday `date` strings from the backend, so the gap is computable client-side by comparing consecutive dates in the sorted array. The correct injection seam is a pure `computeInactivityGaps(dates, thresholdDays)` helper that returns typed gap descriptors, plus a custom Recharts `<Customized>` component (or equivalent SVG overlay) that renders the glyph above the x-axis. A `<ReferenceLine>` with a `content` render prop is another viable approach already used in `EndgameClockDiffOverTimeChart.tsx`.

**Change 2 (ELO Timeline default series):** `EndgameEloTimelineSection.tsx` already has `computeDefaultHidden()` and the `MAX_DEFAULT_VISIBLE = 3` constant. The change is to lower `MAX_DEFAULT_VISIBLE` to `1` and re-verify the active-weeks filter still works correctly for the single-series edge case. The toggle mechanism, state reset on data change, and single-series auto-case are unchanged.

**Change 3 (Endgame Overall Performance restructure):** `EndgameOverallPerformanceSection.tsx` is an orchestrator that mounts five sub-components into a 3-column `lg:grid-cols-3` grid with a `ConnectorArrows` SVG overlay. The restructure removes the 3-column grid, the `ConnectorArrows` component, and the `useRef` + `relative` positioning; replaces them with a single `charcoal-texture` card using a `flex flex-col md:flex-row` (or `lg:grid-cols-2`) layout with CSS dividers. `EndgameOverallConnectorArrows.tsx` is deleted entirely. `EndgameOverallCard`, `EntryCard`, and `ScoreGapRow` sub-components survive unchanged.

**Primary recommendation:** Implement as three Wave tasks. Waves 1 and 2 can run in parallel; Wave 3 is independent. The inactivity-gap annotation is the most novel piece of work and warrants the most test coverage.

### Pattern 1: Inactivity Gap Detection

**What:** Given the sorted `date[]` array (ISO-Monday strings), detect consecutive pairs where the gap exceeds a threshold. Return descriptors for each such gap.

**When to use:** In `useMemo` inside each chart component, before rendering, using the same `dates` array already computed for `createDateTickFormatter`.

**Implementation approach:**
```typescript
// Source: [ASSUMED — authored pattern for this phase]
const INACTIVITY_THRESHOLD_WEEKS = 8; // gap > ~2 months triggers annotation

interface InactivityGap {
  afterIndex: number;   // index in sorted dates[] after which the gap occurs
  gapDays: number;      // calendar days between dates[afterIndex] and dates[afterIndex+1]
}

function computeInactivityGaps(
  sortedDates: string[],
  thresholdDays: number,
): InactivityGap[] {
  const gaps: InactivityGap[] = [];
  for (let i = 0; i < sortedDates.length - 1; i++) {
    const a = parseDate(sortedDates[i]!);
    const b = parseDate(sortedDates[i + 1]!);
    const days = Math.round((b.getTime() - a.getTime()) / (1000 * 60 * 60 * 24));
    if (days > thresholdDays) {
      gaps.push({ afterIndex: i, gapDays: days });
    }
  }
  return gaps;
}
```

`parseDate` already exists in `frontend/src/lib/utils.ts` as a module-private function. If the helper is extracted to a new `inactivityGaps.ts`, `parseDate` should be exported from `utils.ts` or duplicated (it is three lines). [ASSUMED — parseDate is currently unexported]

### Pattern 2: Break-Glyph Rendering in Recharts

**What:** Render a visible axis-break annotation (zigzag or double-slash) at each inactivity gap position on the ordinal x-axis.

**Viable approaches (from best to simplest):**

**Option A — `<Customized>` component (SVG overlay inside ComposedChart):** Recharts supports a `<Customized>` element as a direct child of `<ComposedChart>`. It receives chart layout props (`xAxisMap`, `yAxisMap`, `width`, `height`) and can render arbitrary SVG. This is the cleanest path: the glyph SVG lives inside the chart canvas, respects the chart margin, and requires no DOM measurement outside Recharts.

**Option B — `<ReferenceLine>` with `content` render prop:** `EndgameClockDiffOverTimeChart.tsx` already uses `<ReferenceLine>` (a horizontal line at y=0). Recharts `<ReferenceLine>` supports `x={dateString}` for a vertical line at a given x-axis value plus a `content` render prop for a custom SVG shape. This is the pattern already in the codebase and requires zero new Recharts API surface. The label (e.g. "≈3.1y inactive") can be rendered via the `label` prop or within the `content` renderer.

**Option C — DOM overlay (outside ComposedChart, `position: absolute`):** Requires DOM measurement to align with chart pixels — the same pattern `ConnectorArrows` uses (which this phase is removing). Avoid.

**Recommended: Option B** — `<ReferenceLine>` with `x={dates[gap.afterIndex]}` and a custom `content` renderer. One `<ReferenceLine>` per gap, rendered as direct children of `<ComposedChart>` (not inside `flatMap` wrapper — Recharts requires direct children). The `content` renderer receives `{x, y, width, height}` computed by Recharts and draws the zigzag SVG path or double-slash symbol plus the compact label.

**Label format:** `≈{N}y inactive` for years (rounded to 1 decimal), `≈{N}mo inactive` for months. Thresholds: years if gap >= 365 days, months otherwise.

**Font size constraint:** CLAUDE.md mandates `text-sm` as the minimum for regular UI. Glyph labels sit above the x-axis within the chart SVG canvas — they are transient "structural annotation" akin to tooltip-popover content. The decisions file (Thread 2) says annotations "must be small enough to read and not crowd the axis on mobile." Treat glyph labels as equivalent to the `text-xs` tooltip exception (HelpCircle trigger pattern) — short, opt-in structural information rendered in SVG. Use `font-size: 11` or `fontSize={11}` in the SVG `<text>` element (below Tailwind's `text-xs` 12px floor but inside the chart SVG). If in doubt, use `fontSize={12}` to stay clearly within the text-sm equivalent for SVG.

**Crowding prevention:** On 375px screens, if multiple gaps are close together, their labels may overlap. Mitigation: render the label only when the gap is the widest one within a certain ordinal window, or shorten the label to `≈{N}y`. The planner should include a mobile-legibility verification step.

### Anti-Patterns to Avoid

- **DOM measurement for layout:** `ConnectorArrows` used `getBoundingClientRect()` + `ResizeObserver` to position arrows. The new 2-column layout must not use any DOM measurement — pure CSS/Flexbox is the right tool.
- **Wrapping Recharts direct children:** The `EndgameScoreOverTimeChart` and `EndgameEloTimelineSection` files have explicit comments warning that `<Area>` and `<Line>` must be direct children of `<ComposedChart>`. `<ReferenceLine>` elements for gap annotations are also direct children — do not wrap them in fragments or extra elements.
- **Duplicating gap-detection logic:** Both charts need the same logic. Extract to `@/lib/inactivityGaps.ts` (or export from `utils.ts`) so both import the same function. Do not copy-paste into each chart file.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date arithmetic for gap detection | Custom week/month counter | `Date.getTime()` difference (already in `utils.ts:dateRangeDays`) | Off-by-one errors with ISO week boundaries; millisecond arithmetic is exact |
| Break-glyph SVG path | Custom SVG zigzag from scratch | Standard `<path d="M0,0 L4,8 L8,0 ..." />` double-slash pattern (3-4 lines of SVG) | Zigzag is simple enough to hand-code; no library needed |

| Legend toggle state | Custom state machine | Existing `hiddenKeys: Set<string>` + `setHiddenKeys` (already in the component) | Changing `MAX_DEFAULT_VISIBLE` is a one-line constant change |

## Common Pitfalls

### Pitfall 1: Recharts `<ReferenceLine x={...}>` must use the exact dataKey value
**What goes wrong:** The x-axis `dataKey="date"` uses ISO-Monday strings (e.g. `"2024-01-08"`). A `<ReferenceLine x="2024-01-08" />` resolves correctly only if that exact string appears in the chart data. For the gap annotation, the reference line should be positioned at `dates[gap.afterIndex]` — the last point BEFORE the gap — not at a computed midpoint date that may not exist in the data.
**Why it happens:** Recharts resolves categorical x-axis positions by matching the value against the data array. Non-existent values produce no visible line.
**How to avoid:** Use `x={sortedDates[gap.afterIndex]}` where `sortedDates` is the same array passed to `createDateTickFormatter`.
**Warning signs:** Reference line renders at position 0 or is invisible.

### Pitfall 2: ELO Timeline single-series active-weeks filter behavior
**What goes wrong:** When `MAX_DEFAULT_VISIBLE = 1`, the active-weeks filter (`MIN_ACTIVE_WEEKS_RATIO = 0.33`) may exclude the only qualifying combo if it has very few active weeks relative to a sparse "leader." But when there is only one combo, `leaderWeeks = combo.points.length` and the combo trivially passes `>= 0.33 * leaderWeeks`. The current `if (combos.length <= MAX_DEFAULT_VISIBLE)` comment-only block does NOT short-circuit. Verify the logic is correct for 1-combo case after the constant change.
**Why it happens:** The comment says "No need to hide anything; the cap can't bite" but the code falls through to the ranking logic. With 1 combo and `MAX_DEFAULT_VISIBLE = 1`, `ranked.slice(0, 1)` = [the 1 combo], `visible` = {combo_key}, and `hidden` = {}. This is correct — but the comment is now misleading since it no longer applies at threshold 1.
**How to avoid:** Add a test for 1-combo payload that asserts `hiddenKeys` is empty.
**Warning signs:** Single-series chart shows nothing (all series hidden).

### Pitfall 3: ConnectorArrows removal leaves orphaned `useRef` import
**What goes wrong:** `EndgameOverallPerformanceSection.tsx` imports `useRef` from React solely for `gridRef` passed to `ConnectorArrows`. After removing the `ConnectorArrows` usage, the `gridRef` declaration and the `ref={gridRef}` prop on the grid `<div>` must also be removed. Forgetting the `useRef` import causes a `ty check` error or knip warning.
**Why it happens:** `useRef` has no other use in the file.
**How to avoid:** Remove `useRef` from the import, remove `const gridRef = useRef(...)`, remove `ref={gridRef}` from the outer container div, and remove `relative` from the container's class list (it was there to anchor the absolute-positioned ConnectorArrows overlay).
**Warning signs:** `ty check` "unused variable" error or knip detecting dead import.

### Pitfall 4: Inner card charcoal-texture double-nesting
**What goes wrong:** `EndgameCard` and `EntryCard` currently render their own `<div className="charcoal-texture rounded-md p-4">` wrapper. If the new outer card also uses `charcoal-texture`, the texture doubles up visually.
**Why it happens:** The sub-components were designed as standalone tiles. Inside a containing charcoal card, their own wrapping is redundant.
**How to avoid:** Either (a) strip the wrapper from `EndgameCard` and `EntryCard` (they become bare `<div className="flex flex-col gap-4">` containers) or (b) use `charcoal-texture` only on the outer card and give sub-components a simpler inner background. Option (a) is cleaner; confirm visually.
**Warning signs:** Noticeably darker/denser texture on sub-areas of the card.

### Pitfall 5: Test-id assertions break after layout restructure
**What goes wrong:** `EndgameOverallPerformanceSection.test.tsx` asserts testids like `tile-games-without-endgame`, `tile-at-endgame-entry`, `tile-games-with-endgame`, and `endgame-score-differences`. These testids live on the sub-component root elements, which are currently the charcoal card wrappers. If those wrappers are stripped, the testids must move to replacement wrapper elements.
**Why it happens:** Tests query by `data-testid` on DOM elements whose structure changes.
**How to avoid:** Preserve all existing `data-testid` values on replacement `<div>` elements. The testids on `EndgameCard` (`tileTestId` prop), `EntryCard` (`data-testid="tile-at-endgame-entry"`), and the Score Differences block (`data-testid="endgame-score-differences"`) must survive the restructure.
**Warning signs:** Test suite failures in `EndgameOverallPerformanceSection.test.tsx`.

## ELO Timeline Default Series — Current Behavior Confirmed

The `computeDefaultHidden()` function in `EndgameEloTimelineSection.tsx` currently uses `MAX_DEFAULT_VISIBLE = 3`, which shows up to 3 combos by default and hides the rest. The decision record mandates defaulting to **1 series** (the most-active). The change is:

```typescript
// Before
const MAX_DEFAULT_VISIBLE = 3;

// After
const MAX_DEFAULT_VISIBLE = 1;
```

The "most active" series is already correctly identified by the current ranking logic: `totalGames` (sum of `per_week_total_games`) descending. The highest-games combo is the most-active. No algorithmic change is needed. The active-weeks pre-filter (`MIN_ACTIVE_WEEKS_RATIO = 0.33`) is correct for a 1-series default — it ensures the single shown series has meaningful data density.

**Is there per-series activity/count data available client-side?** Yes — `combo.points.length` (active weeks) and sum of `combo.points[i].per_week_total_games` (total games) are both computed inside `computeDefaultHidden()` already. No new data is needed.

## Code Examples

### Gap detection helper

```typescript
// Source: [ASSUMED — new file @/lib/inactivityGaps.ts]
// Uses parseDate pattern from frontend/src/lib/utils.ts

export const INACTIVITY_GAP_THRESHOLD_DAYS = 56; // 8 weeks = ~2 months

export interface InactivityGap {
  afterIndex: number;  // gap is AFTER sortedDates[afterIndex]
  gapDays: number;
  label: string;       // e.g. "≈3.1y inactive" or "≈4mo inactive"
}

export function computeInactivityGaps(
  sortedDates: string[],
  thresholdDays: number = INACTIVITY_GAP_THRESHOLD_DAYS,
): InactivityGap[] {
  const gaps: InactivityGap[] = [];
  for (let i = 0; i < sortedDates.length - 1; i++) {
    const a = new Date(sortedDates[i]!);
    const b = new Date(sortedDates[i + 1]!);
    const days = Math.round((b.getTime() - a.getTime()) / 86400000);
    if (days > thresholdDays) {
      const label = days >= 365
        ? `≈${(days / 365).toFixed(1)}y inactive`
        : `≈${Math.round(days / 30)}mo inactive`;
      gaps.push({ afterIndex: i, gapDays: days, label });
    }
  }
  return gaps;
}
```

## Open Questions

1. **Break-glyph symbol — zigzag vs double-slash**
   - What we know: The decision record says "axis-break glyph (standard zigzag / double-slash convention)"
   - What's unclear: Which symbol to implement first; the zigzag is more visually recognizable but requires a custom SVG path; the double-slash is two short diagonal lines achievable with simpler SVG
   - Recommendation: Implement double-slash first (simpler), then upgrade to zigzag if user requests

2. **Gap threshold value**
   - What we know: The user's account has ">3y of inactivity inside a ~5y span" — a gap of that size is clearly significant
   - What's unclear: Whether 8-week (56-day) is the right threshold, or if something shorter (e.g. 4 weeks) or longer should be used
   - Recommendation: Use 8 weeks as a named constant; the planner can include a note to confirm with the user during UAT

3. **`EndgameCard` / `EntryCard` inner wrapper removal**
   - What we know: They currently own their own `charcoal-texture rounded-md p-4` wrapper
   - What's unclear: Whether the sub-components should be reusable standalone tiles (keep wrapper) or purely internal to the 2-column layout (remove wrapper)
   - Recommendation: Remove wrappers since these components have exactly one caller; cleaner and avoids texture double-nesting
