# Phase 88.3: Code Review Report

**Reviewed:** 2026-05-18
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

The 88.3 endgame-stats viz refinements introduce a shared `inactivityGapReferenceLines` helper and roll Palmtree break annotations across six timeline charts. Core gap-detection logic (`computeInactivityGaps`) and the helper are clean, well-tested, and correctly handle empty/single/short arrays. No security issues, no data-loss risks, no crashes found.

The defects are quality and robustness concerns: an invalid Tailwind class that silently breaks Y-axis label vertical alignment in one chart, a project-convention violation (`computeInactivityGaps` requires sorted input but two of six callers pass raw timeline order without the documented defensive sort that the other callers apply), `new Date(YYYY-MM-DD)` UTC-vs-local parsing fragility, duplicated and behaviorally-divergent `useIsMobile` across five files, and several `text-xs` uses outside the documented tooltip exception. None are blockers; all should be fixed.

## Narrative Findings (AI reviewer)

## Warnings

### WR-01: `pt-33` is not a valid Tailwind spacing class — Y-axis label misaligns

**File:** `frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:175`
**Issue:** The vertical "Score" Y-axis label uses `className="... pt-33 -mr-1"`. Tailwind's default spacing scale has `pt-32` and `pt-36` but **no `pt-33`**. The sibling charts in this same phase use existing values (`EndgameClockDiffOverTimeChart` → `pt-32`, `EndgameEloTimelineSection`/`ScoreChart` → `pt-40`), confirming the default scale with no custom `33` extension. `pt-33` therefore produces no padding rule, the label collapses to `padding-top: 0`, and it renders vertically misaligned versus every other timeline chart. This is a real rendering defect on every desktop render of this chart, not a style nit.
**Fix:**
```tsx
// use a value that exists on the spacing scale; match the chart's plot offset
className="flex items-center text-sm text-muted-foreground shrink-0 pt-32 -mr-1"
```

### WR-02: `computeInactivityGaps` requires sorted dates; two of six callers pass raw order

**File:** `frontend/src/components/charts/InactivityGapReferenceLines.tsx:21`; `EndgameScoreOverTimeChart.tsx:139,264`; `EndgameClockDiffOverTimeChart.tsx:133,244`
**Issue:** `InactivityGapReferenceLinesProps.dates` is documented "MUST be ascending-sorted ... caller's responsibility" and `computeInactivityGaps` computes `Date(dateB) - Date(dateA)` per consecutive pair — an unsorted pair yields a negative diff and the gap is silently missed (dropped annotation, no error). `RatingChart` adds a defensive `sortedGapDates = [...allDates].sort()` with a comment explaining its `Map`-insertion-order source is "generally already sorted" but not guaranteed; `ScoreChart` and `EndgameEloTimelineSection` also `.sort()` their date set. But `EndgameScoreOverTimeChart` (`dates = data.map((p) => p.date)`) and `EndgameClockDiffOverTimeChart` (`dates = data.map((p) => p.date)`) pass the raw `timeline` order through with **no defensive sort**, relying solely on an undocumented backend ordering guarantee. Inconsistent contract enforcement: four charts hardened, two not — the two unhardened ones silently drop break annotations if a timeline ever arrives out of order, the exact failure `RatingChart`'s comment warns about.
**Fix:** Prefer sorting the chart `data`/`timeline` at the data-shaping step in `EndgameScoreOverTimeChart` and `EndgameClockDiffOverTimeChart` so the plotted series AND the dates agree (sorting only inside the helper would desync the ReferenceLine `x` from the X-axis category order — Recharts Pitfall 1 noted in 88.3-01-PLAN.md). Document the chosen contract enforcement consistently across all six callers.

### WR-03: `new Date('YYYY-MM-DD')` UTC parse — gap-day arithmetic is timezone-fragile

**File:** `frontend/src/lib/inactivityGaps.ts:48`
**Issue:** `new Date(dateB).getTime() - new Date(dateA).getTime()` parses bare `YYYY-MM-DD` as UTC midnight (ES spec). The subtraction is internally UTC-consistent so the day count is correct, but this is the exact string→Date pattern the project's own test fixtures route around (`EndgameEloTimelineSection.test.tsx:371` deliberately uses `'2026-01-05T00:00:00Z'` + `setUTCDate`). The 56-day threshold has wide margin, but the `days >= 365` and `Math.round(days / 30)` label boundaries can tick by one across a DST/parse edge. Low blast radius; the bare-string construction should be made explicit to match the codebase's UTC discipline.
**Fix:**
```ts
const days = Math.round(
  (Date.parse(`${dateB}T00:00:00Z`) - Date.parse(`${dateA}T00:00:00Z`)) / 86400000,
);
```

### WR-04: Duplicated and behaviorally-divergent `useIsMobile` across five components

**File:** `EndgameClockDiffOverTimeChart.tsx:63-76`; `EndgameEloTimelineSection.tsx:52-64`; `EndgameScoreOverTimeChart.tsx:39-51`; `ScoreChart.tsx:32-44`; `frontend/src/components/stats/RatingChart.tsx:17-29`
**Issue:** Five near-identical `useIsMobile()` copies that are not consistent: `EndgameClockDiffOverTimeChart` uses `MOBILE_BREAKPOINT_PX = 640` and adds a `typeof window === 'undefined'` guard inside the effect; the other four use `768` and omit the in-effect SSR guard. The 640-vs-768 divergence is undocumented — a reader cannot tell if it is intentional or copy-paste drift. CLAUDE.md flags exactly this ("Search for duplicated markup before considering a change complete").
**Fix:** Extract `frontend/src/lib/useIsMobile.ts` exporting `useIsMobile(breakpointPx = 768)` with the SSR guard in both initializer and effect; import in all five, passing `640` explicitly (with a comment) from the clock-diff chart if that breakpoint is deliberate.

### WR-05: Inactivity-gap label has no right-edge clamp — overflows for recent layoffs

**File:** `frontend/src/components/charts/InactivityGapReferenceLines.tsx:50-84`
**Issue:** `LabelContent` computes `labelX = x + 4`, `textX = labelX + glyphSize + 4` from the viewBox and draws the Palmtree glyph + text rightward unconditionally with no edge detection. When a gap lands near the right plot edge (a long break immediately before the latest week — a common real "returning player" case, and exactly the scenario the annotation exists to surface), the text is clipped by the SVG viewport or overlaps the right-side hidden volume-bar axis. Not a crash, but the annotation becomes unreadable in its highest-value case.
**Fix:** Detect proximity to the right edge using `viewBox.width` / plot width and flip the label leftward (`textAnchor="end"`, glyph + text left of the line) when it would overflow.

### WR-06: Unchecked `as number` assertions on Recharts payloads bypass type safety

**File:** `EndgameClockDiffOverTimeChart.tsx:304`; `EndgameEloTimelineSection.tsx:492-495,509`; `ScoreChart.tsx:146,154`
**Issue:** `(payload.avg_clock_diff_pct as number) ?? 0` — the `?? 0` only catches `null`/`undefined`; a non-numeric payload value is silently asserted to `number` and flows into `.toFixed(1)` / `zoneColor()`. `ScoreChart` does `item.value as number` and `item.payload[...] as number` with no guard. CLAUDE.md: "Avoid `any`, prefer explicit types ... leverage TypeScript's type system fully." The ELO tooltip at least guards with `=== undefined` before use (acceptable); the clock-diff `dot` and `ScoreChart` paths do not.
**Fix:**
```ts
const raw = payload.avg_clock_diff_pct;
const diff = typeof raw === 'number' ? raw : 0;
```

### IN-01: `text-xs` used outside the documented tooltip exception

**File:** `EndgameEloTimelineSection.tsx:414,446`; `EndgameScoreOverTimeChart.tsx:175,315,335`; `ScoreChart.tsx:120`
**Issue:** CLAUDE.md sets `text-sm` as the hard floor, exception only for "hover/tap-activated info tooltips (Radix popover bodies)". These `text-xs` uses are persistent UI: the ELO legend row (414), rotated axis labels (446 / 175 / 120 — "ELO"/"Score"), the score-timeline legend strip (315), and the "Week (rolling average ...)" footnote caption (335). `EndgameClockDiffOverTimeChart.tsx:177` correctly uses `text-sm` for its rotated label, proving `text-sm` is achievable here.
**Fix:** Bump persistent-UI `text-xs` to `text-sm` (legends, rotated axis labels, footnote caption). Leave only the Radix/in-chart hover tooltip bodies at `text-xs`.

### IN-02: Missing `data-testid` / action ARIA on the custom ELO legend

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:413-438`
**Issue:** Per-combo legend `<button>`s carry `data-testid` + `aria-pressed` (good) but the wrapping container `<div>` has no `data-testid`, and the buttons lack an `aria-label` describing the toggle action (the swatch is decorative, label text is truncated). CLAUDE.md Browser Automation Rules require `data-testid` on major layout containers and meaningful labels on interactive controls.
**Fix:** Add `data-testid="endgame-elo-timeline-legend"` to the wrapper div and `aria-label={\`Toggle ${comboLabel} visibility\`}` to each button.

### IN-03: `connectNulls` contradicts the inactivity-gap annotation it sits beside

**File:** `EndgameEloTimelineSection.tsx:601,619,636,651` (`connectNulls={true}`) vs `EndgameScoreOverTimeChart.tsx:281,292,305` (`connectNulls={false}`)
**Issue:** Both charts render sparse weekly series AND Palmtree break annotations, yet make opposite `connectNulls` choices. `EndgameEloTimelineSection` bridges the line straight through a gap that is simultaneously labeled "≈3mo inactive" — a continuous line through a marked break is visually self-contradictory. Not a correctness bug, but the two charts should agree, and `false` is the choice consistent with annotating breaks.
**Fix:** Standardize on `connectNulls={false}` for series in charts that also render inactivity-gap annotations.

### IN-04: Dead empty-`if` block in `computeDefaultHidden`

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:117-121`
**Issue:** `if (combos.length <= MAX_DEFAULT_VISIBLE) { /* comment only */ }` — an empty conditional whose body is just a comment explaining it intentionally falls through. Reads as an unfinished early-return; has zero runtime effect.
**Fix:** Delete the `if`; move the rationale to a plain comment above the accumulation loop.

### IN-05: Magic `barMax * 5` repeated across three charts

**File:** `EndgameClockDiffOverTimeChart.tsx:232`; `EndgameEloTimelineSection.tsx:474`; `EndgameScoreOverTimeChart.tsx:203`
**Issue:** `domain={[0, barMax * 5]}` verbatim in three charts pins volume bars to the bottom 20% of the canvas. The `5` is a magic number (CLAUDE.md: "No magic numbers"); intent only lives in scattered "Pattern 3 in 57.1-RESEARCH.md" comments.
**Fix:** Add `export const VOLUME_BAR_CANVAS_FRACTION = 5;` to `theme.ts` (or a chart-constants module) and reference it in all three.

### IN-06: `BREAK_LABEL_FONT_SIZE` doubles as glyph pixel size

**File:** `InactivityGapReferenceLines.tsx:54`; `frontend/src/lib/theme.ts:78`
**Issue:** `const glyphSize = BREAK_LABEL_FONT_SIZE;` reuses a font-size constant as the Palmtree `width`/`height`. The constant's name and theme.ts doc describe it strictly as a font size; a future font tweak would silently resize the glyph too. Minor coupling smell.
**Fix:** Rename to `BREAK_LABEL_SIZE` (documenting it governs both) or add a sibling `BREAK_LABEL_GLYPH_SIZE`.
