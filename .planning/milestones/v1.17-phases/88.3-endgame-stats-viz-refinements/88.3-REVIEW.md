# Phase 88.3: Code Review Report

**Reviewed:** 2026-05-18
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Reviewed the Endgame stats visualization refinements: the new `computeInactivityGaps`
helper, two Recharts timeline charts (`EndgameScoreOverTimeChart`,
`EndgameEloTimelineSection`), and the SC-3 two-column `EndgameOverallPerformanceSection`
with its sub-cards. The code is generally careful (good `noUncheckedIndexedAccess`
narrowing, explicit memo return types, thorough tests) and free of security issues
(no injection surface, no secrets, no `eval`/`innerHTML`). However there are several
correctness and consistency defects worth fixing:

- An asymmetric neutral-band boundary in `gapZoneColor` that mis-colors values at the
  exact band edge and is inconsistent with the popover's own neutral bounds.
- Date parsing in `computeInactivityGaps` is timezone-dependent, so a near-threshold
  gap can flip on/off depending on the runner's local timezone.
- A correctness gap between `gapZoneColor` and the color shown in the popover (zone
  vs. sig-gated) that can show a colored number with a "not significant" popover.
- Several `text-xs` uses in non-tooltip primary content that violate the project's
  `text-sm` floor.
- A few maintainability issues (magic numbers, dead branch, duplicated mobile hook,
  unnecessary `as` casts in the tooltip).

No blockers: none of these corrupt data or crash, but the boundary and timezone
issues are real logic bugs that produce wrong visuals at edges.

## Warnings

### WR-01: `gapZoneColor` neutral-band boundary is asymmetric

**File:** `frontend/src/components/charts/EndgameOverallPerformanceSection.tsx:50-54`
**Issue:** The classifier uses `value < neutralMin` for the danger side but
`value >= neutralMax` for the success side. With `SCORE_GAP_NEUTRAL_MIN = -0.1` /
`SCORE_GAP_NEUTRAL_MAX = 0.1`, a gap of exactly `+0.10` is painted `ZONE_SUCCESS`
while the mirror-image gap of exactly `-0.10` is left uncolored (because
`-0.1 < -0.1` is false). The two band edges are meant to be symmetric ("neutral
unless clearly above/below"); the inclusive/exclusive split makes them behave
differently and is almost certainly unintended. The same helper is reused for the
Achievable Score Gap (`±0.05`), so the asymmetry affects both rows.
**Fix:** Make both edges consistent (either both inclusive of the neutral band or
both exclusive):
```ts
function gapZoneColor(value: number, neutralMin: number, neutralMax: number): string | undefined {
  if (value <= neutralMin) return ZONE_DANGER;
  if (value >= neutralMax) return ZONE_SUCCESS;
  return undefined;
}
```

### WR-02: `computeInactivityGaps` date diff is timezone-dependent

**File:** `frontend/src/lib/inactivityGaps.ts:45-49`
**Issue:** `new Date('2024-05-01')` parses an ISO date-only string as **UTC
midnight**, but `new Date(dateB).getTime() - new Date(dateA).getTime()` is then
divided by `86400000` and `Math.round`-ed. For pure date-only strings both parse as
UTC so the subtraction is exact — but the helper's contract only says "ISO
YYYY-MM-DD". If any caller ever passes a date with a time component (or a
non-zero-padded value), DST transitions inside the interval shift the millisecond
delta by ±3600000, and a gap exactly one day either side of the 56-day threshold can
flip. The chart callers currently pass `pt.date` which is date-only, so this is
latent rather than active, but the rounding-at-the-threshold design is fragile. The
test suite only exercises clean UTC date-only inputs so it would not catch a
regression here.
**Fix:** Compute the difference on UTC calendar days, independent of time/DST:
```ts
function utcDays(iso: string): number {
  const [y, m, d] = iso.split('-').map(Number);
  return Date.UTC(y, (m ?? 1) - 1, d ?? 1) / 86400000;
}
const days = utcDays(dateB) - utcDays(dateA);
```
(or assert/validate the date-only shape at the boundary).

### WR-03: Zone color vs. popover significance can disagree (score gap rows)

**File:** `frontend/src/components/charts/EndgameOverallPerformanceSection.tsx:72-91, 163-243`
**Issue:** The Score-Gap numbers are colored purely by zone
(`gapZoneColor`, "zone-only, no sig test per D-04"), but the adjacent
`MetricStatPopover` receives `level={...GapLevel}` derived from the p-value via
`deriveLevel`. A user can therefore see a bright green/red gap number whose popover
simultaneously says the result is not statistically significant (low confidence).
This is the exact "misleads users" failure mode the project guards against
elsewhere. It may be the intended D-04 behavior, but the two surfaces telling
different stories about the same number is a UX correctness smell that should be
explicitly reconciled (e.g. a neutral/!confident style, or a popover note) rather
than left implicit.
**Fix:** Confirm D-04 truly wants zone-only coloring with a possibly-contradictory
popover; if so, add a short popover line clarifying the number is tinted by zone
regardless of significance. Otherwise gate the tint on `isConfident(gapLevel)` like
the score cards do.

### WR-04: `text-xs` used in non-tooltip primary content

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:415, 447, 487`;
`frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:176, 215, 327, 347`
**Issue:** CLAUDE.md sets `text-sm` as the hard floor and explicitly limits the
`text-xs` exception to "hover/tap-activated info tooltips (Radix popover bodies
with the HelpCircle trigger pattern)". These usages are not that pattern:
- Custom legend rows (`text-xs pt-3 justify-center`, ELO/Score axis labels,
  `endgame-score-timeline-legend`) are persistent primary chrome.
- The chart **hover tooltip** containers (`...px-3 py-2 text-xs shadow-xl`) are
  custom Recharts tooltips, not Radix info popovers — the exception is scoped to
  the info-popover pattern, and tooltip data here (ELO values, game counts) is
  primary content users read to interpret the chart.
- The "Week (rolling average...)" caption uses `text-xs` (line 347).
**Fix:** Bump these to `text-sm` (legends, axis labels, caption, custom chart
tooltips). If density is a concern, reduce columns/whitespace per the CLAUDE.md
guidance rather than shrinking type. Note `EndgameOverallEntryCard`/`Card` already
correctly use `text-sm` — apply the same standard here.

### WR-05: Unnecessary `as` type assertions mask the tooltip row type

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:482-496`
**Issue:** The tooltip casts every field access
(`dateRow[`${combo.combo_key}_endgame_elo`] as number | undefined`,
`label as string`, `dateRow.per_week_total_visible as number | undefined`). The row
type is a `Record<string, string | number | [number, number] | undefined>`, so
these casts silently assert away the `[number, number]` band tuple possibility. If a
key collision ever made `..._endgame_elo` resolve to a tuple, the cast would hide it
and the rendered ELO would be `[object]`/`NaN` instead of failing typecheck. This is
a robustness/type-safety regression versus the project's "avoid `as`, prefer
explicit types" rule.
**Fix:** Narrow with a runtime check instead of asserting, e.g.
`const endgameElo = typeof v === 'number' ? v : undefined;` (the component already
uses this exact `typeof n === 'number'` pattern for the volume-bar sum on line 274),
or give the merged row a precise typed shape.

## Info

### IN-01: Dead/no-op branch in `computeDefaultHidden`

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:117-121`
**Issue:** The `if (combos.length <= MAX_DEFAULT_VISIBLE) { ... }` block contains
only comments and no statements — it never short-circuits and the code below runs
unconditionally anyway. It reads as if it should `return new Set()` early but
doesn't. It's harmless (the general path produces the same result) but confusing
dead structure.
**Fix:** Delete the empty `if` and keep the explanatory comment as a plain comment
above the loop.

### IN-02: Magic number `* 5` for the volume-bar axis envelope

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:475`;
`frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:204`
**Issue:** `domain={[0, barMax * 5]}` hard-codes the "bars occupy bottom 20%"
factor as a bare `5` in two files. CLAUDE.md: "extract thresholds... into named
constants".
**Fix:** Extract a shared `const VOLUME_BAR_AXIS_HEADROOM = 5;` (bottom 1/5 of
canvas) and use it in both charts.

### IN-03: Duplicated `useIsMobile` hook

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:52-64`;
`frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:39-51`
**Issue:** The exact same `useIsMobile()` implementation (and
`MOBILE_BREAKPOINT_PX = 768`) is copy-pasted in both chart files. Code duplication;
a breakpoint change must be made in two places.
**Fix:** Promote to a shared hook (e.g. `@/lib/useIsMobile`) and import in both.

### IN-04: Non-deterministic `key={i}` on gradient `<stop>` elements

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:586`;
`frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:256`
**Issue:** Gradient stops use the array index as the React key. Stops are
recomputed wholesale on data change and there is no reordering, so this is
functionally safe, but index keys on a list whose contents change between renders
is a known React anti-pattern and the lint config may flag it.
**Fix:** Key by a stable composite, e.g. `key={`${stop.offset}-${stop.color}-${i}`}`.

### IN-05: `barMax` via spread of a potentially large array

**File:** `frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:134`
**Issue:** `Math.max(1, ...data.map((r) => r.per_week_total_games))` spreads every
weekly row as call args. For multi-year weekly timelines this is hundreds of
elements — fine today, but `Math.max(...arr)` can hit the argument-count limit on
very large arrays and is an O(n) allocation each render (not memoized, unlike the
sibling chart which computes `barMax` in `useMemo`). Out-of-scope as a perf issue,
but the inconsistency with `EndgameEloTimelineSection`'s memoized `barMax` is a
maintainability smell.
**Fix:** Use a reduce and/or wrap in `useMemo` to match the sibling chart's pattern.

### IN-06: Comment claims a guarantee the code does not enforce

**File:** `frontend/src/components/charts/EndgameScoreOverTimeChart.tsx:100-103`
**Issue:** The comment "Plan 01 guarantees endgame_score / non_endgame_score are
present on every point — no fallback needed" then does
`Math.round(p.endgame_score * 100)`. If the backend contract ever regresses to
`null`/`undefined`, `Math.round(null * 100)` yields `0` (silently plots 0%) and
`undefined` yields `NaN` (silently drops the point) with no error surfaced. The
type system enforces non-null here so this is informational, but a defensive
assertion or Sentry breadcrumb would fail loudly instead of mis-plotting.
**Fix:** Optionally assert finiteness once per render and capture to Sentry if
violated, consistent with the project's "fail loudly" error-handling posture.
