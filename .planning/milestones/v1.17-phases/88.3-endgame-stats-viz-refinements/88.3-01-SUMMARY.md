# Phase 88.3 Plan 01: Inactivity-Gap Annotations on Timeline Charts — Summary

Pure helper `computeInactivityGaps` + axis-break ReferenceLine annotations on both ordinal-axis over-time charts, surfacing long inactive periods that previously silently compressed into a single ordinal step.

## What Was Built

**`frontend/src/lib/inactivityGaps.ts`:** Pure helper with zero imports. Exports `INACTIVITY_GAP_THRESHOLD_DAYS = 56`, the `InactivityGap` interface (`afterIndex`, `gapDays`, `label`), and `computeInactivityGaps(sortedDates, thresholdDays?)`. Iterates consecutive date pairs, returns gap descriptors for pairs exceeding the threshold. Labels use "≈Ny inactive" format for >= 365 days and "≈Nmo inactive" for shorter gaps. Mirrors the `signedBandGradient.ts` pure-helper structure exactly.

**`EndgameScoreOverTimeChart.tsx`:** Added `ReferenceLine` to the recharts import and `computeInactivityGaps` import. The `inactivityGaps` useMemo is placed before the `if (timeline.length === 0) return null` early return to satisfy react-hooks/rules-of-hooks. Each gap renders a dashed `ReferenceLine` with `yAxisId="value"`, 30% opacity, and an `insideTopRight` label. The ordinal x-axis and `createDateTickFormatter` are unchanged.

**`EndgameEloTimelineSection.tsx`:** Same pattern. `inactivityGaps` useMemo added after the existing `allDates` useMemo. ReferenceLine annotations use `yAxisId="elo"` (matching the named YAxis in this chart). Placed before `<Bar>` as direct ComposedChart children so annotations sit behind data series in SVG z-order.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] useMemo after early-return violated react-hooks/rules-of-hooks**

- **Found during:** Task 2 lint check
- **Issue:** Plan's action placed `inactivityGaps = useMemo(...)` after `if (timeline.length === 0) return null`, which ESLint's react-hooks/rules-of-hooks correctly rejects (hooks must be called in the same order on every render)
- **Fix:** Moved `dates`, `inactivityGaps`, and `formatDateTick` declarations before the early return in `EndgameScoreOverTimeChart`. The ELO Timeline Section has no early return in the same render path (it returns different branches before reaching the chart JSX via `isError` / `isLoading` guards that precede the hooks zone), so no change was needed there.
- **Files modified:** `EndgameScoreOverTimeChart.tsx`
- **Commit:** 643f4890

**2. [Rule 3 - Blocking] Worktree test execution via duplicate React instance**

- **Found during:** Task 1 test run
- **Issue:** The worktree had its own `node_modules` directory created by `npm install` (triggered by the plan's `cd frontend && npm install`). This produced a duplicate React instance when vitest loaded the worktree's React alongside Recharts' React, causing `TypeError: Cannot read properties of null (reading 'useMemo')` at test time.
- **Fix:** Removed the worktree's `node_modules` directory and replaced it with a symlink to the main repo's `frontend/node_modules`. This is the correct worktree testing pattern — the worktree shares the main repo's installed packages.
- **Files modified:** none (filesystem-level fix only)

**3. [Rule 3 - Blocking] @vitest-environment comment text parsed as directive**

- **Found during:** Task 1 RED phase test run
- **Issue:** The initial test file header comment contained the phrase "No @vitest-environment directive" which vitest 4.x parsed as `@vitest-environment directive` — attempting to load an environment named "directive" from the filesystem, causing an "unhandled error" that prevented the test file from loading.
- **Fix:** Rewrote the comment to "Pure function tests — no DOM environment needed" — removing the `@vitest-environment` token from the comment text.
- **Files modified:** `inactivityGaps.test.ts`
- **Commit:** eea7cd7d
