## 375px mobile parity of the new ScoreGapRow on each rendered class card

**Why manual:** Real-device only — CSS interactions with adjacent Conv|Recov gauges and the WDL bar at small viewports can introduce wrap / overflow / whisker-clipping that jsdom does not surface.

**Test instructions:**
1. Run the frontend dev server (`npm run dev` in `frontend/`) and open `http://localhost:5173/endgames` with the device toolbar set to 375px width (iPhone SE / 13 mini emulation in Chrome DevTools).
2. Sign in as a user with sufficient endgame data (≥ 10 spans per rendered class) so the new row is visible.
3. On each of the 5 rendered endgame class cards (rook / minor piece / pawn / queen / mixed), confirm:
   - The new "Score Gap: ±X%" row sits in the slot between the Conv|Recov gauge row and the WDL bar (or standalone Games link if `SHOW_WDL_BAR_IN_TYPE_CARDS=false`).
   - The MiniBulletChart bullet does not horizontally overflow the card.
   - The CI whiskers (vertical ticks at `ciLow` / `ciHigh`) are visible and do not clip the card border.
   - The HelpCircle popover trigger (info-icon) is tap-accessible without overlapping the value text or the next row.

**Pass criteria:** Row renders cleanly at 375px with no visible clipping, the row order (gauges → asg row → wdl/games) matches the desktop layout, and the popover trigger is reachable.

## Sigmoid-bias caveat copy lands on every popover and reads naturally

**Why manual:** Subjective copy review — automated tests confirm the literal string is present in the rendered DOM, but the prose needs to read as an aside, not as primary content.

**Test instructions:**
1. On any rendered EndgameTypeCard with a Score Gap row, hover or tap the popover trigger (`*-asg-info`).
2. Confirm the popover body contains the sentence: "Note: the Lichess sigmoid under-weights endgame eval advantages, so absolute values are scale-compressed; rely on the zone bands, not the raw magnitude."
3. Confirm the sentence reads as a caveat / footnote, not as a primary explanation. It should sit at or near the end of the explanation block.
4. Repeat for each rendered endgame class (5 popovers total).

**Pass criteria:** Caveat present on every popover, reads as an aside (not a primary explanation), and does not contradict or duplicate the surrounding methodology block.

## Sign convention reads naturally (positive value = right of zero = green)

**Why manual:** Visual interpretation — the bullet primitive renders the bar in chart coordinates which differ from the percent label sign.

**Test instructions:**
1. Find a known-positive user (good rook conversions producing a positive type_achievable_score_gap_mean on the rook card).
2. Confirm the rook card's Score Gap row:
   - Renders the value with a leading `+` (e.g. `+8%`).
   - Renders the bar centered on 0 with the colored segment extending to the right of zero.
   - Renders the value text in `ZONE_SUCCESS` (green) when the gap > +5pp neutral band upper.
3. Find or construct a known-negative case (or use a class where the user underperforms, e.g. queen endgames for a positional player). Confirm:
   - Value renders with a leading `-` (no separate minus sign artifact).
   - Bar extends left of zero.
   - Value text renders in `ZONE_DANGER` (red) when below -5pp.

**Pass criteria:** Sign + bar direction + color all agree (positive → right → green; negative → left → red). Inside the ±5pp neutral band the text renders in the default foreground color (no tint).

## Verify LLM narration uses the dual-label rule (Plan 04)

**Why manual:** Subjective copy review — the LLM follows the prompt instructions probabilistically, so a real generation against production-shaped data is the only way to confirm the dual-label rule lands. Added by Plan 04.

**Test instructions:**
1. Trigger an `/api/insights/endgame` request for a user with ≥ 10 endgame spans across at least 2 endgame types (rook + queen is a good combination for contrast).
2. Bypass the cache (the prompt version bump to `endgame_v29` should auto-invalidate prior cached reports, but force a fresh generation if needed).
3. In the resulting `type_breakdown` section text, confirm:
   - When narrating a specific endgame type (rook / queen / pawn / minor piece / mixed) inside a "card context" sentence, the LLM uses the short form **"Score Gap"** (e.g. "your rook Score Gap sits at +6%").
   - When introducing the metric for the first time in the narrative, or when comparing it against the page-level Achievable Score Gap, the LLM uses the full form **"Endgame Type Score Gap"**.
   - The LLM does NOT use the internal identifier `type_achievable_score_gap` in narration.
   - The LLM does NOT use forbidden coinages: "ΔES", "delta_es", "dES".
4. Confirm the LLM does not mention a p-value or significance test in connection with the Score Gap — the cohort band is the only significance signal per `feedback_llm_significance_signal.md`.

**Pass criteria:** Both label forms appear at least once in a 3+ endgame-type narrative (full form on first mention, short form afterwards). No internal identifiers leak into user-facing prose. No p-value editorialising on the Score Gap.
