# Phase 87.1 Plan 03: Frontend Card Extension Summary

**One-liner:** Added per-span Score Gap row to EndgameTypeCard (positioned between Conv|Recov gauges and WDL bar) using lifted ScoreGapRow + MetricStatPopover with dual-label terminology and sigmoid-bias caveat.

## Outcome

Every rendered `EndgameTypeCard` now carries a `ScoreGapRow` bullet for the new per-type Score Gap metric, slotted directly between the gauge row and the WDL bar (or standalone Games link when `SHOW_WDL_BAR_IN_TYPE_CARDS=false`). The row:

- Reuses the shipped `ScoreGapRow` primitive verbatim (no new component, no inline reinvention).
- Wires to the global `ENDGAME_TYPE_SCORE_GAP_NEUTRAL_MIN/MAX` neutral band from Plan 01's codegen.
- Hides at `type_achievable_score_gap_n === 0` (same doctrine as the score row).
- Tints `ZONE_SUCCESS` (green) when the gap >= neutralMax, `ZONE_DANGER` (red) when < neutralMin, and renders the default foreground color inside the neutral band — zone-only tinting, no p-value gate on the row.
- Surfaces the dual-label popover via `MetricStatPopover` with `name="Score Gap"`, vocabulary="score", and the mandatory sigmoid-bias caveat one-liner appended to the explanation copy.
- Carries the testid sub-elements per D-08: `${tileTestId}-asg-bullet` (wrapper), `${tileTestId}-asg-value` (value span), `${tileTestId}-asg-info` (popover trigger).

### Auto-fixed Issues

**1. [Rule 1 - Bug] jsdom oklch() decimal normalization**
- **Found during:** Task 2 GREEN phase first test run
- **Issue:** Zone-tint tests using `.toBe(ZONE_SUCCESS.toLowerCase())` failed because jsdom serializes `oklch(0.50 ...)` as `oklch(0.5 ...)` (trailing zero stripped), while the source constant in `lib/theme.ts` keeps the `0.50` form. Strict string equality mismatched.
- **Fix:** Replaced strict-equality with a `normalize` helper that strips trailing zeros from decimal fractions before comparison. Same fix for ZONE_DANGER assertion.

## Key Decisions Made

1. **Global neutral band, not per-class.** Used `ENDGAME_TYPE_SCORE_GAP_NEUTRAL_MIN/MAX` (global) rather than `PER_CLASS_GAUGE_ZONES[class].achievable_score_gap` (per-class). Per Plan 01's codegen all per-class bands were initialized to ±0.05 placeholder anyway, so the global is functionally equivalent. Switching to per-class is a one-line change if `/benchmarks` §3.4.2 (out of scope for Plan 01 — placeholder values shipped) produces meaningful per-class variation later.
2. **Em-dash usage in explanation copy.** Replaced the first em-dash in the popover explanation with a comma to honor CLAUDE.md "use em-dashes sparingly" guideline; kept the math minus sign (`exit − entry expected`) and the methodology line's `½ draws` unicode characters since they read as math notation, not English punctuation.
3. **Hover vs click in popover test.** The Radix popover uses a 100ms hover delay; the test fires both `mouseEnter` and `click` to reliably trigger the open state in jsdom without flakiness.
