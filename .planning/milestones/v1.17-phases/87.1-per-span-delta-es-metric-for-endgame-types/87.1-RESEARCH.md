## Summary

Phase 87.1 adds a continuous per-endgame-class performance metric — mean per-span Achievable Score
Gap — to the Endgame Type Breakdown cards introduced in Phase 87. The metric addresses cohort
thinning and threshold-cliff problems in Conversion/Recovery by using every span (not just ones
crossing ±1.0 cp), measuring eval movement within the span rather than game outcome, and producing
a continuous score on a signed scale centred on 0 (positive = user outperformed Stockfish baseline).

All prerequisite infrastructure is confirmed present: `eval_cp_to_expected_score` and
`eval_mate_to_expected_score` in `eval_utils.py`, `compute_paired_difference_test` in
`score_confidence.py`, the index `ix_gp_user_endgame_game INCLUDE(eval_cp, eval_mate)` that makes
span-level eval reads index-only, and `ScoreGapRow` / `MetricStatPopover` as the visual primitive.

The primary research unknowns (D-04 and D-06) are now resolved: D-06 recommends extending
`query_endgame_entry_rows` in place with a `LEAD()` window function for next-span eval; D-04
recommends extending `PerClassBands` with an `achievable_score_gap` field and adding §3.4.2 to the
benchmarks skill. A critical Phase 87 drift was found: the shipped `EndgameTypeCard` has a single
chess-score bullet (not two peer-diff bullets), which changes the insertion point for the
ScoreGapRow.

**Primary recommendation:** Four plans, Wave A (Plans 1+2 parallel) then Wave B (Plans 3+4
parallel), with benchmarks-skill + codegen in Plan 1 as the prerequisite for zone bands.

## D-06: Repo Query Strategy (Extend In Place — RECOMMENDED)

### Current shape of `query_endgame_entry_rows`

Confirmed at lines 140-230 of `endgame_repository.py`:

1. **Span subquery** (`span_subq`): `GROUP BY (game_id, endgame_class)`, `HAVING count(ply) >= 6`.
   Uses `array_agg(eval_cp ORDER BY ply)[1]` to extract entry eval. Does NOT capture `min(ply)`.
2. **Main query**: joins Game to span_subq on `game_id`. Returns 6 columns.
   `apply_game_filters()` applied to the main query.

### The extension needed

To compute `gap_span` for transitory spans, the service layer needs the NEXT span's entry eval in
the same game. This requires ordering spans within a game by their first ply — which is NOT
currently captured (only the eval at index [1] of the ordered array_agg).

**Required change to span_subq:** Add `func.min(GamePosition.ply).label("span_min_ply")` as a
selected column. The index already covers ply ordering; this is computed during the GROUP BY at
no additional scan cost.

**Window function layer:** Wrap span_subq in an outer select that applies `LEAD()`:

```python
# Source: verified against existing SQLAlchemy 2.x patterns in endgame_repository.py
from sqlalchemy import func, over

span_with_next = (
    select(
        span_subq.c.game_id,
        span_subq.c.endgame_class,
        span_subq.c.entry_eval_cp,
        span_subq.c.entry_eval_mate,
        span_subq.c.span_min_ply,
        func.lead(span_subq.c.entry_eval_cp)
            .over(
                partition_by=span_subq.c.game_id,
                order_by=span_subq.c.span_min_ply.asc(),
            )
            .label("next_entry_eval_cp"),
        func.lead(span_subq.c.entry_eval_mate)
            .over(
                partition_by=span_subq.c.game_id,
                order_by=span_subq.c.span_min_ply.asc(),
            )
            .label("next_entry_eval_mate"),
    ).select_from(span_subq)
).subquery("span_with_next")
```

The main query then joins Game to `span_with_next` instead of `span_subq`. Returns 8 columns.

### Impact on existing callers

`query_endgame_entry_rows` is called in exactly one place: `_aggregate_endgame_stats` in
`endgame_service.py`. That caller already unpacks the 6 columns by name (`row.game_id`,
`row.endgame_class`, `row.result`, `row.user_color`, `row.eval_cp`, `row.eval_mate`). Adding 2
extra columns to the result (`next_entry_eval_cp`, `next_entry_eval_mate`) does NOT break the
caller — SQLAlchemy Row objects support attribute access; unused columns are simply ignored.

The Conv/Recov builder in `_aggregate_endgame_stats` does not reference `next_entry_eval_*` and
will continue working without change.

**Tests:** `tests/test_endgame_service.py` tests `_aggregate_endgame_stats` via a fixture that
mocks the repository call. After extension, the mock must return 8-column rows (the extra columns
default to NULL in terminal spans — which is correct terminal semantics).

### Why not a sibling query (Option B)?

A sibling `query_endgame_entry_rows_with_next()` would duplicate `apply_game_filters` plumbing,
the `array_agg(eval_cp ORDER BY ply)[1]` pattern, and the Game join — all of which would need to
stay in sync with the original. It would also require a second DB round-trip per request. The
in-place extension adds two columns with zero extra DB round-trips and keeps the filter logic in
one place. [VERIFIED: codebase inspection]

### Index impact

The existing index `ix_gp_user_endgame_game INCLUDE(eval_cp, eval_mate)` covers
`(user_id, endgame_class, game_id)` with `eval_cp` and `eval_mate` as included columns.
Adding `min(ply)` to the span_subq SELECT adds a `min(ply)` aggregate — ply is the sort key in
the index, so this is free (the index already scans in ply order for the array_agg). The LEAD()
window runs on the subquery result set (one row per span), not on game_positions directly; cost
is O(spans_per_game), which is at most 6 rows. [VERIFIED: codebase inspection]

## Service-Layer Math

### Terminal vs transitory classification

```python
# Source: verified against D-07 spec and eval_utils.py function signatures
def _compute_span_gap(
    entry_eval_cp: int | None,
    entry_eval_mate: int | None,
    next_entry_eval_cp: int | None,
    next_entry_eval_mate: int | None,
    result: str,           # "win" | "draw" | "loss"
    user_color: str,       # "white" | "black"
) -> float | None:
    """Return per-span gap (exit - entry expected score), or None if eval unavailable."""
    if entry_eval_cp is None and entry_eval_mate is None:
        return None  # D-07: skip NULL-eval spans

    # Entry expected score (Lichess sigmoid, user-perspective)
    if entry_eval_mate is not None:
        es_entry = eval_mate_to_expected_score(entry_eval_mate, user_color)
    else:
        es_entry = eval_cp_to_expected_score(entry_eval_cp, user_color)  # type: ignore

    # Exit score: transitory if next span exists, terminal if game ended
    is_transitory = (next_entry_eval_cp is not None or next_entry_eval_mate is not None)
    if is_transitory:
        if next_entry_eval_mate is not None:
            exit_score = eval_mate_to_expected_score(next_entry_eval_mate, user_color)
        else:
            exit_score = eval_cp_to_expected_score(next_entry_eval_cp, user_color)  # type: ignore
    else:
        # Terminal: game result as score
        result_to_score = {"win": 1.0, "draw": 0.5, "loss": 0.0}
        exit_score = result_to_score[result]

    return exit_score - es_entry  # positive = outperformed
```

## Common Pitfalls

### Pitfall 1: Forgetting `min_ply` in the span subquery
**What goes wrong:** The LEAD window function requires ordering spans within a game by their
temporal position. Without `min(ply)` in the span subquery, the LEAD() ORDER BY has no valid
column.
**Why it happens:** The current `span_subq` only captures `entry_eval_cp/mate` via `array_agg()[1]`
(which sorts by ply internally) but does not expose `min_ply` as a named column.
**How to avoid:** Add `func.min(GamePosition.ply).label("span_min_ply")` to the span_subq SELECT
and use it as the LEAD ORDER BY.
**Warning signs:** `sqlalchemy.exc.CompileError` or `ProgrammingError` referencing missing ORDER BY
column in the window function.

### Pitfall 2: Sign convention mismatch between SEED-016 and CONTEXT.md
**What goes wrong:** SEED-016 defines `ΔES_span = ES_sigmoid(entry_eval) − exit_score` (positive
= gave back score). CONTEXT.md D-07 flips to `gap_span = exit_score − ES_entry` (positive =
outperformed). The page-level `achievable_score_gap` uses `exit_score − ES_entry`.
**Why it happens:** SEED-016 used "gave back" framing; CONTEXT.md switched to "outperformed"
framing for consistency with the page-level metric.
**How to avoid:** Use `exit_score - ES_entry` (matching CONTEXT.md D-07 and page-level metric).
**Warning signs:** Zone assignments inverted (strong users showing "weak", weak users showing
"strong").

### Pitfall 3: Treating NULL `next_entry_eval_cp` as terminal when it could be a NULL eval
**What goes wrong:** A span row may have `next_entry_eval_cp = NULL` either because it's the last
span in the game (terminal) OR because the next span's entry eval was not backfilled.
**Why it happens:** The LEAD() window returns NULL for the last row in the partition AND for rows
where the referenced column was NULL.
**How to avoid:** The 100% eval coverage in prod (337,345/337,350 ≥6-ply codebase spans have
eval — effectively 0 missing) means this is a theoretical edge case. In code, treat any span with
`next_entry_eval_cp = NULL AND next_entry_eval_mate = NULL` as terminal. The 5 missing evals out
of 337k will be excluded by the NULL-entry check (D-07: skip if `entry_eval_cp` and
`entry_eval_mate` are both NULL).
**Warning signs:** Terminal spans computing non-game-result exit scores.

### Pitfall 4: Emitting `endgameZones.ts` drift without running codegen
**What goes wrong:** CI fails with "endgameZones.ts out of sync" if `gen_endgame_zones_ts.py` is
modified but the TS file is not regenerated.
**Why it happens:** The script is not auto-run on save; the drift check is a CI gate only.
**How to avoid:** After any change to `endgame_zones.py` or `gen_endgame_zones_ts.py`, run
`python scripts/gen_endgame_zones_ts.py` and commit the updated `endgameZones.ts`.

### Pitfall 5: Forgetting `ty` compliance on new Python fields
**What goes wrong:** `uv run ty check app/ tests/` fails in CI on the 5 new `EndgameCategoryStats`
fields if type annotations are missing or wrong.
**Why it happens:** ty enforcement is strict; `float | None` and `int | None` are mandatory.
**How to avoid:** Follow the existing pattern in `EndgamePerformanceResponse` for the page-level
`achievable_score_gap_*` fields (already typed as `float | None`).

### Pitfall 6: `EndgameTypeCard` insertion point
**What goes wrong:** Inserting the ScoreGapRow "below peer bullets" (as stated in Phase 87.1
CONTEXT.md) fails to find peer-bullet elements because they don't exist in the shipped card.
**Why it happens:** Phase 87 Plan 02 replaced peer bullets with a single chess-score bullet.
**How to avoid:** Insert ScoreGapRow between the gauge row (`.flex.gap-4` containing Conv+Recov
gauges) and the WDL bar / score bullet block. Planner must specify the exact JSX insertion point.
