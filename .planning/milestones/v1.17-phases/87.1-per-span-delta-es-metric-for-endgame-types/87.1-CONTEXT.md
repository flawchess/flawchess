# Phase 87.1: Per-span ΔES metric for endgame types - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a new per-endgame-type continuous performance metric to the Phase 87 per-type cards: **mean per-span expected-score delta** aggregated by endgame class. SEED-016 names this `ΔES`; per user direction at discuss-phase, **the math, visual primitive, and statistical helper reuse the existing "Achievable Score Gap" pattern** (Phase 85.1 / SEC1-10/11) — same `compute_paired_difference_test` helper, same `MiniBulletChart` / `ScoreGapRow` visual, same `higher_is_better` sign convention. User-facing naming uses a **dual-label scheme** (amended 2026-05-15 to disambiguate from the page-level metric without bloating the card visual):

- **Endgame statistics concepts section (glossary):** "Endgame Type Score Gap" — full qualifier. The concepts surface stands alone and sits next to two existing "Score Gap" terms ("Endgame Score Gap" = Endgame − Non-Endgame Score; "Achievable Score Gap" = Actual − Expected), so a bare "Score Gap" would conflate.
- **Per-type card row label:** "Score Gap" — short form. The card title ("Rook Endgames" etc.) supplies the disambiguating context. Matches the established page convention where concepts use full forms and card UIs use abbreviated forms.
- **Internal identifier (unchanged):** `endgame_type_achievable_score_gap` in code / zone registry / schema field prefix. Preserves the math-family link to the page-level `achievable_score_gap` for code readers. Internal/external label divergence is intentional: developers grep `achievable_score_gap` to find related code; users see "Score Gap" / "Endgame Type Score Gap" to avoid ambiguity with the page-level metric. Add a one-line comment near the `MetricId` literal and the schema field cluster explaining the discrepancy.

**Mechanics (per SEED-016, sign flipped to match the existing Achievable Score Gap convention `actual − expected`):**

For each endgame sequence (one per `(game_id, endgame_class)` per the existing codebase span definition, ≥6 plies):

```
gap_span = exit_score_span − ES_sigmoid(entry_eval_span, user_perspective)
```

where:
- **`entry_eval_span`** — `(eval_cp, eval_mate)` at the first ply of the span (already surfaced by `query_endgame_entry_rows` and clipped via the existing `EVAL_CLIP_MAX_CP` / `_signed_pawns_from_mate` convention).
- **`ES_sigmoid`** — Lichess winning-chances sigmoid from `app/services/eval_utils.py` (same biased-but-monotonic sigmoid used by `entry_expected_score`; calibration caveat documented in `.planning/notes/lichess-sigmoid-endgame-calibration.md`, mitigated by percentile-derived zone bands and InfoPopover copy).
- **`exit_score_span`** —
  - **Transitory span** (followed by another span in the same game): `exit_score = ES_sigmoid(entry_eval_of_next_span, user_perspective)`.
  - **Terminal span** (last span in the game): `exit_score = game_result_score` (1.0 / 0.5 / 0.0 mapped from `Game.result` × `Game.user_color`).
- Spans where either endpoint's eval is NULL are excluded from the cohort (per SEED-016; non-issue in practice — ≥6-ply prod coverage is 100%).

**Aggregation per (user, endgame_class):** `mean(gap_span)` over all qualifying spans, with paired one-sample two-sided z-test against `H0: mean = 0` via `compute_paired_difference_test` from `app/services/score_confidence.py:338` — the exact helper used by Phase 85.1 SEC1-10 for the existing page-level Achievable Score Gap. Returns `(mean, p_value, ci_low, ci_high)` with the standard `CONFIDENCE_MIN_N = 10` p-value gate and `n >= 2` CI gate.

**Sign convention** (LOCKED at discuss): positive = user outperformed Stockfish expectation across spans of this type (recovered eval, decisive conversions); negative = user gave back expected score (drifted, failed conversions). Matches `higher_is_better` on the existing `achievable_score_gap` ZoneSpec — the visual reads identically: right of zero is green, left is red.

**Display surface (LOCKED):** one new horizontal bullet row appended **below the side-by-side Conv | Recov gauge row** on every `EndgameTypeCard` (Phase 87). Visual primitive = `MiniBulletChart` via the existing `ScoreGapRow` pattern (the same pattern Phase 85.1 ships for the page-level Achievable Score Gap). NOT a third gauge. CI whiskers, neutral band tinting, `MetricStatPopover` info trigger — all reused verbatim.

**Out of scope:**
- Refit of an endgame-specific sigmoid — `.planning/notes/lichess-sigmoid-endgame-calibration.md` tracks this as a research follow-up; percentile zones absorb the bias for V1.
- Strict-contiguous-run span definition — SEED-016 §Open Details recommends codebase span (`GROUP BY game_id, endgame_class`) for cohort consistency with Conv/Recov; defer the strict alternative.
- Lowering `ENDGAME_PLY_THRESHOLD` below 6 plies — keep the cohort definition consistent across the per-type card's three metrics (Conv, Recov, this).
- Per-span (vs per-move) normalisation — explicitly rejected in SEED-016; per-span matches the Conv/Recov cohort grain.
- Sub-`ScoreGapRow` peer comparison — ΔES has no natural opponent mirror; the metric is self-vs-Stockfish-baseline. Single-bullet doctrine satisfied by the gap bar itself (centred on 0).
- Mobile WDL-bar drop decision — handed back to Phase 87's `SHOW_WDL_BAR_IN_TYPE_CARDS` HUMAN-UAT. Phase 87.1 inherits whatever Phase 87 lands on; if 87 ships with `SHOW_WDL_BAR_IN_TYPE_CARDS = false`, the new ΔES row simply slots into the same layout slot directly below the gauges.

## Implementation Decisions

### Visual treatment (LOCKED — user direction "reuse Achievable Score Gap")

- **D-01: ΔES renders as a `ScoreGapRow` bullet, not a gauge.**
  - Visual primitive: `MiniBulletChart` (the same primitive used by `ScoreGapRow` in `EndgameOverallPerformanceSection.tsx:170-200`). Horizontal bar centred on 0, CI whiskers, neutral-band tinting, sig-gated font color on the headline value.
  - Position on `EndgameTypeCard` (Phase 87): **directly below the side-by-side Conv | Recov gauge row**, above (or replacing the layout slot of) the WDL bar — final ordering is planner's call, but the row lives in the "Score Gap" semantic slot of the card.
  - **UI label on the card row: "Score Gap"** (short form — the card title already announces the endgame type). No "Achievable", no "ΔES" / "delta_es" in user-facing copy.
  - Rationale: three side-by-side gauges per card was the dense-mobile alternative and was rejected; a bullet row below the gauges keeps the per-type card hierarchically "two-metric gauges + one delta bar", which is the same grammar as the Section 1 card (`EndgameOverallEntryCard` / `EndgameOverallPerformanceSection`). Reuses an already-shipped visual primitive instead of inventing a per-card variant.

- **D-02: Dual-label naming scheme (AMENDED 2026-05-15 — was "Achievable Score Gap" verbatim).**
  - **Glossary / concepts section label:** **"Endgame Type Score Gap"** (full qualifier). Used wherever the metric is defined in isolation — the endgame statistics concepts panel, the page-level `InfoPopover` body, any documentation prose. Disambiguates from the two existing page-level "Score Gap" terms (Phase 85's "Endgame Score Gap" and Phase 85.1's "Achievable Score Gap").
  - **Per-type card row label:** **"Score Gap"** (short form). The card title ("Rook Endgames" etc.) supplies the type context, so the qualifier is redundant at this scope. Matches the page-wide convention where concepts use full forms and card UIs use abbreviated forms.
  - **Popover prose (running text in `MetricStatPopover`):** Use "Score Gap" verbatim to match the card row label the user just clicked. Optionally include one sentence relating the metric to the page-level Achievable Score Gap family ("This is the per-span, per-type version of the page-level Achievable Score Gap."). See D-09 for the full copy.
  - **Internal identifier (CODE — unchanged from pre-amendment):** `endgame_type_achievable_score_gap` in `MetricId` Literal, zone registry, schema fields, LLM payload field names. Preserves the math-family grep-ability with the page-level `achievable_score_gap`. The internal/external divergence is intentional and worth a one-line comment at the `MetricId` literal site:
    ```

The existing `PER_CLASS_GAUGE_ZONES` registry covers conv/recov; ΔES needs per-class bands too because the metric is per (user, class).

### Statistical treatment (LOCKED — SEED-016 + Phase 85.1 SEC1-10 parity)

- **D-03: Reuse `compute_paired_difference_test` (`app/services/score_confidence.py:338`) — no new statistical math.**
  - Input: a `Sequence[float]` of per-span `gap_span` values for the (user, class) cohort.
  - Output: `(mean, p_value, ci_low, ci_high)`. p-value gated to None at `n < CONFIDENCE_MIN_N = 10`; CI gated at `n < 2`. Variance-0 trap (all identical diffs) already handled per the helper's docstring.
  - This is the **same helper** Phase 85.1 uses for the page-level Achievable Score Gap; the wire shape downstream (frontend types, `MetricStatPopover` props) mirrors that exactly. Per memory `feedback_wilson_chess_score.md` and `feedback_llm_significance_signal.md`: trust the established stat method, no parallel verdict field.
  - Sign convention (per the Phase boundary): per-span input to the helper is `exit_score − ES_entry` (NOT SEED-016's `ES_entry − exit_score`), so positive = user outperformed (right of zero on the bullet = green, matches the existing `achievable_score_gap` `higher_is_better` direction).

### Zone bands (LOCKED — percentile-derived from /benchmarks, no editorial guess)

- **D-04: New `/benchmarks` skill metric `per_span_achievable_score_gap_by_class` (or sibling name).**
  - Compute the per-span gap series for each user in the benchmark cohort, partition by `endgame_class`, surface per-(rating × TC × class) p25/p50/p75 and Cohen's-d collapse verdict per axis.
  - Output: regenerate `frontend/src/generated/endgameZones.ts` with new per-class bands (e.g. `PER_CLASS_GAUGE_ZONES[<class>].achievable_score_gap` or a sibling map keyed off the new `MetricId`). Planner: pick the registry shape that matches collapse verdict (per-(class) global vs per-(class × TC) vs per-(class × ELO)).
  - Per memory `feedback_zone_band_judgement.md`: tighten the band if Cohen's-d collapse verdict says "review/keep separate" AND the editorial read is that small effects are meaningful here. The existing page-level `achievable_score_gap` band tightened from ±10pp to ±5pp under a similar judgement (260514-kei). The new per-type metric will likely sit in a tighter band again because (a) the Lichess sigmoid compresses endgame eval differentials, and (b) per-type cohorts are smaller so individual span quality matters more.
  - **Hidden classes (`pawnless` per `HIDDEN_ENDGAME_CLASSES`)** still get a registry entry — keeps the bucket type-system uniform — but the entry is unused at the UI layer.

### Backend schema & wiring (LOCKED — Phase 86/87 parity)

- **D-05: Extend `EndgameCategoryStats` (`app/schemas/endgames.py`) with the new per-class fields.**
  - New fields (all `float | None` except `_n: int`): `type_achievable_score_gap_mean`, `type_achievable_score_gap_n`, `type_achievable_score_gap_p_value`, `type_achievable_score_gap_ci_low`, `type_achievable_score_gap_ci_high`.
  - TS type mirror: extend `EndgameCategoryStats` in `frontend/src/types/endgames.ts` with the same five fields. Pattern matches the existing page-level Achievable Score Gap field naming on `EndgameOverallStats` (`achievable_score_gap`, `achievable_score_gap_p_value`, `achievable_score_gap_ci_low`, `achievable_score_gap_ci_high`) — the `_mean` suffix here disambiguates because the unsuffixed name is "too generic" at the category level.
  - **Service wiring site:** same per-class category builder in `app/services/endgame_service.py` that Phase 87 augments with the conv/recov peer-diff fields (Phase 87 D-01). Phase 87.1 adds a third per-class computation inside the same loop: collect per-span `gap_span` values, call `compute_paired_difference_test`, populate the five new fields. Comment block cites Phase 87.1 + SEED-016 + D-03/D-05.

- **D-06: Repo query strategy — deferred to researcher / planner per user direction.**
  - The current `query_endgame_entry_rows` (`app/repositories/endgame_repository.py:140`) returns `(game_id, endgame_class, result, user_color, eval_cp, eval_mate)`. Phase 87.1 additionally needs the **next span's `entry_eval_cp` / `entry_eval_mate` per game**, computed via window function over `(game_id, span_min_ply ASC)`. Terminal spans return NULL on the next-eval fields and the service-layer falls back to `(result, user_color)` for `exit_score`.
  - **Options the planner picks between:**
    - **(a) Extend `query_endgame_entry_rows` in place** — add `next_entry_eval_cp`, `next_entry_eval_mate` columns to the existing tuple shape via a window function. Changes the row shape for all current callers (the Conv/Recov categories builder + Phase 87's peer-diff wiring); they ignore the new columns.
    - **(b) Sibling query** — `query_endgame_span_gap_rows` (or similar) that returns the per-span (entry, next_entry) eval pair plus result/user_color. Keeps callers separated; duplicates `apply_game_filters` plumbing and the entry-eval array_agg trick.
  - Planner should weigh: (i) does Phase 87 land first and is the existing tuple shape locked in tests, (ii) is the window-function cost negligible on the existing index `ix_gp_user_endgame_game INCLUDE(eval_cp, eval_mate)`, (iii) does the service-layer want a clean single-pass over spans or two passes.
  - SEED-016 V1 sketch and the ROADMAP success criteria both accept either path; the math is identical.

### Backend math (LOCKED helper, planner picks wrapper site)

- **D-07: Per-span `gap_span` computation lives in the service layer, not the repo.**
  - Inputs from the repo row: `(entry_eval_cp, entry_eval_mate, next_entry_eval_cp, next_entry_eval_mate, result, user_color)`.
  - The service computes `ES_entry` and `exit_score` per span using the existing helpers:
    - **`ES_entry`** = `lichess_winning_chances(_signed_pawns_from_eval(entry_eval_cp, entry_eval_mate, user_color))` — same helper chain used by `entry_expected_score` (Phase 83 D-17).
    - **`exit_score`** = `lichess_winning_chances(_signed_pawns_from_eval(next_entry_eval_cp, next_entry_eval_mate, user_color))` if next-eval is non-NULL, else `_game_result_score(result, user_color)` (the same `result × user_color → {1.0, 0.5, 0.0}` mapping used by the existing `achievable_score_gap` aggregator on `bucket_rows`).
  - **Wrapper:** planner may add a thin per-class helper `_compute_type_achievable_score_gap_from_rows(rows) -> tuple[float | None, int, float | None, float | None, float | None]` for readability (mirrors the Phase 85.1 pattern), or call `compute_paired_difference_test` inline in the categories builder. No new statistical primitive.
  - **Mate at endpoints:** existing `_signed_pawns_from_mate` convention saturates via the sigmoid (mate-in-N maps to ±1_000_000 cp magnitude → sigmoid clamps to ~1.0 or ~0.0). No new mate handling.
