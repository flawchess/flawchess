## Decisions Made

1. **In-renderer synthesis, not service-layer emission.** The plan's `files_modified`
   field explicitly excludes `app/services/insights_service.py`, so I added the
   per-class Score Gap emission inside `_assemble_user_prompt`. The synthesizer
   reads `findings.type_categories` directly (the same data Plan 02 populates).
   This keeps the change isolated to the prompt layer; a future refactor can
   migrate the emission to `insights_service._findings_*` if other tabs need
   the same data shape. Trade-off: the test pipeline needs `type_categories`
   in the fixture, but every existing per-type test already passes that.

2. **Global `assign_zone` for synthesized findings, per-class inline-band
   dispatch in `_format_zone_bounds`.** Both `ZONE_REGISTRY["endgame_type_achievable_score_gap"]`
   and `PER_CLASS_GAUGE_ZONES[<class>].achievable_score_gap` currently hold the
   same ±5% placeholder, so a single global `assign_zone` call produces the
   same zone label that a per-class dispatcher would. The inline `(typical LO to UP)`
   band reads per-class today (so the wiring is in place for the post-§3.4.2
   recalibration); the zone-label dispatcher should follow once the bands diverge.
   Comment in the synthesizer flags this for the future change.

3. **Phase 85.1 page-level `achievable_score_gap` still not wired into LLM payload.**
   Out of scope for Plan 04. The plan only requires the per-type version. The
   page-level version's LLM wire-up remains a separate follow-up (likely Phase 85.1
   Plan 05 if/when the team decides to surface it in the prompt).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing Critical] Synthesizer inside `_assemble_user_prompt`**
- **Found during:** Task 1 (implementation)
- **Issue:** The plan's pseudocode (CONTEXT D-10 + PATTERNS) describes payload
  assembly per category but the actual codebase architecture uses the
  `SubsectionFinding` pipeline. Without a SubsectionFinding emission, the v29
  bump invalidates caches but the LLM never receives the new metric — the
  ROADMAP §6 success criterion ("LLM payload type_breakdown carries
  type_achievable_score_gap per class") would not be met in production.
- **Fix:** Added `_synthesize_endgame_type_achievable_score_gap_findings(findings)`
  inside `insights_llm.py` (the plan's only-modify file) that reads
  `findings.type_categories` and emits one `SubsectionFinding` per non-pawnless
  class with a populated mean. The existing renderer pipeline handles the rest.
- **Files modified:** `app/services/insights_llm.py` (helper + wiring in `_assemble_user_prompt`)
- **Verification:** New tests in `TestEndgameTypeAchievableScoreGapPayload` pass; the rendered prompt includes `[summary endgame_type_achievable_score_gap | endgame_class=<class>]` for each populated class.
- **Committed in:** `662a3fff`

**2. [Rule 2 - Missing Critical] Per-class inline-band dispatch in `_format_zone_bounds`**
- **Found during:** Task 1 (implementation)
- **Issue:** `_format_zone_bounds` only dispatched per-class for
  `conversion_win_pct` / `recovery_save_pct` — the new metric would route through
  the global `ZONE_REGISTRY` path, producing the same band globally. Once
  benchmark §3.4.2 recalibrates per-class bands the global path would silently
  contradict the per-class label.
- **Fix:** Added an `elif` branch for `endgame_type_achievable_score_gap` that
  reads `PER_CLASS_GAUGE_ZONES[<class>].achievable_score_gap` and synthesizes a
  `ZoneSpec(lo, hi, "higher_is_better")`. Currently produces the same `(typical
  -5 to +5)` band because the placeholder is uniform, but the wiring is in
  place for the future recalibration.
- **Files modified:** `app/services/insights_llm.py` (`_format_zone_bounds`)
- **Verification:** `test_gap_inline_band_dispatches_per_class` asserts the
  per-class registry is the source.
- **Committed in:** `1497b76d`

## Issues Encountered

- **Plan's pseudocode mismatched architecture.** The plan's CONTEXT D-10 +
  PATTERNS Delta 1 wrote pseudocode treating the LLM "payload" as a dict-shape
  JSON object (`type_breakdown.<class>.type_achievable_score_gap = { mean, n, zone, neutral_band }`).
  The actual codebase architecture uses a `SubsectionFinding` pipeline that
  renders prose `[summary ...]` blocks; there is no dict payload exposed to the
  agent. Resolved by treating the pseudocode as semantic intent and emitting
  one `SubsectionFinding` per class with the equivalent fields (mean -> value,
  n -> sample_size, zone -> zone, neutral_band -> inline `(typical LO to UP)`
  via `_format_zone_bounds`). The plan's grep acceptance criteria all match
  the chosen implementation.
