## 1. FIDE Performance Rating with Laplace Smoothing — Edge Cases

### 1a. Formula canonicity

The CONTEXT-locked form is:

```
PR = R_opp_avg + 400 · log10(s* / (1 − s*))
s* = (n · s + 1) / (n + 2)
```

This is the textbook Laplace rule-of-succession applied to a Bernoulli/multinomial likelihood: prior pseudo-count of 1 success and 1 failure (the "+1" numerator and "+2" denominator), shifting the MLE toward 0.5. Equivalence to the CONTEXT design-note framing `s* = (wins + 0.5·draws + 1) / (n + 2)` is algebraic when `score = (wins + 0.5·draws) / n` — confirmed by inspection. **VERIFIED via codebase scan + math derivation.** [VERIFIED: derivation]

For draws, the +1 pseudo-count is split implicitly into +0.5 pseudo-win and +0.5 pseudo-loss (chess scoring is half-integer-valued, but Laplace's rule generalises naturally to bounded continuous outcomes — the pseudo-count is added to the score moment, not the success count).

### 1b. n=100 bound

Worst-case scores at `n=100`:
- `s = 0` → `s* = 1/102 = 0.00980`; PR delta = `400 · log10(0.00980 / 0.99020) = −801.73 ELO`
- `s = 1` → `s* = 101/102 = 0.99020`; PR delta = `400 · log10(0.99020 / 0.00980) = +801.73 ELO`

[VERIFIED: Python `math.log10`]

The design note's "**bounds PR delta near ±900 ELO at n=100**" (`endgame-elo-pr-direct-rebuild.md:90`) is conservative — the true bound is **±802 ELO**. Off by ~100 but still inside "no realistic user reaches this" territory. The CONTEXT D-01 line "Bounds PR delta near ±900 ELO at n=100; no clamping at output" should land at ±900 verbally but the planner may want to write the docstring with the exact ±802 number for accuracy.

**Planner action:** PR helper docstring can state "PR delta bounded at ±802 ELO at n=100" with full precision; the design note's ±900 number is an upper-bound estimate.

### 1c. Numerical stability of `log10(s/(1-s))` vs split form

Tested at extreme inputs (`s ∈ {0.001, 0.01, 0.5, 0.99, 0.999}`):

```
form A = 400 * math.log10(s_star / (1 - s_star))
form B = 400 * (math.log10(s_star) - math.log10(1 - s_star))
diff   = 0.0 in all cases
```

[VERIFIED: Python double-precision arithmetic]

CPython's `math.log10` is monotone correctly-rounded for both forms. With Laplace smoothing pinning `s* ∈ [1/(n+2), (n+1)/(n+2)]` and `n ≥ 10`, the argument `s*/(1-s*) ∈ [1/11, 11]` at worst — comfortably inside double-precision sweet spot. **No catastrophic cancellation possible.** Use the simple inline form `math.log10(smoothed / (1 - smoothed))`; no stability rewrite. [VERIFIED: derivation + numerical test]

### 1d. Laplace vs Bayesian-toward-0.5

The CONTEXT-locked Laplace prior of `(1, 1)` shifts `s` toward `1/2`. A Bayesian shrinkage prior toward `0.5` (e.g. Beta(α, α) with α > 1 for stronger shrinkage) would be a generalisation. Both are valid; the design note picks Laplace as the simplest interpretable rule with a single rationale ("Add one synthetic win and one synthetic loss"). Given `MIN_GAMES_FOR_TIMELINE = 10` is already enforced upstream by `_compute_score_gap_timeline` (`endgame_service.py:947-948`) and the smoothing only matters at the `n=100` boundary where `s` approaches `0` or `1`, the choice is functionally equivalent to any small-α Beta prior at the precision the chart renders to (rounded `int` ELO).

**Verdict:** Laplace is the right call. Don't second-guess this. Document the choice in the helper docstring as "add 1 win and 1 loss as priors to handle 100/0 streaks." [ASSUMED — equivalence at low precision verified numerically; absent published precedent for "Laplace on FIDE PR" specifically since FIDE itself uses table lookups, not the closed-form smoothing.]

## 2. The Midpoint Property — Tolerance Bounds

### 2a. Equal opp pools, symmetric score gap (the canonical case)

With `s_E = s_all + g/2`, `s_N = s_all − g/2`, `n_E = n_N = 100`, `R_opp_avg` equal on both sides:

| `s_all` | `gap = ±0.25` | `gap = ±0.20` | `gap = ±0.10` | `gap = 0` |
|---------|---------------|---------------|---------------|-----------|
| 0.50 | drift = 0.00 ELO | 0.00 | 0.00 | 0.00 |
| 0.55 | drift = +1.05 | n/a | 0.00 | −0.34 |
| 0.60 | drift = +2.26 | n/a | +0.01 | −0.70 |
| 0.76 (1700 vs 1500 pool) | n/a | n/a | drift = +0.80 (at g=−0.10), −1.59 (at g=−0.05), −2.37 (at g=0) | — |

[VERIFIED: Python simulation, see Bash output 2026-05-17]

The drift `|midpoint(PR_E, PR_N) − PR_all|` is **≤ 2.3 ELO** across all tested points with `|g| ≤ 0.20` and `s_all ∈ [0.5, 0.6]`. The design-note 5-ELO test tolerance is comfortable.

**At s_all = 0.947 (a 2000-rated user vs 1500 pool — unrealistic but stress-test territory)**, drift at `g = ±0.10` blows out to **+71.58 ELO**. This is the "logit nonlinearity bites at extremes" regime the design note flags (`endgame-elo-pr-direct-rebuild.md:64`). **Test fixtures must keep `s_all` in [0.4, 0.6] OR use rating-matched opp pools (so `s_all` ≈ 0.5 by construction) to stay inside the 5-ELO bound.** [VERIFIED: simulation]

### 2b. Asymmetric opp pools

Tested with `R_opp_avg_E ≠ R_opp_avg_N`, user playing at rating level in both subsets (`s_X = expected_score(actual_elo, R_opp_X)`):

| pool spread (E − N) | midpoint | drift from `actual_elo` |
|---------------------|----------|-------------------------|
| 0 | 1500.00 | +0.00 |
| 20 | 1500.00 | +0.00 |
| 50 | 1500.00 | +0.00 |
| 100 | 1500.00 | +0.00 |
| 200 | 1500.00 | +0.00 |

[VERIFIED: Python simulation]

**This is the key insight: when the user performs at rating level in both subsets, the midpoint stays anchored at `actual_elo` regardless of how far apart the per-side opp pools drift.** The math: `PR_E = R_opp_E + 400·log10(s_E*/(1-s_E*))`, and if `s_E ≈ expected_score(actual_elo, R_opp_E)`, the second term simplifies to `actual_elo − R_opp_E` exactly (by the Elo logistic-skill assumption — `expected_score(R_u, R_o) = 1/(1+10^((R_o-R_u)/400))` so `log10(s/(1-s)) = (R_u - R_o)/400`). The `R_opp_E` cancels: `PR_E = actual_elo` exactly.

The same cancellation makes `PR_N = actual_elo` exactly, so `midpoint = actual_elo` for any pool spread.

**Test assertion implication:** the design-note 5-ELO tolerance is fine. In synthetic fixtures with `s_E = s_all + g/2` and equal pools, the property is exact (drift = 0 ELO) for any `s_all = 0.5`; near-exact for `s_all ∈ [0.4, 0.6]`; preserves the bound for pool spreads up to ≥200 ELO. **The planner can write the test as `abs(midpoint - actual_elo) <= 5` and it will pass with margin.** [VERIFIED: simulation]

### 2c. Test fixture recommendation

When writing the midpoint property unit test (CONTEXT §Tests bullet 2), construct fixtures with:
- `actual_elo = R_opp_avg` (so user is at pool level, `s_all = 0.5` by construction)
- `s_E = 0.5 + g/2`, `s_N = 0.5 - g/2` for `g ∈ {-0.20, -0.10, 0, +0.10, +0.20}`
- Equal `endgame_opp_rating_avg = non_endgame_opp_rating_avg = R_opp_avg`

The drift is 0.00 ELO at all five points. The 5-ELO tolerance leaves headroom for any future fixture variation. **Do NOT extend the property test to `s_all = 0.95` regime** — nonlinearity kills it there, but that regime is outside the §3.1.6 score-gap data range, so users will never trigger it.

| Aspect | Level | Reason |

| Equal-pool tolerance | HIGH | Exact symbolic derivation, simulation 0 ELO drift |
| Asymmetric-pool tolerance | HIGH | Exact cancellation via Elo logistic identity |

## 3. Recharts `<Area>` + `<linearGradient>` Layering

### 3a. Multi-gradient pattern is well-supported

[CITED: leanylabs.com/blog/awesome-react-charts-tips/, recharts.github.io/en-US/api/ComposedChart] Recharts supports multiple `<Area>` components inside a single `<ComposedChart>`, each with its own `fill={`url(#${gradientId})`}` referencing a `<linearGradient>` inside `<defs>`. The standard SVG-defs rule applies: gradient IDs must be unique within the document, otherwise the second `<linearGradient>` with the same `id` silently shadows the first (browser behaviour, not Recharts-specific).

The codebase already has the right idiom: `EndgameScoreOverTimeChart.tsx:98-99` uses `useId()` to derive a per-mount unique prefix:

```ts
const rawId = useId();
const gradientId = `score-gap-gradient-${rawId.replace(/[^a-zA-Z0-9]/g, '_')}`;
```

`useId()` returns a stable but unique ID per React mount (e.g. `":r3:"`), and the regex strips colons because SVG `id` must match XML NCName rules. The non-alphanumeric strip is **load-bearing** — without it `url(#:r3:)` fails to resolve in some browsers.

### 3b. Per-combo gradient ID derivation

The Phase 87.6 chart has up to 8 combos (4 TCs × 2 platforms) each needing its own gradient. The pattern is:

```ts
const rawId = useId();
const baseId = `endgame-elo-band-${rawId.replace(/[^a-zA-Z0-9]/g, '_')}`;
// Per-combo:
const gradientId = `${baseId}_${combo.combo_key}`;  // e.g. "..._chess_com_blitz"
```

`combo_key` is already underscore-joined (e.g. `chess_com_blitz`) per `EndgameEloTimelineCombo` schema docstring (`app/schemas/endgames.py:579+`), so it's NCName-safe. **PATTERNS pattern 6 already calls this out** (`87.6-PATTERNS.md:103`) — research confirms it's necessary, not optional. [VERIFIED: codebase + SVG spec]

### 3c. Document order = z-order

[CITED: recharts.github.io/en-US/api/ComposedChart] Recharts renders children in the order they appear in the React child array. SVG draws later children on top of earlier ones. So for a per-combo emission of `[<Area/>, <Line PR_E/>, <Line PR_N/>, <Line Actual/>]`, the order matches CONTEXT D-04: Area (background band) → fine PR lines → bold Actual ELO on top. Confirmed.

`data.combos.flatMap(combo => [...])` produces a flat array where Recharts can locate each `<Line>` / `<Area>` via its child-traversal (`generateCategoricalChart.findAllByType`). **The existing comment block in `EndgameEloTimelineSection.tsx:447-453` documents this exact contract** — extend it, don't change it. Wrapping with `<React.Fragment>` or `<g>` breaks the type-discovery scan (Phase 68 UAT pitfall, documented in `EndgameScoreOverTimeChart.tsx:86-95`).

### 3d. Multiple Areas — pitfalls

Two pitfalls to flag for the planner:

1. **Sibling-but-not-direct-child Areas:** if a future refactor extracts the 4-element-per-combo emission into a sub-component (`<CombinedSeries combo={...}/>`), Recharts' `findAllByType` scan over `Children.toArray(props.children)` will miss them. **PATTERNS pattern 6 already calls this out** (anti-pattern bullet at `87.6-PATTERNS.md:115-116`). Research confirms: the pattern *requires* direct children. Do not factor out.

2. **`<defs>` placement:** the existing chart places `<defs>` inside `<ComposedChart>` at `EndgameScoreOverTimeChart.tsx:288`. This works because Recharts renders the chart's SVG root and all children land in the same `<svg>` document. **Defining `<defs>` multiple times (one per combo) is fine** — SVG allows multiple `<defs>` blocks; they merge by document position. Cleaner pattern: single `<defs>` containing all 8 `<linearGradient>` children, generated via `data.combos.map(combo => <linearGradient.../>)`. Both work; one-`<defs>`-eight-gradients is the more conventional choice.

### 4c. Extension vs parallel-type recommendation

**Verdict: extend `ScoreGapTimelinePoint` directly.** Rationale:

1. **Backward compat is free** — Pydantic v2 lets the field default `= 0.0` (matching the `per_week_endgame_games: int = 0` precedent at line 345). Old constructors (tests, fixtures) keep working.

3. **A parallel `EndgameEloInputPoint`** would require:
   - A second producer path (or a transform from `ScoreGapTimelinePoint` → `EndgameEloInputPoint` at the boundary)
   - A second Pydantic class with overlapping fields (`date`, `endgame_score`, `non_endgame_score`)
   - A second frontend type mirror
   - More code, no benefit — the score-gap consumers won't accidentally use opp-rating fields they don't reference.
4. **PATTERNS pattern 4** already calls this exact decision out as YAGNI (`87.6-PATTERNS.md:81`).

The CONTEXT discretion bullet (CONTEXT line 120: "the former is simpler; the latter is cleaner if the score-gap consumers shouldn't be exposed to opp-rating fields") errs on the side of cleanliness. Research disagrees politely — the cost (2 schemas, 2 mirrors, threading both through 3 files) is real and the cleanliness benefit is zero given the score-gap consumers are 1 backend service method and 1 frontend chart, both with named-field access patterns.
