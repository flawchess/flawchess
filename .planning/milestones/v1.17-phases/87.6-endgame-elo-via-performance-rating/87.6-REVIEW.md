# Phase 87.6: Code Review Report

**Reviewed:** 2026-05-17T08:06:50Z
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

Phase 87.6 swaps the additive `K · eg_score_gap` mapping for FIDE Performance Rating computed independently per side (endgame games / non-endgame games), threads `non_endgame_elo` through the API, schemas, TypeScript types, and LLM payload, and rebuilds the timeline chart as a 3-line + signed-band layout. The mechanics are sound and tested.

The defects are concentrated in three areas:

1. **Quiet population change at a shared query boundary.** The repository's `query_endgame_performance_rows` gained `white_rating IS NOT NULL AND black_rating IS NOT NULL` WHERE clauses. Two callers (`get_endgame_performance` for WDL comparison, plus the new ELO use) share this query; the WDL comparison silently loses any games with NULL ratings. There is no test asserting either the prior or the new population — the change is invisible to the suite.
2. **Overclaimed "midpoint property" in user-facing copy.** The chart popover, the prompt, and at least one docstring assert that "Actual ELO sits inside the band by construction" or "midpoint(endgame_elo, non_endgame_elo) ≈ actual_elo by construction." The schema docstring qualifies this with "|gap| <= 0.20 and equal opp pools"; the popover and prompt drop the conditions. Whenever the user's endgame opp pool diverges from the non-endgame opp pool (common in practice — e.g. arena vs casual mix), Actual ELO can sit outside the band and the LLM will narrate impossible claims.
3. **Stale documentation and zero-valued defaults.** Three docstrings still cite `K · eg_score_gap`; `ScoreGapTimelinePoint.endgame_opp_rating_avg: float = 0.0` defaults silently produce PR ≈ 0 if a caller forgets to populate them; `_endgame_elo_per_bucket` derives endgame_elo arithmetically while the new `_non_endgame_elo_per_bucket` reads the field directly, opening a 1-ELO rounding gap at monthly aggregation. None block ship by themselves, but together they erode the contract.

The PR helper itself, the per-side window accumulators, the gradient extraction, and the LLM prompt's THREE-summary-block emission are clean.

## Critical Issues

### CR-01: `query_endgame_performance_rows` NULL-rating filter changes WDL aggregation population

**File:** `app/repositories/endgame_repository.py:514-518`

**Issue:** Phase 87.6 added two WHERE clauses to the shared `game_cols` select:

```python
Game.white_rating.isnot(None),
Game.black_rating.isnot(None),
```

`game_cols` is reused for both the endgame and non-endgame row streams; both are consumed by `_get_endgame_performance_from_rows` (via `get_endgame_performance` at `endgame_service.py:2160`) for the endgame/non-endgame WDL comparison, and separately by the new ELO timeline path. The WDL path has nothing to do with PR / opponent ratings, but it now silently drops any game with a NULL `white_rating` or `black_rating` — a behavior change to the user-visible Endgame Performance card and every downstream stat fed by these rows. The repo's own docstring at line 487 still claims "Existing callers ignore columns past index 2," which conflates "extra columns" with "row population change" — those are different things.

There is no test that asserts either the prior or the new population (`tests/test_endgame_repository.py` seeds all games with default 1500/1500 ratings, so the NULL case never fires). Production data: ratings are usually present for chess.com/lichess, but historical PGN imports, certain unrated formats, and some early imports do leave NULLs. Affected users will see a quiet drop in counted games in `EndgamePerformanceResponse` without any version-bump signal.

**Fix:** Decouple the filter from the shared select. Either:

```python
# Option A: keep base game_cols unfiltered for ratings; add the rating-NULL guard
# only when building the rows that feed the PR computation. The orchestrator at
# get_endgame_elo_timeline can drop NULL-rating rows in Python after partitioning
# (cheap; rating-NULL is rare).

# Option B: split the repo into two queries — query_endgame_performance_rows (no
# rating filter, used by WDL and time pressure) and query_endgame_pr_rows (with
# rating filter, used by the timeline).

# Option C (minimum-change): add an explicit assertion + test that exercises a
# game with NULL ratings, verifies it still appears in WDL aggregation, and
# verifies it is excluded from the ELO timeline (with the exclusion happening at
# the service layer, not the repo).
```

Whatever option, also add a regression test that seeds a NULL-rating game and asserts both call sites' behavior.

### CR-02: User-facing "midpoint property" claims are unconditional but the math is not

**File:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx:249-251`, `app/prompts/endgame_insights.md:139,150,431`, `app/services/insights_llm.py:_PROMPT_VERSION changelog`

**Issue:** Multiple user-facing surfaces assert the midpoint property without the conditions that make it hold:

- Popover (`EndgameEloTimelineSection.tsx:249-251`): "Actual ELO sits inside the band by construction."
- Prompt (`endgame_insights.md:139`): "the midpoint property holds: `midpoint(elo, non_eg_elo) ≈ actual rating` by construction."
- Prompt (`endgame_insights.md:150`): "The midpoint property holds: `midpoint(endgame_elo, non_endgame_elo) ≈ actual_elo` by construction."
- Prompt (`endgame_insights.md:431`): "The midpoint property holds: `midpoint(endgame_elo, non_endgame_elo) ≈ actual_elo` by construction."
- `_PROMPT_VERSION` changelog line: "The midpoint property holds: `midpoint(endgame_elo, non_endgame_elo) ≈ actual_elo` by construction."

The schema (`schemas/endgames.py:559-561`) correctly qualifies the property — "approximates actual_elo within +-5 ELO for |score gap| <= 0.20 with equal opp pools." The user-facing surfaces drop both conditions.

Counter-example (realistic for FlawChess users): a 1800-rated player who plays endgames primarily against weaker arena pool (opp_avg = 1500, score 0.6 → PR_E ≈ 1572) and non-endgame games against equal-rated peers (opp_avg = 1800, score 0.5 → PR_N = 1800). Band = [1572, 1800], Actual ELO = 1800 — actual sits at the top edge, not "inside by construction." If the endgame opp pool dips lower, Actual ELO sits **above** the band entirely. The LLM will be instructed to narrate "your endgame is lifting your rating" when in fact the actual rating is above both PRs.

The sentence skeleton ("Your Endgame ELO sits at X, vs Y in non-endgame games — your endgame play is [lifting / holding back] your rating by ~Δ ELO") amplifies the issue because the lifting/holding distinction reduces to "compare PR_E and PR_N," not "compare PR_E and actual_elo," which is the conceptually-intended diagnostic.

**Fix:** Either (a) tighten the math by computing opp pools with explicit equal-weighting, then re-derive `endgame_elo - actual_elo` instead of `PR_E - PR_N` for the "lifting/holding" verdict, or (b) drop the unconditional "by construction" framing from user-facing copy and replace it with the conditional version from the schema docstring. The chart popover line "Actual ELO sits inside the band by construction" must be removed or reworded — it is provably false for any user with asymmetric opp pools.

### WR-02: `ScoreGapTimelinePoint.endgame_opp_rating_avg: float = 0.0` default fails silently

**File:** `app/schemas/endgames.py:357-358`

**Issue:** The two new fields default to `0.0`. The docstring at line 352 says "Default 0.0 for back-compat with older fixtures... a real timeline emission ALWAYS writes the rolling mean. Tests that exercise the PR math must supply non-zero values." If a future caller (production OR test) constructs a `ScoreGapTimelinePoint` without supplying these and then passes it to `_compute_endgame_elo_weekly_series`, `_performance_rating` is called with `opp_rating_avg=0.0` and emits PR ≈ 0 ELO (logistic delta + 0) instead of failing loudly. This is the same anti-pattern as silent-numeric-fallbacks discussed in the team's CR notes: prefer crashes to silently wrong analytics.

**Fix:** Either (a) make these fields required (no default) and update the one in-tree fixture that needs them, OR (b) default to `None` (typing `float | None = None`) and have `_performance_rating` raise `ValueError("opp_rating_avg required")` on None. Option (a) is cleanest because the docstring already declares the field is always written in real emissions.

### WR-03: `_endgame_elo_per_bucket` derives arithmetically, `_non_endgame_elo_per_bucket` reads directly — 1-ELO rounding skew possible at monthly aggregation

**File:** `app/services/insights_llm.py:1166-1184` vs `1297-1313`

**Issue:** Phase 87.6 adds `pt.endgame_elo` and `pt.non_endgame_elo` as direct fields on `TimePoint`. The new helper `_non_endgame_elo_per_bucket` reads `pt.non_endgame_elo` directly. The pre-existing `_endgame_elo_per_bucket` still derives `endgame_elo = pt.actual_elo + pt.value`, where `pt.value` is the (gap = endgame_elo - actual_elo) computed at the service layer and **not rounded** during weekly emission but **rounded into actual_elo** in the all_time monthly aggregation at `insights_service.py:1341-1345`. After independent rounding, `mean_elo + mean_value` can differ from `mean_e_elo` by up to ~1 ELO. The two summary blocks in the prompt (`endgame_elo` and `non_endgame_elo`) can therefore disagree with each other by ~1 ELO on the same combo at the all_time monthly bucket, and the LLM may notice and narrate inconsistently.

**Fix:** Replace the derived path with the direct path. `_endgame_elo_per_bucket` should read `pt.endgame_elo` and skip points where it is None (mirror the new helper). The two helpers can then be collapsed into a single `_pr_per_bucket(points, field_name)` parameterized helper, which also removes the ~75 lines of duplicate code between `_render_endgame_elo_summary_block` and `_render_non_endgame_elo_summary_block`.

### WR-04: `_render_non_endgame_elo_summary_block` duplicates `_render_endgame_elo_summary_block` body verbatim

**File:** `app/services/insights_llm.py:1316-1399`

**Issue:** The new helper is a near-verbatim copy of the existing endgame_elo helper. Bodies differ only in (a) the bucket extractor function and (b) the header literal string. Approximately 75 lines of duplicated logic spanning trend computation, std computation, sample-mismatch detection, and shift computation. Any future bug-fix to the trend/std/within-noise rules must be applied in two places — a violation of CLAUDE.md's "refactor bloated code on sight" guideline.

The `mean=<weighted_mean:+.0f> Elo` format string in both helpers prints a forced `+` sign on an absolute ELO value (e.g. `mean=+1500 Elo`), which is conceptually wrong for an absolute rating. Pre-existing in the endgame_elo helper; the new helper inherits the bug.

**Fix:** Extract a single parameterized helper:

```python
def _render_pr_summary_block(
    *,
    metric_label: str,  # "endgame_elo" | "non_endgame_elo"
    extractor: Callable[[list[TimePoint] | None], list[tuple[float, int]]],
    dim_key: str,
    all_time_finding: SubsectionFinding | None,
    ...
) -> list[str]: ...
```

Drop the forced `+` from `mean={weighted_mean:+.0f} Elo` for both helpers — absolute ELOs don't carry a sign.

### WR-06: `_opp_rating_for` else-branch silently accepts unexpected `user_color`

**File:** `app/services/endgame_service.py:870-884`

**Issue:** The helper has the structure:

```python
if user_color == "black":
    return float(white_rating) if white_rating is not None else None
return float(black_rating) if black_rating is not None else None
```

If `user_color` is anything other than `"black"` (including `None`, `"unknown"`, or a typo), the function falls through to the else branch and returns black_rating. `derive_user_result` constrains `user_color ∈ {"white", "black"}` in practice, but the helper doesn't enforce that and there's no `assert` or explicit check.

**Fix:** Make the contract explicit:

```python
if user_color == "black":
    return float(white_rating) if white_rating is not None else None
if user_color == "white":
    return float(black_rating) if black_rating is not None else None
return None  # unknown user_color — cannot derive opp_rating
```

### WR-07: `signedBandGradient` produces a phantom flip-stop at offset 100 for sign=-1 → sign=0 transitions

**File:** `frontend/src/lib/signedBandGradient.ts:79-93`

**Issue:** Walking the algorithm for rows `[-1, 0]`:
- currentColor = negative (first non-zero sign)
- Stop at offset 0, color=negative
- Segment 0→1: dA=-1, dB=0, colorA=negative, colorB=positive. colorA !== colorB → enter branch.
- t = -1 / (-1 - 0) = 1; offsetPct = (0+1)/1 * 100 = 100
- Push (100, negative), (100, positive). currentColor = positive.
- Final stop at offset 100, color=positive.

Result: 4 stops `[0/neg, 100/neg, 100/pos, 100/pos]`. The final stop at offset 100 switches to positive even though sign=0 should inherit from the prior sign (negative). The flip is a zero-width region so the visual is invisible, but the stop sequence is logically wrong — sign=0 at the right endpoint of the band should not introduce a color it never visibly occupies. Future readers debugging gradient artifacts will be misled.

This case is not tested in `signedBandGradient.test.ts`. The existing `sign-zero-leading with all negatives` test for `[0, 0, -1, -1]` only verifies "no green leakage," masking the symmetric bug here.

**Fix:** When `colorA !== colorB` and `dB === 0`, treat the segment as monochromatic in `currentColor`. Simplest patch:

```typescript
if (colorA !== colorB) {
  // A sign=0 row inherits the previous segment's color; only emit a flip
  // when both endpoints have a definite (non-zero) sign.
  if (dB === 0) continue;
  if (dA === 0) {
    // sign=0 → non-zero: flip at the start of this segment.
    const offsetPct = (i / denom) * 100;
    stops.push({ offset: offsetPct, color: currentColor });
    stops.push({ offset: offsetPct, color: colorB });
    currentColor = colorB;
    continue;
  }
  // Two non-zero signs of opposite sign: normal crossover at t = dA/(dA-dB).
  ...
}
```

Add a unit test for `[-1, 0]` and `[1, 0]` that asserts the stop sequence does not flip away from the dominant color at offset 100.

### IN-02: `_make_elo_combo` sets `non_endgame_elo = actual_elo` — unrealistic test fixture shape

### IN-04: `_performance_rating` accepts score outside [0, 1] without validation

**File:** `app/services/endgame_service.py:_performance_rating`

**Issue:** No bounds check on `score`. If a future caller passes `score = 1.5` (e.g. via a faulty score aggregator that returned percentage instead of proportion), `smoothed = (n*1.5 + 1) / (n+2)` could exceed 1.0, making `1 - smoothed` negative and triggering `math.log10` of a negative number — `ValueError` at runtime. The current sole call site at `endgame_service.py:1407-1413` passes `pt.endgame_score` / `pt.non_endgame_score`, which are upstream means of `{0.0, 0.5, 1.0}` per game so they are bounded in [0, 1]. A defensive assertion or clamp would prevent regressions if the input contract ever changes.

**Fix:** Add a single guard at the top of `_performance_rating`:

```python
if not (0.0 <= score <= 1.0):
    raise ValueError(f"score must be in [0, 1], got {score}")
if n < 0:
    raise ValueError(f"n must be non-negative, got {n}")
```
