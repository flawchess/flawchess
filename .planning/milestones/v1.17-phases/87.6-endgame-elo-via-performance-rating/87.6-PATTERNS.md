### 2. Trailing-window accumulator extension — `_compute_score_gap_timeline`

**Analog:** Same function body, lines 833-950 (the existing accumulator).

**Mirror:**
- Two existing trailing windows: `endgame_window: list[float]` (line 887) and `non_endgame_window: list[float]` (line 888). Pattern is **slice-truncation** (`endgame_window = endgame_window[-window:]` on line 903), not `collections.deque`. **Add two parallel `opp_rating` windows the same way** — `endgame_opp_window: list[float]` and `non_endgame_opp_window: list[float]`, slice-truncated identically.
- The event tuple at line 873 (`tuple[Any, Literal["endgame", "non_endgame"], float]`) grows a fourth slot for `opp_rating: float`. Derive it from the row inside `_score_for` (or a sibling helper) following the `query_endgame_elo_timeline_rows` pattern: row index for `white_rating` / `black_rating` is appended at the END of the SELECT tuple (see Pattern 4 below), and `opp_rating = white_rating if user_color == 'black' else black_rating` (CONTEXT §decisions).
- Per-week mean computation: emit `statistics.mean(endgame_opp_window)` and the non-endgame counterpart into `data_by_week[...]` next to `endgame_score` / `non_endgame_score` (lines 940-941). Match the `round(..., 4)` precision style for `score`; opponent ratings should be `round(mean, 1)` (one decimal — the downstream PR helper only uses them as a float anchor).
- The MIN_GAMES_FOR_TIMELINE drop filter at the bottom (lines 944-950) stays unchanged — same gate covers the new opp-rating sums (no separate threshold needed).

**Anti-pattern to avoid:**
- Do NOT compute opp-rating averages in a **second pass** over the event stream. Extend the single-pass accumulator.
- Do NOT introduce a parallel `_compute_opp_rating_timeline` producer. CONTEXT D-04 is explicit: the score-gap producer is the single threading point for both metric families.
- Do NOT swap `list[float]` for `collections.deque(maxlen=window)`. The codebase's established trailing-window idiom is slice-truncation; consistency wins over micro-optimisation.

## Shared Patterns

### Pure-function math helper convention
**Source:** `app/services/endgame_service.py` `_endgame_elo_from_score_gap` (lines 1314-1335), `_classify_endgame_bucket`, `derive_user_result` style.
**Apply to:** New `_performance_rating` helper.
**Style:** Module-private (`_` prefix), `int` or `float` return, no I/O, no session/state dependency, docstring carries the formula + a one-line pointer to the design note. Unit test class `TestX` in `tests/test_endgame_service.py`.

### Trailing-window slice-truncation idiom
**Source:** `_compute_score_gap_timeline` lines 887-906 (existing `endgame_window` / `non_endgame_window`).
**Apply to:** New opp-rating windows in same function.
**Style:** `window: list[float] = []` + `window.append(x); window = window[-WINDOW_SIZE:]`. **Not** `collections.deque(maxlen=WINDOW_SIZE)`. Project convention is stable across `_compute_endgame_elo_weekly_series`, the type-timeline helpers, and Clock Pressure timelines.

### Schema field-addition with backward-compat default
**Source:** `ScoreGapTimelinePoint.per_week_endgame_games: int = 0` (line 345) — Phase 87.5 D-06 follow-up.
**Apply to:** New opp-rating-avg fields on `ScoreGapTimelinePoint`; new `non_endgame_elo` on `EndgameEloTimelinePoint`.
**Style:** Default to a neutral sentinel (`0.0` for float means; `0` for ints) when older fixtures construct without the field, plus an inline comment naming the phase that added it.

### Frontend type mirror
**Source:** `frontend/src/types/endgames.ts` (manually maintained mirror of `app/schemas/endgames.py`).
**Apply to:** All Pydantic schema changes in Pattern 4.
**Style:** Hand-update lines after the Python schema lands. Mirror Phase-tagging in field comments. No codegen — knip catches accidental dead exports.

### Theme constant for chart colors
**Source:** `frontend/src/lib/theme.ts:171-172` (`SCORE_TIMELINE_FILL_ABOVE` / `SCORE_TIMELINE_FILL_BELOW`).
**Apply to:** New `signedBandGradient` helper — accept colors as a `{ positive, negative }` parameter so callers pass theme constants. Don't import theme inside the helper (per CLAUDE.md Frontend §"Theme constants in theme.ts" — colors with semantic meaning live in `theme.ts`, helpers stay color-agnostic).
