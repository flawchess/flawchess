### WARN-02 — Symmetric-s=0.7 PR assertion is weak
- **Severity:** WARNING
- **Plan:** 01 Task 1
- **Dimension:** test_coverage_adequacy
- **CONTEXT §Tests** explicitly enumerates: `symmetric s=0.7 / s=0.3 ⇒ PR symmetric around R_opp`.
- **Plan 01 Task 1** specifies `test_symmetric_scores_around_half` asserting `abs((pr_07 - 1500) + (pr_03 - 1500)) <= 1`. This is only an additive-symmetry check (sum of deltas ≈ 0). It does NOT verify the magnitude of either delta. A bug that returned `pr_07 == pr_03 == 1500` would pass this test.
- **Remediation:** Add a second assertion: `abs((pr_07 - 1500) - (1500 - pr_03)) <= 1` AND `pr_07 > 1500` AND `pr_03 < 1500`. Or split into two assertions: `pr_07 - 1500 > 100` (real magnitude) AND `pr_07 - 1500 == -(pr_03 - 1500) ± 1` (mirror).

### WARN-03 — Plan 02 widens `chartData` row type to include tuple values but `barChartData`'s `&` intersection still constrains it
- **Severity:** WARNING
- **Plan:** 02 Task 2 step (d)
- **Dimension:** task_completeness / type-safety
- **Plan 02 says** to widen `chartData` row type to `Record<string, string | number | [number, number] | undefined>` AND to update `barChartData`'s explicit return type with the same widening. Currently `barChartData` uses `Array<Record<string, string | number | undefined> & { per_week_total_visible: number }>`. The widened version must still satisfy the `& { per_week_total_visible: number }` intersection — `per_week_total_visible` is `number`, which narrows in. OK in principle.
- **Risk:** Recharts' `data` prop is loosely typed (`object[]`), so tsc won't catch a runtime case where a row's `_band` is read into Recharts' renderer for a Line series (which expects `number | string | undefined`, not a tuple). The Area's `dataKey={`${combo_key}_band`}` correctly consumes the tuple, but a future grep-and-rename could accidentally point a Line at the `_band` key.
- **Remediation:** Plan 02 should add a verification step: `grep -n "_band" frontend/src/components/charts/EndgameEloTimelineSection.tsx` should show the key referenced exactly once as the `<Area>`'s `dataKey`. Add this grep to the Task 2 `<verify>` block.
