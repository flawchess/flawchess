### Endgame ELO mapping (LOCKED — FIDE Performance Rating, no free constants)

For each (platform, TC) combo, at each weekly emission point with a trailing 100-game window:

```
endgame_elo     = R_opp_avg_E + 400 · log10(s_E* / (1 − s_E*))
non_endgame_elo = R_opp_avg_N + 400 · log10(s_N* / (1 − s_N*))
```

- `s_E*` and `s_N*` are **Laplace-smoothed** per-side scores: `s* = (wins + 0.5·draws + 1) / (n + 2)` where `n` is per-side game count in the trailing window. Bounds PR delta near ±900 ELO at n=100; no clamping at output.
- `R_opp_avg_E` and `R_opp_avg_N` are per-side average opponent ratings over the same trailing window.
- The `400` is fixed by Elo's logistic-skill assumption. Not a calibration knob.
- Both PRs are rounded to int at the emission boundary. Identical rounding contract to the existing `endgame_elo` int field.

At score = 0.5, PR equals `R_opp_avg` exactly. By the midpoint property documented in the design note (actual_elo ≈ midpoint(PR_E, PR_N) in the |gap| < 0.23 regime that covers §3.1.6 p05–p95), the Actual ELO line is bracketed by Endgame ELO and Non-Endgame ELO by construction. **No anchoring math** — plot all three lines on the same axis.

## Specific Ideas

### PR helper suggested signature

```python
def _performance_rating(score: float, n: int, opp_rating_avg: float) -> int:
    """FIDE Performance Rating with Laplace smoothing.

    PR = R_opp_avg + 400 · log10(s* / (1 − s*))
    s* = (n · score + 1) / (n + 2)   # Laplace-smoothed; bounds PR delta at ~±900 for n=100

    The `400` is fixed by Elo's logistic-skill assumption — not a knob.
    See `.planning/notes/endgame-elo-pr-direct-rebuild.md` for derivation.
    """
    smoothed = (n * score + 1) / (n + 2)
    return int(round(opp_rating_avg + 400 * math.log10(smoothed / (1 - smoothed))))
```

The `n * score + 1` form is equivalent to `(wins + 0.5 * draws + 1) / (n + 2)` when `score = (wins + 0.5*draws) / n`. Planner picks whichever form reads clearer to them.

### Signed-band-gradient helper suggested shape

```typescript
export type GradientStop = { offset: number; color: string };

export type SignedBandRow = {
  x: number;            // domain x value, used to compute offset percentage
  sign: 1 | -1 | 0;     // direction at this row (0 inherits previous-segment color)
};

export function signedBandGradient(
  rows: SignedBandRow[],
  xDomain: [number, number],   // [min, max] for offset% calculation
  colors: { positive: string; negative: string },
): GradientStop[] { ... }
```

Move the body from `EndgameScoreOverTimeChart.tsx:101-165` verbatim, parameterising the two colors so the same helper serves both charts (the existing chart uses WDL-derived green/red constants; the new chart can pass the same constants for consistency).

### Midpoint property tolerance

Synthetic-data tests: with score_gap ∈ {-0.20, -0.10, 0.0, +0.10, +0.20} and equal-mean opp_rating_avg on both sides, assert `|midpoint(PR_E, PR_N) − actual_elo| ≤ 5 ELO`. Beyond |gap| > 0.25 the linearization breaks; do not assert there.
