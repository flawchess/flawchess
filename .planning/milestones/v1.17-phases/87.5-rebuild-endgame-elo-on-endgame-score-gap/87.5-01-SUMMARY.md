# Phase 87.5 Plan 01: Backend Rewire (Additive K Mapping + Schema Rename) Summary

**One-liner:** Rebuild Endgame ELO on the Endgame Score Gap series via the additive `actual_elo + K · eg_score_gap` formula; delete Phase 87.4 Conv-ΔES sigmoid helpers wholesale; rename the schema field end-to-end.

## What Was Built

Phase 87.5 Plan 01 lands the backend half of the Endgame ELO rebuild. The Phase 87.4 conv-ΔES → affine-recenter → Phase 57 multiplicative-formula pipeline is replaced by a single additive mapping `endgame_elo = round(actual_elo + K · eg_score_gap)` fed directly from the per-week windowed Endgame Score Gap series already produced by `_compute_score_gap_timeline`. The per-combo invariant — Endgame ELO for combo X is derived only from combo X's own score-gap series — is enforced by partitioning endgame / non-endgame rows by `(platform, time_control_bucket)` inside the orchestrator and calling `_compute_score_gap_timeline` once per combo.

## Decisions Made

1. **K = 450, locked.** Derived from the §3.1.6 pooled benchmark Lichess percentile table (n=1,632) — the same table the researcher cited as the back-of-envelope basis. At K=450: p05 -0.227 → -102 ELO, p25 -0.104 → -47 ELO, p50 -0.014 → -6 ELO, p75 +0.073 → +33 ELO, p95 +0.202 → +91 ELO. Typical points land in the targeted 30-60 ELO band; extremes land near ±100 ELO. No further eyeball needed; the §3.1.6 percentiles are themselves derived from real benchmark users. `test_K_constant_locked` guards future drift.

2. **Bucket-row subquery deleted from the repo query (RECOMMENDED → MANDATED).** The plan's "RECOMMENDED" optimization was promoted to mandatory in the planner-revised text. After `_windowed_conv_delta_es` was deleted, the `eval_cp / eval_mate / result` columns the bucket subquery projected had zero consumers. Deleting them keeps both the SQL query and the Python signature lean.

3. **Row-shape extension for `query_endgame_performance_rows`.** Rather than fetch the endgame / non-endgame partition twice (once for the WDL summary, once per combo for score-gap derivation), the existing query was extended with two trailing columns (`platform`, `time_control_bucket`). The orchestrator reuses the same fetched rows. `_build_wdl_summary` was updated to index access; all other consumers ignore the new columns by reading only indices 0/1/2.

4. **Use `pt.score_difference` (pre-rounded) instead of recomputing `endgame_score - non_endgame_score`.** Both the Pydantic field `score_difference` and the per-side `endgame_score` / `non_endgame_score` are rounded to 4 decimals by `_compute_score_gap_timeline`. Recomputing the difference introduces sub-ulp noise that flips the rounding boundary at half-integer K·gap (e.g. K=450, gap=0.05 → 22.500000000000004 rounds up to 23 instead of banker-rounding to 22). Using `pt.score_difference` directly aligns the producer with the canonical pre-rounded value. Test asserts the same identity.

5. **Helper renames deferred to Plan 03.** Per the plan's atomic-edit-scope rule, helper function names (`_findings_conversion_elo_timeline`, `_conversion_elo_per_bucket`, `_render_conversion_elo_summary_block`), the rendered `[summary conversion_elo` header literal, and the prompt prose in `app/prompts/endgame_insights.md` are Plan 03's territory. Plan 01 only flipped what ty required to stay green.
