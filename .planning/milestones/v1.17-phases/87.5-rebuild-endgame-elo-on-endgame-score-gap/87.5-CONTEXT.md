## Phase Boundary

Replace the Phase 87.4 Conversion ELO Timeline (driven by Conv ΔES + affine recenter + Phase 57 multiplicative formula) with an **Endgame ELO Timeline** driven by the existing Endgame Score Gap series and a single additive K mapping. Both backend (service + repo + schema), frontend (component + theming/copy), and LLM payload (MetricId + SubsectionId + prompt prose + glossary + prompt version) are in scope.

</domain>

<decisions>
## Implementation Decisions

### Input metric (LOCKED)
- The Endgame ELO Timeline reads its input from the **per-week windowed Endgame Score Gap** series already produced by `_compute_score_gap_timeline` in `app/services/endgame_service.py` (currently emitted as `score_gap_material.timeline[*].score_difference`, `endgame_score`, `non_endgame_score`).
- Per-week input value is `eg_score_gap = endgame_score − non_endgame_score`, where each side is the trailing-100-game (`SCORE_GAP_TIMELINE_WINDOW`) rolling mean of outcome scores `{win: 1.0, draw: 0.5, loss: 0.0}`. Both terms are realized outcomes — no sigmoid baseline, no eval reference, no per-bucket subset.
- Sample-quality / sparse-window handling, ≥`MIN_GAMES_FOR_TIMELINE`-per-side floor, and ISO-week Monday bucketing are inherited as-is from the existing producer. No new sample-quality logic.

### ELO mapping (LOCKED — additive, single global K)
- Formula: `endgame_elo = round(actual_elo + K · eg_score_gap)`.
- At `eg_score_gap = 0`, `endgame_elo == actual_elo` exactly. Neutral is literal zero, not a benchmark-derived constant.
- `K` is a single global float constant.
- Starting candidate `K ≈ 450` from §3.1.6 percentile mapping (p95 +0.20 ≈ +91 ELO; p05 −0.23 ≈ −102 ELO; p50 −0.01 ≈ −6 ELO). Finalized value chosen during planning per the calibration procedure in `.planning/todos/pending/calibrate-endgame-elo-k.md`.
- Rounding to int, plus the sensible floor/ceiling already used by the existing endgame_elo_timeline contract (positive int, do not output negative or absurd-high values).

### Phase 87.4 helpers / constants to delete (LOCKED — delete, do not retune)
From `app/services/endgame_service.py`:
- `PIVOT: float = -0.0474`
- `ALPHA: float = 2.025`
- `CALIBRATION_VERSION: str = "conv_delta_v1_260516"`
- `_affine_recenter_conv_delta(...)` helper
- `_windowed_conv_delta_es(...)` bucket-row aggregator
- `_conversion_elo_from_skill(...)` (renamed from `_endgame_elo_from_skill` in 87.4) — the Phase 57 `400·log10(s/(1−s))` formula is retired entirely; replaced by the additive K mapping.
- `_compute_conversion_elo_weekly_series` (or whatever the 87.4 weekly-series producer was named) is replaced by a new `_compute_endgame_elo_weekly_series` (or equivalent) that consumes the score-gap series directly.

### Naming + chart relocation (LOCKED)
- Chart name: **"Endgame ELO Timeline"** everywhere (chart title, popover, LLM glossary, CHANGELOG, Vitest assertions).
- The renamed chart moves *out* of the "Endgame Metrics and ELO" section and *into* "Endgame Overall Performance", positioned **directly below** the existing "Endgame Score Gap over Time" chart in `frontend/src/pages/Endgames.tsx`.
- File rename: `frontend/src/components/charts/ConversionEloTimelineSection.tsx` → `EndgameEloTimelineSection.tsx`. Props/types updated. Knip clean.
- Backend schema rename: `EndgameOverviewResponse.conversion_elo_timeline` → `endgame_elo_timeline`. Frontend `frontend/src/types/endgames.ts` updated to match.
- Repo-wide grep `Conversion ELO` (case-insensitive) returns only CHANGELOG history + the supersession comment block in code after this phase.

### LLM payload (LOCKED)
- `MetricId` literal: `conversion_elo_gap` → `endgame_elo_gap`.
- `SubsectionId` literal: `conversion_elo_timeline` → `endgame_elo_timeline`.
- Section grouping: subsection moves from `metrics_elo` to `overall`.
- Prompt prose rewritten to reference Endgame Score Gap as the input series; "lifts up / holds back actual rating" headline metaphor restored (Phase 87.4 had traded this away for "what your ELO would be if everyone played the way you do when up material").
- `_PROMPT_VERSION` bumps from `endgame_v32` → `endgame_v33`.
- The Phase 87.4 Section-3 LLM surface scaffolding (`[summary endgame_elo | …]` literal, `### Subsection:` header, glossary entry) is kept structurally; only the metric/prose/section grouping changes.

### Tests (LOCKED — required coverage)
- Pytest unit tests for the additive K mapping on synthetic windowed score-gap series: zero gap ⇒ ELO equals actual; ±p95 gap ⇒ ELO equals actual ± `round(K·p95)`.
- Pytest assertion that the new `_endgame_elo_from_score_gap` rounds correctly and clamps at any floor/ceiling.
- Regression test (Python or grep-based CI) asserting no rendered surface contains "Conversion ELO" outside the supersession comment + CHANGELOG.

### Conv display-centering (UNCHANGED)

<specifics>
## Specific Ideas

### K calibration eyeball criteria
From the calibration todo: pull 5–10 representative users spanning rating buckets from dev or benchmark DB; compute the proposed series; check that (a) typical points sit within ~30–60 ELO of actual_elo, (b) visible score-gap trends produce visibly-believable ELO trends, (c) extremes read as "this user is 100 ELO over/underperforming their rating on endgames" — not 300, not 50. Adjust K to whatever lands. Document the chosen value with one line of rationale next to the constant.

### Suggested rounding contract
`round(actual_elo + K · eg_score_gap)` → `int`. Phase 57 timeline contract already returns positive int. If a sensible floor/ceiling is needed (e.g. clamp at `[100, 3500]` to avoid pathological renderings), match the existing endgame_elo_timeline behaviour exactly — do not introduce a new clamp policy.

### Suggested file structure
Either keep everything in `app/services/endgame_service.py` (alongside the deleted Phase 87.4 helpers — drop-in replacement) OR extract into `app/services/endgame_elo.py` with a 5-line module docstring referencing this phase. Planner picks; both are acceptable.

### LLM prompt prose pattern
"Your Endgame ELO is your actual rating plus a small offset proportional to your Endgame Score Gap. Positive Endgame Score Gap means your endgame games score better than your non-endgame games — your endgame is *lifting* your rating. Negative means your endgame is *holding back* your rating. The constant K is calibrated so a typical Endgame Score Gap (≈ ±10pp at p25/p75) maps to ±33–47 ELO; extreme gaps (±p05/p95 ≈ ±23pp) map to ±100 ELO."

The exact phrasing is the planner's discretion; the requirement is that the "lifts up / holds back" metaphor is restored.

</specifics>

<deferred>
## Deferred Ideas

- **Per-(platform, TC) K tuning** — flagged as a future option only if UAT reveals systematic per-TC drift after this phase ships. Benchmark §3.1.6 collapse verdict (ELO d=0.17 collapse, TC max d=0.34 review) justifies a single global K for v1.
- **Section-header rename "Endgame Metrics and ELO" → "Endgame Metrics"** — suggested in the design note but optional; planner decides whether to fold in or defer to Phase 89 polish.
- **Drop the ELO axis, ship Score Gap only** — considered and rejected because actual ELO + Endgame ELO are load-bearing for the LLM-narrated section ("your endgame is lifting your rating by 35 ELO" is the story users understand). Keep ELO units, just feed the right metric.
- **Smooth peer-conditional PIVOT (still using Conv ΔES)** — rejected; still inherits the sigmoid bias and quietly redefines the ELO scale.
- **Self-baseline at row level (Conv ΔES − user's own non-endgame Conv ΔES per row)** — rejected; equivalent in spirit to Endgame Score Gap but reinvents the aggregation and gets noisy at low sample counts.
- **Conv / Recov gauge re-centering** — rejected as out of scope; §3.2.2 explicitly shows the bands *should* be off-center and re-centering would lie about what's typical.

</deferred>
