## Summary

Phase 87.5 is a surgical refactor that replaces one ELO-derivation pipeline (Phase 87.4's conv-ΔES-driven affine recenter + Phase 57 multiplicative formula) with a simpler additive formula (`endgame_elo = round(actual_elo + K * eg_score_gap)`) fed directly from the Endgame Score Gap series already computed by `_compute_score_gap_timeline`. The refactor spans three layers: backend (service arithmetic + helper deletion + schema rename), frontend (component file rename + section repositioning), and LLM payload (MetricId/SubsectionId rename + `_SECTION_LAYOUT` entry move + prompt version bump).

The empirical basis: §3.1.6 of `reports/benchmarks-latest.md` shows the Endgame Score Gap metric has ELO Cohen's d = 0.17 (essentially flat across rating buckets), eliminating the systematic strong-player bias that Phase 87.4's Conv ΔES metric (d = 1.62) introduced. The K = 450 starting candidate maps p95 gap (+0.202) to approximately +91 ELO delta, and p05 gap (-0.227) to approximately -102 ELO delta — both in the "believable endgame over/underperformance" range.

The "Endgame Metrics and ELO" section in the frontend retains only the three Conv / Parity / Recov gauges after the chart moves. The section-header rename ("→ Endgame Metrics") is discretionary for the planner. The repo-wide grep `Conversion ELO` must return only CHANGELOG history + one supersession comment block after this phase lands.

**Primary recommendation:** Plan 3 waves (backend rewire + deletion + schema rename, then frontend chart rename + repositioning + tests, then LLM payload rename + prompt rewrite + CHANGELOG). Backend is a prerequisite for frontend and LLM (schema field rename propagates to TypeScript types); frontend and LLM payload can run in parallel after backend lands.

## 2. The Existing Endgame Score Gap Producer

### `_compute_score_gap_timeline`

- **File:** `app/services/endgame_service.py`
- **Lines:** 826–933
- **Signature:** `(endgame_rows, non_endgame_rows, window, cutoff_str=None) -> list[ScoreGapTimelinePoint]`
- **Called from:** `get_endgame_overview` at line 2640, which passes `SCORE_GAP_TIMELINE_WINDOW` as window.
- **`SCORE_GAP_TIMELINE_WINDOW = 100`** at line 336.

**Per-week record shape** (the `ScoreGapTimelinePoint` dict built at lines 917–925):

| Field | Type | Description |
|-------|------|-------------|
| `date` | `str` (YYYY-MM-DD) | ISO Monday of the week |
| `score_difference` | `float` | `endgame_score - non_endgame_score`, 4 decimals |
| `endgame_game_count` | `int` | games in trailing endgame window |
| `non_endgame_game_count` | `int` | games in trailing non-endgame window |
| `per_week_total_games` | `int` | endgame + non-endgame games this ISO week |
| `endgame_score` | `float` | absolute rolling-window mean score for endgame side |
| `non_endgame_score` | `float` | absolute rolling-window mean score for non-endgame side |

**Phase 87.5 input:** `eg_score_gap = endgame_score - non_endgame_score` (equivalently `score_difference`) from each point.

**Sparse-window handling:**
- Emission requires `endgame_game_count >= MIN_GAMES_FOR_TIMELINE` AND `non_endgame_game_count >= MIN_GAMES_FOR_TIMELINE` (line 930–931). `MIN_GAMES_FOR_TIMELINE` is imported from `app/services/openings_service` (line 71). No new sample-quality logic is needed.
- `cutoff_str` pre-fill pattern: `get_endgame_overview` calls `_compute_score_gap_timeline` with `endgame_rows_all` (all-time unfiltered rows) so rolling windows are pre-filled, but passes `cutoff_str` so output points before the cutoff are dropped. The call is at line 2640.

### `ScoreGapTimelinePoint` schema

- **File:** `app/schemas/endgames.py`
- **Lines:** 308–344
- All 7 fields documented in Section 2 above. No changes needed to this schema.

### Relationship to `score_gap_material`

The `_compute_score_gap_material` function at line 1099 receives the `timeline: list[ScoreGapTimelinePoint]` as its `timeline` parameter and passes it through to `ScoreGapMaterialResponse.timeline`. The Endgame ELO timeline phase reads from this same data structure — `score_gap_material.timeline` on the `EndgameOverviewResponse` is already the source.

**Phase 87.5 implication:** The new `_compute_endgame_elo_weekly_series` (or however the planner names it) will receive the `ScoreGapTimelinePoint` list (or the raw endgame/non-endgame rows directly) and derive `eg_score_gap` from each point's `endgame_score` and `non_endgame_score`. Two implementation options:

1. Consume `score_gap_timeline: list[ScoreGapTimelinePoint]` directly from the orchestrator — zero re-aggregation, single pass.
2. Keep the existing `get_endgame_elo_timeline` orchestrator that fetches `query_endgame_elo_timeline_rows` for the Actual ELO line, then join the two series on ISO week — preserves the asof-join accuracy for `actual_elo`.

Option 2 is almost certainly required: the Actual ELO line still needs the per-combo asof-join from `query_endgame_elo_timeline_rows`, and the score-gap series must be per-combo-filtered too (the existing `score_gap_material.timeline` is cross-combo, not per-(platform, TC)). The existing `get_endgame_elo_timeline` orchestrator already handles per-combo partitioning of the all_rows for the Actual ELO line. Phase 87.5 must also produce a per-combo score-gap input, meaning either a re-query or a per-combo partition of the endgame/non-endgame rows that `_compute_score_gap_timeline` already accepts. The cleanest path is to derive the per-combo score-gap series inside `get_endgame_elo_timeline` by calling `_compute_score_gap_timeline` per combo using the existing per-combo row partitions.

The pooled (excl. sparse) n=1,632, mean = -0.0134. eg_mean = 0.5105, non_eg_mean = 0.5239. These come from the benchmark DB sample of Lichess users.

### Calibration procedure

From `.planning/todos/pending/calibrate-endgame-elo-k.md`:
1. Pull 5–10 real users from dev/prod DB spanning rating buckets.
2. For each user, compute the proposed `endgame_elo` series using the existing `score_gap_material.timeline` data (or re-run `_compute_score_gap_timeline` against their rows).
3. Eyeball criteria: typical points within ~30–60 ELO of `actual_elo`; extremes around ±100 ELO; no 300–500 ELO gaps.

**Recommended calibration mechanism:** The planner should include a Wave 0 step or a standalone one-off script (not a pytest test) to compute the proposed series for a sample of dev/benchmark users and inspect the output. The dev DB is accessible via the `flawchess-db` MCP; the benchmark DB via `flawchess-benchmark-db` (requires `bin/benchmark_db.sh start`).

**Sample SQL to derive eg_score_gap for a user:**
The `score_gap_material.timeline` is already computed by the service; to calibrate K without running the full service, the planner can query `game_positions` joined to `games` and replicate the per-week rolling-window logic, or more practically: call the `/api/endgames/overview` endpoint for a sample of users and inspect `score_gap_material.timeline[*].score_difference` × K against `conversion_elo_timeline[*].actual_elo`.

**Where K should live:** Alongside the new `_endgame_elo_from_score_gap` helper, either in `app/services/endgame_service.py` (alongside the deleted Phase 87.4 constants) or in a small `app/services/endgame_elo.py`. The constant should carry a comment block: `K = <value>  # §3.1.6 p95 ≈ +0.20 → +<K*0.20> ELO delta; typical ±10pp → ±<K*0.10> ELO`.

## 12. Open Questions for the Planner

**Q1: Should `_compute_endgame_elo_weekly_series` derive its score-gap input from the existing `score_gap_timeline` parameter, or from re-calling `_compute_score_gap_timeline` per combo?**

The existing `get_endgame_elo_timeline` orchestrator has per-combo row partitions (`all_by_combo`) for the Actual ELO asof-join. To derive a per-combo score-gap series, it also needs per-combo endgame/non-endgame rows. Two options:
- Option A: Pass the endgame/non-endgame row partitions (from the existing `query_endgame_elo_timeline_rows` `all_rows` + a separate endgame-filtered subset) and call `_compute_score_gap_timeline` per combo inside `get_endgame_elo_timeline`.
- Option B: The `get_endgame_elo_timeline` remains its own separate query (`query_endgame_elo_timeline_rows`) which currently returns `(bucket_rows, all_rows)`. Phase 87.5 could add endgame/non-endgame partitions to the same query, or run a second query for the score-gap rows. Option A seems cleanest since `_compute_score_gap_timeline` already accepts generic row sequences.

**Recommendation:** Option A — reuse `_compute_score_gap_timeline` per combo inside `get_endgame_elo_timeline`. The per-combo endgame rows and all_rows are already partitioned in `all_by_combo`; classify endgame games from there (or partition `bucket_rows_all` separately). The weekly helper then receives both the score-gap series and the asof-dates/ratings and does the point-by-point lookup and K multiplication.

**Q2: Does `query_endgame_elo_timeline_rows` need to be simplified?**

After Phase 87.5, `bucket_rows` (the 9-column rows with `eval_cp`, `eval_mate`, `result`) are needed only if the new producer still uses them. If the new weekly producer derives `eg_score_gap` from the score-gap timeline points (not from bucket_rows), then `bucket_rows` are no longer needed. Simplifying the query to return only `all_rows` would reduce DB work. However, this is a secondary optimization — the planner may defer it.

**Q3: Should the section-header rename ("Endgame Metrics and ELO" → "Endgame Metrics") be folded into this phase?**

CONTEXT.md marks this as Claude's Discretion. The `h2` at `Endgames.tsx` line 490 reads `"Endgame Metrics and ELO"`. After the ELO chart moves out, the header is inaccurate. The rename is a 1-line change in `Endgames.tsx` and a corresponding update in the `SectionInsightSlot sectionId="metrics_elo"` comment (the `sectionId` value itself doesn't need to change since it maps to `SectionId` in the LLM schema). **Recommendation: fold it in** — the one-line rename is trivial, the mismatch post-phase is obvious, and deferral to Phase 89 creates a visible inconsistency.

**Q4: Where should the K constant live?**

CONTEXT.md says Claude's Discretion. Two options:
- `app/services/endgame_service.py` alongside the other endgame_elo constants (`ENDGAME_ELO_TIMELINE_WINDOW`, `_ENDGAME_ELO_COMBO_ORDER`) — consistent with existing structure, no new file.
- `app/services/endgame_elo.py` — cleaner separation, but creates a file with ~30 lines of content.

**Recommendation:** Keep in `app/services/endgame_service.py`. The deleted Phase 87.4 constants lived there; the replacement K constant and `_endgame_elo_from_score_gap` function should drop in at the same location (~line 1270). A separate module is only worthwhile if the function grows or accumulates siblings.

**Q5: Should `EndgameEloTimelinePoint.endgame_elo` or `EndgameEloTimelinePoint.conversion_elo` rename trigger a DB migration?**

No — `EndgameEloTimelinePoint` is a Pydantic response schema (wire format), not an ORM model. The rename changes the JSON response field name. Frontend must be deployed atomically with the backend; the standard CI/CD pipeline handles this.

**Q6: Should `_series_for_endgame_elo_combo` in `insights_service.py` be renamed?**

That function (imported by `tests/services/test_insights_service_series.py` at line 32) computes the gap series for the LLM. Its name already uses `endgame_elo` (not `conversion_elo`). Check: the function at line 861 is called inside `_findings_conversion_elo_timeline`. Its name may already be `_series_for_endgame_elo_combo` (from the Phase 87.4 partial rename).
