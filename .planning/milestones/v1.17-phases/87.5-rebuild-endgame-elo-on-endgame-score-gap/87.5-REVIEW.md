# Phase 87.5: Rebuild Endgame ELO on Endgame Score Gap — Code Review

## Summary

The refactor lands cleanly at the surface: the additive `endgame_elo = round(actual_elo + K · eg_score_gap)` mapping is correct, the helper signature is clean, Phase 87.4's `PIVOT`/`ALPHA`/`CALIBRATION_VERSION`/`_affine_recenter_conv_delta`/`_windowed_conv_delta_es`/`_conversion_elo_from_skill` are gone, `K = 450.0` is documented with provenance, `int(round(...))` rounds correctly at the gap=0 invariant, the per-combo isolation is enforced inside the orchestrator, schemas + LLM payload + frontend types are renamed end-to-end, the supersession comment block reads well, and ruff/ty/pytest/knip are clean. However the rewire silently redefined the semantics of `per_week_endgame_games` from "this ISO week" to "trailing 100-game window count" — the schema docstring, frontend type comment, and chart tooltip copy ("Games this week: N") all still assert the original semantics, so the volume bars and tooltip now mislead users. Two smaller-scope regressions in the chart's parent-gating cause its empty-state UI to become unreachable. Outside those issues the work is in good shape.

## Critical Findings

### CR-01: `per_week_endgame_games` semantics silently flipped — volume bars + tooltip now show wrong numbers

**File:** `app/services/endgame_service.py:1402-1403` (with knock-on at `app/schemas/endgames.py:558-561`, `frontend/src/types/endgames.ts:295`, `frontend/src/components/charts/EndgameEloTimelineSection.tsx:383, 416`)

**Issue:** The new `_compute_endgame_elo_weekly_series` emits `per_week_endgame_games=pt.endgame_game_count`, where `pt.endgame_game_count` is the **trailing-window** count (≤ 100, capped by `_compute_score_gap_timeline`'s window) — NOT the per-ISO-week count. The pre-Phase-87.5 producer maintained a separate `per_week_count: dict[(iso_year, iso_week), int]` accumulator and emitted that distinct value. The new implementation lost the per-ISO-week accumulator entirely, so both `endgame_games_in_window` and `per_week_endgame_games` now carry the same trailing-window number on every point.

Downstream consequences:
1. `EndgameEloTimelinePoint` schema docstring (`app/schemas/endgames.py:558-561`) still says "count of endgame games for THIS specific ISO week (NOT the trailing window). Frontend uses this for the muted volume-bar series so users can see at a glance whether a weekly point is well-supported (50+ games this week) or marginal (just over the 10-game floor)." — now factually wrong.
2. Frontend `EndgameEloTimelinePoint` TSDoc comment (`frontend/src/types/endgames.ts:295`) says the same — also wrong.
3. The volume bars in `EndgameEloTimelineSection.tsx` aggregate `per_week_endgame_games` across visible combos (line 161-169) and feed `barChartData.per_week_total_visible`. After the rewire every weekly bar plateaus at roughly the trailing window size × number of visible combos, instead of showing actual weekly activity. The bars become a near-constant ribbon, defeating the documented "well-supported vs marginal" affordance.
4. The chart tooltip (`EndgameEloTimelineSection.tsx:383`) renders the same value as `Games this week: {perWeekTotal}` — the user is told they played ~100 games this week regardless of whether they played 0 or 200.
5. LLM payload: `_series_for_endgame_elo_combo` (`app/services/insights_service.py:1284`) uses `p.per_week_endgame_games` as the `n` for monthly aggregation weighting. With `n` ≈ constant ≈ window size, the weighted monthly mean degenerates to a simple mean — losing the volume-weighted aggregation contract.

The docstring of `_compute_endgame_elo_weekly_series` itself (lines 1348-1352) acknowledges the carry-over: "the score-gap producer does not split per-week-this-bucket vs trailing-window counts, so this carries the trailing-window count too; the volume-bar UI still reads as 'supported window size'." — but the schema, frontend types, and tooltip copy were not updated to match this redefinition. Either the producer needs to preserve the per-ISO-week count (matching the previous Phase 57.1 D-06 design) or every other surface needs to be updated in lockstep. Shipping the mismatch is a user-visible UX bug.

**Fix:** Restore per-ISO-week counting by augmenting `_compute_score_gap_timeline` (or accumulating in `_compute_endgame_elo_weekly_series` from the raw endgame rows the orchestrator already partitions per combo) so `ScoreGapTimelinePoint` carries `endgame_per_week_count` (and ideally `non_endgame_per_week_count`) distinct from `endgame_game_count` / `non_endgame_game_count`. Then emit:

```python
per_week_endgame_games=pt.endgame_per_week_count,  # NOT pt.endgame_game_count
```

If preserving the prior contract is too invasive for this phase, the alternative is to update the schema docstring, the frontend TSDoc, and the tooltip copy in lockstep ("Games in trailing 100-game window") and rethink the volume-bars UI (they no longer carry weekly information). Either path closes the mismatch; shipping it open is the bug.

## Warning Findings

### WR-01: Endgames page wrapper now hides the chart's empty/error/loading UI

**File:** `frontend/src/pages/Endgames.tsx:490`

**Issue:** The relocated chart is gated on `eloTimelineData && eloTimelineData.combos.length > 0`. Before Phase 87.5 the wrapper was gated only on `scoreGapData && perfData`, so when overview had loaded but `combos` was empty the component's own "Not enough endgame games yet for a timeline. Import more games or loosen the recency filter." empty state (lines 269-282 of `EndgameEloTimelineSection.tsx`) was reachable. After the rewire, an empty-combos response now silently renders nothing — users with sparse endgame history lose the helpful guidance and just see the chart vanish. The component's `isLoading={overviewLoading}` and `isError={overviewError}` props are also dead — `eloTimelineData` is `undefined` in both states, so the parent gate short-circuits before the component renders its skeleton or `endgame-elo-timeline-error` UI. The component's class-level docstring (lines 22-25) explicitly claims to own these branches; that promise is no longer met by any production caller.

**Fix:** Change the wrapper gate to render the component whenever the parent has a non-loading overview, letting the component decide between chart / empty / error internally:

```tsx
{eloTimelineData && (
  <div
    className="charcoal-texture rounded-md p-4"
    data-testid="endgame-elo-timeline-section"
  >
    <EndgameEloTimelineSection
      data={eloTimelineData}
      isLoading={overviewLoading}
      isError={overviewError}
    />
  </div>
)}
```

The page-level `overviewLoading ?` branch at line 364 still catches the global loading state; once that clears, `eloTimelineData` is always defined (the schema guarantees `endgame_elo_timeline: EndgameEloTimelineResponse`) and the component's internal `data.combos.length === 0` empty state at line 269 handles the no-combos case cleanly. Vitest `Endgames.overallPerformance.test.tsx` will need a paired empty-combos assertion.

### WR-02: `EndgameEloTimelinePoint.date` documented inconsistently between backend (Sunday) and frontend (Monday)

**File:** `frontend/src/types/endgames.ts:287`

**Issue:** The frontend interface comment says `date: Monday of ISO week, YYYY-MM-DD.` for `EndgameEloTimelinePoint`. The backend schema docstring (`app/schemas/endgames.py:537`) and the producer (`app/services/endgame_service.py:1378, 1399`) emit the **Sunday** of the ISO week (`sunday = monday + timedelta(days=6)`, `date=sunday.isoformat()`) — intentionally, so a daily rating chart at the same date shows the same actual-rating value. The Monday claim is left over from the `ScoreGapTimelinePoint` semantics; copying it into `EndgameEloTimelinePoint`'s TSDoc is misleading.

**Fix:** Update line 287 of `frontend/src/types/endgames.ts`:

```ts
 *  date: Sunday of ISO week (end of week), YYYY-MM-DD. Aligned with the asof
 *    rating moment so a daily rating chart at the same date shows the same value.
```

## Info Findings

### IN-01: `_compute_endgame_elo_weekly_series` docstring contradicts the implementation

**File:** `app/services/endgame_service.py:1340-1341`

**Issue:** The docstring says `endgame_elo = round(actual_elo + K · eg_score_gap)` where `eg_score_gap` is `pt.endgame_score - pt.non_endgame_score`. The implementation (line 1394) uses `pt.score_difference` instead, which is independently rounded to 4 decimals (and may differ from `pt.endgame_score - pt.non_endgame_score` by sub-ulp noise). 87.5-01-SUMMARY Decision #4 is explicit about why `score_difference` is the canonical pre-rounded value. The docstring should match the implementation to avoid future re-introduction of the half-integer rounding boundary bug.

**Fix:** Replace `pt.endgame_score - pt.non_endgame_score` in the docstring with `pt.score_difference` (and optionally re-state the half-integer rationale).

### IN-02: Inline `{"win": 1.0, "draw": 0.5, "loss": 0.0}` literal duplicates the module-level `_ACTUAL_SCORE_BY_OUTCOME`

**File:** `app/services/endgame_service.py:867`

**Issue:** `_score_for` inside `_compute_score_gap_timeline` declares a fresh dict literal `{"win": 1.0, "draw": 0.5, "loss": 0.0}` to look up the outcome score, but the module already defines `_ACTUAL_SCORE_BY_OUTCOME` (line 826) with the same mapping and a typed `dict[Literal["win", "draw", "loss"], float]` signature for exactly this purpose. The inline literal reallocates on every row, loses the `Literal` typing guard, and is the kind of subtle duplication that drifts when one copy is updated and the other isn't.

**Fix:** Use the module-level constant: `return _ACTUAL_SCORE_BY_OUTCOME[outcome]`. (Pre-existing — not introduced by 87.5 — but the surrounding code is in scope for this refactor; surface for cleanup or accept as out of scope.)

### IN-03: Per-call repository fetch of `query_endgame_elo_timeline_rows` is unchanged but no longer load-balanced

**File:** `app/services/endgame_service.py:2312-2322`

**Issue:** `get_endgame_elo_timeline` still issues its own `query_endgame_elo_timeline_rows` call for the per-combo Actual ELO asof-arrays, even though the orchestrator already has all the row data it needs in `endgame_rows_all` / `non_endgame_rows_all`. The all-rows query is only needed for the per-combo `(white_rating, black_rating)` columns, which `query_endgame_performance_rows` does not currently project. Performance optimization (v1 out-of-scope), but worth flagging as a follow-up: extending `query_endgame_performance_rows` row shape with the rating columns would let the orchestrator drop the second query entirely. Filed for a future cleanup phase, not this one.

### IN-04: `try: monday = datetime.fromisoformat(pt.date).date() except ValueError: continue` silently swallows malformed dates

**File:** `app/services/endgame_service.py:1371-1374`

**Issue:** The defensive `except ValueError: continue` is fine for expected-input parsing, but `pt.date` is produced exclusively by `_compute_score_gap_timeline` (which emits `monday.isoformat()`), so a parse failure indicates a real data corruption, not an expected condition. A silent `continue` will drop the point with zero observability. Consider either asserting (since the producer guarantees the format) or calling `sentry_sdk.capture_exception` per CLAUDE.md's "non-trivial except block" rule. Low priority — the chance of a parse failure in this code path is near-zero given the producer contract.

**Fix:** Either drop the try/except (rely on the producer's invariant) or capture to Sentry:

```python
try:
    monday = datetime.fromisoformat(pt.date).date()
except ValueError as exc:
    sentry_sdk.set_context("endgame_elo_timeline", {"pt_date": pt.date})
    sentry_sdk.capture_exception(exc)
    continue
```
