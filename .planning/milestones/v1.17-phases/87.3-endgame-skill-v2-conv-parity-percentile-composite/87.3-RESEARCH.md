## Summary

Phase 87.3 redefines Endgame Skill by replacing the current rate-based equal-weighted Conv/Parity/Recovery composite with the pooled **percentile rank** of a Conv ΔES + Parity ΔES composite (Recovery excluded — opponent-confounded per LOCKED SPEC). The gauge changes from a rate gauge (0–1 scale, bands ~0.47–0.55) to a percentile gauge (0–100 scale, center at 50). The `_endgame_elo_from_skill` formula (Phase 57) is reused unchanged: `s` is fed the percentile as a fraction in [0,1] — this is the invariant the plan must assert. Recovery survives as a standalone descriptive tile with no changes to its ΔES Score Gap bullet or rate gauge.

The only fundamentally new artifact is the frozen, versioned quantile table: a Python constant module (`endgame_skill_reference.py`) generated from the benchmark DB by a new gen script (`scripts/gen_endgame_skill_reference.py`), with a CI drift-gate that parallels the existing `scripts/gen_endgame_zones_ts.py` / `git diff --exit-code frontend/src/generated/endgameZones.ts` pattern — but applied to the Python module itself (not a TS mirror). The Skill neutral-band zone constant (D-04) flows into `endgameZones.ts` via the existing `ZONE_REGISTRY["endgame_skill"]` entry (updated bounds) and the existing codegen, requiring no new TS surface.

**Primary recommendation:** The composite and percentile computation are pure backend additions. On the backend, the critical seam is replacing `endgame_skill_rate_mean` (the current rate-based gauge driver in `ScoreGapMaterialResponse`) with a new `endgame_skill_percentile: float | None` field. On the frontend, `EndgameSkillCard.tsx` swaps its gauge input from the rate to the percentile, changes zone constants from `ENDGAME_SKILL_ZONES` to a new 0–100 percentile zone, and updates its headline copy. The ELO timeline path (`_endgame_skill_from_bucket_rows` + `_compute_endgame_elo_weekly_series`) gets a parallel percentile path using the frozen reference.

### Pattern 2: Conv+Parity composite (dropping Recovery)

**What:** The current `_compute_skill_score_gap` aggregates three buckets. Phase 87.3 needs a 2-bucket version (Conv+Parity) both at the aggregate level and inside the rolling window for the ELO timeline.

**Critical seam — `_endgame_skill_from_bucket_rows`:**

The current function (lines 1369–1441) computes rate-based skill from raw bucket rows. It feeds `_compute_endgame_elo_weekly_series` at line 1525. The Phase 87.3 replacement must:
1. Compute Conv ΔES mean + Parity ΔES mean from the window's per-span gap series.
2. Return the composite as a float → feed `interpolate_percentile()` → return `s`.

Two implementation options:
- **Option A (recommended):** Replace `_endgame_skill_from_bucket_rows` with a new `_endgame_skill_percentile_from_window(window_rows)` that (a) accumulates per-span gaps per bucket (same logic as `_aggregate_endgame_stats`'s `gaps_by_bucket` but restricted to Conv+Parity), (b) computes the equal-weighted mean, (c) applies `interpolate_percentile`. The ≥10-span floor per active bucket is checked before including a bucket (matching `CONFIDENCE_MIN_N = 10`).
- **Option B:** Reuse the existing `gaps_by_bucket` accumulator from `_aggregate_endgame_stats` and pass it into the ELO timeline path. This avoids recomputing spans but requires threading the dict through additional layers — adds complexity.

Option A is simpler and consistent with the existing rolling-window pattern.

**Rate-based skill gauge driver:** The current `endgame_skill_rate_mean` field in `ScoreGapMaterialResponse` (line 436 of `endgames.py`) was restored in quick task `260515-wye` specifically to drive the Skill gauge after Phase 87.2 dropped the rate-composite field. Phase 87.3 replaces the gauge driver with `endgame_skill_percentile`. The `endgame_skill_rate_mean` field can be dropped — no other consumer exists after the gauge switches. [VERIFIED: codebase grep — only `EndgameSkillCard.tsx` reads `skill` prop from `endgame_skill_rate_mean` via `EndgameMetricsSection.tsx:180`]

### Pattern 3: `_compute_skill_score_gap` — dropping Recovery from the ΔES Skill aggregate

The current `_compute_skill_score_gap` (lines 1095–1153) iterates `("conversion", "parity", "recovery")`. Phase 87.3 changes this loop to `("conversion", "parity")`. The field `section2_score_gap_skill_mean` becomes the Conv+Parity equal-weighted mean (not Conv+Parity+Recovery). This is the ΔES bullet for the Skill card — distinct from the percentile gauge driver. Both the gauge (percentile) and the bullet (ΔES score gap) are displayed on the same card; the bullet changes from 3-bucket to 2-bucket; the gauge changes from rate to percentile.

**Recovery's ΔES bullet survives.** `section2_score_gap_recov_mean` and all five `section2_score_gap_recov_*` fields remain on `ScoreGapMaterialResponse`, unchanged. The Recovery card still renders its ScoreGapRow using these fields (D-06).

### Anti-Patterns to Avoid

- **Do not add `verdict`/`p_value` to the LLM Skill payload** — the zone band is the significance signal (per memory `feedback_llm_significance_signal`).
- **Do not mirror the quantile table to TS** — only the resulting percentile float and the neutral-band bounds (via `ZONE_REGISTRY["endgame_skill"]` → codegen) reach the frontend.
- **Do not auto-regenerate the reference on benchmark rebuild** — manual-refresh-only versioning (D-03) prevents historical ELO timeline drift.
- **Do not re-derive Recovery recovery as a skill metric** — the confound is confirmed by the benchmark spike (reports/spike-clamped-recovery-elo-ramp-2026-05-16.md); the LOCKED SPEC is authoritative.
- **Do not use asyncio.gather on the same AsyncSession** — sequential awaits only (CLAUDE.md constraint).

## Frozen Reference Artifact Mechanics

### Gen Script Specification

**File:** `scripts/gen_endgame_skill_reference.py`

Parallels `scripts/gen_endgame_zones_ts.py` in structure:
1. Accepts `--check` flag for CI verification mode (exits 1 if output differs from committed file).
2. Queries the benchmark DB via `mcp__flawchess-benchmark-db__query` at gen time (not at runtime).
3. Computes per-user Conv+Parity ΔES composite for all benchmark users passing the floor.
4. Derives ~100 quantile knots (p0, p1, ..., p100 or similar density).
5. Writes `app/services/endgame_skill_reference.py` with `REFERENCE_VERSION`, `QUANTILE_KNOTS`, and `interpolate_percentile()`.

**Benchmark query pattern** (from SKILL.md §3.2.2):
The query structure mirrors §3.2.2's per-bucket ΔES query: `spans` CTE → `spans_with_next` (LEAD) → `gap_rows` (with bucket assignment via `_classify_endgame_bucket` logic) → `per_user_bucket` (mean per user per bucket, floor ≥ 20 spans) → `per_user_skill` (equal-weighted mean of Conv+Parity buckets only, requiring ≥10 spans per active bucket). The standard CTE `selected_users` (with `bic.status = 'completed'` filter) and the equal-footing filter (`abs(opp_rating − user_rating) ≤ 100`) are mandatory.

**Sparse cell exclusion:** `NOT (rating_bucket = 2400 AND tc_bucket = 'classical')` in marginals/pooled aggregation (per SKILL.md §1.1.4).

**Knot derivation:** Use `percentile_cont(k/N) WITHIN GROUP (ORDER BY composite_value)` for k in 0..N (e.g. N=100) to get evenly-spaced percentile knots. The resulting `(composite_value, k/N)` pairs are the QUANTILE_KNOTS.

**Median invariant check:** Assert that `interpolate_percentile(median_composite) ≈ 0.5` — this is the SC#3 invariant. The planner should include a verification step that queries the benchmark median and checks the round-trip.

### CI Drift-Gate Extension

**File:** `.github/workflows/ci.yml`, line 47–50

**Current "Zone drift check" step:**
```yaml

  run: |
    uv run python scripts/gen_endgame_zones_ts.py

```

**Extension:** Add the reference artifact check to the same step (or a parallel "Reference drift check" step):
```yaml
    uv run python scripts/gen_endgame_skill_reference.py
    git diff --exit-code app/services/endgame_skill_reference.py
```

The gen script must be runnable in CI without benchmark DB access — it should support a `--check` mode that reads the committed file and diffs against the expected output. However, since the benchmark DB is not available in CI, the check must be a file-diff check (same as the zone script): regenerate locally, commit, CI checks that the committed artifact matches what `gen_endgame_skill_reference.py` would produce. The check can skip the DB query in `--check` mode and simply compare the committed file's content against a format-validity check. Or more simply: the `git diff --exit-code` after running the gen script works because CI has the committed artifact — if the script is deterministic given the same QUANTILE_KNOTS constant, the diff will be empty. This is exactly how `gen_endgame_zones_ts.py` works: the DB query is done locally; the committed output is diffed in CI.

**Important distinction:** The gen script for the zone file reads from Python constants (no DB). The gen script for the quantile table must query the benchmark DB — but this is a local-only operation. The CI drift check only validates that the committed artifact matches what would be regenerated from the same Python source (the constants are embedded in the module). Since the QUANTILE_KNOTS are hardcoded after gen, re-running the gen script in CI (without DB) can re-emit the same Python module from the already-embedded constants. This is equivalent to the zone gen pattern.

**Planner decision:** Whether the gen script re-queries the DB on every run or only when explicitly invoked with a `--refresh` flag. The D-03 constraint (manual-refresh-only) means the planner should add a guard: the script refuses to query the DB unless `--refresh` is passed, and in default (or `--check`) mode it only validates/re-emits the current constants without a DB round-trip.

## Open Questions

1. **D-08 neutral band exact bounds.** The band must be derived from the benchmark Conv+Parity composite distribution (d≈1.39 separating ELO cohorts). The plan-phase must run the §3.2.2 benchmark query extended to the 2-bucket composite to derive [p25, p75] or a tighter editorial band. The current `ZONE_REGISTRY["endgame_skill"]` bounds (0.47–0.55) are on the rate scale and will be incorrect on the percentile scale.
   - What we know: d≈1.39 for the combined composite — the band should be meaningfully tighter than naive 40–60 to honor `feedback_zone_band_judgement`.
   - Recommendation: query the benchmark during Plan 1, use [p25, p75] as the starting point, tighten if the spread is too wide given d≈1.39.

2. **In-memory LEAD computation for the rolling window.** The ELO timeline rolling window is an ordered list of per-game rows. The next-span eval (LEAD) can be computed in-memory by scanning consecutive elements with the same game_id — but the window can contain games that span non-consecutive plies. The planner must verify the window's row ordering (chronological by `played_at`, not by ply-within-game) and determine whether in-memory LEAD is reliable or whether extending the query is safer.
   - What we know: the rolling window is built from `bucket_rows` which are `(played_at, platform, tc, user_color, white_rating, black_rating, eval_cp, eval_mate, result)` — one row per game (not per span within a game). Computing LEAD within a game requires knowing the game's entry spans in order; if the window has only one row per game, the "next span" is the next row with the same game_id, which is never in the same window position.
   - Resolution: the window is per-game (one endgame entry per game). "Next span within the same game" does not apply here because the window is one row per game. The "next entry eval" needed for `_compute_span_gap` transitory exit would need to be the next game's entry eval — which is not meaningful. **For the ELO timeline path, terminal span handling (game result) is the only exit-score method.** This simplifies the rolling window: `_compute_span_gap` uses `game_result_score` (not the next span's eval) because each game is a single entry in the window. The composite is then `gap = game_result_score - ES_entry` per game, restricted to Conv/Parity buckets.

3. **`_endgame_skill_from_material_rows` in `insights_service.py`.** This function recomputes skill from already-aggregated `MaterialRow` data (rates, not percentiles). After Phase 87.3, this function is stale (it returns a rate, not a percentile). The recommended fix (Seam 9, Option A) is to read `endgame_skill_percentile` directly from `response.score_gap_material`. If the planner adopts Option A, `_endgame_skill_from_material_rows` can be removed.
   - Risk: other callers of `_endgame_skill_from_material_rows` — check via grep before deletion.
