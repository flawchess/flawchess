## Decisions Made

- **Captured `expected_score_i` in a per-branch local.** The alternative — re-running the sigmoid in a separate post-loop pass — would compute the same value twice per row and create a maintenance risk (two sites to keep aligned if the eval helpers' contract changes). The single-pass-with-capture form is also the cleanest seam for the d_n == ex_n invariant.
- **Refactored the existing ex_sum/ex_n loop body to early-continue style.** The Phase 83 original had an `if … elif … # else skip` shape with the accumulator update inside each branch. Adding the paired-diff append after every accumulator increment without re-introducing duplication required restructuring to a single post-branch `ex_sum += …; ex_n += 1; diffs.append(…)` block guarded by an early-continue on the NULL-both branch. Same observable behavior, but the d_n == ex_n invariant is now obvious from the control flow.
- **Inlined the `outcome → actual_score` mapping** instead of factoring out a helper. The dict literal `{"win": 1.0, "draw": 0.5, "loss": 0.0}` lives at exactly one call site, is one line long, and benefits from being adjacent to the loop. Per CLAUDE.md "don't invent context dataclasses to make signatures fit" — single-use 3-entry dict goes inline.
