## Math Derivations

### Test 1: Independent Two-Sample Z-Test (Endgame Score Gap)

**What we're testing:**  
H0: `score_eg − score_ne = 0`  
where `score_eg = (W_eg + 0.5·D_eg) / N_eg` and `score_ne = (W_ne + 0.5·D_ne) / N_ne`.

**Trinomial empirical variance** (per side) [VERIFIED: `score_confidence.py:219`]:
```
var_i = max(0, (W_i + 0.25·D_i) / N_i  −  score_i²)
```
This is the uncentered second moment of the trinomial distribution on (1, 0.5, 0):
- A win contributes `1² = 1` to `E[X²]`, giving `W/N` as its term.
- A draw contributes `0.5² = 0.25` to `E[X²]`, giving `0.25·D/N`.
- `score_i²` is the squared mean term (variance = E[X²] − (E[X])²).

**Standard error of the difference:**
```
SE_diff = sqrt(var_eg / N_eg  +  var_ne / N_ne)
```

**Z-statistic:**
```
z = (score_eg − score_ne) / SE_diff
p_value = erfc(|z| / sqrt(2))    # two-sided
```

**95% Wald CI on the difference:**
```
diff ± CI_Z_95 * SE_diff
ci_low  = (score_eg − score_ne) − 1.96 * SE_diff
ci_high = (score_eg − score_ne) + 1.96 * SE_diff
```

**Why Wald here, not Wilson:**  
Wilson applies to testing a single proportion against a fixed H0. The difference of two independent proportions does not reduce to a single-proportion problem; Wald (normal approximation on the difference) is the textbook choice. At `min(N_eg, N_ne) >= 10` the normal approximation is adequate. [ASSUMED — standard stats; no citation needed for elementary frequentist theory]

**Helper signature:**
```python
def compute_score_difference_test(
    eg_w: int, eg_d: int, eg_l: int, eg_n: int,
    ne_w: int, ne_d: int, ne_l: int, ne_n: int,
) -> tuple[float | None, float | None, float | None]:
    """Return (p_value, ci_low, ci_high) for the independent two-sample z-test.

    p_value: None when eg_n < MIN_N or ne_n < MIN_N (SEC1-08 gate).
    ci_low/ci_high: None when eg_n < 2 or ne_n < 2 (SEC1-09 gate).
    All three can be None simultaneously (degenerate input).
    """
```

Where `MIN_N = 10` (= `OPENING_INSIGHTS_CONFIDENCE_MIN_N`).

**Edge cases to unit-test:**
- `eg_n = 0` or `ne_n = 0` → all three `None`.
- Either `n < 10` → `p_value = None`; CI gated separately at `n < 2`.
- All-wins one side, all-losses other side: `var_i = max(0, 1·1/1 − 1²) = 0` for all-wins. `SE_diff = 0` → variance-0 trap. When `SE_diff = 0`, `score_eg != score_ne` → return `p_value = 0.0`; `score_eg == score_ne` → return `p_value = 1.0`. CI half-width = 0 in this branch (point estimate, no spread).
- Identical distributions both sides (`score_eg == score_ne`): `z = 0`, `p_value = 1.0`.

**Variance-0 trap:** When all games in one side have the same outcome (all wins or all losses), the empirical variance is 0 exactly (clamped by `max(0, ...)` per the existing code). `SE_diff = 0` → z is undefined. The implementation must short-circuit: if `SE_diff == 0`, set `p_value = 0.0` if scores differ, else `p_value = 1.0`. This pattern already exists in `eval_confidence.py:116-119` [VERIFIED: `eval_confidence.py:116-119`].

### Test 2: Paired One-Sample Z-Test (Achievable Score Gap)

**What we're testing:**  
H0: `mean(d_i) = 0`  
where `d_i = actual_score_i − expected_score_i`.

`actual_score_i` = `{win: 1.0, draw: 0.5, loss: 0.0}[derive_user_result(row.result, row.user_color)]`  
`expected_score_i` = `eval_cp_to_expected_score(row.eval_cp, row.user_color)` or `eval_mate_to_expected_score(row.eval_mate, row.user_color)`.

**Collecting the pairs:**  
The `bucket_rows` loop in `_get_endgame_performance_from_rows` (lines 1780-1796) already iterates all bucket rows and computes `expected_score_i` via the sigmoid. We need to extend this loop to also record `actual_score_i = derive_user_result_to_score(row.result, row.user_color)` and accumulate the difference `d_i`. No new query is needed.

**Accumulation:**
```python
d_sum = 0.0
d_sumsq = 0.0
d_n = 0
# inside the existing loop:
d_i = actual_score_i - expected_score_i
d_sum += d_i
d_sumsq += d_i * d_i
d_n += 1
```

**Mean and SE (Bessel-corrected):**
```
mean_d = d_sum / d_n
var_d = max(0, (d_sumsq - d_n * mean_d²) / (d_n - 1))   # Bessel correction for n >= 2
se_d = sqrt(var_d / d_n)
z = mean_d / se_d
p_value = erfc(|z| / sqrt(2))   # two-sided
ci_low  = mean_d - CI_Z_95 * se_d
ci_high = mean_d + CI_Z_95 * se_d
```

The Bessel-corrected sample variance is the same formula used in `eval_confidence.py:113` [VERIFIED: `eval_confidence.py:113`].

**Helper signature:**
```python
def compute_paired_difference_test(
    diffs: Sequence[float],
) -> tuple[float, float | None, float | None, float | None]:
    """Return (mean_d, p_value, ci_low, ci_high).

    mean_d: always returned (0.0 when len(diffs) == 0).
    p_value: None when len(diffs) < 10 (SEC1-10 gate).
    ci_low/ci_high: None when len(diffs) < 2.
    """
```

The helper takes the already-computed diff sequence (not raw (actual, expected) pairs) to keep it testable in isolation.

**Why not pass `(actual, expected)` pairs?**  
Single responsibility: the helper only needs the diffs to compute the test. The `actual_score_i` → `d_i` transformation stays in the service where it can be audited alongside the existing expected-score loop. The helper is a pure statistics function.

**`actual_score_i` derivation:**  
The function `derive_user_result` (imported from `openings_service` at line 71) returns `"win"`, `"draw"`, or `"loss"`. Map to score via `{"win": 1.0, "draw": 0.5, "loss": 0.0}`. This mapping already exists inline at `endgame_service.py:1560` [VERIFIED: line 1560]. Do not duplicate — extract a small helper or reuse the inline dict.

**Edge cases to unit-test:**
- `diffs = []` (n=0) → `mean_d=0.0`, all three `None`.
- `diffs = [0.1]` (n=1) → `mean_d=0.1`, `p_value=None`, `ci_low=None`, `ci_high=None` (Bessel undefined at n=1; forced low by N gate anyway).
- All-equal pairs (`d_i = 0` for all): `var_d = 0`, `se_d = 0` → same variance-0 trap as above. `mean_d = 0 == H0` → `p_value = 1.0`.
- All pairs identical but nonzero (`d_i = c ≠ 0`): `var_d = 0`, `se_d = 0` → `p_value = 0.0` (perfectly determined signal).
- Missing `expected_score_i`: rows with both `eval_cp = None` and `eval_mate = None` are already excluded from the `ex_n` loop at lines 1788-1796. The paired diff loop must apply identical exclusions so `d_n == ex_n` by construction.

**`d_n` alignment with `ex_n`:** The paired diff accumulator must run on the exact same row set as the `ex_sum`/`ex_n` accumulator, so `d_n == ex_n` holds. This simplifies gating: re-use `ex_n` for the p-value / CI gate rather than tracking a separate counter. (Or track `d_n` independently as a safety check — planner decides.)

## `bucket_rows` Cursor Shape

`query_endgame_bucket_rows` returns one row per endgame game. Each row has labeled columns [VERIFIED: `endgame_repository.py:252`, `endgame_repository.py:307-312`]:

```
game_id        int
endgame_class  int   (first endgame class for the game)
result         str   ("1-0" | "0-1" | "1/2-1/2")
user_color     str   ("white" | "black")
eval_cp        int | None   (white-perspective, no sign flip in SQL)
eval_mate      int | None   (white-perspective, no sign flip in SQL)
```

The sign flip (`sign = 1 if user_color == "white" else -1`) is applied in the service layer (endgame_service.py:1731).

**What `entry_expected_score` is:** Mean over `bucket_rows` of `eval_cp_to_expected_score(eval_cp, user_color)` (Lichess sigmoid, user-perspective) for non-mate rows, and `eval_mate_to_expected_score(eval_mate, user_color)` for mate rows. Rows with both NULL are excluded. This is already computed in the `ex_sum`/`ex_n` loop at lines 1780-1796 [VERIFIED].

**How to pull per-game `(actual, expected)` pairs without a new query:**  
The `ex_sum`/`ex_n` loop already loops `bucket_rows` once for expected scores. Extend it to also accumulate `d_i = actual_score_i − expected_score_i`. The actual score needs `derive_user_result(row.result, row.user_color)` → score map. The two loops can be merged into one:

```python
# Extend the existing ex_sum/ex_n loop:
for row in bucket_rows:
    expected_score_i: float
    if row.eval_mate is not None:
        expected_score_i = eval_mate_to_expected_score(row.eval_mate, row.user_color)
    elif row.eval_cp is not None:
        if abs(row.eval_cp) >= EVAL_CLIP_MAX_CP:
            continue
        expected_score_i = eval_cp_to_expected_score(row.eval_cp, row.user_color)
    else:
        continue
    ex_sum += expected_score_i
    ex_n += 1
    # NEW: paired diff accumulation
    outcome = derive_user_result(row.result, row.user_color)
    actual_score_i = {"win": 1.0, "draw": 0.5, "loss": 0.0}[outcome]
    d_i = actual_score_i - expected_score_i
    d_sum += d_i
    d_sumsq += d_i * d_i
```

The `d_n` counter is implicit: `d_n == ex_n` by construction (same filter logic). No second pass over `bucket_rows` is needed.

**Warning:** `bucket_rows` is a `Sequence[Row[Any] | tuple[Any, ...]]`. The code iterates it once (lines 1722-1735 for eval, 1780-1796 for expected score). The current implementation uses two separate loops over `bucket_rows`. This is valid because `bucket_rows` is a materialized list (returned as `list(result.fetchall())` at line 329 [VERIFIED]) — not a generator. Merging the two loops is a clean refactor if the planner chooses to, but iterating twice is also safe.

## D-04 Reopen: Recommendation on Gap Font Coloring

**Phase 85 D-04 decision (locked):** "Zone-color only — no sig gating." Gap font color is determined solely by whether the value is above/below `SCORE_GAP_NEUTRAL_MIN/MAX` (`±0.10`).

**What reopening would mean:** Replace zone-only coloring with the standard triple gate: `n >= MIN_GAMES_FOR_RELIABLE_STATS AND p < 0.05 AND value outside neutral band`. This is what Cards 1 and 3 (score rows) use.

**Arguments for reopening (tighten):**
- Consistency within the card: the two `ScoreGapRow` rows would use the same coloring convention as every other sig-gated element in v1.17.
- Avoids misleading green/red for users with 5 endgame games.
- SEC1-11 defaults to "informational only" but explicitly permits reopening.

**Arguments against (keep zone-only):**
- D-04 was a deliberate design decision made during Phase 85 planning with the user's review.
- The gap rows already get CI whiskers (this phase), which provides the uncertainty signal visually.

## Out-of-Scope Flags

### Sigmoid Bias Caveat (InfoPopover Note, not Test Input)

The Achievable Score Gap test treats Stockfish-derived expected scores as ground truth. The Lichess sigmoid was calibrated on a broad population; for a specific rating bucket (e.g., 1500 rapid) the sigmoid may systematically over- or under-predict conversion probability. This means a negative `achievable_score_gap` may partly reflect sigmoid miscalibration rather than pure user underperformance.

This is documented in the ROADMAP as FUT-05: "quantify sigmoid bias on the benchmark cohort and adjust the test." For Phase 85.1, the caveat belongs in the `InfoPopover` on the Achievable Score Gap row, not in the test computation. The existing `AchievableScorePopover.tsx` (referenced in Phase 85 REVIEW at WR-04) may be updated or its copy augmented. The current popover copy [VERIFIED: `EndgameOverallPerformanceSection.tsx:157-162`] mentions "2300+ rated rapid player" — adding one sentence noting that the baseline is a Lichess population estimate and may vary by rating tier is sufficient.

## Common Pitfalls

### Pitfall 1: Iterating `bucket_rows` Twice

`bucket_rows` is a materialized list — safe to iterate twice. But the Phase 83 code already has two separate `for row in bucket_rows:` loops (lines 1722-1735 and 1780-1796). Adding a third loop for the paired diff is valid but wasteful. The cleaner solution is merging the expected-score and diff accumulation into one loop. Watch that the sign flip for `eval_cp` (line 1731 in the eval loop) uses `signed_cp` not `raw eval_cp` — the expected-score loop calls `eval_cp_to_expected_score(row.eval_cp, row.user_color)` which does the sign flip internally [VERIFIED: function signature uses user_color]. Don't double-apply the sign.

### Pitfall 2: Variance-0 When SE_diff = 0

When one cohort has all wins or all losses, `var_i = max(0, 1 - 1²) = 0`, so `SE_diff = 0`. The test is undefined. Short-circuit: if `SE_diff == 0`, check whether `score_eg == score_ne` and return `p_value = 1.0` (no difference detectable) or `p_value = 0.0` (perfect separation). This pattern is already established in `eval_confidence.py:116-119` [VERIFIED].

### Pitfall 3: `None` vs `undefined` at the Frontend Boundary

Schema fields `float | None` become `number | null` in TypeScript. `MiniBulletChart.ciLow` is typed `number | undefined` (optional prop). The conversion `data.achievable_score_gap_ci_low ?? undefined` is required at every `ScoreGapRow` call site. Missing this coercion causes TypeScript errors that block the build.

### Pitfall 4: Frontend Derivation Removal

`EndgameOverallPerformanceSection.tsx:80-82` currently derives `achievableGapValue = withScore - achievableScore`. After this phase, `achievableGapValue` should be `data.achievable_score_gap` (server-computed). The `withScore` local variable (line 71-75) may still be needed for other uses — check before removing. Looking at the file: `withScore` is used only for `achievableGapValue` (line 81) and nothing else in the rendered output. After the schema change, `withScore` becomes dead code if `achievableGapValue` is replaced wholesale. Remove it to keep knip clean.

### Pitfall 5: `ScoreGapMaterialResponse` Is Constructed in `_compute_score_gap_material`, Not `_get_endgame_performance_from_rows`

The two response types are built in different functions. The independent test must be added to `_compute_score_gap_material` (line 723) where both `endgame_wdl` and `non_endgame_wdl` are in scope. The paired test must be added to `_get_endgame_performance_from_rows` (line 1687) where `bucket_rows` is in scope. Do not accidentally add the wrong test to the wrong function.

### Pitfall 6: `ex_n` as Gate vs Separate `d_n`

If the diff accumulation is merged into the `ex_n` loop, the `d_n` is identical to `ex_n`. Using `ex_n` as the gate for both `achievable_score_gap_p_value` and `achievable_score_gap_ci_*` is correct and eliminates a duplicated counter. Add a comment: `# d_n == ex_n by construction (same filter applied to both accumulators)`.

### Pitfall 7: `score_difference` Scale

`score_difference = endgame_score - non_endgame_score` is on the 0–1 scale (e.g., `+0.05` = 5 percentage points). The CI bounds are also on this scale. The frontend formats the gap as `Math.round(value * 100)%` (line 61 of `EndgameOverallPerformanceSection.tsx`). CI bounds should NOT be multiplied by 100 before passing to `MiniBulletChart` — the chart operates in raw 0–1 units matching `value`.
