# Phase 85.1: Code Review Report

**Reviewed:** 2026-05-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 85.1 implements two new hypothesis tests (independent two-sample z-test on the WDL chess-score difference; paired one-sample z-test on per-game actual-minus-expected diffs) plus 95% Wald CIs, and migrates the Achievable Score Gap from a frontend derivation to a server-authoritative paired-difference statistic. The math is correctly implemented: the variance-0 traps mirror the established `eval_confidence.py:116-119` pattern, the n-gates (p at n>=10, CI at n>=2) are applied independently per spec, and `compute_paired_difference_test` correctly short-circuits the n=1 Bessel singularity. Test coverage is dense, including degenerate cases (all-wins-vs-all-losses, all-draws both sides, n=1, empty input) and a hand-computed Bessel sanity check that explicitly rejects the naive `/n` variance. The `PVALUE_RELIABILITY_MIN_N` constant cleanly retires the four prior bare-`10` gates (carry-forward from Phase 85 REVIEW IN-01).

Three warnings worth fixing before this card ships to users:

1. **Center vs CI bound numerical inconsistency** on `score_difference`: the displayed center is built from `_wdl_to_score` (rounded `win_pct`/`draw_pct` to 1 decimal place), while the helper recomputes from raw W/D counts. Up to ~0.001 disagreement between the bullet's value and its CI midpoint.
2. **Stale popover prose** on the Endgame Score Differences card: text still describes the legacy "Endgame Score minus the Achievable Score" derivation; the new server value is `mean(actual_i - expected_i)`, which is a different statistic when the per-game cohort differs from `endgame_wdl.total`.
3. **Loose dict type** on `_ACTUAL_SCORE_BY_OUTCOME` (declared `dict[str, float]` instead of `dict[Literal["win", "draw", "loss"], float]`) — violates CLAUDE.md "Type safety / never use bare `str` for fields with a fixed set of values."

The four info items are cleanup opportunities (duplicate constant, redundant CI computation at sub-gate sample sizes, eval_mate=0 edge case in upstream helper that's now exercised by the new path, and a comment claim that doesn't quite match the implementation).

## Warnings

### WR-01: `score_difference` center can disagree with CI midpoint due to upstream rounding

**File:** `app/services/endgame_service.py:772-789`
**Issue:**
`_compute_score_gap_material` builds `score_difference` as `_wdl_to_score(endgame_wdl) - _wdl_to_score(non_endgame_wdl)`, and `_wdl_to_score` uses the *rounded* (`round(wins / total * 100, 1)`) `win_pct` / `draw_pct` from `_build_wdl_summary` (line 583-585). Meanwhile `compute_score_difference_test` recomputes both scores from the raw integer W/D/N counts:
```python
score_eg = (eg_w + 0.5 * eg_d) / eg_n   # unrounded
```
So the CI is centered on the unrounded diff, but the chart plots `value=scoreGap.score_difference` (rounded). Two sources of truth for the same scalar. Worst-case visual disagreement is ~0.001 (max ±0.05% rounding on each side, then a 0.5× draw weight), which is invisible on the bullet chart but is a latent landmine if the CI ever gets re-derived from the center (e.g. via `(ci_low + ci_high) / 2`) and compared to the field.

**Fix:**
Use raw integer counts for `score_difference` too, so the response field and the CI come from the same expression:
```python
endgame_score = (
    (endgame_wdl.wins + 0.5 * endgame_wdl.draws) / endgame_wdl.total
    if endgame_wdl.total > 0
    else 0.0
)
non_endgame_score = (
    (non_endgame_wdl.wins + 0.5 * non_endgame_wdl.draws) / non_endgame_wdl.total
    if non_endgame_wdl.total > 0
    else 0.0
)
score_difference = endgame_score - non_endgame_score
```
Or extract a `_wdl_to_score_raw(wdl) -> float` helper that consumes the integer fields directly. The existing rounded `_wdl_to_score` can stay for any non-stat-test consumers, but the score that's compared against a CI must use the same precision as the CI's center.

### WR-02: Endgame Score Differences popover prose describes the old derivation

**File:** `frontend/src/components/charts/EndgameOverallPerformanceSection.tsx:152-167`
**Issue:**
The InfoPopover for the **Achievable Score Gap** row reads:
> "Your Endgame Score minus the Achievable Score from your endgame-entry positions. Positive means you converted your endgame entry positions better than a 2300+ rated rapid player would on average..."

That described the previous frontend derivation `withScore - entry_expected_score`. The new server value `data.achievable_score_gap` is `mean(actual_score_i - expected_score_i)` over the per-game pairs — a paired difference, not the difference of two cohort means. These two quantities are only equal when the cohorts match exactly. They don't: `withScore` is computed over `endgame_wdl.total` games, while the paired mean is over `entry_expected_score_n` games (which drops mate-NULL-both and `|eval_cp| >= 2000` rows; mate is INCLUDED — so the surviving cohorts are strict subsets, not equal).

The test `EndgameOverallPerformanceSection.test.tsx::"achievable score gap shows signed data.achievable_score_gap..."` explicitly demonstrates the migration with a fixture where the server value (0.07, -0.15) diverges from the legacy derivation. The user-facing prose, however, still claims the latter.

**Fix:**
Reword the popover to describe the actual paired statistic:
```tsx
<p>
  Your average per-game score minus the achievable score from each
  endgame-entry position. Positive means you converted your endgame entry
  positions better than a 2300+ rated rapid player would on average;
  negative means you converted them worse.
</p>
```
The "calibration drift" caveat in the second paragraph still applies and can stay verbatim.

### WR-03: `_ACTUAL_SCORE_BY_OUTCOME` uses bare `str` keys instead of the `Literal` it lookups

**File:** `app/services/endgame_service.py:1815`
**Issue:**
```python
_ACTUAL_SCORE_BY_OUTCOME: dict[str, float] = {"win": 1.0, "draw": 0.5, "loss": 0.0}
...
outcome = derive_user_result(row.result, row.user_color)  # returns Literal["win", "draw", "loss"]
actual_score_i = _ACTUAL_SCORE_BY_OUTCOME[outcome]
```
CLAUDE.md "Type safety" rule: "Never use bare `str` for fields with a fixed set of values — use `Literal["a", "b", "c"]`." The producer (`derive_user_result`) returns `Literal["win", "draw", "loss"]`; the dict is keyed by exactly those three strings; nothing else should ever index it. The current `dict[str, float]` declaration silently permits a future caller to index with an arbitrary string and KeyError at runtime instead of being caught by ty/mypy at edit time.

A related minor: the variable is named `_ACTUAL_SCORE_BY_OUTCOME` (SCREAMING_SNAKE, leading underscore — module-private constant convention) but it's declared inside a function body, freshly allocated on every call to `_get_endgame_performance_from_rows`. The constant naming is misleading; either lift it to module scope (zero runtime cost, matches the existing `_MATERIAL_BUCKET_LABELS` pattern at line 616) or rename to a regular local variable (`outcome_to_actual_score`).

**Fix:**
Lift to module scope and tighten the type:
```python
# Module scope, near _MATERIAL_BUCKET_LABELS
_ACTUAL_SCORE_BY_OUTCOME: dict[Literal["win", "draw", "loss"], float] = {
    "win": 1.0,
    "draw": 0.5,
    "loss": 0.0,
}
```
Drop the in-function declaration. The dict is read-only and re-allocating on every request is unnecessary.

## Info

### IN-01: `PVALUE_RELIABILITY_MIN_N` and `CONFIDENCE_MIN_N` are duplicate constants

**File:** `app/services/endgame_service.py:191`, `app/services/opening_insights_constants.py` (referenced)
**Issue:**
`PVALUE_RELIABILITY_MIN_N: int = 10` (endgame_service.py) and `OPENING_INSIGHTS_CONFIDENCE_MIN_N = 10` (imported as `CONFIDENCE_MIN_N` inside `score_confidence.py`) both encode the same n>=10 wire-format reliability gate, with identical semantics. The carry-forward comment acknowledges this: "currently 10 — REVIEW IN-01 carry-forward". A reader has to discover that both names refer to the same threshold by reading three modules.

**Fix:**
Either (a) import `CONFIDENCE_MIN_N` directly into `endgame_service.py` and drop `PVALUE_RELIABILITY_MIN_N`, or (b) define `PVALUE_RELIABILITY_MIN_N = CONFIDENCE_MIN_N` so the equality is enforced at module-load time and a future refactor of the opening-insights constant doesn't silently drift. The current state (two `= 10` literals living in different files) is the brittle option. This is fine to defer if the next phase has a planned consolidation — flag in a follow-up ticket otherwise.

### IN-02: CI bounds computed and then discarded when `min_n < 2`

**File:** `app/services/score_confidence.py:215-241`
**Issue:**
`compute_score_difference_test` computes `ci_low` and `ci_high` unconditionally (lines 227-229), then gates them to `None` if `min_n < 2`. Same pattern in `compute_paired_difference_test` lines 304-311. At `min_n=1` this is wasted arithmetic (and a `math.sqrt` call); not a correctness issue but symmetric with how `entry_eval_ci_*` in `_get_endgame_performance_from_rows` skips the computation entirely with an `if eval_n >= 2:` guard.

**Fix:**
Defer the CI arithmetic behind the gate, matching the existing `_get_endgame_performance_from_rows` pattern:
```python
ci_low_out: float | None
ci_high_out: float | None
if min_n < 2:
    ci_low_out = None
    ci_high_out = None
else:
    ci_half_width = CI_Z_95 * se_diff
    ci_low_out = diff - ci_half_width
    ci_high_out = diff + ci_half_width
```
Same edit applies to `compute_paired_difference_test`. Minor — the cost of the discarded CI is negligible. Skip if you prefer the straight-line form for readability.

### IN-03: Docstring claim "`d_n == ex_n` by construction" is true but easy to violate on edit

**File:** `app/services/endgame_service.py:1814, 1851`
**Issue:**
The merged accumulator relies on appending to `diffs` only on the same branches that increment `ex_n`. Currently this holds: every path that reaches `ex_n += 1` also reaches `diffs.append(...)` immediately after (lines 1835-1843). But the invariant is structural, not enforced — a future edit that adds a third filter (e.g. drop rows where `derive_user_result` somehow returns an unexpected value) inside this block could silently break the invariant.

The test `test_achievable_score_gap_d_n_equals_ex_n_invariant` covers the happy path with the current three exclusion rules, but doesn't assert `len(diffs) == ex_n` directly — it only checks the resulting mean is the expected value, which would still pass even if both counters drifted by the same amount.

**Fix:**
Either (a) replace the parallel counter `ex_n` with `len(diffs)` at the end (`ex_n = len(diffs)`), so the invariant is enforced by construction; or (b) add an `assert len(diffs) == ex_n` after the loop as a runtime guard. Option (a) is preferable — one less counter to maintain.

### IN-04: `eval_mate_to_expected_score(0, ...)` returns 0.0 — now reachable from the paired-diff accumulator

**File:** `app/services/eval_utils.py:93-97` (called from `app/services/endgame_service.py:1822`)
**Issue:**
`eval_mate_to_expected_score` returns 0.0 for `eval_mate == 0`, because neither `eval_mate > 0` nor `eval_mate < 0` matches. In practice `eval_mate == 0` should not occur (mate is "in N", N>=1), but the existing helper does not validate. The new paired-diff path adds another consumer of this branch: if a row ever has `eval_mate=0`, the diff `actual_score_i - 0.0` will heavily skew the mean toward `actual_score_i` for that game.

This is upstream of Phase 85.1, but Phase 85.1's new code path increases the exposure surface. Not in this phase's scope to fix — flag for upstream tightening.

**Fix:**
In a follow-up: change `eval_mate_to_expected_score` to either raise `ValueError` for `eval_mate == 0` or treat it the same way `_classify_endgame_bucket` does. If the schema can guarantee `eval_mate != 0` at the DB level (CHECK constraint or NOT NULL + Stockfish contract), do that instead.
