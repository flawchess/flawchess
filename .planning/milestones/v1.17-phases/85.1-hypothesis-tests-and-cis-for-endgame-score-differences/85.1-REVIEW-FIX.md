## Fixed Issues

### WR-01: `score_difference` center can disagree with CI midpoint due to upstream rounding

**Files modified:** `app/services/endgame_service.py`
**Commit:** 19d6044f
**Applied fix:** Replaced `_wdl_to_score(endgame_wdl) - _wdl_to_score(non_endgame_wdl)` (which rounds `win_pct`/`draw_pct` to 0.1% before dividing) with raw-integer-count expressions `(wins + 0.5 * draws) / total` for both sides, gated on `total > 0`. The response field `score_difference` and the CI midpoint now come from a single source of truth (the unrounded raw counts), matching the formula already used inside `compute_score_difference_test`. Existing tests still pass: `test_score_gap_material_score_difference_negative` constructs WDL fixtures with `_make_wdl_pct` where integer counts coincide with the pct values (`40% of 100 = 40`), so the expected `-0.15` value is unchanged under the new formula. Left `_wdl_to_score` in place for non-stat-test consumers and the timeline series (no behavior change there).

### WR-02: Endgame Score Differences popover prose describes the old derivation

**Files modified:** `frontend/src/components/charts/EndgameOverallPerformanceSection.tsx`, `frontend/src/pages/Endgames.tsx`
**Commit:** 63b423ad
**Applied fix:** Reworded the Achievable Score Gap popover from "Your Endgame Score minus the Achievable Score..." to "Your average per-game score minus the achievable score from each endgame-entry position..." to describe the actual paired-difference statistic `mean(actual_i - expected_i)` rather than the legacy cohort-difference derivation `withScore - entry_expected_score`. The "calibration drift" caveat in the second paragraph was kept verbatim. Also applied the same correction to the matching prose block in `Endgames.tsx` (the page-level Info section) per CLAUDE.md "always apply changes to mobile too / search for duplicated markup before considering a change complete" — the two blocks describe the same metric and would otherwise drift. No test asserted the exact wording, so no test updates were required.

### WR-03: `_ACTUAL_SCORE_BY_OUTCOME` uses bare `str` keys instead of the `Literal` it lookups

**Files modified:** `app/services/endgame_service.py`
**Commit:** c6db290a
**Applied fix:** Lifted `_ACTUAL_SCORE_BY_OUTCOME` from a per-call local in `_get_endgame_performance_from_rows` to module scope, immediately after `_MATERIAL_BUCKET_LABELS`, and tightened its type from `dict[str, float]` to `dict[Literal["win", "draw", "loss"], float]`. `Literal` was already imported in the module. The lookup site (`_ACTUAL_SCORE_BY_OUTCOME[outcome]` where `outcome` comes from `derive_user_result` which returns `Literal["win", "draw", "loss"]`) is now type-safe by construction — a future caller indexing with an arbitrary string is caught by ty at edit time. The reallocation-per-call cost is also eliminated. Replaced the in-function declaration with a one-line comment pointing at the module-scope definition. ty check passes; tests pass.
