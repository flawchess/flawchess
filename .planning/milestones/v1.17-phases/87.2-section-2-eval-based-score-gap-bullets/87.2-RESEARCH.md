## Summary

Phase 87.2 backports Phase 87.1's per-span ΔES bullet pattern to the four Section 2 cards (Conversion, Parity, Recovery, Endgame Skill), restricted to each card's eval-entry bucket. **All prerequisite per-span infrastructure is shipped:** `_compute_span_gap` (`endgame_service.py:260`), `_classify_endgame_bucket` (`endgame_service.py:196`), `compute_paired_difference_test` (`score_confidence.py:338`), the LEAD-based `query_endgame_entry_rows` row shape (8 columns incl. `next_entry_eval_cp/mate`), `ScoreGapRow` (`EndgameOverallScoreGapRow.tsx`), `MiniBulletChart`, `MetricStatPopover`. **No new statistical primitive is needed** for per-bucket cohorts; equal-weighted Skill aggregation uses three independent paired-z calls (recommendation §Open Questions §3) — the existing `compute_paired_difference_test` signature already supports this with no math invention.

The phase's primary novelty over Phase 87.1 is the bucket filter in `_aggregate_endgame_stats` and the deletion sweep across the rate-based mirror-bucket plumbing (`_MIRROR_BUCKET`, `_compute_skill_wire`, `compute_skill_diff_test`, `compute_per_bucket_diff_test`, frontend `MIRROR_BUCKET`/`userRate`/`opponentRate`/`formatDiffPct` etc., `MaterialRow.{opponent_score, opponent_games, diff_p_value, diff_ci_low, diff_ci_high}`, `ScoreGapMaterialResponse.{skill, opp_skill, skill_diff_*}`, and TS-type mirrors). Cross-phase grep confirms ZERO non-Section-2 consumers — the deletion is safe and clean.

**Critical drift to flag for the planner:** the CONTEXT D-09 framing ("swap 4 rate-diff fields for 4 ΔES fields in the LLM payload") implies rate-diff fields currently flow to the LLM. **They do not.** Section 2's LLM emission is rate-only (`endgame_skill`, `conversion_win_pct`, `parity_score_pct`, `recovery_save_pct` findings via `_findings_endgame_metrics`). The peer-diff fields (`opponent_score`, `diff_p_value`, `skill_diff_p_value`) exist on the wire schema but have no LLM consumer. D-09 should be re-framed as "ADD 4 ΔES Gap fields to the Section 2 / overview payload"; the prompt-version bump still happens.

**Primary recommendation:** 4 plans in 2 waves. Wave A (parallel): Plan 1 (benchmarks skill §3.4.4 + zone codegen) and Plan 2 (backend math + schema + service + dedupe of skill helper unit tests). Wave B (parallel after Wave A merges): Plan 3 (frontend refactor + aggressive deletion + Vitest) and Plan 4 (LLM payload + prompt v31 + insights findings extension). Optional Plan 5: HUMAN-UAT sweep.

### Pattern 2: Equal-weighted Skill aggregator with denominator-drop fallback

**What:** Compute `Skill_ΔES = (m_c + m_p + m_r) / n_active` where each component is the per-bucket mean from `compute_paired_difference_test`, and `n_active` counts only buckets with `n >= CONFIDENCE_MIN_N` (=10).

**When to use:** Phase 87.2 only. Mirrors the existing rate-based Skill formula's philosophy.

**Example (locked by CONTEXT D-01; CI propagation in Open Question §3):**

```python
# Source: CONTEXT D-01 locked decision; primitives in score_confidence.py:338 [VERIFIED]
from app.services.score_confidence import compute_paired_difference_test, CONFIDENCE_MIN_N

results: dict[MaterialBucket, tuple[float, float | None, float | None, float | None]] = {
    b: compute_paired_difference_test(gaps_by_bucket[b])
    for b in ("conversion", "parity", "recovery")
}
# Each result: (mean, p_value, ci_low, ci_high). n is len(gaps_by_bucket[b]).

# Equal-weighted Skill mean — drop components below CONFIDENCE_MIN_N (D-01 edge case):
active_components: list[float] = [
    results[b][0]  # mean
    for b in ("conversion", "parity", "recovery")
    if len(gaps_by_bucket[b]) >= CONFIDENCE_MIN_N
]
skill_mean: float | None = (
    sum(active_components) / len(active_components)
    if active_components
    else None
)
# Skill CI / p_value: see Open Question §3 for the propagation method.
```

### Anti-Patterns to Avoid

- **Threading a "context dataclass" through helpers that only have 2 fields.** Don't invent `Section2GapContext` just to make signatures tidy (per CLAUDE.md "Don't invent context dataclasses"). Pass `gaps_by_bucket` and the per-bucket result tuples directly.
- **Treating `gap_span = exit − entry` as positionally interchangeable with the old `score − opponent_score` Gap.** The math is genuinely different — Stockfish-baseline anchor vs same-game mirror identity. The visual axis is identical (`MiniBulletChart` centered on 0 with CI whiskers) but the **interpretation** differs. The popover copy must say this explicitly (no "vs opponents" framing per D-08).
- **Adding a `verdict` or per-card `p_value` field to the LLM payload.** Per memory `feedback_llm_significance_signal.md` — zone band IS the significance signal. The 4 new fields per card on the wire are `{mean, n, zone, neutral_band}` only.
- **Trying to keep `_compute_skill_wire` "in case Section 3 uses it".** Cross-phase grep confirmed it's Section-2-only. Delete.
- **Hand-editing `frontend/src/generated/endgameZones.ts`.** Always regenerate via `uv run python scripts/gen_endgame_zones_ts.py`. CI catches drift.

## Common Pitfalls

### Pitfall 1: Bucketing the wrong span when a game has multiple spans

**What goes wrong:** `_aggregate_bucket_counts` (`endgame_service.py:921`) picks ONE bucket per GAME for the rate-based Conv/Parity/Recov counts (priority order: Conv > Recov > Parity). If Phase 87.2 reuses that logic for ΔES bucketing, the per-bucket cohort would aggregate at game-grain rather than span-grain — losing the per-span resolution Phase 87.1 introduced.

**Why it happens:** Reading `_aggregate_bucket_counts` and assuming the same per-game bucket reduction applies. It does for rates (one game per bucket); it does NOT for ΔES (every qualifying span per bucket).

**How to avoid:** Use `gaps_by_bucket` keyed by the per-SPAN `_classify_endgame_bucket(eval_cp, eval_mate, user_color)` call evaluated on each row inside the `for row in rows:` loop at `endgame_service.py:380-470`. Phase 87.1's `gaps_by_class` already lives in that loop — Phase 87.2 piggybacks on the same iteration, just keyed by bucket instead of class.

**Warning signs:** Per-bucket span counts that suspiciously equal per-bucket game counts. Test with a fixture where one game has spans in two different eval-entry buckets (e.g. starts +2 rook conversion, transitions to −1 pawn recovery via a blunder) — the same game should contribute one span to each bucket cohort.

### Pitfall 2: Equal-weighted Skill denominator when buckets are sparse

**What goes wrong:** Computing `Skill = (mean_c + mean_p + mean_r) / 3` when one bucket has `n = 0` makes `mean_b = 0.0` (returned by `compute_paired_difference_test` for empty input — verified at `score_confidence.py:375-376`), polluting the average.

**Why it happens:** Naively dividing by 3.

**How to avoid:** Per D-01, count `n_active = number of buckets with n >= CONFIDENCE_MIN_N`. Divide by `n_active`. If `n_active == 0`, Skill is None (card hides with opacity-50).

**Warning signs:** Users with mostly-parity history showing Skill ≈ (Parity mean) / 3 instead of Skill ≈ Parity mean.

### Pitfall 3: Frontend `opponent_games` / `MIN_OPPONENT_BASELINE_GAMES` references surviving in `EndgameMetricsSection.tsx`'s `buildZeroRow`

**What goes wrong:** `EndgameMetricsSection.tsx:111-126` constructs a `buildZeroRow` for buckets missing from the response, populating `opponent_score: null` and `opponent_games: 0`. After D-04 deletes those fields from `MaterialRow`, this zero-row builder must be updated. Likewise `EndgameMetricsSection.tsx:23` imports `MIRROR_BUCKET` and `EndgameMetricsSection.tsx:153` uses it to look up the mirror — both go.

**Why it happens:** Easy to miss the orchestrator's `buildZeroRow` because the deletion focus is on the card components.

**How to avoid:** Knip CI gate. Also: grep `MIRROR_BUCKET\|opponent_games\|opponent_score` in `frontend/src/` after deletion — must return zero hits outside generated files and tests.

**Warning signs:** TS compile errors on `buildZeroRow` after the type deletion.

### Pitfall 4: ty errors on the schema field deletion

**What goes wrong:** `MaterialRow.opponent_games` is `int` (not optional). After deletion, any test fixture or constructor with `opponent_games=0` will fail ty.

**Why it happens:** Pydantic field deletion is a wire-shape change; the field has dozens of references in tests.

**How to avoid:** Search-and-remove via `grep -rn "opponent_games\|opponent_score" tests/` first. Update all test fixtures in the same plan (Plan 2 or Plan 3 depending on where the deletion lands). Same for `diff_p_value`, `diff_ci_low`, `diff_ci_high`, `skill`, `opp_skill`, `skill_diff_p_value`, `skill_diff_ci_low`, `skill_diff_ci_high`.

**Warning signs:** ty CI fails with "missing required argument 'opponent_games' for MaterialRow(...)" — fix every fixture, not the schema.

### Pitfall 5: LLM payload "swap" framing (D-09 drift)

**What goes wrong:** CONTEXT D-09 says "remove the 4 rate-diff fields; add the 4 ΔES fields." But the rate-diff fields (`opponent_score`, `diff_p_value`, `skill_diff_p_value`, etc.) are NOT currently emitted to the LLM. The current Section 2 LLM emission is rate-only (`endgame_skill`, `conversion_win_pct`, `parity_score_pct`, `recovery_save_pct` findings via `insights_service._findings_endgame_metrics`).

**Why it happens:** D-09 was drafted assuming the diff fields were on the wire. They're on the wire schema (`MaterialRow.diff_p_value`, `ScoreGapMaterialResponse.skill_diff_*`) but the `insights_service` and `insights_llm` chains never consume them.

**How to avoid:** Treat D-09 as **"ADD 4 new findings + glossary entry, BUMP version."** No removal of rate-diff fields from the LLM payload (they aren't there). The wire-schema deletion (D-05) is a backend/frontend wire-only cleanup, not an LLM payload cleanup.

**Warning signs:** Trying to find a `material_rows[*].diff_p_value` reference in `insights_llm.py` and not finding one.

### Pitfall 6: Skill aggregator running paired-z on a "synthetic mean-of-means" vector

**What goes wrong:** If the planner mis-implements equal-weighted Skill by constructing a per-bucket-mean vector `[m_c, m_p, m_r]` (length 3) and passing it to `compute_paired_difference_test`, the helper computes `mean(m_c, m_p, m_r)` (correct) BUT the SE / CI is over n=3 with Bessel correction (`(n-1)=2`), which is statistically meaningless.

**Why it happens:** The signature accepts a `Sequence[float]` so this compiles cleanly.

**How to avoid:** Skill aggregation is NOT a paired-z call. It's a deterministic aggregation of three independent per-bucket means. The CI / p propagation is the separate Open Question §3 — and either option (variance-of-sum/9 OR per-game reduction) avoids the "n=3 paired test" trap.

**Warning signs:** Skill p-values that look identical to (or stronger than) the strongest per-bucket p-value — symptom of a degenerate n=3 sample variance.

## Open Questions

### 1. Should the planner pick D-02 option (a) before /benchmarks §3.4.4 runs, or wait for the verdict?

- **What we know:** Option (a) is simpler and accommodates both collapse and divergence outcomes. Option (b) requires inventing a new dispatch shape with no precedent.
- **What's unclear:** Whether /benchmarks §3.4.4 will produce 4 identical bands (Cohen's d < 0.2 across all axes), making the 4-MetricId expansion look like over-engineering.
- **Recommendation:** Lock option (a) now. Plan 1's benchmark calibration produces the actual band values regardless of shape choice. Even if all 4 bands end up at `(-0.05, 0.05)`, the 4-MetricId expansion costs ~12 lines of Python and ~8 lines of TS — negligible vs the cognitive cost of inventing a new dispatch.

### 2. Should `_aggregate_endgame_stats` return a tuple (per-class results + per-bucket gaps) or should `_compute_score_gap_material` re-iterate the row set?

- **What we know:** `_aggregate_endgame_stats` is called from `_compute_score_gap_material` (verified at line 1195 via `_aggregate_bucket_counts` and the surrounding orchestration). Single call site.
- **What's unclear:** Whether the planner prefers a tighter `_aggregate_endgame_stats` signature (just `list[EndgameCategoryStats]`) with a sibling helper for the bucket gaps, or a single tupled-return.
- **Recommendation:** Tupled return. Single iteration over rows, single helper to maintain. The sibling-helper alternative doubles row iteration cost and splits the bucket-classification logic.

### 3. Skill ΔES CI propagation — variance-of-sum/9 (option A) vs per-game equal-weighted reduction (option B)

**Option A: variance-of-sum/9 on three independent paired-z helper outputs**
- For each active bucket b: run `compute_paired_difference_test(gaps_by_bucket[b])` → `(mean_b, p_b, ci_low_b, ci_high_b)`.
- Recover SE_b from the CI: `SE_b = (ci_high_b − ci_low_b) / (2 × CI_Z_95)`.
- `mean_skill = sum(mean_b for b in active) / n_active`.
- `SE_skill = sqrt(sum(SE_b ** 2)) / n_active`.
- `ci_low = mean_skill − CI_Z_95 × SE_skill; ci_high = mean_skill + CI_Z_95 × SE_skill`.
- `p_value = erfc(|mean_skill / SE_skill| / sqrt(2))`.
- Assumes per-bucket means are mutually independent.

**Option B: per-game equal-weighted ΔES reduction (one value per game)**
- For each game with any qualifying ΔES span, compute that game's per-bucket means restricted to spans of each bucket that exist in that game (n_active varies per game; typically 1 or 2 buckets per game).
- Reduce each game to a single equal-weighted scalar `g_i = (mean_conv_in_game + mean_parity_in_game + mean_recov_in_game) / n_active_in_game`.
- Run `compute_paired_difference_test(g_for_all_games)`.
- Cleaner statistical interpretation; assumes the per-game equal-weighted reduction is the metric of interest.

**Recommendation:** Option A. Reasons:
1. **Independence assumption is defensible** because each span contributes to exactly one bucket cohort (no overlap), and the per-bucket means are uncorrelated across players in the benchmark distribution (Cohen's d analysis from §3.4.4 will quantify this).
2. **Reuses the existing helper unchanged**, no new statistical math.
3. **Population invariant**: option A's Skill_mean equals `(m_c + m_p + m_r) / 3` — exactly the D-01 formula. Option B's Skill_mean is a different quantity (the user mean of game-level equal-weighted ΔES scalars), which CONTEXT D-01 does NOT lock to.
4. **Option B requires reshaping the span population per game** — additional service-layer plumbing, more test surface.

Option A's downside (mild correlation between per-bucket means via shared game outcomes) is small and acceptable for a v1.17 heuristic composite — same reasoning as the existing `compute_skill_diff_test` independence caveat at `score_confidence.py:464-469`.

### 4. Section 2 `endgame_skill` rate finding (existing) — keep, deprecate, or hide alongside the new Skill ΔES finding?

- **What we know:** `insights_service._findings_endgame_metrics:720` emits an `endgame_skill` finding with value = mean of per-bucket rates. This is the composite rate (Conv-rate + Parity-rate + Recov-rate)/3, which is what `data.skill` is on the wire (now scheduled for D-05 deletion).

The Skill gauge on the Endgame Skill card still renders the rate-based composite (`(Conv + Parity + Recov) / 3` from per-bucket rates) — only the bullet swaps to ΔES. The LLM benefits from both signals: the rate-based Skill gives the gauge value the user sees; the ΔES Skill gives the Stockfish-anchored performance signal.
