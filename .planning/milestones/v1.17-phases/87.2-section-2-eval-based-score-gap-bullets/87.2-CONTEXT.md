# Phase 87.2: Section 2 — eval-based ΔES Score Gap on Conv/Parity/Recov + Endgame Skill cards - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Backport the Phase 87.1 per-span ΔES Score Gap bullet pattern to **Section 2** (the four cards `EndgameMetricCard` × 3 + `EndgameSkillCard`). Retire the rate-based mirror-bucket peer-diff Gap row (`You / Opp / Gap` + `MiniBulletChart` on `userRate − opponentRate`) on every Section 2 card, replacing each with a per-card **eval-based ΔES Score Gap bullet** restricted to that card's eval-entry bucket. The Phase 87.1 per-span infrastructure (LEAD-based next-span eval, `_compute_span_gap`, `compute_paired_difference_test`, `ScoreGapRow` / `MiniBulletChart` visual primitive, dual-label scheme) is fully shipped and reused as-is — Phase 87.2 adds bucket filtering on top of it.

**Bucket assignment per card** (already encoded in `_classify_endgame_bucket(eval_cp, eval_mate, user_color)` at `app/services/endgame_service.py:196`):

| Card | ΔES cohort (filter applied to per-span gap series) |
|---|---|
| Conversion | spans with entry eval `≥ +EVAL_ADVANTAGE_THRESHOLD` (user-perspective) |
| Parity | spans with entry eval inside `(-EVAL_ADVANTAGE_THRESHOLD, +EVAL_ADVANTAGE_THRESHOLD)` |
| Recovery | spans with entry eval `≤ -EVAL_ADVANTAGE_THRESHOLD` |
| Endgame Skill | **equal-weighted mean of the three bucket means** (NOT pooled all-spans — see D-01) |

**Why the rework:** the current rate-based peer-diff is mathematically degenerate by construction:
- `Conv-Gap ≡ Recov-Gap` by mirror-bucket same-game symmetry (`opp_conv_win% = user_recov_loss%`).
- `Parity-Gap = 2·user_parity − 1` — a deterministic affine of the absolute Parity gauge.
- `Skill-Gap = (2·Conv-Gap + Parity-Gap)/3` — inherits both pathologies.

The ΔES Score Gap measures eval delta during the span (entry-eval expected-score vs exit-score), not symmetric outcomes of a shared game set, so Conv-bucket spans and Recov-bucket spans yield genuinely independent values.

**Out of scope:**
- Re-deriving any per-span math — Phase 87.1's `_compute_span_gap` + LEAD-based next-eval already produce the per-span gap series; this phase only adds the bucket filter on top.
- Lichess-sigmoid refit — deferred (tracked in `.planning/notes/lichess-sigmoid-endgame-calibration.md`).
- Sub-6-ply spans — keep `ENDGAME_PLY_THRESHOLD = 6` for cohort consistency with Phase 87.1.
- Section 3 (`EndgameTypeCard`) — already migrated to the ΔES pattern in Phase 87.1; not touched here.
- Section 1 page-level Achievable Score Gap (Phase 85.1) — unaffected; sits in its own card.

## Implementation Decisions

### Skill card cohort (LOCKED)

- **D-01: Endgame Skill ΔES = equal-weighted mean of the three bucket means.**
  - Formula: `Skill_ΔES = (mean_ΔES_conv + mean_ΔES_parity + mean_ΔES_recov) / 3` over the three Section 2 buckets. **Not** the pooled all-spans mean, and **not** a games-weighted mean.
  - **Rationale:** by telescoping, `Σ_spans (exit_score − ES_entry)` over a single game equals `game_result − ES_first_entry`. So a pooled all-spans mean over the user reduces (up to a span-count reweighting) to the page-level **Achievable Score Gap** (Phase 85.1) — which already has its own bullet on `EndgameOverallPerformanceSection`. Equal-weighted bucket mean is independent from that signal because each bucket has its own anchor (Stockfish baseline conditioned on entry-eval bucket), and it mirrors the existing rate-based Skill formula `mean(Conv_rate, Parity_rate, Recov_rate)` — same equal-importance philosophy.
  - **Edge cases:** if a bucket has `n < CONFIDENCE_MIN_N` for the user, that component is dropped from the equal-weighted average (denominator becomes 2 or 1); if all three are below the floor, Skill ΔES is null (card renders with opacity-50 and the existing "Not enough data yet" treatment from `EndgameSkillCard.tsx:122`). Planner: confirm CI propagation method for an equal-weighted mean of three paired-z tests (per-bucket variance pooled / 3²; or compute the Skill CI from a per-game pooled-bucket reduction — pick whichever the existing `compute_paired_difference_test` signature accommodates without inventing new math).
