# Phase 88.4: Code Review Report

**Reviewed:** 2026-05-19
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

A focused frontend refactor of the Time Pressure card layout (SC-1/SC-2/SC-3). No security issues, no data-loss risk, no crashes. Core logic (sparse handling, unit conversions, zone tinting, Q4 filtering) is sound and well-tested. The findings are quality/robustness concerns: a duplicate-key risk in the custom dot renderer, a `text-xs` violation against the project font-size floor that is not covered by the documented tooltip exception, a test that asserts against an inline re-implementation rather than the production tooltip, and the per-card `MIN_GAMES_PER_TC_CARD` gate being silently load-bearing for the section's `grandTotal` denominator.

## Warnings

### WR-01: `text-xs` in the score-gap tooltip violates the font-size floor and is not covered by the documented exception

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:188`
**Issue:** The tooltip body uses `className="... text-xs ..."`. CLAUDE.md sets `text-sm` as the hard floor for new code. The documented exception is narrowly scoped to "hover/tap-activated info tooltips (Radix popover bodies with the HelpCircle trigger pattern — `MetricStatPopover`, `WdlConfidenceTooltip`, `EvalConfidenceTooltip`, `AchievableScorePopover`, etc.)". A Recharts chart hover tooltip is not a Radix popover body and is not in that enumerated list, so the exception does not cleanly apply. This is primary content (the per-bucket score gap, opponent score, and n) displayed on chart hover. It should be `text-sm` unless the user explicitly extends the exception to chart tooltips.
**Fix:**
```tsx
<div
  className="rounded-lg border border-border/50 bg-background px-3 py-2 text-sm shadow-xl space-y-1"
  data-testid="score-gap-tooltip"
>
```
If `text-xs` is intentionally desired here, get explicit sign-off and add a comment citing the extended exception so the next reviewer does not re-flag it.

### WR-02: Custom dot `key` collides when the chart receives duplicate or empty labels

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:222-231` (also `220`)
**Issue:** The dot renderer keys each `<circle>` by `payload.label`. Labels come from `PRESSURE_LABELS[bin.quintile_index as 0|1|2|3] ?? ''` in `toChartData` (line 99). The cast `as 0|1|2|3` is unchecked: the filter only guarantees `quintile_index <= MAX_VISIBLE_QUINTILE_INDEX (3)`, not `>= 0`. A backend payload with a negative or fractional `quintile_index`, or two bins mapping to the same label, produces duplicate React keys (`''` or a repeated label), which React surfaces as a key-collision warning and can cause incorrect dot reconciliation (stale `cx`/`cy`/color). The same fragile keying exists in the no-dot branch on line 220. The label string is the wrong key source; the ordinal index is the stable identity.
**Fix:** Carry `quintile_index` through `ChartPoint` and key by it (it is unique per visible bin and order-stable):
```tsx
// in ChartPoint
quintile_index: number;
// in toChartData map()
quintile_index: bin.quintile_index,
// no-dot branch
return <g key={`nodot-${payload?.quintile_index ?? cx}`} />;
// dot branch
key={`score-gap-dot-${payload.quintile_index as number}`}
```
Also harden the label lookup so an out-of-range index is dropped rather than silently mapped to `''` (see WR-03).

### WR-03: Out-of-range `quintile_index` is silently coerced to an empty-label point instead of being filtered

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:84-99`
**Issue:** `toChartData` filters `quintile_index <= MAX_VISIBLE_QUINTILE_INDEX` and `n > 0`, but never asserts `quintile_index >= 0`. `PRESSURE_LABELS[bin.quintile_index as 0|1|2|3] ?? ''` then yields `''` for any index outside `{0,1,2,3}`. The result is a chart point with an empty x-axis label, a real `delta`, and (per WR-02) a colliding key. The `as 0|1|2|3` cast actively defeats `noUncheckedIndexedAccess` here — the codebase rule says to narrow, not cast. Backend currently emits 0–4 so this is latent, but the cast hides the only thing that would catch a contract drift.
**Fix:** Replace the filter + cast with an explicit narrowing that drops unknown indices:
```ts
.filter((bin): bin is PressureQuintileBullet & { quintile_index: 0 | 1 | 2 | 3 } =>
  bin.quintile_index >= 0 && bin.quintile_index <= MAX_VISIBLE_QUINTILE_INDEX)
.filter((bin) => bin.n > 0)
.map((bin) => ({ label: PRESSURE_LABELS[bin.quintile_index], /* ... */ }))
```
This removes the `?? ''` fallback entirely (label is now provably present) and makes a contract drift fail loudly (point dropped) rather than rendering a blank-labelled dot.

### WR-04: `grandTotal` denominator silently diverges from visible cards because the per-card hide gate lives in the child

**File:** `frontend/src/components/charts/EndgameTimePressureSection.tsx:46-51` and `frontend/src/components/charts/EndgameTimePressureCard.tsx:211`
**Issue:** `EndgameTimePressureSection` computes `grandTotal = data.cards.reduce((acc, c) => acc + c.total, 0)` over **all** cards and passes it to every card for the "Games: X% (N)" percentage. The comment asserts "backend pre-filters by `MIN_GAMES_PER_TC_CARD`, so `data.cards.length` is the visible count — do not inspect null-returning children." But `EndgameTimePressureCard` independently re-applies `if (card.total < MIN_GAMES_PER_TC_CARD) return null;` (line 211). These two assumptions are coupled by an undocumented invariant: if the backend ever sends a card with `total < MIN_GAMES_PER_TC_CARD`, the section still adds it into `grandTotal` and renders the grid cell, but the child returns `null` — producing an empty grid slot AND inflated denominators so every visible card's percentage is understated and the percentages no longer sum to ~100%. The defensive guard in the child is correct to keep, but the section must compute the denominator over the same predicate, otherwise the guard and the math contradict each other.
**Fix:** Make the section the single source of truth for visibility and filter before both the grid and the sum:
```tsx
const visibleCards = data.cards.filter((c) => c.total >= MIN_GAMES_PER_TC_CARD);
const visibleCount = visibleCards.length;
const grandTotal = visibleCards.reduce((acc, c) => acc + c.total, 0);
// ...map over visibleCards
```
(Import `MIN_GAMES_PER_TC_CARD` from `@/generated/endgameZones` as the card already does.) Keep the child guard as defense-in-depth.

## Info

### IN-01: Test asserts against an inline re-implementation of the tooltip, not the production component

**File:** `frontend/src/components/charts/__tests__/ScoreGapByTimePressureChart.test.tsx:172-236`
**Issue:** The "tooltip surfaces per-bucket stats" test defines a local `ScoreGapTooltipContent` that re-implements the formatting logic and renders that, instead of exercising the real `ChartTooltip content` function in `ScoreGapByTimePressureChart.tsx:176-203`. The production tooltip wraps values in `tabular-nums`/zone-color markup and uses `point.opp_score` keyed off the Recharts payload; the test's copy could pass forever while the real tooltip regresses (e.g. a sign-format or `opp_score`-null change). This is a tautological test.
**Fix:** Extract the tooltip content into an exported pure function/component (`ScoreGapTooltipContent`) and import it in both the chart and the test, or drive the assertion through the rendered chart with a simulated active Recharts tooltip. At minimum, add a comment flagging that this assertion does not cover the production path.

### IN-02: Magic numbers in chart geometry not extracted to named constants

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:128,162,228,239` (and `131` margins)
**Issue:** `h-44`, `width={44}` (YAxis), `r={2.5}` (dot radius), `width={4}` (ErrorBar cap), and the `margin={{ top: 5, right: 10, left: 0, bottom: 10 }}` literal are inline. CLAUDE.md "No magic numbers" calls for named constants for layout/threshold values. The neutral-band and domain constants are correctly extracted; these visual constants are not. Lower severity because they are presentational and single-use, but a `DOT_RADIUS`/`ERROR_BAR_CAP_WIDTH`/`Y_AXIS_WIDTH` cluster would match the file's own stated convention and the sibling `pressureBulletConfig.ts` pattern.
**Fix:** Hoist to module constants with brief comments, mirroring how `Y_DOMAIN`, `ZONE_OPACITY`, and `MAX_VISIBLE_QUINTILE_INDEX` are already defined in this file.

### IN-03: `(payload.delta as number) ?? 0` is a no-op nullish coalesce after a type assertion

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:222`
**Issue:** `const delta = (payload.delta as number) ?? 0;` — once `payload.delta` is asserted `as number`, the `?? 0` cannot trigger from the type system's view, and if `payload.delta` is actually `undefined` at runtime the assertion has already lied. The intent (default a missing delta to 0 for color) is reasonable but the construct is self-contradictory and will not survive a strict lint pass cleanly.
**Fix:**
```ts
const raw = payload.delta;
const delta = typeof raw === 'number' && Number.isFinite(raw) ? raw : 0;
```

### IN-04: Duplicated `PRESSURE_LABELS` / `makeBin` label tables across modules and tests

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:56-61`, test files (`EndgameTimePressureCard.test.tsx:71`, `ScoreGapByTimePressureChart.test.tsx:74`)
**Issue:** The quintile label strings exist in at least three forms: `PRESSURE_LABELS` (`'0-20% Time'` …), and two test fixtures (`'0-20%'` …). The component comment explicitly justifies redeclaration ("self-contained per RESEARCH.md line 450"), which is an accepted decision, so this is informational only. The risk is the test fixtures use a different label format than production (`'0-20%'` vs `'0-20% Time'`), so a label-format regression in production would not be caught by any fixture-based assertion. Acceptable given the explicit "build independently" guidance; noting for traceability.
**Fix:** No action required if the independence decision stands. If label correctness matters, add one assertion that the rendered x-axis tick text matches `PRESSURE_LABELS` values.

### IN-05: `ChartTooltip content` function exceeds inline-render readability; consider extracting

**File:** `frontend/src/components/charts/ScoreGapByTimePressureChart.tsx:175-204` and dot renderer `213-232`
**Issue:** Two inline render-prop closures (tooltip content, custom dot) carry real logic (null guards, sign formatting, finite checks, zone color). The component is within LOC limits overall, but both closures mix data shaping with JSX inside the declarative tree, which the CLAUDE.md guidance flags as a split seam ("pull large Recharts subtrees / custom tooltips into named sub-components"). Extracting `ScoreGapTooltipContent` and `ScoreGapDot` would also resolve IN-01 (testability) and WR-02 (keying) in one move.
**Fix:** Extract both into named, unit-testable sub-components in the same file.
