# Phase 122 — UI Design Contract

> Visual and interaction contract for the in-app feedback button (SEED-049). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Two surfaces: (1) a **global floating trigger** (bottom-right, every page, auto-hide on scroll, yields to overlays, safe-area aware, mini/secondary weight) and (2) a **submit modal** (required freeform text + optional 3-point sentiment, primary brand-brown submit). Plus bottom scroll padding on long containers.
>
> This contract adapts to FlawChess's REAL design system (theme.ts tokens, shadcn `Button`/`Dialog`, forced dark mode, `tailwindcss-safe-area`). It invents NO parallel palette.

## Spacing Scale

4px grid (Tailwind default). Declared values used by this phase (all multiples of 4):

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px (`gap-1`) | Sentiment-button icon/label gap |
| sm | 8px (`gap-2`, `p-2`) | Sentiment segment internal padding, modal field gap |
| md | 16px (`p-4`, `gap-4`) | Dialog content padding (shadcn `DialogContent` is `p-4`), trigger inset from edge (`bottom-4 right-4` on desktop) |
| lg | 24px | Vertical rhythm between modal header / body / footer (shadcn `gap-4` default is acceptable; do not exceed) |
| xl | 32px | n/a this phase |

**Exceptions (required by D-02):**
- **Floating trigger tap target ≥ 44×44px** even though the visible icon is `size-4` (16px). Use a `size-11` (44px) or larger hit area. shadcn `Button size="icon"` is `size-8` (32px) — too small; this phase **must** use a ≥44px square (`size-11` / `h-11 w-11`) wrapping the icon, NOT the stock `icon` size. This 44px target is the one deliberate spacing exception and exists for touch a11y.
- **Trigger vertical offset on mobile** clears the fixed `MobileBottomBar` (4rem tall, `z-40`): position at `bottom-[4.5rem]` (72px) on mobile so the button sits **above** the bottom bar, `bottom-4` (16px) on `sm:` and up. Combine with safe-area (see Interaction Contract).
- **Bottom scroll padding (D-02):** long scroll containers get `pb-20` (80px) added to clear the floating trigger so the last row never hides under it. On mobile this stacks with the existing `<main className="pb-16">` — apply the extra padding to the scrollable list/content region, not double-applied to `<main>`.

**Accent reserved for:** the modal **Submit** button only (`Button variant="default"`, brand brown). The floating **trigger** is explicitly NOT brand-brown / NOT a high-contrast primary FAB (D-02 mini/secondary). **Sentiment control color semantics (optional, coarse — NOT a verdict palette):** the 3 sentiment segments are **neutral by default and on selection** (use `--toggle-active` for the selected segment, `--secondary` for unselected). ## Sentiment Control — LOCKED Design Decision (resolves D-03 open question)

**Decision: 3-point sentiment**, encoded `Literal["negative", "neutral", "positive"]` (nullable / optional). NOT thumbs 2-state, NOT 1–5 stars.

Rendered as a horizontal **segmented control** of 3 labelled buttons:

| Segment | lucide icon | Visible label | Stored value |
|---------|-------------|---------------|--------------|
| left | `Frown` | "Negative" | `negative` |
| middle | `Meh` | "Suggestion" | `neutral` |
| right | `Smile` | "Positive" | `positive` |

**Rationale (why 3-point over thumbs):**
- A feedback channel collecting *likes / dislikes / suggestions* (SEED-049 title) has a genuine third state: a neutral suggestion that is neither praise nor complaint. Thumbs-up/down forces those into a false binary and depresses the signal. The middle segment is labelled "Suggestion" (not a vague "neutral") to make its meaning unambiguous.
- It still avoids the "what does 3 mean?" star paralysis the seed warns about: each of the 3 points has an explicit icon **and** word label, so there is no numeric scale to interpret.
- It yields a more useful coarse trend over time (the seed's stated goal for the optional rating) than a 2-state thumb.

**Interaction:**
- **Optional.** Default state = nothing selected. The form is submittable with no sentiment chosen (text is the only required field).
- **Single-select, toggle-off allowed.** Tapping the selected segment again clears it back to unselected (so a misclick is recoverable without forcing a choice).
- Selected segment uses `--toggle-active` bg + `--toggle-active-foreground`; unselected segments use `--secondary`. Neutral hues only (no red/green — see Color section).
- Each segment is a semantic `<button type="button">` with `aria-pressed`, wrapped in an `aria-label`'d group.

**Storage note for planner (non-UI, for alignment):** column wide enough for the longest token — `String(16)` per RESEARCH Pattern 2 comfortably fits `"negative"`. Keep the backend `Literal` in lockstep with these three values.

## Interaction Contract — Floating Trigger (resolves D-02)

The trigger's positioning and visibility are the phase's real design risk (seed + RESEARCH Pitfall 4). This contract is prescriptive on **WHEN/HOW it hides** and the **stacking relationship**; the **detection mechanism** (zustand `useOverlayStore` vs radix DOM-presence query) is left to the planner (D-02 discretion, RESEARCH A3) but MUST satisfy this contract.

### Stacking / z-index (grounded in real values)

Observed z-index ladder in the codebase:

| Layer | z-index | Source |
|-------|---------|--------|
| Dialog / Sheet / Drawer overlay + content | `z-50` | `ui/dialog.tsx:28,50`, `ui/drawer.tsx:43,62` |
| `MobileBottomBar` (mobile only) | `z-40` | `App.tsx:260` |
| iOS install banner | `z-30` | `InstallPromptBanner.tsx:50` (sits at `bottom-16`) |
| **Feedback trigger (this phase)** | **`z-20`** | NEW — must sit BELOW all of the above |

**Contract:** the trigger lives at **`z-20`** — below the mobile bottom bar (`z-40`), below the install banner (`z-30`), and far below any overlay (`z-50`). Combined with the yield rule below, this guarantees the button can never paint on top of a drawer, modal, bottom sheet, the bottom nav, or the install banner.

### Position

- Desktop (`sm:` and up): `fixed bottom-4 right-4`.
- Mobile (`< sm`): `fixed bottom-[4.5rem] right-4` so it clears the 4rem `MobileBottomBar`. Combine the inset with safe-area: add `pb-safe`-equivalent bottom inset so it clears the iOS home indicator in installed PWA mode. **Use the `tailwindcss-safe-area` plugin** (`pb-safe` / safe-area utilities already imported in `index.css`) — do NOT hand-roll `style={{ paddingBottom: 'env(safe-area-inset-bottom)' }}` (RESEARCH "Don't Hand-Roll"). If the inset must be composed with the `bottom-[4.5rem]` offset, the planner may use a `calc(... + env(safe-area-inset-bottom))`, but prefer the plugin utility where it expresses the same thing.

### Visibility rules (the button is HIDDEN when ANY of these is true)

1. **Scrolling down** — hide on scroll-down, show on scroll-up (D-02). Show whenever scroll direction is up or the page is at the top. A new `useScrollDirection` hook supplies this (no existing analog; RESEARCH Pattern 9).
2. **An overlay is open** — when ANY drawer / modal / bottom sheet is open (the `z-50` layer), the trigger hides entirely. It never "sits behind, peeking out"; it is fully hidden so it can't fight the overlay for the bottom-right corner. This includes the filter/bookmark drawers on `/openings`, the `MobileMoreDrawer`, the install drawer, and any `Dialog`.
   - The button's OWN feedback modal opening also hides the trigger — harmless and expected (RESEARCH Pattern 9).
3. (Implied) When the trigger's own modal is open, keep it hidden until close.

**Transition:** fade/slide the trigger in/out (e.g. `transition-all`, opacity + small translate-y) rather than hard-popping, so hide-on-scroll feels intentional. Keep it short (≤200ms).

**Detection (planner's choice, must meet the contract):** RESEARCH recommends either (a) a tiny global zustand `useOverlayStore` that overlays increment/decrement on open/close (explicit, requires wiring each drawer), or (b) a radix DOM-presence check (`document.querySelector('[role="dialog"], [data-slot="dialog-content"], [data-slot="drawer-content"]')`) (no per-drawer wiring, more brittle). Either is acceptable; the yield behavior above is the binding requirement, not the mechanism.

### Bottom scroll padding (D-02)

Long scrollable containers add `pb-20` (80px) so their last row clears the floating trigger. Apply to the scroll region of dense pages (e.g. the openings move list, library game list, endgame tables). Verify against the actual long lists at plan/impl time; do not blanket-apply to `<main>` (it already carries `pb-16` on mobile for the bottom bar).

## Copywriting Contract

| Element | Copy |
|---------|------|
| Trigger `aria-label` | `Send feedback` |
| Modal title | `Send feedback` |
| Modal description | `What's working, what's not, or what you'd like to see. We read every note.` |
| Text field label | `Your feedback` |
| Text field placeholder | `Tell us what you think about this page…` |
| Sentiment group label | `How do you feel? (optional)` |
| Sentiment options | `Negative` · `Suggestion` · `Positive` |
| Primary CTA (submit) | `Send feedback` |
| Secondary action | `Cancel` |
| Disabled-submit reason (a11y / title) | Submit stays disabled until the text field is non-empty (after trimming whitespace). No inline error needed for empty — the disabled state communicates it. |
| Validation error (if user submits over the max length, server 422) | `Feedback is too long. Please shorten it and try again.` |
| Submit error (network / 500) | `Couldn't send your feedback. Something went wrong. Please try again in a moment.` |
| Rate-limit error (429, D-07) | `You've sent a lot of feedback recently. Please try again later.` |
| Success | Toast (sonner): `Thanks for the feedback!` — then close the modal. |
| Empty state | **N/A** — this is an action surface, not a data view; there is no empty/list state in scope (read UI is deferred). |
| Destructive action | **None.** This phase has no destructive action. Cancel/close simply dismisses; no confirmation dialog. |

Copy style (CLAUDE.md): no em-dashes (use commas/periods/ellipsis), no sycophancy, concise. The success toast is the one light, warm note and is fine.

### Validation / submit-enable behavior (D-03)

- **Required text:** Submit is `disabled` while `text.trim().length === 0`. Enables as soon as non-whitespace text exists.
- **Max length (D-07):** soft client cap mirrors the server `_MAX_FEEDBACK_LEN` (a named constant on the backend, RESEARCH Pattern 3 suggests 2000). If shown, a character counter is `text-sm` (not `text-xs`). The hard guard is server-side Pydantic `max_length` → 422; the client copy above handles that response.
- **Optional sentiment:** never blocks submit.
