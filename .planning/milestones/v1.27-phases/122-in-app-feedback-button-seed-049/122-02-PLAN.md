<objective>
Build the frontend vertical slice for in-app feedback: a global floating trigger button mounted
in `ProtectedLayout`, the two genuinely-new hooks it needs (`useScrollDirection` for
hide-on-scroll-down and `useOverlayOpen` for yield-to-overlay), and a submit modal (required
freeform text + optional 3-point sentiment) wired to `POST /api/feedback` via a TanStack mutation
that captures the page URL from the router at submit time.

Purpose: deliver the discoverable, non-intrusive submission UI (D-01/D-02) and the form (D-03)
under project conventions (D-06), mirroring the request contract from Plan 01. The two no-analog
pieces (scroll-direction + overlay-yield) are the de-risk targets.

Output: `useScrollDirection`, `useOverlayOpen`, `useFeedback`, `feedbackApi`, `FeedbackButton`,
`FeedbackModal`, types; `FeedbackButton` mounted in `App.tsx`; bottom scroll padding on long
containers; frontend tests.

    Re-evaluate on a `MutationObserver` watching `document.body` childList/subtree (overlays portal
    into the body), so the signal updates when any radix Dialog/Drawer mounts or unmounts.
    Disconnect the observer on unmount. Explicit return type `(): boolean`. Note: the button's OWN
    modal opening also flips this true — harmless (the trigger hiding while its modal is open is
    expected, UI-SPEC).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → feedback text/sentiment | Untrusted text entered in the modal; sent to the backend (validated server-side in Plan 01) |
| client → POST /api/feedback | Mutation sends body over the existing authenticated apiClient (JWT) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-122-F1 | Tampering | feedback text rendered in the modal | accept (low) | React default-escapes text in JSX; the textarea value is never injected as HTML; no `dangerouslySetInnerHTML` anywhere in the feedback components |
| T-122-F2 | Spoofing | page_url / user attribution from the client | mitigate | `page_url` is captured from `useLocation()` (router state), not user-typed; `user_id` is NOT sent by the client — the backend derives it from the JWT (Plan 01 T-122-01) |
| T-122-F3 | Information Disclosure | double Sentry capture leaking duplicate events | mitigate | no `Sentry.captureException` in the modal; the global `MutationCache.onError` is the single capture point (Pitfall 1 / CLAUDE.md) |
| T-122-SC | Tampering | npm installs | accept | no packages installed this phase (all libs in lockfile — RESEARCH Package Legitimacy Audit: N/A) |
</threat_model>
