## Tests

### 1. End-to-end big-first-import latency reduction
expected: For a >= 300-game first import with an entry-capable worker running, `evals_completed_at` populates faster than server-pool-only, and `entry_eval_leased_by` shows distinct worker IDs in the DB.
result: [passed] — Verified on a real 5,132-game first import (user 28, 2026-06-16: 4,355 chess.com + 777 lichess). Entry-ply Stockfish drain split server-pool (2,573 games) / worker `ws80` (1,800 games) — ws80 carried ~41% of the load with identical 3.7 evals/game density, confirming the fan-out and that the server derives the same eval targets regardless of who drains. User reported the drain finished considerably faster than server-pool-only. `entry_eval_leased_by` shows distinct IDs (`server-pool`, `ws80`). 100% `evals_completed_at` (5,132/5,132), zero stuck or expired-unstamped leases. CR-01 livelock fix verified live: 9 zero-target leased games (2 server-pool + 7 ws80) stamped complete instead of re-leased forever.

### 2. Mixed-fleet backward compat
expected: An un-upgraded worker (no scope param, no X-Worker-Id header) still drains full-ply, never touches entry-ply, and `leased_by` falls back to "remote-worker".
result: [passed] — Prod `eval_jobs` shows 26 completed full-ply jobs under the `remote-worker` fallback `leased_by` (un-upgraded worker, no `--worker-id`), alongside 43 server-pool. Crucially, zero `remote-worker` values appear in the entry-ply `games.entry_eval_leased_by` column, confirming the un-upgraded worker drained full-ply only and never touched entry-ply. (Evidence from prior prod observation 2026-06-15, not a controlled A/B in the 2026-06-16 ws80 run.)
