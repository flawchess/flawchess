**Composite-PK + column-types pattern** — combine the `opening.py` `__table_args__`

- `metric` → reuse the `benchmark_metric` Postgres ENUM (see below) or `String`;
  the `CdfMetricId` value set is the same one stored in `user_benchmark_percentiles.metric`.
- `anchor_elo` → `SmallInteger` (800..2600 grid, fits i16).
- `tc` → `time_control_bucket_enum`.
- `percentile` → `SmallInteger` (1..99 per `BREAKPOINT_PERCENTILES`).
- `value` → `Double` / `double precision` (0-1 score-gap float, matches `CdfTable.breakpoints` element type).
- `n_users` → `Integer` (audit; nullable is planner's call per Discretion).
- `snapshot_month` → `String` (e.g. "2026-05"; plain audit column, NOT idempotency-relevant per notes #1; nullable is planner's call).

**`metric` column note:** `user_benchmark_percentiles` already persists the same
`CdfMetricId` family strings against a Postgres `benchmark_metric` ENUM (see the
migration `52c928794fe7` which added the bare rate family names
`conversion_rate`/`parity_rate`/`recovery_rate`). The planner should decide
whether `benchmark_cohort_cdf.metric` reuses that ENUM (consistency, but couples
to its value list) or a plain `String` (simpler for an analysis table). Either is
defensible; flag the choice. The 11 `CdfMetricId` values live in
`global_percentile_cdf.py` lines 101-115.

**Model registration:** confirm the new model is imported wherever `Base.metadata`
is assembled for Alembic autogenerate (the same place `Opening` / `UserRatingAnchor`

### `app/repositories/benchmark_cohort_cdf_repository.py` (CREATE — repository)

**Analog:** `app/repositories/openings_repository.py::query_openings_by_hashes`
(lines 761-787) — the cleanest "batch `IN` query → grouped dict" pattern in the
repo. The new loader is the **only** async DB access for this phase (D-04).

**Signature (D-04 / D-03):**
```python
async def load_cohort_cells(
    session: AsyncSession,
    anchors: Sequence[int],
    tcs: Sequence[TimeControlBucket],
) -> dict[tuple[CdfMetricId, int, TimeControlBucket], CdfTable]:
```

**Batch-IN + group-into-dict pattern to mirror** (`openings_repository.py` lines 761-787):
```python
async def query_openings_by_hashes(session, full_hashes):
    if not full_hashes:
        return {}
    stmt = select(Opening).where(
        Opening.full_hash.is_not(None),
        Opening.full_hash.in_(full_hashes),
    )
    rows = await session.execute(stmt)
    by_hash: dict[int, Opening] = {}
    for opening in rows.scalars():
        ...
        by_hash[opening.full_hash] = opening
    return by_hash
```

**Loader-specific construction notes:**
- Early-return `{}` on empty `anchors`/`tcs` (mirrors the `if not full_hashes` guard).
- One `SELECT ... WHERE anchor_elo = ANY(:anchors) AND tc = ANY(:tcs) ORDER BY
  metric, anchor_elo, tc, percentile` (Discretion: exact `IN`/`= ANY` SQL is the
  planner's call). The `ORDER BY percentile` is **load-bearing**: `CdfTable.breakpoints`
  is a tuple indexed parallel to `BREAKPOINT_PERCENTILES` (p1..p99), so rows must
  be assembled in percentile order to reconstruct a valid table.
- Group rows by `(metric, anchor_elo, tc)`, collect 99 `value`s into a tuple, then
  rebuild `CdfTable(breakpoints=tuple(values), n_users=..., snapshot_month=...)`.
  `n_users`/`snapshot_month` are identical across a cell's 99 rows — take from any
  (e.g. first) row.
- **No SQL in the service** (CLAUDE.md "repositories/ DB access; no SQL in
  services"). The grouping/`CdfTable` rebuild lives here, not in
  `user_benchmark_percentiles_service.py`.
- Import `CdfTable`, `CdfMetricId`, `TimeControlBucket` from
  `app.services.global_percentile_cdf` (the module stays the type source — see
  notes "The type `CdfMetricId` is imported widely; only the data moves").

**Apply to:** the new model's `tc` column and the migration (reference, don't

**Source:** migration `52c928794fe7` (added bare `CdfMetricId` family names) +
`user_benchmark_percentiles.metric`.
**Apply to:** the new model's `metric` column — planner picks ENUM reuse
(consistency) vs `String` (decoupled analysis table). Flag the trade-off; not

### Idempotent seed via `ON CONFLICT DO UPDATE`
**Source:** `scripts/seed_openings.py` lines 86-104.
**Apply to:** `seed_cohort_cdf.py`. This is the **sole** idempotency mechanism
(notes #3) — no hash gate, no `snapshot_month` gate.

**Source:** `user_benchmark_percentiles_service.py` per-cell try/except + top-level
handler (lines 437-459, 519-539), CLAUDE.md Backend Rules.
**Apply to:** preserved as-is when wiring the prefetch — variables go via
`set_context`, never in the message; `CancelledError` always re-raises.

### Deterministic regen ordering
**Source:** `gen_global_percentile_cdf.py` `_emit_registry_block` sort keys (lines
534-553) + `_build_cohort_cdf_for` `(|dist|, user_id)` tiebreak (lines 480-482).
**Apply to:** the new TSV emitter — same iteration/sort order so regen is
byte-identical and the drift test is meaningful.
