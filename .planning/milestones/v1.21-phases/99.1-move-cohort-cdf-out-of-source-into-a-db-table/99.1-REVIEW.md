# Phase 99.1: Code Review Report

**Reviewed:** 2026-05-31
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Phase 99.1 relocates the ~130k-line in-source `COHORT_PERCENTILE_CDF` literal
into a `benchmark_cohort_cdf` Postgres table, leaving `global_percentile_cdf.py`
as a pure/sync interpolation module. I reviewed the locked-design invariants
specifically: SQL parameterization, the batched-prefetch query, ON CONFLICT
idempotency, and the shrunk module's interpolation purity.

**No correctness or security BLOCKERS were found.** The relocation is sound:

- **SQL injection:** clean. The seed script binds every column via named
  parameters (`text(_INSERT_SQL), params`), the repository uses the ORM
  `select()` API with `.in_()` (no string interpolation), and the service's
  raw `text()` SQL binds `user_id` via `.bindparams()` exactly as before.
- **Batched prefetch:** correct. `ORDER BY metric, anchor_elo, tc, percentile`
  feeds positional `CdfTable.breakpoints` reconstruction; the composite PK
  guarantees a complete cell is exactly p1..p99 once each, so the 99-row
  positional assembly cannot mis-map.
- **Idempotency:** the `ON CONFLICT (...) DO UPDATE` upsert is genuinely
  re-runnable; the TSV always populates `n_users`/`snapshot_month` so the
  empty-string coalesce paths are dead but harmless.
- **Purity:** `interpolate_cohort_percentile(value, table)` and helpers are
  side-effect-free, sync, and import nothing from the DB layer. The repository
  is the sole async DB access point per D-04. The `None`-table suppression and
  NaN/tail-clamp logic are preserved.

Findings below are 3 WARNINGs (maintainability/robustness) and 6 INFO (doc
drift and minor consistency). Several doc-drift items matter because the module
docstrings are the stated source of truth for future recalibration.

## Warnings

### WR-01: Misleading `inserted` counter — counts upserts, not inserts

**File:** `scripts/seed_cohort_cdf.py:76`, `90-92`, `97-101`
**Issue:** The variable is named `inserted` and incremented when
`result.rowcount > 0`, but with `INSERT ... ON CONFLICT DO UPDATE` PostgreSQL
returns `rowcount == 1` for *both* a fresh insert and a conflict-update. On a
re-run against an already-seeded table, every row reports `rowcount == 1`, so
the script logs `"123552 upserted"` even though zero rows are new. The log text
says "upserted" (correct) but the variable name `inserted` and the return value
(`return inserted`) imply insert semantics. Any caller or operator who reads
"123552 inserted" on a second run will be misled into thinking the table was
empty. There is no way, as written, to distinguish a first seed from a re-seed.
**Fix:** Rename for accuracy and (optionally) split the counts:
```python
upserted = 0
# ...
result = await session.execute(text(_INSERT_SQL), params)
if result.rowcount and result.rowcount > 0:
    upserted += 1
# ...
logger.info("Seed complete: %d rows upserted, %d errors", upserted, errors)
return upserted
```
If first-vs-re-seed disambiguation is wanted, add `RETURNING (xmax = 0) AS inserted`
to the statement and count true/false separately. At minimum, rename the
variable to `upserted` to match reality.

### WR-02: Per-row execute in a single unbounded transaction (~123k round-trips, OOM-adjacent)

**File:** `scripts/seed_cohort_cdf.py:79-94`
**Issue:** The seed issues one `await session.execute(...)` per TSV row inside
a single transaction that only commits after all ~123,552 rows. This is a
correctness/robustness concern, not just performance: (1) a single long
transaction holding 123k pending writes is exactly the class of operation that
triggered the documented Postgres OOM-kills on this project (CLAUDE.md notes
multiple FLAWCHESS-3Q recurrences from large single operations), and the
production runbook explicitly runs this *inside the backend container*; (2) if
the process dies at row 100k, the entire transaction rolls back with no
progress and no partial-seed signal. The `errors` counter accumulates parse
failures but the run still commits everything-or-nothing.
**Fix:** Use `executemany` (single statement, list of param dicts) and/or
periodic intermediate commits in batches, mirroring the chunked approach used
elsewhere on this project:
```python
BATCH_SIZE = 5000
batch: list[dict[str, object]] = []
for row_num, row in enumerate(reader, start=2):
    ...
    batch.append(params)
    if len(batch) >= BATCH_SIZE:
        await session.execute(text(_INSERT_SQL), batch)  # executemany
        await session.commit()
        batch.clear()
if batch:
    await session.execute(text(_INSERT_SQL), batch)
await session.commit()
```
`session.execute(text(sql), list_of_dicts)` runs as a single multi-row
prepared statement, collapsing 123k round-trips to ~25 and bounding the open
transaction size.

### WR-03: Missing secondary index — query filters on non-leading PK columns

**File:** `app/models/benchmark_cohort_cdf.py:43-62`, `app/repositories/benchmark_cohort_cdf_repository.py:69-81`
**Issue:** The composite PK is `(metric, anchor_elo, tc, percentile)`, with
`metric` as the leading column. `load_cohort_cells` filters on
`anchor_elo IN (...) AND tc IN (...)` with no `metric` predicate. Because the
PK index is prefixed by `metric`, PostgreSQL cannot use it to satisfy this
filter and must fall back to a full sequential scan of a ~123k-row table on
every import (Stage A and Stage B each issue this query). On a pure-relocation
refactor the prior in-source dict lookup was O(1) in-memory; the DB replacement
silently introduces a seq-scan per import. (Pure runtime cost is out of v1
scope, but the *missing index* is a schema-design defect under CLAUDE.md
"Database Design Rules," and it is a robustness regression vs. the relocated
behavior.)
**Fix:** Add a composite index on the actual filter columns in the model and a
matching migration op:
```python
__table_args__ = (
    Index("ix_benchmark_cohort_cdf_anchor_tc", "anchor_elo", "tc"),
)
```
```python
# in upgrade()
op.create_index(
    "ix_benchmark_cohort_cdf_anchor_tc",
    "benchmark_cohort_cdf",
    ["anchor_elo", "tc"],
)
# and drop it in downgrade()
```

## Info

### IN-01: Model docstring grid math is wrong (181 vs 36 anchors)

**File:** `app/models/benchmark_cohort_cdf.py:18-19`
**Issue:** The docstring states "≤ 181 anchor_elo grid points" and "~130k rows".
The committed TSV uses a 50-Elo grid over 800..2600, yielding 36 anchor points
(`(2600-800)/50 + 1 = 37`, of which 36 survive the K-floor; min is 850). 181
corresponds to a 10-Elo grid that does not exist. The row count is plausible
(11 × 36 × 4 × 99 ≈ 156k upper bound, ~123k after suppression), but the
181-anchor claim is simply incorrect and will mislead anyone sizing the table.
**Fix:** Change to "≤ 37 anchor_elo grid points (800..2600 on a 50-Elo grid;
see `COHORT_ANCHOR_STEP_ELO`)".

### IN-02: `_build_cohort_cdf_for` docstring references `numpy.percentile`, code uses pure Python

**File:** `scripts/gen_global_percentile_cdf.py:458`
**Issue:** The docstring step 4 says "Compute 99-breakpoint CDF via
`numpy.percentile`", but the implementation uses `_percentile_linear`
(pure-Python, since numpy is intentionally not a dependency — stated in the
same file's other docstring at line 405). Reader-misleading doc drift.
**Fix:** "Compute 99-breakpoint CDF via `_percentile_linear` (numpy-free,
matches numpy's `method='linear'`)".

### IN-03: Stale "8 to 8" / "16 to 8" comments after collapse to 11

**File:** `app/services/global_percentile_cdf.py:89`, `user_benchmark_percentiles_service.py:143-145`
**Issue:** The `CdfMetricId` comment says "collapsed from 16 to 8 (CONTEXT
D-13)" and "each metric has at most 4 cells per anchor", and the service says
"the 8-value CdfMetricId Literal lifts TC out of the metric name", but the
Literal now has 11 values (Phase 99 appended 3 rate families). The module-level
docstring at the top (lines 21, 89) already says "16 → 11", so the file
contradicts itself.
**Fix:** Update the inline comments to "collapsed 16 → 11" and "11-value
CdfMetricId Literal" for internal consistency.

### IN-04: Seed script uses legacy `sqlalchemy.orm.sessionmaker` instead of `async_sessionmaker`

**File:** `scripts/seed_cohort_cdf.py:31`, `73-74`
**Issue:** Every other async script in the repo (`gen_global_percentile_cdf.py`,
`backfill_eval.py`, `gen_benchmarks.py`, `select_benchmark_users.py`) uses
`sqlalchemy.ext.asyncio.async_sessionmaker`. This new script uses the legacy
`sqlalchemy.orm.sessionmaker(engine, class_=AsyncSession)` form. It works (and
mirrors the older `seed_openings.py`), but it is the deprecated 1.x-style
factory and is inconsistent with the sibling generator script written in the
same phase.
**Fix:** `from sqlalchemy.ext.asyncio import async_sessionmaker` and
`async_session = async_sessionmaker(engine, expire_on_commit=False)`.

### IN-05: `run_local.sh` count-probe masks all psql failures as "empty table"

**File:** `bin/run_local.sh:47-51`
**Issue:** `COHORT_CDF_COUNT=$(... psql ... 2>/dev/null || echo "0")` forces a
re-seed whenever `psql` is absent or the query errors for any reason other than
an empty table. The seed is idempotent so the consequence is only a spurious
(slow) re-seed, and this exactly mirrors the pre-existing openings probe two
lines above, so it is consistent rather than newly broken. Noting for
completeness since the per-row seed (WR-02) makes a spurious re-run more costly.
**Fix:** Optional — distinguish "missing psql / connection error" from "table
empty" before deciding to seed, or accept the established pattern.

### IN-06: `n_users` NULL-coalesce to 0 silently discards audit signal

**File:** `app/repositories/benchmark_cohort_cdf_repository.py:124-127`
**Issue:** `n_users: int = n_users_raw or 0` coalesces a NULL `n_users` to 0.
Since `CdfTable.n_users` is audit-only and does not affect interpolation, parity
is unaffected (as the comment states). But `or 0` also collapses a legitimate
stored `0` to `0` (harmless) and, more subtly, makes "unknown cohort size"
indistinguishable from "zero-user cohort". The TSV never emits NULL/0 today, so
this is latent. Prefer an explicit None-check if the audit field is ever
surfaced to operators.
**Fix:** `n_users: int = n_users_raw if n_users_raw is not None else 0` (avoids
the falsy-0 conflation) — or leave as-is given the field is audit-only.
