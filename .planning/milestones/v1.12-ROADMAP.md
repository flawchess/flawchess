## Overview

Ship the infrastructure half of SEED-002: a separate `flawchess-benchmark` PostgreSQL instance, a third read-only MCP server, an Alembic migration extending the canonical `games` table with eval-source metadata, and a resumable Lichess monthly-dump ingestion pipeline that stratifies subsampling on (rating × time_control). Pipeline correctness verified by smoke test.

The applied-analytics half of SEED-002 — classifier validation at scale, rating-stratified offset analysis, Parity validation, `/benchmarks` skill upgrade, and rating-bucketed zone recalibration — is deferred to a future milestone. See **Deferred Work** below.

### Why the milestone was scoped down (2026-04-26)

The original v1.12 plan bundled five phases (69-73): infrastructure (69) plus four applied-analytics phases (70-73). The actual operational step of populating the benchmark DB by running the ingestion pipeline against a Lichess monthly dump takes days of wall-clock time on Adrian's machine. Treating that ingest as a milestone gate blocked all other GSD work for its duration — including v1.13 opening insights, which has zero structural dependency on the populated benchmark DB.

The split:
- **v1.12 (this milestone)** — Phase 69 only. Pipeline correctness shipped, smoke-test verified. Population is operational, not a milestone gate.
- **SEED-006 (future milestone)** — Phases 70-73 reframed. Trigger: full benchmark ingest complete.
- **v1.13 (next milestone)** — Opening insights (SEED-005). Independent of either of the above.

## Phases

### Phase 69: Benchmark DB Infrastructure & Ingestion Pipeline

**Goal:** A separate `flawchess-benchmark` PostgreSQL instance, queryable via a third MCP server, populated by a resumable Lichess-monthly-dump ingestion script that uses the existing `game_positions.eval_cp` / `eval_mate` columns (plus newly added `games.eval_depth` / `games.eval_source_version` columns from the INGEST-06 migration) and stratifies subsampling on (rating_bucket × time_control) at the player-opportunity level.

**Depends on:** Nothing (parallel-isolated from prod)
**Requirements:** INFRA-01, INFRA-02, INFRA-03, INGEST-01, INGEST-02, INGEST-03, INGEST-04, INGEST-05, INGEST-06
**Plans:** 6 (69-01..69-06; see `.planning/phases/69-benchmark-db-infrastructure-ingestion-pipeline/`)

**Success Criteria** (what must be TRUE):
1. `docker compose -f docker-compose.benchmark.yml up -d` starts a `flawchess-benchmark` PostgreSQL 18 instance running the SAME Alembic migrations as dev/prod/test, with no benchmark-only schema additions (Lichess `[%eval` populates the existing `game_positions.eval_cp` and `eval_mate` columns).
2. The third MCP server `flawchess-benchmark-db` (read-only) is registered and documented in `CLAUDE.md` Database Access section, and a quick MCP query returns row counts from `games` and `game_positions`.
3. Interrupting the ingest with SIGINT mid-dump and re-running it from the same checkpoint resumes without duplicating rows and without redoing parsed games (verified by row-count parity vs an uninterrupted run on a small dump).
4. An Alembic migration adds `eval_depth` (SmallInteger, nullable) and `eval_source_version` (String, nullable) columns to the canonical `games` table (applied to dev/prod/test/benchmark uniformly), the import pipeline populates them from Lichess API metadata when present, and a one-off centipawn convention verification note (or test) documents that signed-from-white POV centipawns vs pawn-units agrees with a known sample.

**Note on dropped criteria (2026-04-26):** the original criterion 3 ("≥1k queen-endgame samples per cell at full storage") and criterion 5's storage-target tail ("50-100 GB at end of phase") were dropped in the scope-down. Criterion 3 was already obsolete after the per-cell sample-unit pivot (D-12 / sample unit = distinct players per cell, not games per endgame type). The storage figure is operational guidance for the actual ingest run, not a Phase 69 gate. Future-population sample-size and storage targets, if needed, become entry criteria for the SEED-006 phases that consume the populated DB.
