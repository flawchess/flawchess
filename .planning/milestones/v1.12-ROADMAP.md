# Milestone v1.12: Benchmark DB & Population Baselines

**Status:** Active (planning)
**Phases:** 69-73
**Source:** `.planning/seeds/SEED-002-benchmark-db-population-baselines.md`
**Requirements:** `.planning/REQUIREMENTS.md` (17 requirements: 3 INFRA, 6 INGEST, 4 VALID, 4 BENCH)

## Overview

Replace self-referential endgame baselines with a Lichess-derived population dataset stratified by rating x time control, validate the material-vs-eval classifier at 10-100x larger scale than the 2026-04-07 report, and recalibrate the Conversion / Recovery / Endgame Skill gauge zones per rating bucket. Self-referential analysis remains the primary UI signal; population data is an overlay used to (a) calibrate zone constants and (b) layer absolute framing onto the existing per-user comparisons.

The milestone follows the SEED-002 A-E decomposition mapped onto Phase numbers 69-73. Phase 70 is a hard quantitative gate: if the benchmark-scale classifier validation surfaces material differences from the 2026-04-07 small-sample estimates on Pawn / Rook / Minor cells, Phases 71-73 pause until the offset shift is investigated and the rating-stratified analysis is re-grounded against a confirmed offset.

## Phase Decomposition Notes (departures from SEED-002)

The seed prescribes Phases A-E as a clean 1:1 mapping. We adopt that mapping unchanged at the phase level (69 -> A, 70 -> B, 71 -> C, 72 -> D, 73 -> E). Specifically:

- **Phase 69 NOT split into 69 + 69.1.** Phase 69 covers BOTH the docker-compose / MCP-server scaffolding (INFRA-01..03) AND the ingestion pipeline (INGEST-01..06). They are tightly coupled: the eval pre-filter (INGEST-01) presupposes the schema decision (INFRA-02), the resumability checkpoint (INGEST-04) lives in the benchmark DB itself, and the eval-metadata schema migration + centipawn-convention check (INGEST-06) is the natural smoke test for the deployed instance. Splitting would force a fake handoff. If `/gsd-discuss-phase 69` surfaces a 1-2 week effort delta between scaffolding and pipeline that warrants splitting, insert Phase 69.1 mid-milestone via `/gsd-insert-phase` rather than pre-committing to it now.
- **Phase 70 success criterion 4 is the documented atomic gate decision.** Per the seed Notes section, Phase B is a checkpoint. We make this concrete: the validation report MUST end with a "Gate Verdict" section that states PASS or FAIL on each of Pawn / Rook / Minor cells and an overall verdict authorizing or pausing Phases 71-73. This is part of the phase, not a separate process step.
- **Phase 73 includes shipping zone constants in code.** Per BENCH-03, the milestone is INCOMPLETE until rating-bucketed zone thresholds land in `frontend/src/lib/theme.ts` (or the equivalent constants location). Emitting a report from `/benchmarks` is necessary but not sufficient. If the rating-bucketed thresholds need larger UI plumbing than a constants edit (e.g., user-rating-aware lookup at render time), `/gsd-discuss-phase 73` should split that out into Phase 73.1 rather than expanding Phase 73 mid-flight.

## Phases

### Phase 69: Benchmark DB Infrastructure & Ingestion Pipeline

**Goal:** A separate `flawchess-benchmark` PostgreSQL instance, queryable via a third MCP server, populated by a resumable Lichess-monthly-dump ingestion script that uses the existing `game_positions.eval_cp` / `eval_mate` columns (plus newly added `games.eval_depth` / `games.eval_source_version` columns from the INGEST-06 migration) and stratifies subsampling on (rating_bucket x time_control) at the player-opportunity level.

**Depends on:** Nothing (parallel-isolated from prod)
**Requirements:** INFRA-01, INFRA-02, INFRA-03, INGEST-01, INGEST-02, INGEST-03, INGEST-04, INGEST-05, INGEST-06
**Plans:** TBD

**Success Criteria** (what must be TRUE):
1. `docker compose -f docker-compose.benchmark.yml up -d` starts a `flawchess-benchmark` PostgreSQL 18 instance running the SAME Alembic migrations as dev/prod/test, with no benchmark-only schema additions (Lichess `[%eval` populates the existing `game_positions.eval_cp` and `eval_mate` columns)
2. The third MCP server `flawchess-benchmark-db` (read-only) is registered and documented in `CLAUDE.md` Database Access section, and a quick MCP query returns row counts from `games` and `game_positions`
3. Running the ingestion script against a single Lichess monthly dump completes a stratified subsample where every (rating_bucket x time_control) cell of the 5x4 = 20 grid contains enough games that the rarest endgame type (queen, ~2%) clears the 10-game min-sample threshold by a comfortable margin (target: >= 1k queen-endgame samples per cell at full storage), with each game's per-side stats independently bucketed by `WhiteElo` and `BlackElo`
4. Interrupting the ingest with SIGINT mid-dump and re-running it from the same checkpoint resumes without duplicating rows and without redoing parsed games (verified by row-count parity vs an uninterrupted run on a small dump)
5. An Alembic migration adds `eval_depth` (SmallInteger, nullable) and `eval_source_version` (String, nullable) columns to the canonical `games` table (applied to dev/prod/test/benchmark uniformly), the import pipeline populates them from Lichess API metadata when present, a one-off centipawn convention verification note (or test) documents that signed-from-white POV centipawns vs pawn-units agrees with a known sample, and total benchmark DB storage at end of phase falls within the 50-100 GB target

### Phase 70: Classifier Validation Replication at Scale (GATE)

**Goal:** Re-run the 2026-04-07 material-vs-eval validation against the benchmark DB with 10-100x more samples per (endgame_type) cell, document the offset estimates and confidence intervals, and emit a documented PASS/FAIL gate verdict that authorizes or pauses Phases 71-73.

**Depends on:** Phase 69
**Requirements:** VALID-01, VALID-02
**Plans:** TBD

**Success Criteria** (what must be TRUE):
1. A report exists at `reports/classifier-validation-benchmark-YYYY-MM-DD.md` whose section structure parallels `reports/endgame-conversion-recovery-analysis.md` (sections 1-4: threshold comparison, persistence comparison, eval validation head-to-head, decision), so side-by-side comparison is trivial
2. The report contains a per-(endgame_type) offset table comparing the new benchmark-DB point estimates against the 2026-04-07 estimates AND against the 2026-04-07 95% CI bounds, for both Conversion and Recovery, on the t=100 + 4-ply configuration
3. The report ends with an explicit `## Gate Verdict` section that states PASS or FAIL on each of Pawn / Rook / Minor cells using the locked rule: FAIL only if (a) the new point estimate falls outside the 2026-04-07 95% CI for that cell AND (b) |delta| > 2pp absolute
4. The Gate Verdict section concludes with a single overall verdict line: either "Phases 71-73 authorized to proceed" or "Phases 71-73 paused for offset re-investigation". The verdict is persisted in `.planning/STATE.md` and PR-reviewed before downstream phases start
5. Sample size in the benchmark replication for Pawn / Rook / Minor / Mixed cells is at least 10x the 2026-04-07 cell sizes (Pawn: ~1k -> >= 10k; Rook: ~3k -> >= 30k; Minor: ~2.5k -> >= 25k; Mixed: ~28k -> >= 280k); Queen sample reaches at least 1k (vs 28-40 in the original report)

### Phase 71: Rating-Stratified Offset Analysis

**Goal:** Extend the classifier validation with the rating-bucket breakdown the 2026-04-07 report lacked: produce per-(rating_bucket x endgame_type) material-vs-eval offset tables for both Conversion and Recovery, and surface whether the t=100 + 4-ply proxy needs a rating-dependent correction.

**Depends on:** Phase 70 (gate-pass)
**Requirements:** VALID-03
**Plans:** TBD

**Success Criteria** (what must be TRUE):
1. A report extension (either appended to the Phase 70 report or written as `reports/classifier-validation-rating-stratified-YYYY-MM-DD.md`) contains a 5x6 offset table (5 rating buckets x 6 endgame types) for both Conversion and Recovery, populated from the benchmark DB
2. Each cell reports point estimate AND 95% CI, with a flag where CI width >= 5pp (sample-quality marker for the eventual baseline use)
3. The report concludes with an explicit verdict on the hypothesis "the t=100 + 4-ply offset varies materially by rating": either (a) "no rating-dependent correction needed; offsets are within +/- 2pp across all rating buckets for at least Pawn / Rook / Minor", or (b) "rating-dependent correction recommended for ", with the affected cells named
4. If verdict (b) fires, the report names the concrete remediation paths (rating-bucketed offset adjustment in zone calibration vs deferring to v1.13 hybrid classifier) without committing to one in this phase

### Phase 72: Parity Proxy Validation

**Goal:** Validate the Parity proxy ("even material at endgame entry" -> Stockfish eval in [-50, +50]cp at endgame entry) at benchmark-DB scale, document any systematic offset analogous to the Conversion / Recovery offsets, and reassess the composite Endgame Skill formula if Parity has its own offset.

**Depends on:** Phase 70 (gate-pass) — independent of Phase 71 in principle, but the SEED-002 ordering keeps it after rating stratification
**Requirements:** VALID-04
**Plans:** TBD

**Success Criteria** (what must be TRUE):
1. A report (extension of Phase 70 report or `reports/parity-validation-benchmark-YYYY-MM-DD.md`) documents the agreement rate between "material imbalance in [-100, +100]cp at endgame entry" and "Stockfish eval in [-50, +50]cp at endgame entry", per endgame type, with sample size and 95% CI per cell
2. The report names any systematic Parity offset (proxy over- or under-reads vs eval) per endgame type, in the same format as the Conversion / Recovery offset tables in the 2026-04-07 report
3. The report explicitly reassesses the composite Endgame Skill formula: either confirms the current `(Conv + Par + Rec) / 3` (or the actual current formula, verified during the phase) is sound under the measured offsets, OR proposes a re-weighting and documents the rationale
4. If a re-weighting is proposed, the report names the affected zone constants and which subsequent Phase 73 step they depend on (so Phase 73 can carry the change through to `theme.ts`)

### Phase 73: `/benchmarks` Skill Upgrade & Zone Re-Calibration

**Goal:** Upgrade the `/benchmarks` skill to query the benchmark DB and produce per-(rating_bucket x TC x platform x endgame_type) population baselines for Conversion / Parity / Recovery / composite Endgame Skill / clock-at-entry / timeout rates, compute rating-specific zone thresholds using the median-user-at-warning/success-boundary rule, AND ship those rating-bucketed zone constants in `frontend/src/lib/theme.ts` (PR-reviewed, deployed). Extend the `/db-report` skill so DB-health reports cover the benchmark database alongside dev/prod.

**Depends on:** Phase 70 (gate-pass), Phase 71 (rating offsets), Phase 72 (parity verdict)
**Requirements:** BENCH-01, BENCH-02, BENCH-03, BENCH-04
**Plans:** TBD

**UI hint:** yes

**Success Criteria** (what must be TRUE):
1. Running the upgraded `/benchmarks` skill produces a report containing population baselines for Conversion / Parity / Recovery / composite Endgame Skill / clock at endgame entry / timeout rates, broken down per (rating_bucket x time_control x platform x endgame_type), with sample size and 95% CI per cell, AND a rating-bucketed zone-threshold table covering Conversion (warn/succ), Recovery (danger/warn, warn/succ), and Endgame Skill (warn/succ)
2. The rating-bucketed zone-threshold values from the skill report land in `frontend/src/lib/theme.ts` (or the equivalent constants location) and are reviewed in a merged PR — the milestone is INCOMPLETE until the constants ship in code, not just the report

4. Running `/db-report` returns a single output describing both prod and benchmark databases (table counts, row counts, last-updated timestamps), so DB-health checks are unified
5. If Phase 72 proposed a composite-formula re-weighting, the new formula is reflected in the skill output, the zone registry, and `theme.ts`; otherwise the existing `(Conv + Par + Rec) / 3` (or current actual) formula stands unchanged
