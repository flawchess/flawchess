> free-analysis `/analysis` board. Generated by gsd-ui-researcher, verified by

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / row text | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Dense engine-surface text (approved exception) | 12px (`text-xs`) | 400 / 600 | 1 (`leading-4`, matching `EngineLines` chips) |
| Card header caption | 14px (`text-sm`) | 600 (semibold, `font-medium`/`font-semibold`) | 1.2 |
| Score-pair badge numbers | 12px (`text-xs`) | 600 (`font-semibold`) | 1 |

Only 2 weights used: 400 (regular body/chips) and 600 (semibold headers/badges) — matches the existing `EngineLines`/`Analysis.tsx` convention exactly. The `text-xs` dense-engine-surface exception (CLAUDE.md) extends to the new `FlawChessEngineLines` card because it is structurally the same surface type as `EngineLines` (PV chips + eval badges) — toggle labels stay `text-sm` per the CLAUDE.md floor. **Correction (verified against shipped code):** the existing "Maia"/"SF" eval-bar caps use `text-xs font-medium leading-none` (`Analysis.tsx` `evalBarCap`, a UAT-driven "make the labels smaller" local exception, not `text-sm`). The new "FC" cap MUST match that existing `text-xs` cap so the three caps are visually uniform — do NOT introduce a `text-sm` cap here (this supersedes the earlier draft of this line, which incorrectly claimed the caps were `text-sm`; see 155-RESEARCH.md Common Pitfall 4).

## Color

| Role | Value | Usage |
|------|-------|-------|

### New theme.ts tokens (add alongside `STOCKFISH_ACCENT`/`MAIA_ACCENT`, same file, same export style)

```ts
// FlawChess Engine source accent (Phase 155 D-05). Brand brown — third source
// accent alongside STOCKFISH_ACCENT (blue) and MAIA_ACCENT (violet). Tints the
// FlawChessEngineLines card frame + header caption + the "FC" eval-bar fill/cap.
export const FLAWCHESS_ENGINE_ACCENT = 'oklch(0.55 0.09 55)'; // brand brown, matches --brand-brown (#8B5E3C) in oklch

// Subtle gold/bronze highlight reserved for the practical-score headline number
// ONLY (Phase 155 D-05) — never a card-wide glow (fights the ~150ms live-refresh
// churn). Distinct hue from FLAWCHESS_ENGINE_ACCENT (brown, 55) so the "practical"
// number pops against its own brown-accented card without becoming another brown.
export const FLAWCHESS_ENGINE_HEADLINE_ACCENT = 'oklch(0.75 0.14 85)'; // bronze/gold
```

Accent reserved for (10% — explicit list, nothing else):
1. `FlawChessEngineLines` card frame border + header caption text ("FC" label, mirroring how `STOCKFISH_ACCENT` tints `ENGINE_NAME` in the existing `analysis-engine-info` header).
2. The shared left-slot `EvalBar`'s `accentColor` prop when it is showing the FC source (precedence over Maia per D-04).
3. The **practical** score number inside each line's score-pair badge (bronze/gold, `FLAWCHESS_ENGINE_HEADLINE_ACCENT`) — the objective number stays `STOCKFISH_ACCENT` (blue-tinted, per D-06).
4. Nowhere else. No card-wide glow, no background tint change, no border-radius change. (Board-arrow color is explicitly out of scope — Phase 156.)

Destructive: not applicable — this phase has no destructive actions (see Copywriting Contract).

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|
| shadcn official | `switch` (new — for the 3 card-header toggles, D-03) | not required (official registry, no vetting gate per the researcher's scope) |
| third-party | none | not applicable |

No third-party registries declared this phase. `switch` is pulled from the shadcn **official** registry only (`ui.shadcn.com`) — the registry vetting gate (view + flag-scan) applies only to third-party registries per the researcher contract, so it is not triggered here. If the executor instead hand-rolls a switch (e.g. reusing `toggle.tsx`'s CVA pattern) rather than running `npx shadcn add switch`, that is an acceptable substitution — CVA-styled hand-rolled Radix wrapper is the existing project convention for every other `ui/` primitive (`toggle.tsx`, `tabs.tsx`, etc. are all hand-authored, not raw shadcn pulls) and no registry issue is introduced either way. **Recommendation: hand-roll it as `src/components/ui/switch.tsx`** using `@radix-ui/react-switch` (already a transitive dep via `radix-ui`) styled with the project's existing CVA/token conventions (brand-brown `bg-brand-brown` when on, matching `Button`'s `default` variant fill), rather than pulling a shadcn preset that would need re-theming to match `radix-nova`'s existing neutral-base look. Either path is design-contract-compliant; the planner picks one.

## Component Inventory (new/modified this phase)

Beyond the standard template sections, the executor needs concrete component contracts for the two new surfaces this phase introduces. This section is the authoritative visual/behavioral spec — do not re-derive from the CONTEXT.md prose alone.

### 1. `FlawChessEngineLines.tsx` (new, `frontend/src/components/analysis/`)

Structurally a sibling of `EngineLines.tsx` — same card-row skeleton, same chip/badge/hover-preview pattern — with these deltas:

- **Card placement (D-01):** new `Card` wrapped around this component, stacked directly **above** `MaiaHumanPanel` inside the existing `analysis-human-column` div (desktop, `frontend/src/pages/Analysis.tsx` ~line 1436) and the mobile "Human" tab content (~line 1264-1281). Both desktop and mobile get the card — no desktop-only exception (per CLAUDE.md mobile-parity rule and D-01's explicit "apply to both").
- **Card header:** `CardHeader size="compact"` (matches `analysis-engine-info`'s pattern) containing: a new `Switch` (D-03, replaces a plain toggle button) + `"FlawChess Engine"` caption text styled `style={{ color: FLAWCHESS_ENGINE_ACCENT }}` `className="text-sm font-semibold"`.
- **Line count (D-08):** top **3** ranked lines (not 2) — `MAX_LINES = 3` local constant, distinct from `EngineLines.tsx`'s `MAX_LINES = 2`. Do not change the shared constant; this is a phase-155-only value on the new component.
- **Score-pair badge (D-06, DISPLAY-03):** replaces the single-number `BADGE_CLASS` pill with a two-number badge: `{objectiveScore}` (blue-tinted, `STOCKFISH_ACCENT`) + a visual separator (e.g. a thin `/` or two adjacent pill segments) + `{practicalScore}` (bronze-tinted, `FLAWCHESS_ENGINE_HEADLINE_ACCENT`). Both on the pawn scale, both white-POV, same `+X.X`/mate-`#±N` format as `formatScore()` in `EngineLines.tsx` — reuse that formatter verbatim rather than reimplementing (it already takes `evalCp`/`evalMate`; convert `RankedLine.practicalScore` (0–1 root-STM) to a white-POV cp-equivalent via the inverse of the project's existing sigmoid util before calling it, per D-06's copy nuance — do not literally invert POV per side-to-move).
- **Modal path (D-07):** identical `MAX_PLIES = 5` chip pattern to `EngineLines.tsx`, but chips render `RankedLine.modalPath` (already computed root-relative). Reuse `replayPvLine()`/hover `MiniBoard` preview/`ChevronDown` expand exactly — this is a verbatim structural clone, not a new pattern.
- **Click-to-graft (D-10):** identical `onMoveClick(uciMoves)` wiring to `EngineLines.tsx` — same handler shape (`playUciLine` from `useAnalysisBoard`), so `Analysis.tsx` passes the same callback to both components.
- **Live-refine cadence (D-09):** first paint shows `EngineLinesSkeleton` (reused component, no new skeleton needed) sized for 3 rows instead of 2 (pass a `rows` prop or accept the taller default — planner's call on the exact prop shape); subsequent snapshots batch-update at the existing `RAPID_STEP_DEBOUNCE_MS = 150` cadence via the new hook, not a new debounce constant.
- **Empty/off state:** `"FlawChess Engine off"` text row, exact visual clone of the existing `"Engine off"` `<div className="flex h-full items-center px-2 text-sm text-muted-foreground">` block.
- **`data-testid`s:** `analysis-flawchess-card`, `analysis-flawchess-info` (header), `flawchess-line-{n}-move-{m}` (chips, mirroring `engine-line-{lineIndex}-move-{moveIndex}`), `flawchess-line-{n}-expand`, `btn-analysis-flawchess-toggle` (the new switch).

### 2. Eval-bar left-slot precedence (D-04)

- No new component — `EvalBar.tsx` is reused unchanged (`accentColor`/`whiteFraction`/`testId` props already sufficient).
- `Analysis.tsx`'s left-slot `EvalBar` call (currently hard-wired to Maia, `testId="analysis-maia-eval-bar"`) becomes conditional: when the FlawChess Engine is enabled, render it with `accentColor={FLAWCHESS_ENGINE_ACCENT}`, `whiteFraction` derived from the top `RankedLine.practicalScore` (root-STM → white-POV, same conversion as the badge), and `testId="analysis-flawchess-eval-bar"`; else fall back to the existing Maia-accented bar unchanged. The right-slot Stockfish bar's `evalCp`/`evalMate` source swaps to `RankedLine.objectiveEvalCp` of the top line while the FlawChess Engine runs (same `testId="analysis-eval-bar"`, no visual change — only the data source changes, per D-04/154 D-03 handoff).
- Bar cap labels ("FC"/"Maia"/"SF") — small `text-sm` caption above each bar; exact copy is Claude's-discretion per CONTEXT.md ("FC" confirmed by D-04's own prose; "SF" and "Maia" already exist as the current caps — no change to those two).

### 3. Three-toggle refactor (D-03)

- All three engine cards (`Stockfish`, `Maia`, `FlawChess`) get a `Switch` in their `CardHeader`, replacing the current icon-`Button` pattern (`btn-analysis-engine-toggle`'s `Cpu` icon) with a real on/off switch component. Visual weight: switch first (left), caption text after — same left-to-right order the current icon-button+caption already uses, just swapping the control type.
- Switch "on" fill color: use each card's own accent (`STOCKFISH_ACCENT` blue / `MAIA_ACCENT` violet / `FLAWCHESS_ENGINE_ACCENT` brown) for the switch's checked-state track, so the switch itself reinforces the source-color coding already established by the bars/captions — do not default to a single neutral "on" color across all three (that would erase the existing color-coding convention this phase is extending).
- Binding (Claude's Discretion, resolved): the Maia card's switch continues to gate `useMaiaEngine` (the chart worker) **and** the existing `useStockfishGradingEngine` (which colors the Moves-by-Rating chart) — both are Maia-chart-surface concerns, so bundling them under the Maia toggle is correct; the FlawChess Engine's internal `maiaQueue` instance (Phase 154) is a **separate** worker instance and must NOT be gated by this switch (per CONTEXT.md's explicit note — turning off the Maia card must not starve the FlawChess Engine's policy source). The Stockfish card's switch keeps gating `useStockfishEngine` (the eval-bar/PV search) exactly as `engineEnabled` does today; it does NOT gate the FlawChess Engine's own `workerPool` instance either (that's the new FlawChess switch's job).

## Interaction Contract

- **Hover:** modal-path chips get the same `MiniBoard` hover-preview tooltip as `EngineLines` (desktop only, `Tooltip` suppresses on touch — no change needed, reused verbatim).
- **Click:** modal-path chips graft the clicked line's prefix into the board tree via `onMoveClick`, exactly like `EngineLines` — same `playUciLine` handler, same "graft whole prefix up to and including the clicked move" semantics (D-10).
- **Toggle:** clicking/tapping a card's `Switch` flips that card's `enabled` boolean state; no confirmation, no undo needed (non-destructive, instantly reversible) — matches every other toggle already on this page (`engineEnabled`, board-flip, etc.).
- **Expand chevron:** per-line expand/collapse of the modal path beyond `MAX_PLIES = 5`, identical to `EngineLines`' `ChevronDown`/`aria-expanded` pattern — state is per-row, keyed by line index, persists across live-refine snapshot updates (same as the existing component's `expanded` state persisting across streamed depth updates).
- **Live update cadence:** lines reorder/update at the 150ms batched cadence (D-09) — no user-facing loading spinner during a re-order, only the initial pre-first-snapshot skeleton. Reordering must not steal focus or scroll position (no scroll-into-view on reorder).
