<objective>
Close UAT gap 1 (major, test 2): the analysis board does not flip to the player's colour
when opened from a finished bot game via "Analyze this game".

Root cause (already diagnosed — do NOT re-investigate): the bot Analyze button navigates to
`/analysis` in **free-play (`?line=`) mode**, but `Analysis.tsx`'s only auto-flip effect is
gated on `isGameMode` (`?game_id=`) and sources colour from the backend's
`gameData.user_color`. Free play has **no orientation input at all** — the URL contract has
no colour param — so `boardFlipped` stays at its `useState(false)` default.

Fix: make free-play mode **orientable**. Add an orientation param to the `/analysis` URL
contract, pass the bot game's `settings.userColor` into it, and collapse the two orientation
sources into a single `autoOrientation` value driving the one existing flip effect.

Purpose: a player who just played Black must see their own game from their own side. This is
the one thing an analysis deep-link cannot get wrong.
Output: an orientable free-play `/analysis` URL contract, wired end-to-end from `Bots.tsx`.
</objective>

<constraint priority="hard">
## DO NOT DO THIS (the tempting shortcut, explicitly ruled out)

**Do NOT re-point `handleAnalyze` at `buildGameAnalysisUrl(storedGameId)`** to get the
existing game-mode auto-flip "for free".

Why it is forbidden:
- It is ruled out by **D-20 / D-21**: the store is queued and may still be pending (offline,
  401-retry, guest). The primary CTA would sometimes have nothing to point at.
- It is **pinned by test V-17** (`GameResultDialog.test.tsx:94` — *"Analyze this game" still
  fires onAnalyze when storeSucceeded is true — not re-pointed at the stored game*). That test
  must stay green and must not be edited.
- Analyze must work for **guests** and **when the store failed**.

Therefore free-play mode is what becomes orientable. There is no second path.
</constraint>
