# Phase 94.2: Pooled-Per-User Percentile Redesign — Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace Phase 94.1's per-cell stratified percentile methodology with a one-point-per-user pooled model on both the CDF construction side and the per-user lookup side. For every subject (cohort user during CDF generation; app user at lookup time), pool their games across all TCs played, capped at the most recent 1000 per TC and ≤36 months old, then compute the metric once on that pool. Build the CDF as `percentile_cont` over those one-point-per-user values — a single globally pooled CDF per metric, representing the whole benchmark cohort's *current* state rather than an average across each user's rating journey.

The app user looks up their pooled value against the same global CDF, so the chip answers "where I currently sit on this metric vs all benchmarked Lichess players, on my recent play" rather than today's "average across the `(elo_bucket, tc_bucket)` cells I've ever played in." Inclusion floor is a single ≥30-games threshold on the pooled set. Rating-correlated metrics will read systematically higher for high-rated players — intentional, disclosed in tooltip copy per metric.

**Locked carryover from Phase 94.1 (no schema change):** `user_benchmark_percentiles` table shape, Stage A/B trigger pattern in import + eval-drain pipelines, backfill script harness (`scripts/backfill_user_percentiles.py --target dev|prod`), shared `app/services/canonical_slice_sql.py` module location, Postgres ENUM for `metric`, `cdf_snapshot` column semantics.

**Out of scope:** Phase 95 LLM payload + prompt rework. Per-type / Time Pressure / Opening Insights percentile chips. Scheduled CDF-snapshot re-lookup job.

</domain>

<decisions>
## Implementation Decisions

### Sparse-cell exclusion (discussed)

- **D-1: Sparse-cell `(2400, classical)` exclusion lives only at cohort selection (dedup-aware).** On the CDF-construction side, dedup `benchmark_selected_users` to one row per `lichess_username`; a user only selected via the `(2400, classical)` slot drops, a user selected via that slot AND another slot stays. **No game-row WHERE filter in the pooled per-user compute on either side** — the per-row `sparse_exclusion_sql(elo_col, tc_col)` filter (currently invoked in `_per_user_cte_*` builders) is removed. Rationale: sparse-cell exclusion was a per-cell stability fix; the pooled CDF has no cells to stabilise. Pooling already mixes rating buckets together, so a single user's 2400-classical games are a minority slice of their pool and don't destabilise their pooled value. The 1000/TC cap + 36-month recency already shape the pool.

### Distinct-user-count investigation (discussed)

- **D-2: Researcher records distinct-user and eval-coverage counts in RESEARCH.md; no formal blocker.** During `/gsd-plan-phase` research, run two SQL queries against the benchmark DB:
  1. `SELECT COUNT(DISTINCT lichess_username) FROM benchmark_selected_users` (and the same filtered to passing `status='completed'` + ±100 + standard-variant filters per metric).
  2. Per-benchmark-user eval coverage on the 36-month window: how many users pass the ≥30 floor for `achievable_score_gap` / `section2_score_gap_conv` / `section2_score_gap_parity` (these require Stockfish evals on entry-eval rows in the benchmark DB).
  
  Numbers land in RESEARCH.md so the planner and user can eyeball them. **No hard threshold gate** — ROADMAP success criterion 8 is satisfied by capturing the investigation, not by enforcing a TBD blocker. If a metric's cohort looks egregiously small, surface it as a planning concern then; otherwise ship the four metrics together.

### Cutover ordering (discussed)

- **D-3a: Single PR. Plans wave-ordered so no green commit ever holds an incoherent state (new CDF literal + old per-cell compute, or vice versa).** Recommended plan boundaries (planner finalises):
  1. Refactor `app/services/canonical_slice_sql.py` to drop `apply_floor` dual-mode, drop `n_cells_floor`, drop per-row sparse-cell exclusion, add the recent-1000-per-TC cap and 36-month recency filter, collapse `per_user_cte_*` to one pooled aggregate per subject.
  2. Rewrite `scripts/gen_global_percentile_cdf.py:_build_metric_breakpoint_query` to consume the pooled shape; regenerate `app/services/global_percentile_cdf.py:GLOBAL_PERCENTILE_CDF`; regenerate the byte-identical regression test goldens (`tests/scripts/test_gen_global_percentile_cdf_unchanged.py`).
  3. Rewrite `app/services/user_benchmark_percentiles_service._compute_metric_for_user` to call the new pooled SQL once and store the value; drop `apply_floor` argument from its public surface.
  4. Update `.claude/skills/benchmarks/SKILL.md` methodology chapter.
  5. Update the four tooltip copies on `EndgameOverallScoreGapRow.tsx` + `EndgameMetricCard.tsx` per D-4.
  6. Rerun `scripts/backfill_user_percentiles.py --target dev` (and `--target prod` after sign-off) as the final plan.
  
  Plans 1–3 commit together at every green state (e.g. plan 1's commit keeps old CDF + new builder both behind a feature flag, or simply lands all three in adjacent commits inside the same plan if simpler — planner's call). The key constraint: no merged commit produces a broken chip for users.

- **D-3b: No explicit rollback plan.** The chip is non-critical product surface (visible but not breaking). If post-deploy numbers look obviously wrong, `git revert` the PR and accept that the next backfill rerun restores 94.1 (value, percentile) values. No pre-deploy snapshot file or emergency revert script.

### Tooltip copy direction (discussed)

- **D-4: Tooltip copy explicitly discloses four things per metric:**
  1. **Benchmark composition** — "Calibrated against benchmarked Lichess players of all ratings and time controls."
  2. **Recent-games basis** — "Uses your most recent 1000 games per played time control (last 36 months)."
  3. **Filter independence** — "UI filters do not affect this percentile."
  4. **Per-metric rating-correlation framing** — calibrated per metric from `reports/benchmarks-gap-metrics-percentile-candidacy.md`:
     - `score_gap` (Endgame Score Gap, d=0.19, rating-invariant) — no special note
     - `achievable_score_gap` (d=0.32, mild coupling) — soft mention
     - `section2_score_gap_parity` (d=0.30, small effect) — soft mention
     - `section2_score_gap_conv` (d=1.37, heavy rating-proxy) — honest framing per the candidacy report (e.g. "this metric tracks rating strongly")
  
  This is a **deliberate exception to `feedback_popover_copy_minimalism`** for the percentile chip surface. The chip is itself the methodology-disclosure surface; without per-metric framing a 1200 user reads Conversion-Score-Gap percentile as a skill judgment it can't make. The exception is captured in `feedback_percentile_chip_tooltip_disclosure` (project memory). All other popover/tooltip copy in the app continues to obey minimalism. Exact strings are planner's call within these four bullets.

### Amendment 2026-05-24: `n_games` column re-add (correction to D-9)

- **D-9-amend: `n_games` column is RE-ADDED via a small Alembic migration in Plan 03 (correction to D-9's "no Alembic migration" claim).** D-9 listed the table shape as `(user_id, metric, value, percentile, n_games, cdf_snapshot, computed_at)` but that list was wrong — the actual current shape post-94.1-13 is `(user_id, metric, value, percentile, cdf_snapshot, computed_at)`. The history: 94.1-04 added `n_games` → 94.1-11 renamed to `n_cells_floor` → 94.1-13 dropped it entirely. Under the pooled methodology, `n_games` is semantically meaningful again (one count per user per metric, not per cell), so we restore it. Per-metric meaning:
  - `score_gap` → endgame games count (the binding floor — paired non-endgame count goes in the backfill summary log, not the table)
  - `achievable_score_gap` → endgame-entry games with non-null `d_i`
  - `section2_score_gap_conv` → conversion-span count
  - `section2_score_gap_parity` → parity-span count
  
  `_compute_metric_for_user` widens return to `tuple[float, int] | None`. `upsert_percentile` widens to accept `n_games`. ENUM unchanged. The new Alembic migration is single-column `ADD COLUMN n_games INT NOT NULL DEFAULT 0` followed by an immediate post-deploy backfill via Plan 06 (the value is recomputed during the backfill rerun, so default-0 rows only exist for the deploy-to-backfill-completion window — acceptable for non-critical chip surface). D-9's "schema unchanged" intent (no major redesign, no PK change) is preserved; the literal "no migration" clause is corrected.

  Leave per-cell distribution analysis methodology intact. Total SKILL.md diff: ~30 LOC, not a chapter rewrite. This trims ROADMAP success criterion 6 to "add SKILL.md callout + update any production-CDF-correspondence claims" rather than "rewrite Chapter 4 cohort-construction and diagrams." ### Locked by ROADMAP / design note (not re-discussed)

- **D-5: One-point-per-user pooling on both sides.** Per subject: take games per TC played ordered by `played_at DESC`, cap at most-recent 1000 per TC, drop >36 months from snapshot date, pool the per-TC subsets, apply universal filters (`rated=true`, non-computer, ±100 equal-footing, standard variant), compute the metric once on the pool. CDF = `percentile_cont` over the one-point-per-user values. Source: `.planning/notes/per-user-percentile-pooled-redesign.md` §Redesign. Locked by ROADMAP success criteria 1, 2.
- **D-6: Inclusion floor = ≥30 games of the metric-relevant unit on the pooled set.** Per metric:
  - `score_gap`: ≥30 endgame games AND ≥30 non-endgame games
  - `achievable_score_gap`: ≥30 endgame-entry games with non-null `d_i`
  - `section2_score_gap_conv`: ≥30 spans in the Up-entry-eval bucket
  - `section2_score_gap_parity`: ≥30 spans in the Equal-entry-eval bucket
  
  Failure: no row stored (chip suppressed). Same suppression contract as Phase 94.1 Plan 13. No partial-row-with-NULL-percentile in 94.2 (the cell-based model that motivated NULL-percentile rows is gone). Locked by ROADMAP success criterion 1 + design note §Inclusion floor.

### Claude's Discretion

The user picked the 4 areas above for discussion. Remaining HOW choices the planner can resolve in PLAN.md without coming back to the user:

- **`n_games` column semantics in the pooled model** — single integer per row. Recommend per-metric meaning: for `score_gap` store the endgame-games count (the constraint is the binding one for tooltips/diagnostics — paired non-endgame count goes in the backfill summary log, not the table); for `achievable_score_gap` store endgame-entry-with-d_i count; for section2_* store the relevant bucket span count. Document the per-metric meaning in `app/models/user_benchmark_percentiles.py` docstring.
- **Recent-1000-per-TC cap implementation** — `ROW_NUMBER() OVER (PARTITION BY user_id, tc_bucket ORDER BY played_at DESC) <= 1000` inside a CTE before the universal-filter join. Apply before the 36-month recency filter or after — either order produces the same set (the cap is independent of recency).
- **36-month recency anchor** — `played_at >= NOW() - INTERVAL '36 months'` on the per-user compute side (per-user snapshot is "now"); on the CDF-generation side, anchor to a snapshot date passed via CLI (mirrors the existing `gen_global_percentile_cdf.py` snapshot semantics).
- **Plan boundaries inside the phase** — the 6-step ordering in D-3a is a recommendation; planner can merge/split (e.g. fold the SKILL.md update into the same plan as the regen-report capture). The constraint is D-3a's "no green commit holds an incoherent state."

Planner may drop if it adds friction.
- **Tooltip copy strings** — exact wording is planner's call within D-4's four bullets and `feedback_percentile_chip_tooltip_disclosure`. Mobile vs desktop variants treated identically.
- **Backfill operational details** — same defaults as Phase 94.1 (per-user commits, creation-asc ordering, per-100-users summary log line).
- **Regen test goldens** — regenerate `tests/scripts/test_gen_global_percentile_cdf_unchanged.py` fixtures against the new methodology; the test exists (catches accidental drift in the SQL builder), only the fixtures change.
