**Researched:** 2026-05-24

## Summary

Phase 94.2 is a methodology refactor, not a feature build. The schema, table, triggers, backfill harness, and frontend chip component are all 94.1-stable. What changes is *what number lands in `user_benchmark_percentiles.value` and `.percentile`*: the per-cell-then-averaged value is replaced by a single pooled value over the user's recent-1000-per-TC × 36-month game window. The CDF is regenerated identically over the deduped benchmark cohort using the same recipe, so drift between the two paths remains structurally impossible.

The refactor surface is narrow and well-mapped: ~5 backend Python files (one SQL-builder module, one CDF-regen script, one service module, one literal, one backfill script), one regenerated regression-test golden, one SKILL.md chapter, and four tooltip copy strings on two frontend components. No Alembic migration. No router changes. No new packages.

**Primary recommendation:** Land Plans A+B+C as one wave (single commit or contiguous-commits inside one branch with no merge-to-main between them), because any intermediate state is broken by construction — the byte-identical CDF regression test requires both the new builder *and* the new literal, and the service stores values that must be interpreted against the matching CDF. Plan D (SKILL.md), Plan E (tooltip copy), and Plan F (backfill rerun) can be sequenced safely after the A+B+C wave merges.

### Pattern 1: Pooled-per-user CTE shape (replaces per-cell shape)

**What:** A single CTE chain that produces one row per subject (one cohort user for CDF, one app user for lookup) with one pooled `metric_value` column. No `elo_bucket` / `tc_bucket` projection, no `sparse_exclusion_sql`, no `HAVING` per-cell gate.

**When to use:** Replaces every `per_user_cte_*` builder in `app/services/canonical_slice_sql.py`.

**Example (sketch — score_gap, single_user):**
```sql
WITH selected_users AS (SELECT CAST(:user_id AS int) AS user_id),
recent_games AS (
  SELECT g.*, row_number() OVER (PARTITION BY g.user_id, g.time_control_bucket
                                  ORDER BY g.played_at DESC) AS rn
  FROM games g
  JOIN selected_users su ON su.user_id = g.user_id
  WHERE g.rated AND NOT g.is_computer_game
    AND g.white_rating IS NOT NULL AND g.black_rating IS NOT NULL
    AND g.played_at >= NOW() - INTERVAL '36 months'
    AND abs(...) <= 100   -- equal-footing
),
filtered AS (
  SELECT * FROM recent_games WHERE rn <= 1000
),
endgame_game_ids AS (
  SELECT game_id FROM game_positions WHERE endgame_class IS NOT NULL
  GROUP BY game_id HAVING count(*) >= 6
),
scored AS (
  SELECT
    user_id,
    CASE WHEN (result = '1-0' AND user_color = 'white')
           OR (result = '0-1' AND user_color = 'black') THEN 1.0
         WHEN result = '1/2-1/2' THEN 0.5 ELSE 0.0 END AS score,
    (eg.game_id IS NOT NULL) AS has_endgame
  FROM filtered f
  LEFT JOIN endgame_game_ids eg ON eg.game_id = f.id
),
per_user AS (
  SELECT
    user_id,
    avg(score) FILTER (WHERE has_endgame)     AS eg_score,
    avg(score) FILTER (WHERE NOT has_endgame) AS non_eg_score,
    count(*)   FILTER (WHERE has_endgame)     AS eg_n,
    count(*)   FILTER (WHERE NOT has_endgame) AS non_eg_n
  FROM scored
  GROUP BY user_id
  HAVING count(*) FILTER (WHERE has_endgame)     >= 30
     AND count(*) FILTER (WHERE NOT has_endgame) >= 30
)
SELECT (eg_score - non_eg_score) AS metric_value, eg_n AS n_games
FROM per_user
```

For the benchmark path the `selected_users` CTE switches to the dedup-aware cohort SELECT (one row per `lichess_username`, dropping `(2400, classical)`-only members per D-1; see "Selected-users dedup" subsection below).

### Pattern 2: Dedup-aware cohort selection for the CDF path (D-1)

**What:** On the benchmark side, before pooling, dedup `benchmark_selected_users` to one row per `lichess_username` and drop users whose only selection slot is `(2400, classical)`.

**Sketch:**
```sql
selected_users AS (
  SELECT u.id AS user_id
  FROM users u
  WHERE lower(u.lichess_username) IN (
    SELECT lower(bsu.lichess_username)
    FROM benchmark_selected_users bsu
    JOIN benchmark_ingest_checkpoints bic
      ON bic.lichess_username = bsu.lichess_username
     AND bic.tc_bucket = bsu.tc_bucket
     AND bic.status = 'completed'
    GROUP BY lower(bsu.lichess_username)
    HAVING bool_or(NOT (bsu.rating_bucket = 2400 AND bsu.tc_bucket = 'classical'))
  )
)
```
The `bool_or(...)` keeps any user with at least one non-sparse selection slot. A user *only* selected via `(2400, classical)` is dropped from the cohort entirely.

### Pattern 3: Snapshot-date parameterisation (planner discretion)

**Recommendation:** Add an optional `snapshot_date` parameter to the pooled SQL builder (`None` = use `NOW()` on the per-user lookup path; explicit `date` = use that on the CDF-construction path). The CLI in `scripts/gen_global_percentile_cdf.py` already mirrors `BENCHMARK_DB_SNAPSHOT_MONTH` (currently `'2026-03'`); extend it to pass an explicit `snapshot_date` so the 36-month recency window anchors to the dump month, not the operator's current clock.

**Example signature:**
```python
def per_user_cte_for(
    metric_id: CdfMetricId,
    *,
    source: Literal["benchmark", "single_user"],
    snapshot_date: date | None = None,  # None = NOW() (single_user); date = explicit (benchmark)
) -> str:
    ...
```

Inside the CTE, render the recency predicate as either `NOW() - INTERVAL '36 months'` (None) or `DATE '2026-03-31' - INTERVAL '36 months'` (explicit). Avoid threading a SQL bindparam for this — the snapshot date is a build-time artifact, not user-controlled input. Keeping it as a Python f-string into the SQL is consistent with the existing `BREAKPOINTS` and `BENCHMARK_DB_SNAPSHOT_MONTH` constants in the script.

### Anti-Patterns to Avoid

- **Re-introducing `apply_floor` dual-mode.** D-8 explicitly removes it. The 94.1 motivation (writing a value with `percentile=NULL` below floor) is gone — 94.2 suppresses the row entirely.
- **Keeping `sparse_exclusion_sql` at the per-row level.** D-1 explicitly removes per-row sparse-cell filtering. The function `sparse_exclusion_sql` itself can stay in `canonical_slice_sql.py` if any external caller imports it (none found), but no production builder should invoke it after Plan A.
- **Touching the Stage A/B trigger sites.** `compute_stage_a` and `compute_stage_b` in `user_benchmark_percentiles_service.py` are unchanged in surface. Only the internal `_compute_metric_for_user` body changes.
- **Adding or removing metric IDs.** The Postgres ENUM `benchmark_metric` stays at 4 values: `score_gap`, `achievable_score_gap`, `section2_score_gap_conv`, `section2_score_gap_parity`. Same `CdfMetricId` Literal. Adding a metric is a Phase 95+ concern, not 94.2.
- **Carrying over Plan 13 floor-routing dead code.** Plan 13's `apply_floor=True`-only branch was a per-cell correctness fix; under 94.2 there are no cells. The dual-query / `n_cells_floor` machinery is gone — don't preserve scaffolding for it.
- **Re-adding `elo_bucket`/`tc_bucket` projections to `per_user_values`.** The pooled CTE emits one column (`metric_value`) and optionally `n_games`. The `_build_per_bucket_sanity_query` diagnostic still computes `elo_bucket` independently — that's the only place these projections remain.

      ON bic.lichess_username = bsu.lichess_username

## Common Pitfalls

### Pitfall 1: Inconsistent snapshot date between CDF construction and per-user lookup
**What goes wrong:** CDF built against a 36-month window anchored at `2026-03-31`, but per-user lookups use `NOW()` (e.g. 2026-05-24). A user whose recent games span late-2023 → mid-2026 hits a different game set than the cohort used for the breakpoints; their percentile is biased.
**Why it happens:** The two anchors look symmetric in code but mean different things: CDF anchors to the benchmark DB snapshot month; per-user lookup anchors to "now."
**How to avoid:** Build the optional `snapshot_date` parameter into the shared SQL builder (see Pattern 3). On the per-user side pass `None`/`NOW()` — this is correct by design for "where am I now." Document the asymmetry in the regenerated SKILL.md chapter: the CDF cohort is a 36-month snapshot of the benchmark month; per-user is a 36-month look-back from "now."
**Warning signs:** A user's stored value swings noticeably between two backfill runs separated by a few weeks despite no new imports.

### Pitfall 2: Sparse-cell exclusion accidentally re-applied at the row level
**What goes wrong:** A reviewer "fixes" the missing `sparse_exclusion_sql` invocation by re-adding it inside the rows CTE, silently dropping all `(game-time-2400, classical)` games even when the user has games in other (rating, TC) combinations.
**Why it happens:** The 94.1 muscle memory says "every per-row CTE has sparse exclusion." D-1 explicitly reverses that for 94.2.
**How to avoid:** Delete `sparse_exclusion_sql` calls from every `per_user_cte_*` builder; leave a deliberate comment at each former call site (`# D-1: sparse-cell exclusion lives in cohort selection only; no per-row filter under pooled methodology`). Add a builder-shape unit test asserting that the generated SQL string does NOT contain the substring `2400 AND tc_bucket = 'classical'`.
**Warning signs:** Per-user n_games for a 2400-classical-only user reads zero under the new methodology when the cohort-level dedup would otherwise have dropped them.

### Pitfall 3: Forgetting to dedup `lichess_username` on the benchmark cohort
**What goes wrong:** A user qualified into 3 TC slots (e.g. bullet + blitz + rapid) contributes 3 CDF points instead of 1 — exactly the 94.1 cell-row inflation bug 94.2 is fixing, just relocated.
**Why it happens:** The natural join `benchmark_selected_users → users` doesn't dedup by default.
**How to avoid:** The cohort CTE for `source="benchmark"` MUST `GROUP BY lower(bsu.lichess_username)` (or a subquery `IN (SELECT DISTINCT lower(...))`) before joining to `users`. Validate by `COUNT(DISTINCT user_id)` after the cohort CTE equals the deduped count from Probe 1.
**Warning signs:** The new `n_users` in the regen report exceeds the raw `COUNT(DISTINCT lichess_username)` from Probe 1.

### Pitfall 4: Stage A/B return-value contract drift
**What goes wrong:** `_compute_metric_for_user` is rewritten and its return type changes (e.g. returns a 3-tuple `(value, percentile, n_games)` instead of `float | None`). `compute_stage_a` / `compute_stage_b` callers break silently because their try/except swallows the exception and Sentry sees a flood of errors.
**Why it happens:** The phase researcher (this RESEARCH.md) and the user's instructions both flag this as a load-bearing contract. Plan C is tempting to over-refactor.
**How to avoid:** Keep `_compute_metric_for_user` signature returning `float | None` (the raw pooled value). The interpolation step `interpolate_percentile(metric, value)` already lives in the caller. If the planner wants to surface `n_games` for the UPSERT, return `tuple[float, int] | None` and update the two call sites — but document the change explicitly. Verify by running `tests/services/test_user_benchmark_percentiles_service.py` and the two `test_import_service_stage_a.py` / `test_eval_drain_stage_b.py` integration tests.
**Warning signs:** Stage A/B raise `TypeError` in production logs because `interpolate_percentile` got a tuple instead of a float.

### Pitfall 5: Byte-identical regression test left with stale goldens after refactor
**What goes wrong:** `tests/scripts/test_gen_global_percentile_cdf_unchanged.py` is the canary against accidental drift in `_build_metric_breakpoint_query`. After 94.2 it asserts against new methodology goldens. If Plan A lands without Plan B's golden regen, the test fails immediately. If Plan B lands without Plan A's builder rewrite, the test catches the inconsistency but the regen run produces nonsense values.
**Why it happens:** Plan A and Plan B touch separately-owned files (`canonical_slice_sql.py` vs `tests/scripts/...` + `app/services/global_percentile_cdf.py`) and look mergeable independently.
**How to avoid:** Land Plans A + B + C in the same wave (single commit, or contiguous commits on the same branch with no merge to main between them). The byte-identical test is the explicit gate — make it part of the same commit's CI run.
**Warning signs:** A `green` commit on a feature branch with the regression test failing on the next commit; CI on main goes red after a merge.

### Pitfall 6: `_build_per_bucket_sanity_query` diagnostic interpreted as the production CDF
**What goes wrong:** The regen report still emits per-bucket median/skew/kurtosis tables; a reader (or LLM) interprets them as "this is what the CDF measures." Under the pooled model the CDF is over deduped users, but the sanity-check is still bucketed by `elo_bucket` per cell-row — they're informative for distribution shape, but they do NOT correspond to what the production lookup sees.
**Why it happens:** Visual continuity with prior reports; the table format looks authoritative.
**How to avoid:** Either drop the diagnostic (CONTEXT.md allows it under "Claude's Discretion") OR emit it with a prominent header in the regen report explaining the distinction. Recommendation: keep it with the header — useful for sanity-checking distribution shape across rating buckets even though it no longer reflects the production CDF.
**Warning signs:** Plan 95 LLM payload work or future percentile-related plans cite the sanity-check numbers as authoritative.

### Pitfall 7: Backfill summary log columns sum to the wrong denominator
**What goes wrong:** Backfill emits "users-included" + "floor-rejected" per metric; under the pooled model the denominator is the total users in the app DB, not a per-cell membership. If the planner forgets to update the summary log, the numbers look right but represent a different quantity than the user expects.
**Why it happens:** The script delegates to the service helper; the summary counters are aggregated in the script, not the service.
**How to avoid:** Update the per-metric summary table emission in `scripts/backfill_user_percentiles.py` (the `_MetricSummary` class around line 296). The semantic shift: `skipped_no_canonical_games` becomes `skipped_below_pooled_floor` (single integer per user-metric, not per cell). For `score_gap`, also log the secondary constraint (whichever of `eg_n` / `non_eg_n` failed). For Stage B metrics still log `skipped_no_eval` separately (users with zero pending evals but also zero qualifying eval'd games).
**Warning signs:** Sum of `users-included + skipped_below_pooled_floor` doesn't equal the deduped app-user count (modulo Stage B's `skipped_no_eval` exclusion).

### Pitfall 8: SKILL.md left describing per-cell methodology while code is pooled
**What goes wrong:** The next LLM session (or a human reading) consults `.claude/skills/benchmarks/SKILL.md` to understand the percentile methodology and gets the 94.1 explanation. Worse: an LLM-generated patch references methods that no longer exist (e.g. `per_user_cte_for(metric, apply_floor=False)`).
**Why it happens:** SKILL.md changes feel optional/documentation-only and slip out of the wave; the rest of the system is "working" without them.
**How to avoid:** Make Plan D's SKILL.md rewrite a hard gate in the phase. The Chapter 1 rewrite is substantial (cohort construction + per-cell stratification → pooled + dedup); the Chapter 3 inclusion-floor table needs the 4 metrics updated to ≥30 (was 30/30/20/20 in 94.1). Flag the prior 94.1 phase context as superseded methodology at the top of those chapters.
**Warning signs:** A subsequent plan-checker or research agent quotes the 94.1 per-cell description as current.

### Pooled per-user CTE (single_user path, score_gap)

```sql
-- Source: composed by hand for this research from existing
-- canonical_slice_sql.py:185-252 (score_gap) + the new pooling recipe in
-- .planning/notes/per-user-percentile-pooled-redesign.md §Redesign.
-- Verified against existing schema columns: games.played_at, time_control_bucket,
-- white_rating, black_rating, result, user_color, user_id, rated, is_computer_game.

WITH selected_users AS (
  SELECT CAST(:user_id AS int) AS user_id
),
recent_capped AS (
  SELECT g.id, g.user_id, g.user_color, g.result,
         (CASE WHEN g.user_color = 'white' THEN g.white_rating
               ELSE g.black_rating END) AS user_rating,
         (CASE WHEN g.user_color = 'white' THEN g.black_rating
               ELSE g.white_rating END) AS opp_rating
  FROM (
    SELECT g.*,
           row_number() OVER (PARTITION BY g.user_id, g.time_control_bucket
                              ORDER BY g.played_at DESC) AS rn
    FROM games g
    JOIN selected_users su ON su.user_id = g.user_id
    WHERE g.rated AND NOT g.is_computer_game
      AND g.white_rating IS NOT NULL AND g.black_rating IS NOT NULL
      AND g.played_at >= NOW() - INTERVAL '36 months'
  ) g
  WHERE g.rn <= 1000
    AND abs(
      (CASE WHEN g.user_color = 'white' THEN g.white_rating ELSE g.black_rating END)
      - (CASE WHEN g.user_color = 'white' THEN g.black_rating ELSE g.white_rating END)
    ) <= 100
),
endgame_game_ids AS (
  SELECT game_id FROM game_positions
  WHERE endgame_class IS NOT NULL
  GROUP BY game_id HAVING count(*) >= 6
),
scored AS (
  SELECT
    rc.user_id,
    CASE WHEN (rc.result = '1-0' AND rc.user_color = 'white')
           OR (rc.result = '0-1' AND rc.user_color = 'black') THEN 1.0
         WHEN rc.result = '1/2-1/2' THEN 0.5
         ELSE 0.0
    END AS score,
    (eg.game_id IS NOT NULL) AS has_endgame
  FROM recent_capped rc
  LEFT JOIN endgame_game_ids eg ON eg.game_id = rc.id
),
per_user AS (
  SELECT
    user_id,
    avg(score) FILTER (WHERE has_endgame)     AS eg_score,
    avg(score) FILTER (WHERE NOT has_endgame) AS non_eg_score,
    count(*)   FILTER (WHERE has_endgame)     AS eg_n
  FROM scored
  GROUP BY user_id
  HAVING count(*) FILTER (WHERE has_endgame)     >= 30
     AND count(*) FILTER (WHERE NOT has_endgame) >= 30
)
SELECT (eg_score - non_eg_score) AS metric_value, eg_n AS n_games
FROM per_user
```

### Service-level Python (Plan C — `_compute_metric_for_user` rewrite sketch)

```python
# Source: simplified from existing app/services/user_benchmark_percentiles_service.py:93-128.

async def _compute_metric_for_user(
    session: AsyncSession,
    user_id: int,
    metric_id: CdfMetricId,
) -> tuple[float, int] | None:
    """Run the pooled-per-user CTE for one user and one metric.

    Returns (metric_value, n_games) where n_games is the per-metric count
    on the pooled set (binding-constraint count for score_gap; eval'd
    endgame entries for achievable; span count for section2_*). Returns
    None if the user is below the metric's ≥30 floor (CTE emits no row).
    """
    su_cte = selected_users_cte(source="single_user")
    pooled_cte = per_user_cte_for(metric_id, source="single_user")
    sql = (
        f"WITH {su_cte},\n{pooled_cte}\n"
        f"SELECT metric_value::float AS value, n_games::int AS n_games\n"
        f"FROM per_user_values"
    )
    row = (await session.execute(text(sql).bindparams(user_id=user_id))).fetchone()
    if row is None or row.value is None:
        return None
    return float(row.value), int(row.n_games)
```

Caller (`compute_stage_a` / `compute_stage_b`) destructures the tuple and passes `n_games` to `upsert_percentile`.

## Open Questions

1. **Per-user n_games semantics across the 4 metrics — single column, divergent meanings.**
   - What we know: D-9 keeps `n_games` as an INT column; CONTEXT.md "Claude's Discretion" specifies per-metric meaning (endgame count for `score_gap`, eval'd endgame-entry count for `achievable`, bucket span count for `section2_*`).
   - What's unclear: Whether to also log the secondary constraint (e.g. non-endgame count for `score_gap`) in a backfill summary, or to skip it.
   - Recommendation: Plan A docstring on `app/models/user_benchmark_percentile.py` documents the per-metric meaning; Plan F backfill summary emits the binding constraint count plus a per-metric extra column (e.g. `score_gap` row also logs non-endgame n as a secondary column in the log line, not the table).

2. **Does `_build_per_bucket_sanity_query` need its own pooled rewrite to keep the regen report useful, or is the diagnostic value preserved by running it over the pre-94.2 cell shape?**
   - What we know: CONTEXT.md "Claude's Discretion" allows keeping with a disclaimer or dropping.
   - What's unclear: A reader expects the per-rating-bucket median/skew/kurtosis to reflect the cohort the CDF was built from. If we keep it bucketed by `elo_bucket` per cell-row, it shows distribution shape across ELO but doesn't correspond to what the CDF measures. If we re-bucket each user once (e.g. by their median game-time ELO across the pool), the diagnostic loses its per-cell shape but matches the cohort.
   - Recommendation: Keep the current implementation (per cell-row distribution) with a prominent regen-report header. Skew/kurt by elo_bucket is the most useful sanity check for "did the pooled methodology accidentally amplify or hide a tail?" — it's not measuring the production CDF, but it answers a distinct question that's still valuable.

3. **Plan ordering inside the wave: A+B+C in one commit, or three contiguous commits on the same branch with squash-merge?**
   - What we know: D-3a says single PR; no incoherent-green-state.
   - What's unclear: GSD typically does one plan per commit. Three commits on the same branch keep the file-level diffs small and reviewable; one squash-merge to main means main never sees an intermediate state.
   - Recommendation: Three commits inside the same branch (one per plan A, B, C); branch-level CI runs the byte-identical test on each commit but the developer treats a red CI on commit A as expected (Plan B+C still in flight). PR description calls out the wave structure. Squash-merge to main produces a single coherent state. If the planner is uncomfortable with intermediate-red CI on the feature branch, fall back to a single combined commit.
