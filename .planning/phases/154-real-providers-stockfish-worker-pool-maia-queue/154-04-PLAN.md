<objective>
Close the reproduced pre-ready worker-init-error deadlock in `maiaQueue.ts` (CR-03): a
`{type:'error'}` message arriving before `isReady` becomes true currently drains only
`currentBatch` (null pre-ready), never touches the `pending` array, and never resets
`worker`/`isReady` — so every queued `policy()` call AND every future `policy()` call hangs
forever behind a permanently-dead worker. Also add the missing `worker.onerror` handler for
asynchronous script-load failures (WR-03), which `new Worker` never throws for synchronously.

Purpose: Phase 155's `mctsSearch.dispatchExpansion` `await`s `policy()` on every expansion.
Per CLAUDE.md project history, ONNX/WebGPU worker-init failures are a real field-observed
failure class (maia-worker.js posts `{type:'error'}` on session/model-load failure). A single
init error currently deadlocks the entire engine search on its first expansion, permanently.
Output: modified `maiaQueue.ts` + `maiaQueue.test.ts`. No signature changes, no new exports,
no wiring into any hook (still Phase 155).
</objective>
