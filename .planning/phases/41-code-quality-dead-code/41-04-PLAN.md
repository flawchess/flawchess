    **Pattern A — Array access where length is checked or index is logically guaranteed (zobrist.ts):**
    Use non-null assertion `!` when the index is known to be in bounds by algorithm invariant:
    ```typescript
    // Before: const val = arr[index];  // string | undefined
    // After:  const val = arr[index]!; // non-null assertion — index guaranteed in bounds
    ```
    Use this sparingly and ONLY when you can verify the index is guaranteed valid (e.g., iterating within bounds, after length check).

    **Pattern B — Record<string, T> key access:**
    Use nullish coalescing for fallback:
    ```typescript
    // Before: const label = TYPE_LABELS[key];       // string | undefined
    // After:  const label = TYPE_LABELS[key] ?? key; // fallback to key itself
    ```

    **Pattern C — Array destructuring with known length:**
    ```typescript
    // Before: const [first, second] = arr;  // first: T | undefined
    // After: if (arr.length < 2) return;
    //        const first = arr[0]!;
    //        const second = arr[1]!;
    ```

    **Pattern D — Local variable narrowing for computed property access:**
    ```typescript
    // Before: if (record[key]) { use(record[key]); } // still T | undefined inside
    // After:  const val = record[key]; if (val) { use(val); } // narrowed to T
    ```

    For each file, read it, identify the error locations, apply the appropriate pattern. NEVER use `// @ts-ignore` — use `!` assertions (when logically safe), `?? fallback` (when a default makes sense), or explicit undefined checks (when the value might genuinely be missing).

  <action>
    Fix all remaining `noUncheckedIndexedAccess` type errors in component and page files.

    **Openings.tsx (17 errors — highest count):**

    The dominant pattern is `Record` key access after a `.filter()` guard. TypeScript does not narrow computed property access through filter chains.

    Fix pattern (from RESEARCH.md Pitfall 2):
    ```typescript
    // Before:
    .filter((b) => wdlStatsMap[b.id] && wdlStatsMap[b.id].total > 0)
    .map((b) => ({ stats: wdlStatsMap[b.id] }))  // TS2532: possibly undefined

    // After: assign to local variable so TS can narrow
    .filter((b): b is typeof b & { __hasStats: true } => {
      const s = wdlStatsMap[b.id];
      return s !== undefined && s.total > 0;
    })
    .map((b) => {
      const stats = wdlStatsMap[b.id]!; // safe: filter guaranteed existence
      return { stats };
    })
    ```

    Or more simply:
    ```typescript
    .flatMap((b) => {
      const s = wdlStatsMap[b.id];
      if (!s || s.total <= 0) return [];
      return [{ ...b, stats: s }];
    })
    ```

    Use whichever pattern is cleaner for each specific case. The `!` non-null assertion is acceptable after a filter that provably checks the same key.
