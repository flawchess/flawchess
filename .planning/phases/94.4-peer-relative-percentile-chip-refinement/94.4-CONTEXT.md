# Phase 94.4: Peer-Relative Percentile Chip Refinement - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Pivot the percentile chip from global-pool ("vs all FlawChess benchmark users") to peer-relative ("vs same-rated-cohort users") and rescue the metrics that the v1 global-pool framing dropped. Two structural shifts plus one schema reshape:

1. **Cohort CDF family replaces `GLOBAL_PERCENTILE_CDF`.** Per-(metric, ELO anchor, TC) CDFs built as sliding windows of K=200 floor-passing users per 50-Elo anchor (anchors at 800, 850, …, 2400), ±150-Elo max window, suppress otherwise. Approximate count: ~8 metric families × ~33 anchors × 4 TCs ≈ ~1,050 CdfTable instances (minus suppressed anchors at extremes). Per the locked schema (see below) metrics are stored TC-orthogonally, so the CDF index is `(metric, anchor, tc)` not `(metric_with_tc_suffix, anchor)`.

2. **Per-(user, TC) rating anchor compute at import time.** Single anchor per (user, TC) — median game-time rating over the user's latest ~1000 games at that TC, Lichess wins precedence over chess.com. chess.com inputs convert via the **ChessGoals Table 2** snapshot (https://chessgoals.com/rating-comparison/) shipped as a hardcoded Python constant in `app/services/chesscom_to_lichess.py`. Daily chess.com TC returns None (chip suppresses for that user/TC). The live user's per-TC anchor rounds to nearest 50 and looks up directly — no interpolation step. Up to 4 anchors per user, stored in a new `user_rating_anchors` table.

3. **`user_benchmark_percentiles` schema reshape — TC becomes a column, not part of the metric ENUM.** PK changes from `(user_id, metric)` to `(user_id, metric, time_control_bucket)`. The 12 composite ENUM values introduced in Phase 94.3 (`time_pressure_score_gap_bullet`, …, `net_flag_rate_classical`) collapse to 3 generic ENUM values (`time_pressure_score_gap`, `clock_gap`, `net_flag_rate`); the 4 existing values (`score_gap`, `achievable_score_gap`, `section2_score_gap_conv`, `section2_score_gap_parity`) become per-TC (up to 4 rows per user per metric family). Every metric stores per-TC sub-percentiles directly; page-level ΔES chips render a single aggregated chip via game-count-weighted mean across the user's above-floor TC rows (the aggregation happens at API/render time, not write time). No data preservation discipline on the table — drop/recreate or TRUNCATE + ALTER, whichever is mechanically simplest; backfill restores all rows.

4. **Metric eligibility expanded to 8 families.** Endgame Score Gap, Achievable Score Gap, Parity Score Gap, Conversion Score Gap, **Recovery Score Gap (rescued)** + Time Pressure Score Gap, Clock Gap, Net Flag Rate. Recovery rescue accepts that within-cohort comparison normalises opponent-rating selection bias; opponent-selection (challenging up vs down) remains a residual confound, surfaced honestly via cohort-relative tooltip framing. Approximate chip surface: 5 page-level aggregated ΔES chips + 12 per-TC Time Pressure chips ≈ 17 visible chip positions.

5. **Chip UX shrink: `p23` bare-percentile pill.** No direction word — color band (red < p25 / neutral / green > p75) carries direction. **No flame icons** — the flame tier overlay was removed in commit `6766898c refactor(percentile-chip): remove flame icons and tier logic` and must NOT be reintroduced under peer-relative framing. Tooltip-only explanation. Visual emphasis stays on metric value + zone band.

6. **Tooltip body reworked under peer-relative framing.** Drops the v1 per-metric rating-correlation bullet (cohort absorbs ELO coupling); adds a rating-anchor disclosure bullet. 4-bullet contract preserved: (1) cohort framing + TC + anchor-rating phrasing; (2) recent-games basis (TC-scoped); (3) filter independence; (4) rating-anchor source disclosure (Lichess vs chess.com-converted-via-ChessGoals). Per-metric rating-correlation copy in `RATING_NOTE_BY_FLAVOR` retires.

**Locked carryover (no re-discussion):** sliding-window cohort protocol (K=200 / ±150 Elo / 50-Elo anchors), ChessGoals Table 2 + Table 1 conversion sources, single anchor per (user, TC), Lichess precedence, per-TC sub-percentile mechanism, Phase 94.3 per-TC SQL builder pattern, Stage A/B trigger pattern, backfill harness, `PercentileChip` component shell.

**Out of scope:** Phase 95 LLM payload + prompt rework. Per-type / per-class chips. Opening Insights chips. Per-(TC, ELO) cells within a cohort (cohort is the anchor; no further ELO split). Cross-platform refit from FlawChess prod data (future trigger — dual-platform N still too thin). Intra-chess.com forward-fit refit (deferred — included only if planning capacity allows).

## Implementation Decisions

### Storage schema reshape (locked — user-volunteered constraint)

- **D-01: `user_benchmark_percentiles` PK becomes `(user_id, metric, time_control_bucket)`.** `time_control_bucket` matches the column of the same name on `games`. Every metric family stores up to 4 per-TC sub-percentile rows. The 12 Phase 94.3 composite ENUM values (`time_pressure_score_gap_bullet`, `time_pressure_score_gap_blitz`, …, `net_flag_rate_classical`) retire; replaced by 3 generic ENUM values (`time_pressure_score_gap`, `clock_gap`, `net_flag_rate`). The 4 Phase 94.2 ENUM values (`score_gap`, `achievable_score_gap`, `section2_score_gap_conv`, `section2_score_gap_parity`) stay but become per-TC by adding the orthogonal column. Final ENUM: 7 values × up to 4 TCs = up to 28 rows per user (vs the prior 4 + 12 = 16 ENUM × 1 row shape).
- **D-02: No in-place migration discipline on `user_benchmark_percentiles`.** Drop and recreate the table (or TRUNCATE + ALTER), whichever is mechanically simplest in Alembic. The chip is non-critical product surface and `scripts/backfill_user_percentiles.py` is the canonical recovery path — data regenerates from games + benchmark CDF within one backfill rerun. ENUM reshape can be `DROP TYPE ... CASCADE` + `CREATE TYPE` rather than the per-value `ADD VALUE` discipline that Phase 94.3's TPCTL-02 forced. Planner picks the simplest Alembic shape; downgrade can be a no-op stub.
- **D-03: Page-level ΔES chips aggregate per-TC sub-percentiles at API/render time, not write time.** Storage keeps the 4 per-TC rows verbatim per (user, metric); the API layer computes the displayed chip's percentile via game-count-weighted mean across the user's above-floor TC rows. Aggregation is recomputable from storage. Per-TC sub-percentiles remain available for future per-TC chip surfaces or LLM payload extensions without a second backfill. (The Time Pressure family chips read their per-TC row directly — no aggregation.)
- **D-04: New `user_rating_anchors` table.** PK `(user_id, time_control_bucket)`. Columns: `anchor_rating INT`, `source_platform` (Postgres ENUM `anchor_source` with values `lichess` / `chesscom`), `n_games INT`, `computed_at TIMESTAMPTZ`. Up to 4 rows per user. Anchor compute runs at import time alongside (or inside) Stage A — independent of eval drain, so it can light up immediately. New Alembic migration adds the table + ENUM.

### Metric eligibility (locked)

- **D-05: All 8 metric families ship under peer-relative.** Endgame Score Gap, Achievable Score Gap, Parity Score Gap, Conversion Score Gap, **Recovery Score Gap (rescued)** + Time Pressure Score Gap, Clock Gap, Net Flag Rate. Per-TC dimensionality (4 TCs) gives ~28 storage rows per user (some suppressed below floor) and ~17 visible chip surfaces (5 aggregated page-level + 12 per-TC Time Pressure).
- **D-05a: Recovery rescue rationale and acceptance.** Under global, Recovery's d=0.95 inverted + opponent-confounded drove the v1 drop. Under peer-relative, same-rated-cohort users play same-rated opponents on average, so the rating component of opponent strength normalises naturally. Residual confound: opponent-selection (challenging up vs farming down) survives cohort comparison and reads honestly in the tooltip ("how you recover from material deficit vs other ~1600-rated players"). Accepted as a known tradeoff; no special suppression path.
- **D-05b: Conversion rescue rationale.** Sigmoid bias dissolves under peer-relative (every cohort member is also in the "up material" slice when Conversion is computed); the d=1.37 rating coupling absorbs into the cohort. Tooltip honesty does not require the v1 "tracks rating strongly" hedge. Standard chip framing applies.

### Chip pill format (locked)

- **D-06: Pill text is bare `p23` form — no direction word.** Color band (red < p25 / neutral p25-p75 / green > p75) carries direction. **No flame icons** — the prior p90/p95/p99 flame tier overlay was removed in commit `6766898c refactor(percentile-chip): remove flame icons and tier logic` and must NOT be reintroduced. Pill is ~4-5 chars wide, mobile-comfortable, ~24px tall. Replaces the v1 `Top 23%` / `Bottom 23%` direction-aware verbose form.
- **D-06a: `MIN_PERCENT` floor preserves at 1.** The shipped `Math.max(MIN_PERCENT, Math.round(value))` floor avoids `p0` at edge percentiles. `p99` is the ceiling, mirroring the CDF's p1..p99 breakpoint range.
- **D-06b: aria-label preserves a direction word for screen readers.** Visual chip is `p23`; aria-label reads "percentile 23, bottom of cohort" (or equivalent). Direction-blind screen readers get the disambiguating word that sighted users derive from the color band.

### Rating anchor + tooltip body (locked)

- **D-07: Tooltip body is 4 bullets under peer-relative framing:**
  1. **Cohort framing** — "Compared to other ~{anchor_rating}-rated players in {tc}." Anchor + TC inline so the cohort scope is concrete and the user can read why this number is what it is.
  2. **Recent-games basis** (TC-scoped) — "Your most recent 1000 rated games in {tc} over the last 36 months, vs opponents within ±100 Elo."
  3. **Filter independence** — "UI filters do not affect this percentile." (Verbatim from shipped contract.)
  4. **Rating-anchor disclosure** — "Anchored on your Lichess {tc} rating." OR "Anchored on your chess.com {tc} rating, converted to Lichess-equivalent via the ChessGoals snapshot."
- **D-07a: Per-metric rating-correlation copy retires.** `COPY_RATING_NOTE_*` constants and the `RATING_NOTE_BY_FLAVOR` map in `PercentileChip.tsx` are removed. Cohort absorbs ELO coupling, so the v1 hedge ("this metric is mostly independent of rating" / "tracks rating strongly") is no longer load-bearing. The `feedback_percentile_chip_tooltip_disclosure` memory updates to reflect the new 4-bullet contract.
- **D-07b: Aggregated-chip tooltip cohort framing.** Page-level ΔES chips (5 aggregated metrics) display one chip per metric but draw from up to 4 per-TC rows. Tooltip cohort framing names the active TC(s) honestly — exact copy is planner's call, but the principle is: "Compared to other ~{anchor}-rated players, aggregated across the time controls you play." The aggregation is honest about not being a single-cohort comparison without being verbose.

### Aggregation weight for page-level ΔES chips (locked)

- **D-08: Page-level ΔES chips aggregate per-TC sub-percentiles by raw game-count uniformly.** Every metric weights its per-TC sub-percentiles by the user's raw game count in that TC, regardless of the metric's per-TC denominator (endgame games for `score_gap`, endgame-entry games with `d_i` for `achievable`, span counts for Section-2). Trade-off vs the methodologically-stricter per-metric-floor-N weighting: simpler code, uniform across metrics, and the bias toward TCs the user plays most volume in is the realistic user expectation ("my main TC dominates my headline chip"). Section-2 metrics with sparse spans in a high-volume TC tilt toward that TC; that's a feature, not a bug — the user invested most of their play there.
- **D-08a: Below-floor TC rows drop out of the weighted mean.** If a user has below-floor span counts in classical for `section2_score_gap_conv`, that TC row simply isn't stored (D-09 of Phase 94.2); the aggregation sees no row for that (metric, classical) and the weight is implicitly zero. No `IF n_games < FLOOR THEN exclude` branch needed at the aggregation layer.
- **D-08b: Aggregation happens in the backend API response shaper, not in the database.** Read all 4 per-TC rows for the (user, metric), apply weighted mean in Python, ship one scalar percentile per ΔES row in the API response. Frontend doesn't see the per-TC breakdown for aggregated chips — only the rendered scalar. Per-TC chips (Time Pressure family) bypass aggregation entirely and read their (user, metric, tc) row directly.

### Cohort CDF artifact (locked from SEED-026 v2)

- **D-09: `GLOBAL_PERCENTILE_CDF` retires and is replaced by a `COHORT_PERCENTILE_CDF` family.** Index shape: `Mapping[CdfMetricId, Mapping[tuple[int, TimeControlBucket], CdfTable]]` — outer key is the 7-value metric ENUM (post-D-01 reshape), inner key is `(anchor_rating, tc)`, value is the 99-breakpoint CDF. Approximate ~1,050 CdfTable instances (7 metrics × ~33 anchors × 4 TCs minus suppressed anchors at extremes). Memory footprint: 99 floats × ~1,050 ≈ ~800 KB; trivial.
- **D-09a: `interpolate_percentile` signature widens to accept (metric, value, anchor, tc).** New helper signature: `interpolate_cohort_percentile(metric: CdfMetricId, value: float, anchor: int, tc: TimeControlBucket) -> float | None`. The existing `interpolate_percentile(metric, value)` helper retires (no callers after the cutover). Anchor rounds to nearest 50; if no CDF exists at that anchor (suppressed), returns None and the caller suppresses the chip for that (user, TC).
- **D-09b: CDF generation script.** Extend `scripts/gen_global_percentile_cdf.py` (preserve filename for git-history continuity; or rename to `gen_cohort_percentile_cdf.py` — planner's call) to emit the new index shape. Per (metric, anchor, tc): drop benchmark users below per-metric games floor, rank remaining users by `|user_anchor − anchor|`, take closest K=200; suppress if K-th distance > 150 Elo. Otherwise compute 99-breakpoint CDF. Regenerate `app/services/global_percentile_cdf.py` (file may be renamed to `cohort_percentile_cdf.py` — planner's call).

### Post-research user decisions (2026-05-26, locked)

- **D-11: K=200 / ±150-Elo cohort policy — strict default with regen-report deferred relaxation.** Plan E ships strict K=200 / ±150-Elo defaults. The regen report (a Plan E output) MUST surface suppression flags per (metric, anchor, TC) as a top-line table — not buried in an appendix. The user reviews the report and decides at that point whether to relax (e.g., K=150 or ±200 Elo) before Plan G's backfill runs. The decision is reversible by regenerating the CDF, so the constants live in `gen_global_percentile_cdf.py` as module-level constants (`COHORT_K_USERS_PER_ANCHOR`, `COHORT_MAX_WINDOW_ELO`) for one-line tuning. Per the benchmark DB probe in RESEARCH Pattern 4, only ~6 of 36 sampled (anchor, TC) cells pass strict K=200/±150; expect heavy suppression in bullet, blitz, and most classical anchors, with rapid mid-range as the only band with comfortable coverage.
- **D-12: Lichess-precedence rule — "any Lichess games in TC ⇒ Lichess source".** The anchor compute for (user, TC) selects Lichess as the source platform if the user has **any** Lichess games in that TC; otherwise chess.com (and that anchor converts to Lichess-equivalent via `convert_chesscom_to_lichess`). Simplest rule, mixes-two-ladders avoidance preserved, reversible via the model docstring + a one-line predicate change. Documented in `app/models/user_rating_anchors.py` docstring + the anchor compute service.
- **D-13: ENUM count locked to 8 values (resolves the CONTEXT D-01 prose vs D-05 list inconsistency).** `benchmark_metric` ENUM final values: `score_gap`, `achievable_score_gap`, `section2_score_gap_parity`, `section2_score_gap_conv`, `recovery_score_gap`, `time_pressure_score_gap`, `clock_gap`, `net_flag_rate`. 8 values × up to 4 TCs = up to 32 rows per user (vs CONTEXT D-01's earlier "7 × 4 = 28" approximation). The 8-value count agrees with D-05's metric-family list and the cohort CDF + storage + frontend touchpoints.

### Claude's Discretion

The user picked the 5 areas above (schema reshape, metric eligibility, pill format, anchor storage + tooltip, aggregation weight) plus the no-migration shortcut. Remaining HOW choices the planner resolves in PLAN.md without coming back to the user:

- **Exact column type for `anchor_rating`** — `INTEGER` matches `games.white_rating` / `games.black_rating`. SmallInt would suffice numerically but breaks symmetry with the games table. Recommend INTEGER.
- **`anchor_source` ENUM placement** — define alongside `benchmark_metric` in `app/models/user_benchmark_percentiles.py` or in a sibling `app/models/user_rating_anchors.py`. Recommend the sibling file (separation of concerns).
- **Anchor compute SQL** — `percentile_cont(0.5)` median over the same recent-1000-per-TC, 36-month-windowed CTE that powers the percentile compute. Reuse `_recent_capped_per_tc_cte` from `canonical_slice_sql.py`; add a sibling builder that emits the median rating rather than the per-metric scalar.
- **chess.com Daily handling** — confirmed: return None from `convert_chesscom_to_lichess` for Daily; caller suppresses the anchor for that user/TC. Daily is uncommon in the prod population (verify approximately during planning).
- **Aggregation copy detail** — exact wording of the page-level ΔES tooltip's "across the time controls you play" phrasing is planner's call within `feedback_percentile_chip_tooltip_disclosure` discipline. Single TC user: name the TC concretely ("…in rapid"). Multi-TC user: phrase honestly ("…across the time controls you play").
- **Tooltip per-TC anchor disclosure for aggregated chips** — page-level ΔES chips draw from up to 4 anchors. Recommend disclosing the dominant TC's anchor (highest game count) inline + footnoting "+ N other TCs" if multi-TC, rather than listing 4 anchors. Planner verifies it reads on mobile.
- **CDF script CLI** — match the existing `scripts/gen_global_percentile_cdf.py` CLI (snapshot date, dev/benchmark/prod target). No new args needed for the cohort dimension.
- **`PercentileChipFlavor` enum reshape** — collapse from 16 variants back to 7-8 (`score-gap`, `achievable`, `parity`, `conversion`, `recovery`, `time_pressure_score_gap`, `clock_gap`, `net_flag_rate`) + a `tc` prop on per-TC chips. Component-level rather than flavor-level TC routing. Planner picks the prop signature.
- **No flame icons (locked, not discretionary).** The flame tier overlay was removed from `PercentileChip.tsx` in commit `6766898c` and must NOT be reintroduced. The peer-relative pivot does not change this — color band carries direction, and tail emphasis is no longer expressed via icon. Planner MUST NOT add `Flame`/`flame` props, imports, or tier thresholds; tests MUST NOT assert any flame-tier rendering. The seed's earlier "preserve flames" recommendation is superseded.
- **Backfill operational details** — same defaults as Phase 94.3 (per-user commits, creation-asc ordering, per-100-users summary log line, dev-then-prod-after-sign-off).
- **Plan A's research SQL** — recommend the candidacy report extension is deferred (or trimmed); under peer-relative the per-metric rating-correlation column in the tooltip retires, so the candidacy table loses its load-bearing role. Researcher should sanity-check that the per-(metric, anchor, TC) cohort sizes meet K=200 at common anchors (1200, 1600, 2000) and flag the suppression edges (sub-900, super-2200) for the regen report.
- **Per-TC anchor recomputation on re-import** — when a user re-imports (chess.com / lichess monthly archive update), the anchor recomputes alongside the percentiles. Stage A trigger fires on import completion; anchor compute can be a sibling task or inline with `compute_stage_a`. Planner's call on inline vs sibling.
- **Tests** — Vitest assertions for `p23` rendering + new tooltip body; pytest for `convert_chesscom_to_lichess` table inversion + Daily-returns-None path; pytest goldens for the cohort CDF script (mirror the existing byte-identical regression pattern).
