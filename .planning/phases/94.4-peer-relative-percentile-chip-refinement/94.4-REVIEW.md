# Phase 94.4: Code Review Report

**Reviewed:** 2026-05-27
**Depth:** standard
**Files Reviewed:** 17 (core code touchpoints across Plans 01–08)
**Status:** issues_found

## Summary

Phase 94.4 ships a methodology pivot from global-pool to cohort-relative percentile
chips with three coordinated structural shifts: cohort CDF (8 metrics × ~33 anchors
× 4 TCs), per-(user, TC) rating anchor compute, and `user_benchmark_percentiles`
PK widening. The implementation is clean and tightly aligned with the CONTEXT
decisions (D-01..D-14). No critical bugs or security vulnerabilities surfaced —
the V4 keyword-only `user_id` discipline, V5 bindparams on user input, and
Sentry context-not-message pattern are all preserved across new code.

Findings are all WARNING or INFO tier: project-convention drift (`# type: ignore`
instead of CLAUDE.md's mandated `# ty: ignore`), one misleading documentation
comment that could fool future readers into restructuring data flow, a
tiebreak-comment in the dominant-TC picker that misstates "volume" ordering,
and a few minor robustness concerns in the gen script's URL handling and tmp
file cleanup.

## Warnings

### WR-01: `# type: ignore` violates CLAUDE.md project convention (must use `# ty: ignore[rule-name]`)

**Files:**
- `scripts/backfill_user_percentiles.py:416,417,458,459,803,804,805`
- `scripts/gen_global_percentile_cdf.py:637,638,639`

**Issue:** CLAUDE.md (ty compliance section) explicitly mandates
`# ty: ignore[rule-name]` and forbids the bare `# type: ignore` form, plus
requires the rule name and a brief reason. Phase 94.4 introduces 9 new
`# type: ignore[assignment|arg-type|index]` annotations in script code. ty
is enabled in CI between ruff and pytest and uses the `ty:` prefix; these
annotations are silently ignored by ty (no suppression) and only silence
upstream mypy-style checkers that aren't actually wired in. So they read
as "suppressed" but in practice they're either dead or hiding the wrong tool's
errors.

**Fix:** Replace each with the canonical project form, e.g.:
```python
target: Literal["dev", "prod"] = args.target  # ty: ignore[assignment] argparse returns Any
```
Where suppression is genuinely necessary (argparse `Namespace.*` is `Any`),
prefer narrowing via an explicit cast or a runtime validation check that ty
can follow:
```python
raw_target = args.target
assert raw_target in ("dev", "prod")
target: Literal["dev", "prod"] = raw_target
```

### WR-02: Misleading documentation: PercentileChip claims Net Flag Rate CDF is "inverted" but no inversion exists or is needed

**File:** `frontend/src/components/charts/PercentileChip.tsx:19-22, 125-127`
**Also:** `frontend/src/components/charts/EndgameTimePressureCard.tsx:249-251`

**Issue:** Multiple comments claim:
> "Net Flag Rate's inversion is handled at the Plan 04 CDF-gen layer — its
> cohort CDF is already inverted, so smaller raw values map to higher
> percentiles."

Inspection of `app/services/canonical_slice_sql.py::per_user_cte_net_flag_rate`
(lines 794-806) shows `metric_value = (timeout_wins − timeout_losses) / total`,
which is naturally higher-is-better (more opponent flags, fewer self flags is
good). The cohort CDF gen script (`scripts/gen_global_percentile_cdf.py`) does
NO inversion — it ranks raw `metric_value` ascending and builds an
ascending-percentile CDF. Higher raw value → higher percentile, which is what
"higher is better" wants. Same is true of `clock_gap` (more time advantage =
better) and every other metric in the 8-value enum.

So the chip rendering is correct, but the documentation creates a phantom
invariant. A future reader investigating the Net Flag Rate chip will look for
inversion logic that doesn't exist; if someone "fixes" the inversion thinking
it's missing, the chip semantics flip silently.

**Fix:** Replace the comments with the accurate statement:
```tsx
// All 8 metrics are naturally higher_is_better in their raw form:
//   - score_gap / achievable / section2_* / recovery: higher = outperformance
//   - time_pressure_score_gap / clock_gap: higher = more advantage
//   - net_flag_rate = (wins − losses) / total: higher = fewer self-flags
// No CDF-side inversion is needed; the CDF ranks ascending and the chip
// renders higher-percentile = green directly.
```

### WR-03: Misleading tiebreak comment in `pickDominantTcAnchor` — "bullet → blitz → rapid → classical" claimed as "volume-weighted ordering most players follow"

**File:** `frontend/src/lib/percentileAnchor.ts:29-31, 36`

**Issue:** The function iterates TCs in order `['bullet', 'blitz', 'rapid', 'classical']`
and breaks ties on `n_games` by keeping the earlier-iterated TC (strict `>`
comparison at line 41). The docstring claims this ordering "matches the
volume-weighted ordering most players follow." In practice most chess.com
and lichess users play more blitz and rapid than bullet (bullet is a smaller
audience). The tiebreak rule is fine as an arbitrary deterministic ordering,
but framing it as matching "what users play" is empirically misleading and
risks a future reviewer "fixing" the order without realizing the tests then
break.

Note: this is a tiebreak only — strict-greater comparison means ties favor the
earlier TC. Real users typically have one dominant TC with no tie, so the
visible behavior is determined by `n_games`. The misleading prose risks a
"fix" PR that doesn't change observable behavior but does invalidate snapshot
tests.

**Fix:** Replace the docstring rationale with the honest one:
```ts
// Tie-break: when two TCs have identical n_games, the earlier TC in the
// canonical order ['bullet', 'blitz', 'rapid', 'classical'] wins.
// Order matches the ROADMAP-canonical TC sweep used elsewhere; the actual
// dominant-TC behavior is driven by n_games strict-greater comparison.
```

### WR-04: `_compute_median_anchor_for_platform` and `_user_has_lichess_games_in_tc` open separate text queries from inside the long-running anchor compute loop, multiplying round-trips per user

**File:** `app/services/user_benchmark_percentiles_service.py:239-302, 350-402`

**Issue:** `compute_anchors_for_user` is a per-(user × TC) loop (4 iterations
per user) where each iteration issues:
1. `SELECT 1 FROM games ...` Lichess existence probe (one query)
2. `WITH selected_users_cte, _recent_capped_per_tc_cte, per_user_anchor SELECT ...`
   median anchor compute (one query)

Then `compute_stage_a` adds a third per-TC query for `score_gap`. So a single
user's Stage A path issues 12 sequential round-trips (3 × 4 TCs) inside one
session. Stage B adds 7 families × 4 TCs = 28 more for a total of 40 queries
per user in the steady-state import-completion path. Plan 06's recorded
runtime (`441s for 19 users` = ~23s/user) is consistent with this round-trip
cost.

This is out of scope for v1 review (performance), but flag it as a v1
constraint surface: at scale (Phase 91 stress test target of ~11 g/s with
two concurrent platforms), Stage A latency per user could dominate
post-import wall time. The query shape is amenable to consolidation — a
single `SELECT user_id, time_control_bucket, has_lichess_games, anchor_rating, n_games FROM ...`
could replace the 8 anchor-path queries per user.

**Fix (optional / future):** Consolidate `_user_has_lichess_games_in_tc` and
`_compute_median_anchor_for_platform` into a single 4-TC query that returns
`(tc, has_lichess, lichess_anchor, lichess_n, chesscom_anchor, chesscom_n)`
per TC; choose source platform in Python from the consolidated result. This
collapses 8 round-trips to 1 per user. Not required for correctness; flagged
because the steady-state cost is non-trivial and the read pattern is uniform.

### IN-03: `tmp.py.tmp` file leaks if ruff format/check fails in `_write_module`

**File:** `scripts/gen_global_percentile_cdf.py:551-559`

**Issue:** `tmp = OUTPUT_MODULE_PATH.with_suffix(".py.tmp")` is written before
`subprocess.run(["ruff", "format"], check=True)`. If ruff fails (non-zero
exit), `CalledProcessError` propagates and the tmp file is left behind. Next
run re-creates it but the orphan adds noise to git status.

**Fix:** Wrap the ruff calls in try/finally to remove tmp on failure:
```python
try:
    subprocess.run(["uv", "run", "ruff", "format", str(tmp)], check=True)
    subprocess.run(["uv", "run", "ruff", "check", "--fix", str(tmp)], check=True)
    tmp.replace(OUTPUT_MODULE_PATH)
except Exception:
    tmp.unlink(missing_ok=True)
    raise
```

### IN-04: `_db_url` rebuilds netloc by concatenation without URL-encoding password

**Files:**
- `scripts/gen_global_percentile_cdf.py:217`
- `scripts/backfill_user_percentiles.py` (`_db_url` equivalent path)

**Issue:** `new_netloc = f"{parsed.username}:{parsed.password}@localhost:{port}"`
assumes password has no URL-reserved characters (`@`, `:`, `/`, etc.). If the
operator's dev DB password contains `@`, the resulting URL parses incorrectly
and asyncpg connects to the wrong host. Same pattern in the prod tunnel path.

**Fix:** Use `urllib.parse.quote(parsed.password, safe="")` when constructing
the new netloc. For local Docker dev DB this is unlikely to bite (passwords
are alphanumeric), but it's a small change to harden the path.

### IN-05: `_invert_intra_tc` claims monotone non-decreasing assumption but doesn't validate it

**File:** `app/services/chesscom_to_lichess.py:317-325, 336-339`

**Issue:** The docstring says "Table 1 columns are monotone non-decreasing in
chess.com Blitz (modulo the one tied row at 1100/1150)". `bisect_left` only
works correctly if this invariant holds. The snapshot data appears to honor
the invariant per inspection, but the code never asserts it — if a future
re-fetched snapshot has a non-monotone column (data correction or new
methodology), `bisect_left` silently returns a wrong index and the
interpolation produces a nonsense Blitz equivalent.

**Fix:** Add a `_validate_monotone` module-load assertion (executes once at
import time) that scans each Table 1 column for monotone non-decrease, e.g.:
```python
def _validate_monotone_columns() -> None:
    for col in ("bullet", "rapid"):
        vals = [CHESSCOM_INTRA_TC[k][col] for k in _CHESSCOM_INTRA_KEYS]
        for i in range(1, len(vals)):
            if vals[i] < vals[i - 1]:
                raise RuntimeError(
                    f"CHESSCOM_INTRA_TC[{col!r}] not monotone at idx {i}: "
                    f"{vals[i - 1]} → {vals[i]}; bisect_left would mis-invert"
                )
_validate_monotone_columns()
```

### IN-06: `cast(Literal[...], ...)` runtime no-op masks invalid backend values

**Files:**
- `app/services/endgame_service.py:2054, 2778, 2779`

**Issue:** `cast(Literal["bullet", "blitz", "rapid", "classical"], tc)` is a
typing hint with no runtime check. If a DB row ever returns a TC value
outside the closed set (mis-migration, raw SQL bypass, future bug), the cast
silently passes through and the downstream consumer fails opaquely. The
casts are at internal trust boundaries (TC values coming from the games
table), so the risk is low, but the pattern teaches readers to use casts
instead of validation.

**Fix (optional):** Replace with an assertion at the boundary:
```python
assert tc in ("bullet", "blitz", "rapid", "classical"), f"unknown TC: {tc!r}"
tc_literal: TimeControlBucket = tc
```
Negligible runtime cost; immediate failure on mis-typed data; ty narrows
correctly after the assert.
