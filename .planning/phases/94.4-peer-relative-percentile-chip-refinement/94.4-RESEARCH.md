# Phase 94.4: Peer-Relative Percentile Chip Refinement — Research

**Researched:** 2026-05-26
**Domain:** Cohort-CDF replacement of the global-pool CDF + per-(user, TC) rating anchor compute + per-(metric, anchor, TC) lookup + per-TC sub-percentile aggregation for 4 page-level ΔES metrics + chip UX shrink to `p23` pill + tooltip body rework + ChessGoals empirical conversion module + `user_benchmark_percentiles` schema reshape
**Confidence:** HIGH for code touchpoints (every file in scope read at the post-94.3 state); HIGH for cohort-size feasibility (benchmark DB probed at 9 anchors × 4 TCs); HIGH for ChessGoals snapshot extraction (Table 1 + Table 2 fetched verbatim from source URL on 2026-05-26); MEDIUM for per-(metric, anchor, TC) cell counts under per-metric floors (probed only `score_gap`-style endgame ≥30 floor as a proxy; eval-dependent metric cells will be smaller subsets).

## Summary

Phase 94.4 is a methodology pivot, not an additive feature. Three structural shifts cooperate inside one wave so no green commit holds an incoherent state:

1. **Cohort CDF replaces global CDF.** `GLOBAL_PERCENTILE_CDF: Mapping[CdfMetricId, CdfTable]` becomes `COHORT_PERCENTILE_CDF: Mapping[CdfMetricId, Mapping[tuple[int, TimeControlBucket], CdfTable]]` — three keys deep. The 50-Elo sliding-window protocol (K=200 floor-passing users / ±150 Elo max window) drives per-(metric, anchor, TC) generation; suppressed anchors emit no entry. `interpolate_percentile(metric, value)` becomes `interpolate_cohort_percentile(metric, value, anchor, tc) -> float | None`.
2. **Per-(user, TC) rating anchor compute** at import time. A new `user_rating_anchors` table stores up to 4 anchors per user (median rating over the latest ~1000 games at that TC, ±100 footing, 36-month window). Lichess wins precedence when a user has both platforms in a TC. chess.com inputs convert via a hardcoded snapshot of ChessGoals Table 2 (Lichess-equivalent) preceded by Table 1 inversion (intra-chess.com TC offset) for chess.com Bullet/Rapid sources. **chess.com Daily** appears in storage as `time_control_bucket='classical'` (per `app/services/normalization.py:57-59`) — so the anchor compute must look at the raw `games.time_control_str` to detect `'1/%'` patterns and suppress those games from the anchor median when the player's classical bucket is dominated by Daily (or skip them at the query level — see Pitfall 11). Stage A trigger fires the anchor compute alongside `score_gap` — eval-independent, so it lights up immediately.
3. **`user_benchmark_percentiles` reshapes from `(user_id, metric)` to `(user_id, metric, time_control_bucket)`.** The 12 composite TC-suffixed ENUM values from 94.3 (`time_pressure_score_gap_bullet`, …) collapse to 3 generic values (`time_pressure_score_gap`, `clock_gap`, `net_flag_rate`); the 4 Phase 94.2 values stay but become per-TC; a new `recovery_score_gap` is added. Final ENUM = 8 values. Page-level ΔES chips aggregate per-TC sub-percentiles via game-count-weighted mean **at the API shaper layer** (D-08, D-08b), not at write time — storage keeps per-TC rows verbatim. The migration drops and recreates the table + uses `DROP TYPE benchmark_metric CASCADE` + `CREATE TYPE` (D-02); backfill restores all rows.

Frontend shrinks the chip from `Top 23%` / `Bottom 23%` to `p23`, retires `RATING_NOTE_BY_FLAVOR`, and reshapes the 16-flavor enum back to ~8 + a `tc` prop on per-TC chips. **No flame icons** — the flame tier overlay was removed post-94 in commit `6766898c` and remains forbidden.

**Primary recommendation:** Adopt the CONTEXT.md A–H plan boundary verbatim, with three modifications: (1) Plan A's deliverable is now the ChessGoals snapshot extraction + cohort-size sanity table (NOT the per-(metric × TC) Cohen's d candidacy extension — that retires under peer-relative per the CONTEXT "Claude's Discretion" note); (2) Plans D, E, and F form the atomic Wave 1 cutover trio (mirroring 94.2 D-3a / 94.3 Wave 1) — `canonical_slice_sql.py` per-TC ΔES builders + cohort CDF regen + schema/service/aggregation rewrite must land together; (3) Add an explicit median-rating CTE builder in `canonical_slice_sql.py` so the anchor compute and the percentile compute share infrastructure (already 60% there via `_recent_capped_per_tc_cte` — just needs a sibling builder that emits `percentile_cont(0.5)` on `user_elo_at_game`).

### Pattern 1: Cohort CDF generation — per-(metric, anchor, TC) sliding window

**What:** Replace the single 1-level loop in `scripts/gen_global_percentile_cdf.py` with a 3-level loop over `(metric, anchor, tc)` triples. For each triple:

1. Run the per-TC builder for `metric` (one of the 4 new per-TC ΔES builders OR one of the 3 existing 94.3 per-TC builders). Result: per-user `(user_id, metric_value, n_games)` rows.
2. JOIN against `per_user_cte_median_anchor(tc, source="benchmark", snapshot_date=...)`. Result: each user gets a `(metric_value, anchor_rating, n_games)` triple.
3. Filter to users with `n_games >= floor` (per-metric floor — unchanged).
4. Rank by `|user_anchor − target_anchor|`, take closest K=200.
5. If K-th distance > 150 Elo: suppress (do not emit a CDF entry for this `(metric, anchor, tc)`).
6. Else compute 99-breakpoint CDF via `percentile_cont` over the K users' metric values.

**SQL shape (sketch):**
```sql
WITH selected_users AS (...),  -- deduped benchmark cohort
{per_user_cte_score_gap_tc('bullet', source='benchmark', snapshot_date='2026-03-31')},
{per_user_cte_median_anchor('bullet', source='benchmark', snapshot_date='2026-03-31')},
joined AS (
  SELECT puv.metric_value, pua.anchor_rating, puv.n_games
  FROM per_user_values puv
  JOIN per_user_anchor pua ON pua.user_id = puv.user_id  -- requires user_id to be exposed; see Pitfall 1
),
ranked AS (
  SELECT *, abs(anchor_rating - :anchor) AS dist,
         row_number() OVER (ORDER BY abs(anchor_rating - :anchor)) AS rk
  FROM joined
),
top_k AS (
  SELECT metric_value FROM ranked WHERE rk <= 200
),
gate AS (
  SELECT max(dist) AS kth_dist FROM ranked WHERE rk = 200
)
SELECT
  percentile_cont(ARRAY[0.01, 0.02, ..., 0.99]::double precision[])
    WITHIN GROUP (ORDER BY metric_value) AS breakpoints,
  count(*) AS n_users,
  (SELECT kth_dist FROM gate) AS kth_distance_elo
FROM top_k
```

**Per-anchor suppression check** happens in Python after fetching `(breakpoints, n_users, kth_distance_elo)`: if `kth_distance_elo > 150`, skip the entry. If `n_users < 200`, also skip (could only happen if the per-metric floor is more restrictive than the K floor — defensive).

**Pitfall — `per_user_values` no longer exposes `user_id`:** The existing per-TC builders (`per_user_cte_time_pressure_score_gap`, etc.) emit `per_user_values(metric_value, n_games)` — `user_id` was stripped in Phase 94.2 D-5. To join the metric value with the anchor on the same user, the builder must be either (a) rewritten to expose `user_id` in `per_user_values`, OR (b) the JOIN happens at the per-user-aggregate level before the `per_user_values` projection. **Recommendation:** widen the per-TC builders to project `user_id` in `per_user_values` (adds one column to every existing builder — minor change, internal-only). This unblocks the median-anchor JOIN without forcing a different join shape per metric.

**Migration tactic for the existing 94.3 builders:** the 3 existing 94.3 per-TC builders (`per_user_cte_time_pressure_score_gap`, `per_user_cte_clock_gap`, `per_user_cte_net_flag_rate`) already strip `user_id`. Add `user_id` projection to all 7 per-TC builders (3 existing + 4 new) inside `per_user_values`. The downstream `scripts/gen_global_percentile_cdf.py` Phase 94.3 query (lines 310-317) currently does `SELECT percentile_cont(...) FROM per_user_values` — it ignores `user_id` because there's only one row per user; the column is just a no-op extra projection at the percentile_cont level. Same for the per-user lookup path: `_compute_metric_for_user` does `SELECT metric_value::float, n_games::int FROM per_user_values LIMIT 1` — adding `user_id` to the SELECT list doesn't break either. **Net effect:** widening `per_user_values` to include `user_id` is structurally safe.

**Output shape:**

```python
def per_user_cte_median_anchor(
    tc: TimeControlBucket,
    *,
    source: Literal["benchmark", "single_user"],
    snapshot_date: date | None = None,
    min_games: int = MEDIAN_ANCHOR_MIN_GAMES,
) -> str:
    """Per-(user, TC) median rating anchor over the recent-1000-per-TC × 36-month pool.

    Output: per_user_anchor(user_id, anchor_rating INT, n_games INT)
    HAVING: count(*) >= min_games

    `anchor_rating` = `percentile_cont(0.5) WITHIN GROUP (ORDER BY user_elo_at_game)::int`
    -- discrete int median; ties broken by Postgres semantics; rounds to nearest int.

    The caller subsequently rounds anchor_rating to the nearest 50 for CDF lookup.

    Lichess-precedence note: this builder reads `user_elo_at_game` from games.{white,black}_rating
    pooled across PLATFORMS by default. For per-(user, TC) precedence per CONTEXT D-04, the
    caller passes a sibling `selected_users_cte` variant that restricts to Lichess games when
    the user has any Lichess games in this TC; falls back to chess.com games otherwise.

    Daily-handling note: chess.com Daily games carry chess.com Daily ratings (not Live Blitz);
    these are bucketed `classical` (per normalization.py:57-59) but have no ChessGoals
    conversion. The WHERE clause `AND NOT (g.platform = 'chess.com' AND g.time_control_str LIKE '1/%%')`
    drops them from the anchor compute. See Pitfall 11.
    """
    _ = source
    return f"""{_recent_capped_per_tc_cte(snapshot_date, tc)},
recent_capped_no_daily AS (
  SELECT rc.* FROM recent_capped rc
  JOIN games g ON g.id = rc.id
  WHERE NOT (g.platform = 'chess.com' AND g.time_control_str LIKE '1/%%')
),
per_user_anchor AS (
  SELECT
    user_id,
    percentile_cont(0.5) WITHIN GROUP (ORDER BY {_user_elo_at_game_expr()})::int AS anchor_rating,
    count(*) AS n_games
  FROM recent_capped_no_daily rc
  JOIN games g ON g.id = rc.id  -- needed to access white_rating/black_rating
  GROUP BY user_id
  HAVING count(*) >= {min_games}
)"""
```

**Open question: should the median be over *all* games or only Lichess games when a user is dual-platform?** CONTEXT D-04 says Lichess wins precedence per TC. **Recommendation:** the precedence is at the PLATFORM-SOURCE level, not the per-game level. The anchor compute reads from whichever platform contributes the majority of recent-1000 games per (user, TC); if Lichess has ≥1 game, the anchor is `lichess`-source and only Lichess games count; otherwise `chesscom`-source. The Lichess-precedence implementation lives in the calling Python code, not in the SQL builder — the builder takes a `platform: Literal["lichess", "chesscom"]` parameter and restricts accordingly. This keeps the SQL simple and the precedence policy explicit in Python.

```python
def _aggregate_per_tc_percentile(
    per_tc_rows: Mapping[TimeControlBucket, PercentileRow] | None,
) -> float | None:
    """Game-count-weighted mean of per-TC percentiles per CONTEXT D-08.

    Used for the 5 page-level aggregated ΔES chips (score_gap, achievable_score_gap,
    section2_score_gap_conv, section2_score_gap_parity, recovery_score_gap).

    Weighted-mean formula: Σ(percentile_i × n_games_i) / Σ(n_games_i), where the

    weighted_sum = 0.0

    for tc_row in per_tc_rows.values():
        if tc_row.percentile is None:

        # n_games per CONTEXT D-08: raw game count uniformly across metrics.
        # The PercentileRow dataclass must expose n_games — see Pattern 5

        weighted_sum += tc_row.percentile * tc_row.n_games

    if weight_sum == 0:
        return None
    return weighted_sum / weight_sum
```

## Common Pitfalls

### Pitfall 1: `per_user_values` strips user_id — needs widening for the anchor JOIN

**What goes wrong:** The existing 94.2/94.3 per-user CTE builders emit `per_user_values(metric_value, n_games)` and explicitly drop `user_id` (D-5 from Phase 94.2). Phase 94.4 needs to JOIN per-user metric values with per-user anchor values for the cohort CDF build — without `user_id` in the projection the JOIN cannot happen at the per-user grain.

**Why it happens:** Phase 94.2 stripped `user_id` because the consumer (`percentile_cont` for the global CDF) ignored it. Phase 94.4's cohort CDF needs to filter per-user metric values to the K=200 users closest to the anchor — a per-user operation requiring `user_id`.

**How to avoid:** Plan E (CDF rewrite) widens ALL per-TC builders (the 3 existing 94.3 ones + the 4 new 94.4 ones) to project `user_id` in `per_user_values`. The downstream `_compute_metric_for_user` does `LIMIT 1 SELECT metric_value, n_games` — adding `user_id` to the projection is a no-op at that consumer. The CDF generator stops using `percentile_cont` directly on `per_user_values` and instead uses the K=200 sliding-window protocol (Pattern 1).

**Warning signs:** The CDF build query needs to "see" individual users (one row per user) but the existing CTE collapses to a single percentile_cont aggregate. The test `tests/services/test_canonical_slice_per_user_vs_benchmark_parity.py` should be extended to assert that `per_user_values` exposes `user_id` on both `source="benchmark"` and `source="single_user"` paths.

### Pitfall 2: ChessGoals Daily handling — Daily is bucketed `classical` in storage

**What goes wrong:** Per `app/services/normalization.py:57-59`, chess.com Daily games are normalized to `time_control_bucket='classical'` in storage. The naive Daily-handling logic ("convert chess.com ratings except for Daily TC") fails silently because the source TC at the anchor compute level is always one of {bullet, blitz, rapid, classical} — there is no "daily" bucket to filter on.

**Why it happens:** chess.com tracks 4 separate rating ladders (Bullet, Blitz, Rapid, Daily); Daily ratings are NOT comparable to Lichess ratings (different rating system, different time scale). When the median-anchor CTE reads `games.white_rating` / `games.black_rating` for classical-bucketed games, chess.com Daily games contribute their Daily ratings to the median — but ChessGoals provides no Daily → Lichess conversion.

**How to avoid:** The median-anchor builder (`per_user_cte_median_anchor`) must drop chess.com Daily games at the SQL level. Detection: `g.platform = 'chess.com' AND g.time_control_str LIKE '1/%'` (the `1/SECONDS` format is the chess.com Daily PGN convention per `normalization.py:37`). Add this exclusion to `recent_capped` before the median is taken. Pattern 6 shows the corrected SQL.

**Subtler issue:** what if a user has 100 chess.com classical games (Daily) and 0 lichess classical and 0 chess.com Live Classical (rare but possible)? The classical-bucket anchor compute would emit no row (all games dropped) → no anchor → no chip for that user's classical TC. That's the correct behavior (D-04: anchor unavailable → suppress chip), but the planner should call this out in Plan E's task notes.

**Warning signs:** Backfill summary shows unexpectedly high suppression for classical-bucket chips on users with chess.com-only history.

### Pitfall 3: Atomic-cutover sequencing — Plans D + E + F + G must coordinate

**What goes wrong:** Plan E (CDF regen) lands on `main` with the new 3-level COHORT_PERCENTILE_CDF, but Plan F (storage + service rewrite) is still in review. The deployed backend imports `interpolate_percentile` (signature unchanged in Plan E only if you keep the old name) but the registry literal is the new 3-level shape — runtime TypeError at the first chip lookup.

**Why it happens:** The cohort-relative pivot touches 3 coupled surfaces — the SQL pool definition (Plan D), the CDF artifact shape (Plan E), and the per-user lookup + storage shape (Plan F). All three must agree on the new shape simultaneously.

**How to avoid:** Plans D + E + F land together on the same feature branch in sequential commits. The byte-identical regression test in `tests/scripts/test_gen_global_percentile_cdf_unchanged.py` is EXPECTED RED between Plan D and Plan E (the goldens regenerate at the end of Plan E). Squash-merge of the branch to main produces a single coherent commit. **This mirrors Phase 94.2 D-3a and Phase 94.3 Wave 1.** Plan G (backfill) lands in Wave 2 after Wave 1 merges to main — necessarily after, because the backfill writes the new shape that only the new code can produce.

**Plan H (frontend) is independent** — it can land before or after the Wave 1 cutover. Before: the chips render with the old API contract (no `tc` prop, etc.) but the chip suppresses gracefully when the API returns `null`. After: chips render with the new API contract.

**Warning signs:** Pre-merge CI status on the feature branch shows red on the regen test in the middle of the branch (expected) but green at the tip.

### Pitfall 4: K=200 / ±150 Elo is more restrictive than the seed assumes

**What goes wrong:** Per Pattern 4's probe, only 6 of 36 sampled (anchor, TC) cells pass K=200 / ±150 Elo. The seed's "passes at common anchors" assumption is empirically too optimistic on the current benchmark cohort. If the planner ships strict K=200 / ±150 without surfacing the suppression rate, users at most rating bands see no chip on most metrics → product surface becomes a near-empty wasteland.

**Why it happens:** The benchmark cohort is moderately sized; per-(TC, anchor) density at the 50-Elo granularity is below 200 in many bands. Rapid TC is the only TC with comfortable coverage across mid-range anchors; the other three TCs barely sustain K=200 at any anchor.

**How to avoid:** Plan E's regen report MUST surface per-(metric, anchor, tc) suppression flags as a top-line table — not buried in an appendix. The user reviews the report before Plan G's backfill runs and chooses whether to:
- Ship strict K=200 / ±150 (accepts heavy suppression).
- Relax to K=150 / ±200 (broader coverage, slightly noisier breakpoints).
- Two-stage relaxation (default K=200 / ±150, fall back to K=150 / ±200).

**Recommendation:** ship the regen report with the strict default; offer the relaxation path as an inline planning question that the user resolves before Plan G executes. Either way the design is reversible by regenerating the CDF — Plan E is the bottleneck, not a commit.

**Warning signs:** Plan E's regen report shows >80% suppression in a TC × metric × anchor combination.

### Pitfall 5: Lichess-precedence policy lives in Python, not SQL

**What goes wrong:** The CONTEXT D-04 "Lichess wins precedence per TC" sounds like a SQL clause (`WHERE platform = 'lichess' OR NOT EXISTS (SELECT 1 ... WHERE platform = 'lichess')`). Implementing precedence per-game is structurally wrong: a user with 500 Lichess blitz + 500 chess.com blitz should get an anchor based on Lichess only, not the union.

**Why it happens:** The seed describes a per-(user, TC) PRECEDENCE — when both platforms exist, use only one. Per-game logic mixes the two.

**How to avoid:** The median-anchor builder takes `platform: Literal["lichess", "chesscom"]` as a parameter. Python-side wrapper:
```python
for tc in TC_BUCKETS:
    # Check if user has Lichess games in this TC
    has_lichess = await _has_lichess_games_in_tc(session, user_id, tc)
    source_platform = "lichess" if has_lichess else "chesscom"
    anchor_sql = per_user_cte_median_anchor(tc, source_platform=source_platform, ...)
    # If chess.com: read raw rating, then convert via chesscom_to_lichess
    # If lichess: read rating directly as anchor
    # Store one row per (user, TC) with source_platform tag
```

**Warning signs:** A dual-platform user's anchor for a TC reads as an obvious average of their two platforms instead of the Lichess value (verifiable by hand against a test fixture).

### Pitfall 7: Drop/recreate migration deletes dev data silently

### Pitfall 8: Cohort CDF generation cost — 1,050 queries vs 8

**What goes wrong:** Phase 94.3's CDF generation ran 16 queries against the benchmark DB. Phase 94.4's cohort CDF generation runs ~1,050 queries (8 metrics × ~33 anchors × 4 TCs, minus suppressed). Each query computes per-user metric + per-user anchor + K=200 ranking + percentile_cont. If each query takes 500ms, the full regen is ~9 minutes.

**Why it happens:** The 3-level loop multiplies the query count by ~130×. Each query is harder than the 94.3 one (extra JOIN against per_user_anchor + WHERE rk <= 200).

**How to avoid:** One option — fetch all per-user `(user_id, metric_value, n_games, anchor_rating)` tuples ONCE per (metric, tc) into Python, then do the K=200 ranking + percentile breakpoints in Python for each anchor. This collapses to ~32 (metric × tc) queries instead of ~1,050. Memory: ~1,500 users × 8 metrics × 4 TCs × ~50 bytes/tuple ≈ 2.5MB — trivial.

**Recommendation:** in-Python K=200 ranking + breakpoint computation is the cleaner shape. The SQL builder emits `(user_id, metric_value, n_games, anchor_rating)` per (metric, tc); Python iterates anchors in 50-Elo steps and runs `sorted(users, key=lambda u: abs(u.anchor - target))[:200]` + `numpy.percentile(values, range(1, 100))`. 32 queries × ~500ms = 16s total. Much better than the 9 minutes.

**Warning signs:** Plan E's CDF regen takes >5 minutes; operator complains.

### Pitfall 9: Backfill order — anchors must compute before percentiles

**What goes wrong:** Plan G's backfill calls `compute_stage_a(user_id)` which (extended) computes the anchor AND the percentile. If the percentile compute happens before the anchor is written, the anchor lookup returns None → percentile is None → row written with `percentile=NULL` but `value` is meaningful. Chip suppresses.

**Why it happens:** Inside `compute_stage_a`, the anchor compute must commit (or at least be in the same transaction) before the percentile lookup runs.

**How to avoid:** The Stage A extension structure:
```python
async def compute_stage_a(user_id):
    async with maker() as session:
        # 1. Compute and upsert per-(user, TC) anchors first.
        anchors = await compute_anchors_for_user(session, user_id)
        # 2. For each TC where an anchor exists, compute per-TC score_gap percentile.
        for tc, anchor in anchors.items():
            result = await _compute_metric_for_user_per_tc(session, user_id, "score_gap", tc)
            if result is None: continue
            value, n_games = result
            percentile = interpolate_cohort_percentile("score_gap", value, anchor, tc)
            await upsert_percentile(session, user_id=user_id, metric="score_gap",
                                    time_control_bucket=tc, value=value, n_games=n_games,
                                    percentile=percentile, cdf_snapshot=date.today())
        await session.commit()
```

The anchor compute happens in-memory (no commit needed before the lookup); both anchor and percentile writes commit together at the end.

**Warning signs:** Backfill log shows `percentile=NULL` rows on the first run; second run "fixes" them (because anchor was written by the first run).

### Pitfall 10: Frontend type codegen sequencing

**What goes wrong:** Plan F's schema change (no new fields per Pattern 5 — the existing `*_percentile` fields ship aggregated scalars) doesn't actually require a type regen for the chip slots. BUT the API may add an `anchor_rating` / `anchor_source` block (per Pattern 7) that the frontend needs for tooltip disclosure. If the planner adds that block in Plan F and Plan H imports it without regen, TS build fails.

**Why it happens:** Type codegen runs from FastAPI OpenAPI; new schema fields don't appear in `frontend/src/types/endgames.ts` until regen.

**How to avoid:** Plan F task list ends with `npm run generate-types` (or equivalent — verify via `frontend/package.json`). Commit the regenerated file. Plan H opens with "confirm new fields exist in types/endgames.ts."

**Warning signs:** `Property 'anchor_rating' does not exist on type 'EndgameOverviewResponse'` during Plan H's TS compile.

### Pitfall 11: chess.com Daily classical anchor — anchor compute may emit no row for some users

**What goes wrong:** A user with 100% chess.com Daily games in classical bucket → anchor compute drops all games → `HAVING count(*) >= 30` fails → no row in `user_rating_anchors` for (user, classical). All classical-bucket chips suppress.

**Why it happens:** Daily games carry chess.com Daily ratings, which have no ChessGoals conversion. The seed's design accepts this — "Daily returns None and the caller suppresses."

**How to avoid:** This is correct behavior. Document explicitly in the model docstring + tooltip copy (the tooltip should say "No classical anchor available for your account — chess.com Daily ratings are not compared cross-platform"). Plan H tooltip update covers this edge case.

**Warning signs:** Users with chess.com-only history report "I have no classical chip." Backfill log surfaces this as `skipped_no_anchor` count per TC.

```python
def _build_per_user_with_anchor_query(metric_id, tc, *, snapshot_date) -> str:
    """Fetch (user_id, metric_value, n_games, anchor_rating) for one (metric, tc) — all benchmark users above floor."""
    return f"""
WITH {selected_users_cte(source="benchmark")},
{per_user_cte_for(metric_id, source="benchmark", tc=tc, snapshot_date=snapshot_date)},
{per_user_cte_median_anchor(tc, source="benchmark", snapshot_date=snapshot_date)}
SELECT puv.user_id, puv.metric_value, puv.n_games, pua.anchor_rating
FROM per_user_values puv
JOIN per_user_anchor pua USING (user_id)
""".strip()

# Per-Python:
async def _build_cohort_cdf_for(session, metric, tc, snapshot_date) -> dict[int, CdfTable]:
    rows = (await session.execute(text(_build_per_user_with_anchor_query(metric, tc, snapshot_date=snapshot_date)))).all()
    result: dict[int, CdfTable] = {}
    for anchor in range(COHORT_ANCHOR_MIN_ELO, COHORT_ANCHOR_MAX_ELO + 1, COHORT_ANCHOR_STEP_ELO):
        # Rank users by |anchor_rating - target_anchor|
        ranked = sorted(rows, key=lambda r: abs(r.anchor_rating - anchor))[:COHORT_K_USERS_PER_ANCHOR]
        if len(ranked) < COHORT_K_USERS_PER_ANCHOR:
            continue  # not enough users at this anchor
        kth_dist = abs(ranked[-1].anchor_rating - anchor)
        if kth_dist > COHORT_MAX_WINDOW_ELO:
            continue  # window too wide; suppress
        values = [r.metric_value for r in ranked]
        breakpoints = tuple(float(numpy.percentile(values, p)) for p in range(1, 100))
        result[anchor] = CdfTable(breakpoints=breakpoints, n_users=len(ranked))
    return result
```

(Use `statistics.quantiles` or `numpy.percentile` for the in-Python breakpoint computation — either is faster than 99 separate Postgres percentile_cont calls.)

## Open Questions

1. **K=200 / ±150-Elo relaxation policy.**
   - What we know: Per Pattern 4 probe, only ~6 of 36 sampled (anchor, TC) cells pass strict K=200 / ±150. Relaxing to K=150 OR ±200 Elo passes substantially more cells.
   - What's unclear: Whether the user accepts heavy suppression (strict default) or wants relaxation (which slightly widens per-anchor SE).
   - Recommendation: Plan E's regen report surfaces suppression rate per (metric, anchor, TC) as a top-line table. The user reviews and chooses before Plan G's backfill executes. Reversible by regenerating.

2. **Lichess-precedence implementation: per-(user, TC) or per-game?**
   - What we know: CONTEXT D-04 says "Lichess wins precedence" per TC. Per-game logic would mix two ladders (wrong); per-(user, TC) means "if user has any Lichess games in this TC, anchor reads Lichess-only."
   - What's unclear: Whether the precedence requires a minimum Lichess-game threshold before precedence kicks in (e.g., 30 games), or fires on the first Lichess game.
   - Recommendation: Plan B picks the simplest rule (≥30 Lichess games in TC → Lichess source; else chess.com). Document in the model docstring. Reversible — the policy lives in Python.

3. **`recovery_score_gap` per-TC builder — is it a sibling of `per_user_cte_section2_tc` with `bucket_label='recovery'`, or a distinct family?**
   - What we know: The existing `per_user_cte_section2` (94.2) supports `bucket_label: Literal["conversion", "parity"]`. Recovery was deliberately omitted in 94.2 / 94.3 (D-12 of Phase 94.1). The peer-relative pivot rescues it.
   - What's unclear: Whether the existing per_user_cte_section2 can simply extend its bucket_label Literal to `Literal["conversion", "parity", "recovery"]`, or whether recovery needs a distinct SQL shape.
   - Recommendation: Extend the existing builder — recovery is the third bucket in the same `bucket` derivation expression (see `canonical_slice_sql.py:480-507`; the `bucket` CASE already emits `'recovery'`). The bucket label just needs to be accepted by the `WHERE bucket = '{bucket_label}'` filter at line 554.

4. **API contract for `anchor_rating` + `anchor_source` disclosure.**
   - What we know: The chip popover (Pattern 7) needs the anchor rating, source, and (for chess.com) raw rating to render the 4th bullet.
   - What's unclear: Whether to surface these on `EndgameOverviewResponse` as a top-level `rating_anchors: dict[TimeControlBucket, RatingAnchorOut]` block, or embed per-chip on the existing `*_percentile` fields (would require widening every `*_percentile` field to a structured object).
   - Recommendation: Top-level `rating_anchors` block on `EndgameOverviewResponse`. The frontend looks up per-chip's TC; for aggregated chips, picks the dominant TC. Simpler than per-chip embedding.

5. **Plan A scope — is the ChessGoals snapshot extraction the only research deliverable?**
   - What we know: CONTEXT.md "Claude's Discretion" trims Plan A — the per-(metric × TC) Cohen's d candidacy table is no longer load-bearing under peer-relative (cohort absorbs the coupling).
   - What's unclear: Whether a sanity-check on per-(metric, anchor, TC) cohort viability (a smaller version of Pattern 4 against eval-dependent metrics, not just the score_gap-floor proxy) is worth a planning probe before Plan E.
   - Recommendation: this RESEARCH.md Pattern 4 + Pattern 8 close the primary unknowns. The eval-dependent metric viability is a Plan E regen-report output, not a pre-flight check. **Plan A retires entirely — the research-time deliverables (ChessGoals snapshot + cohort sizing) are captured in this RESEARCH.md.**
