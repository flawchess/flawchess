  - Rejected: a hand-curated static book, incl. reusing the harness's 16-opening `OPENING_BOOK` (narrow, hand-maintained, the human deviates out of it immediately).

Note for the planner: ECO's 3,641 lines cover nearly every sane early position (even 1.h4 is a named opening), so the membership test filters very little in the first few plies. The book's real jobs are (a) a principled stop signal, (b) making the opening nameable so a persona can bias it later, (c) keeping the bot inside theory.
- **D-04 Calibration is NOT perturbed**: `scripts/calibration-harness.mjs` already starts every anchor game from a mid-opening FEN drawn from `scripts/lib/calibration-openings.mjs`'s `OPENING_BOOK`, so the calibrated ELO was never measured on plies 1-8 in the first place. - **D-05 Sampling reuse**: reuse `botSampling.ts`'s existing raw-policy sampling (`samplePolicy`, the `blend <= 0` regime), renormalized over the in-book candidate subset. Do **NOT** use `applyPolicyTemperature` from `policyTemperature.ts` — Phase 166 D-02 locks that transform out of the bot path entirely.
- **D-06 Troll/persona book is out of scope** here, but the design must leave the hook: `frontend/src/data/trollOpenings.ts` + `frontend/src/lib/trollOpenings.ts` (`deriveUserSideKey`) already exist, and SEED-098's Trickster persona plans to bias the book toward them. ## Summary

The book is a pure-function problem layered on top of code that already exists and is already proven correct for this exact corpus. `frontend/src/lib/openings.ts` already fetches and parses `openings.tsv` once (3,641 rows) into a `Map<sanSequence, Opening>` keyed by the full SAN line; the book needs the inverse query (given the SAN history so far, which legal moves extend a book prefix) and can be built from the **exact same parsed data**, in the same lazy fetch, with zero extra network requests. I wrote and ran a throwaway verification script (`frontend/scratch-verify-san.mjs`, deleted after use) that replayed all 3,641 TSV lines through the *real* project `chess.js` (v1.4, ^1.4.0) and compared every token's canonical `move.san` output against the TSV's own SAN token: **0 illegal moves, 0 SAN mismatches across all 3,641 lines and every ply within them.** This resolves the single highest-risk unknown in the phase — the TSV's SAN notation (disambiguated knight moves, `O-O`/`O-O-O`, check `+` suffixes) is byte-identical to what `chess.js` generates, so a plain string-keyed `Set<string>` of every line's prefixes is a correct, zero-conversion index.

The Maia policy the app already fetches via `EngineProviders.policy()` is **UCI-keyed** (`maiaQueue.ts` converts SAN→UCI internally via `sanToUci` before resolving), and it is a **full, un-truncated softmax over every legal move** (`maskAndSoftmax` in `maiaEncoding.ts` — no top-k truncation). This means the book's candidate-restriction step is a pure filter-and-reuse of code that already exists: compute the in-book candidate UCIs (from the SAN prefix-set membership test against `chess.moves({verbose:true})`'s `.lan` field), filter the raw Maia policy `Record<uci, prob>` down to that subset, and hand it directly to `botSampling.ts`'s existing `samplePolicy()` — whose cumulative-weight walk is scale-invariant, so the "renormalization" D-05 asks for is free (no new math). The exit-rule policy floor must be checked against the **raw** (un-renormalized, full-legal-move) probabilities, not the subset-renormalized ones — comparing renormalized probabilities against a floor would make a single obscure book candidate always pass (its renormalized share is always 1.0 if it's the only book move), defeating the entire point of the floor.

The strongest structural finding is on wiring: `selectBotMove` cannot see the app vs. harness distinction (it is imported unchanged by both), but `useBotGame.ts` categorically CAN — the calibration harness has its own from-scratch `playGame`/`playCell` loop in `scripts/calibration-harness.mjs` that calls `selectBotMove` directly and never imports the browser-only, Worker/DOM-dependent `useBotGame` hook at all. Putting the book ahead of `selectBotMove`, inside `runBotTurn`, therefore makes harness-safety a non-issue by construction (the harness cannot reach book code, not "must be told not to"), reuses the exact `Promise.all(book-or-search, revealDelay)` → `chargeableElapsedMs()` → `flagIfOutOfTime` → `commitMove` pipeline already built for honest-clock accounting (SC1's "small fraction of clock" requirement falls out for free), and is exercised by the exact same `vi.mock('@/lib/engine/selectBotMove', ...)` test seam `useBotGame.test.ts` already uses (SC1's "assert search was never called" test needs zero new mocking infrastructure).

Engine prewarm is two asymmetric problems. Maia warms "by necessity" (as CONTEXT.md itself predicts) because the book's own weighting step calls `deps.policy()` on essentially every bot turn while in book (ECO's 3,641 lines filter almost nothing in the first few plies, so a policy call happens on move 1 in virtually every game). Stockfish does NOT warm by necessity — `WorkerPool.grade(fen, [])` (an empty candidate list, the shape a naive "prewarm ping" might try) returns early *before* even calling `ensureSpawned()` (a deliberate empty-searchmoves guard, WR-05), so it spawns nothing. The clean fix is a new, trivial `warm()` method on `WorkerPool` (and, for symmetry/robustness against a ply-cap-0 edge case, on `MaiaQueue` too) that calls the pool's already-existing lazy `ensureSpawned()` directly, with no search and no wasted `movetime`. Actual cold-load timings for both Stockfish WASM and Maia ONNX are **NOT recorded anywhere in this repo** — Phase 151's own measurement table (`151-MAIA-MEASUREMENTS.md`) has every cold-load cell explicitly marked "NOT YET MEASURED" rather than fabricated, and I found no equivalent Stockfish number in Phase 136's research either. I am not fabricating a number here; §"Engine prewarm" documents what IS knowable (asset sizes, spawn code paths) and flags the ms figure as unmeasured.

**Primary recommendation:** Add a pure `frontend/src/lib/engine/openingBook.ts` module (prefix-set candidate filter + a swappable weighting-function seam + exit-rule check), extend `frontend/src/lib/openings.ts` to build the prefix `Set<string>` from its already-fetched/parsed data (not a second fetch), wire the book into `useBotGame.ts`'s `runBotTurn` ahead of the `selectBotMove` call (never inside `selectBotMove` itself), and add a minimal `WorkerPool.warm()` (+ `MaiaQueue.warm()`) called once in the existing provider-bring-up effect.

### Pattern 1: Prefix-set index, derived from the SAME parse `openings.ts` already does

**What:** `openings.ts`'s `buildLookup()` today returns `Promise<Map<string, Opening>>` keyed by the FULL SAN line (`"e4 c6 d4 d5"` → `{eco, name}`). Extend it to also accumulate, while it's already iterating each line's tokens, every non-empty PREFIX of that token array into a `Set<string>`.

**When to use:** Any time the app needs "is this move-history-plus-one-candidate-move still inside some line of book theory" — a pure O(1) hash-set membership check, no incremental per-ply state.

**Example (illustrative shape, not the final diff):**
```typescript
// Source: extending frontend/src/lib/openings.ts's existing buildLookup()
// (this repo, verified structure — see the current buildLookup() at
// frontend/src/lib/openings.ts:16-45)

export interface OpeningLookup {
  fullLineMap: Map<string, Opening>;
  prefixSet: Set<string>;
}

let lookupPromise: Promise<OpeningLookup> | null = null;

function buildLookup(): Promise<OpeningLookup> {
  if (lookupPromise) return lookupPromise;
  lookupPromise = fetch('/openings.tsv')
    .then((r) => r.text())
    .then((text) => {
      const fullLineMap = new Map<string, Opening>();
      const prefixSet = new Set<string>();
      const lines = text.split('\n');
      for (let i = 1; i < lines.length; i++) {
        const line = lines[i]!.trim();
        if (!line) continue;
        const parts = line.split('\t');
        const eco = parts[0];
        const name = parts[1];
        const pgn = parts[2];
        if (!eco || !name || !pgn) continue;
        const tokens = pgnToSanSequence(pgn).split(' ').filter(Boolean);
        fullLineMap.set(tokens.join(' '), { eco, name });
        // Insert every prefix, not just the full line — this is the delta
        // vs. today's buildLookup(). O(tokens^2) string-join work per line,
        // but tokens is small (max 36 in the shipped TSV) and this runs once.
        for (let len = 1; len <= tokens.length; len++) {
          prefixSet.add(tokens.slice(0, len).join(' '));
        }
      }
      return { fullLineMap, prefixSet };
    });
  return lookupPromise;
}
```

**Empirical size check (this repo, 2026-07-13):** `wc -l frontend/public/openings.tsv` → 3,641 rows + 1 header. Longest line: `awk -F'\t' ...` → 36 SAN tokens (Ruy Lopez Marshall Attack, C89, 18 full moves). Total prefix-set insertions is bounded by `sum(tokens_i)` across all lines — a few tens of thousands of short strings, built once, cached module-scope. This is trivial relative to the ~59-70 MB Maia model the app already downloads.

### Pattern 2: SAN prefix membership → UCI candidate list (the SAN/UCI conversion boundary)

**What:** `chess.moves({verbose:true})` returns each legal move with both `.san` (for the prefix-set check) and `.lan` (UCI, for the sampling/weighting step and for `chess.move()`/`commitMove` downstream, which already work in UCI per `selectBotMove.ts`'s D-08 contract). No manual SAN↔UCI conversion is needed in the book module itself — chess.js's own `Move` object already carries both, verified against the real binary:

```
// Verified this repo, node --input-type=module:
{
  "color": "w", "from": "a2", "to": "a3", "piece": "p", "flags": "n",
  "san": "a3", "lan": "a2a3", ...
}
```

```typescript
// Source: frontend/src/lib/engine/openingBook.ts (proposed, pure)
export interface BookCandidate {
  uci: string;
  san: string;
}

/** Stateless — a pure function of the current move history and legal moves;
 * needs no incremental trie-walk state (Pattern 1's tradeoff note). */
export function getBookCandidates(
  moveHistorySan: readonly string[],
  legalMoves: readonly { san: string; lan: string }[],
  prefixSet: ReadonlySet<string>,
): BookCandidate[] {
  const candidates: BookCandidate[] = [];
  for (const move of legalMoves) {
    const key = [...moveHistorySan, move.san].join(' ');
    if (prefixSet.has(key)) candidates.push({ uci: move.lan, san: move.san });
  }
  return candidates;
}
```

**Pitfall avoided by this shape:** `openings.ts`'s existing `findOpening()` walks `len` from `moveHistory.length` DOWN to find the longest match against the FULL-LINE map (naming an already-played sequence). The book's check is the opposite direction and granularity — it tests `moveHistory + [oneCandidateMove]` against the PREFIX set, once per legal move, not once per history length. Do not conflate the two lookup shapes; they read the same parsed data but query it differently.

### Pattern 3: Restrict + weight the Maia policy over book candidates (D-02/D-05, no new sampling code)

**What:** `EngineProviders.policy(fen, elo, side): Promise<Record<string, number>>` is documented (`types.ts` D-08) as "UCI-keyed Maia move-probability distribution... for `side` to move," and its real implementation (`maiaQueue.ts` → `maskAndSoftmax` in `maiaEncoding.ts`) is a **full softmax over every legal move at that position** — no top-k truncation (verified by reading `maskAndSoftmax`'s body: it iterates `chess.moves({verbose:true})` with no slicing, then `exps.reduce`/normalizes over ALL of them). This means `rawPolicy[bookCandidate.uci]` is defined for essentially every book candidate (the only escape hatch is `sanToUci`'s `try/catch` returning `null` for a genuinely unconvertible SAN inside `maiaQueue.ts`'s own conversion — WR-07 — which cannot happen for a move `maskAndSoftmax` itself generated from `chess.js`'s legal-move enumeration).

```typescript
// Source: frontend/src/lib/engine/openingBook.ts (proposed, pure)

/** Pluggable weighting seam (D-06 hook) — today only maiaPolicyWeighting is
 * wired; a future persona layer supplies an alternative implementing the
 * same signature without touching getBookCandidates() or the exit-rule check. */
export type BookWeightingFn = (
  candidates: readonly BookCandidate[],
  rawPolicy: Record<string, number>,
) => Record<string, number>;

/** D-02/D-05: restrict the raw Maia policy to book candidates. `weightedPick`
 * inside samplePolicy() sums whatever weights it's given and draws
 * proportionally (draw = rng() * total) — a filtered-but-unnormalized subset
 * IS the renormalized distribution D-05 asks for; no extra math needed. */
export const maiaPolicyWeighting: BookWeightingFn = (candidates, rawPolicy) => {
  const restricted: Record<string, number> = {};
  for (const { uci } of candidates) {
    const p = rawPolicy[uci];
    if (p !== undefined) restricted[uci] = p;
  }
  return restricted;
};
```

Then the actual move choice is a direct call into the UNCHANGED, already-tested `samplePolicy`:

```typescript
import { samplePolicy } from './botSampling'; // Source: frontend/src/lib/engine/botSampling.ts:55-57
const chosenUci = samplePolicy(maiaPolicyWeighting(candidates, rawPolicy), rng);
```

**Load-bearing correctness point (not obvious from the ticket text):** the D-03 exit-rule policy floor MUST be evaluated against the RAW `rawPolicy` values (each candidate's unconditional probability among ALL legal moves), never against the `restricted`/renormalized values. If only one book candidate exists, its renormalized share inside `restricted` is trivially 1.0 regardless of how implausible Maia thinks the move actually is — checking the floor post-renormalization would make the exit rule un-triggerable in exactly the single-candidate case it exists to catch. See Pattern 4 below.

### Pattern 4: Exit rule (D-03) — floor check on raw probabilities + a ply cap + a one-way latch

```typescript
// Source: frontend/src/lib/engine/openingBook.ts (proposed, pure)

/** Tuned by feel (D-03, Claude's discretion) — no repo precedent for either
 * value; both are [ASSUMED] starting points, not verified against real play.
 * See Assumptions Log A1/A2. */
export const BOOK_POLICY_FLOOR = 0.05;
export const BOOK_PLY_CAP = 16; // 8 full moves — comfortably short of the TSV's 36-ply (18-move) longest line

export interface BookDecision {
  uci: string | null; // null => caller must fall through to selectBotMove and set the one-way latch
}

export function selectBookMove(
  moveHistorySan: readonly string[],
  legalMoves: readonly { san: string; lan: string }[],
  prefixSet: ReadonlySet<string>,
  rawPolicy: Record<string, number>,
  rng: () => number,
  weighting: BookWeightingFn = maiaPolicyWeighting,
): BookDecision {
  if (moveHistorySan.length >= BOOK_PLY_CAP) return { uci: null };
  const candidates = getBookCandidates(moveHistorySan, legalMoves, prefixSet);
  if (candidates.length === 0) return { uci: null };

  // D-03: the floor is checked against RAW probabilities (Pattern 3's
  // correctness note) — never the weighting function's output, which may
  // already be renormalized/reshaped by a future persona layer (D-06).
  const clearsFloor = candidates.some((c) => (rawPolicy[c.uci] ?? 0) >= BOOK_POLICY_FLOOR);
  if (!clearsFloor) return { uci: null };

  const restricted = weighting(candidates, rawPolicy);
  return { uci: samplePolicy(restricted, rng) }; // samplePolicy's own null (all-zero/empty) also falls through
}
```

`useBotGame.ts`'s `runBotTurn` owns the one-way latch (it is React/turn-loop state, not something the pure book module should track — see §"Exit rule mechanics" below for why this can't be a stateless derive-from-history function in the general case):


### Anti-Patterns to Avoid

- **Do not put book logic inside `selectBotMove.ts` or thread it through `BotMoveDeps`.** The module header explicitly locks this file to "ONLY the impure orchestration (one `policy()` OR one `search()` call, plus regime branching)" (D-09) — the book needs a policy call PLUS a completely different pure pipeline (prefix filter → floor check → weighted sample) that doesn't fit the existing three-regime (`blend<=0`/`blend>=1`/`0<blend<1`) dispatch, and `deps.search` is already occupied by `createDeadlineSearch` for the honest-clock model (D-16) — a second competing meaning for that seam would be a real design collision, not just an aesthetic one.
- **Do not call `pool.grade(fen, [])` to "warm" Stockfish.** `WorkerPool.grade()`'s own WR-05 guard returns `Promise.resolve(new Map())` for an empty `candidateUcis` array BEFORE `ensureSpawned()` runs at all (see `workerPool.ts:359-362`) — this is a deliberate no-op, not an oversight, and using it as a prewarm trigger silently does nothing.
- **Do not check the exit-rule policy floor against renormalized/subset probabilities.** See Pattern 3's correctness note.
- **Do not build a second `fetch('/openings.tsv')` for the book.** Extend `openings.ts`'s single cached `lookupPromise`.
- **Do not derive the one-way "left book" state purely from replaying move history at every turn** (see §"Exit rule mechanics" for the precise reason this fails for the floor-exit case specifically).

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Weighted random move selection | A new cumulative-distribution sampler | `botSampling.ts`'s `samplePolicy`/`weightedPick` | D-05 explicitly mandates reuse; `weightedPick` already handles NaN/Infinity/empty-weight degeneracy (D-13) and UCI-ascending tie-break determinism (D-12) — reimplementing loses both for free |
| Legal move / SAN / UCI derivation | Custom move-parsing | `chess.js` `.moves({verbose:true})` | Already the project's sole rules engine; empirically verified against the full TSV corpus in this research (0 mismatches) |

**Key insight:** this entire phase is deliberately "boring" — every hard sub-problem (weighted sampling, legal-move enumeration, provider lifecycle, honest-clock debiting) already has a correct, tested implementation in this codebase; the only genuinely new code is the prefix-set index and the thin glue that filters candidates through it.

## Common Pitfalls

### Pitfall 1: Treating the exit-rule state as fully derivable from move history alone
**What goes wrong:** A design that recomputes "are we still in book" purely from `moveHistory` at every turn (no persisted "left book" bit) can silently RE-ENTER book after the search has already played a move, violating D-03's explicit one-way lock.
**Why it happens:** D-03's own note that "ECO's 3,641 lines cover nearly every sane early position" means the prefix-membership test alone is a very weak filter in the first several plies — it is entirely possible for the game to wander back onto a cataloged ECO line's prefix even after the bot has left book and started searching (e.g., both players independently play moves that happen to match a different, less-common ECO line). Ply-cap exits ARE safely re-derivable from `moveHistory.length` alone (monotonic), but floor-exits are NOT — the floor check is a fresh per-position evaluation with no memory of prior positions.
**How to avoid:** Keep a single one-way ref/flag (`hasLeftBookRef` in `useBotGame`, mirroring the existing `outcomeRef` latch at line 246) that, once set true (by either the floor-miss or the ply-cap branch), is checked BEFORE even attempting the book candidate/floor computation on every subsequent bot turn, and is reset only in `newGame()`.
**Warning signs:** A game log showing the bot returning to fast (~500-1500ms), book-typical move times after it has already had one or more multi-second searched moves.

### Pitfall 2: Empty-`candidateUcis` "prewarm" no-op
**What goes wrong:** Calling `pool.grade(fen, [])` (or any call with an empty candidate list) expecting it to spawn the Stockfish worker pool as a side effect — it does not; `workerPool.ts:359-362`'s WR-05 guard returns before `ensureSpawned()` is ever reached.
**Why it happens:** The guard exists to protect the PUBLIC `EngineProviders.grade` surface from a caller accidentally letting Stockfish search ALL moves and burn its full `movetime` budget on an empty `searchmoves` list — a correct and desirable guard for its own purpose, but an unintuitive trap for anyone reusing `grade()` itself as a prewarm signal.
**How to avoid:** Add a dedicated `warm()` export to `WorkerPool` that calls the pool's existing internal `ensureSpawned()` directly, bypassing `grade()`'s request/dispatch machinery entirely.
**Warning signs:** `pool.warm()`-via-`grade([])` "works" in that it doesn't throw, but the first REAL search after book exit still pays full worker-spawn cold start — the bug is silent because nothing errors.

### Pitfall 3: Floor check on the wrong (renormalized) probability set
**What goes wrong:** Checking `Math.max(...Object.values(restricted)) >= BOOK_POLICY_FLOOR` (the WEIGHTED/filtered subset) instead of the raw full-legal-move policy — with only one book candidate remaining, its renormalized share is always exactly 1.0, so the floor can never fire for that position no matter how implausible Maia actually rates the move.
**Why it happens:** It's the more "obvious" variable to check if you write the weighting step before the exit-rule check and reach for whatever's already in scope.
**How to avoid:** Structure the code so the floor check reads `rawPolicy` (the direct `deps.policy()` result), never the `weighting(...)` output — see Pattern 4's `selectBookMove` above, where `clearsFloor` is computed before `weighting` is even called.
**Warning signs:** The bot never leaves book via the floor path in any test/manual session, only ever via the ply cap.

### Pitfall 4: Suppressing the D-01 post-move draw-accept grade call without deciding to
**What goes wrong:** `useBotGame.ts`'s `runBotTurn` already fires a best-effort `pool.grade(fen, [uci])` AFTER every commit (line ~805-819) to refresh `lastRootPracticalScoreRef` for the draw-offer heuristic — this call is UNRELATED to move selection and predates this phase, but it IS a real Stockfish leaf eval, and it WILL fire after a book move too if left untouched. A literal reading of SC1 ("no Stockfish leaf evals... while in book") could be read as requiring this call to be suppressed during book plies as well.
**Why it happens:** SC1's wording doesn't distinguish "evals used for move selection" from "evals used for an orthogonal post-hoc heuristic."
**How to avoid:** This is a genuine judgment call for the planner, not something this research can resolve unilaterally — recommend treating SC1 as scoped to the MOVE-SELECTION path (leave the D-01 call unchanged; it predates this phase, is orthogonal to search/node-budget, and suppressing it would require new gating logic and would leave `lastRootPracticalScoreRef` stale through the whole book window, degrading the draw-accept heuristic for no clear benefit) — but flag it explicitly in the plan/verification so it's an INTENTIONAL decision, not an oversight either way.
**Warning signs:** A test asserting "zero `pool.grade` calls during book plies" fails against unmodified `runBotTurn` — if the planner wants that literal invariant, the D-01 call needs its own book-aware gate, which is additional (small) scope beyond what CONTEXT.md's decisions describe.

### Verifying the SAN-matching risk empirically (already run, not hypothetical)

```javascript
// Ran as frontend/scratch-verify-san.mjs (deleted after use — not committed).
// Source: this research session, 2026-07-13, real frontend/node_modules/chess.js (^1.4.0).
import { Chess } from 'chess.js';
import { readFileSync } from 'fs';

const text = readFileSync('frontend/public/openings.tsv', 'utf8');
const lines = text.split('\n');
function pgnToSanSequence(pgn) {
  return pgn.replace(/\d+\./g, '').replace(/\s+/g, ' ').trim();
}

let total = 0, mismatches = 0, illegal = 0;
for (let i = 1; i < lines.length; i++) {
  const line = lines[i].trim();
  if (!line) continue;
  const [eco, name, pgn] = line.split('\t');
  if (!eco || !name || !pgn) continue;
  total++;
  const sanSeq = pgnToSanSequence(pgn).split(' ').filter(Boolean);
  const chess = new Chess();
  let ok = true;
  const replayed = [];
  for (const tsvSan of sanSeq) {
    let move;
    try { move = chess.move(tsvSan); } catch { ok = false; illegal++; break; }
    replayed.push(move.san);
  }
  if (ok) {
    for (let j = 0; j < replayed.length; j++) {
      if (replayed[j] !== sanSeq[j]) { mismatches++; break; }
    }
  }
}
console.log(`Total: ${total}, illegal: ${illegal}, mismatches: ${mismatches}`);
// Actual output (this repo, 2026-07-13):
// Total lines checked: 3641
// Illegal (chess.js threw): 0
// SAN string mismatches (parsed legal but different canonical SAN): 0
```

## Open Questions

1. **Should the D-01 post-commit draw-accept `pool.grade` call be suppressed during book plies?**
   - What we know: it's an existing, orthogonal (pre-Phase-169.5) call that fires after EVERY commit regardless of how the move was chosen.
   - What's unclear: whether SC1's "no Stockfish leaf evals... while in book" is meant to cover it.
   - Recommendation: leave it unchanged by default (Pitfall 4's reasoning); the planner should make this an explicit, stated decision in the plan rather than silently pick one interpretation.

2. **Exact `BOOK_POLICY_FLOOR`/`BOOK_PLY_CAP` values.**
   - What we know: both are explicitly "tuned by feel" per D-03; A1/A2 above give defensible starting points with reasoning.
   - What's unclear: no play-testing has happened yet to validate these against actual bot-at-various-ELO behavior.
   - Recommendation: ship the starting constants, treat them as trivially re-tunable named constants (already satisfies CLAUDE.md's no-magic-numbers rule), and consider a manual UAT pass (per existing 169-07-PLAN.md precedent for pacing-feel checkpoints) rather than gating the phase on precise numeric validation.

3. **Should `MaiaQueue` also get a `warm()` export, or is "warm by necessity" sufficient?**
   - What we know: in the overwhelmingly common case (ply cap > 0, book actually engaged on move 1), the book's own `deps.policy()` call warms Maia for free.
   - What's unclear: whether the edge case (a hypothetical `BOOK_PLY_CAP` of 0, or a future config where book is disabled) is worth defending against with an explicit `queue.warm()`.
   - Recommendation: add it anyway — it's the same trivial `ensureSpawned()`-forwarding shape as `WorkerPool.warm()`, costs nothing, and removes a latent "Maia is warm only if the book happened to run" invariant that could silently break under future config changes.

### Revert-provability note (CLAUDE.md `feedback_mutation_test_gap_closures` memory)

Two of the above assertions are only revert-provable if written carefully, per this project's own hard-won lesson (mutation-test gap closures must fail when the fix is reverted, not just "pass vacuously"):

- **SC1 "no search while in book":** asserting `mockSelectBotMove` was called ZERO times during the first N plies is vacuously true if the book branch is never reached at all (e.g., a bug where `hasLeftBookRef` starts `true`). Pair it with a POSITIVE assertion later in the same test — `mockSelectBotMove` IS called at least once after the exit condition is forced (e.g., stub `mockPolicy` to return an empty/all-below-floor distribution) — so a test that "passes" by never engaging the book at all also fails.
- **SC3 "variety, weighted not uniform":** asserting "≥2 distinct first moves across N seeds" alone does NOT distinguish a correctly Maia-weighted sampler from a bug that silently degrades to uniform-random-over-book-candidates (a uniform sampler also produces multiple distinct moves over many draws). The test MUST additionally assert the empirical frequency of each candidate tracks its STUBBED policy weight within a tolerance band — reverting the `weighting`/`samplePolicy` call to `candidates[Math.floor(rng()*candidates.length)]` (uniform) must make this specific assertion fail, even though the "at least 2 distinct moves" assertion would still pass.
