# Phase 123.1 Context — Opening-eval dedup cache table (SEED-053)

Standalone infra phase (post-v1.27, milestone TBD). Source seed: `.planning/seeds/SEED-053-opening-eval-dedup-cache-table.md`. Designed interactively 2026-06-17; this file is the locked-decision record so execution can run after a `/clear`.

## Problem

The full-eval drain's per-game dedup lookup (`_fetch_dedup_evals`, `app/services/eval_drain.py:270`) is the single largest consumer of total DB server time in prod.

- **db-report 2026-06-17** (window since 2026-06-15 reset): `DISTINCT ON (full_hash) … eval_cp, eval_mate, best_move` — **8.4 s avg, 10,655 calls, 89M ms (~24.7 h cumulative)**.
- Called once per game to recover a position's own eval + best_move from an existing engine eval at the same `full_hash` (cross-user dedup, D-116-02), so the drain can skip Stockfish.

## Root cause (measured)

Low-ply **fan-out** under **post-move storage**, not a missing index:

- A position's eval lives on its *predecessor* row, so the lookup is a self-join (`ix_gp_full_hash_opening` finds the hash → join back one ply for `eval_cp`). No covering index can serve it (an `INCLUDE(eval_cp)` on the hash row carries the *next* position's eval).
- Common opening positions share one `full_hash` across nearly all games. Prod dup factors at ply≤20:

  | ply | distinct hashes | dup factor |
  |---|---|---|
  | 0 | 46 | 13,534× |
  | 2 | 522 | 1,191× |
  | 4 | 9,133 | 68× |
  | 6 | 42,634 | 14.5× |
  | 13+ | 335k–522k | 1.2–1.8× (≈unique) |

- The bound is already in place (`nxt.ply <= DEDUP_MAX_PLY`=20 in the WHERE, and `ix_gp_full_hash_opening` is partial `WHERE ply <= 20`). The bound is not the bottleneck; the posting-list length of common hashes is. Tightening the cap backfires — the expensive hashes are the low plies, which is exactly where dedup pays off.

## EXPLAIN evidence (prod, real 21-hash batch incl. ply-0)

| Approach | Exec time | Buffers |
|---|---|---|
| Current `DISTINCT ON` self-join | 16.4 s | 6.5M |
| `LATERAL … LIMIT 1` rewrite (no new table) | 7.8 s | 4.2M |
| Cache table (projected) | ~1–5 ms | 21 PK probes on an ~80MB resident table |

The rewrite hits a 2× structural ceiling (post-move storage forces the join). Only a hash-unique relation collapses it.

## Sizing

Prod `game_positions` ply≤20: 12.9M rows → 4.68M distinct hashes → **1.06M cacheable** (distinct hashes with an our-engine eval). Cache table ≈ **80MB with PK index**, smaller than `shared_buffers` (2GB) → fully resident, never touches the 10GB `game_positions` nor fights the drain for buffers.

## Why now

Tier-3 drain backlog at design time: **245,225 engine-eval games + 21,488 lichess-pv games pending** (of 376k non-guest). The drain is far from steady state, so the 8.4 s/game lookup is a live, present bottleneck. (`pg_stat_statements` was reset 2026-06-17; the db-report should be re-checked post-drain, but the backlog justifies acting now.)

## Locked Decisions

- **D-123.1-01 — Position-keyed cache table.** `opening_position_eval(full_hash BIGINT PRIMARY KEY, eval_cp SmallInteger NULL, eval_mate SmallInteger NULL, best_move VARCHAR(5) NULL)`. `best_move` width matches `game_positions.best_move` (4 chars normal, 5 promotion). No FK, no cascade, **no invalidation logic** — the cached value is position-intrinsic and immutable (modulo eval non-determinism we already accept project-wide); a dangling entry after game/user deletion is still a correct eval.
- **D-123.1-02 — Reject the materialized view.** A matview is the same structure but `REFRESH` is a full recompute that grows with the DB, and staleness between refreshes erodes the dedup hit rate during exactly the active-drain window we care about. pg_ivm rejected too (extension + triggers on bulk-`COPY`'d `game_positions`). The incremental table buys always-fresh + zero refresh orchestration for a one-line write-path addition.
- **D-123.1-03 — Seed from prod our-engine data only; NOT the benchmark DB.** Benchmark has 5.3× more opening-eval coverage but (a) no `best_move`/`pv` columns, so it can't let the drain skip the Stockfish pass (eval-only entries save zero engine work), and (b) its evals are lichess %eval, which the dedup deliberately excludes via `has_engine_full_evals` (D-117-07). Mixing would create two-tier provenance.
- **D-123.1-04 — Write only freshly-computed opening engine misses.** The write set per game = `engine_targets` with `ply <= DEDUP_MAX_PLY`, not terminal, non-null eval. NOT "insert all ~21 opening positions" (≈90% would conflict → ~4.6M speculative-insertion dead tuples over the backfill) and NOT dedup transplants (already cached). One **batched** multi-row `INSERT … ON CONFLICT (full_hash) DO NOTHING` inside the existing Step-4 write transaction. Self-limiting: insert volume falls as the cache fills.
- **D-123.1-05 — Read path keeps signature + ALL guards.** `_fetch_dedup_evals` keeps its signature and every read-side guard (`ply <= DEDUP_MAX_PLY`, `not in flaw_adjacent_plies`, `not is_terminal`). Only the backing query changes (cache `full_hash = ANY(:hashes)` vs self-join). eval/flaw/pv output must be byte-identical.
- **D-123.1-06 — Backfill is a standalone idempotent script**, not the Alembic migration. `scripts/backfill_opening_eval_cache.py --db dev|prod` (prod needs `prod_db_tunnel.sh`), `INSERT … SELECT DISTINCT ON (full_hash) … ON CONFLICT DO NOTHING`. Keeping it out of the migration avoids running a 1M-row data op on the test template and every env. (Mirrors `scripts/backfill_eval.py`.)
- **D-123.1-07 — SEED-054 is OUT OF SCOPE.** The opening-flaw `pv` gap (~3.5% / 5,570 engine-game flaws registered without a refutation `pv`) is pre-existing, inherited unchanged by the cache (it's a drop-in for the same dedup decisions), and is a tactic-tagging concern. Tracked in `.planning/seeds/SEED-054-opening-flaw-pv-gap-engine-games.md`.

## Out of scope

- SEED-054 pv-gap fix (separate seed).
- Denormalizing `has_engine_full_evals` onto `game_positions` (a marginal ~16s→10s self-join win, mooted by the cache).
- Any change to `DEDUP_MAX_PLY` or the dedup decision logic.
