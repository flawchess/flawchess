# Phase 28.1: Import Lichess Analysis Metrics - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Import lichess per-player analysis metrics (ACPL, inaccuracy count, mistake count, blunder count) from the API response into the `games` table. Storage only — no display, no API endpoints, no UI changes. These metrics are pre-computed by lichess and returned alongside accuracy when `accuracy=true` is set (already wired up in Phase 28 + the accuracy fix).

Chess.com does NOT provide these metrics via their public API (confirmed — API is in bug-fix-only mode). This is lichess-only data, but columns use generic names.

</domain>

<decisions>
## Implementation Decisions

### Column Schema
- **D-01:** Eight new nullable columns on `games` table, generic names (not lichess-prefixed):
  - `white_acpl` (SmallInteger) — average centipawn loss, white
  - `black_acpl` (SmallInteger) — average centipawn loss, black
  - `white_inaccuracies` (SmallInteger) — inaccuracy count, white
  - `black_inaccuracies` (SmallInteger) — inaccuracy count, black
  - `white_mistakes` (SmallInteger) — mistake count, white
  - `black_mistakes` (SmallInteger) — mistake count, black
  - `white_blunders` (SmallInteger) — blunder count, white
  - `black_blunders` (SmallInteger) — blunder count, black
- **D-02:** All columns nullable — NULL for chess.com games and unanalyzed lichess games. SmallInteger is sufficient (ACPL rarely exceeds 999; move counts are small integers).
- **D-03:** Both sides stored (white + black) consistent with accuracy columns, for opponent scouting.

### Data Extraction
- **D-04:** Extract from lichess API response at `players.{color}.analysis.{acpl,inaccuracy,mistake,blunder}`. This data arrives in the same response as accuracy — no additional API parameter needed beyond `accuracy=true` (already set).
- **D-05:** Add extraction to `normalize_lichess_game()` in `normalization.py`, same pattern as accuracy extraction just added.
- **D-06:** Chess.com normalizer unchanged — these fields don't exist in chess.com API response.

### Backfill
- **D-07:** No separate backfill needed. Existing lichess games get these metrics when users re-sync (the admin re-import script from Phase 28 plan 28-03 handles full re-import). New imports get them automatically.

### Scope Boundary
- **D-08:** Storage only this phase. Display (game cards, stats) and API endpoints for these metrics deferred to future phases.
- **D-09:** Update REQUIREMENTS.md out-of-scope note to clarify: "per-move quality classification" (us computing it) remains out of scope; importing pre-computed game-level counts from lichess is in scope.

### Claude's Discretion
- Alembic migration implementation details
- Test structure and coverage approach
- Whether to add a comment on the `games` model clarifying lichess-only provenance

</decisions>
