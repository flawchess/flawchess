# Phase 117.1: Flaw-Eval Convention Fix - Context

**Gathered:** 2026-06-14
**Status:** Ready for planning
**Source:** Discussion (locked decisions) + SEED-044

<domain>
## Phase Boundary

Fix a confirmed off-by-one in the flaw pipeline: the engine drain stores `eval_cp` for the **pre-push** position (eval BEFORE the move) while `classify_game_flaws` assumes **post-move** (eval AFTER the move). This is correct for lichess `%eval` games and wrong for every chess.com (engine-evaluated) game — the majority of the dataset. The result is missing or misattributed flaws (a single real blunder can surface as a spurious adjacent pair, or as nothing) on the Flaw-Stats panel, opponent comparison, and every flaw-derived metric for chess.com users.

**In scope:** the eval-storage convention (write side), the opening-region dedup transplant, the terminal-position eval, regression tests, a clean-slate migration, and the post-deploy re-eval/verification.

**Out of scope:** the classifier's flaw-detection logic itself (no threshold/severity changes — it already assumes post-move and needs no logic change); lichess `%eval` data (already post-move, untouched); Phase 118 Demand UX; SEED-039 tactic-motif classification.

**Source of the bug:** the Phase 116/117 engine drain met the Phase 108/113 flaw classifier (which predates it). Live on `main`/prod now.
</domain>

<decisions>
## Implementation Decisions (LOCKED)

### Canonical convention = POST-MOVE storage everywhere
- Standardize on the lichess post-move convention: `eval_cp[k]`/`eval_mate[k]` = eval of the position AFTER move k, at every row, for ALL sources.
- Rationale: the classifier and the lichess import are already post-move; a single convention removes a forever per-source branch in the classifier. Lichess `%eval` games cannot be re-evaluated (evals come from lichess annotations, not our Stockfish), so lichess is fixed and post-move must be the canonical target.
- **Classifier (`app/services/flaws_service.py` `_run_all_moves_pass`, ~304-327): NO logic change.** It already reads post-move. Once engine storage is post-move, it is correct for both sources with no branch. Only update stale comments if any reference the old engine pre-push convention.

### Drain write convention (`app/services/eval_drain.py`)
- Store `eval_cp`/`eval_mate` post-move (= eval of the resulting position after the move).
- Underlying Stockfish work is unchanged: each pre-push position is still evaluated once (needed to produce `best_move` from the decision position). The eval NUMBER for a position is identical regardless of convention — only the ROW it is written into changes. Post-move: `eval_cp[k]` := eval of the position after move k (which equals the pre-push eval of position k+1).
- **Evaluate the terminal position too** (currently skipped at `eval_drain.py:168` — "board is now terminal — NOT added"). Under post-move the last move's "after" eval = the terminal position's eval; store it as the last move's `eval_cp` so the last move of every game is flaw-assessable (game-ending blunders are exactly the ones to flag). One extra position-eval per game; acceptable cost.
- `best_move` and `pv` stay keyed to the **decision ply** (the move-played row k): `best_move[k]` = the better alternative to move k = best move from the pre-push position `full_hash[k]`. These do NOT shift. So a row k becomes "the assessment of move k": `move_san[k]` played, `best_move[k]` was better, `eval_cp[k]` is the eval that resulted.

### Dedup transplant rework (`app/services/eval_drain.py` `_fetch_dedup_evals` / `_resolve_full_eval`)
- The opening-region dedup (`ply <= _DEDUP_MAX_PLY=20`, gated on `lichess_evals_at IS NULL`) currently transplants `(eval_cp, eval_mate, best_move)` by `full_hash`, relying on engine rows being **pre-push position-keyed** (`eval_cp[k]` describes `full_hash[k]`). Post-move storage makes that false: `eval_cp[k]` then describes `full_hash[k+1]`.
- **Rework with a one-ply shift** so a *position's* eval is recovered correctly from post-move donors. Of the transplanted tuple:
  - `best_move` stays pre-push/decision-keyed → its transplant is UNCHANGED (recovered by direct `full_hash`).
  - Only `eval_cp`/`eval_mate` need the shift on the donor read (recover the position's own eval from the post-move column).
- Encapsulate the convention shift in ONE place/helper so the convention lives in exactly one spot, not smeared across the code.
- **Rewrite the `eval_drain.py:182-191` comment** (currently documents "engine rows are pre-push position-keyed, safe to transplant by full_hash") to document the post-move convention + the shift.
- Handle the first/last-ply edges of the shift (no donor eval for the position that has no stored predecessor/successor row).

### Migration (clean slate)
- For engine games (`lichess_evals_at IS NULL`) with `full_evals_completed_at IS NOT NULL`:
  - NULL `game_positions.eval_cp`, `eval_mate`, `best_move`, `pv`.
  - Clear `games.full_evals_completed_at` and `games.full_pv_completed_at`.
  - DELETE their `game_flaws` rows.
- TRUNCATE the `eval_jobs` queue/lease table (clears stale leases/jobs so games re-enqueue cleanly under the new convention).
- Do NOT touch lichess games' eval data.
- The background drain re-materializes everything under the new convention after deploy.

### Re-eval + verification (post-deploy)
- Clean-slate re-eval runs automatically via the background drain on prod. The prior engine eval run took ~24h; losing it is explicitly acceptable (user-confirmed).
- After re-eval progresses, re-verify the ~32% flaw-PV coverage TODO (git `4b40960e`) — the off-by-one is the suspected cause.
</decisions>
