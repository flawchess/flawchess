# Phase 41.1: Import Speed Optimization - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Optimize the game import processing pipeline to increase throughput from ~20 games/sec to ~40+ games/sec. Focus on eliminating code-level inefficiencies in `_flush_batch` and the PGN processing path. No concurrent archive fetching, no architectural pipeline restructuring, no new user-facing features.

</domain>

<decisions>
## Implementation Decisions

### PGN Processing Consolidation
- **D-01:** Replace the triple PGN parse with a single unified function that walks the mainline once, yielding (hashes, classification, eval, clock, move_san) per ply. `hashes_for_game` and the separate classification/eval loops in `_flush_batch` are replaced by this new function.
- **D-02:** `move_count` is derived from the unified function output (`(len(hash_tuples) - 1 + 1) // 2`), and `result_fen` is already returned. The third PGN parse for these is eliminated entirely.

### DB Roundtrip Reduction
- **D-03:** Eliminate the SELECT for Game.pgn after bulk_insert_games. Instead, build a `platform_game_id -> PGN` lookup dict from the batch, then do a lightweight SELECT for `(id, platform_game_id)` to map new game IDs back to their PGNs.
- **D-04:** Replace per-game UPDATE for move_count/result_fen with a bulk UPDATE using CASE expressions — one query per batch instead of 10-30 individual UPDATEs.

### Batch Size
- **D-05:** Increase `_BATCH_SIZE` from 10 to 25-30. The position INSERT chunking (1,700 rows/chunk) already prevents oversized queries. Larger batches mean fewer DB commits and round-trips.

### Scope Boundaries
- **D-06:** Focus on processing speed only — no concurrent archive fetching. Chess.com rate limits and ban risk outweigh the ~10% gain from parallel fetching.
- **D-07:** No multiprocessing or ProcessPoolExecutor for CPU-bound work — keep the optimization within the existing async/single-thread model.
- **D-08:** Re-sync speed is already fast and not a concern. Optimization targets initial imports of large game libraries (1,000+ games).

### Claude's Discretion
- Exact batch size within the 25-30 range
- Internal API of the unified PGN processing function (return type, parameter design)
- Whether to keep `hashes_for_game` as a thin wrapper or replace it entirely
- Test strategy for verifying correctness after refactoring

</decisions>

<code_context>
## Existing Code Insights

### Current Bottlenecks (measured from prod import_jobs)
- chess.com: ~20 games/sec for initial imports (9,324 games in 432s)
- lichess: ~16-24 games/sec for initial imports (9,994 games in 411s)
- Processing dominates over network I/O: archive fetching is ~30s of a 432s import

### Triple PGN Parse (main inefficiency)
1. `hashes_for_game(pgn)` — parses PGN, walks mainline for Zobrist hashes + move_san + clock
2. `chess.pgn.read_game()` in `_flush_batch` — re-parses for `classify_position` + eval extraction
3. `chess.pgn.read_game()` in `_flush_batch` — re-parses for `move_count` + `result_fen`

### Unnecessary SELECT
- After `bulk_insert_games` returns new IDs, code does `SELECT Game.id, Game.pgn WHERE id IN (new_ids)` even though PGN was already available in the batch

### Per-game UPDATEs
- Individual `UPDATE Game SET move_count=X, result_fen=Y WHERE id=Z` for each new game (10+ per batch)

### Existing Patterns to Preserve
- `bulk_insert_positions` chunking at 1,700 rows (asyncpg 32,767 arg limit)
- ON CONFLICT DO NOTHING for duplicate game deduplication
- Incremental progress persistence (counters updated after each batch)
- Sentry context tagging for error reporting
- Position classification includes: material_count, material_signature, material_imbalance, has_opposite_color_bishops, piece_count, backrank_sparse, mixedness, endgame_class
- Eval extraction from PGN %eval annotations (lichess games)
- Clock seconds from %clk annotations

### Integration Points
- `_flush_batch` is the only caller of `hashes_for_game` and `classify_position` during import
- `hashes_for_game` is also called in tests — maintain backward compatibility or update tests
- `compute_hashes` (underlying Zobrist function) is used independently by the analysis endpoints

</code_context>

<specifics>
## Specific Ideas

- Production data shows ~20 games/sec baseline. Target is ~40+ games/sec (2x improvement).
- The OOM incident at batch_size=50 was before position INSERT chunking was added. With chunking, larger batches should be safe.
- The unified PGN function should expose all data currently extracted by three separate parses: Zobrist hashes (white/black/full), move_san, clock_seconds, board states for classification, node evals, move_count, and result_fen.

</specifics>

<deferred>
## Deferred Ideas

- **Concurrent archive fetching** — Fetch multiple chess.com monthly archives in parallel. Estimated ~10% gain but requires careful rate limit handling. Worth revisiting if processing optimizations don't meet target.
- **Profiling instrumentation** — Add timing metrics to `_flush_batch` sub-operations for data-driven optimization. Not needed for the obvious wins in this phase, but useful for future fine-tuning.
- **ProcessPoolExecutor for CPU-bound work** — Offload PGN parsing and position classification to separate processes. Complex, uncertain benefit given GIL behavior with python-chess.

</deferred>
