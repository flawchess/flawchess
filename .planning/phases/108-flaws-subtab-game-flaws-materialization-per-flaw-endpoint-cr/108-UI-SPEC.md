## Flaw-Filter Control — Detailed Contract

### Purpose and placement

`FlawFilterControl` is a **dedicated** control, separate from the game-metadata `FilterPanel`
(which handles platform / time-control / rated / recency / opponent). It lives inside the existing
`LibraryFilterPanel` sidebar/drawer chrome, rendered ABOVE the standard metadata filters — the same
position the boolean severity toggle occupied in Phase 107. The Flaws tab introduces no new sidebar
or drawer chrome.

### Layout structure

```
┌─────────────────────────────────────┐
│ [section label] "Show flaws with:"  │
│ ─────── severity ───────────────    │
│  [Blunders]  [Mistakes]             │  ← h-11 sm:h-7 buttons, gap-2
│                                     │
│ [divider]                           │
│                                     │
│ [family group] Timing               │
│  [low-clock] [impatient] [considered│  ← family-colored toggle buttons
│                                     │
│ [family group] Opportunity          │
│  [miss] [lucky-escape]              │
│                                     │
│ [family group] Impact               │
│  [result-changing] [while-ahead]    │
│                                     │
│ [clear link] "Clear flaw filter"    │  ← shown only when any selection active
└─────────────────────────────────────┘
```

Desktop: rendered inside a `<aside>` sidebar (same as existing `LibraryFilterPanel`).
Mobile: rendered inside the `Drawer` (same component, no separate mobile copy).

### Severity section

Label: `"Show flaws with:"` — `text-sm text-muted-foreground`.

Two buttons: **Blunders** and **Mistakes** (`text-sm font-bold`).

Behavior:
- Default: both selected (M+B). This is NOT the same as "both deselected = no filter" from
  the Phase 107 boolean toggle. In the new control, "both deselected" means no severity filter
  (show all, same as both selected in practice since the table only contains M+B). The planner
  must decide whether to enforce both-always-selected or allow deselect — recommended: keep both
  selected by default and prevent deselecting both (at least one must remain selected). Deselecting
  one to get only blunders or only mistakes is the useful action.
- Either or both selected: filters `game_flaws` to matching severity rows.
- Active style: `border-toggle-active bg-toggle-active text-toggle-active-foreground` (brand-brown,
  matching existing FilterPanel button pattern).
- Inactive style: `border-border bg-inactive-bg text-muted-foreground`.
- Height: `h-11 sm:h-7` — matches existing FilterPanel touch targets.
- `data-testid="filter-flaw-severity-blunder"` / `data-testid="filter-flaw-severity-mistake"`.
- `aria-pressed` reflecting active state on each button.

### Tag-family sections

Three family groups, each with a label and a row of tag buttons.

**Family group label:** `text-sm text-muted-foreground font-bold uppercase tracking-wide` — small
caps style (matches Flaw-Stats zone labels from Phase 107). Labels:

| Family | Group label | Tags |
|--------|------------|------|
| Tempo | "Timing" | low-clock, impatient, considered |
| Opportunity | "Opportunity" | miss, lucky-escape |
| Impact | "Impact" | result-changing, while-ahead |

Note: phase tags (`opening`, `middlegame`, `endgame`) are NOT included in the filter control.
They appear only in the Flaw-Stats panel's tag distribution histogram, not as filter predicates.

**Tag buttons:** render identically to the read-only `TagChip` on game cards (same icon, same label,
same family color), but as interactive toggle buttons rather than popover triggers.

- Unselected: `border-border bg-inactive-bg text-muted-foreground` (no family color).
- Selected: family-color border + background (`FAM_TEMPO` / `FAM_OPPORTUNITY` / `FAM_IMPACT`
  from `theme.ts`) + foreground color. Same visual as a `TagChip` on a game card — the selection
  state IS the chip appearance.
- Icon: same lucide icon as the corresponding `TagChip` (Clock / Zap / Brain / Target / Clover /
  Swords / TrendingDown).
- Height: `h-11 sm:h-7` (touch targets); chip shape: `rounded-full` with `px-3 py-0.5`.
- Flex-wrap allowed within each family row.

`data-testid` per tag button (kebab-case, component-prefixed):

| Tag | data-testid |
|-----|-------------|
| low-clock | `filter-flaw-tag-low-clock` |
| impatient | `filter-flaw-tag-impatient` |
| considered | `filter-flaw-tag-considered` |
| miss | `filter-flaw-tag-miss` |
| lucky-escape | `filter-flaw-tag-lucky-escape` |
| result-changing | `filter-flaw-tag-result-changing` |
| while-ahead | `filter-flaw-tag-while-ahead` |

`aria-pressed` on each button reflecting selection state.
`aria-label="Filter by tag: {tag}"` on each button.

### Boolean semantics (for tooltip and affordance copy accuracy)

The filter predicate uses OR within a family, AND across families. The control's grouping
makes this legible without requiring copy explanation:

- Selecting `low-clock` + `impatient` within Timing = "either timing tag" (OR).
- Selecting `result-changing` (Impact) + `low-clock` (Timing) = "a single flaw that is both
  result-changing AND low-clock" (AND across families, one flaw satisfies both).

No tooltip or explanatory text is needed for the boolean logic — the family grouping IS the
affordance. Do not add inline OR/AND labels.

### Clear affordance

When any flaw filter selection is non-default (any tag selected, or severity not both-M+B):
show a `"Clear flaw filter"` text link below the tag section.

- Style: `text-sm text-muted-foreground underline cursor-pointer`.
- `data-testid="btn-clear-flaw-filter"`.
- `aria-label="Clear all flaw filter selections"`.
- Resets to default: both severity buttons selected, no tags selected.

This is separate from the main "Reset Filters" button (which clears game-metadata filters too).
"Clear flaw filter" clears ONLY the flaw filter state, preserving game-metadata filters.

### Deep-link landing state

When the Flaws tab is reached via `/library/flaws?tag={TAG}` (from a Games-card chip click):

- The URL param `tag={TAG}` pre-populates the corresponding tag button as selected.
- Severity defaults to M+B (both selected).
- No other tag buttons are selected.
- The control renders visually with the pre-selected tag button highlighted in its family color.
- The "Clear flaw filter" link appears (since a tag is selected = non-default state).

The control must read URL params on mount and initialize state from them. On Flaws tab, URL stays
synced: when the user changes filter selections, `?tag=` and `?severity=` params update without
a page reload (replace-state, not push-state).

On the Games tab, URL is NOT synced (D-04 from CONTEXT.md, mirrors Phase 107 severity precedent).
State is shared via `useFlawFilterStore` (mirrors `useFilterStore`), so switching Games↔Flaws
preserves selections in memory.

## Interaction Contract

### Multi-select behavior

- Tag buttons toggle independently (tap/click = select/deselect).
- Multiple tags may be selected simultaneously across families (AND across families).
- Multiple tags within the same family may be selected (OR within family).
- Selecting within-family unsatisfiable combos (e.g. `low-clock` + `impatient` — mutually
  exclusive on a single flaw) is technically allowed in the UI; the result set will simply be
  empty (the backend returns zero rows). No client-side guard needed — the empty state copy
  handles this case.

### Filter application timing

- Desktop sidebar: apply live (same as GamesTab desktop behavior).
- Mobile drawer: apply on drawer close (same as GamesTab mobile behavior).
- On filter change: Flaws tab list query refetches, page resets to offset 0.
- On filter change from Games tab: does NOT trigger a Flaws tab refetch (lazy — only fetches when
  the Flaws tab is active).

### Shared state across tabs

Flaw filter state lives in a Zustand store (`useFlawFilterStore`), mirroring the existing
`useFilterStore`. Both tabs read from and write to the same store. Switching tabs preserves
selections in memory.

The game-metadata `FilterState` (platform / TC / rated / recency / opponent) is ALSO shared
across Games and Flaws tabs — the `useFilterStore` already handles this.

### URL sync (Flaws tab only)

On mount, read `?tag={TAG}&severity={SEV}` from the URL and initialize store from them.
On filter change (while Flaws tab is active), update URL params via `replace` (not `push` —
avoids polluting browser history with every toggle click).

URL param shapes:
- `?tag=low-clock&tag=result-changing` — multi-value (one `tag=` per selected tag).
- `?severity=blunder&severity=mistake` — multi-value (one `severity=` per selected severity).
- Default state (both severities, no tags) produces no URL params (clean URL).

### Tab trigger active state

The "Flaws" `TabsTrigger` uses the existing `variant="brand"` `TabsList` style. When active,
it uses the same brand-brown filled state as the other active triggers.

`data-testid="tab-flaws"` (desktop) / `data-testid="tab-flaws-mobile"` (mobile).

### Chip deep-link (Games tab → Flaws tab)

Phase 107 shipped tag chips as display-only (`cursor: pointer`, no navigation). In Phase 108,
chips become navigation triggers:

- Clicking a chip navigates to `/library/flaws?tag={TAG}`.
- The Flaws tab renders with that tag pre-selected, severity defaulted to M+B.
- `aria-label` changes from the Phase 107 `"Tag: {tag} — {definition}"` to
  `"Filter flaws by tag: {tag}"` — accurate for the navigable state.
- `cursor: pointer` remains (no change to visual affordance).

The chip's `data-testid="chip-{tagName}-{gameId}"` is unchanged.

### Returning user default subtab

Unchanged: `/library/games` for users with games, `/library/import` for new users. The Flaws
tab is accessible via the tab row, not the default redirect.
