<objective>
Add the new per-flaw list endpoint `GET /library/flaws` (D-05/D-07/D-08): the Pydantic schemas (`FlawListItem`, `LibraryFlawsResponse`), the repository `query_flaws` (a `SELECT f.* FROM game_flaws JOIN games` using the SAME `build_flaw_filter_clauses` from Plan 03, ordered recent-first, paginated), the service method, and the thin router route. Default severity M+B, page size 20, user-scoped.

Purpose: this is the data backbone of the Flaws subtab — a flat, paginable, one-row-per-flawed-position list. Reusing Plan 03's predicate builder enforces the cross-tab unification (Games EXISTS and Flaws SELECT share one WHERE-clause builder, SEED-038).
Output: schemas + repository query + service + route + router/IDOR/pagination tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → GET /library/flaws | untrusted query params (severity, tag, pagination) cross here |
| user → game_flaws rows | response must contain only the requesting user's flaws |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-108-10 | Information Disclosure (IDOR) | GET /library/flaws | mitigate | user_id is taken from `current_active_user` (never a request param); `query_flaws` filters `GameFlaw.user_id == user_id`. Verified by the IDOR isolation test (Task 3). |
| T-108-11 | Tampering | SQL injection via tag/severity | mitigate | `severity`/`tag` typed as `Literal` Query params (FastAPI 422 on invalid); flow through parameterized `build_flaw_filter_clauses`. Phase-tag rejection asserted. |
| T-108-12 | Denial of Service | unbounded pagination | mitigate | `limit` bounded `ge=1, le=100`; `offset` `ge=0`; default page size 20. |
</threat_model>
