<objective>
Build the backend vertical slice for in-app feedback: a new `feedback` table, its Alembic
migration, the request/response schemas (with a typed sentiment `Literal`), a thin
`POST /api/feedback` router guarded by a per-user rate limit, a repository writing the row,
and a service that derives cohort context (platform + ELO bucket) from the authenticated user
record and pushes a non-exception Sentry signal.

Purpose: deliver the durable system-of-record + the Sentry push signal (D-04, D-05) behind an
abuse-guarded endpoint (D-07) that accepts guests (D-08). This is a near-mechanical clone of the
existing `position_bookmarks` vertical slice (see PATTERNS.md).

Output: `feedback` table + migration; `Feedback` model; `FeedbackCreate`/`FeedbackResponse`
schemas; `feedback_repository.create_feedback`; `feedback_service.elo_bucket` +
`push_sentry_signal`; `feedback_limiter`; `POST /api/feedback`; backend tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/feedback | Untrusted JSON body (text, sentiment, page_url) crosses here; the authenticated identity comes from the JWT, not the body |
| backend → Sentry | Cohort tags (username/platform/elo_bucket) leave the trust boundary to a third-party sink |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-122-01 | Spoofing/Elevation | feedback row attribution | mitigate | `user_id` is taken from `current_active_user`, NEVER from the request body — the schema has no user_id field; repository receives `user.id` as an argument (D-04 / V4) |
| T-122-02 | Denial of Service | POST /api/feedback flooding | mitigate | per-user sliding-window rate limit (`feedback_limiter`, 5/3600s) → 429 (D-07) |
| T-122-03 | Tampering/DoS | oversized text or page_url payload | mitigate | Pydantic `Field(max_length=_MAX_FEEDBACK_LEN)` on text + `max_length=_MAX_PAGE_URL_LEN` on page_url → 422 at the boundary |
| T-122-04 | Tampering | SQL injection via text/page_url | mitigate | SQLAlchemy ORM parameterizes; no raw SQL in the feedback path |
| T-122-05 | Information Disclosure | PII leakage to Sentry | accept (low) | tags carry only username/platform/elo_bucket (intentional cohort context, D-05); raw email and full feedback text are NOT placed in tags — text/url/sentiment go in `set_context` only, message string is static |
| T-122-06 | Tampering | stored XSS via feedback text rendered later | accept (deferred) | no read UI in v1 (deferred per CONTEXT.md); React default-escaping + escape-on-render is required when the deferred admin read-UI phase is built — noted for that phase |
| T-122-SC | Tampering | npm/pip installs | accept | no packages installed this phase (RESEARCH Package Legitimacy Audit: N/A — all libs in lockfile) |
</threat_model>
