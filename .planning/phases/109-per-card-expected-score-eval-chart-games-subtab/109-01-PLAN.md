<objective>
Build the backend eval-chart series builder that extends the `GET /api/library/games`
`GameFlawCard` payload with three new inline fields — `eval_series` (white-perspective ES
per ply), `flaw_markers` (both players' B/M/I dots with an `is_user` discriminator), and
`phase_transitions` (≤2 transition plies) — for analyzed games only, with Wave 0 unit tests.

Purpose: Deliver the data contract the frontend EvalChart consumes (LIBG-10), reusing the
existing `flaws_service` mover-POV kernel and `eval_utils` sigmoid rather than re-deriving
either (D-01/D-04). No schema change, no migration, no new endpoint, no N+1 (D-05/D-10).

Output: `EvalPoint`/`FlawMarker`/`PhaseTransitions` Pydantic models, extended `GameFlawCard`,
`fetch_page_eval_positions` repository function, `_build_eval_series`/`_build_opponent_tags`
service helpers, an extracted `_resolve_increment` helper, and `test_eval_chart_service.py`.
</objective>

| client → GET /library/games | authenticated request; user_id derives from the JWT, never from a client param |
| service → game_positions query | new batched read of per-ply rows must stay scoped to the requesting user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-109-01 | Information Disclosure (IDOR) | `fetch_page_eval_positions` | mitigate | `GamePosition.user_id == user_id` WHERE clause (same pattern as `fetch_page_game_flaws`); user_id from `current_active_user`, not request body. A Wave 0 / router test asserts cross-user game_ids return no positions. |
| T-109-02 | Tampering (SQLi) | game_id IN list | mitigate | SQLAlchemy `.in_()` with a Python `list[int]`; parameterized, no f-string interpolation. |
| T-109-03 | Denial of Service (payload bloat) | new inline series fields | accept | Bounded by existing 20-game pagination; per-page positions ≈ 20 × ~100 plies; payload delta measured in plan 03 (D-05). No new input surface. |
