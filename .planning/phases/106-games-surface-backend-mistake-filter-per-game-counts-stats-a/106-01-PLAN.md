<objective>
Build the shared severity-math seam that both Phase 106 endpoints depend on: (1) the additive
`count_game_severities` kernel helper for per-game B/M/I counts (incl. inaccuracy), (2) the
`mistake_severity` boolean-EXISTS extension of `apply_game_filters`, and (3) the single SQL
transcription of the ES-drop math (sigmoid + mate Option-B + eval-AFTER N-1/N pairing) in the
new `library_repository`, guarded by the load-bearing SQL<->kernel cross-check fixture test.
This plan also lays down the Wave-0 test scaffolds from 106-VALIDATION.md so downstream plans
have sampling targets.

Per D1 (locked): the SQL handles ONLY the EXISTS filter, per-severity flagged-row count
aggregates, and the >=90% analyzed denominator. Tags come from re-calling the Phase 105 kernel
downstream. Do NOT refactor 105's private tag functions into a public row-shaped API.

CRITICAL user-color scoping (criterion 5): both the count helper AND the EXISTS filter are
USER-scoped. The kernel re-call (`classify_game_mistakes`, line 475-477) skips plies where
`mover != game.user_color`; the SQL EXISTS MUST apply the same restriction so that a game where
only the OPPONENT blundered does NOT match `severity=["blunder"]`. The mover parity convention is
`mover = "white" if ply even else "black"` (kernel line 210) — the EXISTS restricts to plies whose
parity equals the game's `user_color`. "EXISTS matches" means "the game contains >=1 USER ply of
the selected severity," consistent with the user-scoped per-game card counts.

Purpose: this is the one place where severity math is duplicated (SQL vs Python). Pinning both
to the kernel constants and the user-color scope, and guarding the seam with the cross-check test,
is what prevents drift (criterion 5).
Output: `count_game_severities` + `SeverityCounts`, the extended `apply_game_filters`, the new
`library_repository` SQL builders, and four named Wave-0 test groups.
</objective>
