# Phase 64 Plan 02: Alembic Migration for llm_logs Summary

Shipped Wave 1 of Phase 64: registered `LlmLog` in `alembic/env.py`, autogenerated + hand-edited the `create_llm_logs` migration (revision `85dfef624a19`), applied it to the dev DB successfully, verified the resulting schema exactly matches LOG-01 + LOG-03 (18 columns, FK `ON DELETE CASCADE`, 5 named indexes with `created_at DESC` preserved on the three composites), and landed a session-scoped smoke test that inspects the test-DB schema after `alembic upgrade head`. Full 945-test suite green; ty + ruff project-wide clean.

## What Changed

### Task 1: Register LlmLog in `alembic/env.py` + autogenerate the migration

Appended one line to `alembic/env.py` after the existing side-effect import block:

```python
from app.models.llm_log import LlmLog  # noqa: F401
```

Ran `uv run alembic revision --autogenerate -m "create llm_logs"` which produced `alembic/versions/20260420_211450_85dfef624a19_create_llm_logs.py` with `down_revision='179cfbd472ef'` (the phase-start head). Autogenerate correctly emitted:

- `op.create_table('llm_logs', ...)` with all 18 columns
- `postgresql.JSONB(astext_type=sa.Text())` on `filter_context`, `flags`, `response_json`
- `sa.Integer()` on `user_id` (matching `users.id` ‚Äî RESEARCH.md Pitfall 1 avoided)
- `sa.ForeignKeyConstraint(['user_id'], ['users.id'], ondelete='CASCADE')`
- 5 `op.create_index` calls with all locked names
- `postgresql_ops={'created_at': 'DESC'}` on the 3 composite indexes

This last item was a pleasant surprise ‚Äî RESEARCH.md ¬ßPitfall 2 flagged Alembic issues #1166/#1213/#1285 as DESC-dropping, but the current Alembic (‚â•1.13) respected the declared kwarg on this run. Task 2's hand-edit became a smaller cleanup pass than planned.

Autogenerate also emitted **unrelated drift** ‚Äî three `alter_column` calls changing `REAL ‚Üí Float(precision=24)` on `game_positions.clock_seconds`, `games.white_accuracy`, `games.black_accuracy`. Root cause: pre-existing mismatch between the ORM's `Float` declarations and the dev DB's historic `REAL` columns. Per plan db_notes, this is not Phase 64 scope ‚Äî Task 2 removes it from the migration.

Commit: `dddac62` ‚Äî `chore(64-02): register LlmLog in alembic/env.py + autogen llm_logs migration`

### Task 2: Hand-edit migration ‚Äî scope to llm_logs only + style cleanup

Rewrote the migration file to:

1. **Remove unrelated drift.** Deleted the three `alter_column` calls from both `upgrade()` and `downgrade()` ‚Äî those REAL‚ÜîFloat type changes on unrelated tables do not belong in a `create_llm_logs` migration.
2. **Normalize `cache_hit` server_default.** Changed `server_default='false'` (bare string) to `server_default=sa.text("false")` for consistency with `created_at`'s `server_default=sa.text("now()")` and other migrations in `alembic/versions/`.
3. **Standardize formatting.** Alphabetized imports (`sqlalchemy` before `alembic`), switched single quotes ‚Üí double quotes to match project ruff format.

What was **retained unchanged** from autogenerate:

- `postgresql.JSONB(astext_type=sa.Text())` on all 3 JSON columns
- `postgresql_ops={"created_at": "DESC"}` on exactly 3 composite indexes (upgrade half) + the 3 matching drop_index calls (downgrade half)
- `sa.Integer()` on the `user_id` FK column
- Inline `sa.ForeignKeyConstraint(["user_id"], ["users.id"], ondelete="CASCADE")` inside `op.create_table`
- All 18 column definitions and their exact types

Final shape: 18 `sa.Column` entries, 5 `op.create_index` calls, 5 `op.drop_index` calls, 1 `op.drop_table`. ruff check + ruff format clean.

Commit: `5972eb1` ‚Äî `refactor(64-02): scope llm_logs migration to the new table only`

## Deviations from Plan

**1. [Rule 3 ‚Äî Blocking issue] Removed unrelated autogenerate drift from the migration**
- **Found during:** Task 1 (autogenerate output review)
- **Issue:** Autogenerate emitted three `alter_column` REAL‚ÜíFloat diffs on `game_positions.clock_seconds`, `games.white_accuracy`, `games.black_accuracy` ‚Äî pre-existing drift between the ORM's `Float` declarations and the dev DB's historic `REAL` columns. Leaving these in the migration would mutate unrelated tables, violating the plan's scope.
- **Fix:** Task 2 removed all six `alter_column` calls (three in `upgrade()`, three in `downgrade()`) during the hand-edit pass. The migration now only creates `llm_logs` and its 5 indexes.
- **Files modified:** `alembic/versions/20260420_211450_85dfef624a19_create_llm_logs.py`
- **Commit:** `5972eb1`
- **Matches plan db_notes verbatim:** "If autogenerate picks up unrelated drift (e.g. from other models being out of sync with DB), scope the generated migration to the llm_logs create_table + 5 indexes only ‚Äî the plan is explicit about not modifying other tables."
- **Follow-up note:** The REAL vs Float drift is real (both ORM models use `Float` but the columns are `REAL` in Postgres). This should be handled in a dedicated future migration, not smuggled in under `create_llm_logs`. Logging under "Deferred" below.

**2. [Positive deviation ‚Äî plan assumption too pessimistic] Autogenerate preserved DESC kwargs**

- **Observation:** RESEARCH.md ¬ßPitfall 2 warned that Alembic autogenerate drops `postgresql_ops={"created_at": "DESC"}`. On this run (Alembic ‚â•1.13 with Python 3.13 + sqlalchemy 2.x), the kwarg was preserved verbatim in the generated file.
- **Action:** Task 2 retained the autogenerate output as-is; no re-addition needed. This is a style/defensive-code observation, not a correctness deviation ‚Äî the same three composite indexes still ship with DESC.
- **Commit:** n/a (no corrective change)

**3. [Task 3 has no file changes ‚Äî no task-commit]**
- **Found during:** Task 3 execution
- **Rationale:** The plan's Task 3 explicitly marks `<files>(no file changes ‚Äî DB side-effect + verification)</files>`.
