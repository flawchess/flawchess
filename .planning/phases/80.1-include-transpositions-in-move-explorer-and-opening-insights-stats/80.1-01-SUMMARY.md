# Phase 80.1 Plan 01: Repository SQL Plumbing — Summary

Added two sibling repository functions that compute resulting-position WDL aggregates, with Wave 0 fixtures covering convergence, single-order, filter-parity, color-passthrough, and edge cases. These functions are the SQL plumbing for Plans 02 and 03.

## What Changed

### `app/repositories/openings_repository.py`

Two new async functions inserted between `query_transposition_counts` and `query_opening_transitions`:

1. **`query_transposition_wdl(session, user_id, result_hash_list, ...filters) -> dict[int, tuple[int, int, int]]`**
   Sibling of `query_transposition_counts`; returns `{result_hash: (wins, draws, losses)}` per resulting position. Used by Plan 80.1-02 (Move Explorer field swap).

2. **`query_resulting_position_wdl(session, user_id, hash_list, ...filters) -> dict[int, tuple[int, int, int]]`**
   Same SQL shape; parameter named `hash_list` to reflect compute_insights call-site semantics (list of `resulting_full_hash` values from `query_opening_transitions`). Used by Plan 80.1-03 (Opening Insights field swap).

Both functions reuse the W/D/L `case()` expressions from `query_next_moves` (lines 401-420), call `apply_game_filters` with the identical 8-parameter set as their parent queries, and short-circuit on empty input lists. Missing hashes are omitted from the dict so callers can fall back to move-played WDL defensively.

### Implementation choices

**1. [Plan deviation - test scaffolding] Filter-parity test uses in-test SQL UPDATE**

- **Found during:** Task 3 (initial test run failed on filter-parity)
- **Issue:** `_seed_game_with_positions` in `tests/repositories/test_opening_insights_repository.py` hardcodes `time_control_bucket="blitz"` (only `time_control_seconds` is parameterized). `apply_game_filters` filters on `Game.time_control_bucket`, not `time_control_seconds`, so the originally-planned approach (passing `time_control_seconds=1500` to get rapid) silently produced two blitz games and the rapid filter dropped both.
- **Fix:** After seeding, run `UPDATE games SET time_control_bucket = 'rapid' WHERE id = :gid` against the test session to override the bucket on Game A. Documented inline.
- **Files modified:** `tests/repositories/test_opening_insights_repository.py` (test class only; helper signature unchanged).
- **Why not extend the helper:** Out of scope for this plan, which is constrained to two new repo functions + their tests. A helper signature change would touch all existing callers.

### User IDs

- Plan suggested user_id=80101 / 80102; I used user_id=1 (TestQueryTranspositionWdl) and user_id=10 (TestQueryResultingPositionWdl) to match the autouse fixtures in each test file. Per-test rollback isolates fixtures regardless. No functional impact.
