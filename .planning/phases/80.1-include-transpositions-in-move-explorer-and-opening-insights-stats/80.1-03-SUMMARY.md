# Phase 80.1 Plan 03: Opening Insights Service Field Swap — Summary

`compute_insights` now consumes the new `query_resulting_position_wdl` (Plan 01) and feeds resulting-position W/D/L/N into `_classify_row`, `_compute_score`, `compute_confidence_bucket`, `wilson_bounds`, and the `OpeningInsightFinding` field population via a `SimpleNamespace` adapter. The four downstream helpers stay byte-identical (D-07). Surfacing gate (HAVING n>=20) stays on move-played count per D-04. The 57%→61% click-through mismatch on the Opening Insights findings is closed by construction, paralleling the Move Explorer fix in Plan 02.

## What Changed

### `app/services/opening_insights_service.py`

**Imports (top of file):**
- Added `import logging` and `from types import SimpleNamespace` (Phase 80.1 adapter).
- Added `query_resulting_position_wdl` to the existing `app.repositories.openings_repository` import group.
- Module-level `logger = logging.getLogger(__name__)` declared after all module constants (no prior logger; consistent with `app/services/openings_service.py` pattern from Plan 02).

**`compute_insights` (function modified):**

Two changes inside the `try:` block:

1. **New batch call** before the per-color findings loop (after `parents_by_hash` is built, before the `sections` dict initialization):
   ```python
   all_resulting_hashes: list[int] = list(
       {int(r.resulting_full_hash) for color_rows in rows_by_color.values() for r in color_rows}
   )
   pos_wdl: dict[int, tuple[int, int, int]] = {}
   if all_resulting_hashes:
       pos_wdl = await query_resulting_position_wdl(...)
   ```
   Filter parameters are byte-for-byte identical to `query_opening_transitions` (Pitfall 1 mitigation; threat T-80.1-08). `color=None` per the call-site contract (mirrors `query_opening_transitions` filter behavior — both white-perspective and black-perspective games visiting the same position contribute to the same resulting-position summary).

2. **Inner loop body**: at the top of the `for row in rows_by_color[color]:` loop, lookup `pos_wdl.get(int(row.resulting_full_hash))`:
   - When present: `pos_row = SimpleNamespace(w=pos_w, d=pos_d, l=pos_l, n=pos_n)`
   - When missing (filter-mismatch defensive fallback per Pitfall 4): `logger.warning(...)` and `pos_row: Any = row`
   - All subsequent WDL-touching calls use `pos_row` instead of `row`:
     - `_classify_row(row)` → `_classify_row(pos_row)`
     - `_compute_score(row)` → `_compute_score(pos_row)`
     - `compute_confidence_bucket(row.w, row.d, row.l, row.n)` → `compute_confidence_bucket(pos_row.w, pos_row.d, pos_row.l, pos_row.n)`
     - `wilson_bounds(score, row.n)` → `wilson_bounds(score, pos_row.n)`
     - `OpeningInsightFinding(..., n_games=pos_row.n, wins=pos_row.w, draws=pos_row.d, losses=pos_row.l, ...)`
   - Non-WDL fields on `row` (`move_san`, `entry_san_sequence`, `entry_hash`, `resulting_full_hash`) continue to come from `row`.

**No changes** to: `_classify_row`, `_compute_score`, `_replay_san_sequence`, `_apply_display_name_parity`, `_compute_prefix_hashes`, `_attribute_finding`, `_dedupe_within_section`, `_robustness`, `_dedupe_continuations`, `_rank_section`, `_apply_eval_metrics_to_finding`, the eval pillar (`query_opening_phase_entry_metrics_batch` is already keyed on `resulting_full_hash` per Phase 76 / quick task 260506-u2b), or the section-cap / response-wrapping logic.

**No frontend changes.** Per D-07, schema field names (`n_games / wins / draws / losses / score / confidence / p_value / ci_low / ci_high`) stay identical; only their semantics shift. `OpeningFindingCard` score-badge tooltip copy is unchanged.

**New test class `TestComputeInsightsTranspositionWdl` (3 tests):**

1. **`test_finding_uses_resulting_position_wdl_when_diverges_from_move_played`** (canonical D-03/D-05 case):
   - Move-played row: `w=15, d=3, l=2, n=20` → score = 0.825 (major strength)
   - pos_wdl: `DIVERGE_HASH → (24, 6, 10)`, n=40 → score = 0.675 (still major strength)
   - Asserts: `finding.n_games == 40`, `finding.wins == 24`, `finding.draws == 6`, `finding.losses == 10`, `finding.score ≈ 0.675`, `classification == "strength"`, `severity == "major"`.

2. **`test_finding_classification_uses_resulting_position_score`** (classification flip):
   - Move-played row: `w=12, d=1, l=7, n=20` → score = 0.625 (minor strength under move-played)
   - pos_wdl: `FLIP_HASH → (10, 10, 20)`, n=40 → score = 0.375 (major weakness under pos)
   - Asserts: finding appears in `white_weaknesses` (NOT `white_strengths`); `classification == "weakness"`, `severity == "major"`, `score ≈ 0.375`, `n_games == 40`.

3. **`test_finding_n_games_validator_satisfied_when_pos_n_gt_move_played_n`** (Pitfall 3 regression):
   - Move-played row: `n=20` (exactly at the surfacing-gate boundary)
   - pos_wdl: `VALIDATOR_HASH → (15, 0, 10)`, n=25 → score = 0.60 (major strength)
   - Asserts the Pydantic `Field(ge=OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE)` validator does not raise; `finding.n_games == 25 >= MIN_GAMES_PER_CANDIDATE`.

## Deviations from Plan

### Auto-fixed Issues

None — plan executed as written.

### Implementation choices

**1. [Implementation choice — module logger]** Added `import logging` and a module-level `logger = logging.getLogger(__name__)` in `opening_insights_service.py` (no prior logger). This is the standard Python pattern, consistent with the parallel addition in Plan 80.1-02's `openings_service.py`. Required for the defensive fallback `logger.warning(...)` call.

**2. [Implementation choice — fallback type annotation]** The defensive fallback branch needs `pos_row: Any = row` to satisfy ty's union narrowing rules: `pos_row` flows into `_classify_row(pos_row: Any)` (which takes Any) and `_compute_score(pos_row: Any)`, but the conditional initialization with `SimpleNamespace` in one branch and `row` (a SQLAlchemy `Row[Any]`) in the other would otherwise give `pos_row` the joined type, which doesn't expose `.w/.d/.l/.n` as concrete attributes. Annotating as `Any` is consistent with the helpers' parameter types and keeps ty satisfied without exposing additional types.

**3. [Implementation choice — inline patch block extensions]** Six tests don't use `_run_compute` (they need `side_effect` on `mock_transitions` or `mock_attribution`). Each was extended with a new `patch(...)` for `query_resulting_position_wdl` and a `mock_pos_wdl.return_value = {...}` line mirroring the move-played WDL. The alternative — refactoring those tests to use `_run_compute` — would require expanding `_run_compute`'s signature with `transitions_side_effect`, `attribution_side_effect`, and similar plumbing, which is a larger refactor than the in-place patch addition.

### Plan compliance

- The plan listed the eight inline patch blocks roughly correctly (the plan said "six" but eight is the actual count). All eight were extended.
- Plan mentioned a new top-level test class. Implemented as `class TestComputeInsightsTranspositionWdl` with three pytest-asyncio test methods.
- Hash namespace 0xDD** chosen per the plan's suggestion (DIVERGE_HASH = 0xDD01, FLIP_HASH = 0xDD02, VALIDATOR_HASH = 0xDD03).

## Threat Surface Scan

No new attack surface. The new `query_resulting_position_wdl` call:
- Filters on `GamePosition.user_id == user_id` (same user-isolation key as siblings)
- Applies `apply_game_filters` with the identical 8-parameter set as `query_opening_transitions` (filter parity; mitigates T-80.1-08 tampering threat)
- Operates purely as an additive read-only aggregation on the same user-scoped tables

The defensive `logger.warning` on missing pos_wdl entry mitigates T-80.1-09 (repudiation) by surfacing the filter-mismatch anomaly with the missing `resulting_full_hash` for post-hoc debugging.

No threat flags.
