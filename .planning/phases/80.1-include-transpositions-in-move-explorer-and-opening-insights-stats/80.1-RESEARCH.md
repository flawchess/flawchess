## Summary

This phase is a targeted SQL aggregation rewrite on two backend surfaces with no frontend code changes. The core problem: Move Explorer rows and Opening Insights findings both currently compute WDL/score over games that played the exact candidate move from the exact entry position (move-played). After clicking through to a position summary, users see a different, higher (or lower) score because the summary counts all games that ever reached the position via any move order. This phase makes the row's headline number match the destination by switching both surfaces to resulting-position WDL.

**Surface 1 — Move Explorer (`get_next_moves` in `openings_service.py`):** `query_next_moves` already returns `result_hash` per row. `query_transposition_counts` already fetches total-games-reaching-position per `result_hash` under the same filters. The planner extends (or adds a sibling to) `query_transposition_counts` to also return W/D/L per `result_hash`. Then `get_next_moves` replaces the move-played W/D/L fields with the resulting-position values while keeping `game_count` unchanged.

**Surface 2 — Opening Insights (`compute_insights` in `opening_insights_service.py`):** `query_opening_transitions` already returns `resulting_full_hash` per row; the surfacing gate (n >= 20 HAVING clause) and the HAVING effect-size gate remain on move-played W/D/L (`n`, `w`, `d`, `l`). A second aggregation is added (either as a sub-CTE or a second pass after the query returns) that joins `game_positions` on `resulting_full_hash` to produce resulting-position W/D/L per row. Those resulting-position counts replace the values fed to `_classify_row`, `compute_confidence_bucket`, `wilson_bounds`, and `OpeningInsightFinding` field population. The existing eval pillar is already keyed on `resulting_full_hash` (from quick task 260506-u2b) and requires no changes.

**Primary recommendation:** For the Insights surface, implement the resulting-position WDL as a sub-CTE inside `query_opening_transitions` (single round trip, consistent filters, mirrors the eval-metrics pattern from `query_opening_phase_entry_metrics_batch`). For the Move Explorer surface, extend `query_transposition_counts` to return per-result_hash W/D/L alongside the existing transposition count (minimal change, reuses existing filter plumbing).

## Detailed Code Analysis

### Surface 1: Move Explorer — Current Shape

**`query_next_moves` (`openings_repository.py:372-454`)** [VERIFIED: direct read]

Returns one row per `(move_san, result_hash)` with columns:
```
(move_san, result_hash, game_count, wins, draws, losses)
```
Uses `COUNT(DISTINCT Game.id)` with `CASE WHEN` for W/D/L. Applies `apply_game_filters` after the JOIN. `result_hash` is `gp2.full_hash` (the position AFTER the candidate move).

**`query_transposition_counts` (`openings_repository.py:457-508`)** [VERIFIED: direct read]

Returns `{result_hash: transposition_count}`. Structure:
```python
select(
    GamePosition.full_hash.label("result_hash"),
    func.count(GamePosition.game_id.distinct()).label("transposition_count"),
)
.join(Game, Game.id == GamePosition.game_id)
.where(GamePosition.user_id == user_id, GamePosition.full_hash.in_(result_hash_list))
.group_by(GamePosition.full_hash)
```
Applies same `apply_game_filters` call. This is the function to extend with W/D/L per result_hash.

**`get_next_moves` (`openings_service.py:378-488`)** [VERIFIED: direct read]

Current assembly loop (lines 452-480):
```python
for row in move_rows:
    gc = row.game_count
    w, d, lo = row.wins, row.draws, row.losses  # <-- currently move-played
    wp = round(w / gc * 100, 1) if gc > 0 else 0.0
    score = (w + 0.5 * d) / gc if gc > 0 else 0.5
    confidence, p_value, _se = compute_confidence_bucket(w, d, lo, gc)
    moves.append(NextMoveEntry(
        move_san=row.move_san,
        game_count=gc,                  # <-- stays move-played
        wins=w,                          # <-- switch to pos W
        ...
        transposition_count=trans_counts.get(row.result_hash, gc),
    ))
```

After the change: `game_count` stays from `row.game_count`; W/D/L/score/confidence/p_value come from the extended transposition query keyed on `row.result_hash`. The denominator for percentages and Wilson becomes the resulting-position total (W+D+L from the position, not `gc`).

**Key implementation note:** The mute trigger in `MoveExplorer.tsx` reads the row's WDL denominator. After the switch that denominator is resulting-position N, which is always >= move-played N. This is the intended behavior per D-07.

### Surface 2: Opening Insights — Current Shape

**`query_opening_transitions` (`openings_repository.py:511-715`)** [VERIFIED: direct read]

Reads `row.w`, `row.d`, `row.n`. After D-03 these must be resulting-position counts. The function itself needs no code change — just new inputs.

**`compute_insights` inner loop (lines 498-565)** [VERIFIED: direct read]

Currently feeds `row.w`, `row.d`, `row.l`, `row.n` directly from `query_opening_transitions` into `_classify_row`, `compute_confidence_bucket`, `wilson_bounds`, and `OpeningInsightFinding`. After the change these four attribute accesses switch to resulting-position W/D/L/N.

Signature accepts `(w, d, losses, n)`. Computes Wilson score-test using only `w`, `d`, and `n` (losses not used in formula). Can be called identically on resulting-position W/D/L/N without modification.

**`wilson_bounds(p, n)`** [VERIFIED: direct read]

**`_rank_section`** [VERIFIED: direct read]

Reads `finding.score` and `finding.n_games`. Both flip to resulting-position; no change to the helper.

**`_robustness`** [VERIFIED: direct read]

Reads `finding.score` and `finding.n_games` via `wilson_bounds`. Follows automatically.

### `OpeningInsightFinding` Schema — No Structural Change

**`n_games: int = Field(ge=OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE)`** [VERIFIED: direct read]

The validator `ge=20` (current floor). After D-05, `n_games` becomes resulting-position N. Since every game that played the candidate move also has a `game_positions` row at the resulting position, resulting-position N >= move-played N always. The validator stays satisfied. No schema edit needed.

### `apply_game_filters` — Filter Consistency

**`apply_game_filters` (`query_utils.py:12-77`)** [VERIFIED: direct read]

**Critical for correctness:** The new resulting-position WDL aggregation (for both surfaces) MUST receive the identical `apply_game_filters` call as the parent query. Mismatched filters would produce WDL that disagrees with other filtered stats on the same page.

## Implementation Patterns

### Pattern 1: Extend `query_transposition_counts` (Move Explorer)

Add W/D/L case expressions matching `query_next_moves:401-420` exactly. The existing `COUNT(DISTINCT GamePosition.game_id)` becomes `COUNT(DISTINCT Game.id)` with three CASE WHEN branches:

```python
# Source: app/repositories/openings_repository.py:401-420 (verified)
win_case = case(
    (
        ((Game.result == "1-0") & (Game.user_color == "white"))
        | ((Game.result == "0-1") & (Game.user_color == "black")),
        Game.id,
    ),
    else_=None,
)
draw_case = case((Game.result == "1/2-1/2", Game.id), else_=None)
loss_case = case(
    (
        ((Game.result == "1-0") & (Game.user_color == "black"))
        | ((Game.result == "0-1") & (Game.user_color == "white")),
        Game.id,
    ),
    else_=None,
)
```

Return type changes from `dict[int, int]` to `dict[int, tuple[int, int, int, int]]` (total, W, D, L) or a TypedDict/dataclass. The existing `transposition_count` for the tooltip popover is total = W+D+L.

**Alternatively:** Add a sibling `query_transposition_wdl` that returns `dict[int, tuple[int, int, int, int]]`. Keeps the existing function signature intact for any other callers. The planner chooses based on whether other callers of `query_transposition_counts` exist.

### Pattern 2: Sub-CTE for resulting-position WDL in `query_opening_transitions` (Insights)

The eval pillar already demonstrates this pattern in `query_opening_phase_entry_metrics_batch` (`stats_repository.py:452`): it builds a `dedup_select` CTE of `(full_hash, game_id, user_color)` using the same `apply_game_filters`, then aggregates over it.

For resulting-position WDL in `query_opening_transitions`, the new aggregation needs:
- JOIN `game_positions` ON `full_hash = resulting_full_hash` (per-row, not per-game-id)
- Same user_id and game filters
- GROUP BY `resulting_full_hash`

The planner can implement this as either:
1. A sub-CTE within the same SQL statement (single round trip, cleaner)
2. A second Python-level query after `query_opening_transitions` returns (two DB round trips, but simpler SQL, analogous to how `get_next_moves` calls `query_transposition_counts` separately)

Option 2 (separate query) mirrors the existing Move Explorer pattern and requires minimal changes to the complex `query_opening_transitions` function body. Option 1 (sub-CTE) is a single round trip but increases the complexity of an already-complex function. The planner picks the cheaper shape per D-06.

### Pattern 3: Field Swap in `get_next_moves`

```python
# Current (move-played):
gc = row.game_count
w, d, lo = row.wins, row.draws, row.losses
wp = round(w / gc * 100, 1) if gc > 0 else 0.0
score = (w + 0.5 * d) / gc if gc > 0 else 0.5
confidence, p_value, _se = compute_confidence_bucket(w, d, lo, gc)

# After (resulting-position, using new dict lookup):
gc = row.game_count                   # unchanged
pos = pos_wdl.get(row.result_hash)   # from extended transposition query
pos_total, w, d, lo = pos if pos else (gc, row.wins, row.draws, row.losses)
wp = round(w / pos_total * 100, 1) if pos_total > 0 else 0.0
score = (w + 0.5 * d) / pos_total if pos_total > 0 else 0.5
confidence, p_value, _se = compute_confidence_bucket(w, d, lo, pos_total)
```

Note: the fallback to move-played values when `pos` is not found is defensive. In practice, if `query_next_moves` returns `result_hash`, the position must exist and `query_transposition_wdl` (under the same filters) will return at least `game_count` games for it.

### Pattern 4: Field Swap in `compute_insights` inner loop

```python
# Current (move-played):
cls = _classify_row(row)   # reads row.w, row.d, row.n
...
score = _compute_score(row)   # reads row.w, row.d, row.n
confidence, p_value, se = compute_confidence_bucket(row.w, row.d, row.l, row.n)
ci_low, ci_high = wilson_bounds(score, row.n)
finding = OpeningInsightFinding(
    n_games=row.n,
    wins=row.w,
    draws=row.d,
    losses=row.l,
    score=score, ...
)

# After: row.w/d/l/n come from resulting-position lookup (pos_wdl dict)
# The surfacing gate (HAVING n >= 20) still uses move-played n in SQL.
# _classify_row, compute_confidence_bucket, wilson_bounds, OpeningInsightFinding
# all receive pos_w, pos_d, pos_l, pos_n instead.
```

The cleanest approach: after `query_opening_transitions` returns its rows, run a batch query that returns `{resulting_full_hash: (pos_n, pos_w, pos_d, pos_l)}` — then substitute in the inner loop. This preserves the function's current structure while cleanly separating the two N sources.

## Common Pitfalls

### Pitfall 1: Filter Mismatch Between Move-Played and Resulting-Position Queries

**What goes wrong:** The new resulting-position WDL query uses different filters than `query_next_moves` / `query_opening_transitions`. Users see scores that don't match the position summary they click through to.

**Why it happens:** Copy-pasting the query without forwarding all filter parameters (e.g., forgetting `opponent_gap_min`/`opponent_gap_max`).

**How to avoid:** Call `apply_game_filters` with all 8 parameters identical to the parent query. Add a test that exercises the filter round-trip.

**Warning signs:** Score on the row differs from the position summary under any non-default filter combination.

### Pitfall 2: `_classify_row` Still Reading Move-Played `row.n`/`row.w`/`row.d`

**What goes wrong:** HAVING clause gates on move-played n (correct), but `_classify_row` is fed the same move-played values, so the switch has no effect on classification.

**Why it happens:** Passing `row` directly to `_classify_row` instead of a resulting-position-substituted object.

**How to avoid:** Either (a) create a small `SimpleNamespace` / dataclass with resulting-position W/D/L/N and pass that, or (b) add `w_pos`, `d_pos`, `l_pos`, `n_pos` columns to the `query_opening_transitions` return and update `_classify_row` to read those. Document clearly which `.n`/`.w`/`.d`/`.l` field is used where.

**Warning signs:** Classification output in tests matches old values when it should differ (transposition test fixture with two move orders converging on same hash).

### Pitfall 3: `n_games` Field Validator Fires

**What goes wrong:** `OpeningInsightFinding(n_games=pos_n, ...)` raises a Pydantic `ValidationError` when `pos_n < OPENING_INSIGHTS_MIN_GAMES_PER_CANDIDATE`.

**Why it can't happen in normal flow:** By construction, resulting-position N >= move-played N (every game that played the candidate move also visits the resulting position). Since the HAVING clause gates on move-played n >= 20, resulting-position n >= 20 always holds.

**Edge case to watch:** If `apply_game_filters` for the resulting-position query accidentally excludes some games that the move-played query included (filter mismatch), pos_n could fall below 20. This surfaces as a Pydantic validation error in tests, making Pitfall 3 a useful smoke-test indicator for Pitfall 1.

### Pitfall 4: WDL Fallback When `result_hash` Not Found in Transposition Dict

**What goes wrong:** `pos_wdl.get(row.result_hash)` returns `None` for a valid result_hash, silently using move-played WDL as a fallback. The row appears correct but hasn't actually switched.

**Why it happens:** Filtering mismatch or empty IN list edge case in the batch query.

**How to avoid:** Log or assert when the fallback path is hit in tests. The canonical test (two move orders converging on one result_hash) would catch this — if game_count != WDL total, the fallback is working.

### Pitfall 5: Deduplicated Resulting-Position WDL Double-Counts

**What goes wrong:** Two candidate rows with the same `resulting_full_hash` (different entry/candidate combinations) both fetch resulting-position WDL for that hash. The WDL is correct per-hash, but the dedup step (`_dedupe_within_section`) still runs on correct values, so no actual double-count. This is not a real bug.

**Confirmation:** The resulting-position WDL is keyed on `resulting_full_hash` and each hash has one aggregate value. Multiple candidate rows pointing to the same hash correctly see the same resulting-position WDL (consistent with the position summary).

### Key New Test Case (Canonical Transposition Verification)

The canonical test for "transposition inclusion is working" requires seeding two games that reach the same `result_hash` via different move orders:

```
Game A: plays "e4" from source_hash → result_hash (win)
Game B: reaches result_hash at some ply without going through source_hash (loss)
```

Then `get_next_moves(source_hash)` should return:
- `e4.game_count == 1` (only Game A played e4 from source)
- `e4.wins == 1` (Game A + Game B combined: 1 win + 1 loss... wait: both games visit result_hash)
- `e4.wins + e4.losses == 2` (both games visited result_hash under the same filters)

This is the signal test. A fixture without a genuine second move order is not testing transposition inclusion — it's testing the existing behavior. The planner must ensure at least one integration test seeds two converging paths.

## Performance Considerations

### Index Coverage for Resulting-Position WDL Queries

**Move Explorer (`query_transposition_wdl`):** [VERIFIED: model read]

The existing `query_transposition_counts` joins `GamePosition` on `(user_id, full_hash)`. The index `ix_gp_user_full_hash_move_san` covers `(user_id, full_hash, move_san)` — this index serves the new W/D/L aggregation as well (W/D/L come from the `Game` join, not from `game_positions` columns). The query is an IN-list lookup on result_hash values (typically 1-20 distinct hashes per page view), so the index scan is narrow.

**Opening Insights (second-pass or sub-CTE on `resulting_full_hash`):**

The insights query operates on a post-HAVING filtered set of rows (typically 5-50 candidates). The resulting-position WDL aggregation needs to join `game_positions` on `full_hash = resulting_full_hash` under `user_id`. The `ix_gp_user_full_hash_move_san` index on `(user_id, full_hash, move_san)` serves this lookup directly.

**No new indexes needed.** Both aggregations hit index patterns already used by existing queries. The position-summary query (which powers the click-through and already shows resulting-position WDL) uses the same index for the same user + full_hash pattern.

**Risk:** If implemented as a sub-CTE inside `query_opening_transitions`, PostgreSQL must evaluate the sub-CTE while still processing the complex transitions CTE + LAG window function. The planner should `EXPLAIN ANALYZE` on a heavy-user partition in the dev database or use the prod-DB tunnel (read-only) to confirm the plan doesn't degrade. The alternative (separate Python-level query after `query_opening_transitions` returns) avoids this risk entirely at the cost of one additional round trip.

## `query_opening_transitions` Structural Options for Resulting-Position WDL

The function is complex (205 lines, LAG window function, phase 71 hotfix comments, etc.). Two shapes for adding resulting-position WDL:

**Option A: Sub-CTE (single round trip)**

Add a `pos_wdl_cte` after the transitions CTE:
```sql
WITH transitions AS (...),
pos_wdl AS (
    SELECT
        gp.full_hash AS resulting_full_hash,
        COUNT(DISTINCT CASE WHEN ... THEN g.id END) AS pos_w,
        COUNT(DISTINCT CASE WHEN ... THEN g.id END) AS pos_d,
        COUNT(DISTINCT CASE WHEN ... THEN g.id END) AS pos_l
    FROM game_positions gp
    JOIN games g ON g.id = gp.game_id
    WHERE gp.user_id = :user_id
      AND gp.full_hash IN (SELECT DISTINCT resulting_full_hash FROM transitions)
      -- apply_game_filters here (same params)
    GROUP BY gp.full_hash
)
SELECT ..., pos_wdl.pos_w, pos_wdl.pos_d, pos_wdl.pos_l
FROM transitions
JOIN game ON ...
LEFT JOIN pos_wdl ON pos_wdl.resulting_full_hash = transitions.resulting_full_hash
GROUP BY ...
HAVING ...
```

The HAVING clause keeps reading move-played `n`/`score` (unchanged). The new `pos_w`/`pos_d`/`pos_l` columns are aggregated via `func.min()` (safe because all rows sharing the same resulting_full_hash have identical pos_wdl values from the CTE).

**Option B: Separate query (two round trips)**

After `query_opening_transitions` returns rows, collect distinct `resulting_full_hash` values and run a second batch query analogous to `query_transposition_counts`:
```python
async def query_resulting_position_wdl(
    session, user_id, hash_list, time_control, platform, rated,
    opponent_type, recency_cutoff, color, opponent_gap_min, opponent_gap_max
) -> dict[int, tuple[int, int, int]]:  # {resulting_full_hash: (w, d, l)}
```

Then in `compute_insights`, substitute the resulting-position values before calling `_classify_row`.

**Planner recommendation:** Option B is safer given the complexity of `query_opening_transitions`. It isolates the change, is easier to test independently, and follows the established pattern from `get_next_moves` + `query_transposition_counts`. Option A is more efficient (single DB round trip) but adds risk to an already-complex function. Given that insights are already < 1s even for heavy users (per Phase 70 context), the extra round trip is unlikely to matter.

## Open Questions

1. **`query_transposition_counts` return type change vs. sibling function**
   - What we know: it's imported only by `get_next_moves`
   - What's unclear: planner preference for extending vs. adding sibling (both acceptable per D-02)
   - Recommendation: add a sibling `query_transposition_wdl` returning `dict[int, tuple[int, int, int]]` (W, D, L); keep the existing function intact for the `transposition_count` used in `NextMoveEntry.transposition_count`

2. **Option A vs. Option B for `query_opening_transitions`**
   - What we know: both are acceptable per D-06
   - What's unclear: query planner behavior with IN-subquery referencing a previous CTE (known to cause issues in some PostgreSQL versions — see `stats_repository.py:500-528` comment about planner Nested Loop hangs)
   - Recommendation: Option B (separate query) for safety; Option A if planner confirms clean EXPLAIN

3. **`_classify_row` receives row object vs. substituted values**
   - What we know: `_classify_row` reads `row.w`, `row.d`, `row.n`
   - What's unclear: whether to pass a `SimpleNamespace` with pos values, or add new attributes to the row
   - Recommendation: use a `SimpleNamespace(w=pos_w, d=pos_d, l=pos_l, n=pos_n)` adapter; this avoids modifying `_classify_row` and keeps the substitution explicit
