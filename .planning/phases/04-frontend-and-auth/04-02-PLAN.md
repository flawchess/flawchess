    2. Copy the POLYGLOT_RANDOM_ARRAY from the python-chess source as an array of 781 BigInt literals. The source is `chess/polyglot.py` in the python-chess repo. Each value must be written as `0x...n` (BigInt literal notation). This is the largest part of the file — approximately 800 lines of data.

    To extract the values, the executor should run:
    ```bash
    uv run python -c "
    import chess.polyglot
    for i, v in enumerate(chess.polyglot.POLYGLOT_RANDOM_ARRAY):
        comma = ',' if i < 780 else ''
        print(f'  0x{v:016X}n{comma}')
    "
    ```
    And wrap in `const POLYGLOT_RANDOM_ARRAY: bigint[] = [...]`.

    3. Implement `computeHashes` function:
       ```typescript
       import { Chess, Square, Color, PieceSymbol } from 'chess.js';

       // Piece type mapping: chess.js uses 'p','n','b','r','q','k'
       const PIECE_TYPE_MAP: Record<PieceSymbol, number> = {
         p: 1, n: 2, b: 3, r: 4, q: 5, k: 6,
       };

       // Square to index (0-63): chess.js uses 'a1'='a1' etc.
       // Polyglot square mapping: a1=0, b1=1, ..., h1=7, a2=8, ..., h8=63
       function squareToIndex(square: Square): number {
         const file = square.charCodeAt(0) - 97; // 'a' = 0
         const rank = parseInt(square[1]) - 1;    // '1' = 0
         return rank * 8 + file;
       }

       function toSignedBigInt64(n: bigint): bigint {
         const unsigned = BigInt.asUintN(64, n);
         if (unsigned >= 9223372036854775808n) {
           return unsigned - 18446744073709551616n;
         }
         return unsigned;
       }

       export interface ZobristHashes {
         whiteHash: bigint;
         blackHash: bigint;
         fullHash: bigint;
       }

       export function computeHashes(chess: Chess): ZobristHashes {
         let whiteHash = 0n;
         let blackHash = 0n;
         let fullHash = 0n;

         const board = chess.board(); // 8x8 array of {type, color, square} | null

         for (const row of board) {
           for (const piece of row) {
             if (!piece) continue;
             const pieceType = PIECE_TYPE_MAP[piece.type];
             const colorPivot = piece.color === 'w' ? 0 : 1;
             const squareIdx = squareToIndex(piece.square as Square);
             const index = 64 * ((pieceType - 1) * 2 + colorPivot) + squareIdx;
             const hash = POLYGLOT_RANDOM_ARRAY[index];

             fullHash ^= hash;
             if (piece.color === 'w') {
               whiteHash ^= hash;
             } else {
               blackHash ^= hash;
             }
           }
         }

         return {
           whiteHash: toSignedBigInt64(whiteHash),
           blackHash: toSignedBigInt64(blackHash),
           fullHash: toSignedBigInt64(fullHash),
         };
       }
       ```

       IMPORTANT: The `fullHash` computed this way (XOR of all individual piece hashes) will NOT match `chess.polyglot.zobrist_hash(board)` because the polyglot full hash also includes castling rights, en passant, and turn. However, the Python backend's `_color_hash` function (used for white_hash and black_hash) works the same way — pure piece XOR. And the Python `full_hash` uses `chess.polyglot.zobrist_hash(board)` which includes extra bits.

       To match the Python backend exactly:
       - `whiteHash` and `blackHash`: piece-only XOR — the code above is correct.
       - `fullHash`: Must also XOR in castling, en passant, and turn bits from the polyglot array. The polyglot array indices 768-780 encode these:
         - 768: white king-side castling
         - 769: white queen-side castling
         - 770: black king-side castling
         - 771: black queen-side castling
         - 772-779: en passant file (a=772, b=773, ..., h=779)
         - 780: turn (XOR if black to move)

       For the analysis use case, the backend analysis endpoint accepts `target_hash` which is compared against `white_hash`, `black_hash`, or `full_hash` depending on `match_side`. Since most users will use "white" or "black" match_side, and those are pure piece hashes, the simpler piece-only approach works. For `full` match_side, we need the extra bits.

       Add castling/en-passant/turn XOR to the fullHash computation. Use `chess.getCastlingRights()` or parse the FEN string to extract castling availability. Chess.js provides `chess.fen()` — parse the castling field (3rd space-separated field).

    4. Export a convenience function for the analysis API:
       ```typescript
       export function hashToString(hash: bigint): string {

       ```
       The API request will send the hash as a string. The backend `AnalysisRequest.target_hash` is `int` — need to also add a `@field_validator` on the backend to accept string input and convert to int. OR just use `Number()` knowing precision issues. Best approach: modify `AnalysisRequest.target_hash` to `int | str` with a validator, or just `int` and have the frontend send `Number(signedBigint)`. Given many hashes exceed Number.MAX_SAFE_INTEGER, the safest is:
       - Backend: change `target_hash: int` to accept string: add `@field_validator('target_hash', mode='before')` that converts str to int.
       - Frontend: send hash as string in the JSON body.

       Update `app/schemas/analysis.py` to add this validator:
