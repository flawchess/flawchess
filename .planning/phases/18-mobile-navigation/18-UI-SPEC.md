## Spacing Scale

Declared values (multiples of 4, using Tailwind defaults):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps within bottom bar buttons (`gap-1`) |
| sm | 8px | Bottom bar vertical padding (`py-2`) |
| md | 16px | Mobile header horizontal padding (`px-4`), drawer content padding |
| lg | 24px | Drawer section padding |
| xl | 32px | — |
| 2xl | 48px | — |
| 3xl | 64px | Bottom bar height budget (bottom bar buttons `py-2` + icon 20px + label 12px + gap = ~56px; `pb-16` main content offset) |

Exceptions:
- Touch targets: bottom bar buttons must be minimum 44px tall (per CLAUDE.md UX-01 requirement, addressed in Phase 19, but the Phase 18 bottom bar should not be shorter than 56px to stay within range). Use `py-2` (8px top + 8px bottom) with icon 20px + label 12px + gap 4px = 52px rendered height — acceptable.
- Safe-area offset: `pb-safe` on bottom bar adds `env(safe-area-inset-bottom)` (0px on non-notched devices, ~34px on iPhone X+ in standalone PWA mode). Main content uses `pb-16 sm:pb-0` to clear bottom bar on mobile; no desktop change.
- Z-index: bottom bar `z-40`; drawer overlay `z-50` (above bottom bar).

## Color

All colors use the established shadcn CSS custom property system from `index.css`. Light and dark modes are handled automatically via the `.dark` class.

| Role | CSS Token | Light Value | Dark Value | Usage |
|------|-----------|-------------|------------|-------|
| Dominant (60%) | `--background` | oklch(1 0 0) ≈ #ffffff | oklch(0.145 0 0) ≈ #1a1a1a | Page backgrounds, bottom bar background, mobile header background |
| Secondary (30%) | `--secondary` / `--muted` | oklch(0.97 0 0) ≈ #f7f7f7 | oklch(0.269 0 0) ≈ #383838 | Drawer content area background |
| Accent (10%) | `--primary` | oklch(0.205 0 0) ≈ #282828 | oklch(0.922 0 0) ≈ #e8e8e8 | Active tab indicator in bottom bar (active icon + label color) |
| Destructive | `--destructive` | oklch(0.577 0.245 27.325) ≈ #e53935 | oklch(0.704 0.191 22.216) ≈ #ef5350 | Logout button in More drawer (destructive variant or text-destructive) |
| Border | `--border` | oklch(0.922 0 0) ≈ #ebebeb | oklch(1 0 0 / 10%) | Bottom bar top border (`border-t border-border`); mobile header bottom border |
| Muted foreground | `--muted-foreground` | oklch(0.556 0 0) ≈ #757575 | oklch(0.708 0 0) ≈ #aaaaaa | Inactive tab icons and labels in bottom bar |

Accent reserved for:
- Active bottom bar tab: icon and label text color (`text-primary` when `isActive()` returns true)
- Active route indicator in the More drawer (same `text-primary` class on the active link)

No new accent usage is introduced. Destructive color is used solely for the Logout button in the More drawer.

## Component Inventory

New components introduced in this phase:

### MobileBottomBar

Fixed bottom navigation bar, visible only on mobile (<640px).

| Property | Value |
|----------|-------|
| Position | `fixed bottom-0 inset-x-0` |
| Visibility | `flex sm:hidden` |
| Z-index | `z-40` |
| Background | `bg-background` |
| Border | `border-t border-border` |
| Safe-area | `pb-safe` (from tailwindcss-safe-area) |
| Height | auto — driven by content (~52px rendered + safe-area offset) |
| Layout | `flex` with 4 equal children (`flex-1` each) |

Tab items (3 direct + 1 More):

| Tab | Icon (Lucide) | Label | Route | data-testid |
|-----|---------------|-------|-------|-------------|
| Import | `DownloadIcon` (or `ImportIcon`) | "Import" | `/import` | `mobile-nav-import` |
| Openings | `LayoutIcon` (or `GridIcon`) | "Openings" | `/openings` | `mobile-nav-openings` |
| Global Stats | `BarChartIcon` (or `TrendingUpIcon`) | "Global Stats" | `/global-stats` | `mobile-nav-global-stats` |
| More | `MenuIcon` | "More" | opens drawer | `mobile-nav-more` |

Icon size: `h-5 w-5` (20px). Label: `text-xs`. Gap between icon and label: `gap-1` (4px). Button padding: `py-2` (8px each side).

Active state: `text-primary`. Inactive state: `text-muted-foreground`.

Container `data-testid`: `mobile-bottom-bar`.

### MobileHeader

Simplified header bar, visible only on mobile (<640px).

| Property | Value |
|----------|-------|
| Visibility | `block sm:hidden` |
| Background | `bg-background` |
| Border | `border-b border-border` |
| Safe-area | `pt-safe` (from tailwindcss-safe-area) |
| Padding | `px-4 py-3` |
| Layout | `flex items-center justify-between` |

Contents:
- Left: Brand text "Chessalytics" — `text-xl font-semibold tracking-tight text-foreground` (`data-testid="mobile-header-brand"`)
- Right: Current page title (top-level route name only, e.g. "Openings" not "Openings > Moves") — `text-sm text-muted-foreground` (`data-testid="mobile-header-page-title"`)

Page title source: a route-to-title map keyed off `useLocation().pathname`, same approach as existing `isActive()`.

Container `data-testid`: `mobile-header`.

### MobileMoreDrawer

Bottom sheet drawer opened by "More" tab. Built with `shadcn add drawer` (vaul 1.1.2).

| Property | Value |
|----------|-------|
| Component | shadcn Drawer with `direction="bottom"` |
| Open control | Controlled — `open` / `onOpenChange` props |
| Dismiss | Tap backdrop or swipe down — vaul built-in |
| Z-index | `z-50` (vaul overlay handles) |

Contents (top to bottom):
1. Username/email — `text-sm font-medium text-foreground` (source: `useAuth` hook; check if profile is available or extend hook)
2. Separator line
3. All nav links (Import, Openings, Global Stats) — each wrapped in `<DrawerClose asChild>` so tap closes drawer + navigates. Active route highlighted with `text-primary`.
4. Separator line
5. Logout button — `variant="ghost"` with `text-destructive` class, or destructive variant. Wrapped in `<DrawerClose asChild>`.

Each nav link `data-testid`: `drawer-nav-{route-slug}` (e.g. `drawer-nav-import`, `drawer-nav-openings`, `drawer-nav-global-stats`).
Logout button `data-testid`: `drawer-logout`.
Drawer content `data-testid`: `mobile-more-drawer`.

### NavHeader (modified)

Existing component — add `hidden sm:flex` to become desktop-only. No functional changes.

### ProtectedLayout (modified)

Add `pb-16 sm:pb-0` to `<main>` wrapper to prevent content hiding behind fixed bottom bar on mobile. Add `MobileHeader`, `MobileBottomBar`, and `MobileMoreDrawer` renders.

## Interaction Contracts

### Bottom Bar Active State

- `isActive()` logic: reuse verbatim from `NavHeader`. `/openings` uses `startsWith('/openings')`; all others use exact match.
- Active tab: `text-primary` on icon + label.
- Inactive tab: `text-muted-foreground` on icon + label.
- No animation on tab switch — instant color change only.

### More Drawer Open/Close

- Trigger: tap "More" button in bottom bar.
- Open animation: slide up from bottom — vaul default spring animation (no custom timing needed).
- Close triggers: tap backdrop, swipe down, tap any nav link (via `DrawerClose asChild`), tap Logout (via `DrawerClose asChild`).
- Controlled state: `const [moreOpen, setMoreOpen] = useState(false)` in `ProtectedLayout`. Pass `open={moreOpen}` and `onOpenChange={setMoreOpen}` to `MobileMoreDrawer`.

### Responsive Switching

Use CSS-only Tailwind breakpoint classes. No JavaScript media query listeners.

| Component | Mobile (<640px) | Desktop (>=640px) |
|-----------|-----------------|-------------------|
| NavHeader | hidden (`hidden sm:flex`) | visible |
| MobileHeader | visible (`block sm:hidden`) | hidden |
| MobileBottomBar | visible (`flex sm:hidden`) | hidden |
| MobileMoreDrawer | rendered (controlled hidden/shown) | never opened (bottom bar hidden) |
| Main content pb offset | `pb-16` | `pb-0` (`sm:pb-0`) |

### Safe-Area Insets

- Bottom bar: `pb-safe` class (from tailwindcss-safe-area).
- Mobile header: `pt-safe` class (from tailwindcss-safe-area).
- `viewport-fit=cover` is already set in `index.html` (Phase 17).
- CSS import: add `@import "tailwindcss-safe-area";` to `index.css` after existing imports.

## Copywriting Contract

This phase introduces navigation UI only. No data-entry forms, no empty states, no destructive confirmations beyond Logout.

| Element | Copy | Notes |
|---------|------|-------|
| Bottom bar — "More" label | "More" | Source: CONTEXT.md locked decision |
| More drawer — header | No title text (username/email as identity anchor) | Drawer shows user identity, not a generic "More" heading |
| More drawer — logout button | "Logout" | Matches existing desktop nav (`nav-logout`) |
| Mobile header — brand | "Chessalytics" | Matches existing desktop brand text |
| Mobile header — page titles | "Import" / "Openings" / "Global Stats" | Top-level names only; source: NAV_ITEMS labels |
| Drawer nav links | "Import" / "Openings" / "Global Stats" | Same labels as bottom bar and desktop nav |

No empty state copy needed — navigation is always populated for authenticated users.
No error state copy needed — navigation itself has no error conditions in this phase.
No destructive confirmation needed — Logout executes immediately on tap (matches existing desktop behavior; no confirmation dialog).

## Accessibility Contracts

- Bottom bar container: `<nav aria-label="Mobile navigation">`.
- "More" button (no visible text context): `aria-label="More navigation options"` — satisfies CLAUDE.md icon-only button rule (even though it has text label, the aria-label is belt-and-suspenders).
- Drawer focus trap: handled by vaul/Radix internally.
- Drawer close on ESC: handled by vaul internally.
- All bottom bar tab icons: `aria-hidden="true"` (decorative, text label present).
- Active tab: no explicit `aria-current` required — color contrast differentiates active state visually; screen readers read link labels naturally.

## Known Limitations (Out of Scope for Phase 18)

- **Android keyboard overlap**: When a software keyboard opens, `position: fixed` bottom bars reposition on Android Chrome. No inputs exist on the pages with bottom bar in this phase, so this is a non-issue. Flag for Phase 19 if inputs are added.
- **Context-sensitive bottom bar (Openings page)**: Replacing nav buttons with board controls on the Openings page is marked experimental in CONTEXT.md. Deferred to post-Phase 18 experiment.
- **`useAuth` username/email**: The More drawer spec shows username/email at the top. Whether `useAuth` exposes this needs verification during implementation (RESEARCH.md open question #2). Implementer should check `useAuth` hook and extend if needed or fall back to showing a generic "Account" label.
