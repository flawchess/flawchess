# Phase 151.1 Plan 03: Grading engine hook + MockWorker unit tests Summary

**A second, fully independent Stockfish WASM Web Worker (`useStockfishGradingEngine`) that grades Maia candidate moves via ONE `searchmoves`-restricted MultiPV root search, streaming a pv[0]-keyed, white-POV-normalized grade cache — structurally isolated from the primary eval-bar engine and proven under a MockWorker harness that reproduces Plan 01's confirmed multipv-reordering landmine.**

## Accomplishments

- `useStockfishGradingEngine.ts` — second `new Worker('/engine/stockfish-18-lite-single.js')`
  instance, a structural sibling of `useStockfishEngine.ts`: same idle/thinking/stopping state
  machine, same `stopPendingRef` stop-before-go serialization (FLAWCHESS-7V guard), same tab-hide
  pause and unmount cleanup — but a completely separate instance with its own UCI configuration.
  Never imports, mutates, or reads the primary engine's state (SC3 isolation by construction).
- Per-search dynamic `setoption name MultiPV value <N>` (N = candidate count) followed by
  `go depth 14 searchmoves <ucis> movetime 2500` — ONE root search per settled `(fen,
  candidateSans)` pair, not N child searches (D-04).
- Grade map keyed by `pv[0]`'s SAN (via a local UCI->SAN replay helper mirroring `Analysis.tsx`'s
  `bestSanFromPv`), NEVER by the `multipv` index — the exact caveat Plan 01's real-binary spike
  confirmed (multipv is an eval rank, not a stable move identity).
- Per-FEN grade cache (`Map<fen, Map<san, MoveGrade>>`, FIFO-capped at `GRADE_CACHE_MAX=256`,
  mirrors `useMaiaEngine`'s cache pattern) with an ELO-drag short-circuit: re-requesting an
  already-graded candidate subset for the same FEN serves from cache with zero worker round-trip.
- Named degradation knobs (`GRADING_TARGET_DEPTH=14`, `GRADING_MOVETIME_SAFETY_CAP_MS=2500`,
  `GRADE_CACHE_MAX=256`, `RAPID_STEP_DEBOUNCE_MS=150`) — tunable for SC4 without touching logic.
- MockWorker test suite (10 tests) proving: classic-Worker instantiation, the no-setoption-at-init
  sequence, the exact search command shape, pv[0]-vs-multipv-index keying under a simulated
  mid-search reordering, white-POV sign normalization, the cache short-circuit, the FLAWCHESS-7V
  stale-guard/deferred-re-go, tab-hide pause, and unmount cleanup.

## Decisions Made

- MultiPV is set dynamically per-search rather than once at init, since the candidate count (and
  therefore the required MultiPV value) varies per board position — unlike the primary engine's
  fixed `MULTIPV = 2`.
- The pv[0]->SAN replay helper is a private local function inside the hook file rather than an
  import from `Analysis.tsx`'s `bestSanFromPv` — hooks should not depend on page-level modules;
  the ~15-line replay logic is mirrored, not shared, keeping the hook self-contained.
- The search-command's `searchmoves` list is built from the UNION of already-cached + newly-needed
  SANs (not just the ungraded subset) so a single coherent MultiPV ranking is produced even when a
  later request adds one new candidate to an already-partially-graded position.
