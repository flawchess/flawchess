# Phase 172 ‚Äî UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope note (read first)

This phase is ~85% invisible background scheduling (SEED-106 D1‚ÄìD5). The **entire** user-visible
surface this contract governs is:

1. A new **book marker** on `VariationTree.tsx` (`resolveMarkerIcon`) and the board corner marker
   (`boardMarkers.tsx`) ‚Äî precedence `severity > gem > book` (D8).
2. Gem badges (existing icon/color, unchanged) now **appearing progressively ahead of the cursor**
   instead of lazily at it ‚Äî a timing change, not a visual redesign of the gem badge itself.
3. **No new page, no new modal, no new CTA, no new empty/error state, no progress UI.** The
   unanalyzed-game lazy behavior + Analyze pill (D3) is existing UI, untouched.

Several template sections below are therefore marked N/A by design ‚Äî this is intentional, not an
omission. Do not invent surface area to fill them.

## Color

The app's page-level 60/30/10 (forced dark mode, unchanged, not touched by this phase):

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `oklch(0.145 0 0)` (`--background`) | Page background |
| Secondary (30%) | `oklch(0.205 0 0)` (`--card`) | Cards, move-list panel, popovers |
| Accent (10%) | `#8B5E3C` (`--brand-brown`) | Primary CTA, brand nav ‚Äî **not used anywhere in this phase** |
| Destructive | shadcn `--destructive` token (`oklch(0.577 0.245 27.325)`, already aliased as `CLOCK_LOW_TIME_URGENT`) | Not used ‚Äî this phase adds no destructive action |

**New token this phase adds ‚Äî the book-marker color.** Corner-marker "identity" colors already
follow a strict single-hue-per-kind convention in `theme.ts` (severity red/orange/yellow, Maia gem
violet `MAIA_ACCENT` chroma 0.20, Stockfish blue chroma 0.16, FlawChess gold chroma 0.14). The book
marker must read as **quieter than a gem** and must not be mistaken for a flaw. Two established
"quiet/neutral" precedents already exist in `theme.ts` at deliberately near-zero chroma ‚Äî
`MOVE_QUALITY_PENDING = 'oklch(0.65 0.02 260)'` and `EVAL_CHART_PHASE_LABEL = 'oklch(0.65 0 0)'` ‚Äî
both used exactly for "this is context, not a signal" surfaces. The book marker follows the same
idiom: lowest chroma of any corner-marker color, so it visually recedes behind every alert/insight
badge.

| Constant | Value | File | Rationale |
|----------|-------|------|-----------|
| `BOOK_MARKER_COLOR` | `oklch(0.60 0.04 250)` (muted slate-blue) | `frontend/src/lib/theme.ts` (new export, grouped near `MAIA_ACCENT`/`STOCKFISH_ACCENT`) | Chroma 0.04 ‚Äî deliberately the lowest of any corner-marker hue (gem 0.20, Stockfish 0.16, FlawChess 0.14, severities 0.13‚Äì0.19). Hue 250 sits in the existing cool-blue family (adjacent to Stockfish 255 / gauge-neutral 260) without colliding ‚Äî reads as "informational," not as a second Stockfish/engine identity. Lightness 0.60 keeps the white glyph legible (matches the existing badge-fill lightness band of ~0.40‚Äì0.72 used by gem/severity/FlawChess). |

**Accent reserved for:** `BOOK_MARKER_COLOR` is reserved **exclusively** for the book/opening-theory
corner marker and its `VariationTree` list-icon twin (`BookIcon`, mirroring `GemIcon`). It must not be
reused for any other affordance (buttons, chips, chart lines) ‚Äî this is a marker-family identity
color, not a general-purpose "muted blue," exactly like `MAIA_ACCENT`/`FLAWCHESS_ENGINE_ACCENT` are
reserved for their respective engine identities and nothing else.

**Gem badge color: unchanged.** `MAIA_ACCENT` / `GEM_GLYPH` stay exactly as shipped (Phase 163). This
phase changes gem **timing** (progressive sweep vs. lazy-at-cursor), never gem **appearance**.

## Component Contract ‚Äî the book marker

### New files (mirror the existing gem precedent exactly ‚Äî "one record, two consumers")

| File | Purpose | Mirrors |
|------|---------|---------|
| `frontend/src/lib/bookGlyph.ts` | `export const BOOK_GLYPH: { color: string } = { color: BOOK_MARKER_COLOR };` | `gemGlyph.ts` |
| `frontend/src/components/icons/BookIcon.tsx` | White lucide `BookOpen` on a `BOOK_GLYPH.color` circle, same props shape (`className`, `style`, `aria-hidden`) as `GemIcon`/`BlunderIcon`/`MistakeIcon`, same 24-unit viewBox, `<title>Opening theory</title>` for the SC 163-REVIEW IN-02 accessible-name precedent | `GemIcon.tsx` verbatim (swap `Gem` ‚Üí `BookOpen`, swap `GEM_GLYPH` ‚Üí `BOOK_GLYPH`, swap the title string) |

**No `data-testid` on the book glyph ‚Äî deliberate, not an omission.** CLAUDE.md mandates `data-testid` on
every *interactive* element. The book marker is non-interactive: no popover, no click target, glance-only,
`aria-hidden` with an accessible `<title>`. It therefore follows the `BlunderIcon` / `MistakeIcon` /
`GemIcon` precedent, none of which carry a testid. Tests assert the marker via its rendered glyph, not a
testid. Do not add one "for consistency."

**Icon choice: `BookOpen` (lucide-react).** Already the codebase's established glyph for "opening" ‚Äî
used in `App.tsx` nav, `Import.tsx`, `GameCard.tsx`/`LibraryGameCard.tsx` opening-name rows,
`tagVisuals.ts`, `flawComparisonMeta.ts`. Reusing it (rather than a new icon) keeps "book marker" and
"opening" reading as the same concept everywhere in the app. Do not use the plain `Book` glyph ‚Äî
`BookOpen` is the shipped convention.

### No popover on the book marker (deliberate, matches the severity-glyph precedent)

Gems get an `InfoPopover` (`GemMoveBadge`) because there are two real numbers to explain (ELO rung +
Maia probability). The book marker has nothing comparable to disclose ‚Äî "this ply is inside known
opening theory" is the entire fact. **Do not build a `BookMoveBadge` popover.** Render the plain
`BookIcon` exactly like `BlunderIcon`/`MistakeIcon` render today (no popover, no extra tap target) ‚Äî
this is the "quieter than gem" requirement expressed structurally, not just chromatically: gem is the
only corner-marker kind with an interactive disclosure; book, like severity, is glance-only.

### Precedence wiring (D8 ‚Äî `severity > gem > book`)

**`VariationTree.tsx`'s `resolveMarkerIcon`** (real branching logic, since `FlawMarkerEntry` can
legitimately carry both a `severity` and a `gem` field simultaneously):

```typescript
function resolveMarkerIcon(flaw: FlawMarkerEntry | undefined): {
  show: boolean;
  Icon: typeof BlunderIcon;
  isGem: boolean;
  isBook: boolean; // new
} {
  if (flaw != null && (flaw.severity === 'blunder' || flaw.severity === 'mistake')) {
    return { show: true, Icon: flaw.severity === 'blunder' ? BlunderIcon : MistakeIcon, isGem: false, isBook: false };
  }
  if (flaw?.gem) return { show: true, Icon: GemIcon, isGem: true, isBook: false };
  if (flaw?.book) return { show: true, Icon: BookIcon, isGem: false, isBook: true };
  return { show: false, Icon: MistakeIcon, isGem: false, isBook: false };
}
```

`MoveListMarker` branches on `isGem` for the popover-wrapped path (unchanged) and falls through to
the plain `<Icon className={className} aria-hidden />` render for both severity AND book (book needs
no new branch in `MoveListMarker` itself ‚Äî it already takes the "plain icon" path today). Add
`book?: boolean` to `FlawMarkerEntry` (`VariationTree.tsx:140-179`), set from
`plyIndex < opening_ply_count` at the `Analysis.tsx` call site that builds `moveListMarkers`.

**`boardMarkers.tsx`'s `SquareMarker`/`SquareMarkerBadge`** (construction-time exclusivity, not
runtime branching ‚Äî mirrors the existing `gem`/`severity` pattern exactly):

```typescript
export interface SquareMarker {
  square: string;
  severity?: FlawSeverity;
  gem?: boolean;
  book?: boolean; // new ‚Äî mutually exclusive with severity/gem by construction, like gem is today
  label?: string;
  labelColor?: string;
}
```

`SquareMarkerBadge` gets a third branch (after the existing `if (marker.gem)` block, before the
`severity` fallback), rendering the same circle-badge shape with `BOOK_GLYPH.color` fill and a
`BookOpen` glyph at the same `GEM_ICON_DIAMETER_RATIO` sizing. At the `Analysis.tsx` call site
(`boardSquareMarkers`), append the book marker only when the square carries **neither** an existing
severity **nor** gem marker for that square ‚Äî the same guard shape the gem append already uses:

```typescript
if (
  isBookPly &&
  lastMove != null &&
  !base.some((m) => m.square === lastMove.to && (m.severity != null || m.gem === true))
) {
  return [...withGem, { square: lastMove.to, book: true }];
}
```

## Sweep visibility contract (D5 / SEED-106 discretion questions)

Two open UX questions from the phase brief, settled here:

| Question | Decision | Rationale |
|----------|----------|-----------|
| **Transition/animation as a gem or book badge lands ahead of the cursor?** | **No animation.** Badges render exactly the way they do today when the lazy per-node resolution completes ‚Äî an instant appear, no fade/pop/highlight. | The sweep's entire design goal (D5) is that the page "does not feel slower than it does today" ‚Äî and, symmetrically, it should not feel *busier* or *more animated* either. A landing transition would draw attention to background work happening, which cuts against "quiet, ahead-of-cursor delivery." It is also the zero-cost implementation: the sweep writes into the exact same `gemByNode`/marker-map render path the lazy resolution already uses, so no new CSS/animation library is needed. Matches project precedent (`feedback_no_flame_icons`, `feedback_popover_copy_minimalism`) of resisting decorative chrome that doesn't carry information. |
| **Is sweep progress surfaced (spinner, percentage, toast)?** | **No progress UI of any kind.** No spinner, no toast, no counter, no "Analyzing gems‚Ä¶" copy anywhere. | Explicitly the preferred answer per CONTEXT.md's Claude's Discretion note ("no spinner was specified"). A visible progress indicator (a) implies a duration worth watching, undermining the "shouldn't feel slower" goal, (b) has no natural home ‚Äî there is no header/toolbar region on `/analysis` reserved for background-task chrome, and adding one would be new UI surface this phase explicitly must not create, and (c) the sweep's own contract (D5) already guarantees it never blocks anything the user is doing ‚Äî there is nothing actionable a progress UI would let the user do differently. |

**Consequence for the executor:** the sweep is invisible except for its output (badges appearing on
plies not yet visited). No loading state, no skeleton, no disabled state anywhere in this contract.
