<objective>
Build `useGemSweep` ‚Äî the background cascade with its own dedicated workers and its own scheduler.

**Design choice (CONTEXT puts the scheduler's shape under Claude's Discretion; recording it here explicitly):** dedicated worker instances, NOT a migration onto `workerPool.ts`'s existing (unused, unit-tested) priority queue. Rationale: `workerPool.dispatchNext()` only assigns pending work to IDLE slots and never preempts an in-flight low-priority slot, so under saturation a newly-arrived live request still queues behind whatever finishes first ‚Äî priority alone does not satisfy "never starve the live position". Dedicated instances make starvation structurally impossible instead of merely unlikely, at the cost of one extra Worker thread, which is bounded by the mobile gate, the one-in-flight cap, the idle pacing, and the reduced movetime. The pool-based unification stays available as a follow-up once a second real priority consumer exists.

Purpose: This is ~the whole phase's risk (D-05). Everything else is mechanical.
Output: A hook that resolves gems ahead of the cursor without the live position ever waiting on it.
</objective>
