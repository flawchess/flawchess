<objective>
Let the remote eval worker claim tier-1 (and tier-2) single-game "analyze" requests through the existing lease/submit contract, not just the tier-3 idle backlog. Three tightly-coupled backend changes, no DB migration:

1. Wire `lease_eval_game` to `claim_eval_job` (tier-1 > tier-2 > tier-3, lease_expiry + SKIP LOCKED, stale-lease sweep) instead of calling `_claim_tier3_derived` directly.
2. Thread an opaque `job_id` token through the lease → submit round-trip so the submit handler can stamp `eval_jobs.status='completed'` for tier-1/tier-2 claims, while tier-3 (`job_id=None`) keeps writing nothing.
3. Drop the worker idle poll `DEFAULT_IDLE_SLEEP` from 5.0 to 1.0 so an idle remote worker picks up a freshly-enqueued tier-1 quickly.

Purpose: shorten click-to-pickup latency in the server-busy overflow case (FCFS) — when the server pool is mid-game on another job, a second idle machine can pick up a freshly-enqueued single-game analysis. The server's in-process drain still usually wins tier-1 when idle (no network hop), so this targets only the overflow case.

Output: edits to four existing files (one test file, two app files, one script). No new files, no migration, no frontend change.

Out of scope (explicit DEFERRED fence from CONTEXT.md — DO NOT implement): bias-to-fast-box / server grace-yield, and interruptible tier-3 / ply-batch chunking.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| remote worker → /api/eval/remote/lease | Operator-authenticated machine claims a job; server assigns the opaque job_id |
| remote worker → /api/eval/remote/submit | Operator-authenticated machine echoes a worker-controllable job_id and eval payload |
| claim_eval_job ↔ server in-process drain | Two concurrent claimants compete for the same eval_jobs rows (FCFS) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-121-01 | Tampering | submit_eval — attacker/buggy worker echoes an arbitrary `job_id` | mitigate | Stamp UPDATE is guarded by `WHERE id == job_id AND status == 'leased'`. A crafted job_id can at worst mark an already-`leased` job `completed` (skipping its eval); it cannot touch a pending/completed/unrelated job. Acceptance: R5 late-submit test asserts a non-'leased' row is unchanged. |
| T-121-02 | Spoofing / Access Control | both lease + submit endpoints | mitigate | `require_operator_token` dependency is UNCHANGED on both handlers (constant-time hmac compare, fail-closed 403/401). Task 2 explicitly forbids weakening it. Acceptance: existing auth tests in test_eval_worker_endpoints.py stay green. |
| T-121-03 | Elevation of Privilege | tier-1 claim via claim_eval_job vs EVAL_AUTO_DRAIN_ENABLED gate | accept | claim_eval_job gates ONLY tier-3 on EVAL_AUTO_DRAIN_ENABLED; tier-1/tier-2 are intentionally ungated (a freshly-enqueued explicit request must be claimable). No new privilege — the same gate the server drain uses. Rationale: tier-1 is a user-initiated request, not idle backlog. |
| T-121-04 | Tampering (double-claim) | server drain + remote worker both claim same job | mitigate | tier-1/tier-2 use `SELECT FOR UPDATE OF ej SKIP LOCKED` inside claim_eval_job — PostgreSQL guarantees one claimant per row. tier-3 double-claim is pre-existing and accepted (idempotent eval). Acceptance: SKIP LOCKED is DB-level, covered by existing QUEUE-06 tests; soak (manual UAT) confirms under live concurrent load. |
| T-121-05 | Denial of Service | oversized submit body via job_id path | accept | `SubmitRequest.evals` already capped at `MAX_SUBMIT_EVALS=1024`; adding a scalar `job_id: int | None` does not widen the body. No new DoS surface. |
| T-121-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase (edits to existing files only). No legitimacy checkpoint needed; no new dependency added. |
