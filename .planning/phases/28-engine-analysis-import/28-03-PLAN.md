  <action>
    **1. Create scripts/reimport_games.py (per D-11):**

    Structure mirrors `scripts/backfill_positions.py` exactly. The script:

    a) **CLI args (per D-11):** `--user-id N` (single user) or `--all` (all users), mutually exclusive, required. Add `--yes` / `-y` flag to skip confirmation prompt.

    b) **For each target user:**
    - Query `import_job_repository` (or directly query ImportJob model) to find completed import jobs for the user. Each job has `platform` and `username` fields -- these tell us which platforms the user has imported from and their username on each platform.
    - Print: "User {user_id}: {N} games across platforms: {platforms}. This will DELETE all games and re-import from scratch."
    - If `--yes` not provided, prompt for confirmation: "Proceed? [y/N] "
    - Call `game_repository.delete_all_games_for_user(session, user_id)` to delete all games + positions (CASCADE handles positions).
    - Commit the delete.
    - For each platform the user has imported from, trigger a fresh import. The simplest approach: reset the import job's `last_synced_at` to None and `status` to a state that allows re-import, then call the import service. OR call the platform clients directly.

    **Important design decision:** The re-import should use the existing import pipeline (`import_service.py`) to ensure all the new eval extraction and accuracy logic runs. The cleanest approach:

    Option A: Delete games, then programmatically create new import jobs and run them through `_run_import_job`. This requires understanding the full job lifecycle.

    Option B: Delete games, reset the existing import job's `last_synced_at` to None, and let the next normal import cycle pick them up. But this doesn't actively trigger re-import.

    Option C: Delete games, then directly call the platform fetch + flush functions. This is more self-contained.

    **Recommended: Option A** -- use the existing `_run_import_job` function. The script should:
    1. Find the user's completed import jobs (one per platform)
    2. Delete all games for the user
    3. For each import job, reset `last_synced_at` to None and set `status` back to `pending` (or whatever the initial state is)
    4. Run `_run_import_job` for each job

    The executor should read `import_service.py` fully to understand the job lifecycle and determine the correct approach. Key functions to examine: `_run_import_job`, `start_import`, `ImportJobState`.

    c) **Progress reporting:** Print per-user progress: "Deleting {N} games for user {id}...", "Re-importing from {platform}...", "Done: {N} new games imported."

    d) **Error handling:** Per-user try/except. If one user fails, log warning and continue to next user. Print final summary: "{N} users processed, {M} failed."

    e) **Constants:** `_BATCH_SIZE = 10` (per STATE.md critical constraint, same as backfill script).

    f) **Script header:**
    ```python
    """Re-import games for user(s) to populate engine analysis data.

    Deletes all existing games and positions for specified user(s), then
    re-imports from scratch using the updated pipeline (which now extracts
    lichess per-move evals and chess.com accuracy scores).

    Usage:
        uv run python scripts/reimport_games.py --user-id 42
        uv run python scripts/reimport_games.py --all --yes
    """
    ```
