# Phase 94.1: Code Review Report

**Reviewed:** 2026-05-23
**Depth:** standard
**Files Reviewed:** 11 source files + 1 migration

## Summary

The implementation is structurally sound and the high-risk concerns flagged in the brief have largely been mitigated: SQL injection is prevented via SQLAlchemy bindparams, the import-completion `create_task` fires OUTSIDE the orchestrator's session scope (verified at `import_service.py:490-504`), and eval-drain opens a fresh `read_session` for the user-distinct query before firing Stage B (`eval_drain.py:556-564`). Sentry rules are followed (no f-strings in error messages, contexts attached via `set_context`).

One concrete bug exists in the eval-drain hook: the per-user `count_pending_evals` call is awaited sequentially **inside** an `async with read_session` block, and the `compute_stage_b` `create_task` is also created from within the same `async with`. While `create_task` is not itself an await of a query, the broader issue is that `count_pending_evals(read_session, user_id)` repeatedly hits the DB for every distinct user in a drain batch — fine for one user per batch, but a 10-user batch yields 10 sequential round-trips inside one session, blocking the drain loop. This is N+1-shaped but worse: it is on the hot drain path. See BLOCKER below for a stronger correctness concern.

Several semantic mismatches are also surfaced (the `n_games` column stores cell-count, not game-count; the chip is gated on `value != None` but the DB column allows `0` n_games rows; `cdf_snapshot` records the local compute date, not the CDF artifact's snapshot month).

The 4 plans 03-08 dovetail cleanly. The structural-finding mode flagged no cross-module hazards beyond what is captured below.

## Critical Issues

### CR-01: `n_games` column stores `n_cells_floor`, not game count — semantic mismatch with column name, schema, and chip gating

**File:** `app/services/user_benchmark_percentiles_service.py:196, 250`
**Issue:** The migration (`alembic/versions/20260524_000000_add_user_benchmark_percentiles.py:76`) defines `n_games` as `Integer NOT NULL` documented as "Number of canonical-slice games that contributed to ``value``" (per repository docstring `user_benchmark_percentiles_repository.py:61`). However, both `compute_stage_a` and `compute_stage_b` write `n_games=n_cells_floor`, where `n_cells_floor` is the **count of (elo_bucket, tc_bucket) cells that passed the per-metric HAVING floor** — not a game count. The two quantities differ by up to ~25× (max 5 elo × 4 TC cells vs. dozens-hundreds of games per cell).

Downstream consequences:

1. The CONTEXT brief lists this as "agent deviation". It is not just a label drift — the column is now semantically misleading for any future consumer who queries it as a game count (e.g., a future operator dashboard, a CSV export, a debug log, or a future "show n_games in the chip popover" feature). The column comment in the migration says nothing about cells.

2. When `value` is computable but `n_cells_floor == 0` (below floor), the row is written with `n_games=0` AND `percentile=NULL`. The chip front-end gates on `percentile != null` (`endgame_service.py:1376-1378`, `2345-2347`), so the chip is suppressed correctly. However, a row with `n_games=0` is still a record of "this user attempted to materialise but was below floor" — and the migration's `NOT NULL` constraint accepts the misleading 0.

3. The backfill script's outcome classifier (`backfill_user_percentiles.py:431-452`) infers "below_floor" by querying `percentile IS NULL` post-hoc rather than reading `n_games`. So the column has no current consumer — but that means the bug is latent until a future consumer trusts the column name.

**Fix:** Either (a) rename the column to `n_cells_floor` in a follow-up migration so the name matches the value, or (b) compute and store actual canonical-slice game counts. Option (b) is correct semantically (it matches the docstring and the column name), but adds one extra `COUNT(*)` per metric per user — likely cheap. Pick one and update the docstrings, the migration column comment, and the README in the phase artifact.

```python
# Option (a) — rename in follow-up migration:
sa.Column("n_cells_floor", sa.Integer(), nullable=False),

# Option (b) — actually compute game count (single COUNT in the same CTE):
# Add a third sub-query alongside value_raw / n_cells_floor returning
# `count(distinct game_id)` over the canonical slice.
```

## Warnings

### WR-01: Per-user `count_pending_evals` called sequentially inside an active read session — N+1 on the hot eval-drain path

**File:** `app/services/eval_drain.py:556-564`
**Issue:** After draining a batch (up to `_DRAIN_BATCH_SIZE = 10` games), the loop opens one `read_session`, then for each distinct user_id in the batch issues `count_pending_evals(read_session, user_id)` sequentially. With `_DRAIN_BATCH_SIZE=10`, the worst case is 10 round-trips inside one session before the next drain tick can begin. Each query is on the indexed `(user_id, evals_completed_at)` column pair so per-query cost is small, but the loop is on the hot path and blocks the next batch pick. A single aggregated query would return the answer in one round-trip:

```sql
SELECT user_id, count(*) FILTER (WHERE evals_completed_at IS NULL) AS pending
FROM games
WHERE user_id = ANY(:affected_ids)
GROUP BY user_id
HAVING count(*) FILTER (WHERE evals_completed_at IS NULL) = 0
```

The brief's concern 5 explicitly called this pattern out as a concern; the `.distinct()` was applied at the user_id collection step, but the per-user pending count is still N+1.

**Fix:** Replace the loop with a single aggregated query that returns only the users whose pending count is zero; iterate that list to fire `create_task(compute_stage_b(uid))`.

### WR-02: `cdf_snapshot` is set to `date.today()` rather than the actual CDF artifact's snapshot month

**File:** `app/services/user_benchmark_percentiles_service.py:197, 251`
**Issue:** The column docstring says ``cdf_snapshot`` is "Date of the CDF artifact used for interpolation" (`user_benchmark_percentiles_repository.py:63`). The CDF artifact carries `BENCHMARK_DB_SNAPSHOT_MONTH = "2026-03"` (constant in `global_percentile_cdf.py:42`). But the service writes `cdf_snapshot=date.today()` — i.e., the local compute date, not the CDF snapshot month.

After CDF regen (which only happens at SKILL.md-driven intervals), users computed before regen will still have their `cdf_snapshot` set to whatever `date.today()` was at compute time — totally unrelated to which CDF version they were interpolated against. The intended use case ("inspect which CDF a stored percentile came from") cannot be answered by this column.

**Fix:** Either store `BENCHMARK_DB_SNAPSHOT_MONTH` parsed as a date (e.g., `"2026-03"` → `date(2026, 3, 1)`) so the column actually identifies the CDF, or repurpose the column to a `computed_on` semantic and rename + update the docstring. The cleaner option:

```python
from app.services.global_percentile_cdf import BENCHMARK_DB_SNAPSHOT_MONTH
# parse "YYYY-MM" → date(YYYY, MM, 1) once at module scope:
_CDF_SNAPSHOT_DATE = date.fromisoformat(f"{BENCHMARK_DB_SNAPSHOT_MONTH}-01")
# ...
cdf_snapshot=_CDF_SNAPSHOT_DATE,
```

### WR-03: `selected_users_cte(source="single_user")` produces `SELECT :user_id::int` — `tc_bucket` column absent but referenced downstream

**File:** `app/services/canonical_slice_sql.py:127, 208, 280, 354, 372, 417`
**Issue:** For `source="single_user"`, `selected_users_cte` emits `SELECT :user_id::int AS user_id` only — no `tc_bucket` column. Downstream the per-metric CTEs reference `su.tc_bucket` in their projection (`per_user_cte_score_gap` line 208: `su.tc_bucket AS tc_bucket`, also line 280 in achievable, lines 354/372/417 in section2). For `source="single_user"`, these references resolve to a column that does not exist in the `selected_users` CTE.

Reading more carefully: the per-metric CTE wraps the `selected_users` join via `JOIN selected_users su ON su.user_id = g.user_id` (line 217), and then projects `su.tc_bucket AS tc_bucket`. PostgreSQL will raise `column su.tc_bucket does not exist` for the single_user variant. The test suite (per Plan 03 SUMMARY) verifies byte-identical CDF output for `source="benchmark"` only; the `source="single_user"` path is exercised by the new tests in Plan 05, which presumably pass — meaning either (a) I am missing something, or (b) the canonical_slice_sql.py rendering coincidentally works (e.g., `tc_bucket AS tc_bucket` resolves to a different column from `g`/elsewhere).

Examining more carefully: `per_user` GROUP BY uses `tc_bucket` (line 230), and `per_user_values` selects `tc_bucket` (line 233). If `su.tc_bucket` does not exist, the query fails at parse time — unless `g.time_control_bucket` is being projected and that's the lookup... but it isn't. The CTE explicitly writes `su.tc_bucket AS tc_bucket`. This needs end-to-end verification against a real DB (run the single_user query for a real user_id on dev DB) — if Plan 05's tests pass against a real Postgres, then this is a false alarm. If they pass only against mocks, this is a runtime BLOCKER.

**Fix:** Validate by running `compute_stage_a(user_id=<known_dev_user>)` against the dev DB (not against mocks). If the query indeed fails, add a `NULL::text AS tc_bucket` column to the single_user `selected_users` CTE so downstream references resolve:

```sql
selected_users AS (SELECT :user_id::int AS user_id, NULL::text AS tc_bucket)
```

### WR-04: Repository docstring claims `PercentileRow` is a TypedDict — actually a frozen dataclass

**File:** `app/repositories/user_benchmark_percentiles_repository.py:105`
**Issue:** The `fetch_for_user` docstring says: "values are PercentileRow TypedDicts." `PercentileRow` is a `@dataclass(frozen=True)` (line 25), not a TypedDict. ### WR-05: Implicit `CREATE CAST (varchar AS benchmark_metric) ... WITH INOUT AS IMPLICIT` weakens type safety project-wide

**File:** `alembic/versions/20260524_000000_add_user_benchmark_percentiles.py:62`
**Issue:** The migration installs an implicit cast from any varchar to the `benchmark_metric` ENUM. The justification ("asyncpg sends Python `str` bind parameters as `varchar`") is real, but the standard fix is to add a single explicit `CAST(:metric AS benchmark_metric)` at each raw SQL call site — which is what `backfill_user_percentiles.py:443, 475` already does. Once those sites use `CAST(...)`, the implicit cast is no longer required for any code path in the codebase, but remains installed as a Postgres-wide rule.

Consequences:

1. Future ENUMs may follow this precedent (one ENUM, one implicit-cast migration), encouraging a pattern that erodes the type-safety guarantee Postgres ENUMs are designed to provide.
2. Postgres will silently accept `'arbitrary_string'::varchar` wherever a `benchmark_metric` is expected at the prepared-statement level. The "still raises at runtime on invalid string" claim is correct for INSERTs/UPDATEs (Postgres validates the cast on row write), but the implicit cast also affects predicate evaluation (`WHERE metric = 'typo'` now type-checks and silently returns zero rows instead of erroring early).

**Fix:** Remove the `CREATE CAST` and rely on the explicit `CAST(:metric AS benchmark_metric)` at the (currently 2) raw-SQL call sites. SQLAlchemy ORM operations already use the ENUM type descriptor and do not need the implicit cast. If the cast must stay, document at the migration head why all 4 alternatives (per-site cast, SQLAlchemy ORM, `astext`-style comparison, server-side `LIKE` predicate) are infeasible.

### WR-06: `selected_users_cte` and `per_user_cte_for` cache and reuse a single `selected_users` CTE name across two compute queries in `_compute_metric_for_user` — confirmed safe, but no test asserts it

**File:** `app/services/user_benchmark_percentiles_service.py:130-152`
**Issue:** `_compute_metric_for_user` builds two SQL strings (`raw_sql` and `floor_sql`), each starting with `WITH selected_users AS (SELECT :user_id::int AS user_id), per_user_values AS (…)`. SQLAlchemy `text(...).bindparams(user_id=user_id)` is called twice on two distinct strings. This works, but it is a maintenance hazard: a future change that adds a 3rd query (e.g., a count of canonical-slice games for `n_games` per CR-01) duplicates the bindparam plumbing 3×. A small helper, or a single `WITH selected_users AS (...), per_user_values AS (...) SELECT … UNION ALL SELECT …` query, would centralise the bindparam.

This is not a bug per se, but the brief flagged "two queries instead of single dual-CTE" as an agent deviation; the deviation increases the surface area of bindparam plumbing and the cost of the single-user compute by ~2× round-trips.

**Fix:** Collapse the two queries into one using a single `WITH` block that emits both metrics. Sketch:

```sql
WITH selected_users AS (SELECT :user_id::int AS user_id),
     per_user_values_raw  AS ( ... apply_floor=False ... ),
     per_user_values_floor AS ( ... apply_floor=True ... )
SELECT
  (SELECT avg(metric_value)::float FROM per_user_values_raw)  AS value_raw,
  (SELECT count(*)::int FROM per_user_values_floor)           AS n_cells_floor;
```

### IN-03: `_compute_and_count.row_before` value used only when `row_after` is True — the `not row_before` branch always returns True post-row check

**File:** `scripts/backfill_user_percentiles.py:418-453`
**Issue:** The function executes 3 queries (row_exists pre, compute, row_exists post, then a percentile lookup) to classify the outcome. The final logic (lines 447-453) returns `True` whether `row_before` was True or False, so the pre-compute probe at line 419 is unused in the actual return path. Reading carefully:

- If `row_after` is False → return None (no row written, never read row_before)
- If `row_after` is True and `pctl_row is None` (which means the percentile lookup itself returned no row — but row_after just said there IS a row, so this is unreachable unless the row was deleted between the two queries) → return None
- Otherwise → return True

`row_before` is queried but never used to differentiate the return. This is dead-ish code that adds an extra query per user per metric (currently 1 user has 4 metrics × 2 stages × probes = ~8 unnecessary queries). The function's docstring says it returns `False` for "below floor", but no code path actually returns False — that classification is delegated to the percentile-null caller logic in `_backfill_user`. Wait — re-reading `_backfill_user:373-378`: it does check the return value `False` branch:

```python
if upserted is None: ... no_canonical_games
elif upserted:       ... upserted
else:                ... skipped_below_floor
```

But `_compute_and_count` never returns False explicitly (only True or None). So `skipped_below_floor` is unreachable and the count will always be 0.

**Fix:** The function's classifier is broken. When `row_after` is True AND `pctl_row.percentile IS NULL`, it should return `False` (below-floor). Currently it returns True. Update:

```python
if pctl_row is None or pctl_row.percentile is None:
    return False  # row exists but below-floor (percentile=NULL)
return True
```

Removing the unused `row_before` probe simplifies the function further.
