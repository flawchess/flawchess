Per the user decision in the gap-closure scope ("rename column, NOT add a new game-count column — the stored value already matches the new name"), this plan ships a reversible Alembic rename migration plus all downstream code/test renames.

This plan MUST land AFTER 94.1-09 (the bindparam fix + dev backfill HUMAN-UAT) so the rename runs against a table that actually contains real rows — providing visible confidence that the rename preserves data, not just empty schema.

Purpose: Eliminate a latent semantic landmine; align the column name with what is physically stored; prevent future operator/dashboard consumers from trusting a misleading name.
Output: 1 new migration + ORM/repository/service renames + ~25 test-fixture renames.
