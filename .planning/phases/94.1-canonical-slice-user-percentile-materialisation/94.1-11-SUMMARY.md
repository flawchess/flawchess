### Task 5 — HUMAN-UAT (BLOCKING checkpoint — SUPERSEDED BY PLAN 13)

**Status: SUPERSEDED on 2026-05-23.**

The Task 5 HUMAN-UAT was started by the user but surfaced two correctness issues that fall outside Plan 11's scope:

1. The `value` stored in `user_benchmark_percentiles` was wrong for multi-cell users (user 28 stored `+0.1204` for `achievable_score_gap`; manual floor-respecting computation produced `−0.0322`, a ~4× magnitude error with sign flip).
2. Stage B re-fired multiple times during transient eval-drain states, briefly writing intermediate (wrong) values that were visible to the chip.

Root cause of (1): `_compute_metric_for_user` reads `value_raw` from a CTE with `apply_floor=False`, so cells with `n < 20` are folded into the unweighted average. The CDF was built with `apply_floor=True`, breaking SC-7 parity. Root cause of (2): `users_with_zero_pending` only gates on `count_pending_evals == 0`, not on "no active import for this user", so the gate flickers across eval-drain batches during an active import.

User decision (2026-05-23): the right fix is to **drop the `n_cells_floor` column entirely** rather than rename it (the rename Plan 11 did is reverted by Plan 13's drop migration). The column is write-only from production code's POV — it is only compared `== 0` as a binary gate, and that gate becomes implicit once the value query uses `apply_floor=True` (no row written for below-floor users).

Plan 11's implementation work IS preserved on the branch:
- Task 1 (rename migration): supersededby Plan 13's drop migration. Both stay in history. dev DB will end up without the column.
- Task 2 (source-wide rename): reverted by Plan 13 (all `n_cells_floor` references removed).
- **Task 3 (TanStack endgameOverview invalidation): KEPT.** This fix remains valid and is unrelated to the column issue.
- Task 4 (pre-PR gate): no commit; one-shot verification only.

The HUMAN-UAT in Plan 13 covers what Task 5 here was meant to cover, plus the new correctness checks.
