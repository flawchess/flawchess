<objective>
Extend the remote-worker contract additively so MultiPV=2 second-best flows from upgraded
workers into the JSONB blobs, while un-upgraded workers keep processing full-ply jobs
without error (MPV-02, SC3, D-03/D-04). Three changes: (1) inline optional fields on
`SubmitEval`; (2) thread a parallel `second_best_map` through `_apply_submit` and reuse
Plan 02's `_build_flaw_multipv2_blobs` / `_run_multipv2_pass`; (3) upgrade the worker client
`_eval_positions` to call `evaluate_nodes_multipv2` and emit the second-best keys.

Purpose: Makes "new games on upgraded workers get blobs immediately" true (D-04), and proves
the schema extension is backward-compatible (the load-bearing SC3 property).

Output: `SubmitEval` extension, `_apply_submit` threading, worker `_eval_positions` upgrade,
and backward-compat tests.
</objective>

| submitted second_uci string → JSONB blob → Phase 143 gate | A worker-supplied UCI string is stored and later parsed by the gate/`chess.Move.from_uci` |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-142-03-01 | Spoofing | `/eval/remote/submit` caller | medium | transfer | Endpoint already gated by `require_operator_token` (eval_remote.py:453); new fields ride the same authenticated path — no new auth surface |
| T-142-03-02 | Tampering | malformed/oversized `second_uci` | low | mitigate | Pydantic validates `str \| None`; stored as JSONB data via parameterized CAST (no SQL injection); UCI is never executed — the Phase-143 gate parses it defensively (try/except already in `_detect_tactic_for_flaw`); blob assembly maps None → `su=""` |
| T-142-03-03 | Tampering | out-of-range `second_cp`/`second_mate` | low | accept | White-perspective cp stored as-is; the gate's sigmoid (`eval_cp_to_expected_score`) saturates gracefully on extreme values — a data-quality, not security, concern at ASVS L1; no bound enforced |
