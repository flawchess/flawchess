<objective>
Deliver the backend vertical slice for per-persona win tracking, end-to-end: a client-supplied `persona_id` on the store-game request round-trips to a new nullable `games.persona_id` column (persona games only, custom games stay NULL, pre-existing games never backfilled), and a new authenticated `GET /bots/persona-wins` aggregation endpoint returns per-persona win counts scoped to the current user.

This is the phase's TRACER: it wires one path through every backend layer (schema → router → service → repository → migration/DB → aggregation read) and is verified by a real end-to-end integration test before any frontend expansion (Plan 03) builds on it.

Purpose: give the Bots page a truthful per-persona win signal without a new architectural layer — reuses the Phase 167 post-insert `created`-gated UPDATE pattern and the `stats_repository` win-condition verbatim.
Output: migration + model column + schema field/response + repo write & aggregation + GET endpoint + backend test coverage.

**Spec-less probe fallback:** visibly skipped — this phase has no requirement IDs to probe; no probe-derived edge/prohibition predicates were generated. Coverage is behavior-based per 185-VALIDATION.md.
</objective>

<assumption_delta_decision>
Noun now primary: `game`. Signal detected: `{kind: "chosen", term: "custom"}` from "custom games stay NULL".
Decision: **add-alongside** (nullable `persona_id` is an optional descriptive attribute, NOT part of the game identity/natural key).
Rationale: the natural key stays `(user_id, platform, platform_game_id)`; `persona_id` is null for custom and all pre-existing games and never participates in uniqueness or dedup — the game identity model is unchanged.
What would force a later promote: if personas became first-class DB entities (a `personas` table + real `ForeignKey`), or if win-tracking ever required per-(user, persona) game uniqueness. Neither is in scope (the registry is a frontend-only TS module — RESEARCH A1).
</assumption_delta_decision>

<reversibility>
The Alembic migration (nullable `String(30)` column, no backfill, no server_default) is `reversible` — `downgrade()` is a bare `op.drop_column`. No `checkpoint:decision` gate is warranted (planning context explicitly: do not fabricate gates for a cheaply-reversible nullable column add).
</reversibility>

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-185-01 | Tampering/DoS | StoreBotGameRequest.persona_id | medium | mitigate | `Field(max_length=30)` at the Pydantic boundary — an overlong value 422s before SQL, never crashing the INSERT/UPDATE with a Postgres DataError (CR-01 precedent) |
| T-185-02 | Elevation/Info Disclosure | GET /bots/persona-wins | high | mitigate | user_id derived ONLY from `current_active_user` JWT; no user_id query/path param — blocks cross-user win-count reads (ASVS V4) |
