# Phase 128.1: Add tactic motifs for lichess-theme coverage - Context

**Gathered:** 2026-06-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Close lichess `puzzleTheme.xml` coverage gaps in `detect_tactic_motif` by adding **four**
deterministically-detectable motifs, each following the locked 5-step recipe (Literal +
IntEnum entry, detector function/branch, `motif_theme_map.py` entry, dispatch placement,
precision floor baselined from a fresh TRAIN harness run — never lower an existing floor):

1. **discovered-check** — split out of `discovered-attack` (logic already exists in
   `detect_discovered_attack` Sub-case 1, `tactic_detector.py:547`). Maps to lichess
   `discoveredCheck`.
2. **trapped-piece** — new detector: escape-square enumeration. Maps to `trappedPiece`.
3. **en-passant** — trivial one-line check (the solution move is an e.p. capture). Maps
   to `enPassant`.
4. **promotion** + **under-promotion** — two motifs (see D-01). Map to `promotion` /
   `underPromotion`.

**Out of scope (unchanged from SEED-058):**
- More named-mate patterns (epaulette/corner/killBox/etc.) — BLOCKED on first fixing the
  suppressed arabian/boden/dovetail detectors (zero precision today; SEED-057).
- Non-motif lichess metadata themes (game phase, endgame TYPE, eval bands, length/
  difficulty, source meta) — out of the motif model by design.
- Fuzzy/positional themes (zugzwang, exposedKing, advancedPawn, quietMove, etc.) —
  eval/judgement-driven, not deterministic geometry; low precision.
- **No UI / chip surfacing.** This phase emits + stores motifs and validates precision in
  the harness. The chip-surfacing decision for Phase 129 is *informed* here (D-09) but the
  controls/chips themselves are Phase 129.

<decisions>
## Implementation Decisions

### Promotion / under-promotion (the seed's explicit product call)
- **D-01 (LOCKED):** **Two separate motifs** — `promotion` and `under-promotion`.
  `under-promotion` is the more genuinely-tactical signal and lichess tracks
  `promotion` / `underPromotion` separately, so keep them distinct. `under-promotion`
  **dominates** `promotion` (a promotion to a non-queen tags as `under-promotion`, never
  as `promotion`) — same "more-specific subtype wins" pattern as the named-mate subtypes.
- **D-02 (LOCKED):** **Move-type family.** `en-passant`, `promotion`, and `under-promotion`
  are grouped as a **"move-type" family** that is **excluded from the default tactic-chip
  set**. They are detected + stored for lichess-theme parity and harness coverage; whether/
  how to surface them is a Phase 129 decision (see D-09). discovered-check and trapped-piece
  are **real tactical geometry** and are NOT in the move-type family.

### Dispatch priority placement (single-tag-per-orientation)
- **D-03 (LOCKED — tier assignment):** `detect_tactic_motif` emits exactly **one** tag per
  PV. Lock only the **tier**:
  - `discovered-check` → **Tier 2** (geometric material-winners), adjacent to `double-check`
    (both are check motifs) and ranked **above** `discovered-attack` (a discovering move
    that gives check tags as `discovered-check`, not `discovered-attack`).
  - `trapped-piece` → **Tier 2**, ranked above the Tier-4 `hanging-piece` catch-all (so a
    trapped piece is never mislabeled as merely hanging).
  - `en-passant` / `promotion` / `under-promotion` (move-type) → a **NEW lowest tier**
    below `hanging-piece`. **Real tactics always win** — a fork-via-promotion or
    en-passant-fork tags as the real tactic; move-type only fires when **nothing else** does.
- **D-04 (LOCKED — posture):** "Real tactic wins, move-type lowest." Explicitly NOT
  inverted — move-type tags do not pre-empt real tactics.
- **D-05 (Claude's discretion):** **Exact intra-tier rank** is the planner's call,
  baselined by the harness, matching the existing provisional **D-08** posture for tiers 2-3
  ("reorder in the registry without touching detector bodies"). Lock the tier, not the rank.
- **Consequence to record (NOT a bug):** because move-type sits at the lowest tier under
  single-tag dispatch, `en-passant` / `promotion` / `under-promotion` will be **shadowed by
  any real tactic** and therefore **rarely populate** — they only tag on the residual
  positions where no real motif fired. The planner must NOT treat sparse move-type
  population as a defect, and the harness precision for these is measured **only on that
  residual** (expected ~100%, consistent with their trivial detection).

### trapped-piece detector scope
- **D-06 (LOCKED):** **Precision-first, strict material gate.** Require the piece to be
  **under attack** AND **every legal escape square to lose material** (net material gain for
  the trapper). Mirrors Phase 127's governing value — *a wrong visible tag erodes trust more
  than a missing one*. Recall-leaning (fire on trappable-but-not-yet-attacked pieces) was
  **rejected**.
- **D-07:** Distinct from `hanging-piece` by **definition** (hanging = capturable now for
  free; trapped = has moves but all lose material) and resolved at the **dispatch** layer via
  D-03 priority (trapped-piece Tier 2 beats hanging-piece Tier 4 when both fire).
- **D-08:** **Precision floor baselined from a fresh TRAIN harness run** (`precision_floors.py`,
  D-09 of the tagger architecture). **Never lower an existing floor** to make the suite pass.
  - **Fixture sparsity caveat (verified 2026-06-19 against `fixtures/tagger/detector_fixture_{train,test}.csv`,
    11,855 / 5,164 rows).** Ground-truth positive counts per new theme are wildly uneven —
    the floor mechanism must NOT be applied uniformly:

    | lichess theme | TRAIN | TEST | floor posture |
    |---|---|---|---|
    | `discoveredCheck` | 1004 | 397 | normal % floor — robust |
    | `promotion` (pure queen) | 176 | 57 | normal % floor — adequate |
    | `trappedPiece` | 28 | 11 | % floor OK for **precision** (denominator = detector firings across all ~11.8k rows, not the 28 labels), but **recall is only weakly validated** by 28 positives — do not over-read recall |
    | `enPassant` | 12 | 7 | thin, but ~100% precision by construction (one-line move-type check) — acceptable small sample |
    | `underPromotion` | **4** | **4** | **Too few for a real percentage floor.** Use a no-regression / "must fire on the known 4" floor, OR hand-augment the fixture with curated under-promotion FENs, OR mark as explicit low-confidence coverage. Do NOT assign a percentage floor that pretends to be statistically real. |

  - `underPromotion` is always a **subset** of `promotion` in the fixtures (4 co-tagged, 0
    under-only in each split) — this confirms the D-01 dominance design maps cleanly to lichess
    (every under-promotion puzzle also carries the `promotion` theme).
  - **Why precision survives sparse labels for `trappedPiece`/`enPassant`:** precision = of the
    rows where *our* detector emits the motif, how many are correct. The denominator is our
    firing count over the full TRAIN set, independent of the labeled-positive count. Only
    **recall** confirmation is gated by the thin label counts.

### Chip-surfacing intent (forward-looking — informs grouping, not built here)
- **D-09 (LOCKED — directional intent, executed in Phase 129):** **Split.**
  - `discovered-check` and `trapped-piece` are **first-class tactic chips** (real geometry,
    trainable) — eligible for the default motif chip set in 129.
  - `en-passant` / `promotion` / `under-promotion` are **detection-only / move-type** —
    surfaced **separately or behind a toggle** in 129, **not** in the default motif chip set.
  - This is a 129 UI decision; 128.1 only needs to make the **family grouping** (D-02)
    available so 129 can apply it. No chip code in this phase.

### Claude's Discretion
- Exact intra-tier rank of the new motifs within their assigned tiers (D-05).
- Whether discovered-check is a new standalone detector function or a refactor that lifts
  Sub-case 1 out of `detect_discovered_attack` and returns it as a separate firing (the
  seed notes the logic already exists at `tactic_detector.py:547`).
- Whether the move-type "lowest tier" is implemented as a new candidate-pool tier or as a
  set of post-candidate fallbacks after `hanging-piece` — either satisfies D-03/D-04.
- IntEnum integer assignment for the new motifs (append 25-29 in declaration order;
  preserve existing 1-24).
- Where the "move-type family" set constant lives (alongside `MATE_MOTIFS` in
  `tactic_detector.py`, mirroring that frozenset pattern).
- trapped-piece detector internals (escape-square enumeration, defended-piece handling),
  subject to the D-06 precision gate and D-08 floor.

## Existing Code Insights

### Reusable Assets
- **discovered-check logic already exists** — `detect_discovered_attack` Sub-case 1
  (`tactic_detector.py:547-558`) already isolates "first move gives check, checker ≠ moved
  piece, checker is pov." Splitting it out is a refactor, not new geometry (seed: "nearly
  free").
- **`MATE_MOTIFS` frozenset** (`:167`) is the exact pattern for a new `MOVE_TYPE_MOTIFS`
  family constant (D-02) — fine-grained storage, free coarsening / exclusion at query time.
- **Tier registries are data, not control flow** — `_GEOMETRIC_REGISTRY` /
  `_TIER3_REGISTRY` / the Tier-4 hanging-piece block let the planner add motifs + reorder
  ranks by editing lists, not detector bodies (D-05).
- **The harness TRAIN/TEST split + `precision_floors.py`** (Phase 127) is the ready-made
  baselining mechanism for the five new floors (D-08).

### Established Patterns
- **4-tuple detector contract** `(fired, piece, confidence, depth)` — every new detector
  conforms; move-type detectors return `TACTIC_CONFIDENCE_HIGH` (trivial, ~100% precision).
- **More-specific subtype dominates** (named-mate subtypes short-circuit before generic
  mate) — reused twice: discovered-check before discovered-attack, under-promotion before
  promotion (D-01/D-03).
- **Single tag per orientation, clean `GROUP BY`** — drives the D-03/D-04 priority decision
  and the "move-type rarely populates" consequence; the dispatcher still returns exactly one
  motif per PV.
- **Precision-first, NULL/absent over a wrong tag** (127 governing value) — D-06 strict gate.

### Integration Points
- **Both PV passes get the new motifs for free** — Phase 128's missed (`flaw_ply` PV) and
  allowed (`flaw_ply+1` PV) passes both call the same `detect_tactic_motif`; no per-pass work.
- **No schema/migration change** — new motifs are new integer values in the **existing**
  `allowed_tactic_motif` / `missed_tactic_motif` SmallInteger columns. No new DB columns.
- **Dispatcher wiring** — add detectors to the tier registries + a new lowest tier for
  move-type; add `TacticMotif` Literal + IntEnum values + `motif_theme_map` + `precision_floors`
  entries (the 5-step recipe, per motif).

## Specific Ideas

- The user took the recommended option on three of four questions and chose **"let planner
  decide intra-tier"** for dispatch placement — i.e. lock the *tier*, leave the *rank* to the
  harness-baselined planner, consistent with the existing provisional-D-08 posture.
- Governing value carried from 127: a wrong **visible** tag erodes trust more than a missing
  one. This drives both the strict trapped-piece gate (D-06) and the move-type family's
  exclusion from default chips (D-02/D-09) — move-type tags are mechanically true but not
  what a player wants foregrounded as "the tactic."
- The "move-type rarely populates under single-tag" outcome was surfaced and **accepted** as
  correct behavior, not a defect — recorded under D-05 so the planner doesn't chase it.
