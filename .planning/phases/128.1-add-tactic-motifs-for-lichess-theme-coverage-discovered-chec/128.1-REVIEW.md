## Warnings

### WR-01: `detect_promotion` docstring misrepresents dispatch loop evaluation

**File:** `app/services/tactic_detector.py:1533-1535`

**Issue:** The docstring states "Dispatch order (under-promotion ranked before promotion in `_MOVE_TYPE_REGISTRY`) enforces this: under-promotion is checked first; **this function is only called if** `moves[0].promotion == chess.QUEEN`." This is factually wrong. The Tier-5 dispatch loop in `detect_tactic_motif` unconditionally calls ALL three move-type detector functions regardless of which fires:

```python
for rank, (motif_str, motif_int) in enumerate(_MOVE_TYPE_REGISTRY):
    fn = _MOVE_TYPE_DETECTOR_FNS[motif_str]
    fired, piece, depth = fn(boards, moves, pov)  # all three always called
    if fired:
        candidates.append(...)
```

`detect_promotion` is always called; it simply does not FIRE when `moves[0].promotion != chess.QUEEN`. D-01 dominance is actually guaranteed by mutual exclusion of the detection conditions (`detect_under_promotion` fires only when `promo is not None and promo != chess.QUEEN`; `detect_promotion` fires only when `promo == chess.QUEEN`) — not by registry rank order. The rank order is a redundant secondary defense that would only matter if both detectors could fire on the same move, which they cannot.

The inaccuracy in the docstring means a future developer who adds a fourth move-type detector, or who refactors the loop to short-circuit on first hit, could incorrectly conclude the rank ordering is load-bearing and misplace a new detector.

**Fix:**
```python
def detect_promotion(
    boards: list[chess.Board], moves: list[chess.Move], pov: chess.Color
) -> tuple[bool, int | None, int | None]:
    """Promotion (queen): the first refuting move promotes a pawn to a QUEEN.

    D-01 dominance: a non-queen promotion fires detect_under_promotion (rank 0 in
    _MOVE_TYPE_REGISTRY) and NOT this function — because detect_under_promotion fires
    only when moves[0].promotion is not None AND != chess.QUEEN, which is mutually
    exclusive with this function's condition (moves[0].promotion == chess.QUEEN).
    The registry rank provides a secondary defense but is not the primary enforcement
    mechanism; the detection conditions are fully disjoint.

    Returns (fired, chess.PAWN, depth=0). Confidence: TACTIC_CONFIDENCE_HIGH.
    """
```

### WR-02: `library_repository.py` comment omits motifs 27–29 from the "not yet in chip families" disclosure

**File:** `app/repositories/library_repository.py:63-65`

**Issue:** The comment updated in this phase reads:

```python
# All 24 motif ints (1-24) appear exactly once across all families.
# Motifs 25 (discovered-check) and 26 (trapped-piece) are not yet assigned to a
# chip family — they will be added to FAMILY_TO_MOTIF_INTS in a future phase.
```

Phase 128.1-02 added motifs 27 (en-passant), 28 (promotion), and 29 (under-promotion) to `TacticMotifInt`, but these three are also absent from `FAMILY_TO_MOTIF_INTS` and the comment does not mention them. A developer who reads this comment to understand which motifs need chip-family assignment will get an incomplete picture and may not realise that 27–29 are also unassigned. The existing test `test_family_mapping_covers_all_24_motifs` asserts `sorted(all_ints) == list(range(1, 25))` which will still pass, but it reinforces the misimpression that only the original 24 motifs are relevant.

**Fix:**
```python
# All 24 motif ints (1-24) appear exactly once across all families.
# Motifs 25 (discovered-check) and 26 (trapped-piece) are Tier-2 real-geometry motifs
# not yet assigned to a chip family — deferred to a future phase (Phase 129 directional).
# Motifs 27 (en-passant), 28 (promotion), and 29 (under-promotion) are the MOVE_TYPE_MOTIFS
# family; chip surfacing is a Phase 129 decision (D-09). None of 25-29 appear here.
```

### IN-02: `test_real_tactic_beats_promotion_motif` has misleading comment block

**File:** `tests/services/test_tactic_detector.py:1666-1693`

**Issue:** The test description opens with "Position: the refuting move is a queen promotion that also forks two pieces." and then the code comment explains that the original position "does NOT have a fork — it returns None currently (no real tactic)." The test then switches to a different position (`6r1/k1P5/7p/...`) where an under-promotion fork fires, but the surrounding prose describes the abandoned first-attempt position. The leading docstring and the final chosen FEN are therefore misaligned: the docstring describes a queen-promotion-fork scenario, but the actual fixture is an under-promotion fork. The test behavior is correct (a real tactic beats the move-type tag), but the narrative is confusing.

**Fix:** Replace the misleading comment block with a description that matches the actual fixture:
```python
def test_real_tactic_beats_promotion_motif(self) -> None:
    """D-04: a fork via under-promotion returns the real tactic (fork), not a move-type tag.

    The refuting PV includes an under-promotion (c7c8n) that also forks pieces.
    Fork fires at Tier 2 and beats under-promotion at Tier 5 — move-type is the lowest
    tier and always loses to any real tactic (D-04).
    """
    fen = "6r1/k1P5/7p/B2pp2Q/p3P3/P2P3P/2P2qr1/1R5K w - - 1 30"
    pv = "a5b6 f2b6 c7c8n g8c8 b1b6"
    motif_int, _piece, _conf, _depth = detect_tactic_motif(chess.Board(fen), pv)
    assert motif_int is not None
    assert _INT_TO_MOTIF[motif_int] not in MOVE_TYPE_MOTIFS, (
        f"expected a real tactic, got move-type '{_INT_TO_MOTIF.get(motif_int)}'"
    )
```
