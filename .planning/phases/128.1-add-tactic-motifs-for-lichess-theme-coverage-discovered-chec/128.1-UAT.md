## Verification approach (D-resolution 2026-06-20)

The three items below were originally framed as "import a real game, hit a rare
position, confirm a `game_flaws` row stores motif_int 26/27." That is impractical
(en-passant ~never fires on real tactical positions by design; trapped-piece is
only-FP) and depends on data luck rather than code correctness. Per Adrian's call,
verification was done deterministically instead by **updating
`scripts/tactic_tagger_report.py` (`/tactic-tagger-report`) to render the 5 new
motifs**, scoring them against the committed CC0 lichess puzzle fixtures (17k rows).
The detector runs through the FULL dispatcher (`detect_tactic_motif`) for every
fixture, so a report row IS end-to-end proof that the motif fires (or correctly does
not) and would be stored — the persistence seam is motif-agnostic and proven separately
(`flaws_service.py:496/500` → `game_flaws_repository.py:120/125`, no write-time
suppression per D-11; existing service/repo tests assert a specific int propagates).

Report artifact: `reports/tactic-tagger/tactic-tagger-2026-06-20.md`.
Numbers match the D-08 baseline in `128.1-02-SUMMARY.md` exactly.
