Fix four issues in position_bookmark_repository.py:

**1. Replace get_existing_full_hashes with get_existing_target_hashes:**
Rename function to `get_existing_target_hashes`. Instead of recomputing full_hash from FEN via compute_hashes(), simply read `target_hash` directly from each bookmark row. Return `set[int]` of all target_hash values. This is correct because target_hash already stores the appropriate hash (white_hash for mine+white, black_hash for mine+black, full_hash for both).

**2. Fix get_top_positions_for_color to deduplicate by target hash and exclude by target hash:**
The function must produce results where each returned position has a UNIQUE target hash for the given color. The target hash depends on the color: for white, it is white_hash; for black, it is black_hash (since match_side defaults to "mine" and the user can change it later).

Change the approach:
- Group by the user's color hash only (white_hash for white, black_hash for black) instead of (white_hash, black_hash, full_hash). This aggregates all opponent variations under a single "my pieces" hash.
- Count distinct games per color hash group.
- Add a minimum game count filter of 2 (user requested: "include positions which occur at 2 plays minimum").
- The exclude set now contains target_hashes, so exclude rows where the color hash is in the exclude set.
- For each result, we need a representative full_hash and the complementary hash. After grouping by the color hash, select one representative row to get the other hashes: use a subquery or post-query to fetch one (white_hash, black_hash, full_hash) tuple for each color hash result.
- Return list of (white_hash, black_hash, full_hash, game_count) tuples as before, but now deduplicated by the color-specific hash.

Implementation approach for grouping:
```python
color_hash_col = GamePosition.white_hash if color == "white" else GamePosition.black_hash

stmt = (
    select(
        color_hash_col.label("color_hash"),
        func.count(GamePosition.game_id.distinct()).label("game_count"),
    )
    .join(Game, Game.id == GamePosition.game_id)
    .where(
        GamePosition.user_id == user_id,
        GamePosition.ply.between(ply_min, ply_max),
        Game.user_color == color,
    )
    .group_by(color_hash_col)
    .having(func.count(GamePosition.game_id.distinct()) >= 2)
    .order_by(func.count(GamePosition.game_id.distinct()).desc())
    .limit(over_fetch_limit)
)
```

Then post-filter to exclude hashes in exclude_target_hashes. For each remaining color_hash, fetch a representative row to get the full (white_hash, black_hash, full_hash) triple:
```python
for row in filtered_rows:
    rep_stmt = (
        select(GamePosition.white_hash, GamePosition.black_hash, GamePosition.full_hash)

    rep = await session.execute(rep_stmt)
```

Update the function signature: rename parameter `exclude_full_hashes` to `exclude_target_hashes`.

**3. Fix suggest_match_side heuristic:**
Replace the flawed ratio with a two-granularity comparison as suggested in the debug analysis. Query at two levels for the user's color hash within the suggestion ply range (6-14):
- `my_hash_count`: COUNT(DISTINCT game_id) WHERE color_hash = X AND ply BETWEEN 6 AND 14
- `full_hash_count`: COUNT(DISTINCT game_id) WHERE full_hash = (the representative full_hash) AND ply BETWEEN 6 AND 14

If my_hash_count > 2 * full_hash_count, suggest "mine" (opponent varies a lot, so matching just your pieces captures more games). Otherwise suggest "both" (position is consistent, full match is fine).

Update the function signature to also accept `full_hash: int` and add `ply_min: int, ply_max: int` parameters. The caller in the router already has full_hash available.

**4. Fix create_bookmark sort_order:**
Before creating the bookmark, query `SELECT COALESCE(MAX(sort_order), -1) + 1 FROM position_bookmarks WHERE user_id = :uid`. Use the result as the new bookmark's sort_order.

```python
max_stmt = select(func.coalesce(func.max(PositionBookmark.sort_order), -1) + 1).where(
```
