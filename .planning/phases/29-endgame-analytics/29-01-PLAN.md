Game phase derivation (from Phase 26 summary — must be computed at query time):
- Phase weights: Q=9, R=5, B=3, N=3 (per side). Starting total = 62.
- material_count includes pawns (starting = 7800 cp).
- material_count EXCLUDES king value.
- The PHASE_WEIGHT system (non-pawn/non-king): opening >= 50, middlegame >= 25, endgame < 25.
- In centipawn terms (1 weight unit = 100 cp): endgame < 2500 cp of non-pawn material.
- But material_count INCLUDES pawns, so we cannot use material_count alone for phase.
- ALTERNATIVE: Use material_count directly with calibrated thresholds:
  - Starting material_count = 7800 (all pieces + all pawns).
  - Endgame heuristic: material_count < 2600 (captures significant material from both sides).
  - This threshold is approximate but good enough for statistics grouping.
  - Actually, a better approach: derive from material_signature at query time. Parse the signature
    to determine piece composition and classify.

IMPORTANT: For endgame category classification, use material_signature pattern matching in SQL:
- Endgame detection: material_count is low enough. Use threshold ENDGAME_MATERIAL_THRESHOLD = 2600.
  Rationale: At 2600 cp, roughly Q+R+minor have been traded. This is conservative — many endgames
  start with more material but the key is that significant pieces are off the board.
- Category from material_signature (format: "KQR...P_KBN...P"):
  - pawn: signature matches K[P]*_K[P]* (only kings and pawns)
  - pawnless: no 'P' in signature at all
  - rook: signature has R but no Q/B/N (beyond K and P)
  - minor_piece: signature has B or N but no R/Q (beyond K and P)
  - queen: signature has Q but no R/B/N (beyond K and P)
  - mixed: everything else (has pieces from multiple categories)

Endgame class ordering for display: ["Rook", "Minor Piece", "Pawn", "Queen", "Mixed", "Pawnless"]

For conversion/recovery (CONV-01, CONV-02, CONV-03):
- "Materially up" = material_imbalance > 0 when user is white, or < 0 when user is black
- In user terms: user_material > opponent_material. Since imbalance = white - black:
  - If user_color = "white": up when imbalance > 0
  - If user_color = "black": up when imbalance < 0
- Game phase for grouping: opening/middlegame/endgame derived from material_count threshold
- Opening: material_count >= 6800 (nearly all pieces on board)
- Middlegame: 2600 <= material_count < 6800
- Endgame: material_count < 2600
- Report: for each game phase, what % of games where user was ever materially up did user win?
  And what % of games where user was ever materially down did user draw or win?
- Query approach: find DISTINCT game_ids where user was at some point materially up/down in each phase,
  then get the game result and aggregate W/D/L.

**2. Create `app/repositories/endgame_repository.py`** with two async query functions:

a) `query_endgame_wdl_by_category(session, user_id, time_control_bucket, color)`:
   - Join `game_positions` with `games` on game_id
   - Filter: `game_positions.user_id == user_id`
   - Filter: `game_positions.material_count IS NOT NULL AND game_positions.material_count < 2600` (endgame threshold)
   - Filter: `game_positions.material_signature IS NOT NULL`
   - Optional filter: `games.time_control_bucket == time_control_bucket` if provided
   - Optional filter: `games.user_color == color` if provided (ENDGM-03)
   - Derive endgame_class from material_signature using SQL CASE:
     ```sql
     CASE
       WHEN material_signature NOT LIKE '%Q%' AND material_signature NOT LIKE '%R%'
            AND material_signature NOT LIKE '%B%' AND material_signature NOT LIKE '%N%'
            AND material_signature LIKE '%P%' THEN 'pawn'
       WHEN material_signature NOT LIKE '%P%' THEN 'pawnless'
       WHEN material_signature NOT LIKE '%Q%' AND material_signature NOT LIKE '%B%'
            AND material_signature NOT LIKE '%N%' THEN 'rook'
       WHEN material_signature NOT LIKE '%Q%' AND material_signature NOT LIKE '%R%' THEN 'minor_piece'
       WHEN material_signature NOT LIKE '%R%' AND material_signature NOT LIKE '%B%'
            AND material_signature NOT LIKE '%N%' THEN 'queen'
       ELSE 'mixed'
     END
     ```
   - Use `COUNT(DISTINCT game_positions.game_id)` as per critical constraint in STATE.md
   - Group by derived endgame_class
   - Return: list of tuples (endgame_class, game.result, game.user_color, distinct_game_id) OR
     use subquery approach: first get DISTINCT game_ids per category, then aggregate results.
   - BETTER APPROACH: Use a subquery/CTE to get DISTINCT (game_id, endgame_class) pairs first,
     then join back to games table for result/user_color, then aggregate.
   - Actually, simplest correct approach: SELECT DISTINCT game_id, endgame_class, result, user_color
     from the joined tables (deduplicated because a game can have multiple endgame positions with
     the same or different categories). Group by endgame_class, result, user_color with COUNT(*).
   - IMPORTANT: A game may have positions in multiple endgame categories (e.g., starts as rook endgame,
     trades down to pawn endgame). Use the FIRST endgame position (lowest ply) to classify the game.
     SQL: Use DISTINCT ON (game_id) ORDER BY ply ASC to get the first endgame classification per game.

   Refined SQL approach using SQLAlchemy:
   ```python
   # Subquery: first endgame position per game (lowest ply)
   endgame_class_expr = case(...)  # the CASE expression above

   first_endgame = (
       select(
           GamePosition.game_id,
           endgame_class_expr.label("endgame_class"),
       )
       .where(
           GamePosition.user_id == user_id,
           GamePosition.material_count.is_not(None),
           GamePosition.material_count < ENDGAME_MATERIAL_THRESHOLD,
           GamePosition.material_signature.is_not(None),
       )
       .distinct(GamePosition.game_id)
       .order_by(GamePosition.game_id, GamePosition.ply)
       .subquery()
   )

   # Main query: join with games to get result/user_color, aggregate
   stmt = (
       select(
           first_endgame.c.endgame_class,
           Game.result,
           Game.user_color,
           func.count().label("cnt"),
       )
       .join(Game, Game.id == first_endgame.c.game_id)
       .group_by(first_endgame.c.endgame_class, Game.result, Game.user_color)
   )
   ```
   Apply optional time_control_bucket and user_color filters on the outer query.

b) `query_conversion_stats(session, user_id, time_control_bucket)`:
   - Find games where user was at some point materially up or down, per game phase.
   - Game phase from material_count: opening >= 6800, middlegame 2600-6799, endgame < 2600.
   - "Materially up" for user: if user_color=white then imbalance > 0, if black then imbalance < 0.
     Use a minimum threshold: abs(material_imbalance) >= 100 (at least a pawn advantage) to filter noise.
   - Query: for each game phase, find DISTINCT game_ids where user was up/down.

   Approach: Two separate subqueries (up and down), each returning DISTINCT game_ids per phase.
   Then join to games table for result and aggregate.

   ```python
   MATERIAL_UP_THRESHOLD = 100  # at least 1 pawn advantage
   ENDGAME_MATERIAL_THRESHOLD = 2600
   OPENING_MATERIAL_THRESHOLD = 6800

   game_phase_expr = case(
       (GamePosition.material_count >= OPENING_MATERIAL_THRESHOLD, "opening"),
       (GamePosition.material_count >= ENDGAME_MATERIAL_THRESHOLD, "middlegame"),
       else_="endgame",
   )

   # User is materially up when: (user_color=white AND imbalance > threshold) OR (user_color=black AND imbalance < -threshold)
   # But user_color is on the Game model, not GamePosition. So we need to join.
   # Alternative: compute user-relative imbalance in the query.

   # Subquery for "up" games per phase:
   up_subq = (
       select(
           GamePosition.game_id,
           game_phase_expr.label("phase"),
       )
       .join(Game, Game.id == GamePosition.game_id)
       .where(
           GamePosition.user_id == user_id,
           GamePosition.material_count.is_not(None),
           GamePosition.material_imbalance.is_not(None),
           # User is up: (white AND imbalance >= 100) OR (black AND imbalance <= -100)
           or_(
               and_(Game.user_color == "white", GamePosition.material_imbalance >= MATERIAL_UP_THRESHOLD),
               and_(Game.user_color == "black", GamePosition.material_imbalance <= -MATERIAL_UP_THRESHOLD),
           ),
       )
       .distinct(GamePosition.game_id)
       .subquery()
   )
   ```

   Then aggregate by joining up_subq with games and counting results per phase.

   Do the same for "down" (invert the imbalance condition).

   Apply optional time_control_bucket filter.
