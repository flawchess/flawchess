<objective>
Ship the historical backfill and the correctness-validation script.

This plan creates `scripts/backfill_accuracy_acpl.py` — scaffolding cloned from
`scripts/backfill_full_evals.py` but the read path follows the **server-side-cursor streaming
pattern** in `scripts/backfill_best_move_pv.py` (`_stream_game_batches` + `itertools.groupby`),
NOT full_evals' load-all-ids-then-per-game structure (which would be an N+1 position query per
game at ~718k-game prod scale). One streamed `game_positions ⋈ games` SELECT ordered by
`(game_id, ply)` yields per-game ply-ordered rows; each group calls the SAME Plan 02
`compute_game_accuracy_acpl` the live hook uses, then writes the four canonical columns —
streamed, batch-committed, `--db`-targeted, resumable.
It also creates `scripts/validate_accuracy_acpl.py`, a read-mostly comparison of our computed
values against the preserved `*_imported` values (primary signal: computed ACPL vs lichess's
own `*_acpl_imported`; secondary: accuracy vs lichess `*_accuracy_imported`; chess.com accuracy
divergence reported separately, not a failure). A small dev-DB test proves the backfill fills
columns and honors the hole gate. Running the ~718k-game prod backfill is a separate operator
step and is NOT gated on this phase (D-06).

Purpose: Fill the existing corpus (on dev now; prod later) and provide the empirical
correctness check that our computed numbers track lichess's own.

Output: `scripts/backfill_accuracy_acpl.py`, `scripts/validate_accuracy_acpl.py`, and a backfill test.
</objective>
