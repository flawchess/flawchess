# Phase 57 — UI Design Contract

> Visual and interaction contract for the Endgame ELO Timeline Chart. Generated by gsd-ui-researcher from Phase 57 CONTEXT.md decisions D-01..D-10 plus existing FlawChess frontend patterns. Verified by gsd-ui-checker.

This phase adds one new chart section to the Endgames tab, placed immediately below Phase 56's Endgame ELO breakdown table. The contract is intentionally narrow because the project already has a locked design system (radix-nova shadcn preset, dark mode, `charcoal-texture` card pattern, Recharts line-chart infrastructure). Only genuinely new decisions (combo palette, axis helper, legend mobile layout, info popover copy) are specified in full.

### ELO_COMBO_COLORS — new theme constant for this phase

Add to `frontend/src/lib/theme.ts`. An 8-entry record keyed by `{platform}_{time_control}` combo — 2 platforms × 4 time controls. Hues chosen to sit around L=0.60 for the **bright (Endgame ELO) stroke** on the project's dark surface (`oklch(0.145 0 0)`), which clears WCAG AA 3:1 non-text contrast for data-viz lines. Chroma 0.18–0.22 keeps lines visually distinct without visual noise. Hues are separated by ≥40° so adjacent combos read as different colors.

| Combo key | Bright (Endgame ELO) | Dark (Actual ELO) | Hue rationale |
|-----------|----------------------|-------------------|---------------|
| `chess_com_bullet` | `oklch(0.62 0.22 30)` | `oklch(0.42 0.18 30)` | Red — matches `RatingChart.tsx` bullet |
| `chess_com_blitz` | `oklch(0.65 0.20 260)` | `oklch(0.45 0.16 260)` | Blue — matches `RatingChart.tsx` blitz |
| `chess_com_rapid` | `oklch(0.70 0.18 80)` | `oklch(0.50 0.14 80)` | Amber — matches `RatingChart.tsx` rapid |
| `chess_com_classical` | `oklch(0.60 0.22 310)` | `oklch(0.40 0.18 310)` | Magenta — matches `RatingChart.tsx` classical |
| `lichess_bullet` | `oklch(0.62 0.20 0)` | `oklch(0.42 0.16 0)` | Pink-red (shift from chess.com red) |
| `lichess_blitz` | `oklch(0.62 0.18 220)` | `oklch(0.42 0.14 220)` | Cyan-blue (shift from chess.com blue) |
| `lichess_rapid` | `oklch(0.65 0.18 140)` | `oklch(0.45 0.14 140)` | Green (distinct from chess.com amber) |
| `lichess_classical` | `oklch(0.60 0.18 340)` | `oklch(0.40 0.14 340)` | Violet-pink (shift from chess.com magenta) |

**Bright vs dark encoding — decision:** two explicit theme constants per combo, NOT an opacity modifier.

- Rationale: opacity would composite against the dark surface and drift the hue toward black, weakening the "same color family, two lines" read. Separate lighter/darker L values at matching hue+chroma preserve perceived hue while creating clear brightness separation. Pattern also matches existing theme.ts which uses semantic constant names (`GAUGE_DANGER`, `MY_SCORE_COLOR`) rather than opacity tokens.
- Exported shape (executor plans the exact names):
  ```ts
  export const ELO_COMBO_COLORS: Record<EloComboKey, { bright: string; dark: string }> = { ... };
  ```
- Lines render with `strokeWidth={2}` (bright Endgame ELO) and `strokeWidth={1.5}` + `strokeDasharray="4 2"` (dark Actual ELO) so color-blind users also get a shape cue distinguishing the two lines per combo. This is a small addition beyond the L-value contrast and costs nothing.

All copy follows CLAUDE.md em-dash guidance: at most one em-dash per paragraph, prefer commas/periods. Second-person voice ("you", "your") consistent with every other endgame section.

| Element | Copy |
|---------|------|
| Section h2 header | `Endgame ELO` (shared container; Phase 56 breakdown table and Phase 57 timeline sit inside it with their own h3 subsection headers) |
| Chart heading (h3) | `Endgame ELO Timeline` |
| Chart sub-description | `Endgame ELO versus Actual ELO over time, per platform and time control. Bright lines are Endgame ELO, dark lines are Actual ELO.` |
| Primary CTA | None — this section has no primary CTA; it's a read-only visualization (the global FilterPanel sidebar is the only interactive control that affects it) |
| Empty state (chart-level, all combos drop or `overall` empty) | Heading: `Not enough endgame games yet for a timeline.` &nbsp; Body: `Import more games or loosen the recency filter.` (mirrors CONTEXT.md D-10 verbatim) |
| Empty state layout | Single `<div className="text-center text-muted-foreground py-8">` with the two lines; matches `EndgameTimelineChart`'s existing empty-state shape |
| Per-combo silent drop (combo has 0 qualifying weekly points) | No toast, no inline warning — combo simply absent from legend and chart. Rationale: a visible "chess.com classical hidden (only 4 games in window)" message on every user load would be noise; the breakdown table in Phase 56 already surfaces per-combo game counts for users who want to audit. |
| Error state (React Query `isError`) | Heading: `Failed to load Endgame ELO timeline` &nbsp; Body: `Something went wrong. Please try again in a moment.` (CLAUDE.md frontend rule: every isError branch must surface failure, never fall through to empty-state copy) |
| Destructive confirmation | None — no destructive actions in this phase |
| Info popover body (≤ 4 short paragraphs, JSX `<p>` inside `<div className="space-y-2">`) | See below |

### InfoPopover copy (final prose for the executor)

Render inside `<InfoPopover ariaLabel="Endgame ELO Timeline info" testId="endgame-elo-timeline-info" side="top">` using the standard `<div className="space-y-2">` wrapper. Four paragraphs, one em-dash total:

```jsx
<div className="space-y-2">
  <p>
    <strong>Endgame ELO</strong> is a performance rating derived from your
    Endgame Skill (the average of Conversion Win %, Parity Score %, and
    Recovery Save %). We compute it as
    <em> avg_opponent_rating + 400 &middot; log10(skill / (1 &minus; skill))</em>,
    using the rolling window's opponent pool for the skill and for the
    average opponent rating.
  </p>
  <p>
    The bright line is Endgame ELO, the dark line is your <strong>Actual ELO</strong>
    (average rating over the same rolling window of all games for that combo).
    The gap between the lines is the interesting signal: well above Actual ELO
    means your endgames are pulling your rating up; well below means they're
    pulling it down.
  </p>
  <p>
    Points are emitted weekly and each point looks back at your trailing 100
    endgame games for that platform and time control. Weeks with fewer than
    10 qualifying endgame games are hidden. Skill is clamped to the 5&ndash;95 %
    range so a handful of lucky or unlucky endgames can't produce an
    absurd performance rating at the extremes.
  </p>
  <p>
    Chess.com uses Glicko-1 and lichess uses Glicko-2, so ratings across the
    two platforms aren't directly comparable. Each combo line is self-consistent
    on its own scale.
  </p>
</div>
```

Notes for the executor:
- Use `&middot;` and `&minus;` and `&ndash;` HTML entities to avoid mixing multiplication/minus glyphs with prose hyphens.
- Keep `<strong>` / `<em>` exactly as shown; matches the info-popover prose style elsewhere on the page (`EndgameScoreGapSection.tsx` lines 201–248).
- One em-dash budget, not used — paragraphs are short enough that commas suffice.

## Chart Specification

### Axes

- **X axis:** `dataKey="date"`, `tickFormatter={createDateTickFormatter(allDates)}` — reuse the project helper (`frontend/src/lib/utils.ts` line 29), no new tick logic.
- **Y axis (Elo/rating scale):** introduce a new helper `niceEloAxis(values: number[]): { domain: [number, number]; ticks: number[] }` in `frontend/src/lib/utils.ts`. Contract:
  - Input: flat array of rating numbers (mix of Endgame ELO and Actual ELO values across all visible combos, after legend filtering).
  - Step candidates: `[50, 100, 200, 500]`. Pick the largest step where `range / step >= 4` (aim for 4–8 ticks). This is the same algorithm `RatingChart.tsx` uses inline (lines 84–100); promote it to `utils.ts` so the two charts share it.
  - Fallback when all values equal: expand ±50 around the point so ticks render.
  - Fallback when values array empty: return `{ domain: ['auto', 'auto'], ticks: undefined }` (Recharts auto-scale).
  - Recompute on every legend toggle via `useMemo` keyed on `chartData` + `hiddenKeys`, matching `RatingChart.tsx` pattern.
- `niceWinRateAxis` is **not** used here — it's a 0–1 percent helper.
- Do NOT apply `allowDataOverflow={false}` (keep Recharts default) so rating spikes don't clip.

### Lines

For each `combo_key` that survives filter + `MIN_GAMES_FOR_TIMELINE` culling:
- Bright Endgame ELO line: `<Line dataKey={`${combo}_endgame_elo`} stroke={ELO_COMBO_COLORS[combo].bright} strokeWidth={2} dot={false} connectNulls={true} hide={hiddenKeys.has(combo)} />`
- Dark Actual ELO line: `<Line dataKey={`${combo}_actual_elo`} stroke={ELO_COMBO_COLORS[combo].dark} strokeWidth={1.5} strokeDasharray="4 2" dot={false} connectNulls={true} hide={hiddenKeys.has(combo)} />`
- Legend entry exists per COMBO (not per line). Clicking the legend toggles both lines of that combo as a unit. Implementation: one `hiddenKeys: Set<comboKey>` shared by both line components; `handleLegendClick` pattern lifted from `EndgameTimelineChart.tsx` lines 33–45.

### Tooltip

Multi-line tooltip matching `EndgameTimelineChart`'s structure:
- Date header (`formatDateWithYear(label)`)
- One block per visible combo showing: combo display label, Endgame ELO value (bright swatch), Actual ELO value (dark swatch), gap (Endgame − Actual) with sign, and `(past {games_in_window} games)` in muted-foreground. Example row:
  ```
  chess.com Blitz
    [●] Endgame ELO: 1847
    [◐] Actual ELO:  1792   gap +55 (past 100 games)
  ```
- Wrap in the existing tooltip class string: `rounded-lg border border-border/50 bg-background px-3 py-2 text-xs shadow-xl space-y-1` (copied from `EndgameTimelineChart.tsx` line 133).

### Legend — mobile parity decision

Up to 8 combos × 2 lines = 16 entries is too dense for a 360px mobile viewport. Decision: **compact combo-level legend with responsive layout**, no disclosure button.

- Desktop (≥ `md`, 768px): `ChartLegendContent` renders inline below the chart in a single wrap row — existing shadcn chart legend behavior. One entry per combo, formatted `"chess.com Blitz"` with a split swatch (half bright, half dark) showing both tones at a glance.
- Mobile (< `md`): same `ChartLegendContent` component, but the wrapper applies `className="flex flex-wrap gap-x-3 gap-y-1 text-xs"` so entries flow onto multiple rows gracefully. No horizontal scroll, no disclosure toggle — both of those hide information that users with narrow filter selections actually need.
- When the sidebar platform+time-control filter narrows combos, the legend shrinks automatically. This is the primary lever for mobile users who find 8 combos overwhelming — the filter UI already exists, promote it via the info popover wording if needed in a later phase (not this phase).
- Split-swatch rendering: custom `content` prop on `ChartLegend` that renders `<span style={{ background: `linear-gradient(to right, ${bright} 50%, ${dark} 50%)` }} />` instead of the default single-color dot. Planner details the exact component signature.

**Mobile parity rule honored:** same React component renders both viewports; only Tailwind breakpoint utilities shift the legend wrap behavior. No second copy of the component for mobile.
