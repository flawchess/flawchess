| Token | Value | Usage |

## Component Contracts

### EvalBar (`components/analysis/EvalBar.tsx`)

**Visual layout:**
- Outer container: `relative flex flex-col border border-border rounded overflow-hidden w-4`
  - `w-4` (16px) fixed width. Height determined by Phase 138 page shell via className override.
- Top fill div (white): `absolute inset-x-0 top-0 transition-[height] duration-150` вҖ” height = `whiteFraction * 100%`
  - Background: `EVAL_BAR_WHITE` inline style
- Bottom fill div (black): `absolute inset-x-0 bottom-0` вҖ” height = `(1 - whiteFraction) * 100%`
  - Background: `EVAL_BAR_BLACK` inline style

**Sigmoid formula (implemented in component, not in the hook):**
```typescript
const SIGMOID_SCALE = 400; // centipawns at which the bar shows ~73% dominance
function cpToFraction(cp: number): number {
  return 1 / (1 + Math.exp(-cp / SIGMOID_SCALE));
}
```

**White fraction logic:**
```
evalMate > 0   вҶ’ whiteFraction = 1.0  (white mates, bar fully white)
evalMate < 0   вҶ’ whiteFraction = 0.0  (black mates, bar fully black)
evalMate === 0 вҶ’ whiteFraction = 0.5  (mate-in-0 is terminal; treat as midpoint display)
evalCp !== null вҶ’ whiteFraction = cpToFraction(evalCp)
evalCp === null && evalMate === null вҶ’ whiteFraction = 0.5  (no data yet)
```

**Mate label (D-04 locked: depth >= 8 gate):**
- Render a centered `<span>` overlaid on the bar ONLY when `evalMate !== null && depth >= 8`
- Position: `absolute left-1/2 -translate-x-1/2` placed in the dominant region:
  - `evalMate > 0` вҶ’ `top-2` (near top, in white region)
  - `evalMate < 0` вҶ’ `bottom-2` (near bottom, in black region)
- Content: `M{Math.abs(evalMate)}`
- Style: `text-sm font-semibold` + contrasting color вҖ” use `text-foreground` reversed:
  - In white region: `text-black` (dark text on light fill)
  - In black region: `text-white` (light text on dark fill)
- When `depth < 8` and `evalMate !== null`: DO NOT show the mate label; show centipawn fill as if evalMate were null (use `whiteFraction = 0.5` if evalCp is also null)

**Props:**
```typescript
interface EvalBarProps {
  evalCp: number | null;
  evalMate: number | null;
  depth: number;
  className?: string; // caller overrides width/height (Phase 138)
}
```

**data-testid:** `data-testid="analysis-eval-bar"` on the outer container.

**ARIA:** `role="img"` `aria-label="Engine evaluation: {score}"` where score is derived from evalCp/evalMate. Update `aria-label` on each render.

**White-POV rule (D-04):** EvalBar does NOT know about board orientation. White is always at top. The `orientation` prop from the page is intentionally excluded.

### EngineLines (`components/analysis/EngineLines.tsx`)

**Visual layout:**
```
[outer container вҖ” data-testid="analysis-engine-lines"]
  [when isAnalyzing && pvLines.length === 0]
    <Loader2 animate-spin /> "AnalyzingвҖҰ"
  [for each pvLine (index 0..1)]
    [line header] "+1.42"  "d{depth}"
    [move chips row] "e4"  "e5"  "Nf3"  вҖҰ
    [border-t border-border separator between lines]
```

**Score format rules:**
| Condition | Display |
|-----------|---------|
| `evalMate > 0` | `#+{evalMate}` (e.g. `#+3`) |
| `evalMate < 0` | `#-{Math.abs(evalMate)}` (e.g. `#-2`) |
| `evalCp >= 0` | `+{(evalCp / 100).toFixed(2)}` (e.g. `+1.42`) |
| `evalCp < 0` | `{(evalCp / 100).toFixed(2)}` (e.g. `-0.33`) |
| Both null | `вҖҰ` (ellipsis placeholder) |

Score is per-line (each PvLine carries its own score). Use `pvLine.score` from the engine hook's `PvLine` type.

**Depth badge:** Show `d{depth}` once, in the header of line 0 only (both lines share the same search depth). Class: `text-sm text-muted-foreground`.

**"AnalyzingвҖҰ" state:** When `isAnalyzing && pvLines.length === 0`:
```html
<div class="flex items-center gap-2 text-sm text-muted-foreground p-2">
  <Loader2 class="h-4 w-4 animate-spin" />
  AnalyzingвҖҰ
</div>
```
When `isAnalyzing && pvLines.length > 0`: show pvLines normally (they update progressively). No spinner overlay вҖ” the depth badge incrementing is the progress signal.

**PV move chip interaction (D-03 locked):**
- Each PV move is a `<button>` (not a div/span). Clicking calls `makeMove(from, to)` вҖ” Phase 138 wires this; Phase 137 receives `onMoveClick: (from: string, to: string) => void` as a prop.
- The `from` and `to` squares are derived from UCI move notation (first 4 chars of the move string, e.g. `"e2e4"` вҶ’ `from="e2"`, `to="e4"`). Promotion piece is the 5th char if present.
- Chip style: matches `HorizontalMoveList` chip exactly вҖ” `inline-flex items-center rounded px-1 py-0.5 font-mono text-sm transition-colors hover:bg-accent`.
- No "active" highlight on PV chips (they are suggestion sequences, not the current board position).
- Truncation: show the first 5 plies of each PV line. If a line has fewer than 5 moves, show all. No "вҖҰ" truncation token needed for v1.

**Move number labels:** Use `moveLabel(startPly, index)` from `lib/moveNumberLabel.ts` where `startPly` is derived from the current board ply when the engine was last invoked. Phase 138 provides `startPly` as a prop. Phase 137 can accept it as an optional prop with default 0.

**Props:**
```typescript
interface EngineLinesProps {
  pvLines: PvLine[];       // from useStockfishEngine
  depth: number;           // from useStockfishEngine
  isAnalyzing: boolean;    // from useStockfishEngine
  evalCp: number | null;   // from useStockfishEngine (for score display)
  evalMate: number | null; // from useStockfishEngine (for mate score display)
  startPly?: number;       // for move-number labels; default 0
  onMoveClick: (from: string, to: string) => void; // calls makeMove
}
```

**data-testid:**
- Container: `data-testid="analysis-engine-lines"`
- Each PV move button: `data-testid={`engine-line-${lineIndex}-move-${moveIndex}`}`
- "Analyzing" spinner container: `data-testid="engine-lines-analyzing"`

**ARIA:**
- Container: `aria-label="Engine lines"` `aria-live="polite"` (screen reader gets updates as engine refreshes)
- Each line: `aria-label={`Line ${lineIndex + 1}: ${scoreText}, moves: ${moves.join(' ')}`}`
- Each move button: `aria-label={`Play ${san}`}` (e.g. `aria-label="Play e4"`)

### VariationTree (`components/analysis/VariationTree.tsx`)

**Responsive split (D-02 locked):**
- Mobile (`< sm` breakpoint): extend `HorizontalMoveList` вҖ” flat wrapping chip list with variation inline in parentheses.
- Desktop (`sm` and above): new vertical paired move list (`N. white black`), variation as indented sub-section.

Implementation pattern: single component with two render paths using Tailwind `hidden`/`sm:block`:
```tsx
<>
  <div className="sm:hidden"><MobileTree вҖҰ /></div>
  <div className="hidden sm:block"><DesktopTree вҖҰ /></div>
</>
```
Or use a `useMediaQuery` hook if conditional rendering is cleaner than dual DOM.

**Mobile rendering (MobileTree):**

Extends `HorizontalMoveList`. Build a flat `HorizontalMoveItem[]` array from `nodes + mainLine + activeVariation`:

1. Map `mainLine` node IDs to chips: use `moveLabel(rootPly, index)` for number labels.
2. At the fork point (where the active variation branches off the main line), append:
   - A non-button `<span className="text-muted-foreground select-none">(</span>` as a `trailing` override or inline adjacent to the fork node.
   - Variation node chips (from the active variation chain), each with `ply = nodeId` passed to `onMoveClick`.
   - A closing `<span className="text-muted-foreground select-none">)</span>`.
3. Main line moves AFTER the fork point remain in the list (dimmed with `dimmed: true`) so the full main line is always visible.

Active node: the chip whose `key === currentNodeId` gets `isCurrent: true`.

`HorizontalMoveList` `heightClass` override: `"h-20 sm:h-20"` (taller than the default `h-12` to accommodate the potential two-line wrap with variation). VariationTree sets this as default; callers can override via a `heightClass` prop pass-through.

**Desktop rendering (DesktopTree):**

New vertical paired move list. Each full-move occupies one row:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ 1.  e4            e5                          в”Ӯ
в”Ӯ 2.  Nf3           Nc6                         в”Ӯ
в”Ӯ 3.  Bb5           a6   вҶҗ active (bg-primary)  в”Ӯ
в”Ӯ     (variation)                               в”Ӯ
в”Ӯ         3...      d5   вҶҗ fork chip            в”Ӯ
в”Ӯ         4.  Ba4                               в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

Layout per row:
- Container: `flex items-center min-h-[28px]` (7-unit minimum row height)
- Move number cell: `w-8 shrink-0 text-sm text-muted-foreground select-none`
- White move cell: `flex-1 min-w-0`
- Black move cell: `flex-1 min-w-0`

Move button (in any cell):
```
text-sm font-mono px-1 py-0.5 rounded transition-colors hover:bg-accent
```
Active move button:
```
bg-primary text-primary-foreground hover:bg-primary/90
```
Empty cell (move not played yet): render nothing (no placeholder text).

**Variation sub-section (desktop):** When `currentNodeId` is not in `mainLine`:
- After the row containing the fork node (last mainLine node that is the parent of the variation), render the variation chain as additional rows with `ml-8` left indent.
- Variation rows use the same `N. white black` paired structure.
- The variation's nodes are dimmed until reached: inactive variation chips use `text-muted-foreground` instead of `text-foreground` for non-active chips.

**Auto-scroll:** Both mobile and desktop variants must scroll the active node into view on `currentNodeId` change. Use a `useEffect` with `activeRef.current?.scrollIntoView({ block: 'nearest', behavior: 'smooth' })` (same pattern as `HorizontalMoveList`). Container must be `overflow-y-auto` (desktop); HorizontalMoveList already handles this for mobile.

**Empty state:** When `mainLine.length === 0` and `nodes.size === 0`:
- Mobile: `HorizontalMoveList` renders its default `emptyText="No moves yet"` вҖ” pass this as `emptyText`.
- Desktop: render `<p className="text-sm text-muted-foreground p-2">No moves yet</p>`.

**Props:**
```typescript
interface VariationTreeProps {
  nodes: Map<NodeId, MoveNode>;
  mainLine: NodeId[];
  currentNodeId: NodeId | null;
  rootPly?: number;         // game ply of rootFen, for move-number labels; default 0
  onNodeClick: (nodeId: NodeId) => void; // calls goToNode
  heightClass?: string;     // mobile HorizontalMoveList height override

```

**data-testid:**
- Outer container: `data-testid="analysis-variation-tree"`
- Each node button: `data-testid={`variation-node-${nodeId}`}`
- Mobile scroll box: `data-testid="variation-tree-mobile"`
- Desktop container: `data-testid="variation-tree-desktop"`

**ARIA:**
- Container: `aria-label="Move list"` `role="navigation"`
- Each node button: `aria-label={`Move ${numberLabel} ${san}`}` `aria-current={isCurrent ? 'step' : undefined}`

## Interaction Contract

### Move chip interaction (all components)

| Interaction | Element | Result |
|-------------|---------|--------|
| Click / tap | PV move chip in EngineLines | Calls `onMoveClick(from, to)` вҖ” triggers `useAnalysisBoard.makeMove` |
| Click / tap | Node chip in VariationTree (mobile or desktop) | Calls `onNodeClick(nodeId)` вҖ” triggers `useAnalysisBoard.goToNode` |
| Keyboard (Enter / Space) | Any focusable `<button>` chip | Same as click |

### Hover states

| State | Class applied |
|-------|--------------|
| Chip hover (inactive) | `hover:bg-accent` |
| Active chip | `bg-primary text-primary-foreground hover:bg-primary/90` |
| EvalBar | No hover interaction (presentational only) |

### Focus states

All interactive `<button>` elements must support visible focus ring via Tailwind's `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1` (shadcn default вҖ” no custom override needed).

### Keyboard navigation

Keyboard back/forward navigation (`ArrowLeft`, `ArrowRight`) is handled by `useAnalysisBoard`'s container-scoped keyboard handler (ARCHITECTURE Pattern 3) вҖ” scoped to `containerRef` div, NOT window-level. This applies to the page shell (Phase 138), not to the analysis components themselves. VariationTree and EngineLines do not install keyboard handlers.
