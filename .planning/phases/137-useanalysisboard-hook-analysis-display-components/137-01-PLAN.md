<objective>
Build `useAnalysisBoard.ts` — a NEW React hook (independent of `useChessGame.ts`) that holds a
branching move tree where a mid-line move forks a new child node instead of truncating the line,
stores a full FEN per node for O(1) navigation, and seeds an initial main line via `loadMainLine`.
Reads NO URL and writes NO URL state (D-01: read-only entry-point only; entry-point param reading
is Phase 138's `Analysis.tsx`).

Purpose: this hook is the state engine the Phase 138 `/analysis` page composes with
`useStockfishEngine`. Phase 137 delivers and unit-tests it in isolation (D-05).
Output: `useAnalysisBoard.ts` (hook + `MoveNode`/`NodeId` types) and its Vitest test file.

Implements: BOARD-01 (mid-line fork), BOARD-02 (back/forward/jump/reset navigation),
BOARD-03 (`makeMove(from, to)` is the input-agnostic move entry point both drag-drop and
click-to-click feed — board input wiring is Phase 138), BOARD-04 (per D-01: `loadMainLine` +
`rootFen` entry-point seeding with NO write-back, NO sessionStorage).

  <action>
    Create the hook cloning useTacticLine's ephemeral shape (no sessionStorage, no Zobrist, no
    opening lookup) and extending it from a linear stepper to a branching tree.

    Types (export both): `NodeId = number`; `MoveNode { id, san, fen, from, to, parentId }` exactly
    per ARCHITECTURE Pattern 3 lines 196-206. Internal state via useState: a `Map<NodeId, MoveNode>`
    nodes, `currentNodeId: NodeId | null` (null = at rootFen), `mainLine: NodeId[]`, `rootFen: string`,
    `nextId: number`. Keep a live `Chess` instance only for making new moves; positions for navigation
    come from stored FENs.

    `makeMove(from, to)`: resolve parentFen = currentNodeId != null ? nodes.get(currentNodeId)!.fen :
    rootFen; build `new Chess(parentFen)`; `chess.move({ from, to, promotion: 'q' })`; on null return
    false. On success build a new MoveNode { id: nextId, san: result.san, fen: chess.fen(),
    from: result.from, to: result.to, parentId: currentNodeId }, insert into a cloned nodes Map,
    advance nextId, set currentNodeId = newNode.id, update position + lastMove. This forks because the
    new node is parented to currentNodeId regardless of existing children — the opposite of
    useChessGame's truncation. This is the input-agnostic entry point both drag-drop and
    click-to-click board input feed (BOARD-03; board wiring is Phase 138).

    Navigation (BOARD-02): `goBack()` → set currentNodeId to current node's parentId (null returns to
    rootFen — this is reset-to-start at root level); `goForward()` → first child of currentNodeId in
    insertion order (scan nodes for the lowest-id node whose parentId === currentNodeId), no-op if
    none; `goToNode(id)` → set currentNodeId = id and position = nodes.get(id)!.fen directly with NO
    replay loop (O(1) — the key win over useChessGame; ARCHITECTURE lines 208-209). Each navigation
    updates position + lastMove (lastMove from the target node's from/to, null at rootFen).

    `loadMainLine(sans, newRootFen)` (BOARD-04 / D-01 entry-point seeding): replay each SAN onto a
    `new Chess(newRootFen)` in sequence, creating one MoveNode per move (parentId chained to the prior
    node, first node's parentId = null), and record their IDs in order into mainLine. Set rootFen =
    newRootFen, currentNodeId to the last seeded id (or null if sans empty), reset nextId past the
    seeded ids. Mirror useTacticLine's rootFen-start replay (lines 117-136), but build tree nodes
    rather than a flat history. `isOnMainLine(nodeId)` = mainLine.includes(nodeId).

    Container-scoped keyboard (ARCHITECTURE Pattern 3 / D-01): expose `containerRef:

    callbacks in useCallback.
