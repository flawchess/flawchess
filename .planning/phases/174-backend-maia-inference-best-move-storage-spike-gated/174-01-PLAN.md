<objective>
Port the client's Maia-3 board->tensor encoding to Python and prove, via a
fixture-based parity spike, that Python + onnxruntime==1.20.1 produces `maia_prob`
values that land in the SAME gem/great/neither tier as the client's onnxruntime-web
output, within a loose empirically-measured epsilon (D-01). This spike IS the
phase gate: per D-02, if it cannot pass, the phase pauses for re-scope and no
downstream work proceeds.

Purpose: Everything else in Phase 174 (table, inference-at-eval-apply, classification)
stores or reclassifies `maia_prob`. If the backend's `maia_prob` disagrees with the
client's at a tier boundary, stored gem/great data would silently diverge from the
live board. This gate is the single guarantee that they agree.

Output: `app/services/maia_encoding.py` (the port), `scripts/maia_parity_spike.py`
(the gate), `tests/services/test_maia_encoding.py` (pure-math unit tests, run without
onnxruntime), a committed fixture corpus, and the new isolated uv group with
`onnxruntime==1.20.1` + `numpy`.
</objective>
