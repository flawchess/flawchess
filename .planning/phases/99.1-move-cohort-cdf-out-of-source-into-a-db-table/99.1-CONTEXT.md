# Phase 99.1: Move Cohort CDF Out of Source into a DB Table - Context

**Gathered:** 2026-05-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Internal refactor / tech-debt. Relocate the generated `COHORT_PERCENTILE_CDF`
registry (3.1 MB / ~130,119 float-literal lines) out of
`app/services/global_percentile_cdf.py` into a new `benchmark_cohort_cdf` DB
table, shrinking the module to ~250 lines of logic and unlocking SQL analysis of
the breakpoints. The generator stops rewriting the `.py` and emits a seed file;
an idempotent seed script loads it. The lookup becomes DB-backed (background path
only, ~32 resolves per import). **Pure relocation — no behavior change**; chip
output must be byte-for-byte equivalent to pre-refactor.

The decision log in `.planning/notes/phase-99-1-cdf-db-refactor.md` locks the
bulk of the design. This discussion only resolved the four implementation
choices the log left open (table layout, seed format, lookup shape, async
boundary).

</domain>

<decisions>
## Implementation Decisions

### Table layout (was "planner's call" in notes)
- **D-01:** **Long** table — one row per breakpoint. Columns:
  `metric, anchor_elo, tc, percentile, value, n_users, snapshot_month`.
  PK `(metric, anchor_elo, tc, percentile)`. ~130k rows. Chosen for the
  SQL-analysis side-benefit (the stated goal). Lookup fetches ~99 rows per
  `(metric, anchor, tc)` cell, `ORDER BY percentile`, rebuilds a `CdfTable`.
  `snapshot_month` is a plain audit column, NOT part of any idempotency gate
  (per notes #1).

### Seed file format
- **D-02:** **TSV, long rows** — `app/data/cohort_cdf.tsv`, header
  `metric\tanchor_elo\ttc\tpercentile\tvalue\tn_users\tsnapshot_month`, one row
  per breakpoint (~130k lines), 1:1 with table rows. Reuses the
  `openings.tsv` + `csv.DictReader` streaming pattern that `seed_openings.py`
  already models. The regen drift test diffs **this file's content** (not source
  bytes), per success criterion #4.

### Lookup query shape
- **D-03:** **Batched prefetch.** `compute_stage_a` / `compute_stage_b` issue
  **one** load query per import that pulls all cells for the user's relevant
  `(anchor, tc)` combinations into an in-memory dict
  `dict[(metric, anchor, tc), CdfTable]`; the ~32 per-import resolves hit that
  dict, not the DB. (User chose this over 32 per-call SELECTs despite the extra
  cache-shape layer — accepted.)

### Async / session boundary — **supersedes notes decision #5 literal wording**
- **D-04:** The async DB access lives in a **new repository**, NOT in
  `global_percentile_cdf.py`. Add
  `app/repositories/benchmark_cohort_cdf_repository.py` with an async
  `load_cohort_cells(session, anchors, tcs) -> dict[(metric, anchor, tc), CdfTable]`.
  `global_percentile_cdf.py` **stays 100% pure and synchronous** —
  `interpolate_cohort_percentile` takes a prefetched `CdfTable` (no session, no
  `async`) and runs the existing `_interpolate_with_table` math.
  - This refines notes #5 / ROADMAP success criterion #3, which assumed
    `interpolate_cohort_percentile` itself becomes async and takes a session.
    That literal wording is **overridden** here: keeping the module pure
    preserves its stated purity invariant (module docstring: "no DB or I/O. All
    functions are synchronous and side-effect free") and its sibling parity with
    `endgame_zones.py`. The planner should reconcile criterion #3 to the
    repository-loader shape (module pure, repo async) rather than the literal
    "interpolate_cohort_percentile is async" text.

### Claude's Discretion
- Exact repository file/function naming, dict key tuple ordering, anchor-grid
  rounding location (already exists in current helper), and the precise SQL for
  the batched prefetch (single `IN`/`= ANY` over the needed anchors+TCs).
- Whether `n_users` / `snapshot_month` are NULLable (audit columns — planner's
  call; not idempotency-relevant).
- Column types (e.g. `SmallInteger` for `percentile` 1..99, `anchor_elo`;
  `float8`/`double precision` for `value`) — follow the project's
  "appropriate column types" DB rule.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CdfTable` dataclass + `_interpolate_with_table` in `global_percentile_cdf.py`:
  interpolation math is unchanged; only the data source moves. The repository
  loader rebuilds `CdfTable` instances from long rows.
- `seed_openings.py` end-to-end structure (engine setup, streaming reader,
  upsert, count return, logging) is a near-direct template for `seed_cohort_cdf.py`.

### Established Patterns
- **FK + explicit types DB rules** apply to the new model. Natural key
  `(metric, anchor_elo, tc, percentile)` → enforce via PK or `UniqueConstraint`.
- **Routers/repos split**: DB access belongs in the new repository, not the
  service or the pure module (reinforces D-04).
- **Seed-on-empty count gate** (openings) for `run_local.sh`; in-container prod
  seed command in the seed script's docstring.

### Integration Points
- `compute_stage_a` / `compute_stage_b` already hold an `AsyncSession` → call
  `load_cohort_cells(session, ...)` once, then resolve in-memory.
- Alembic autogenerate picks up the new model → schema-only migration (no data
  INSERTs in the migration; seeding is the seed script's job).

</code_context>
