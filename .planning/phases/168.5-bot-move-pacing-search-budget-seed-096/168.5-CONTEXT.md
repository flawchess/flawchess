# Phase 168.5: Bot Move Pacing & Search Budget (SEED-096) - Context

**Gathered:** 2026-07-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Make a bot move at any `blend > 0` return in seconds instead of the measured
~110s, without changing what strength *means*: (a) settle the clock/
fixed-strength fork (SEED-091's deferred question) and amend Phase 169 SC1 to
match; (b) make harness Stockfish grades deterministic under load and harden
the watchdog (SEED-095); (c) ship grade-margin early stopping + a retuned
`MAX_NODES` in `mctsSearch`/`useFlawChessEngine.ts`, mirrored into the
harness's D-11 constants; (d) run a bounded re-calibration sweep at exactly
the shipped budget (app == harness) producing the first real strength map.

Owns changes to `mctsSearch.ts` (stop rule), `useFlawChessEngine.ts`
(budget/concurrency constants), and the calibration harness
(`scripts/calibration-harness.mjs`, `scripts/lib/calibration-providers.mjs`,
`scripts/lib/stockfish-pool.mjs`, anchors/sweep logic). Does NOT build the
clocked game loop, clocks UI, reveal delay, or synthetic clock debit
(Phase 169 implements those to this phase's recorded model), and does NOT do
user-results curve fitting (deferred per SEED-091 #3).

</domain>

<decisions>
## Implementation Decisions

### Area A — Clock/fixed-strength fork (SC1)
- **D-01: Fixed strength + synthetic bot clock.** One global search budget →
  one calibrated ELO per config (preserves everything SEED-096 locked). The
  bot's search NEVER reads or bends to the clock. The bot's displayed clock is
  debited a scripted amount per move, not real compute time.
- **D-02: Synthetic debit = fraction-of-remaining schedule** (e.g.
  `remaining/N + share-of-increment`; exact form is Phase 169's to implement,
  the model is locked here). By construction the bot cannot lose on time —
  flag-on-time is a user-side-only end condition.
- **D-03: Small TC-independent randomized minimum reveal delay** (~0.5–1.5s
  ballpark) floors near-instant moves (blend=0 computes in ~0.09s and feels
  robotic). Display-only, implemented in Phase 169's game loop; strength
  untouched. SEED-096's TC-scaled pacing envelope stays deferred.
- **D-04: Phase 169 SC1 is amended during this phase's execution** (part of
  168.5 SC1) to: "the bot paces its replies (not instant): search runs at the
  fixed shipped budget regardless of clock, a small randomized reveal delay
  floors fast moves, and the bot's clock is debited a fraction-of-remaining
  synthetic think time (the bot never flags)."
- **D-04b: Pacing theater is UX-only — the harness has NONE of it.** The
  synthetic clock debit (D-02) and reveal delay (D-03) live exclusively in
  Phase 169's game loop. The harness plays moves back-to-back at raw compute
  speed: no artificial waits, no delay floors, no clock simulation (consistent
  with SEED-096 "no per-game time pressure in calibration"). Sweep wall-clock
  and the D-17 moveMs stats therefore measure pure search time.

### Area B — Early-stop rule + budget
- **D-05: Two-sided stop rule — stop when the decision is SETTLED, either way:**
  - *Clear winner*: top candidate's practicalScore leads the runner-up by
    ≥ margin, stable for K applied expansions (obvious recaptures, only-moves).
  - *Near-tie flatness*: top candidates all within a small ε of each other and
    the ranking has stopped churning (opening routine, dead-equal positions) —
    more search won't change anything that matters; for `blend < 1` the softmax
    sampling picks among them anyway.
  Both sides share a min-nodes floor. Rationale: a margin-only rule gets
  openings exactly backwards (near-equal moves → no clear winner → full-budget
  thinks on routine moves). Only genuinely contested positions (moderate gaps,
  best move still flipping) run toward the full budget — user's explicit
  intent: "spend more time in complicated positions, less in opening routine."
- **D-06: The rule is grade-based and deterministic** — evaluated in canonical
  apply order (per SEED-096), no wall-clock input, no explicit Maia-entropy
  signal in v1 (grade-margin dynamics capture complexity implicitly). All
  thresholds (margin, ε, K, min-nodes floor) are named constants tuned via the
  harness.
- **D-07: Pacing target ~5s median, ~15s worst-case** for `blend > 0` on the
  reference box. `MAX_NODES` sized so a full-budget contested move stays ~15s
  (~50 nodes at the measured ~0.28s/expansion — validate, don't trust).
- **D-08: Same stop rule at ALL blends.** The mid-blend softmax-sampling shift
  (early stop changes the score vector feeding the softmax) is accepted and
  re-measured by the SC4 sweep — no special-casing, no blend-scaled thresholds.
- **D-09: Pin bot-play search concurrency to one fixed constant** (bot games
  only; the analysis board keeps device-adaptive `computePoolSize`). The
  harness uses the same pinned value — app == harness holds exactly, effective
  strength is device-independent, and mctsSearch's per-concurrency-level
  determinism becomes full determinism for bot play.

### Area C — SEED-095 deterministic grades + watchdog
- **D-10: Depth-only `go` + `Clear Hash` per `go` in the harness.** ALL
  harness searches (bot grading AND adjudication) become depth-bounded with NO
  movetime cutoff, plus `setoption name Clear Hash` before every `go`. Result
  depends only on (position, depth, clean hash) — load-independent; D-09
  determinism becomes a hard guarantee instead of probabilistic. Measure the
  throughput cost (grading is the sweep bottleneck), don't assume it.
- **D-11: Watchdog = generous timeout + retry-in-place.** Raise the timeout
  well above worst observed; on breach: `stopAndSync` recovery, retry the same
  `go` N times, engine restart as last resort. The in-flight game survives;
  only after N failures does the cell abort. Plus an operational `--resume`
  retry loop for sweeps.
- **D-12: The app keeps its movetime safety cap** (workerPool grading UX guard
  on slow devices). The harness measures the canonical depth-14 uncapped
  strength; rare device-side grading-depth wobble is accepted (status quo
  Phase 168 already shipped).
- **D-13: The D-09 determinism check stays an on-demand script**, run at the
  new budget + stop rule as this phase's SC5 and documented as the pre-sweep
  sanity step. NOT added to CI (engine spawning too heavy per push).

### Area D — Re-calibration sweep (SC4)
- **D-14: Full coarse grid with trimmed anchors.** ELO {1100, 1500, 1900} ×
  blend {0, 0.5, 1} × trimmed anchor set (Maia rungs bracketing each cell;
  Stockfish-skill anchors droppable for this run) × ~10 games/cell.
  blend=0 cells are cheap (no search) and anchor the human-end scale. This is
  the FIRST real strength map — 168-03 never launched the grid run.
- **D-15: Anchor pruning, both mechanisms (user-requested pragmatism).**
  - *Static bracketing*: per bot cell, only schedule anchors within an
    ELO window of the cell's nominal rating.
  - *Dynamic cutoff*: run a cell's anchors in ascending strength order; after
    the bot scores ~0 vs an anchor, skip all stronger anchors for that cell;
    symmetrically, after a 100% sweep, skip weaker anchors.
  Every skipped matchup is logged in run metadata — no silent coverage gaps.
  Window size / cutoff ε are named constants for the planner.
- **D-16: Wall-clock envelope ~24h** (overnight/weekend) on the reference box,
  operator-launched with `--resume` + the D-11 retry loop — plans must NOT
  block an executor agent on the sweep. Size games-per-cell/anchors to fit the
  envelope at measured per-move cost.
- **D-17: SC4 concrete pass criterion.** `blend > 0` per-move median in
  **3–8s** AND p95 ≤ **20s** across all sweep moves on the reference box
  (Adrian's 16-core dev box). The band catches both "still too slow" and a
  degenerate over-eager stop rule.
- **D-18: Accept & record if top-end strength drops.** The map documents real
  strength at the shipped budget — that IS the deliverable. No in-phase retune
  loop; relabeling bots / strength recovery is future work (SEED-091 #3
  deferral). Pacing wins over nominal top-end strength.

### Claude's Discretion
- Exact values of all named constants (margin, ε, stability window K,
  min-nodes floor, pinned concurrency level, retuned `MAX_NODES`, watchdog
  timeout, retry count N, anchor window, cutoff ε, games-per-cell) — set from
  measurement during execution, within the D-07/D-17 targets.
- Whether `MAX_PLIES` (8) also changes — not discussed; default is keep.
- How the harness mirrors the new constants (import vs mirrored-constant
  comment convention already used for D-11).
- Sweep TSV schema extensions (early-stop stats, skipped-matchup log).

## Existing Code Insights

### Reusable Assets
- `mctsSearch.ts` already applies expansions strictly in canonical dispatch
  order (`Promise.all` input order) and fires `onSnapshot` after every applied
  backup — the natural, already-deterministic hook point for the stop rule:
  evaluate D-05 after each applied expansion, break the outer `while` loop.
- The harness already has `--resume` (SEED-097), streaming durable TSV,
  `stopAndSync` recovery, and the stockfish pool — D-11's retry-in-place and
  D-15's pruning are extensions, not new infrastructure.
- `calibration-determinism.check.mjs` exists; SC5 is a re-run at new settings.

### Established Patterns
- **App == harness (D-11 discipline):** harness mirrors
  `FLAWCHESS_ENGINE_MAX_NODES`/`_MAX_PLIES` as commented mirror constants;
  every new knob (stop-rule thresholds, pinned concurrency) must follow the
  same mirroring so the sweep measures the shipped bot.
- **No-reimplementation (CAL-02):** harness imports `selectBotMove` and engine
  fns via `@/` verbatim — the stop rule ships INSIDE `mctsSearch`, so the
  harness inherits it automatically; only constants need mirroring.
- **Named constants for every threshold** (project-wide rule; SEED-096 and
  Phase 168 D-10 both restate it).

### Integration Points
- `SearchBudget` (types.ts) is the seam for new stop-rule knobs → wired in
  `useFlawChessEngine.ts` (app) and `calibration-harness.mjs` (Node) — both
  already construct budgets against the same type.
- Phase 169 consumes this phase's recorded model: D-01/D-02/D-03 define what
  `useBotGame` implements (synthetic debit, reveal delay, no bot flag);
  D-04's amended SC1 is the contract.
- mctsSearch determinism scope note: output is deterministic PER concurrency
  level (module header) — D-09's pinning is what upgrades that to full
  determinism for bot play; do not attempt to equalize levels.

<specifics>
## Specific Ideas

- User's framing of the stop rule (verbatim intent): "spend more time for
  complicated positions (e.g. when there are several plausible moves according
  to Maia) and less in opening routine and simple positions (one move clearly
  best)" — D-05's near-tie side exists specifically so routine openings go
  fast; a margin-only rule would have inverted this.
- User's sweep pragmatism (verbatim intent): "Super strong Stockfish anchors
  vs weak FlawChess bots doesn't make any sense. Maybe we can stop testing a
  FlawChess bot vs even stronger anchors when it lost all games" → D-15.
- ~15s is a target for the contested worst case, not a wall (SEED-096 #2);
  the median is the UX number (D-07/D-17).

</specifics>
