# Phase 168.5 Plan 04: Bot move pacing/search budget — lock the budget from measurement, mirror it into the harness Summary

**Locked FLAWCHESS_BOT_* values from a real-engine measurement (median ~5.4s, worst-case ~12.7s, confirming Plan 03's provisional values were correct), mirrored them into the calibration harness with cited line-number comments, and wired playGame's SearchBudget to carry the stop rule + pinned concurrency instead of the old full-budget/pool.size search.**

## Accomplishments

- Ran a standalone, throwaway probe (`scripts/probe-bot-budget.mjs`, deleted after use — following Plan 02's precedent of scratchpad-only measurement scripts) that calls `mctsSearch` directly with a hand-built `SearchBudget` carrying `stopRule` + pinned `concurrency`, against the REAL Stockfish pool + Maia ONNX session (via the harness's own exported `setupHarnessEngines()`). `playGame` is not stop-rule-wired until Task 2, so this was the only way to genuinely exercise the early-stop dynamics before locking any constant.
- Measured 8 positions (3 near-tie openings, 1 clear-winner tactical position, 1 flat starting position, 2 genuinely-contested/balanced middlegames), run twice for variance confirmation. Every value came back identical to the provisional values already shipped in Plan 03 — no numeric change, only the `PROVISIONAL` markers removed and replaced with a one-paragraph measurement citation in `useFlawChessEngine.ts`.
- Added the `FLAWCHESS_BOT_MAX_NODES`/`_MAX_PLIES`/`_CONCURRENCY`/`_STOP_RULE` mirror block to `calibration-harness.mjs` (D-11 comment convention: "mirrors X — useFlawChessEngine.ts line N"), additive alongside the existing untouched analysis-board `FLAWCHESS_ENGINE_MAX_*` mirrors.
- Rewired `playGame`'s `SearchBudget`: `maxNodes`/`maxPlies` now default from the bot budget (not the old 400-node analysis budget), `concurrency` is PINNED to `FLAWCHESS_BOT_CONCURRENCY` (replacing `pool.size`), and `stopRule: FLAWCHESS_BOT_STOP_RULE` is now set — the harness measures the shipped bot, completing SC3's app==harness half and preparing SC4's sweep.
- Updated the `--resume` budget-mismatch guard and the TSV row's recorded `maxNodes`/`maxPlies` to check/record against the bot budget, so a resumed sweep refuses to mix a prior 400-node run with the new bot budget.
- Smoke-tested the full wiring end-to-end with a real (short) harness cell run — bot move times landed in the 2.18s-13.49s range, matching the target band.

## Measurement (Task 1)

**Method:** standalone probe script (`scripts/probe-bot-budget.mjs`, throwaway — deleted before finishing per the plan's own instruction), reusing `calibration-harness.mjs`'s exported `setupHarnessEngines()` (real 4-process Stockfish pool + real Maia ONNX session + `chess.js`), calling `mctsSearch` DIRECTLY (bypassing `selectBotMove`'s argmax layer, since the goal was to observe raw search/stop-rule diagnostics: `nodesEvaluated`, `stopReason`, `budgetExhausted`) with a hand-built `SearchBudget`:

```js
{ maxNodes: 50, maxPlies: 8, concurrency: 4, elo: { w: 1500, b: 1500 }, stopRule: { marginThreshold: 0.05, epsilonThreshold: 0.02, stabilityWindow: 3, minNodes: 8 } }
```

**Positions and results** (run twice; results below are the second run, both runs agreed within ~1-2% wall-clock, `nodesEvaluated` byte-identical between runs as expected for a fixed concurrency level):

| Position | Type | moveMs | nodesEvaluated | stopReason |
|---|---|---|---|---|
| Italian Game | near-tie opening | 5,891 (5.89s) | 15 | early-stop |
| Sicilian Najdorf | near-tie opening | 3,799 (3.80s) | 8 | early-stop |
| King's Indian | near-tie opening | 6,123 (6.12s) | 19 | early-stop |
| Fried Liver Attack | clear-winner tactical | 1,955 (1.95s) | 8 | early-stop |
| QGD middlegame-ish | mixed | 4,980 (4.98s) | 18 | early-stop |
| Starting position | max-flat opening | 2,318 (2.32s) | 8 | early-stop |
| Berlin Defense endgame | genuinely contested | 9,106 (9.11s) | 50 (budget) | budget |
| Symmetric middlegame | genuinely contested | 12,732 (12.73s) | 50 (budget) | budget |

**Median (n=8):** ~5.44s (sorted: 1.95, 2.32, 3.80, 4.98, 5.89, 6.12, 9.11, 12.73 — avg of 4th/5th = (4.98+5.89)/2). **Worst-case:** 12.73s.

**Two-sided rule separation confirmed:** every near-tie/decisive position (6 of 8) settled via `early-stop` well before the 50-node cap (8-19 nodes); the two positions deliberately chosen to be genuinely balanced/undecided (Berlin Defense endgame, a queenless symmetric IQP-ish middlegame) ran to the FULL 50-node budget without the stop rule ever falsely firing. This is the behavior the plan's `must_haves` required: a genuinely contested position stays near the worst-case target, while routine/settled positions trend toward the median target.

**D-07/D-17 target comparison:** D-07 targets ~5s median / ~15s worst-case; D-17's pass band is 3-8s median / p95<=20s. Measured median ~5.44s sits inside the 3-8s band; measured worst-case 12.73s sits comfortably under the 15s target and the 20s p95 ceiling, giving headroom rather than a knife-edge fit.

**Exploratory calibration note (not a final measurement, discarded):** before settling on maxNodes=50, an exploratory run at maxNodes=80 was used to characterize the two contested positions' natural growth curve without a cap — Berlin Defense ran to 80 nodes/13.82s, the symmetric middlegame to 80 nodes/21.57s (the latter exceeding the 15s target), confirming that a cap in the 45-55 node range was the right ceiling. maxNodes=50 (the value already shipped in Plan 03) was then re-confirmed as the correct choice.

**Environment note:** a pre-existing operator-launched calibration sweep (`reports/data/calibration-harness-2026-07-12T10-24-36-664Z.tsv`, PID 160903, `--blends 0` only) was running throughout this plan's execution, consuming one Stockfish process and modest CPU. Per the plan's environment note, this was left untouched; the measured moveMs numbers above reflect this contention, not an idle machine — actual production worst-case latency on an uncontended box is likely equal or slightly better.

## Decisions Made

- All four `FLAWCHESS_BOT_*` values locked at their Plan 03 provisional values (50/8/4/{marginThreshold:0.05, epsilonThreshold:0.02, stabilityWindow:3, minNodes:8}) — the measurement's role here was validation, not retuning, and it genuinely validated: 6/8 tested positions settled early well under budget, 2/8 deliberately-contested positions ran to the full cap within the 15s target with headroom.
- `scripts/probe-bot-budget.mjs` deleted after use (Claude's discretion, per the plan's explicit option) rather than kept — it is genuinely throwaway (hardcoded FEN list, no CLI, duplicate of harness setup logic) and keeping it would add an unmaintained parallel entry point next to the real harness.
- Old `FLAWCHESS_ENGINE_MAX_NODES`/`_MAX_PLIES` mirror constants in `calibration-harness.mjs` kept (not deleted) even though no code in this file references them anymore after Task 2's edits — they remain valid, cited documentation of the untouched analysis-board budget (matching the hook's own unchanged constants), and the plan explicitly said not to delete them.
- `concurrency: FLAWCHESS_BOT_CONCURRENCY` (pinned 4) replaces `concurrency: pool.size` in `playGame`'s `SearchBudget` — this is a deliberate decoupling from `--stockfish-procs`: running the CLI with a different `--stockfish-procs` value no longer silently changes the bot's search concurrency (and therefore its determinism-per-concurrency-level output), matching D-09's "pinned" requirement exactly.
