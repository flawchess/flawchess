<objective>
Deliver the two operator-facing scripts that make the rollout observable and graded:

1. **`scripts/backfill_multipv.py`** — the SC1 deliverable, reconciled with D-01/D-02. It is a THIN
   observability/kickoff CLI, NOT a server-local engine backfill (there is no second EnginePool — the
   remote fleet does the bulk compute via the tier-4 lottery). It exposes `--status` (progress query:
   how many games/flaws still have `allowed_pv_lines IS NULL`, spanning engine + lichess), `--dry-run`
   (how many flaws would be backfilled), and `--dev-validate` (run the tier-4 lottery + lease + submit
   + per-game retag end-to-end on the dev DB before any `--db prod` rollout). It still honors SHIP-01's
   literal acceptance: a `--db prod` backfill mechanism with the `WHERE allowed_pv_lines IS NULL`
   idempotency guard, not gated on `lichess_evals_at`.
2. **`scripts/snapshot_tactic_counts.py`** — the SC3 deliverable. `--phase before|after` runs the
   per-motif allowed/missed chip-count queries and writes/append a committed
   `reports/retag/rollout-YYYY-MM-DD.md` table.

Purpose: Give the operator a progress gate and a before/after noise-reduction record.
Output: Two scripts following the `backfill_full_evals.py` / `retag_flaws.py` CLI conventions.
</objective>
