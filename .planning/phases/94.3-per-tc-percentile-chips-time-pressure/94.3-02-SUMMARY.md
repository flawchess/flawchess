# Phase 94.3 Plan 02: per-TC pooled-aggregate SQL builders — Summary

Three new per-(metric × TC) pooled-aggregate SQL builder families and 12 new dispatcher arms in `app/services/canonical_slice_sql.py`, with 91 new pytest cases covering SQL shape, per-TC predicate placement, metric-specific HAVING floors, and benchmark-vs-single_user parity. This is the first commit of the Wave-1 atomic-cutover trio (B → C → D); locating the per-TC restriction inside the shared builder makes drift between CDF construction (Plan C) and per-user lookup (Plan D) structurally impossible.

## What shipped

### New module-level constants (block as committed)

```python
# Phase 94.3 per-TC time-pressure builder thresholds (CONTEXT D-6).
# `TIME_PRESSURE_CLOCK_PCT_THRESHOLD` defines the "under time pressure"
# cutoff applied to each side's clock fraction at endgame entry: clock_pct
# < 0.40 = pressured. The remaining three constants are the inclusion
# floors gated via HAVING on the pooled per-user aggregate (≥30 mirrors
# Phase 94.2 D-6 for every other metric in this module).
TIME_PRESSURE_CLOCK_PCT_THRESHOLD: float = 0.40
TIME_PRESSURE_MIN_PRESSURED_N: int = 30
CLOCK_GAP_MIN_POOL_N: int = 30
NET_FLAG_RATE_MIN_POOL_N: int = 30
```

### TypeAlias

```python
TimeControlBucket: TypeAlias = Literal["bullet", "blitz", "rapid", "classical"]
```

Reused across all three new per-TC builder signatures and the `_recent_capped_per_tc_cte` helper. Ensures the leading positional `tc` argument is type-checked.

### New helpers

- `_recent_capped_per_tc_cte(snapshot_date, tc) -> str` — sibling of `_recent_capped_cte` that adds `AND g.time_control_bucket = '{tc}'` to the inner WHERE and collapses the ROW_NUMBER `PARTITION BY` from `(g.user_id, g.time_control_bucket)` to `(g.user_id)` (because the restricted set already contains only one TC).
- `_endgame_entry_clocks_cte() -> str` — per-game endgame-entry clock derivation shared by all three new per-TC builders. For each game with ≥6 endgame plies, emits `(game_id, white_entry_clock, black_entry_clock)` where the white/black clock is the first non-null `clock_seconds` at that parity's plies (mirroring `_extract_entry_clocks` in `endgame_service.py`).

### Three new per-TC builders

| Builder | Signature | Metric value | HAVING floor |
|---|---|---|---|
| `per_user_cte_time_pressure_score_gap` | `(tc, *, source, snapshot_date=None)` | `user_pressured_score − opp_pressured_score` at `clock_pct < 0.40` | dual ≥30: pressured-cell user-side AND opp-side |
| `per_user_cte_clock_gap` | `(tc, *, source, snapshot_date=None)` | `avg((user_clock − opp_clock) / base_time_seconds)` at endgame entry | pooled ≥30 |
| `per_user_cte_net_flag_rate` | `(tc, *, source, snapshot_date=None)` | `(timeout_wins − timeout_losses) / total_endgame_games` | pooled ≥30 |

All three project `per_user_values(metric_value, n_games)` identical in shape to the Phase 94.2 builders. `n_games` for `time_pressure_score_gap` = `least(user_pressured_n, opp_pressured_n)::int` (binding floor on the dual-HAVING gate); for the other two = pooled-set count.

### 12 new dispatcher arms

`per_user_cte_for(metric_id, *, source, snapshot_date)` widened with explicit if-arms (PATTERNS.md recommended style) for the 12 cells, in this order (Plan C must use these exact strings):

```
1.  time_pressure_score_gap_bullet
2.  time_pressure_score_gap_blitz
3.  time_pressure_score_gap_rapid
4.  time_pressure_score_gap_classical
5.  clock_gap_bullet
6.  clock_gap_blitz
7.  clock_gap_rapid
8.  clock_gap_classical
9.  net_flag_rate_bullet
10. net_flag_rate_blitz
11. net_flag_rate_rapid
12. net_flag_rate_classical
```

The fallback `raise ValueError(f"Unknown metric_id: {metric_id!r}")` is unchanged.

## Status of `tests/scripts/test_gen_global_percentile_cdf_unchanged.py`

**The plan predicted this test would be EXPECTED RED after Plan B.** It is in fact still GREEN.

The reason: that test only covers the 4 existing `CdfMetricId` literals (`score_gap`, `achievable_score_gap`, `section2_score_gap_conv`, `section2_score_gap_parity`). Plan B does not modify those four metrics' SQL output — they continue to route to the same `per_user_cte_score_gap` / `per_user_cte_achievable` / `per_user_cte_section2` builders with identical bodies. The plan's prediction confused "the dispatcher is widened" with "the existing 4 metrics' SQL changes" — only the former is true.

Plan C will widen `IN_SCOPE_METRICS` in `scripts/gen_global_percentile_cdf.py` to 16 entries and regenerate the goldens table — at that point the goldens grow from 4 entries to 16, but the existing 4 entries' SQL bodies remain unchanged. No mid-trio RED transition.

This is a documentation deviation from the plan, not an implementation deviation. Plan C should proceed without expecting to fix a RED test from Plan B.

## Deviations from Plan

### [Doc-only deviation] `tests/scripts/test_gen_global_percentile_cdf_unchanged.py` is NOT RED after Plan B

- **Found during:** post-plan verification step ("EXPECTED RED" assertion).
- **Issue:** The plan and its `<verification>` block claim this test is RED after Plan B and stays red until Plan C regenerates goldens. In fact the test continues to pass because (a) Plan B leaves the existing 4-metric SQL bodies completely unchanged and (b) the script's `IN_SCOPE_METRICS` (4 entries) bounds what the regression test exercises.
- **Fix:** None — the implementation is correct; only the plan's prose is slightly inaccurate. Documented in the "Status of …" section above. Plan C should proceed without expecting to flip this test from RED to GREEN.
- **Files modified:** none (documentation note only).

### [Rule 2 — Auto-add missing critical functionality] `_endgame_entry_clocks_cte` helper extraction

- **Found during:** Task 2 implementation.
- **Issue:** The plan describes inlining the endgame-entry clock derivation in each of the three builders. Inlining a 6-line CTE body three times in the same module is a smell — the body is byte-identical, and a future schema change (e.g. switching from ply-parity to a stored `user_clock_seconds` column once it exists) would have to land in three places.
- **Fix:** Centralised as `_endgame_entry_clocks_cte()`, called from all three per-TC builders. Sibling-helper pattern, identical in spirit to how `_recent_capped_cte` and `_recent_capped_per_tc_cte` coexist.
- **Files modified:** `app/services/canonical_slice_sql.py` (one new helper function).
- **Commit:** `512acac5`

### [Style polish] `{:.2f}` formatting on `TIME_PRESSURE_CLOCK_PCT_THRESHOLD` interpolation

- **Found during:** Task 2 GREEN phase (test assertions expected `0.40`, Python's default float repr produced `0.4`).
- **Issue:** `f"... < {TIME_PRESSURE_CLOCK_PCT_THRESHOLD} ..."` rendered as `< 0.4`, breaking the lock to D-6's `0.40` and forcing whitespace-normalised substring assertions to work around the trailing-zero drop. Plan C's golden fixture would also have to strip the trailing zero, making future no-op refactors trigger a regeneration.
- **Fix:** All interpolation sites use `{TIME_PRESSURE_CLOCK_PCT_THRESHOLD:.2f}` explicitly, so rendered SQL always contains `< 0.40`.
- **Files modified:** `app/services/canonical_slice_sql.py` (three sites: WHERE FILTER for user-pressured, opp-pressured avg, and HAVING gate).
- **Commit:** `512acac5`
