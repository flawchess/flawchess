## Summary

Phase 94.3 is a **parameterised extension** of the shipped Phase 94.2 pipeline, not a feature build. Every primitive it needs already exists: the pooled-per-user CTE pattern, the CDF regen script, the Stage A/B trigger + UPSERT + `interpolate_percentile` plumbing, the per-user backfill, and the `PercentileChip` component including the 4-bullet popover body. Phase 94.3 adds:

- **Three new builder families** in `app/services/canonical_slice_sql.py`, each accepting a `tc` parameter, each restricting the existing `recent_capped` CTE to a single TC, each emitting the same `per_user_values(metric_value, n_games)` shape on both `source="benchmark"` and `source="single_user"`.
- **12 new ENUM values** (`{time_pressure_score_gap, clock_gap, net_flag_rate}_{bullet, blitz, rapid, classical}`) added to `benchmark_metric` via a single `ALTER TYPE … ADD VALUE` migration. Reversibility is irreducibly partial (Postgres ENUMs cannot remove values without a type-and-table rewrite — document the cost in the migration docstring).
- **12 new entries** in `GLOBAL_PERCENTILE_CDF` produced by 12 new breakpoint queries dispatched from `scripts/gen_global_percentile_cdf.py`.
- **12 new Stage B compute calls** in `compute_stage_b`, each calling the corresponding shared builder with `source="single_user"`.
- **12 new backfill counters** flowing through the existing `_MetricSummary` machinery.
- **One new `direction` prop** on `PercentileChip` (flavor-bound default per CONTEXT.md "Claude's Discretion"); the flavor union widens from 4 to 16 variants. Three chip slots on each of the 4 `TimePressureTcCard` instances (Clock Gap on `ClockGapHeaderRow`, Net Flag on `NetFlagRateRow`, Time Pressure Score Gap on the Quintiles subtitle line).
- **One candidacy report extension** (12 new (metric × TC) Cohen's d cells) to calibrate the per-chip rating-correlation tooltip line.

The surface is wider than 94.2 (12 metrics vs 4; 3 chip slots × 4 TCs vs 4 chip rows; new directional-inversion logic; new backend metric family for Time Pressure Score Gap that doesn't exist anywhere yet) but the **shape** is parameterised, not novel. Phase 94.2 is the template.

**Primary recommendation:** Adopt the CONTEXT.md A–F plan boundary verbatim, mirroring 94.2's wave structure (Plan A as a research-time candidacy deliverable feeding Plan F's tooltip copy; Plans B+C+D coordinated in one wave for the no-incoherent-green-state contract; Plan E backfill in Wave 2 blocked on Wave 1; Plan F frontend in Wave 2 with the candidacy report's tier copy as its input). Six plans total.

### Pattern 1: Per-TC pooled CTE shape

**What:** A new builder per metric family taking `tc: Literal["bullet", "blitz", "rapid", "classical"]`. Each restricts `recent_capped` to one TC by adding `AND g.time_control_bucket = '{tc}'` to the inner `recent_games` WHERE clause. The cap (`ROW_NUMBER() <= 1000`) and 36-month recency remain, but the partition `(user_id, time_control_bucket)` simplifies to `(user_id)` because there's only one TC inside the restricted set — equivalent to "most recent 1000 games of that TC."

**When to use:** Three new builders. Recommended naming consistent with the existing `per_user_cte_*` family:

- `per_user_cte_time_pressure_score_gap(tc, *, source, snapshot_date=None) -> str`
- `per_user_cte_clock_gap(tc, *, source, snapshot_date=None) -> str`
- `per_user_cte_net_flag_rate(tc, *, source, snapshot_date=None) -> str`

**Dispatcher extension** in `per_user_cte_for`:
```python
def per_user_cte_for(
    metric_id: CdfMetricId,
    *,
    source: Literal["benchmark", "single_user"],
    snapshot_date: date | None = None,
) -> str:
    if metric_id == "score_gap":
        return per_user_cte_score_gap(source=source, snapshot_date=snapshot_date)
    # ... existing 3 dispatch arms ...
    # NEW: 12 dispatch arms for the per-TC metrics.
    if metric_id == "time_pressure_score_gap_bullet":
        return per_user_cte_time_pressure_score_gap("bullet", source=source, snapshot_date=snapshot_date)
    # ... 11 more arms ...
```

**Example (sketch — time_pressure_score_gap, bullet, single_user):**
```sql
-- D-6 inclusion floor: ≥30 endgame-entry games where USER hit clock_pct < 40%
-- AND ≥30 endgame-entry games where OPP hit clock_pct < 40%.
-- The pool: most-recent-1000 bullet games × 36-month window, universal filters.

WITH selected_users AS (SELECT CAST(:user_id AS int) AS user_id),
recent_capped AS (
  SELECT g.id, g.user_id, g.user_color, g.result
  FROM (
    SELECT g.*,
           row_number() OVER (PARTITION BY g.user_id ORDER BY g.played_at DESC) AS rn
    FROM games g
    JOIN selected_users su ON su.user_id = g.user_id
    WHERE g.rated AND NOT g.is_computer_game
      AND g.white_rating IS NOT NULL AND g.black_rating IS NOT NULL
      AND g.played_at >= NOW() - INTERVAL '36 months'
      AND abs((CASE WHEN g.user_color='white' THEN g.white_rating ELSE g.black_rating END)
            - (CASE WHEN g.user_color='white' THEN g.black_rating ELSE g.white_rating END)) <= 100
      AND g.time_control_bucket = 'bullet'        -- NEW: per-TC restriction
  ) g
  WHERE g.rn <= 1000
),
-- endgame-entry derivation reused from achievable / section2 pattern:
endgame_entries AS (
  SELECT gp.game_id,
         (array_agg(gp.user_clock_seconds ORDER BY gp.ply ASC))[1]::float
           / NULLIF((array_agg(gp.base_clock_seconds ORDER BY gp.ply ASC))[1]::float, 0)
             AS user_clock_pct,
         (array_agg(gp.opp_clock_seconds ORDER BY gp.ply ASC))[1]::float
           / NULLIF((array_agg(gp.base_clock_seconds ORDER BY gp.ply ASC))[1]::float, 0)
             AS opp_clock_pct
  FROM game_positions gp
  WHERE gp.endgame_class IS NOT NULL
  GROUP BY gp.game_id
  HAVING count(gp.ply) >= 6
),
joined AS (
  SELECT rc.user_id, rc.result, rc.user_color,
         ee.user_clock_pct, ee.opp_clock_pct,
         CASE WHEN (rc.result='1-0' AND rc.user_color='white')
                OR (rc.result='0-1' AND rc.user_color='black') THEN 1.0
              WHEN rc.result='1/2-1/2' THEN 0.5 ELSE 0.0 END AS user_score
  FROM recent_capped rc
  JOIN endgame_entries ee ON ee.game_id = rc.id
),
per_user AS (
  SELECT
    user_id,
    avg(user_score)         FILTER (WHERE user_clock_pct < 0.40) AS user_pressured_score,
    avg(1.0 - user_score)   FILTER (WHERE opp_clock_pct  < 0.40) AS opp_pressured_score,
    count(*) FILTER (WHERE user_clock_pct < 0.40) AS user_pressured_n,
    count(*) FILTER (WHERE opp_clock_pct  < 0.40) AS opp_pressured_n
  FROM joined
  GROUP BY user_id
  HAVING count(*) FILTER (WHERE user_clock_pct < 0.40) >= 30
     AND count(*) FILTER (WHERE opp_clock_pct  < 0.40) >= 30
),
per_user_values AS (
  SELECT
    (user_pressured_score - opp_pressured_score) AS metric_value,
    least(user_pressured_n, opp_pressured_n)::int AS n_games
  FROM per_user
)
```

⚠️ The exact column names for clock data on `game_positions` are not verified in this research pass (`gp.user_clock_seconds`, `gp.opp_clock_seconds`, `gp.base_clock_seconds`). The planner must verify against the actual schema before writing the builder. The candidacy report extension (Plan A) will need to verify these as well — see Open Landmines.

For **Clock Gap per TC**, the canonical-slice shape is much simpler — the metric is `avg((user_clock - opp_clock) / base_clock)` at endgame entry, restricted to one TC, with the existing pooled-set ≥30 floor:
```sql
per_user AS (
  SELECT
    user_id,
    avg((user_clock_seconds - opp_clock_seconds)::float / NULLIF(base_clock_seconds, 0)) AS clock_gap_frac,
    count(*) AS pool_n
  FROM joined
  GROUP BY user_id
  HAVING count(*) >= 30
),
per_user_values AS (
  SELECT clock_gap_frac AS metric_value, pool_n AS n_games FROM per_user
)
```

For **Net Flag Rate per TC**, the metric is `(timeout_wins − timeout_losses) / total_endgame_games` (matching the existing `net_timeout_rate` field on `TimePressureTcCard`). The shape:
```sql
per_user AS (
  SELECT
    user_id,
    (count(*) FILTER (WHERE termination = 'time forfeit' AND opponent_lost)::float
     - count(*) FILTER (WHERE termination = 'time forfeit' AND user_lost)::float)
    / NULLIF(count(*), 0) AS net_flag_rate,
    count(*) AS pool_n
  FROM joined
  GROUP BY user_id
  HAVING count(*) >= 30
),
per_user_values AS (
  SELECT net_flag_rate AS metric_value, pool_n AS n_games FROM per_user
)
```

⚠️ Exact column names for game termination (`termination`, derived user/opp outcome on time forfeit) need verification against the schema. ### Pattern 2: ENUM migration recipe

**What:** Single Alembic migration adds 12 new values to the `benchmark_metric` ENUM. The pattern mirrors what would have been required if Phase 94.2 had added an ENUM (it didn't — 94.2 kept the 4-value set).

**Recipe:**
```python
# alembic/versions/20260525_extend_benchmark_metric_for_tc_pressure.py

from collections.abc import Sequence
from alembic import op

revision: str = "{generated}"
down_revision: str | Sequence[str] | None = "b8f4d92c1e25"  # n_games add (94.2)
branch_labels = None
depends_on = None

# New ENUM values added in this migration. Ordered for grep-ability;
# Postgres preserves insertion order in pg_enum.
_NEW_VALUES: tuple[str, ...] = (
    "time_pressure_score_gap_bullet",
    "time_pressure_score_gap_blitz",
    "time_pressure_score_gap_rapid",
    "time_pressure_score_gap_classical",
    "clock_gap_bullet",
    "clock_gap_blitz",
    "clock_gap_rapid",
    "clock_gap_classical",
    "net_flag_rate_bullet",
    "net_flag_rate_blitz",
    "net_flag_rate_rapid",
    "net_flag_rate_classical",
)

def upgrade() -> None:
    """Add 12 new ENUM values to benchmark_metric for Phase 94.3 per-TC chips.

    Postgres ALTER TYPE … ADD VALUE is the only supported per-value extension
    path. Each statement is its own transaction (Postgres 12+ allows running
    ADD VALUE inside the migration transaction; Postgres 18 also supports
    IF NOT EXISTS, used here for idempotency).

    REVERSIBILITY CAVEAT: Postgres does NOT support DROP VALUE on an ENUM.
    The downgrade implementation requires creating a new replacement ENUM,
    casting the column, dropping the old type — a full table rewrite. The
    downgrade below is intentionally a no-op stub that documents this cost.
    If a true rollback is needed, prefer `git revert` of the upgrade migration
    BEFORE any prod row is written with one of the new values; once values
    exist on rows, the rollback procedure is the table-rewrite recipe.
    """
    for value in _NEW_VALUES:
        op.execute(f"ALTER TYPE benchmark_metric ADD VALUE IF NOT EXISTS '{value}'")

def downgrade() -> None:
    """No-op stub. Postgres cannot remove ENUM values without a table rewrite.

    To truly roll back: (1) create a new ENUM type with the 4 original values,
    (2) update user_benchmark_percentiles.metric to use the new type, (3) drop
    the old ENUM. The procedure is destructive and out of scope for this
    migration's downgrade. Use `git revert` of the upgrade commit instead, and
    accept that rows written with new ENUM values during the deploy window
    remain in the table with old ENUM values until manual cleanup.
    """
    pass
```

**Verified pattern reference:** The original `20260524_000000_add_user_benchmark_percentiles.py` created the ENUM type via `postgresql.ENUM(...).create(op.get_bind(), checkfirst=True)`. Extending it with `ALTER TYPE ADD VALUE` is the standard Postgres-supported path; SQLAlchemy itself does not wrap this with a high-level API.

**Commit-ordering caveat (see Open Landmines):** the ENUM extension migration MUST land before any code path attempts to use one of the new metric IDs at runtime. Because the migration runs on container startup via `deploy/entrypoint.sh`, and all 12 new code paths land in the same PR, the deploy is atomic — but the CI test suite must pass against a DB with the migration applied. Plan B's pytest run depends on `alembic upgrade head` being applied to the test DB before the per-TC builder tests execute.

## Common Pitfalls

### Pitfall 1: ENUM extension run AFTER code that emits the new ENUM values

**What goes wrong:** Plan D's service rewrite lands before Plan D's migration runs against the target DB. Stage B fires for a user; `_compute_metric_for_user` produces a value for `time_pressure_score_gap_bullet`; `upsert_percentile(..., metric='time_pressure_score_gap_bullet')` raises `InvalidTextRepresentation: invalid input value for enum benchmark_metric: "time_pressure_score_gap_bullet"`.

**Why it happens:** Migrations run on container startup in prod (via `deploy/entrypoint.sh`) — fine. In dev, the operator may forget `uv run alembic upgrade head` between checkout and test exercise.

**How to avoid:** Plan D's task list MUST gate code-level changes behind the migration. Recommend Task 1 = create migration; Task 2 = run `alembic upgrade head` against dev; Task 3 = widen `STAGE_B_METRICS` + write tests. Pytest fixtures that touch `user_benchmark_percentiles` already require migration-applied DB.

**Warning signs:** Sentry alerts for `InvalidTextRepresentation` shortly after deploy if migration ordering is wrong. (In practice, Alembic on container startup makes this near-impossible in prod; the risk is purely local-dev workflow.)

### Pitfall 2: Per-TC restriction injected at the service or script layer

**What goes wrong:** A reviewer "simplifies" the per-TC builders by injecting `AND g.time_control_bucket = '{tc}'` inside `gen_global_percentile_cdf.py` or `_compute_metric_for_user` instead of inside the shared `canonical_slice_sql.py` builder. The CDF construction and per-user lookup paths drift — same metric_id produces different SQL — but the test suite doesn't catch it because the per-TC test fixtures don't cover both consumers.

**Why it happens:** The "simpler" code-path can look right when only one consumer is exercised by tests.

**How to avoid:** Plan B's tests MUST assert that `per_user_cte_for("clock_gap_bullet", source="benchmark", ...)` and `per_user_cte_for("clock_gap_bullet", source="single_user", ...)` produce identical pooled-aggregate bodies (only the cohort CTE differs — see 94.2's existing parity test pattern in `tests/services/test_canonical_slice_per_user_vs_benchmark_parity.py`). Extend that file with 12 new parity assertions.

**Warning signs:** Plan F's chip values don't match Plan A's candidacy table percentile estimates.

### Pitfall 3: PercentileChipFlavor exhaustiveness switches not updated

**What goes wrong:** The current `RATING_NOTE_BY_FLAVOR` map uses `satisfies Record<PercentileChipFlavor, string>` to enforce exhaustiveness. When `PercentileChipFlavor` widens to 16 variants, the build fails until all 16 keys are added — that's the good case. The bad case: a switch statement or `Object.keys(...).forEach(...)` pattern that does NOT use `satisfies` silently drops the new flavors.

**Why it happens:** Phase 94.2's `PERCENTILE_METRIC_LABELS` and `RATING_NOTE_BY_FLAVOR` both correctly use `satisfies`. The new `DIRECTION_BY_FLAVOR` must also use `satisfies`.

**How to avoid:** Audit `frontend/src/components/charts/PercentileChip.tsx` for every `Record<PercentileChipFlavor, ...>` declaration and ensure `as const satisfies Record<PercentileChipFlavor, ...>` is the pattern. Add a frontend lint guard or a test that iterates `Object.keys(DIRECTION_BY_FLAVOR)` and asserts length 16.

**Warning signs:** Compile passes but `DIRECTION_BY_FLAVOR['clock_gap_classical']` returns `undefined` at runtime; chip falls through to a default branch that renders the wrong color band.

### Pitfall 4: Snapshot date inconsistency between per-TC CDF construction and per-user lookup

**What goes wrong:** Same as 94.2 Pitfall 1, but per-TC: the CDF is built with `--snapshot-date 2026-03-31` against the benchmark DB, but a user's lookup uses `NOW()` on prod. A user with 36 months of bullet games ending 2026-05-24 hits a different recency window than the cohort.

**Why it happens:** The `snapshot_date: date | None` parameter on `per_user_cte_for(...)` already handles this asymmetry per 94.2's design — `None = NOW()` on per-user path; explicit date on CDF path. The pitfall is only if the per-TC builder family forgets to thread `snapshot_date` through to `_recent_capped_cte(snapshot_date)`.

**How to avoid:** Each of the 3 new builder families MUST accept `snapshot_date: date | None = None` and pass it to `_recent_capped_cte`. Plan B's test asserts the signature.

**Warning signs:** A user's stored value swings noticeably between two backfill runs separated by a few weeks despite no new imports.

### Pitfall 5: Time Pressure Score Gap column names not verified

**What goes wrong:** Pattern 1's sketch SQL uses `gp.user_clock_seconds`, `gp.opp_clock_seconds`, `gp.base_clock_seconds` — the planner writes the builder with these column names, the test fails, the planner debugs for 30 minutes before reading the schema and discovering the actual column is e.g. `gp.user_clock_ms` or that the base clock lives on the `games` table.

**Why it happens:** This research pass did not run `\d game_positions` against the benchmark DB. The column names are inferred from `endgame_service.py:_compute_time_pressure_cards`'s use of `clock_rows` — but `clock_rows` comes from a repository query that may already do unit conversions.

**How to avoid:** Plan B Task 0 (or Plan A's candidacy SQL) — schema probe: `\d game_positions` + `\d games` against the benchmark DB. Document exact column names + units in the Plan B task notes. Cross-reference with `query_clock_stats_rows` in the repository (called from `endgame_service.py:2844`) which already does the per-game extraction.

**Warning signs:** Builder tests fail with `column gp.user_clock_seconds does not exist`.

### Pitfall 6: Backfill summary log emits 16 rows but the planner expected 12 new

**What goes wrong:** The summary table at the end of the backfill run shows 16 metric rows (the 4 existing + 12 new). The user reads this and assumes the 4 existing values were recomputed against a stale CDF; or they don't see one of the 12 new rows and think it failed.

**Why it happens:** `STAGE_B_METRICS` is a tuple of 15 entries; combined with `STAGE_A_METRIC` it's 16. The loop emits one row per entry.

**How to avoid:** Plan E's task notes explicitly call out that the summary will show 16 rows (`score_gap` + 3 ΔES + 12 per-TC). Plan E's verification asserts:
- The 4 existing rows are present (sanity).
- The 12 new rows are present.
- Each row's `upserted + skipped_below_pooled_floor` sum equals the total user count.

**Warning signs:** A row missing from the summary table — indicates either ENUM extension didn't apply or `STAGE_B_METRICS` tuple is missing a value.

### Pitfall 7: Net Flag Rate 0.0 vs null confusion

**What goes wrong:** A user with 50 bullet endgames and zero timeouts on either side has `net_flag_rate = 0.0` (exactly the midpoint of the distribution). The per-user pooled SQL emits `metric_value = 0.0`, `n_games = 50`, percentile gets computed against the CDF — likely lands near p50. Chip renders "Top 50%" with neutral band — correct.

The pitfall: a user with 5 bullet endgames (below floor) has no row in `user_benchmark_percentiles` because the HAVING clause filters them out. `fetch_for_user` returns no `net_flag_rate_bullet` key; the schema field defaults to `None`; chip suppresses. **Different mechanism than a 0.0 value, but both produce "no chip" in some edge cases — verify the chip rendering distinguishes them.**

**Why it happens:** The chip gate is `card.net_flag_rate_percentile != null`. A `0.0` percentile is not null — chip renders. A `None` percentile (below floor or not yet computed) is null — chip suppresses. This is the correct semantic.

**How to avoid:** Test fixtures cover both cases. Confirm via Plan F's Vitest assertions:
- `percentile=0`  → chip renders with "Top 0%" (clamped to MIN_TOP_PERCENT=1 → "Top 1%"). Actually verify rendering at the floor.
- `percentile=null` → chip absent.

For lower_is_better direction: `percentile=0` → "Top 0%" (clamped) = best possible; `percentile=99` → "Top 99%" = worst possible.

**Warning signs:** A test that conflates "no data" with "perfectly average data."

### Pitfall 8: Classical TC cohort too thin for stable p1..p99

**What goes wrong:** Per-TC bullet cohort is ~600 users; per-TC classical cohort is ~120 users. The CDF for `clock_gap_classical` is constructed over 120 deduped users; p1 and p99 are sensitive to single outliers; a user's percentile for clock_gap in classical reads as wildly different from their bullet/blitz/rapid percentile even though their underlying clock-management skill is similar.

**Why it happens:** SEED-025 estimates ~400–600 / TC but classical specifically may be lower because classical players are rarer in the benchmark cohort.

**How to avoid:** Plan A's candidacy table includes the per-cell `n_users` column. If any per-TC cohort drops below ~200, surface as a planning concern (not a blocker — D-6 accepts tail-SE widening as documented trade-off). Plan B's CDF generator runs without minimum-N guards; the chip suppresses naturally if a user is below the per-TC ≥30 floor. The combination is: small CDF cohort → noisy breakpoints → user percentile interpretation is loose; same as 94.2's "rating coupling accepted as honest" stance.

**Warning signs:** Plan A's classical cells show n_users < 200 across all three metric families. If so, consider per-card chip suppression at classical (drop only the classical chips) as a Plan F fallback — but this is a planning decision, not a research recommendation.

### Pitfall 9: Stage A vs Stage B race on the 12 new metrics

**What goes wrong:** Phase 94.2 D-10 / Pitfall 2 documented the disjoint-metric race: Stage A owns `score_gap` only; Stage B owns the other 3 ΔES metrics. Phase 94.3 keeps this contract — all 12 new metrics are Stage B (per D-10). But if a future plan accidentally moves Clock Gap to Stage A "because it's outcome-only," the two stages can race on the same `(user_id, metric)` PK.

**Why it happens:** The Net Flag Rate metric is genuinely outcome-only (doesn't need Stockfish eval). SEED-025 §Phase 94.2 Integration acknowledges this but explicitly bundles all 12 into Stage B to "avoid a special-case hook." Plan D's task must preserve that bundling.

**How to avoid:** Plan D's `STAGE_B_METRICS` tuple extension covers all 12. Plan D's test asserts `STAGE_A_METRIC == "score_gap"` (unchanged) and the 12 new metrics are NOT in `STAGE_A_METRIC`. No 94.3 changes touch the Stage A trigger site in `import_service.py`.

**Warning signs:** A Stage A trigger fires for `net_flag_rate_bullet` (logged in Sentry).

### Pitfall 10: TypeScript regen timing

**What goes wrong:** The frontend types in `frontend/src/types/endgames.ts` are auto-regenerated from FastAPI OpenAPI. If Plan D's schema change (adding 3 nullable fields to `TimePressureTcCard`) lands after Plan F's frontend code that references those fields, TypeScript compile fails on the frontend branch.

**Why it happens:** The frontend Plan F is in Wave 2; backend schema Plan D is in Wave 1. The dependency chain is correct on paper but the planner must invoke the type-regen script as part of Plan D's task list, OR include it as a Wave 2 sequencing checkpoint.

**How to avoid:** Plan D task list ends with: "Run the type-codegen script (whatever it's called — check `frontend/package.json` for the script name; commonly `npm run generate-types` or `npm run codegen`). Commit the regenerated `frontend/src/types/endgames.ts`." Plan F task list opens with: "Confirm the new fields exist on `TimePressureTcCard` in `frontend/src/types/endgames.ts` before proceeding."

**Warning signs:** `Property 'clock_gap_percentile' does not exist on type 'TimePressureTcCard'` during Plan F's TypeScript compile.

### Verified — Phase 94.2 _compute_metric_for_user (signature unchanged)

```python
# Source: app/services/user_benchmark_percentiles_service.py lines 88-124 (verbatim).

async def _compute_metric_for_user(
    session: AsyncSession,
    user_id: int,
    metric_id: CdfMetricId,                     # ◀── widens to 16-value Literal
) -> tuple[float, int] | None:
    su_cte = selected_users_cte(source="single_user")
    pooled_cte = per_user_cte_for(metric_id, source="single_user", snapshot_date=None)
    sql = (
        f"WITH {su_cte},\n{pooled_cte}\n"
        f"SELECT metric_value::float AS value, n_games::int AS n_games\n"
        f"FROM per_user_values\n"
        f"LIMIT 1"
    )
    result = await session.execute(text(sql).bindparams(user_id=user_id))
    row = result.fetchone()
    if row is None or row.value is None or row.n_games is None:
        return None
    return (float(row.value), int(row.n_games))
```

`per_user_cte_for` is the only changed surface — Plan B widens its dispatcher to handle 12 new metric IDs by routing to the 3 new builder families.

   - What's unclear: The actual classical TC pool size with the per-pressure-cell ≥30 floor applied (the most stringent floor).
   - Recommendation: Plan A's candidacy table captures `n_users` per cell. If classical drops below ~150 across all three families, the planner may consider suppressing classical chips entirely (a Plan F escape hatch) — but the design proceeds without that guard absent evidence.

### Known Threat Patterns for the per-TC builders

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SQL injection via `tc` parameter | Tampering | `tc: Literal["bullet", "blitz", "rapid", "classical"]` — Pydantic + ty enforce the literal set at every call site. F-string into SQL is safe because the type system constrains the value space. |
| SQL injection via `user_id` parameter | Tampering | SQLAlchemy `text().bindparams()` + explicit `CAST(:user_id AS int)` form (94.1 Plan 09 fix; preserved by 94.2; preserved by 94.3). |
| Unintended cross-DB writes from backfill script | Tampering | `--target dev` rejects non-`:5432` URLs; `--target prod` rejects non-`:15432` URLs and requires the tunnel socket to be open. Unchanged from 94.1. |
| Benchmark CDF script accidentally hitting prod | Tampering | `_assert_benchmark_db(url)` requires BOTH `'flawchess_benchmark'` AND `:5433` in the resolved URL. Unchanged from 94.1. |
| ENUM value injection via API | Tampering | All 12 new metric IDs are produced by backend code, never accepted from user input. The `fetch_for_user` repository query filters by `user_id` only; metric is not a query parameter. |
