### [Documentation deviation] Frontend type "regeneration" is a manual edit

- **Found during:** Task 2 implementation.
- **Issue:** The plan and RESEARCH §Pitfall 10 describe running an OpenAPI codegen script (`scripts/generate_openapi_schema.py` + `cd frontend && npm run codegen`) to regenerate `frontend/src/types/endgames.ts`. Neither script exists in this repo — `frontend/package.json` has no `codegen` / `generate-types` script, and there is no Python script under `scripts/` for OpenAPI dump.
- **Fix:** Manually extended `TimePressureTcCard` in the existing hand-maintained types file with 3 new fields, mirroring the Pydantic schema verbatim. Declared as `?: number | null` (optional + nullable) because:
  1. The existing hand-maintained file already uses this idiom for back-compat fields (e.g. it doesn't have all 5 Plan 88-14 fields as `required`).
  2. Pre-94.3 server responses won't emit the keys, so making them required would break frontend builds against an unmigrated backend during the Wave-1 → Plan F transition window.
  3. The Pydantic `= None` default + nullable type maps cleanly to optional-and-nullable in TS.
- **Files modified:** `frontend/src/types/endgames.ts`.
- **Why this is documentation-only:** the file ends up in the same shape the plan wants (3 new fields, frontend tsc clean). The discrepancy is process — the plan assumed codegen, the repo uses manual maintenance. Flagged so a future phase can either (a) build the codegen pipeline or (b) update the RESEARCH §Pitfall 10 stance to match the current manual workflow.
