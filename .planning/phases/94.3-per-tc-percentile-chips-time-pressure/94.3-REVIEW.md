# Phase 94.3: Code Review Report

**Reviewed:** 2026-05-24T18:00:00Z
**Depth:** standard
**Files Reviewed:** 12
**Status:** issues_found

## Summary

The Phase 94.3 widening of the percentile-chip system (4 → 16 metric flavors,
including the new per-(metric × TC) Time Pressure family) is broadly well
constructed: SQL builders use a small, type-constrained `TimeControlBucket`
Literal so f-string interpolation is safe, the ENUM extension follows the
documented `ADD VALUE IF NOT EXISTS` pattern, and the front-end carries
`satisfies Record<...>` constraints on the 16-key dispatch tables so a future
flavor addition cannot silently miss a copy / direction entry.

However, **the `lower_is_better` direction axis on `net_flag_rate_*` chips is
implemented incorrectly in the popover body** — bullet 1 contradicts the new
"Lower is better" preamble (BLOCKER CR-01), telling top-percentile flaggers
they are "better than 1%" of the cohort. Several other issues are quality /
maintainability defects (backfill script unintentionally recomputes all 15
Stage B metrics 15 times per user; subtle invariant differences between the
new per-TC `net_flag_rate` denominator and the existing card-level
`net_timeout_rate` denominator that should be exercised in tests).

## Critical Issues

### CR-01: `lower_is_better` chips show contradictory "better than X%" copy in popover bullet 1

**File:** `frontend/src/components/charts/PercentileChip.tsx:313-319`

**Issue:** For a chip whose `direction` is `lower_is_better` (only the 4
`net_flag_rate_{tc}` flavors today), the popover body renders TWO sentences
about how the user compares to the cohort. Both consume `percentile` but
only one inverts:

```ts
const lowerBetterLine = isLowerBetter && tc !== undefined
  ? `Lower is better — you have fewer net timeouts than ${formatBetterThanPercent(100 - percentile)} of ${tc} players.`
  : null;
const bullet1 = tc !== undefined
  ? `Your ${metricLabel} is better than ${formatBetterThanPercent(percentile)} of benchmarked Lichess players in ${tc}, all ratings.`
  : ...;
```

For a top-flag-resistance user at percentile `5` (their net flag rate sits
lower than 95% of the cohort = excellent), the popover renders:

1. *"Lower is better — you have fewer net timeouts than 95% of bullet players."*
2. *"Your Net Flag Rate (bullet) is better than 5% of benchmarked Lichess players in bullet, all ratings."*

The two sentences contradict each other. Worse, the chip face says `Top 5%`
(via `formatTopXPercent` which correctly uses raw `percentile` for the
lower-better branch), so a careful reader gets the right signal from the
chip and the wrong signal from bullet 1. Per Phase 94.3 D-2 / D-3 the
direction axis must flip the *formatter, band color, AND flame trigger* — it
must also flip the bullet-1 framing.

**Fix:** Mirror the inversion already applied in `lowerBetterLine`. Compute
the displayed comparison percent once at the top of the body and reuse it:

```ts
// Phase 94.3 D-2: same "fraction of cohort you outrank" number is consumed
// by both lowerBetterLine and bullet1. For lower_is_better, that's
// (100 - percentile) — a chip at p5 outranks 95% of the cohort.
const cohortOutranked = isLowerBetter ? 100 - percentile : percentile;
const lowerBetterLine = isLowerBetter && tc !== undefined
  ? `Lower is better — you have fewer net timeouts than ${formatBetterThanPercent(cohortOutranked)} of ${tc} players.`
  : null;
const bullet1 = tc !== undefined
  ? `Your ${metricLabel} is better than ${formatBetterThanPercent(cohortOutranked)} of benchmarked Lichess players in ${tc}, all ratings.`
  : `Your ${metricLabel} is better than ${formatBetterThanPercent(cohortOutranked)} of benchmarked Lichess players of all ELO ratings and time controls.`;
```

Then add a regression test in the PercentileChip test file:

```ts
it('inverts bullet 1 percentage on lower_is_better flavors', () => {
  render(<PercentileChip percentile={5} flavor="net_flag_rate_bullet" metricLabel="Net Flag Rate (bullet)" testId="t" />);
  fireEvent.mouseEnter(screen.getByTestId('t'));
  // chip face: Top 5% — should also read as "better than 95%" in the popover
  expect(screen.getByText(/better than 95%/)).toBeInTheDocument();
  expect(screen.queryByText(/better than 5%/)).not.toBeInTheDocument();
});
```

## Warnings

### WR-01: `_backfill_user` runs `compute_stage_b` 15 times per user, recomputing all 15 metrics each time

**File:** `scripts/backfill_user_percentiles.py:392-410` and `413-446`

**Issue:** `_backfill_user` loops over each metric in `STAGE_B_METRICS` and
calls `_compute_and_count(stage="B", metric=metric_id, ...)` per metric.
`_compute_and_count`'s `stage == "B"` branch unconditionally calls
`compute_stage_b(user_id, ...)`, which itself iterates over ALL 15 Stage B
metrics inside a fresh session, running the full pooled CTE for each and
UPSERTing. Net effect: every user pays `15 × 15 = 225` CTE executions
instead of `15` for Stage B alone, plus 15 UPSERTs per loop iteration that
re-touch every row's `computed_at` (and Sentry capture each time on any
metric that errors).

This is correctness-safe (UPSERTs are idempotent) but a major performance
regression vs. the pre-94.3 (4-metric) shape, and it amplifies any
transient Sentry-noise from a single metric by 15×. The stale comment in
`_compute_and_count` (`# Stage B computes all 3 metrics`) suggests the
amplification was unnoticed during the widening.

**Fix:** Compute Stage B once per user, then probe the row table for each
metric to classify outcomes:

```python
# In _backfill_user:
if metric_filter is None or metric_filter in STAGE_B_METRICS:
    if pending_evals > 0:
        for metric_id in STAGE_B_METRICS:
            if metric_filter is not None and metric_filter != metric_id:
                continue
            summary[metric_id].skipped_no_eval += 1
    else:
        # Compute all Stage B metrics ONCE.
        await compute_stage_b(user_id, session_maker=backfill_session_maker)
        # Then classify per metric via row probe.
        for metric_id in STAGE_B_METRICS:
            if metric_filter is not None and metric_filter != metric_id:
                continue
            if await _row_exists(user_id, metric_id, backfill_session_maker):
                summary[metric_id].upserted += 1
            else:
                summary[metric_id].skipped_below_pooled_floor += 1
```

Then refactor `_compute_and_count` accordingly (it can stay for Stage A
which has only one metric; the Stage B branch becomes dead and should be
deleted, not commented).

### WR-02: New per-TC `net_flag_rate` denominator silently differs from the card's `net_timeout_rate` denominator

**File:** `app/services/canonical_slice_sql.py:753-786` vs `app/services/endgame_service.py:1700-1731,1994-1998`

**Issue:** Two pieces of code claim to compute the "same" net flag rate metric
but their cohorts differ in a way the per-TC chip's promised invariant
("better than X% of bullet players on the *same* metric") does not survive.

Per-TC chip CDF (`per_user_cte_net_flag_rate`):
- Denominator: `count(*) FROM joined` where `joined` requires
  `JOIN endgame_entry_clocks ee ON ee.game_id = rc.id` — implicitly drops
  games that reached an endgame but had NO `clock_seconds` recorded for
  either parity at the entry plies.
- Floor: HAVING `count(*) >= 30` on that filtered count.

Card-level `net_timeout_rate` displayed alongside the chip
(`_iterate_clock_rows`):
- Denominator: `tc_total[tc]`, incremented for every row with
  `base_time_seconds > 0` regardless of whether `_extract_entry_clocks`
  returned non-None clocks. The Plan 88-14 docstring at line 1471-1474 is
  explicit: "total_games: count of all rows in this TC that pass the
  base-clock guard (matches tc_total[tc] semantics)."

Result: for a user with patchy clock data, the chip percentile is computed
against `(timeout_wins - timeout_losses) / endgame_games_with_recorded_clock`,
but the number shown in the same card is computed against
`(timeout_wins - timeout_losses) / all_endgame_games_for_TC`. A user reading
"My card says +3% and the chip says Top 5%" may not be comparing
comparable numbers. The two denominators only match when every endgame
game has clock data.

This is not a SQL injection / data loss bug, but it's a wire-format
correctness gap that contradicts the chip's promise of comparing the user
against the cohort on the value the card displays.

**Fix:** Pick one denominator and use it everywhere. The cleanest options:

1. Drop the `JOIN endgame_entry_clocks ee` from the SQL `joined` CTE for
   `net_flag_rate` — the metric is termination-driven and does not need the
   per-game entry clock at all. Then both sides count "all endgame games
   for this TC". (Symmetric with the Python `tc_total` count.)

2. Or apply the same `endgame_entry_clocks` filter to the Python
   `net_timeout_rate` computation (replace `tc_total[tc]` with
   `tc_clock_agg[tc].clock_games` for the denominator).

(1) is simpler and avoids surprising a user with patchy clocks. Add a
fixture test that builds a user with mixed clock-present / clock-absent
endgame games and asserts both denominators match.

### WR-03: `time_pressure_score_gap` per-TC CDF silently drops rows with one-sided NULL clocks

**File:** `app/services/canonical_slice_sql.py:644-656`

**Issue:** The `per_user` aggregate inside `per_user_cte_time_pressure_score_gap`
is wrapped in:

```sql
WHERE user_clock_pct IS NOT NULL AND opp_clock_pct IS NOT NULL
GROUP BY user_id
HAVING ... user_pressured_n >= 30 AND opp_pressured_n >= 30
```

If a game has only one side's clock recorded (e.g. the user clock at entry
is non-null but the opponent's first odd-ply clock is missing — the
`_endgame_entry_clocks_cte` derivation per-parity is independent), the
entire row is dropped from both filters even though the user-side aggregate
could legitimately include it. With the SQL `FILTER (WHERE ... < 0.40)`
semantic, a NULL would already be filtered out of the FILTER counts, so
the only practical effect of the additional `WHERE` is to drop the row
from the OPPONENT count as well.

The result: a user whose opponents are systematically less likely to have
clock metadata recorded will see a tighter `opp_pressured_n` floor than
expected and the chip will silently suppress more often than the
"Stage B computed → chip shows" promise implies. This is more of a
documentation/observability gap than a logic bug — but the asymmetry
should at minimum be noted in the docstring or removed.

**Fix:** Either:

1. Drop the `WHERE user_clock_pct IS NOT NULL AND opp_clock_pct IS NOT NULL`
   line. The `FILTER (WHERE x < 0.40)` clauses already treat NULL correctly
   (NULL < 0.40 is NULL → row not counted in the FILTER aggregate).
2. Or document the dual-NULL gate in the docstring so the inclusion-floor
   semantics are explicit to future readers comparing against Stage A
   `score_gap` (which has no such restriction).

### WR-04: Migration uses string formatting (low-risk hardcoded constants)

**File:** `alembic/versions/20260524_170733_fd5b551f381c_extend_benchmark_metric_for_tc_pressure.py:70-71`

**Issue:** `op.execute(f"ALTER TYPE benchmark_metric ADD VALUE IF NOT EXISTS '{value}'")`
uses f-string interpolation of a value into raw SQL. The values come from
the module-level `_NEW_VALUES` tuple of hardcoded literals (no user input),
so there is no current injection vector. However, future maintenance that
adds a value with a single-quote character — e.g. `"o'malley_flag_rate"` —
would break the migration silently mid-`upgrade()` and leave the ENUM in
a partially-extended state (some values added, others not). The migration
has no rollback for that state.

This is unlikely in practice (ENUM values for this codebase are always
`snake_case` ASCII), but worth a defensive guard.

**Fix:** Validate or escape the value before composing the SQL:

```python
import re
_SAFE_ENUM_VALUE = re.compile(r"^[a-z_][a-z0-9_]*$")
for value in _NEW_VALUES:
    if not _SAFE_ENUM_VALUE.match(value):
        raise ValueError(f"Unsafe ENUM value would require escaping: {value!r}")
    op.execute(f"ALTER TYPE benchmark_metric ADD VALUE IF NOT EXISTS '{value}'")
```

PostgreSQL ENUM values cannot be parameterised through bindparams in the
DDL path, so a static regex guard is the cleanest defensive fix.

### IN-02: `tcFromFlavor` linear scan is fine but worth a micro-test

**File:** `frontend/src/components/charts/PercentileChip.tsx:242-248`

**Issue:** `tcFromFlavor` iterates the 4 TC suffixes and matches against `endsWith('_${tc}')`. The original 4 kebab-case flavors (`score-gap`, `achievable`, `parity`, `conversion`) cannot collide with the `_${tc}` pattern because they contain no underscore at all (or, in the case of `score-gap`, use a hyphen). Stay safe by asserting this in a test:

```ts
it('returns undefined for all pre-94.3 flavors', () => {
  for (const f of ['score-gap', 'achievable', 'parity', 'conversion'] as const) {
    expect(tcFromFlavor(f)).toBeUndefined();
  }
});
```

### IN-03: `_endgame_entry_clocks_cte` re-computes the 6-ply endgame gate in SQL alongside the Python version

**File:** `app/services/canonical_slice_sql.py:572-585`

**Issue:** The CTE applies `HAVING count(gp.ply) >= 6` for endgame eligibility — replicating the threshold encoded in `ENDGAME_PLY_THRESHOLD` elsewhere as a Python constant. If that threshold ever moves, the SQL CTE will silently disagree with the Python aggregators. Consider hoisting the integer into a Python constant in this module (`_ENDGAME_PLY_THRESHOLD: int = 6`) and threading it through the f-string, so a single grep covers both call sites.
