## Summary

This phase replaces `games.move_count` (full-move count, always `(len(nodes)+1)//2`) with
`games.ply_count` (exact half-move count, `len(nodes)`). The motivation is eliminating the
±1 ambiguity in max ply: a game ending on White's move has `move_count = n` but `max(ply) = 2n - 1`;
a game ending on Black's move has `move_count = n` and `max(ply) = 2n`. Storing the exact half-move
count enables Phase 115's per-color user-move denominator (`floor(ply_count/2)` White,
`ceil(ply_count/2)` Black) without scanning `game_positions`.

The migration follows the project's standard `add nullable → external backfill → enforce NOT NULL`
pattern. The backfill is a standalone script mirroring `scripts/backfill_eval.py`. All readers
(4 backend sites + 3 frontend cards) replace `move_count` with `ply_count`, with `Math.ceil(ply_count/2)`
in the frontend for display. The chapter5.py benchmark drops its 87M-row `game_positions` scan.

**Primary recommendation:** Mirror `backfill_eval.py`'s game_id-range chunked UPDATE pattern
(10,000-game chunks, idempotent WHERE clause, sequential commits). Use `op.alter_column` with
`nullable=False` for Migration B, then `op.drop_column("games", "move_count")`.

### Backfill SQL shape

The recommended UPDATE is a set-based correlated batch over a game_id range (mirrors
`phase_update_sql` in `backfill_eval.py` L622-666):

```sql
UPDATE games g
SET ply_count = sub.max_ply
FROM (
    SELECT game_id, MAX(ply) AS max_ply
    FROM game_positions
    WHERE game_id BETWEEN :lo AND :hi
    GROUP BY game_id
) AS sub
WHERE g.id = sub.game_id
  AND g.ply_count IS NULL
```

The `AND g.ply_count IS NULL` clause is the idempotency guard — safe to re-run.

### Batching strategy

Batch by `game_id` range (same as `PHASE_BACKFILL_CHUNK_SIZE` in `backfill_eval.py`).
Recommended chunk size: **10,000 games** (same as `PHASE_BACKFILL_CHUNK_SIZE = 10_000`).

Loop shape (mirroring `backfill_eval.py` L690-700):

```python
PLY_BACKFILL_CHUNK_SIZE = 10_000

async with session_maker() as session:
    bounds = (await session.execute(text(
        "SELECT COALESCE(MIN(id), 0), COALESCE(MAX(id), 0) FROM games WHERE ply_count IS NULL"
    ))).one()
    lo_total, hi_total = bounds
    if hi_total == 0:
        _log("No rows to backfill (ply_count IS NULL = 0, no-op)")
    else:
        cursor = lo_total
        updated_total = 0
        while cursor <= hi_total:
            chunk_hi = cursor + PLY_BACKFILL_CHUNK_SIZE - 1
            result = await session.execute(ply_update_sql, {"lo": cursor, "hi": chunk_hi})
            updated_total += result.rowcount or 0
            await session.commit()
            _log(f"  chunk [{cursor}, {chunk_hi}]: {result.rowcount} rows updated (total={updated_total})")
            cursor = chunk_hi + 1
```

### Why game_id range (not position-row range)

`game_positions` has 190M rows; iterating by position `id` range requires an expensive range
scan of a 44 GB table. Iterating by `games.id` range lets PostgreSQL use the composite
`(game_id, user_id, ply)` PK index on `game_positions` for the `WHERE game_id BETWEEN :lo AND :hi`
predicate. Each chunk touches at most `10,000 × avg_plies` positions but only writes 10,000
`games` rows.

### Resumability

Re-running after a mid-run kill: the outer loop queries `MIN(id) WHERE ply_count IS NULL`
before starting, so it automatically skips already-backfilled rows. The inner UPDATE's
`AND g.ply_count IS NULL` guard makes each chunk idempotent.

### Sentry in backfill script

Follow `backfill_eval.py` L883-884: `sentry_sdk.init(dsn=settings.SENTRY_DSN, environment=settings.ENVIRONMENT)`
in `main()`. No per-row Sentry capture needed for a pure UPDATE script (no engine errors).

### VACUUM at end

After the UPDATE loop, run `VACUUM ANALYZE games` in autocommit mode (same as `backfill_eval.py`
L800-806 for `game_positions`). This updates planner statistics for the new `ply_count` column.

## Common Pitfalls

### Pitfall 1: `op.alter_column` NOT NULL enforcement

**What goes wrong:** Running `op.alter_column("games", "ply_count", nullable=False)` when any
row still has `ply_count IS NULL` raises `constraint violation` and aborts the migration.

**How to avoid:** The HUMAN gate (verify count = 0 on prod before deploying Migration B) plus
dev verification after the dev backfill.

### Pitfall 2: Frontend type inference

**What goes wrong:** TypeScript's `noUncheckedIndexedAccess` is enabled. If the frontend
renders `Math.ceil(game.ply_count / 2)` inline without a null guard, TypeScript flags it
(`ply_count: number | null`).

**How to avoid:** Keep the `game.ply_count !== null && (...)` guard that currently wraps
`move_count` — same pattern, same location, same null guard.

### Pitfall 3: Test fixture numeric values

**What goes wrong:** Renaming `move_count: 40` to `ply_count: 40` in test fixtures would
silently halve the displayed move count (`Math.ceil(40/2) = 20` instead of `Math.ceil(80/2) = 40`).

**How to avoid:** When renaming fixture values, update the number: `move_count: N` becomes
`ply_count: 2*N` (for even-ply games) or `2*N - 1` (odd-ply games). The `GameCard.test.tsx`
has `move_count: 40` → should become `ply_count: 80`.

### Pitfall 4: `_PositionRowsResult.move_counts` rename propagation

**What goes wrong:** The field rename from `move_counts` to `ply_counts` in `_PositionRowsResult`
must propagate through every reference in `import_service.py` (L102, L756, L768, L860) AND
in tests (L1420-1422, L2879-2913 etc.).

**How to avoid:** Grep for `move_counts` (plural) in `import_service.py` and tests before
closing the task.

### Pitfall 5: `GameProcessingResult` TypedDict key

**What goes wrong:** The TypedDict key `move_count` (L67) and all callers
that access `processing_result["move_count"]` must rename together. A partial rename
causes a `KeyError` at import time (TypedDict is not a class with `__getattr__`).

**How to avoid:** Rename key in TypedDict + all dict literal constructions
(`tests/services/test_import_service.py` L100, L152; `tests/test_import_service.py` L74) simultaneously.

### Pitfall 6: LibraryGameCard.tsx renders `{game.move_count} Moves` (not `{game.move_count} · Moves`)

`LibraryGameCard.tsx` (L351) renders `{game.move_count} Moves` as a single inline string,
while `GameCard.tsx` (L160-162) and `FlawCard.tsx` (L229-231) render the count + a
`<span className="hidden sm:inline">` for the "Moves" label separately. Both patterns need
to derive `Math.ceil(ply_count / 2)` before rendering.
