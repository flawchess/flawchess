# Phase 114.1: Replace `move_count` with exact `ply_count` - Context

**Gathered:** 2026-06-10
**Status:** Ready for planning
**Source:** Orchestrator-synthesized from SEED-041 §9 (user authorized "skip discussion, make decisions yourself" via `/gsd-plan-phase 114.1 --chain`)

<domain>
## Phase Boundary

`games.move_count` is the FULL-move count (`(len(nodes)+1)//2` in `zobrist.py`), so it
pins the half-move total only to ±1 (`max_ply ∈ {2·move_count, 2·move_count−1}`). That
ambiguity blocks an exact per-game **user move count** denominator without scanning the
190M-row / 44 GB `game_positions` table. This phase replaces `move_count` with an exact
half-move `ply_count` so Phase 115's live "you vs opponent" endpoint and the §5 benchmark
chapter get an exact per-game denominator with ZERO `game_positions` access.

**In scope:** ONE Alembic migration (add nullable + backfill + drop `move_count`, see D-02);
import-path write; all backend readers + API schemas; frontend types + the 3 game cards; test
fixtures; chapter5.py simplification + §5 regen.

**Out of scope:** Phase 115's comparison endpoint/UI itself. Any new index. Reworking the
benchmark beyond the §5 `user_moves_per_game` simplification.
</domain>

<decisions>
## Implementation Decisions

### D-01 — API exposes raw `ply_count`; frontend derives the display
Expose raw `ply_count: int` in the API schemas (`app/schemas/openings.py`,
`app/schemas/library.py`) and REMOVE `move_count` from API responses. The frontend derives
the unchanged "N Moves" full-move label as `Math.ceil(ply_count / 2)` (= `(ply_count+1)//2`).
Rationale: Phase 115's live endpoint needs the raw half-move count anyway; a single source
of truth (raw field + client-side derivation) avoids carrying two redundant fields.

### D-02 — Single Alembic migration: add + backfill + drop, in one transaction
**REVISED 2026-06-10 after measuring prod.** The original `add → backfill → enforce` split
(separate resumable script + HUMAN-gated enforce migration) was over-engineering. Measured on
prod (43.9M `game_positions` rows, 590k `games`): the full `SELECT game_id, max(ply) ...
GROUP BY game_id` aggregate runs in **3.2s at 33MB peak memory** (parallel seq scan +
HashAggregate, trivial disk spill); the write touches only ~590k `games` rows. The whole
backfill is a sub-10-second operation — NOT the import-pipeline memory-pressure failure mode
behind the prod OOM history (which was SQLAlchemy connection fanout + Python + Stockfish, not
bulk SQL). Alembic runs at container startup BEFORE uvicorn serves, so a single migration is
safe and has no NULL window.

ONE hand-written Alembic migration (chains from head `d9e3f1a4b5c6`), in a single transaction:
1. `ADD COLUMN ply_count INTEGER` **NULLABLE**.
2. `UPDATE games g SET ply_count = sub.mp FROM (SELECT game_id, max(ply) AS mp FROM
   game_positions GROUP BY game_id) sub WHERE g.id = sub.game_id` (plies 0-based contiguous ⇒
   `max(ply)` = exact half-move total).
3. `DROP COLUMN move_count`.

No standalone backfill script, no second "enforce" migration, no prod runbook, no HUMAN-gating.
On both dev and prod the migration runs the same way via `alembic upgrade head`. Adds ~10s to
the existing per-deploy migration downtime — acceptable and consistent with current behavior.

### D-02a — `ply_count` stays NULLABLE (mirror `move_count`)
Do NOT add a NOT NULL constraint. On prod 971 games have no `game_positions` rows and are
exactly the 971 where `move_count` is already NULL; after backfill those games keep `ply_count
= NULL`. Keeping the column nullable mirrors `move_count`'s current `int | None` shape, so
every reader's existing null-tolerance carries over unchanged and the position-less-game edge
case needs no special handling.

### D-03 — Readers switch directly to `ply_count`, ship in the SAME release as the migration
Backend readers use `ply_count` directly (still `int | None`, same nullability as `move_count`
today — no fallback, no `COALESCE`). Because the reader code and the migration ship together
and Alembic runs before the new app serves, the old schema (with `move_count`) is never seen
by new code and the new schema (without `move_count`) is never seen by old code. No NULL
window, no deploy-ordering runbook.

### D-04 — Import path writes the exact half-move count
`zobrist.py` (~L271): set `ply_count = len(nodes)` (half-moves), NOT `(len(nodes)+1)//2`.
`import_service.py` bulk UPDATE (~L738-770): write `ply_count`. New imports always carry
`ply_count`, so the backfill only ever needs to touch pre-existing rows.

### D-05 — Benchmark §5 follow-on uses the fast path
After ship, simplify `scripts/benchmarks/chapter5.py` `user_moves_per_game` to read
`games.ply_count` (drops its ~87M-row scan), then regenerate §5 via the chapter5 fast path
(`chapter5.compute()`, ~1-2 min), NOT the full ~33-min `gen_benchmarks`. Verify the §5
narrated prose + headline table after the basis change, not just a green diff-gate.

<specifics>
## Specific Ideas

- Backfill correctness identity: plies 0-based contiguous ⇒ `max(ply) = count(ply)` = exact
  half-move total. Verified on benchmark DB: `max_ply ∈ {2·move_count, 2·move_count−1}`.
- Per-color user moves once `ply_count` exists: `floor(ply_count/2)` for White,
  `ceil(ply_count/2)` for Black — no `game_positions` access. This is the Phase 115 payoff.
