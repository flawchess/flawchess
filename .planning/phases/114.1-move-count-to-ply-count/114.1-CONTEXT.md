# Phase 114.1: Replace `move_count` with exact `ply_count` - Context

**Gathered:** 2026-06-10

<domain>
## Phase Boundary

`games.move_count` is the FULL-move count (`(len(nodes)+1)//2` in `zobrist.py`), so it
pins the half-move total only to ±1 (`max_ply ∈ {2·move_count, 2·move_count−1}`). That
ambiguity blocks an exact per-game **user move count** denominator without scanning the
190M-row / 44 GB `game_positions` table. This phase replaces `move_count` with an exact
half-move `ply_count` so Phase 115's live "you vs opponent" endpoint and the §5 benchmark
chapter get an exact per-game denominator with ZERO `game_positions` access.

**In scope:** Alembic add-column + backfill script + enforce migration; import-path write;
all backend readers + API schemas; frontend types + the 3 game cards; test fixtures;
chapter5.py simplification + §5 regen.

**Out of scope:** Phase 115's comparison endpoint/UI itself. Any new index. Reworking the
benchmark beyond the §5 `user_moves_per_game` simplification.
</domain>

<decisions>
## Implementation Decisions

### D-01 — API exposes raw `ply_count`; frontend derives the display
Expose raw `ply_count: int` in the API schemas (`app/schemas/openings.py`,
`app/schemas/library.py`) and REMOVE `move_count` from API responses. The frontend derives
the unchanged "N Moves" full-move label as `Math.ceil(ply_count / 2)` (= `(ply_count+1)//2`).
Rationale: Phase 115's live endpoint needs the raw half-move count anyway; a single source
of truth (raw field + client-side derivation) avoids carrying two redundant fields.

### D-02 — Migration: `add → backfill → enforce`, backfill DECOUPLED from auto-migration
Prod auto-runs Alembic on container startup and has a documented OOM history with heavy
operations, so a 190M-row-touching backfill MUST NOT live inside an auto-running migration.

- **Migration A (cheap, auto-safe):** add `ply_count INT` NULLABLE. Nothing else.
- **Backfill via standalone resumable script** `scripts/backfill_ply_count.py`, mirroring
  `scripts/backfill_eval.py`'s structure (`--db dev|benchmark|prod`, batched, idempotent,
  resumable; prod via `bin/prod_db_tunnel.sh`). Populates `ply_count = max(ply)` from
  `game_positions` grouped per game. Plies are 0-based contiguous, so `max(ply)` = exact
  half-move total.
- **Migration B (enforce):** set `ply_count NOT NULL` + DROP `move_count`. This migration
  is authored in-phase and applied on dev after the dev backfill, but its **prod apply is
  HUMAN-gated**: prod must be fully backfilled (`SELECT count(*) FROM games WHERE ply_count
  IS NULL` = 0) BEFORE the deploy carrying Migration B. The plan documents this prod rollout
  as a runbook and flags prod steps HUMAN-UAT (do not gate dev verification on prod or on
  `bin/reset_db.sh`).

### D-03 — Readers switch directly to `ply_count` (no permanent fallback)
Backend readers use `ply_count` directly. On dev the in-phase backfill runs before reader
verification, so no NULL-tolerance fallback is needed. On prod the runbook orders backfill
before the reader deploy. The planner may add a SHORT-LIVED `COALESCE(ply_count, move_count*2)`
fallback only if it judges the prod deploy gap genuinely needs it; default is no fallback to
avoid dead code.

### D-04 — Import path writes the exact half-move count
`zobrist.py` (~L271): set `ply_count = len(nodes)` (half-moves), NOT `(len(nodes)+1)//2`.
`import_service.py` bulk UPDATE (~L738-770): write `ply_count`. New imports always carry
`ply_count`, so the backfill only ever needs to touch pre-existing rows.

### D-05 — Benchmark §5 follow-on uses the fast path
After ship, simplify `scripts/benchmarks/chapter5.py` `user_moves_per_game` to read
`games.ply_count` (drops its ~87M-row scan), then regenerate §5 via the chapter5 fast path
(`chapter5.compute()`, ~1-2 min), NOT the full ~33-min `gen_benchmarks`. Verify the §5
narrated prose + headline table after the basis change, not just a green diff-gate.

### Claude's Discretion
- Exact Alembic batching loop shape in the backfill script (batch size, game-id ranging).
- Whether to set `ply_count NOT NULL` in Migration B or keep nullable + app-level guarantee.
- Frontend helper location for `(ply_count+1)//2` (shared util vs inline per card).
- Whether a temporary reader COALESCE fallback is warranted (see D-03).
</decisions>

  half-move total. Verified on benchmark DB: `max_ply ∈ {2·move_count, 2·move_count−1}`.

  `ceil(ply_count/2)` for Black — no `game_positions` access. This is the Phase 115 payoff.
