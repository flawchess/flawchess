# Phase 186 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase adds one new component (`ImportFilterCard`) plus per-TC budget-chip rows inside the two existing platform cards on `frontend/src/pages/Import.tsx`. It introduces **zero new visual primitives** — every element below is a reuse of an already-shipped shadcn/project component (`ToggleGroup`, `Select`/plain toggle-button grid, `Card`/`CardHeader`/`CardBody`, `Button`, `InfoPopover`, `Label`). This contract is deliberately narrow and prescriptive because the job is fitting new controls into an established system, not designing a new one.

**Visual anchor:** the per-platform "Sync" buttons remain the Import tab's primary visual anchor and only high-emphasis (brand-brown) elements; the new "Import filters" card is a secondary, lower-emphasis control block that sits above them and never competes for attention (neutral accent only, per the Color contract).

## Typography

This phase introduces **no new type sizes** — it reuses the global `text-sm` (14px) floor exclusively, matching every existing component it sits beside (`FilterPanel.tsx` labels, `Import.tsx` platform-row labels). No Display-scale or larger Heading-scale text is needed for a filter-card scope.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 — inline helper copy, budget-chip text, filter section captions |
| Label | 14px (`text-sm`) | 400 (regular) | 1.4 — matches `FilterPanel`'s existing unweighted `text-muted-foreground` field captions ("Time controls", "Backlog cap") |
| Heading | 14px (`text-sm`) | 600 (semibold) | 1.2 — the "Import filters" card title, rendered via `CardHeader size="compact"` (already `font-semibold text-sm` by default — no override needed) |
| Display | n/a | n/a | n/a — not used; no hero/large text in this phase's scope |

Two weights total: **400** (regular — body/labels/chip numbers) and **600** (semibold — card title only, and the "full/complete" budget state per Color contract below). No third weight introduced.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` / `charcoal-texture` card surface | Page background, the filter card's and platform cards' textured surface (unchanged, inherited) |
| Secondary (30%) | `--border`, `--muted`, `--muted-foreground` | Card borders (`border-border/40`), section-caption text, in-progress budget-chip text |
| Accent (10%) | `--toggle-active` (neutral gray, `oklch(0.55 0 0)` — **not** brand-brown) | Reserved exclusively for: the selected/active state of the TC multiselect buttons and the cap single-select buttons. This matches the existing `FilterPanel.tsx` time-control/platform toggle convention and the shadcn `ToggleGroup` `data-[state=on]`/`aria-pressed` styling verbatim — brand-brown is deliberately NOT used here because it's reserved project-wide for primary CTAs (Sync, Explore buttons), which this phase does not add or restyle. |
| Destructive | `--destructive` | Reserved for PATCH-failure inline error text only (the auto-save-on-toggle mutation failing per the Copywriting Contract's Error state row) — **never** for at-cap/over-cap budget display. Being at or over a backlog cap is an accepted, benign state (D-12: "existing games are never deleted") and must NOT read as an error/danger state; see "full" styling below. |

Accent reserved for: the selected-state fill/border of TC toggle buttons and the cap toggle buttons — nothing else in this phase gets the accent treatment (no accent-colored budget chips, no accent-colored popover trigger — `InfoPopover`'s existing muted-brown hover tint is unchanged, out of scope).

**"Full" budget state (D-11/D-12), explicit default (no destructive red):** a budget chip that has reached or exceeded its cap renders in `text-foreground font-semibold` (600 weight, full-contrast) instead of the in-progress `text-muted-foreground font-normal` (400 weight, secondary contrast) used for chips still filling. This is a **weight/contrast** distinction, not a color-alarm distinction — "full" should read as "complete," matching D-11's explicit requirement ("Full budgets read as complete, not broken"). Over-cap grandfathered numbers (e.g. "Blitz 8000/5000") use the identical "full" treatment — same weight/contrast, no additional red/destructive styling, since D-12 requires the honest count to read reassuringly, not as an error.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | **No new CTA in this phase.** The existing per-platform "Sync" button (`Button` default variant, unchanged) remains the sole trigger per D-03/D-09 — settings auto-save on toggle, nothing runs until Sync is clicked. |
| Empty state heading | n/a — filter controls are always populated with defaults (all-except-bullet + cap 1000, or grandfathered all-TC + 5000); there is no zero-configuration empty state to design. |
| Empty state body | Budget chip before any backlog import: render the plain count `"0/1000"` (or the user's configured cap) — no special "not started yet" copy. Avoids duplicating the platform row's own `"{N} games"` header, which already communicates zero. |
| Error state | New `useImportSettings` query (GET) failure: **"Failed to load import settings. Something went wrong. Please try again in a moment."** — per CLAUDE.md's mandatory `isError` branch rule; never let a settings-fetch failure silently fall back to defaults (that would misrepresent the user's real saved preferences). A PATCH (auto-save-on-toggle) failure should surface as a small inline error near the toggled control (e.g. a `text-destructive text-sm` line below the row) rather than a blocking modal — the toggle itself should revert to its prior state on failure (optimistic-update rollback). |
| Destructive confirmation | **No new destructive action in this phase.** Settings changes are explicitly NOT destructive (auto-save, no confirmation, no dirty state — D-09); the existing "Delete All Games" confirmation dialog is unchanged and out of scope. |
| Locked inline helper copy (D-10) | **"Limits how much older history is imported — new games always sync."** — one line, `text-sm text-muted-foreground`, directly under the filter controls, verbatim per CONTEXT.md. |
| Info popover body copy (D-10, full rule) | Draft, following the project's popover-copy-minimalism convention (WHAT + sign convention only, no jargon, no caveats beyond what's asked): *"Each time control you enable gets its own {cap}-game backlog budget per platform. New games always import regardless of budget. Lowering a setting never deletes games you already have."* Keep to these three sentences — no mention of "backfill," "budget accounting," or internal mechanism names. |

## Component & Layout Notes (non-template, implementation guidance)

These are prescriptive notes for the planner/executor beyond the template's fixed sections, since this phase is 100% new-component-in-existing-page work:

1. **New "Import filters" card (D-08):** use the project's `Card`/`CardHeader size="compact"`/`CardBody` primitive (`frontend/src/components/ui/card.tsx`), title **"Import filters"**, placed in the existing `space-y-8` main column of `Import.tsx`, directly above the `space-y-4` wrapper holding the two platform rows (not inside it).
2. **TC multiselect row:** copy `FilterPanel.tsx`'s raw-`<button>` grid idiom verbatim (`grid grid-cols-4 gap-1`, `h-11 sm:h-7`, `border-toggle-active bg-toggle-active text-toggle-active-foreground` when active / `border-border bg-inactive-bg text-muted-foreground` when inactive, `aria-pressed`, `data-testid="filter-time-control-{tc}"` — reuse the **same** `data-testid` naming convention as the existing filters so automated tooling recognizes it, but scope it under the import-settings context, e.g. `import-filter-time-control-{tc}`, to avoid a duplicate-testid collision with the Library/Openings `FilterPanel` instance rendered elsewhere in the app). Reuse `TimeControlIcon` per chip, matching the existing row exactly.
3. **Cap single-select row:** use shadcn `ToggleGroup type="single"` (matching the `rated`/`opponentType` rows' idiom per RESEARCH.md's explicit recommendation), three items `1000` / `3000` / `5000`, `data-testid="import-filter-cap-{value}"`.
4. **Budget chips (D-11/D-12):** render inline within each existing platform row (`import-platform-chess-com` / `import-platform-lichess`), directly below the existing `"{N} games (last sync: …)"` line — e.g. `"Blitz 1000/1000 · Rapid 640/1000 · Classical 120/1000"` as one `flex flex-wrap gap-x-3 gap-y-1 text-sm` line, dot-separated, each TC's own number weight/contrast following the "full" rule above. `data-testid="import-budget-chip-{platform}-{tc}"` per chip.
5. **HelpCircle popover (D-10):** reuse `InfoPopover` (`frontend/src/components/ui/info-popover.tsx`) verbatim — do not build a new popover shell. Default `icon={HelpCircle}`, `side="top"`, body copy per the Copywriting Contract draft above (may use `text-xs` per the CLAUDE.md hover/tap-tooltip exception — `InfoPopover`'s content is already `text-xs`).
6. **Mobile parity (CLAUDE.md mandatory):** the TC/cap toggle grids and budget-chip rows must render identically on mobile — no separate mobile-only layout branch is needed since `FilterPanel`'s `h-11 sm:h-7` pattern already handles the touch-target sizing responsively within the same markup (unlike `FilterPanel`'s desktop-sidebar-vs-mobile-drawer split, this card is NOT duplicated in a drawer — it lives once, inline, above the platform cards, on both breakpoints).
7. **`data-testid` / ARIA:** every new interactive element needs a `data-testid` (kebab-case, `import-filter-*` prefix) and the TC/cap toggle buttons need `aria-pressed` (already the FilterPanel convention) — no icon-only buttons are introduced by this phase (the HelpCircle trigger already has `aria-label` built into `InfoPopover`).
