<objective>
Establish the per-user import-settings foundation and prove the settings → import-pipeline wiring end-to-end (the tracer): a `user_import_settings` table + grandfathering migration, a settings repository, GET/PATCH endpoints, a derived per-(platform,TC) backlog-count query, and the forward/incremental-sync TC filter that reads those settings.

This is the phase's tracer slice: DB → repository → API → service, wired so that toggling a TC off and running Sync causes the forward pass to stop importing that TC. The backward-fetch backfill machinery (the heavy new path) is deliberately deferred to Plan 02, which builds on this foundation.

Implements: IMPORT-01 (settings model + CRUD, cap semantics per D-01), IMPORT-02 (anchor at users.created_at, TC filter both directions per D-02, correspondence under classical per D-14, NULL-bucket bypass per D-15), IMPORT-04 (backlog counts for UI chips), D-13 (grandfathering), D-16 (guest/registered default parity — one code path).

Purpose: Every later plan (backward walk, frontend chips) depends on this settings row, its defaults, and the derived budget query existing and being correct.
Output: New table + migration, settings repo, settings schemas + endpoints, backlog-count repo function, forward-pass TC filter in import_service.
</objective>
