## Summary

Phase 131 is a precision-first rework of `app/services/tactic_detector.py`. The oracle is lichess-puzzler's `tagger/cook.py` (AGPL-3.0): because the CC0 puzzle fixture labels were generated by cook, the precision ceiling is cook-fidelity. The phase has three interlocking workstreams:

**Workstream A** ports eight cook predicates (skewer, discovered-attack, back-rank-mate, fork, pin, anastasia-mate, hook-mate, discovered-check) plus two shared utilities (`is_defended` ray-aware, `is_in_bad_spot`). Two of the worst motifs (skewer 0.15, discovered-attack 0.17) will be suppressed if full cook fidelity still leaves them below 0.9 on the TEST split — this is an endorsed trade since skewer alone removes ~16% of all tags (mostly false). Volume-weighted held-out precision across the top five firing motifs is ~0.31 today; roughly two of three chips a user sees are wrong.

**Dispatch rework (D-05/D-06/D-07)** inverts the current `(tier, rank, depth)` sort key to `(depth, tier, rank)` for non-mate tactics. This is structurally a ply-outer/detector-inner walk with early-exit rather than the current detector-outer collect-then-sort. Hanging-piece (depth 0) wins when shallowest; mate short-circuits unconditionally first but is now gated on the stored Stockfish mate-in-x score rather than `is_checkmate` at PV end (because PV_CAP_PLIES=12 truncates long forced mates).

**Workstream B** adds a dest-square guard at the call site in `flaws_service.py:424-434` (the missed pass): when the flaw move's destination square equals the best line's first-move destination square, suppress the missed tactic. This catches the wrong-recapture false alarm (player captured the same piece with the wrong piece type). The CC0 fixture cannot cover this bug — it requires hand-built `(flaw_move, best_line)` unit fixtures.

**Primary recommendation:** Port the two shared utilities first (they leak into six detectors), then rebuild each in-scope motif against cook's exact predicate, then invert the dispatch sort key, then add the Workstream B call-site guard. Run `tactic_tagger_report.py --check-goals` after each motif to track progress against the 0.9 TEST-split target.

This section maps the cook-alignment note's pseudocode against both the real cook.py source and our current detector. Every finding is confirmed by direct file read.

### Shared Utilities (D-08)

#### 1. `is_defended` — ray-aware (cook's `util.is_defended`)

cook's `util.is_defended` (util.py:72–85): returns True if `board.attackers(piece.color, square)` is non-empty **OR** any opponent ray attacker on that square can be removed (board copy, `bc.remove_piece_at(attacker)`) and the piece then has a friendly defender. This X-ray detection catches pieces defended only through a friendly ray piece in front.

Our `_is_hanging` (`tactic_detector.py:256–264`) checks `board.attackers(not color, sq)` non-empty AND `board.attackers(color, sq)` empty. It has no X-ray path. [VERIFIED: tactic_detector.py:256]

**Gap confirmed:** Our detector calls pieces "hanging" that cook considers ray-defended. This leaks false positives into: fork (victim hanging check at line 340), hanging-piece (line 370), skewer, pin, and interference consumers. All six consumers must be re-verified after porting.

**Port requirement:** Implement `_is_defended(board, piece, sq)` → bool with the board-copy X-ray path. The negation `not _is_defended(...)` replaces calls to `_is_hanging(...)` for pieces owned by the opponent.

Note: cook's `is_hanging(board, piece, square)` is simply `not is_defended(board, piece, square)`. Our `_is_hanging` is NOT equivalent — it lacks the X-ray defense path.

#### 2. `is_in_bad_spot` — entirely missing (cook's `util.is_in_bad_spot`)

cook's `util.is_in_bad_spot(board, square)` (util.py:98–103):
```
piece = board.piece_at(square)
return (bool(board.attackers(not piece.color, square)) and
        (is_hanging(board, piece, square) or can_be_taken_by_lower_piece(board, piece, square)))
```
Where `can_be_taken_by_lower_piece` checks any opponent attacker with `values[attacker.piece_type] < values[piece.piece_type]` and attacker is not king.

We have no equivalent. [VERIFIED: grep — no is_in_bad_spot in tactic_detector.py]

**Port requirement:** Implement `_is_in_bad_spot(board, sq)` → bool. It is used as:
- A **prune** in `fork`: skip the move if forker lands in a bad spot (cook line 221)
- An **accept** in `skewer`: require `is_in_bad_spot(board_before, capture_sq)` (cook line 491)
- Present in `clearance` (cook line 715) — but clearance is Tier 3, out of scope

#### 3. `king_values` — already exists in our code

`_PIECE_VALUES` dict at `tactic_detector.py:44–51` already has `chess.KING: 99`. [VERIFIED: tactic_detector.py:44–51]

The note states we lack it, but reading the actual code shows it was already added. This is **not** a gap — `_piece_value(chess.KING)` returns 99 today. However, `detect_fork` calls `_piece_value(mover_type)` and `_piece_value(target.piece_type)` at lines 331 and 339 — the king value does flow through correctly. The fork divergence noted below (gap 3 in the alignment note) about king values is already fixed.

**Important note for planner:** The cook-alignment note says we lack king_values, but `_PIECE_VALUES` already contains `KING: 99`. The planner should verify this before adding a "port king_values" task — it only needs a variable rename from `_PIECE_VALUES` to whatever name the refactored code uses.

### Per-Motif Divergences (Confirmed Against cook.py Source)

#### skewer — P(test) 0.154 → target >0.9 or suppress

cook's `skewer()` (cook.py:471–493): fires for a pov move at `mainline[1::2][1:]` (i.e., from 2nd pov move on) that:
1. **Captures** with a ray piece (Q/R/B) and is NOT checkmate
2. `between = SquareSet.between(move.from, move.to)`
3. `op_move` = the immediately preceding opponent move
4. Requires `op_move.to_square != capture_square` (not a recapture of what opp just played) **AND** `op_move.from_square in between` (opp piece moved ACROSS the skewer line, from a between-square)
5. Requires `king_values[moved_piece_type(prev)] > king_values[capture.piece_type]` (the opponent's piece that just moved is MORE valuable than what we're capturing — it was in front) **AND** `is_in_bad_spot(board_before, capture_sq)`

Our `detect_skewer` (tactic_detector.py:482–542): uses a fundamentally different algorithm. It checks for a "piece behind" the captured piece on the same ray (looking at `board_after.attacks(move.to_square)`) or a "high-value piece attacked from the mover's new position." It does NOT implement the cook relational predicate — specifically it has no `op_move.from_square in between` check and no `is_in_bad_spot` requirement. [VERIFIED: tactic_detector.py:496–542]

**Gap confirmed:** Complete rebuild required. The current implementation is not a variant of cook's logic — it's a different algorithm entirely. Expected to be the hardest port.

#### discovered-attack — P(test) 0.167 → target >0.9 or suppress

cook's `discovered_attack()` (cook.py:287–305):
1. If `discovered_check(puzzle)` → return True (sub-case 1, now split to our discovered-check detector)
2. For each pov capture at `[1::2][1:]`:
   - `between = SquareSet.between(move.from, move.to)`
   - If `op_move.to_square == capture_square` → return False (recapture short-circuit)
   - `prev = moves[k-2]` (the earlier pov move)
   - Require: `prev.from_square in between` AND `capture != prev.to` AND `from != prev.to` AND `prev` not castling

Our `detect_discovered_attack` (tactic_detector.py:606–648): handles sub-case 2 correctly in structure — scans `range(2, len(moves), 2)`, checks that `move.from_square != moves[0].to_square` (same piece moved again filter). BUT it lacks the recapture short-circuit (`op_move.to_square == capture_square` → False) and the key geometric test: `prev.from_square in between`. Instead it just checks that the unveiled piece (`board_before.piece_at(move.from_square)`) is a pov piece. [VERIFIED: tactic_detector.py:629–646]

**Gap confirmed:** Missing recapture short-circuit and missing `prev.from_square in between` check. The current sub-case 2 fires on any later pov capture where the capturer is a different piece from the first move — far too broad.

#### back-rank-mate — P(test) 0.271 → target >0.9

cook's `back_rank_mate()` (cook.py:820–848):
1. Final board is checkmate, defender king on its back rank
2. Build the king's three forward squares (one orthogonal + two diagonal, clipped at a/h files)
3. For each such square: if empty, OR holds a pov piece, OR is attacked by pov → **return False**. All three forward squares must be blocked by the defender's OWN pieces (not empty, not pov pieces, not pov-attacked).
4. At least one checker on the back rank.

Our `detect_back_rank_mate` (tactic_detector.py:773–801):
- Checks king rank is back rank ✓
- Checks at least one checker is a pov piece — but requires checker exists on the board, not specifically on the back rank ✗
- Does NOT check the forward-square own-blocker test ✗

[VERIFIED: tactic_detector.py:782–801]

**Gap confirmed:** Missing the own-blocker test (the critical gate that eliminates corner mates) and missing the back-rank-checker requirement. These two absent checks explain the 0.271 precision — it fires on corner mates where the king has no forward squares or where forward squares are empty (stalemate-adjacent positions).

**Implementation note for planner:** Cook's `squares` calculation uses `king + (-8 if pov else 8)` (forward for pov) and diagonals clipped at files 0/7. Need to replicate this relative-rank arithmetic carefully for both pov colors.

#### fork — P(test) 0.400 → target >0.9 or suppress

cook's `fork()` (cook.py:217–239):
1. Scans `mainline[1::2][:-1]` (all pov moves **except the last**)
2. Moved piece is not king
3. **Prune:** if `is_in_bad_spot(board_after, to_square)` → skip (forker is loose)
4. Count victims via `attacked_opponent_squares(to, pov)`, **skipping pawns**
5. Victim qualifies if: `king_values[victim] > king_values[forker]` (higher value, king counts) OR (`is_hanging(board, victim, sq)` AND `sq not in board.attackers(not pov, to_square)` — the hanging victim isn't defending the fork square)
6. Fire if count > 1

Our `detect_fork` (tactic_detector.py:293–346):
1. Scans `range(0, len(moves), 2)` — includes the last pov move ✗ (should exclude it, but see note below)
2. No `is_in_bad_spot` prune for the forker ✗
3. Does not skip pawn victims ✗
4. Uses `_piece_value(target.piece_type) > mover_val` (no king=99 in this comparison path) — but wait, `_piece_value(chess.KING)` IS 99 per the verified `_PIECE_VALUES` dict, so this is actually OK ✓
5. The victim "not an attacker of the fork square" clause is missing ✗
6. `_is_hanging` (non-ray) used instead of ray-aware `is_hanging` ✗

[VERIFIED: tactic_detector.py:313–344]

**Note on last-move scanning:** cook excludes the last pov move (`[:-1]`). Our code scans all pov moves including the last. This was an intentional Phase 124 fix for odd-length PVs. The cook alignment note correctly flags this; aligning with cook here may reduce FP from scenarios where the last pov move "forks" in a non-fork context.

**Gap count: 4 confirmed gaps** (no `is_in_bad_spot` prune, pawn victims not skipped, missing "not attacker of fork square" clause, non-ray `_is_hanging`). King value is NOT a gap.

#### pin — P(test) 0.439 → target >0.9 or suppress

cook fires pin if **either** of two sub-tests passes:

**pin_prevents_attack** (cook.py:577–598): For each pov move, board has an opponent piece pinned (`board.pin(piece.color, sq) != BB_ALL`). That pinned piece attacks a pov piece on a square **outside the pin direction** (`attack not in pin_dir`), and the attacked pov piece is either worth more OR is hanging.

**pin_prevents_escape** (cook.py:601–632): A pinned opponent piece is attacked by a pov attacker **inside the pin direction** (`attacker_sq in pin_dir`). Fires if the pinned piece is worth more than the attacker, OR (pinned piece is hanging AND pinned square is not attacked by opponent where the pov attacker sits AND pinned piece has pseudo-legal moves off the pin line).

Our `detect_pin` (tactic_detector.py:433–479): implements a basic pin finder using `board.pin(not pov, sq)` — iterates board positions, finds opponent pieces with a pin ray, finds the pov pinner along the ray, then applies a relevance gate. This is closer to `pin_prevents_escape` than `pin_prevents_attack`. It does not implement `pin_prevents_attack` at all, and its `pin_prevents_escape` logic is simplified (no "pinned piece has pseudo-legal moves off pin line" check, no "not attacker of forker square" equivalent). [VERIFIED: tactic_detector.py:448–479]

**Gap confirmed:** Full two-sub-test port needed. The pin-direction membership check via `board.pin()` is already used — the port primarily adds the `pin_prevents_attack` sub-test and completes the `pin_prevents_escape` sub-test with the off-pin-line escape check.

#### anastasia-mate — P(test) 0.857 → target >0.9

cook's `anastasia_mate()` (cook.py:851–874): checks the final board's king on a/h file but NOT corner; mating move lands on king's file by Q/R; flips board to h-file if on a-file; requires opponent blocker at `king+1` and pov knight at `king+3`.

Our `detect_anastasia_mate` (tactic_detector.py:864–899): checks king on file 0 or 7, not rank 0/7; checks mating piece is Q/R; looks for any pov knight anywhere on the board (not specifically at `king+3`). [VERIFIED: tactic_detector.py:880–899]

**Gap confirmed:** The "knight at king+3" geometry and "blocker at king+1" are not fully validated. The knight check is too loose (any pov knight vs. specifically at king+3 in the normalized position). Small lift expected.

#### hook-mate — P(test) 0.841 → target >0.9

cook's `hook_mate()` (cook.py visible from the test): final board is checkmate; mating move is a rook adjacent to the king (rook is in king's attacks); a pov knight defends the rook; a pov pawn defends that knight (the chain: pawn←knight←rook→king).

Our `detect_hook_mate` (tactic_detector.py:902–946): checks is_checkmate, last move is a rook, rook adjacent to king, pov knight defends rook, pov pawn defends knight. Logic appears correct in structure. [VERIFIED: tactic_detector.py:910–946]

**Minor gap:** Need to verify the "rook adjacent to king" test uses `boards[-1].attacks(opp_king_sq)` (king's attack range) not Manhattan distance — cook checks `rook_sq not in board.attacks(king_sq)`. #### discovered-check — P(test) 0.833 → hold ≥0.85 (no detector work unless broken by neighbors)

## Dispatch Rework — Structural Analysis (D-05/D-06/D-07)

### Current Sort Key

The current dispatcher (`detect_tactic_motif`, lines 1798–1804) uses:

```python
def _sort_key(c: Candidate) -> tuple[int, int, int]:
    depth_val = c[4] if c[4] is not None else 999999
    return (c[0], c[1], depth_val)  # (tier, rank, depth)
```

`winner = min(candidates, key=_sort_key)`

This means tier dominates, rank second, depth only a tiebreaker within same-tier same-rank. A Tier-2 fork at depth 6 beats a Tier-4 hanging-piece at depth 0. [VERIFIED: tactic_detector.py:1800–1804]

### Target Sort Key (D-05)

New behavior: depth primary, tier/rank as tiebreakers:

```python
def _sort_key(c: Candidate) -> tuple[int, int, int]:
    depth_val = c[4] if c[4] is not None else 999999
    return (depth_val, c[0], c[1])  # (depth, tier, rank)
```

Effect: a hanging-piece at depth 0 beats a fork at depth 2, which beats a pin at depth 4. Equal-depth ties break by tier then rank (same as before).

### Structural Implication

The current flow already collects all non-mate candidates then sorts. The depth-primary sort key is a **one-line change to `_sort_key`** — no architectural restructuring needed. The "ply-outer/detector-inner walk with early-exit" mentioned in D-05 is the ideal implementation but not mandatory; the collect-then-sort approach with the new sort key produces identical results and is far lower blast radius.

**Recommendation for planner:** Change `_sort_key` to `(depth_val, c[0], c[1])`. Do NOT restructure to ply-outer/detector-inner unless performance profiling on the backfill shows the collect-then-sort is a bottleneck. The sort-key change is a 1-line edit; the restructuring is a rewrite of the entire dispatcher.

### Mate Gate (D-06)

Currently all mate detectors require `boards[-1].is_checkmate()`. The PV is capped at 12 plies — a mate-in-7 is 13 half-moves and the PV is truncated, so `boards[-1].is_checkmate()` is False even though Stockfish reported mate. The fix is to gate the mate SHORT-CIRCUIT path on the stored `eval_mate` value from `game_positions`.

Looking at `flaws_service.py:369`, `positions[n - 1].eval_mate` is already available at classify time. The `_detect_tactic_for_flaw` function receives `positions` as a list — the `eval_mate` value for the position being analyzed (flaw ply) is accessible there.

**Required change:** In `detect_tactic_motif`, add an optional `has_mate_score: bool = False` parameter (or pass it as a precondition). When `has_mate_score=True`, run the mate branch even if `boards[-1].is_checkmate()` is False; when False, preserve the current `is_checkmate` gate. The call sites in `_detect_tactic_for_flaw` pass `eval_mate is not None and eval_mate > 0` from the relevant position row.

Alternative (lower blast radius): the mate detectors themselves already check `boards[-1].is_checkmate()`. Since the PV IS 12 plies and the deepest named mates in the fixture have mean TP depth ~2-3, the truncation only affects long forced mates. The fixture precision for mate (1.00/1.00) is already perfect — so the `is_checkmate` gate is fine for the precision harness. The D-06 gate matters for PROD flaws where Stockfish found mate-in-N > 6. The planner should add a separate task for the mate gate change that passes eval_mate through to detect_tactic_motif, separate from the Workstream A predicate ports.

### Hanging-piece First-Class (D-07)

Under the new `(depth, tier, rank)` sort key, hanging-piece (Tier 4) at depth 0 gets `(0, 4, 0)`. A fork (Tier 2) at depth 2 gets `(2, 2, 0)`. Fork loses because 0 < 2. D-07 is automatically satisfied by depth-primary dispatch — no special case needed.

However: a fork at depth 0 would get `(0, 2, 0)` vs hanging-piece at `(0, 4, 0)`. Fork wins `(0,2,0) < (0,4,0)`. This is correct per D-07: the fork IS at depth 0 meaning the very first move is the fork, which is more specific and equally shallow. Hanging-piece only wins when it is shallower than the geometric.

## Workstream B — Dest-Square Gate (D-03/D-04)

### Available Data in the Missed Pass

In `_detect_tactic_for_flaw` with `orientation="missed"` (flaws_service.py:424–434):
- `board_before` — the board BEFORE the flaw move
- `board_before.turn` — the mover (flaw-maker)
- `pv` — the best line PV string (the "instead-of" line from flaw_ply PV)
- The flaw move itself is available in the CALLER context (`_build_flaw_record`) via `positions[n].move_san`

**Current issue:** `_detect_tactic_for_flaw` does not receive the flaw move. The function signature is `(n, fen_map, positions, pv_by_ply, orientation)` — it constructs the board and fetches the PV internally. The flaw move (`positions[n].move_san`) is only parsed in the `orientation="allowed"` branch (line 449).

**Required change for Workstream B:**
1. Parse the flaw move in the missed branch too (already done for allowed at line 449)
2. Parse the first move of the best PV (the "instead-of" first move)
3. Compare `flaw_move.to_square == best_pv_first_move.to_square`
4. If equal, return `(None, None, None, None)` — suppress the missed tactic

**Where to add:** Inside `_detect_tactic_for_flaw`, in the `if orientation == "missed"` branch, after fetching `pv` and before calling `detect_tactic_motif`. Requires parsing `positions[n].move_san` (already available as `positions[n].move_san` from the GamePosition object).

**Unit fixture design (D-04):** Two fixture types needed:
1. **Suppression case:** flaw move = capture on e5 with wrong piece; best line first move = capture on e5. Expected: `(None, None, None, None)`.
2. **Non-suppression case:** flaw move = move to d4; best line first move = capture on e5. Expected: normal tactic detection.
3. **Edge case:** flaw move = non-capture; best line = capture same square (e.g., best line takes a piece the flaw move missed). Expected: suppression should fire.

## Common Pitfalls

### Pitfall 1: Breaking discovered-check while porting discovered-attack
**What goes wrong:** The two detectors share the sub-case 1 / sub-case 2 split established in Phase 128.1. If the discovered-attack port scans a range that includes depth 0, it re-claims discovered-check positions.
**Why it happens:** cook's `discovered_attack` includes `if discovered_check(puzzle): return True` — the whole discovered-check case. Our split puts sub-case 1 into a separate detector; the port must mirror only sub-case 2 (`range(2, len(moves), 2)`).
**How to avoid:** The ported discovered-attack must start its loop from index 2 (skip first pov move). Verify discovered-check precision does not drop below 0.833 after the port.
**Warning signs:** discovered-check precision drops in the harness; ΔP becomes strongly negative.

### Pitfall 2: Shared utility port ripple — all consumers need re-verification
**What goes wrong:** Replacing `_is_hanging` with the ray-aware `_is_defended`-based equivalent changes behavior for fork, hanging-piece, skewer, pin, interference, and clearance simultaneously. A consumer that was correct under the old `_is_hanging` may now over-suppress or over-fire.
**Why it happens:** `_is_hanging` is called from six consumers. A piece that was "hanging" under the old test may be "defended" under the ray-aware test → fewer fork victims counted → fork recall drops. A piece that was "not hanging" under old test may be "hanging" under new test → fork over-fires in different positions.
**How to avoid:** Run `--check-goals` after the shared utility port BEFORE touching individual detectors. Establish a new baseline. Then fix detectors against the new baseline.
**Warning signs:** fork recall drops significantly (it's already 0.78; dropping to 0.60 would signal over-filtering).

### Pitfall 3: Back-rank-mate own-blocker test off-by-one for pov color
**What goes wrong:** cook uses `king + (-8 if pov else 8)` where `pov` is the PLAYER's color. In our convention pov is `board_after_flaw.turn` (the refuting side). The king is the OPPONENT's king (`not pov`). When mapping cook's expression, the forward direction from the opponent's king must be away from the opponent's own back rank, toward pov's territory.
**Why it happens:** cook's pov is who is DELIVERING mate (the puzzler); ours is the same. But the signed offset `(-8 if pov else 8)` assumes White/Black specific rank directions. In python-chess, rank 0 = rank 1 (a1=0), rank 7 = rank 8 (h8=63). `king + 8` moves one rank higher (toward Black's side if White is pov); `king - 8` moves one rank lower.
**How to avoid:** For pov=White (delivering mate), the Black king is on rank 7. Forward squares (toward White's side, away from Black's back rank) are at `king - 8` (one rank lower). For pov=Black, White king on rank 0, forward = `king + 8`. Verify with a known back-rank-mate fixture.
**Warning signs:** back-rank-mate precision does not improve above 0.4 even after the port.

### Pitfall 4: `is_in_bad_spot` uses `values` (no king) while fork uses `king_values`
**What goes wrong:** cook's `is_in_bad_spot` uses `util.values` (no king entry) via `can_be_taken_by_lower_piece`. The fork predicate uses `util.king_values` (king=99) for the victim qualification check. These are different dicts.
**Why it happens:** `is_in_bad_spot` is about "can this piece be profitably captured" — a king can't be captured (so it's not a valid capturer in the lower-value test); `fork` is about "is attacking the king a fork target" — where king=99 makes a king-forking move qualify.
**How to avoid:** In `_is_in_bad_spot`, use the non-king `_PIECE_VALUES` (or a separate `_VALUES_NO_KING` constant without KING entry) for `can_be_taken_by_lower`. In fork's victim qualification, use `_PIECE_VALUES` (which already has KING=99).

### Pitfall 5: Dispatch sort key inversion also re-orders Tier-3 vs Tier-2
**What goes wrong:** Under the new `(depth, tier, rank)` sort key, a Tier-3 deflection at depth 0 gets `(0, 3, 0)`, which beats a Tier-2 fork at depth 2 `(2, 2, 0)`. This is intentional per D-05 (depth primary), but Tier-3 motifs are fuzzy-graded — they may fire at shallow depth for positions that really are forks.
**Why it happens:** All Tier-3 detectors remain active. A deflection at depth 0 in a fork puzzle would shadow the fork.
**How to avoid:** Since Tier-3 motifs are already mostly suppressed at query time via `_TACTIC_CHIP_CONFIDENCE_MIN=70`, this is a lower-urgency concern. Verify the precision harness scores are not degraded for Tier-2 motifs after the dispatch change. If Tier-3 eats Tier-2 precision, a depth floor for Tier-3 may be needed.

### Pitfall 6: Pin depth convention mismatch when integrating the new sub-tests
**What goes wrong:** The existing `detect_pin` returns `max(0, k - 1)` where `k` is the board index. The new `pin_prevents_attack` sub-test fires at a specific board position (after the pov move at node index `i`). The depth must be on the same move-index scale.
**Why it happens:** The SEED-057 depth convention fix (IN-01) established that depth is a move index, not a board index. The board index `k` maps to move `k-1`.
**How to avoid:** Keep returning `max(0, k - 1)` where `k` is the board index at which the pin fires. For `pin_prevents_attack`, the board is `boards[k]` for board index `k`, so depth = `max(0, k - 1)`.

### Pitfall 7: `board.pin()` direction vs `board.pin(color, sq)` calling convention
**What goes wrong:** python-chess `board.pin(color, sq)` returns the pin ray for the piece of `color` at `sq`. In cook's `pin_prevents_attack`, `board.pin(piece.color, square)` where `piece.color` is the OPPONENT (the pinned piece's color). In `pin_prevents_escape`, same call.
**Why it happens:** `board.pin(color, sq)` returns `BB_ALL` if the piece is not pinned; otherwise returns the SquareSet of valid squares the piece can still move to (the pin ray). `attack not in pin_dir` in cook means "this attack square is outside the valid movement ray of the pinned piece" — i.e., the pinned piece cannot capture there.
**How to avoid:** Read the python-chess docs for `Board.pin(color, square)`. The call is on the `board` object with the PINNED piece's color and square. The return value `BB_ALL` means not pinned. Any other value is the valid ray.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| X-ray defense detection | Custom ray tracer | `board.copy(stack=False)`, `remove_piece_at()`, `attackers()` | cook's exact approach; board copy is cheap at single-position scale |
| Pin direction | Manual ray walk | `board.pin(color, sq)` | python-chess native; already used in detect_pin |
| Between-square geometry | Manual square enumeration | `chess.SquareSet.between(sq1, sq2)` | python-chess native; already used in detect_skewer |
| Squares between two squares (collinear check) | Custom collinear | `SquareSet.between()` | Handles diagonal + orthogonal automatically |
| Board copy for hypothetical analysis | Deep copy | `board.copy(stack=False)` | `stack=False` avoids copying move history; ~3x faster |

**Key insight:** The shared utility ports (ray-aware `is_defended`, `is_in_bad_spot`) are the force multiplier. Getting these right propagates precision improvements to six detectors. The per-motif ports are additive on top.

## Architecture Patterns

### The cook index → our index translation

This is the most error-prone part of the port. Every cook predicate uses `mainline[1::2]` (odd = pov's moves) and relative offsets. Our boards/moves arrays use:

```
moves[0] = pov's first move (even index = pov)
moves[1] = opponent's first response
moves[2] = pov's second move
...
```

| cook expression | Our equivalent |
|---|---|
| `mainline[1::2]` | `range(0, len(moves), 2)` |
| `mainline[1::2][1:]` | `range(2, len(moves), 2)` |
| `mainline[1::2][:-1]` | `range(0, len(moves) - 2, 2)` (exclude last pov move) |
| `node.parent.move` = prev opponent move | `moves[k-1]` where k is the current pov move index |
| `node.parent.parent.move` = prev pov move | `moves[k-2]` |
| `node.board()` = board after this pov move | `boards[k+1]` |
| `node.parent.board()` = board before this pov move | `boards[k]` |

For `[1::2][1:]` → `range(2, len(moves), 2)`: the note confirms this is the correct mirror. The first pov move (`moves[0]`) lacks a preceding pov move (`moves[k-2]` would be `moves[-2]` = invalid at k=0), so starting from k=2 is correct.

### Suppression mechanism (Claude's Discretion)

The existing `tactic_confidence` query-suppression lever (`_TACTIC_CHIP_CONFIDENCE_MIN=70` in library_repository.py) works for Tier-3 motifs. For Tier-1 and Tier-2 motifs (confidence always 100), the suppression path is setting the motif to NULL or adding the motif to `SUPPRESSED_MOTIFS` in `precision_floors.py`.

**Recommended pattern for a failing motif (< 0.9 on TEST):** Add to `SUPPRESSED_MOTIFS` in `precision_floors.py`. This removes it from the floor gate. The motif detector itself can remain in the registry — the precision score just won't be gated. If the UI must not show it, add it to `FAMILY_TO_MOTIF_INTS` suppression logic in `library_repository.py`.

For the specific case of skewer: if full cook fidelity still yields < 0.9 on TEST, add skewer to `SUPPRESSED_MOTIFS` and remove it from `PRECISION_FLOOR`. The detector code stays (it's needed for the harness to measure progress); only the CI gate is dropped.

## Code Examples

### Ray-aware `_is_defended` pattern

Based on cook's `util.is_defended` (util.py:72–85). Implement from this description, no source copy:

```python
# Source: cook.py util.is_defended pseudocode (AGPL boundary — reimplemented)
def _is_defended(board: chess.Board, piece: chess.Piece, sq: int) -> bool:
    """True if piece at sq of piece.color has a direct or X-ray defender."""
    # Direct defense
    if board.attackers(piece.color, sq):
        return True
    # X-ray defense: any opponent ray attacker? Remove it and re-check.
    for attacker_sq in board.attackers(not piece.color, sq):
        attacker = board.piece_at(attacker_sq)
        if attacker is not None and attacker.piece_type in _RAY_PIECES:
            bc = board.copy(stack=False)
            bc.remove_piece_at(attacker_sq)
            if bc.attackers(piece.color, sq):
                return True
    return False
```

### `_is_in_bad_spot` pattern

```python
# Source: cook.py util.is_in_bad_spot pseudocode (AGPL boundary — reimplemented)
def _is_in_bad_spot(board: chess.Board, sq: int) -> bool:
    """True if piece at sq is attacked AND (hanging or capturable by lower non-king)."""
    piece = board.piece_at(sq)
    if piece is None:
        return False
    if not board.attackers(not piece.color, sq):
        return False  # not even attacked
    # Hanging: no defense at all
    if not _is_defended(board, piece, sq):
        return True
    # Capturable by a lower-value non-king piece
    for atk_sq in board.attackers(not piece.color, sq):
        atk = board.piece_at(atk_sq)
        if atk is not None and atk.piece_type != chess.KING:
            # Use _PIECE_VALUES without king for the comparator
            if _PIECE_VALUES.get(atk.piece_type, 0) < _PIECE_VALUES.get(piece.piece_type, 0):
                return True
    return False
```
