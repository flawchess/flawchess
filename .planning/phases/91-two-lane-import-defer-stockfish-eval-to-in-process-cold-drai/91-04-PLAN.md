## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → API | Authenticated via `current_active_user`; FastAPI-Users provides the auth chain |
| API → DB | Per-user-scoped `WHERE user_id = :uid` query; SQLAlchemy parameterized |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-14 | Spoofing | Unauthenticated access to `/imports/eval-coverage` | mitigate | `Depends(current_active_user)` raises 401 for missing/invalid token. Tested by `test_eval_coverage_requires_auth`. |
| T-91-15 | Information Disclosure | Cross-user data exposure (User A sees User B's counts) | mitigate | Both `count_games_for_user` and `count_pending_evals` filter on `Game.user_id == user_id`. Tested by `test_eval_coverage_scoped_to_user`. |
| T-91-16 | Denial of Service | COUNT query on large `games` table per-user | accept | At current scale (~50k rows per user, indexed on `user_id`) the per-user COUNT is < 50 ms. Future composite index `(user_id) WHERE evals_completed_at IS NULL` is a follow-up if proven slow. |
| T-91-17 | Tampering (input) | No user input — endpoint is parameterless GET | n/a | No query params, no body, no path params. Attack surface is zero beyond auth. |
| T-91-18 | Information Disclosure (operational metadata) | Pending count reveals user's import recency | accept | The signal is intentional and user-facing (header bar displays it). Not sensitive. |
</threat_model>
