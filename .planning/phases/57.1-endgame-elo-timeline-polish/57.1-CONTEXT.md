<domain>
## Phase Boundary

This phase refines the Endgame ELO Timeline chart shipped in Phase 57. Two coherent changes:

1. **Elo formula anchor change** — The Endgame ELO number is anchored on the user's actual rating at each emitted date (asof-join per combo, forward-fill from the latest prior game's rating when the point's week has no games), NOT on the rolling-mean opponent-rating window Phase 57 used. Skill math (composite of Conversion Win %, Parity Score %, Recovery Save % over `ENDGAME_ELO_TIMELINE_WINDOW`=100 endgame games) is unchanged — only the translation formula's anchor shifts.

2. **Volume bars** — Add a muted per-week endgame game-count bar series at the chart bottom (~20% of chart height) so users can see at a glance whether a weekly point is well-supported or marginal. Requires swapping Recharts `<LineChart>` → `<ComposedChart>` (mixes `<Line>` + `<Bar>`).

Derived: the `endgame_elo` number is no longer a classical Elo performance rating (which anchors on avg_opponent_rating). It is now a **skill-adjusted rating** — "your rating, shifted by how much your Endgame Skill beats or falls short of the 50% neutral mark." The info-popover copy must stop calling it a performance rating.

OUT OF SCOPE:
- Adding new requirement IDs (this is an ELO-05 refinement, not a new capability)
- Changing how skill is computed (still the mean-across-non-empty-buckets composite)
- Changing the 100-game trailing window size
- Changing the [0.05, 0.95] skill clamp
- Changing the line color/stroke convention (bright solid = Actual, dark dashed = Endgame — already landed after Phase 57 corrections)
- Changing the chart's placement in the Endgames page (already moved after Endgame Metrics section)
- Adding opponent-rating anchoring as a secondary display or fallback (decided: clean cut, single definition)
- Per-combo stacked volume bars (v1 ships pooled muted bars; can iterate later)
</domain>

<decisions>
## Implementation Decisions

### Elo anchor (backend)

- **D-01 (revised):** `endgame_elo = round(actual_elo_at_date + 400 * log10(clamped_skill / (1 - clamped_skill)))`. `actual_elo_at_date` is the user's rating at the point's emission date, per combo, sourced via asof-join (latest game's `user_rating` played on or before the ISO-week end date for that combo). Clamp bounds remain `_ENDGAME_ELO_SKILL_CLAMP_LO=0.05` / `_ENDGAME_ELO_SKILL_CLAMP_HI=0.95`.
- **D-02:** Forward-fill when the point's week has no games: use the rating from the most recent prior game in that combo. If no prior game exists (first week of the timeline), the existing ≥10-endgame-games floor already prevents point emission, so this edge case is unreachable.
- **D-03:** `user_rating` is the rating at the **start** of a game (the column existing in `games` today — chess.com/lichess PGN exports don't expose post-game rating). "End-of-week rating" means the rating going into the last game played on or before the week's end — this is "off by at most one game's delta" from a true post-game rating, acceptable.
- **D-04 (revised):** The Actual ELO value on each emitted point is the same `actual_elo_at_date` derived above (not a rolling mean). Both lines (bright Actual ELO and dark dashed Endgame ELO) share this anchor, so they have the same shape and the gap between them IS the skill signal.
- **D-05:** `avg_opponent_rating` is no longer part of the timeline emission and is dropped from the payload. It was only there to serve the old performance-rating formula.

### Volume bars (backend payload)

- **D-06:** Add `per_week_endgame_games: int` to each `EndgameEloTimelinePoint`. This is the count of qualifying endgame games for that (combo, week) bucket — NOT the trailing-window count. Zero-count weeks do not emit points (the existing `endgame_games_in_window >= MIN_GAMES_FOR_TIMELINE` guard still applies).
- **D-07:** Keep `endgame_games_in_window: int` on the point (already shipped in Phase 57) — the tooltip still uses "past N games" text derived from it.

### Volume bars (frontend)

- **D-08:** Swap `<LineChart>` for `<ComposedChart>` in `EndgameEloTimelineSection.tsx`. `<Line>` instances port over with no prop changes.
- **D-09:** Single muted `<Bar>` series per week showing the **pooled** per-week endgame game count across all currently-visible combos. Sum visible combos' `per_week_endgame_games` for each date. Hidden combos (via legend toggle) are excluded from the sum — makes the bars react consistently with the lines.
- **D-10:** Bar visual style: muted fill using `charcoal-texture`-compatible theme color (e.g. `--muted-foreground` at ~25% opacity), no border, no label. Right Y-axis dedicated to bars, hidden (ticks + line styled `hidden`/`visibility: hidden`). Bars occupy ~20% of the chart height via `<YAxis yAxisId="bars">` with a domain that leaves the upper ~80% empty (e.g. `[0, max_count * 5]`).
- **D-11:** Tooltip additions: show the per-week game count as a single line at the top of the tooltip ("Games this week: N (visible combos)"). Bright+dark swatch rows for each combo unchanged.

### Copy rewrite (locked strings)

- **D-12:** Info-popover paragraph 1 (performance-rating framing) rewrite: "**Endgame ELO** is your Actual ELO shifted by how much your Endgame Skill exceeds (or falls short of) the 50% neutral mark. We compute it as *actual_elo + 400 · log10(skill / (1 − skill))*, where skill is the composite of Conversion Win %, Parity Score %, and Recovery Save % over your trailing 100 endgame games."
- **D-13:** Info-popover paragraph 2 (rolling window) rewrite: keep the bright/dark convention but explain the anchor: "The bright line is your **Actual ELO** — your rating at each date from the most recent game on or before that date. The dark dashed line is **Endgame ELO**. If your Endgame Skill is exactly 50% the two lines touch; 75% skill puts Endgame ELO roughly 190 Elo above, 25% skill puts it roughly 190 Elo below. The gap between the lines is the interesting signal."
- **D-14:** Info-popover paragraph 3 (window + clamp) minor rewrite: drop "performance rating" language. "Points are emitted weekly; each Endgame Skill value looks back at your trailing 100 endgame games for that platform and time control. Weeks with fewer than 10 qualifying endgame games are hidden. Skill is clamped to the 5–95 % range so a handful of lucky or unlucky endgames can't produce an absurd rating shift at the extremes."
- **D-15:** Info-popover paragraph 4 (Glicko caveat) unchanged.
- **D-16:** Chart subtitle rewrite: "Actual ELO versus Endgame ELO over time, per platform and time control. Bars show endgame games per week. Bright lines are Actual ELO, dark dashed lines are Endgame ELO."
- **D-17:** HUMAN-UAT.md from Phase 57: supersede item 1 ("bright/dark contrast") and item 3 ("tooltip shows endgame/actual/gap/games") with revised expected strings matching D-12/D-13/D-16. Add a new item "Volume bars render at bottom, scale with visible combos, hidden when combo toggled off". Items 2, 4–8 carry over unchanged.

<specifics>
## Specific Ideas

- **Asof-join in Python (orchestrator) vs SQL (repo):** Python approach — pre-compute per-combo sorted `(played_at, user_rating)` list once, then binary-search for each emitted week's cutoff. Keeps `_compute_endgame_elo_weekly_series` callers minimal and repo-layer SQL unchanged. SQL approach — `LATERAL JOIN` or correlated subquery per week × combo. Planner decides.
- **Per-week game count — source data:** the repository already returns bucket_rows with `played_at`. Group by ISO week + combo in Python (same place the weekly series is already built). No new SQL needed.
- **Volume bar height envelope:** use `ceil(max(visible_weekly_counts) * 1.2)` for the bar Y-axis domain, bounded to max 20% of the chart's visual height via Recharts' `yAxisId` + right-axis `domain={[0, barMax * 5]}`. (If barMax=40, the right axis becomes [0, 200], so the 40-tall bar occupies the bottom 20%.)
- **Skill = undefined case:** if all three material buckets are empty in the window, `_endgame_skill_from_bucket_rows` returns None (existing). `get_endgame_elo_timeline` already skips emission in that case — carries over.
- **Parity skill=0.5 case:** `log10(0.5/0.5) = log10(1) = 0` so `endgame_elo = actual_elo`. Zero-delta is the expected neutral case; tests should assert this literal.
- **Platform bias is no longer an issue:** with the new anchor, cross-platform Endgame ELO values are directly comparable (both are each platform's own Actual ELO plus their own skill delta). The Glicko-1-vs-2 caveat in the info-popover remains because the underlying Actual ELO scales differ, but the gap interpretation is now uniform across platforms.
- **Testid carry-over:** all existing testids from Phase 57 remain (`endgame-elo-timeline-section`, `-chart`, `-info`, `-empty`, `-error`, `-legend-{combo_key}`). New optional: `endgame-elo-volume-bars` on the Bar element (or its wrapping g, if Recharts limits prop passthrough).
</specifics>
