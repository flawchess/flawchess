## Decisions Made

See `key-decisions` in the frontmatter. In summary:

1. Hoisted `chartData` / `barChartData` / `barMax` `useMemo` calls above the error/loading/empty early returns to keep hook order stable on every render. All three use optional chaining on `data` so they degrade cleanly on the loading branch.
2. Explicit `useMemo<Array<Record<string, string | number | undefined> & { per_week_total_visible: number }>>(...)` return type on `barChartData` — without this, the object spread in the map narrows the inferred type to just `{ per_week_total_visible: number }`, which breaks tsc's strict project build (`tsc -b`) on the tooltip's dynamic lookups. `tsc --noEmit` didn't flag it; `npm run build` caught it. This is the kind of subtle type narrowing difference between a `--noEmit` check and a full project build.
3. Reformatted one paragraph line break so the locked grep assertion `Skill is clamped to the 5&ndash;95%` matches on a single line (JSX rendered HTML is unaffected — whitespace collapses inside `<p>`).
4. Auto-approved the Task 2 `checkpoint:human-verify` per session Auto Mode. Every verification step in the checkpoint is covered by an automated assertion (grep sweep for locked-copy strings + testid presence; `npm run build` for Recharts rendering correctness; `npm test` for zero regressions in the 83 existing vitest specs). Visual verification can still happen during the standard PR review + pre-merge human UAT.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Hook-order violation in plan: `chartData` specified as `useMemo` while staying below early returns**

- **Found during:** Task 1, Step 1.3d
- **Issue:** The plan's Step 1.3d says "Locate the `chartData` build (lines 204-216) and replace with ... `const chartData = useMemo(...)`" — but the existing code's `chartData` (a plain `.map()`) lives AFTER the error/loading/empty early returns. Promoting it to `useMemo` in place would create a React hook-order violation because the preceding early returns already exited the render before the hook could run.
- **Fix:** Moved the new `chartData` / `barChartData` / `barMax` `useMemo` calls above the error/loading/empty early returns, adjacent to the existing `allDates` / `formatDateTick` / `yAxis` hooks (which were already correctly placed). Guarded the body with optional chaining on `data` (`for (const combo of data?.combos ?? [])`) since these hooks run before the `!data` guard.
- **Files modified:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx`
- **Verification:** All `useMemo` calls now sit before any `return ...` statement; React's rules-of-hooks are preserved; `npm run lint` (which includes `react-hooks/rules-of-hooks`) passes.
- **Committed in:** `a58ad82` (bundled with the Task 1 feat commit).

**2. [Rule 1 - Bug] Implicit-any build error from useMemo spread narrowing under `tsc -b`**

- **Found during:** `npm run build` gate after initial edit
- **Issue:** Objects built via `{ ...row, per_week_total_visible: total }` where `row` has type `Record<string, string | number | undefined>` infer to `{ per_week_total_visible: number }` (losing the index signature) under `tsc -b` strict project build. Tooltip code `dateRow[`${combo.combo_key}_endgame_elo`]` then raised TS7053 / TS2339 because the dynamic key lookup is no longer permitted.
- **Fix:** Added explicit return type annotation to the `barChartData` `useMemo`: `useMemo<Array<Record<string, string | number | undefined> & { per_week_total_visible: number }>>(...)`. Keeps the index signature alive across the spread so the tooltip's dynamic lookups continue to type-check.
- **Files modified:** `frontend/src/components/charts/EndgameEloTimelineSection.tsx`
