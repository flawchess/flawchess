## Architecture Patterns

### Pattern 1: Python-side asof-join via bisect (RECOMMENDED)

**What:** Pre-compute a per-combo sorted list of `(played_at, user_rating)` from `all_rows` once. For each emitted week, `bisect_right` the week-end date to find the index of the last game on or before that date, read `user_rating` at `index - 1`.

**Why this shape over SQL:**
1. `query_endgame_elo_timeline_rows` already returns `all_rows` chronologically sorted per combo — the input is free.
2. No new SQL = no new migration risk, no new query-planner behavior to verify in prod.
3. O(log N) per week × ~300 weeks max × 8 combos = well under 20k comparisons. Cheap.
4. Matches CLAUDE.md's "execute queries sequentially within the same session" guidance — one less query round-trip.

**Where the computation lives:** The CONTEXT.md "Claude's Discretion" item asks whether the helper grows a new param or the orchestrator pre-computes. **Recommendation: orchestrator builds a per-combo rating map and passes it into the helper.** Rationale:
- `_compute_endgame_elo_weekly_series` already takes 4 params (bucket_rows, all_rows, window, cutoff_str). Growing it further hurts readability.
- The asof lookup needs the complete `all_rows` stream sorted by `played_at` *regardless of cutoff* — the helper already receives this, but adding a callback or map keeps the helper's responsibilities clear (compute skill + window stats) vs. orchestrator (partition + asof).

**Example (for the plan):**
```python
# In get_endgame_elo_timeline, after partitioning all_rows by combo:
import bisect

# Build per-combo asof lookup: two parallel arrays for bisect.
# all_rows tuple shape: (played_at, platform, tc, user_color, white_rating, black_rating)
def _build_rating_asof(all_rows: list[tuple[Any, ...]]) -> tuple[list[Any], list[int]]:
    """Return (dates, user_ratings) parallel arrays sorted by date ASC.
    user_rating derived from user_color + white/black_rating."""
    dates: list[Any] = []
    ratings: list[int] = []
    for row in all_rows:
        played_at, _plat, _tc, user_color, white, black = row[:6]
        if white is None or black is None:
            continue
        user_rating = white if user_color == "white" else black
        dates.append(played_at)
        ratings.append(user_rating)
    # all_rows is already sorted by played_at in the repo (ORDER BY played_at ASC),
    # so no re-sort needed — assert defensively in case repo contract changes.
    return dates, ratings


def _rating_at_date(dates: list, ratings: list[int], cutoff: datetime) -> int | None:
    """Forward-fill rating as of cutoff (inclusive). Returns None if no prior game."""
    # bisect_right returns the insertion point AFTER any entries == cutoff,
    # so dates[idx-1] is the latest game on-or-before cutoff.
    idx = bisect.bisect_right(dates, cutoff)
    if idx == 0:
        return None
    return ratings[idx - 1]
```

Then `_compute_endgame_elo_weekly_series` gains one param — `rating_asof: Callable[[datetime], int | None]` — or simply a pre-computed `dict[tuple[int, int], int]` keyed by `(iso_year, iso_week)` mapping each emitted week to its asof rating. The dict variant is simpler (no callable shape to type-annotate) and works because the set of emitted ISO-weeks is determined during the walk; the orchestrator can either (a) make a two-pass pre-compute or (b) pass the `(dates, ratings)` arrays and let the helper bisect.

**Preferred:** Pass the two arrays `(dates: Sequence[datetime], ratings: Sequence[int])` plus a thin `_rating_at_date` helper. This keeps the helper pure (no callable injection) and avoids two-pass pre-compute.

**When to use:** Any future timeline chart that needs "the value of X at time T for a per-partition series". Common enough to formalize.

### Pattern 2: Per-week count aggregation (trivial add)

**What:** While walking the merged event stream, increment a per-`(iso_year, iso_week)` counter every time the event's side is `"endgame"`. The counter lives on the already-existing `data_by_week` dict — add one key `"per_week_endgame_games"`.

**When to use:** Any helper already producing weekly-bucketed output. The Phase 57 helper already does the ISO-week computation — adding one more accumulator is ~3 lines.

**No off-by-one risk with the cutoff pattern (Phase 57 Pitfall 2):** The per-week count is NOT a trailing-window count — it's the count of events in a single week. The cutoff filter at emission-time (`if cutoff_str is None or data_by_week[key]["date"] >= cutoff_str`) drops the entire week's point, count included. Since the count only sums events that happened in that specific ISO week, there's nothing for pre-filled earlier games to "bleed into". [VERIFIED via read of `endgame_service.py:1034-1047`]

### Pattern 3: ComposedChart + hidden right Y-axis for bar-height clamping

**What:** Add a second `<YAxis yAxisId="bars" orientation="right" hide={true} domain={[0, barMax * 5]} />` next to the existing left axis, then add `yAxisId="bars"` on the `<Bar>` only. Existing `<Line>` components bind to the implicit left axis (no `yAxisId` prop needed — they keep working unchanged).

**Why `max * 5`:** If the bar's natural max is `M` and the axis domain is `[0, M * 5]`, the bar peak reaches `1/5 = 20 %` of the axis height. Recharts maps the axis domain to the full chart height by default. [CITED: Context7 /recharts/recharts "Bi-axial Charts" — each YAxis is independent; `domain` defines the mapping from value → pixel].

**Implementation (for the plan):**
```tsx
import { ComposedChart, Line, Bar, CartesianGrid, XAxis, YAxis } from 'recharts';

// Compute bar data: pooled count across visible combos only.
// chartData already has `date` + per-combo columns; add `per_week_total_visible` per row.
const barChartData = useMemo(() => {
  return chartData.map((row) => {
    let total = 0;
    for (const combo of data.combos) {
      if (hiddenKeys.has(combo.combo_key)) continue;
      const n = row[`${combo.combo_key}_per_week_games`];
      if (typeof n === 'number') total += n;
    }
    return { ...row, per_week_total_visible: total };
  });
}, [chartData, data.combos, hiddenKeys]);

const barMax = useMemo(() => {
  let m = 0;
  for (const row of barChartData) {
    const v = row.per_week_total_visible;
    if (typeof v === 'number' && v > m) m = v;
  }
  return Math.max(m, 1);  // avoid domain [0, 0] → Recharts div-by-zero
}, [barChartData]);

// In the chart body:
<ComposedChart data={barChartData}>
  <CartesianGrid vertical={false} />
  <XAxis dataKey="date" tickFormatter={formatDateTick} tick={{ fontSize: 12 }} />
  <YAxis yAxisId="elo" domain={yAxis.domain} ticks={yAxis.ticks} tick={{ fontSize: 12 }} />
  <YAxis yAxisId="bars" orientation="right" hide domain={[0, barMax * 5]} />
  <Bar
    yAxisId="bars"
    dataKey="per_week_total_visible"
    fill={ENDGAME_VOLUME_BAR_COLOR}  // e.g., oklch(0.55 0 0 / 0.25)
    isAnimationActive={false}
    data-testid="endgame-elo-volume-bars"
  />
  {/* existing Line elements, no changes to them */}
</ComposedChart>
```

**Note on the existing axis:** The shipped code uses `<YAxis domain={...} ticks={...} />` with no `yAxisId`. Recharts defaults the implicit axis to `yAxisId="0"` and all `<Line>` elements without a `yAxisId` prop bind to it automatically. When adding the bar axis, the existing axis needs an explicit `yAxisId="elo"` (or keep `"0"`) and the bar axis must use a different ID — otherwise both axes map to the same scale and the bar-height clamp breaks. **Add `yAxisId="elo"` to the existing YAxis and to all existing `<Line>` elements in the same commit** — this is the one easy miss that would break line rendering. [VERIFIED: Context7 /recharts/recharts "Bi-axial Charts" example uses explicit IDs on all Lines when multiple YAxes exist].

**`connectNulls` interaction with Bar:** Bars do not observe `connectNulls`. A row with `per_week_total_visible === 0` renders as a zero-height bar (invisible) — which is fine for weeks with no games. A row where the bar dataKey is `undefined` renders no bar at all. **Neither of these cases interacts with `<Line connectNulls>`** — the line still bridges missing Elo values for that week independently. [CITED: Context7 /recharts/recharts Bar component docs].

**Custom legend renderer + ComposedChart:** The shipped custom `renderLegend()` passes a DOM tree via `<ChartLegend content={renderLegend()} />`. ComposedChart uses the same `<Legend>` contract as LineChart — no incompatibility [VERIFIED: Context7 /recharts/recharts ComposedChart example uses `<Legend />` with no special prop handling]. The bar series should **not** appear in the legend — achieve this by adding `legendType="none"` to the `<Bar>` (same trick the shipped code uses on the dashed Endgame ELO line).

### Anti-Patterns to Avoid

- **Growing `_compute_endgame_elo_weekly_series`'s parameter list.** It already has 4 params. Adding a `rating_asof_callable` or `per_week_count_dict` is a readability regression. Instead, either (a) pre-compute the rating map in the orchestrator and pass two parallel arrays, or (b) return the per-week count from the helper as part of the existing `data_by_week` dict (zero API change — the point already has a dict shape). **Prefer (b) for the count and (a) for the asof.**
- **Moving the asof-join into SQL with a correlated subquery.** A LATERAL JOIN or window function per combo × week would require query refactoring, prod query-planner verification, and potentially a new index. Python bisect costs <1 ms. Reach for SQL only if a future profiling pass shows Python is the bottleneck.
- **Letting the bar axis share the line axis.** Bars without `yAxisId` will scale to the Elo axis (1000–2800 range), rendering invisibly small. Always give the bar a dedicated `yAxisId="bars"` with its own domain.
- **Re-introducing `avg_opponent_rating` as a schema field.** It's not there in the shipped code and never was (see Pitfall 4). The plan must NOT include a "drop field" step — the field doesn't exist. The plan MUST include an info-popover copy edit that removes the word from the rendered prose.
- **Including the bar in the legend.** Users toggle combos, not bars. The bar is passive context. Use `legendType="none"` on the `<Bar>` element.
- **Using the `payload` arg to the Recharts tooltip content for the bar value.** The shipped tooltip reads from `chartData.find(r => r.date === label)` — this pattern should extend to read `dateRow.per_week_total_visible` for the "Games this week: N" line. Do not re-wire the tooltip to use `payload` — it was dropped in Phase 57 for a reason (see 57-02-SUMMARY.md deviation #1).

## Common Pitfalls

### Pitfall 1: Missing `yAxisId` on existing `<Line>` elements when adding the second YAxis

**What goes wrong:** Adding `<YAxis yAxisId="bars" ... />` alongside the existing unlabeled `<YAxis />` creates TWO axes, but existing `<Line>` elements with no `yAxisId` prop still bind to the default `"0"` axis (which in some Recharts versions silently breaks layout when multiple axes are declared).

**Why it happens:** Recharts' implicit axis ID resolution works when there's exactly one axis. Adding any named second axis changes the contract.

**How to avoid:** When adding `yAxisId="bars"` to `<Bar>` and `yAxisId="bars"` to the new `<YAxis>`, add `yAxisId="elo"` (or keep `"0"` consistently) on the existing `<YAxis>` AND on every existing `<Line>` in the same edit. **Grep assertion in the plan:** every `<Line` in `EndgameEloTimelineSection.tsx` is followed (within its props) by `yAxisId=`.

**Warning signs:** Bars render but lines disappear or overlap the bar axis. Unit test: a snapshot test that renders 2+ data points and asserts the SVG has both axes and two distinct paths per combo.

### Pitfall 2: `bisect_right` off-by-one when week-end equals a game's played_at timestamp

**What goes wrong:** A user plays a game at exactly `23:59:59` on a Sunday (ISO week-end). `bisect_right` with cutoff = Sunday 23:59:59 will or won't include that game depending on millisecond precision. If the week-end cutoff is computed as `midnight of the NEXT Monday minus 1 microsecond`, off-by-one is possible.

**Why it happens:** `datetime` objects stored in Postgres have microsecond precision; chess.com/lichess timestamps tend to be second-granularity but the column is `TIMESTAMPTZ`.

**How to avoid:** Use `bisect_right(dates, cutoff)` where `cutoff` is set to the START of the next day after the ISO week's Sunday (i.e., Monday 00:00:00). This makes "any game played during the week" satisfy `played_at < cutoff` and `bisect_right` returns the index after all Sunday-included games. Alternatively, use `bisect_right(dates, sunday_23_59_59_999999)` — either works as long as the comparison is consistent.

**Recommendation:** Compute cutoff as `(monday_of_next_week_at_midnight)` explicitly. Unit test for "game played on Sunday of the week routes to that week's asof".

**Warning signs:** First emitted week of a combo shows an unexpectedly-high `actual_elo` (pulled from a prior week's game) because the current-week Sunday game was excluded.

### Pitfall 3: Bar color contrast on `charcoal-texture` background

**What goes wrong:** `charcoal-texture` is a noise-modulated dark surface (`oklch(0.205 0 0)` card + texture). A muted bar fill that's too close to the background disappears; too bright and it dominates the chart.

**Why it happens:** The chart surface isn't a flat color — the texture noise modulates the pixel values, so a bar color with low chroma can vanish into the noise lattice.

**How to avoid:** Use an L value ~0.50–0.60 with zero chroma (`oklch(0.55 0 0 / 0.25)` or similar). The 0.25 opacity lets the texture show through, reinforcing "this is context, not data". Add the constant to `theme.ts` as `ENDGAME_VOLUME_BAR_COLOR`. Visual-verify the result against all 8 combo line colors — bars must not read as one of the line colors.

**Warning signs:** Bars appear to "float" above or "disappear into" the texture noise. First Human UAT run will flag this.

### Pitfall 4: Phantom `avg_opponent_rating` schema field

**What goes wrong:** CONTEXT.md D-21 claims `avg_opponent_rating` is a locked per-point field in Phase 57 that needs removing. The planner/executor assumes this and writes a task to "drop avg_opponent_rating from `EndgameEloTimelinePoint`". The grep fails, the executor is confused.

**Why it happens:** The discuss-phase notes conflated the info-popover prose (which mentions `avg_opponent_rating` as part of the old formula explanation) with the schema. Shipped code has no such field in either `app/schemas/endgames.py` or `frontend/src/types/endgames.ts` — it's only a Python-local variable inside `_compute_endgame_elo_weekly_series` and a prose term in the info-popover JSX. [VERIFIED: grep across repo, see Summary].

**How to avoid:** Plan the backend schema migration as "add `per_week_endgame_games: int` only" — no field removal. The schema migration is strictly additive. The only thing being "removed" is the word `avg_opponent_rating` from the info-popover JSX (line 110 of `EndgameEloTimelineSection.tsx`), which is a D-12 copy edit, not a schema migration.

**Warning signs:** Plan task says "drop `avg_opponent_rating` from schema" and has no grep evidence that the field exists.

### Pitfall 5: `per_week_endgame_games` pre-fill vs. cutoff interaction

**What goes wrong:** The Phase 57 cutoff pattern filters *emitted* points to `date >= cutoff_str` while letting earlier games pre-fill the rolling window. For the new `per_week_endgame_games` counter, one might think the same "pre-fill" behavior applies — but it doesn't: weeks before the cutoff are not emitted, so their counts are never observed.

**Why it happens:** Easy to conflate "window state for the ≥10 floor" (which must pre-fill) with "per-week count" (which is an output, not a windowed accumulator).

**How to avoid:** The `per_week_endgame_games` value on each emitted point counts games only for that specific ISO week. It doesn't need cutoff handling because the point itself is filtered by cutoff — if the week is excluded, the count goes with it. Unit test: a user with 15 games in week W-2 and 12 games in week W (cutoff = W-1) — the emitted point for W has `per_week_endgame_games == 12`, not 27.

**Warning signs:** A test for pre-fill semantics fails because the planner over-engineered count handling.

## Code Examples

### Asof-join (pattern for the orchestrator)

```python
import bisect
from typing import Sequence, Any
from datetime import datetime

def _rating_at_or_before(
    dates: Sequence[datetime],
    ratings: Sequence[int],
    cutoff: datetime,
) -> int | None:
    """Forward-filled user rating: latest game's rating played on-or-before cutoff.

    `dates` and `ratings` are parallel arrays sorted by date ASC. Returns None when
    no prior game exists (unreachable in practice because the ≥10-endgame-games
    floor prevents point emission before the window fills).
    """
    idx = bisect.bisect_right(dates, cutoff)
    if idx == 0:
        return None
    return ratings[idx - 1]
```
