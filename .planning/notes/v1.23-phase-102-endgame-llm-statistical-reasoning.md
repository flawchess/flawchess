# v1.23 Phase 102 — Endgame LLM Statistical-Reasoning Rework

Captured during `/gsd-explore` on 2026-06-01. This is the durable record of the scoping
session; it overrides the stale assumptions baked into the original REQUIREMENTS LLM-01..07
text (which predated this session and assumed the full p-value + CI + percentile set).

## The starting hypothesis (user) and what we actually found

**User hypothesis:** the LLM system + user prompts have *diverged* from what the Endgame page
shows, after the v1.17–v1.21 stat-rigor milestones.

**Reality after grounding (the page + concepts section + payload):** the metric *set* is
essentially aligned. An Explore agent initially flagged three "orphan" payload fields as
absent from the page; the user corrected all three — they are all on the page:

- `endgame_elo_gap` → shown as **Endgame ELO / Non-Endgame ELO** in the ELO Timeline (logistic
  spread around Actual ELO; `Endgames.tsx` concepts section).
- `score_gap_conv` / `score_gap_parity` / `score_gap_recov` → shown as **Eval Score Gap** in
  **two** places: the per-phase Endgame Metrics section *and* per-sequence-type in the Endgame
  Type Breakdown.
- `achievable_score_gap` → shown as **Eval Score Gap** across overall endgames (Endgame Score −
  Entry Eval Score) in Overall Performance.

**Lesson for planning:** the Explore agent's frontend mapping was wrong twice. Re-verify any
"present/absent in payload" claim against the live code at plan time, don't trust a prior agent
summary. The authoritative vocabulary source is the **Endgame Statistics Concepts** accordion in
`frontend/src/pages/Endgames.tsx` (~lines 382–580): Score, Score Gap, Eval, Eval Score, Eval
Score Gap, Conversion, Parity, Recovery, Entry Eval, Entry Eval Score, Endgame Score, Endgame
Score Gap, Endgame ELO / Non-Endgame ELO, Percentile Badges, Color Zones, Benchmark.

## The real gap (what Phase 102 actually fixes)

Since the metric set is aligned, the genuine divergence is the **statistical-rigor layer** the
page surfaces but the LLM payload is blind to, plus stale prompt vocabulary, plus time-pressure
narration that was never wired up:

1. **Percentile annotations** — the page shows percentile badges as a headline feature (full
   5-step methodology in the concepts section), but the LLM payload carries no percentile data.
   The model cannot reference the number the user is staring at.
2. **LLM time-pressure narration** — Phase 88's time-pressure surfaces were explicitly left as
   "future phase: LLM narration of time pressure" (see `time-pressure-stats-rework.md` §Out of
   scope). That future phase is now Phase 102.
3. **Prompt vocabulary drift** — the system prompt's UI-vocabulary mapping needs an audit against
   the *current* user-facing copy, then a clean version bump. **The audit must cover BOTH** (a) the
   Endgame Statistics Concepts accordion labels/definitions AND (b) the **tooltip info-icon popover
   bodies** (the HelpCircle-trigger Radix popovers: `MetricStatPopover`, `WdlConfidenceTooltip`,
   `EvalConfidenceTooltip`, `AchievableScorePopover`, percentile-chip tooltip, etc.). The LLM should
   narrate using the same terms and sign conventions the tooltips explain, so the report and the
   hover-help never contradict each other.

## Locked decisions (this session)

- **Core goal = "Both": full alignment + rigor.** But because the metric set is already on the
  page, "alignment" needs **ZERO new frontend cards**. Phase 102 is a pure
  LLM-payload-and-prompt phase. This is a meaningful scope reduction vs what the agent implied.
- **Percentile annotations: YES.** Highest user value, lowest risk. Feed the **page-level,
  game-count-weighted** percentile per metric (the same value the page-level chip shows), so LLM
  and chip agree. Per-TC Time-Pressure chips render their own TC percentile directly (no
  weighting) — match that if time-pressure metrics are narrated per-TC.
- **P-values: NO.** Riskiest, largely redundant with the zone band, and closest to the parallel
  significance signal the `feedback_llm_significance_signal` decision warned against. Do not add.
- **CI bounds: LIKELY NO (user uncertain → recommend out; final lock at discuss-phase).**
  Rationale: the payload already carries `quality` (rich/adequate/thin) + `within-noise` +
  `[near edge]` markers, which cover the precision/hedging dimension CI would inform. Raw bounds
  push the model toward reciting "95% CI [54%, 66%]" — exactly the jargon the popover-copy and
  `feedback_llm_prompt_design` memories steer away from.
- **Zone band stays the GATE.** Preserve `feedback_llm_significance_signal`: the cohort `zone`
  field decides *whether* a metric is narrated; percentile only informs *how* it is phrased once
  the zone has opened the gate. Percentile must NOT double up as a second significance signal
  (don't narrate a "typical"-zone metric just because the percentile is extreme).
- **Time-pressure surfaces to narrate (user-specified):**
  1. **Score Gap by Remaining Time** — the per-bucket decomposition (note: confirm bucket count
     at plan time; `time-pressure-stats-rework.md` describes 5 quintiles of base-clock-remaining,
     the user referred to "4 quintiles" — verify against the live chart before wiring).
  2. **Clock Gap** (`avg_clock_diff_pct`).
  3. **Net Flag Rate** (`net_timeout_rate`).
  Clock Gap + Net Flag Rate scalars MAY already be in the payload; the genuine add is the
  per-bucket Score-Gap-by-time decomposition that Phase 88.1 stripped. Verify current payload
  contents at plan time rather than trusting the agent's "Phase 88.1 removed it" claim.
- **Prompt version bump** from `endgame_v35` → `endgame_v36`; cache invalidation is automatic via
  the `_PROMPT_VERSION` cache key (`app/services/insights_llm.py`). Do not retroactively
  invalidate prior cached reports beyond that mechanism.
- **More narration freedom in "Data Analysis" (the `overview` field).** The report renders three
  stacked cards in `frontend/src/components/insights/EndgameInsightsBlock.tsx` — Player Profile,
  **Data Analysis** (`:286`), Recommendations. "Data Analysis" is the UI title for the LLM
  `overview` field, currently capped at ~300 words / 1-3 paragraphs in `endgame_insights.md`.
  Relax that cap so the model **may** narrate more *when there is enough genuinely interesting
  signal* to justify it (more metrics now in the payload = more to say). Keep the "silence is not
  valid / no fabrication / within-noise" guards — this is permission to go longer when warranted,
  not a mandate to pad. Consistent with memory `feedback_llm_prompt_design` ("longer overviews
  OK"). Final length policy locked at discuss-phase.
- **Heavy UAT.** This phase is UAT-dominated. Budget for multiple UAT passes over representative
  production users (short-history, sparse-section, full-history), same cadence used across the
  `endgame_v23 → endgame_v35` history. Treat UAT as the primary verification, not an afterthought.
