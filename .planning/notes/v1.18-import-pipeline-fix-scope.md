# Phase 90 — scoping decisions

Captured during `/gsd-explore` on 2026-05-20 so the intent behind the Phase 90 ROADMAP entry survives into `/gsd-plan-phase` (which is run after `/clear`).

## In scope (4 items)

1. **Primary leak fix** — `_flush_batch` Stage 5 rewrite to bound-param `executemany`. Critical correctness gotcha: preserve the `result_fen` None-conditional (current code sets `result_fen` only for games that have one via `case(fen_case_map, else_=None)`). A naive `executemany` with `result_fen=bindparam("rf")` would write `NULL` for every game without a fen — silent data regression. Use two executemany groups (move_count only; move_count+result_fen for games with non-null fen) or COALESCE.
2. **Session-recycle (defense-in-depth)** — per-batch `AsyncSession` in `run_import`'s loop.
3. **Orphan-job reaper** — periodic and/or on-DB-reconnect hook.
4. **Resilient failure-state recording** — bounded retry with backoff around the `except Exception` UPDATE in `run_import` (~386–410).

## Out of scope (explicit decisions)

**Atomic duplicate-import guard** — dropped. SEED-018 already demotes it from SEED-017's "load-bearing top priority" to optional UX/data-hygiene after the OOM-causation premise was disproven (single import OOMs alone). It's a real correctness/UX bug but it does not prevent recurrence of the leak. Defer indefinitely; do not gate Phase 90 on it.

**Automated regression test for the leak** — dropped. Two options were considered:
- *Statement-text invariance assertion* (cheap, structural) — catches "someone reintroduced a literal `case()`" but doesn't actually prove memory stays flat.
- *Tracemalloc/RSS growth assertion* (expensive, behavioral) — local tracemalloc run in SEED-018 already timed out at 600s reaching only batch 10, so a behavioral guard would be a 10+ minute test-suite liability.

Decision: skip both. Verify manually once in dev and once in prod after deploy. Cost of getting this wrong is another outage; cost of an automated guard is a slow test or a guard that doesn't actually guard.

## Session-recycle decision

Was initially considered "defense-in-depth, optional" (SEED-018 frames it as phase-sized and warns it touches fragile lifecycle code). Kept in scope because:
- Without a regression test, structural redundancy carries more weight.
- Test-suite impact verified negligible: ~66 import-related tests, all importing small game fixtures (1–2 batches each); per-batch session acquire against test Postgres is ~1–5 ms, well under a second of added pytest time total.
- Bounds *any* future per-connection growth (asyncpg internals, ORM identity map churn) — not just this specific leak.

## Verification plan

- **Dev:** import a real ~5k+ game chess.com account (e.g. via guest user), watch backend RSS via `docker stats` or `ps` across the full import. Expect flat RSS (vs. linear ~0.48 MB/game climb today).
- **Prod:** same after deploy, then resolve Sentry FLAWCHESS-56 (120262007) and FLAWCHESS-3Q (115610288). Watch for recurrence over the next large-import window.
