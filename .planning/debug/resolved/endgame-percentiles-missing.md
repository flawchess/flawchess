# Debug Session: endgame-percentiles-missing

## Symptoms

- **Expected behavior:** After importing games and Stockfish finishing evals, the endgames page shows endgame analysis (WDL by type, conversion/recovery, etc.).
- **Actual behavior:** Endgames page is stuck on the "Analyzing endgames" gate: "Stockfish: 41 / 41 games" with copy "Endgame analysis will be available once Stockfish finishes evaluating your games." — even though evals have long been processed (41/41 complete).
- **Error messages:** None reported (UI gate, not an error). Possible relevance: user 146 had FAILED import_jobs before the successful 41-game import.
- **Timeline:** Regression. "We had this exact problem before which I thought you fixed." So a prior fix may have been undone, incompletely covers this path, or a new code path reintroduced it.
- **Reproduction:** Impersonate user 146 in prod, open the endgames page.

## Environment

- Prod (user 146). Prod read-only DB MCP available via `bin/prod_db_tunnel.sh` → `mcp__flawchess-prod-db__query`.
- 41 games imported successfully, preceded by failed import_jobs.

## Current Focus

- hypothesis: REFINED & CONFIRMED with prod data. The real cause is a FLOOR DIVERGENCE, not a missed trigger. User 146 is ABOVE the 30-game anchor floor (33 rating-eligible rapid games → anchor row exists) but BELOW the per-metric endgame floor: only 22 of 38 rapid games reach an endgame (20 with evals), and every endgame metric (score_gap, conversion, recovery, parity, ...) has its own per-TC ≥30 inclusion floor counting ONLY games that reach an endgame. 22 < 30 ⇒ all metrics return None ⇒ zero `user_benchmark_percentiles` rows BY DESIGN. Stage A ran (committed the anchor) but score_gap was below floor so no row. `tier2` is stuck False because `percentile_ready`=False (no rows) and `below_floor`=False (above the ANCHOR floor). The `is_below_anchor_floor` escape uses the WRONG predicate: it tests anchor candidacy (33≥30) not the endgame-metric floor (22<30), so it fails to unlock a user who can never get rows.
- next_action: Implement the SETTLED-EMPTY escape (anchor row exists + zero percentile rows ⇒ Stage A ran, user too sparse for any metric ⇒ unlock). Do NOT use a recompute trigger as the primary fix — for user 146 it would write zero rows again (22<30 unchanged) and re-schedule every 3s forever (runaway loop). Optional defensive recompute only for the disjoint no-anchor sub-case (Stage A genuinely never ran), which is loop-safe because the committed anchor terminates it.

## Eliminated

- Missed Stage A/B trigger as the cause for user 146 — NO (REVISED). Stage A DID run (anchor committed at 16:51:16). The earlier "missed trigger" hypothesis was superseded by the floor-divergence data: even a perfectly-timed compute writes zero rows because the endgame population (22) is below the per-metric floor (30). A missed trigger is a real bug-class member but is NOT user 146's cause.
- Recompute trigger in get_readiness as the fix — REJECTED. For the stuck state (anchor exists, no rows) the endgame population is provably below floor, so recompute writes nothing, `needs_recompute` stays True, and the 3s poll re-schedules forever (runaway background compute against prod). Replaced by the settled-empty unlock escape.
- Prior fix regressed/reverted — NO. `99f94a61` is intact in the working tree; the `below_floor` branch and `is_below_anchor_floor` are present and unchanged.
- Raw eval incompleteness — NO. 41/41 evaluated; `pending==0`. The Stockfish counter is cosmetic (`EvalCoverageHeader`), not the gate.
- Below-anchor-floor (the user-suggested "not enough games to cross the threshold") as the literal cause — UNLIKELY for user 146. 41 > 30 = `MEDIAN_ANCHOR_MIN_GAMES`, so the cheap global short-circuit does not fire and the per-TC probe must return False for at least one TC unless all 41 games are spread thin across TCs (possible but would itself be a genuine below-floor case the existing escape already handles). The lock therefore points at "above floor but compute never ran", not "below floor".
- Stale in-memory `stage_b_computing` flag — NO. It is an in-memory set cleared on backend restart; it cannot hold tier2 False across a deploy.
- DB-orphaned `in_progress` import job holding tier1 False — NO. `find_active_jobs_for_user` reads the in-memory registry only, which is empty after restart.

## Resolution

- root_cause: FLOOR DIVERGENCE between the rating-anchor floor and the per-metric endgame-population floor. The endgames page gate is `tier2` from `GET /imports/readiness`, which unlocks only on `percentile_ready` (≥1 `user_benchmark_percentiles` row) or the below-anchor-floor escape. User 146 is ABOVE the 30-game anchor floor (33 rating-eligible rapid games → an anchor row was committed by Stage A) but BELOW every endgame metric's per-metric ≥30 population floor: only 22 of 38 rapid games reach an endgame (20 with evals). Each metric (`score_gap`, conversion, recovery, parity, ...) counts only games that reach an endgame, so all return None and Stage A/B write ZERO percentile rows BY DESIGN. `percentile_ready` is therefore False forever, and `is_below_anchor_floor` returns False (it tests anchor candidacy, 33≥30, not the endgame floor 22<30) — so neither unlock condition fires and the page locks permanently. The failed-then-retried imports are NOT the cause (Stage A demonstrably ran — anchor committed at 16:51:16); they were a red herring that made "missed trigger" look plausible. The prior fix (99f94a61) handled the no-anchor below-floor variant (user 145); this is the has-anchor-but-sparse-endgames variant the predicate missed.
- fix: APPLIED. Settled-empty unlock escape (NOT a recompute — recompute would write zero rows again because 22<30 is unchanged, and would re-schedule every 3s forever). A committed anchor row proves `compute_stage_a` ran and committed (Stage A writes anchors + the eval-independent `score_gap` percentiles in one transaction), so "anchor row exists AND zero percentile rows" means no metric can ever qualify (score_gap's game set is a superset of every Stage B metric's set) → unlock.
