## Why

Phase 94.2 swapped the percentile chip from 94.1 per-cell methodology to a
pooled-per-user CDF. The dev DB has been re-backfilled under the new
methodology, but production `user_benchmark_percentiles` rows still carry
94.1 values that no longer reflect what the new `GLOBAL_PERCENTILE_CDF`
expects. Without the prod rerun, the chip would render stale percentile
values against the new CDF — a silent miscalibration.

This is a HUMAN-UAT step per CONTEXT.md D-3a: production data touches need
an explicit hand on the SSH tunnel.

## What

Operator-run from the local repo (NOT inside Claude Code):

```bash
# 1. Open prod tunnel
bin/prod_db_tunnel.sh

# 2. Confirm BACKFILL_PROD_DB_URL is set and includes ":15432"

# 3. Run backfill, log to disk
uv run python scripts/backfill_user_percentiles.py --target prod \
  | tee .planning/phases/94.2-pooled-per-user-percentile-redesign/94.2-06-prod-backfill.log

# 4. Close tunnel
bin/prod_db_tunnel.sh stop
```

## Verify

1. The summary table shows the new pooled counter:
   `score_gap upserted=N/skipped_below_pooled_floor=M`
   (and same per metric for `achievable_score_gap`, `section2_score_gap_conv`,
   `section2_score_gap_parity`).
2. Zero `FAILED users` lines.
3. Spot-check a known prod user via `mcp__flawchess-prod-db__query`:
   ```sql
   SELECT metric::text, value, percentile, n_games, cdf_snapshot
   FROM user_benchmark_percentiles WHERE user_id = <id> ORDER BY metric;
   ```
   - `value` differs from the 94.1 row (methodology shift expected).
   - `cdf_snapshot` = the day of the prod rerun.
4. Visual UAT on https://flawchess.com Endgames page — percentile chips
   render with plausible values and the new 4-bullet popover disclosure.

## When

Before the next `main → production` release that ships Phase 94.2. If
deferred past that point, the chip will display stale 94.1 values against
the 94.2 CDF until the rerun happens.
