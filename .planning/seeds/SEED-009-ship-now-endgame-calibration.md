# SEED-009: Ship-now endgame gauge calibration (pre-SEED-006 cleanup)

## Why This Matters

The 2026-04-29 benchmark report (`reports/benchmarks-2026-04-29.md`) surfaced four calibration findings that do **not** depend on Stockfish-eval validation, rating-bucketed zones, or the skill v2 collapse verdict. They can ship as a quick UI/constants PR without waiting for the full SEED-006 milestone (which needs the classifier-validation gate, rating-stratified offset analysis, and Parity validation — weeks of work).

Splitting these out gives users visible calibration improvements within days while SEED-006 proceeds independently.

## What's In Scope

### 1. Endgame Skill upper bound widening

- File: `frontend/src/components/charts/` (consult `ENDGAME_SKILL_ZONES` constant) and the autogenerated `frontend/src/generated/endgameZones.ts` pipeline (`scripts/gen_endgame_zones_ts.py`).
- Change: upper neutral bound `0.55` → `0.57`.
- Rationale: pooled population p75 = 0.5715. Current band labels the upper quartile of users as "success" — too generous. New bound aligns "success" with the top quartile.
- One-line constant edit if generated, plus regen.

### 2. Per-endgame-class conv/recov neutral zones

- File: `frontend/src/components/charts/EndgameConvRecovChart.tsx` (no per-class zones today).
- Add per-class neutral bands derived from pooled per-class rates ± 5pp (Section 6 of the report has the candidate values):
  - rook: conv `[0.62, 0.72]`, recov `[0.28, 0.38]`
  - minor_piece: conv `[0.60, 0.70]`, recov `[0.32, 0.42]`
  - pawn: conv `[0.64, 0.74]`, recov `[0.27, 0.37]`
  - queen: conv `[0.70, 0.80]`, recov `[0.21, 0.31]`
  - mixed: conv `[0.63, 0.73]`, recov `[0.29, 0.39]`
  - pawnless: conv `[0.69, 0.79]`, recov `[0.22, 0.32]`
- Constants must flow through `app/services/endgame_zones.py` registry → codegen → TS, NOT hard-coded in the React component.
- Pooled bands wouldn't fit (9–10pp spread between classes); per-class is required.

### 3. Clock-pressure per-TC thresholds

- File: `frontend/src/components/charts/EndgameClockPressureSection.tsx` + `frontend/src/generated/endgameZones.ts`.
- Change `NEUTRAL_PCT_THRESHOLD` from a single ±10pp to per-TC: bullet ±4pp, blitz ±6pp, rapid ±8pp, classical ±12pp. (Or accept narrower-but-still-pooled ±7pp as a compromise; per-TC is more honest.)
- Change `NEUTRAL_TIMEOUT_THRESHOLD` from a single ±5pp to per-TC: bullet ±10pp, blitz ±7pp, rapid ±3pp, classical ±1pp. **Single threshold cannot fit** — bullet's p75 is +13.7pp net timeout, classical's is +0.9pp. Current ±5 is wrong by 2× in opposite directions.
- Larger UI plumbing change than #1/#2 — need TC-aware lookup at gauge render time. Worth its own discuss-phase.

### 4. Time-pressure-vs-performance per-TC display

- File: `app/services/endgame_service.py` (`_compute_time_pressure_chart` currently pools all TCs into a single user_series + opp_series).
- Backend change: emit per-TC series (4 user_series, 4 opp_series) or split into chart variants.
- Frontend change: `EndgameTimePressureSection.tsx` consumes per-TC data, displays as overlay or per-TC chart.
- Rationale: pooled chart hides a 13pp gap at extreme time pressure (bucket 0: bullet 0.252 vs classical 0.380). Pooling smooths exactly the buckets users care about.
- Alternative if UI complexity is a concern: keep pooled view as default, add per-TC overlay behind a "details" disclosure.
