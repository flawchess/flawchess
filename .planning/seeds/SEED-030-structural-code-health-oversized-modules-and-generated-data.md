# SEED-030: Structural code-health — split oversized modules + demote generated CDF from source to data asset

**Status:** Open
**Created:** 2026-05-30
**Source:** `/gsd-map-codebase` run (2026-05-30), `.planning/codebase/CONCERNS.md` — repo-quality review. Points 1 and 3 of the post-mapping quality assessment.
**Related:** SEED-027 (DB memory budget guardrail — shares the `import_service.py` hot path), CLAUDE.md "Keep functions small and shallow" + split-recipe rules.

---

## One-liner

Two independent structural-debt tracks surfaced by the codebase map: (A) a handful of modules have grown an order of magnitude past the project's own stated file/function limits and concentrate too many concerns; (B) `app/services/global_percentile_cdf.py` is a 97.6k-line generated lookup table living as Python source, parsed into the heap on every process start. Both are *known, contained* maintainability debt, not bugs. This seed captures them for later phase planning.

---

## Context

The 2026-05-30 codebase mapping rated the repo B+/A−: strong fundamentals (layered architecture, enforced conventions, real-Postgres tests, ty/knip/ruff CI gates), with the main quality drag being concentrated structural debt. Two items stood out as the most actionable and are captured here. Neither is urgent; both are good candidates for discrete GSD refactor phases (per CLAUDE.md "refactor on sight / but flag if out of scope" — these are too large for opportunistic in-task work).

## Track A — Oversized, multi-concern modules

### Problem

Several files breach CLAUDE.md's logic-LOC limits (soft 100 / hard 200) by a wide margin and mix many concerns, making them high cognitive load, merge-conflict magnets, and hard to test per-branch. Worst offenders (LOC at mapping time):

**Backend:**
- `app/services/endgame_service.py` — **3462** (largest hand-written backend module by far; top priority)
- `app/services/insights_llm.py` — **2188** (also carries a ~9 KB single-line `_PROMPT_VERSION` changelog-comment at line 68 — see Track A note below)
- `app/services/insights_service.py` — 1355
- `app/services/canonical_slice_sql.py` — 1115
- `app/services/opening_insights_service.py` — 997
- `app/repositories/endgame_repository.py` — 949
- `app/services/import_service.py` — 929 (overlaps SEED-027; touch carefully)

**Frontend:**
- `frontend/src/pages/Openings.tsx` — **1232**
- `frontend/src/pages/Endgames.tsx` — **965**
- `frontend/src/App.tsx` — 634
- `frontend/src/pages/Home.tsx` — 627

### Proposed approach

Apply the CLAUDE.md split recipes already documented for exactly these shapes:
- **Pipeline orchestrators** (insights, import, normalization): one function per stage (`_fetch`, `_classify`, `_attribute`, `_dedupe`, `_rank`), top-level reads as a list of stage calls.
- **React pages mixing data + JSX**: extract data shaping into `useXyzData` hooks; split desktop/mobile renderers into siblings when both branches exceed ~40 LOC of *logic*; pull large Recharts subtrees into named sub-components only where there's real reuse/complexity-hiding value.
- Heed the CLAUDE.md caveat: don't invent context dataclasses to make signatures fit; if a split needs a state-bag threaded between helpers that always run together, the seam is wrong — split along a different one.

Sequence by risk/payoff: `endgame_service.py` first (biggest, highest-churn), then the two big frontend pages (`Openings.tsx`, `Endgames.tsx`), then the remaining backend services. Treat each module (or small cluster) as its own phase with its tests green before/after — these are behavior-preserving refactors, so the existing suite is the safety net.

**Sub-item:** move the ~9 KB `_PROMPT_VERSION` v14→v35 changelog out of the `insights_llm.py:68` source comment into `CHANGELOG.md` or an `app/prompts/` doc. The version string is a cache key (bumping it busts LLM report caches), so keep the literal but relocate the unreviewable inline history.

## Track B — Demote `global_percentile_cdf.py` from source to data asset

### Problem

`app/services/global_percentile_cdf.py` is **97,598 lines** — a generated `COHORT_PERCENTILE_CDF` lookup table (~99 percentile breakpoints per (metric, anchor, TC) cell) embedded as Python source. It dominates backend LOC, is parsed into the Python heap on every process start, and is imported transitively via the percentile services into the insights path. Regenerated by `scripts/gen_global_percentile_cdf.py`.

This is data masquerading as code: it inflates the repo, skews LOC/quality metrics, and adds import-time parse cost + heap footprint to every cold start (and every test process).

### Proposed approach

Move the table to a binary/columnar data asset loaded lazily at first use rather than imported as source:
- Candidate formats: `.npz` / `.parquet` (numpy/pandas already in the data-science stack), or a dedicated read-only DB table.
- Keep `scripts/gen_global_percentile_cdf.py` as the generator; change its output target from a `.py` module to the chosen asset.
- Provide a thin typed accessor (e.g. `percentile_cdf.lookup(metric, anchor, tc) -> Sequence[float]`) so call sites don't change shape.
- Lazy-load + cache (module-level memoization) so import cost is paid once on first lookup, not at process start — and not at all in tests that don't touch percentiles.

### Open questions (Track B)

- Where does the asset live and how is it shipped? (committed binary in repo vs build-step generated vs seeded into the DB on migration). Affects deploy/`entrypoint.sh`.
- Is the table hot enough that full in-memory load on first use is fine, or should lookups hit disk/DB per cell? (Likely full load is fine — it's a bounded table.)
- Does CI's drift check (the project already CI-checks generated artifacts like `endgameZones.ts`) need an equivalent for the new asset format?

## Acceptance criteria sketch

**Track A:**
- Targeted modules brought within (or substantially toward) CLAUDE.md limits via behavior-preserving splits; no net loss of test coverage.
- `_PROMPT_VERSION` historical changelog relocated out of the source comment; version-string cache-key behavior unchanged.
- ruff / ty / knip / pytest + frontend lint/tests all green; no behavior change observable in UAT.

**Track B:**
- `COHORT_PERCENTILE_CDF` no longer a multi-10k-line `.py` module; served from a data asset via a lazy typed accessor.
- Process start no longer parses the table; percentile/insights outputs byte-for-byte (or within documented eval non-determinism) unchanged.
- Generator script updated; any CI drift check covers the new asset.

---

## Recommendation

Both tracks are LOW-urgency, HIGH-clarity refactors with the existing test suite as a safety net. Suggest scheduling as separate phases (Track A possibly as 2–3 phases given the volume; Track B as one self-contained phase) when there's refactor bandwidth between feature milestones. Not to be planned without explicit go-ahead.
